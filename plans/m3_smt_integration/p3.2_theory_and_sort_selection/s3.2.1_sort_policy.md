## Context

[Q26 working answer](../../open_questions.md#q26--type-mapping):

This stage formalises the policy and audits it on real IR files.

## Tasks

### Task T3.2.1.1 — Pick the policy

`src/ein/smt/sort_policy.py`:

```python
def pick_sort(t: IRType) -> SmtSort:
    if t.positional:
        return SmtSort.int_bounded(t.bounds)
    if t.arity == 0 and len(t.instances) <= 32:
        return SmtSort.datatype(t.instances)
    if t.has_arithmetic:
        return SmtSort.int_unbounded()
    return SmtSort.uninterpreted(t.name)
```

Branching by `has_arithmetic` is for types with arithmetic
constraints attached (e.g. `(* (count …) (count …))` on a future
puzzle); for now everything Zebra-shaped is positional or enum.

Useful when adding a new puzzle: scan the output, sanity-check that
no surprise uninterpreted sorts appear.
