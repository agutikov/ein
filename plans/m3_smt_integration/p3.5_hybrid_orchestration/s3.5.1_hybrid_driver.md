## Tasks

### Task T3.5.1.1 — `(hard-slice …)` IR form

In the IR spec (P1.1): a new form

```ein
(hard-slice
  :scope (and (rel ?r ?a ?b) :where (in arithmetic-rel ?r))
  :reason "global arithmetic; defer to solver")
```

Parser passes it through; M1 engine recognises it as a routing
directive (no semantics in the graph engine — the slice is a hint).

### Task T3.5.1.2 — Driver

`src/ein_bot/engine/hybrid.py`:

```python
def solve_hybrid(g: Graph, registry: Registry, query: Query) -> Verdict:
    while not (verdict := graph_solve_step(g, registry, query)):
        slices = collect_hard_slices(g)
        for s in slices:
            subgraph    = extract_subgraph(g, s)
            subgraph_ir = to_ir(subgraph)
            smt2        = translate(subgraph_ir)
            res         = backend.check(smt2)
            facts       = lift_model(res.model, subgraph_ir, ...)
            g.merge(facts, provenance_kind='smt')
        if not slices:
            break
    return verdict
```

The graph engine runs as usual; whenever it stalls *and* there is
a `(hard-slice …)` annotation in scope, the slice is extracted,
translated, solved, and lifted. M1's saturator then continues on
the augmented graph.

### Task T3.5.1.3 — Conflict resolution

If SMT solves a slice with a value that contradicts a graph-derived
fact: that's a contradiction at the *whole-problem* level. The
hybrid driver surfaces it as a `Contradiction` verdict with the
SMT slice + the graph derivation paths as competing provenance.
