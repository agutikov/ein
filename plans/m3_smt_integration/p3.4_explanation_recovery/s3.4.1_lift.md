# S3.4.1 — Lift model + unsat core to IR

**Phase:** P3.4 (explanation recovery).
**Estimate:** 3-4 days.
**Depends on:** P3.3.
**Resolves:** [Q29](../../open_questions.md#q29--unsat-core-lift).

## Context

The translator produces SMT-LIB; the solver produces a model or
unsat core; this stage reverses the translation, producing IR
edges (with provenance) the M1 trace renderer can already handle.

## Tasks

### Task T3.4.1.1 — Model → reasoning-layer edges

`src/ein/smt/lift.py::lift_model(model, ir, name_map)`:

- For each goal variable's bound value, produce a corresponding
  `(rel …)` edge in the *reasoning* layer.
- `Provenance(kind='smt', backend='z3', solver_id=…, smt2_sha=…)`.
- The full smt2 text is archived under `out/smt-<sha>.smt2` for
  reproducibility.

### Task T3.4.1.2 — Unsat core → derivation DAG

Translate each `:named` label back to its IR source fact (via the
P3.3 name map). Build a `DerivationDAG` with a synthetic root
labelled `(smt-unsat …)` whose premises are the source facts
named in the core.

### Task T3.4.1.3 — Trace formatting

When the trace renderer (M1.P1.6) sees an SMT-kind provenance, it
emits a section:

```markdown
## Step 17 — SMT solve (z3)

> Z3 returned `sat` after 12 ms on the slice covering
> `(House_*, Color)` constraints.

Source assertions used: condition (5), condition (2), …

[full smt2 ↗](smt-3f8a1.smt2)
```

Concise: doesn't dump the SMT model into the prose; links to the
side artefact.

### Task T3.4.1.4 — Tests

`tests/smt/test_lift.py`:

- A Zebra solve via SMT lifts back to an IR with reasoning-layer
  edges matching the canonical answer.
- A `zebra-bad.ein` unsat-core lifts to a DerivationDAG with at
  least the injected bad-fact and condition (5).

## Acceptance

- The lift round-trips cleanly: `lift(check-sat(translate(ir))) ⊇ goal`.
- The SMT-derived trace is *readable* — no SMT-LIB jargon escapes
  into the markdown prose.
