## Lattice re-grounding (2026-06-15)

LCV is a **value-ordering** heuristic — in DPLL it picks the value of the
*current decision variable* that prunes neighbours least. The engine has no
per-variable decision: it enumerates commitment **sets** by cardinality
(Apriori BFS, `architecture_and_algorithms.md` §O7); the only ordering surface
is `apriori.order_candidates` / `hypgen.score_hypothesis` over the layer-1
atoms.

Crucially, **within-layer order cannot change what the complete search
prunes**: all of layer k−1 is processed before layer k, so k−1's nogoods are
available to k regardless of order, and same-size sets can't subsume each
other. So a value order only changes *which solution the fast
`solve(stop_after=1)` path returns first* — it is **inert for the
uniqueness-establishing complete `solve()`** (which exhausts the pruned
lattice anyway). Re-scope E9 to a **fast-path-only** find-first heuristic: no
leverage on the complete search, and none on wall-time there.

## Context

Classic CSP heuristic: when picking a value for the current
variable, prefer the value that prunes the fewest options
from neighboring variables. Composes with M1's existing
most-constrained-object first (S1.5.4 A2) — A2 picks the
variable; LCV picks the value.
