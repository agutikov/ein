## What already ships (2026-06-15)

This entry reads as fully greenfield, but the CDCL-flavoured core landed with
the set-search engine (`S1.5b.6`) and is live:

- **Multi-literal learned clause** — `Clause = frozenset[FactId]`
  (`nogoods.py:52`), stored in `KnowledgeBase._nogoods` (`store.py:166`), kept
  subsumption-minimal on insert. Exactly T1.9.E7.1's "set of literals that
  can't all hold," and **separate from the single-fact `_negated_facts` index**
  (`store.py:203`) — also as T1.9.E7.1 specified.
- **Generation on death** — `emit_nogood(root_kb, frozenset(c), min_size=1)`
  at the dead-commitment handler (`solver.py:805`).
- **Consumption pre-fork** — candidate sets are filtered against `_nogoods`
  *before* any fork/saturate via `apriori.filter_candidate`'s subset test
  (`apriori.py:90`, called from `generate_layer`, `solver.py:1050`) — the
  downward-closure principle. Realises T1.9.E7.3's intent, but at the
  **assembled-set** level, not the per-prospective-hypothesis level the task
  described.

**The genuine residual** is narrow:
- **a.** today's clause is the branch's **path condition** (the committed
  set), *not* a clause derived from the **unsat-core** — E7's actual ask
  (`DeadCommitment.learned_clause == frozenset(commitment)`, `contract.py:102`;
  write-only / diagnostic, never read back to prune).
- **b.** it is not **minimised** ([E19](s1.9.e19_unsat_core_min.md)).
- **c.** it does not persist across calls ([E20](s1.9.e20_conflict_cache.md)).

So E7 is best **re-scoped to "MUS-derived, minimised nogoods"** (folding in
[E6](s1.9.e6_transitive_premise_walk.md)'s premise walk + E19's minimiser), not
"learned clauses from scratch."

> **Doc note:** `nogoods.py`'s module docstring advertises `matches_any_nogood`
> + `build_clause` as the consumption path — both are **dead code** (defined,
> never called); the live path is `apriori.filter_candidate`. Worth fixing when
> this entry is touched.

## Context

When a contradiction fires, the unsat-core (a conjunction of
source facts) is itself a "learned constraint". Future
hypotheses can be tested against it directly, without
re-running the rules that produced the contradiction.

## Tasks

### Task T1.9.E7.1 — Learned-clause representation

A learned clause is a set of `(rel, args)` literals that
*can't all hold simultaneously*. Stored alongside hypgen's
`_negated_facts` index — separate index because the clause is
multi-literal, not a single negation.

### Task T1.9.E7.2 — Generation

After a contradiction fires:

- **a.** Walk `Provenance` (using [E6](s1.9.e6_transitive_premise_walk.md))
  to find the unsat-core.
- **b.** Minimise it (via E19).
- **c.** Store as a learned clause.

### Task T1.9.E7.3 — Consumption

In hypgen, before emitting a candidate, check if adding it
would make any learned clause fully-satisfied → reject.

### Task T1.9.E7.4 — Tests + bench

Hard puzzles: zebra2 + harder. Bench vs E20 (which persists
across `solve()` calls).

## Acceptance

- Learned clauses stored, consumed by hypgen.
- Bench shows ≥ 20% node-count reduction on a contradiction-
  heavy puzzle.
