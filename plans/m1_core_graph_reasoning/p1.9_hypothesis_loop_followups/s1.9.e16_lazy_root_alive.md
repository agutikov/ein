# S1.9.E16 — Lazy `root_alive` materialisation

**Phase:** P1.9, §engineering / UX.
**Estimate:** S (≤ ½ day).
**Value:** L (memory).
**Status:** 📅 parked.
**Catalog row:** [P1.9 README — Engineering / UX](README.md#engineering--ux)
E16.

## Context

T1.5.4.5 materialises the root alive set as a `frozenset[…]`
for hash + reuse. On huge puzzles (M2 NL output), that
frozenset may dominate memory. Alternative: keep it as a
stream / generator with lazy materialisation.

## Tasks

### Task T1.9.E16.1 — API choice

Decide: do callers need a hashable set (e.g., for the
state-hash dedup), or can they accept a stream?

### Task T1.9.E16.2 — Implementation

If stream: replace the frozenset materialisation with a
generator + a "size hint" attribute. The hash consumer
materialises on demand.

### Task T1.9.E16.3 — Tests + bench

Memory bench on a synthetic large puzzle (~10k facts).

## Acceptance

- Memory savings demonstrated on the large synthetic.
- Existing tests unaffected.
