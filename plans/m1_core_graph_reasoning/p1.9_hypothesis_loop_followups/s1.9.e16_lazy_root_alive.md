## Re-grounding (2026-06-03)

Two premises here are stale after P1.7a/b + S1.7.24:

- **There is no `root_alive`.** The alive set is now computed per-KB by
  `_compute_alive(kb) -> frozenset[FactId]`
  (`inference/monotonic/solver.py:455`) — a frozenset of *fact ids*,
  recomputed, not a persistent inherited structure.
- **The hash-reuse rationale is gone.** `state_hash` keys on the KB facts
  only, **not** the alive set (see
  [E22 § Resolution](s1.9.e22_alive_hyps_in_state_hash.md#resolution-2026-06-03)).
  So T1.9.E16.1 ("do callers need a hashable set for the state-hash dedup?")
  is **answered: no** — the dedup never consumes the alive frozenset.

What (weakly) survives: streaming `_compute_alive` to avoid materialising a
large `frozenset[FactId]` on huge puzzles. Re-measure first — it's a fact-id
frozenset, not the heavier `CanonicalSetId` structures, so the memory win is
smaller than the original framing assumed.

## Context (original, 2026-05-24 — pre-P1.7a)

T1.5.4.5 materialises the root alive set as a `frozenset[…]`
for hash + reuse. On huge puzzles (M2 NL output), that
frozenset may dominate memory. Alternative: keep it as a
stream / generator with lazy materialisation.
