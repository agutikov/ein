## Resolution (2026-06-03)

**The shipped engine keys dedup on the KB alone — it does *not* include the
alive set.** `canon.state_hash` (`inference/canon.py:27-46`) hashes only the
KB's propositional fact list (outer set canonicalised, args order preserved):
*"different proof paths leading to the same closed KB should collapse to one
search-tree node."* So the engine took the **"KB ⇒ alive-set"** branch of the
question below — alive is treated as a pure function of the closed KB — and
[S1.7.24](../p1.7_bootstrapping_zebra/s1.7.24_dehardcode_symmetric.md)'s
generic state-hash-keyed lattice (plus its "gaps same-model over-count" fix)
relies on exactly that.

Consequence for this stage: **T1.9.E22.2 (extend the hash with the alive set)
was *not* taken** and is not planned. What remains is **T1.9.E22.3** — write
the KB-only-dedup invariant into
[`docs/kernel/inference/`](../../../docs/kernel/inference/) so the assumption
the lattice relies on is on the record. (If a counter-example to "KB ⇒
alive-set" is ever found, it is a *soundness bug in the shipped dedup*, not a
deferred optimisation — reopen then.)

## Context (original open question, 2026-05-24)

Open question: can two distinct paths reach the same post-
saturation KB but carry **different alive hypothesis sets**?

Under M1's monotonic semantics (append-only KB, no fact
retraction, rules don't depend on path history), the
saturated KB IS a pure function of its fact set — so the
alive set at exit should be a pure function of the KB as well,
regardless of how the KB was reached. *If* that's true, the
current state-hash dedup is sound. *If* not, the dedup is
unsound for the [T1.5.4.5/8 alive-inherit](../p1.5_hypothesis_loop/s1.5.4_hypgen_improvements.md)
optimisation and the fix is to include the alive set in the
canonical hash.
