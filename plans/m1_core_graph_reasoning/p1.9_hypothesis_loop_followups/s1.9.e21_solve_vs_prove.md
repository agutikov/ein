## Superseded by P1.7a (2026-06-03)

This stage proposed splitting one `Mode` into a fast `Mode.SOLVE` and an
exhaustive `Mode.PROVE`. **[P1.7a](../p1.7a_solution_search_refactor/README.md)
(2026-05-31) resolved the same intent a different way, so the shape below did
not ship:**

- **The exhaustive/uniqueness side is now `solve()`.** The completeness
  rework made `solve()` the sound entry — solution = consistent ∧ complete,
  returning `Solution` / `Ambiguity` / `Contradiction`
  (`inference/verdict.py:63/78/92`) with the uniqueness verdict (1 unique →
  Solution, >1 → Ambiguity, 0 → ⊥). That **is** the "exhaustive, with
  uniqueness witness" E21 wanted from `Mode.PROVE` — no new mode required.
- **The fast side already exists** as the monotonic `solve(stop_after=1)`
  path (fast-but-unsound; uniqueness undetermined) — E21's `Mode.SOLVE`.
- **`Mode` was not retired, but its axis is different.** It lives in
  `inference/verdict.py` (not `inference/solver.py`, the path T1.9.E21.1
  cites — there is no `inference/solver.py`), and its three values are
  `SOLVE` / `GAPS` / `CONTRADICTIONS` — idea-03's **task classes**, *not* a
  SOLVE/PROVE exhaustiveness axis. Adding `Mode.PROVE` would cut across it.

**Residual (if any):** an explicit, verdict-level "uniqueness undetermined"
flag for the fast path, *if* the monotonic entry doesn't already surface it.
Everything else here is closed.

## Context (original, 2026-05-24 — pre-P1.7a)

`Mode.SOLVE` today is *operationally* PROVE — alive-branch
termination (S1.5.3) requires fully exploring every alive
branch to ensure uniqueness. The user's framing 2026-05-24:

- **solve** — find any answer ASAP; use heuristics to
  short-circuit; uniqueness undetermined.
- **prove** — exhaustive; the current default.

This stage splits them as distinct modes.

## Tasks (superseded — kept for the record; do not implement as written)

### Task T1.9.E21.1 — Add `Mode.PROVE`

`inference/solver.py` Mode enum gets `PROVE = "prove"`.
Existing `SOLVE` semantics moves under `PROVE`; `SOLVE`
becomes the find-fast variant.

### Task T1.9.E21.2 — Find-fast termination

Under `Mode.SOLVE`: return on first `Solution` verdict, even
if other alive branches remain. The verdict's payload flags
uniqueness as "undetermined".

### Task T1.9.E21.3 — Tests

Smoke: a puzzle with multiple solutions returns one under
`Mode.SOLVE`, returns `Ambiguity` (with both) under
`Mode.PROVE`.

### Task T1.9.E21.4 — Migration

Audit existing call sites: do they want SOLVE-fast or
PROVE-exhaustive? Most M1 tests want PROVE (they assert
uniqueness). Migrate.

## Acceptance

- `Mode.PROVE` ships.
- `Mode.SOLVE` returns faster on multi-solution puzzles.
- Existing tests updated to use PROVE where they need
  uniqueness.
