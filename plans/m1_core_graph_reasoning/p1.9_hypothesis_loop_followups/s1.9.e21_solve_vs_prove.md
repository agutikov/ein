# S1.9.E21 — `solve` vs `prove` mode split

**Phase:** P1.9, §mode taxonomy (added 2026-05-24).
**Estimate:** M (1-3 days).
**Value:** M (UX).
**Status:** 📅 parked.
**Catalog row:** [P1.9 README — Mode taxonomy](README.md#mode-taxonomy--state-hash-design-added-2026-05-24)
E21.

## Context

`Mode.SOLVE` today is *operationally* PROVE — alive-branch
termination (S1.5.3) requires fully exploring every alive
branch to ensure uniqueness. The user's framing 2026-05-24:

- **solve** — find any answer ASAP; use heuristics to
  short-circuit; uniqueness undetermined.
- **prove** — exhaustive; the current default.

This stage splits them as distinct modes.

## Tasks

### Task T1.9.E21.1 — Add `Mode.PROVE`

`inference/solver.py` Mode enum gets `PROVE = "prove"`.
Existing `SOLVE` semantics moves under `PROVE`; `SOLVE`
becomes the find-fast variant.

### Task T1.9.E21.2 — Find-fast termination

Under `Mode.SOLVE`: return on first `Solution` verdict, even
if other alive branches remain. The verdict's payload flags
uniqueness as "undetermined".

### Task T1.9.E21.3 — Tests

Smoke: a puzzle with multiple solutions returns one under
`Mode.SOLVE`, returns `Ambiguity` (with both) under
`Mode.PROVE`.

### Task T1.9.E21.4 — Migration

Audit existing call sites: do they want SOLVE-fast or
PROVE-exhaustive? Most M1 tests want PROVE (they assert
uniqueness). Migrate.

## Acceptance

- `Mode.PROVE` ships.
- `Mode.SOLVE` returns faster on multi-solution puzzles.
- Existing tests updated to use PROVE where they need
  uniqueness.
