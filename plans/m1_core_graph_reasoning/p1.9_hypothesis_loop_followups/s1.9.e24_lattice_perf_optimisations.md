# S1.9.E24 — Lattice perf optimisations (deferred from S1.5b.30)

**Phase:** P1.9 follow-ups (post-P1.5b deferred work).
**Estimate:** S each (½ day per optimisation if triggered).
**Status:** **deferred 2026-05-29** by
[S1.5b.30](../p1.5b_lattice_search/s1.5b.30_lattice_perf_round.md).
**Trigger:** see per-optimisation rules below.

## Context

S1.5b.30's PyPy measurement on
`examples/zebra2.ein` at `--max-set-size 4` found none of
the three optimisations below were warranted on the
shipping reference workload. The data is recorded in
[S1.5b.30 § Measurement results](../p1.5b_lattice_search/s1.5b.30_lattice_perf_round.md#measurement-results-2026-05-29);
each optimisation has its own decision-rule + the data
point that failed to trigger.

These three are catalogued here so a future puzzle / rule
set / measurement that DOES push past the trigger has a
ready landing spot rather than a re-discovery loop.

## F1.a — Subset-trie for Apriori-gen

**Source:** [Q1.5b.2.b](../p1.5b_lattice_search/open_questions.md#q15b2--frontier-strategy).
**Trigger:** zebra2 layer-3 enumeration wall-time > 1 s, or
any future fixture's `|A_{k-1}|` > 200.
**S1.5b.30 measurement:** `|A_{k-1}|` peaked at 19 alive
on zebra2 layer 1 — the naive O(N²) prefix-join is ~360
join attempts, well below the trie's break-even point.

**Implementation sketch:** replace
`inference/apriori.py:apriori_prefix_join` with a
trie-indexed structure keyed by canonical-set prefix.
Lookup becomes O(k) instead of O(N²) for joining at
layer k. Memory cost: one trie node per distinct prefix
across all layers ≈ Σ |A_i| nodes for a depth-d run.

## F1.b — Interned set ids

**Source:** [Q1.5b.2.b](../p1.5b_lattice_search/open_questions.md#q15b2--frontier-strategy)
+ [Q1.5b.3](../p1.5b_lattice_search/open_questions.md#q15b3--lattice-node-representation).
**Trigger:** lattice memory peak on any reference fixture
> 500 MB.
**S1.5b.30 measurement:** zebra2 contradictions+store_lattice
visited 56 commitments; the tuple-based `CanonicalSetId`
overhead is < 20 KB total. Not a memory-pressure
candidate at puzzle scale.

**Implementation sketch:** replace
`CanonicalSetId = tuple[FactId, ...]` with an integer id +
a process-local `dict[int, tuple]` intern. Each
CanonicalSetId becomes a single int (8 bytes vs
~120 bytes/tuple). Saves memory + faster equality /
hashing. Migration touches every CanonicalSetId-typed
field: `apriori.py`, `commitment.py`,
`monotonic/lattice.py`, the `SetNode.canonical_set` /
`labels`, the `DeadCommitment.commitment` /
`learned_clause`.

## F1.c — Commitment-set `try_commitment_set` divergence

**Source:** [Q1.5b.8](../p1.5b_lattice_search/open_questions.md#q15b8--engine-bridge).
**Trigger:** `try_commitment_set` dominates lattice runtime
> 50% via `cProfile` on a reference workload.
**S1.5b.30 measurement:** dominant cost is **saturation
itself** (160–625 ms / entering on PyPy zebra2). Diverging
the lattice's `try_commitment_set` from the shared
`inference/commitment.py` wouldn't move the needle — the
saturator is downstream of `try_commitment_set` and the
shared code paths already produce the right
fork-and-saturate-and-detect sequence.

**Implementation sketch:** copy `try_commitment_set` from
`inference/commitment.py` to
`inference/monotonic/commitment.py` and specialise — skip
the fork-and-copy when state-hash dedup is about to fire
(early state_hash on the partially-built fork's facts),
batch the per-element hypothesis writes. Cost: shared
engine simplicity (the duplicated logic must stay in sync
with the tree-side / monotonic / lattice consumers).
