## Executed (2026-06-15)

**Inventory correction first.** The pre-execution audit (table below) wrongly
marked `back_propagate` **inert**. Its *ancestor-bubble* half is inert
(`_kb_chain_ctx` is never `.set()`), but its **primary `(not h)` write is LIVE**:
`hypgen.py:157` builds its *own* `Lookahead`, so the lookahead-kill path
(`hypgen.py:223`) is reached on the live lattice solve and caches `(not h)` into
`_negated_facts` — and that cache is matchable by rules (`symmetric-negative`), so
deleting it would change behaviour. So this was a behaviour-**preserving**
refactor, not a blind deletion.

  config flags `enable_eager_root_bubble` + `enable_path_condition_nogoods`

- Danglers cleaned: the 3 monotonic tests' kwarg, the demo's `--no-kill-cache`
  flag, and the `store.py` / `solver.py` / `commitment.py` / `provenance.py` /
  `primitives.py` doc-refs.

**Validation:** `ruff` clean; **1216 fast** + **20 slow** zebra2 tests green

## Why this is a stage, not a followup

The post-S1.5b tree-solver removal (commit `8d77b02`) left `back_prop.py` — the
parent-KB / sibling / ancestor-chain back-prop machinery — as **dead/inert
residue** in the tree. This is *not* a parked "someday" kernel-minimisation
audit item: it actively **misleads** (the [S1.5.7](../p1.5_hypothesis_loop/s1.5.7_back_prop_unconditional.md)
doc and the E6 stub describe its `(not h)`-to-parent mechanism as live; it is
not, and there is no parent KB), and E6 cannot be planned honestly while a **dead
`is_unconditional_death`** sits next to the **live** walk
(`commitment._is_unconditional` + `provenance.reaches`). So: a small, immediate
refactor stage, sequenced before E6 — not an entry in the F5 audit catalogue.

The production engine is the commitment-set cardinality lattice
(`inference/monotonic/`): one root KB, commitment sets related by subset, **flat
root-writes + Apriori nogoods** (`monotonic/__init__.py`: *"no per-parent bubble;
saturation commutativity makes the bubble mechanically redundant"*). No parent
KBs, no sibling inheritance, no ancestor chain — the exact concepts `back_prop.py`
implements.
