## "Prune" in an append-only engine (2026-06-15)

The stub's "prune trivially-impossible **facts**" is CSP vocabulary that
mis-describes this engine — which is **append-only** (a saturation only *adds*
facts; retraction = fork a fresh KB). The resolution:

- A CSP *variable* here = an undecided cell (e.g. "House-3's colour"); its
  *domain* = the candidate fillers; the **alive set** (open hypotheses) is the
  candidate `(colour-of House-3 ?c)` facts `hypgen` would propose.
- "Prune value Red from House-3's domain" = establish
  `(not (colour-of House-3 Red))`. The engine does that by **adding** a
  `(not …)` REASONING fact, which `hypgen`'s Filter-C then skips. So *pruning*
  shrinks the **candidate / alive set** (the search space) by **adding
  negatives** — fully monotone, nothing is deleted. The wording should read
  "prune candidate hypotheses (by deriving their negatives)," not "prune facts."

**Stronger point — saturation already does AC-3's job.** AC-3 is generic
constraint propagation; ein-bot's **rule-driven saturation IS that propagation**
over the *same* constraints. The puzzle's elimination rules
(`functional-negative`, `sibling-exclusive`, `domain-elimination`) derive
exactly those `(not h)` negatives at root saturation, to fixpoint. So an AC-3
pre-pass would re-derive what the saturator already derives; its only residual
is negatives a *generic* arc-revision finds that the user's rules don't encode —
none, for a complete ruleset. This is why E14 is cold here: it is largely
*subsumed by saturation*, and `lookahead.py` already supplies the one-step
singleton-consistency case.

## Lattice re-grounding (2026-06-15)

AC-3 as a pre-pass is a **space-shrinker**: pruning trivially-impossible values
**removes alive atoms**, shrinking the cardinality-BFS lattice and so the number
of fork+saturate enterings (the O1+O2 bottleneck) — a complete-search wall-time
lever, not a reorderer.

Note the engine **already has the weakest form**: `lookahead.py` is a one-step
**singleton arc-consistency / forward-checking** pass that kills a candidate
which would die in a single rule firing, before paying for a fork
(`architecture_and_algorithms.md` §O8). E14 generalises that to a **global,
iterated** AC-3 worklist run **once up front** over the relation graph, rather
than one-candidate-at-a-time at branch time. Value scales with puzzle size;
cold on zebra2.

## Context

Classical CSP AC-3 algorithm: iterate "remove value v from
