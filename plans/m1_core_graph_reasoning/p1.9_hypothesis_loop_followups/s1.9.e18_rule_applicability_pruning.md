## Rejected (2026-06-15)

Measured: on `examples/zebra2.ein` (after stdlib import resolution) E18 would
drop **0** of 30 rules. It is structurally inert here, for a reason the stub's
premise ("puzzles import a wider library than they use") missed:

- The stdlib is **generic** — its rules match a relation *parameter* `(?R …)`,
  not a concrete relation, so there are no concrete *unused* match-relations to
  prune in the first place.
- And to *prove* a relation unreachable you must show no rule can ever produce
  it — but any rule with a **variable assert-head** (`symmetric` → `(?rel ?b
  ?a)`; the `implies` idiom → `(?q …)`) can assert *any* relation, so the moment
  one such rule is present (≈ every puzzle) **nothing** is provably-unreachable.

E18 is sound (dropping a truly-unreachable rule can't change results) but never
fires on realistic puzzles, and compile cost is negligible against saturation
anyway (Arch §7). So — like [E14](s1.9.e14_arc_consistency.md) /
[E15](s1.9.e15_path_consistency.md) — rejected rather than shipped as
effectively-dead code. (`demo/` has no tool for this; the measurement was a
one-off reachability count over `kb.rules`.)

## Context

When loading a puzzle, scan each rule's `:match` clause for
relation names. If a referenced relation is absent from the
KB (no declarations + no facts + no rule asserts it), the
rule can never fire — drop it from the compilation cache.

Saves the cost of compiling + watching unreachable rules.

## Tasks

### Task T1.9.E18.1 — Static reachability

Per rule: which relations are referenced in `:match`? Per
KB: which relations are present? Pre-compile, intersect.

### Task T1.9.E18.2 — Drop unreachable rules

In `Engine.compile_all`, skip rules whose required-relation
set isn't a subset of present-relation set.

### Task T1.9.E18.3 — Tests + bench

Confirm rules with absent relations don't fire (which they
already don't, but the engine work is saved).
