## Reframed as rule induction (2026-06-15)

A mutex `{(R a b), (R a c)}` is not data the generator consults — it **is a
rule**: the negative hrule `R(a,b) ⟹ ¬R(a,b_other)`. On zebra2 that rule
already exists, hand-written, as
[`functional-negative`](../../../ein.py/src/ein_bot/stdlib/bijection.ein)
(activated by the `(bijective R)` fan-out); the `(not …)` it derives is what
hypgen skips on. So "extract the mutex" is really *synthesise the property's
negative companion rule* — the same rule-from-rule move that yields
[`symmetric-negative` from `symmetric`](../../../ein.py/src/ein_bot/stdlib/algebra.ein).
The engine already has the *baby* form (`symmetric-negative-setup` derives the
activator `(symmetric-negative R)` from `(symmetric R)`, A9 reflective); the full
form — emitting the companion *body* as data — is
[F5](../../followups/f5_rules_as_data.md)'s `:assert <rule>` (PF5.1), not in M1.

Being **rule-from-rule inference**, the substance is re-homed:

- **[F7 sub-track D](../../followups/f7_rule_induction.md#sub-track-d--companion-rule-synthesis-property-duals)**
  — the property-check → negative-propagation dualisation transform; E5 already
  cross-linked F7 §C (rule-set sufficiency).
- **[F5 — rules as data](../../followups/f5_rules_as_data.md)** — its
  kernel-minimisation audit already poses E5's question ("hypfilter as an ein
  rule, not kernel code?") and answers it with `functional-negative`.
- **[F4 §Q34](../../followups/f4_cross_cutting.md#algebraic-properties-beyond-symmetric-transitive--and-the-2-7-cartesian-product-q34)**
  — the negative companion is a column of the 2⁷ rule-property cartesian product.

The *verification* that a synthesised companion is sound stays cheap — a bounded
chase of the existing saturator over a 2-atom abstract witness (sound on the
monotone, NAF-free fragment, where `functional` / `injective` live). That chase
is the synthesis soundness oracle, not a new engine.

**Residual in P1.9 (dominated).** If one declines the rule form, the static
`(rel, argpos)` table consulted by Python hypgen is a legitimate space-shrinker
— but dominated by `functional-negative` wherever the property is declared,
hence cold. Kept below for the record.

## Lattice re-grounding (2026-06-15)

Like [E11](s1.9.e11_goal_driven_filter.md) / [E4](s1.9.e4_symmetry_class.md),
static mutex pre-analysis is a **space-shrinker**: dropping
`(relation, arg-position)` value-pairs at gen-time **removes atoms / commitment
sets** before the cardinality BFS enumerates them, cutting the number of
fork+saturate enterings (the O1+O2 bottleneck), not merely reordering them.

Today mutex pruning is **dynamic only** — NAF + `lookahead.dies_immediately`
(one rule step before forking) + rule-derived `(not …)`
(`functional-negative` etc.). E5 would lift the easy, *static* mutexes out of
that per-candidate runtime check into a one-time pre-pass. Complete-search
lever; value scales with how many mutexes are statically decidable. Cold on
zebra2.

## Context

Statically analyse the rule set: for each `(relation, arg-
position)`, compute which value pairs would be mutually
exclusive under the loaded rules. Drop hypotheses at gen-time
that would violate any pre-computed mutex.

Pre-emptive cousin of the runtime contradiction detector;
trades startup analysis for runtime pruning.
