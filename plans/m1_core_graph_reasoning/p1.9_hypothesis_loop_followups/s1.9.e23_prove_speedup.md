## Context

`solve()` — P1.7a's sound/complete entry (solution = consistent ∧ complete,
verdict 1-unique / >1-ambiguity / 0-⊥) — explores every alive branch to
establish uniqueness, so it's the exhaustive bottleneck. (Originally framed
against `Mode.PROVE`; that mode never shipped — E21 was superseded by P1.7a —
but the substance is unchanged: the *complete* search is what needs speeding
up.) Open question: is there a way to speed up *exhaustive* search without
giving up the uniqueness guarantee?

User's framing 2026-05-24:

> not doing full search, replace it with what???

Uniqueness is a global property; some form of full coverage
is required. The interesting design question is which form.

## Re-anchor 2 (2026-06-15)

The 2026-06-03 re-anchor (Mode.PROVE → `solve()`) was right but predated the
P1.8a measurement, which is decisive: per
`docs/kernel/inference/architecture_and_algorithms.md` §7, **~95% of a solve is
the matcher inside saturation (O1+O2 — the deductive layer); search is not the
bottleneck.** Consequences for this entry:

- **The four candidates below are all *search-layer* (E7/E11/E14/E4)** (E4 is
  now ⛔ superseded) — they
  cut the branch count, not where the time goes. Still valid as *node-count*
  wins, but low-leverage for wall-time.
- **The headline `solve()` speedup already shipped** in the deductive layer:
  P1.8a cut zebra2 ~3.6× (6.7 s → 1.88 s PyPy `stop_after=1`) via semi-naive
  saturation — see [E8 § Superseded](s1.9.e8_watched_fact.md#superseded-by-p18a-2026-06-15).
- **The named next deductive-layer lever is RETE beta-memories** (persist
  partial joins; the one thing D5 still recomputes — Arch §7), which is **not a
  P1.9 entry** (it's a saturation/perf item, not a hypothesis-loop one) — see
  [P1.9 README — Deductive-layer perf](README.md#deductive-layer-perf-not-a-hypothesis-loop-item-added-2026-06-15).

Net: keep E23 as the *uniqueness-preserving search-pruning* design question,
but recognise it is no longer "the" speedup — saturation was, and P1.8a took it.

## Candidates

- **Learned-clause caching** ([E7](s1.9.e7_learned_clause.md))
  — caches partial proofs; subsequent branches reuse.
- **Goal-driven pruning** ([E11](s1.9.e11_goal_driven_filter.md)
  + [F7 §C](../../followups/f7_rule_induction.md)) — drops
  branches that can't affect the goal.
- **Arc-consistency pre-pass** ([E14](s1.9.e14_arc_consistency.md))
  — tightens alive set before branching.
- **State-space symmetry collapse** ([E4](s1.9.e4_symmetry_class.md))
  — uniqueness up to symmetry instead of literal identity.

Each compounds: a full M1+ engine likely runs all of them.
