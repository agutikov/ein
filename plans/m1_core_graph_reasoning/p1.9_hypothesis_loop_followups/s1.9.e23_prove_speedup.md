## Context

Once E21 lands and `Mode.PROVE` is exhaustive, it's the
bottleneck. Open question: is there a way to speed up
*exhaustive* search without giving up the uniqueness
guarantee?

User's framing 2026-05-24:

> not doing full search, replace it with what???

Uniqueness is a global property; some form of full coverage
is required. The interesting design question is which form.

## Candidates

- **Learned-clause caching** ([E7](s1.9.e7_learned_clause.md))
  — caches partial proofs; subsequent branches reuse.
- **Goal-driven pruning** ([E11](s1.9.e11_goal_driven_filter.md)
  + [F7 §C](../../followups/f7_rule_induction.md)) — drops
  branches that can't affect the goal.
- **Arc-consistency pre-pass** ([E14](s1.9.e14_arc_consistency.md))
  — tightens alive set before branching.
- **State-space symmetry collapse** ([E4](s1.9.e4_symmetry_class.md))
  — uniqueness up to symmetry instead of literal identity.

Each compounds: a full M1+ engine likely runs all of them.

## Tasks

### Task T1.9.E23.1 — Survey + design

Document each candidate's strengths + composition story.

### Task T1.9.E23.2 — Pick + implement

Select one (or a composition) and ship behind a config.

### Task T1.9.E23.3 — Bench

PROVE on a hard puzzle, with vs without the speedup. Confirm
correctness (same unique verdict) + measure time.

## Acceptance

- Speedup landed; bench shows ≥ 2× on at least one hard puzzle.
- Uniqueness preserved.
