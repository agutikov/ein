# S1.9.E2 — `(at-most-one R slot)` activator

**Phase:** P1.9, §closure refinements.
**Estimate:** M (1-3 days).
**Value:** M.
**Status:** ✅ **resolved a different way (2026-06-15)** — at-most-one is the
`functional` / `injective` family (incl. the positional `(functional R 0 1)`
slot form), and "closure derives when the slot saturates" is `std.closure`. A
dedicated `(at-most-one R slot)` activator is unnecessary. See
[§ Resolution](#resolution-2026-06-15).
**Catalog row:** [P1.9 README — Closure refinements](README.md#closure-refinements-q-s154b-descendants)

## Resolution (2026-06-15)

E2's two asks are both met by the P1.8 stdlib, without a new `(at-most-one R
slot)` activator:

- **Per-slot "at most one"** = the cardinality-property family: `(functional
  R)` (≤ 1 image) and `(injective R)` (≤ 1 preimage), with the **positional**
  slot-indexed form `(functional R 0 1)` / `(total R 0)` giving exactly the
  per-slot declaration E2 wanted (`stdlib/elim.ein:56`, `:68`). Cardinality
  ≤ 1 needs *no* counting — `functional` / `injective` express it directly.
- **"Closure derives when every leaf at the slot is bound"** = `std.closure`'s
  `infer-closure`: `functional ∧ total ⇒ (closed R)` (`stdlib/closure.ein:32`)
  — precisely T1.9.E2.2's "derive `(closed R)` when the cardinality saturates."

The only genuinely-absent thing is a **numeric** `(at-most-N)` / `(at-least-N)`
for N > 1, which would need the count predicate the eq/neq matcher lacks
(`stdlib/closure.ein:22-23`) — but that is *beyond* E2's "at-most-**one**"
scope. So E2 is resolved; numeric cardinality is a separate, larger item.

## Context

Generalises `functional` to declare cardinality per slot. E.g.,
`(at-most-one R 0)` says "for any value at slot 1, there's ≤ 1
distinct value at slot 0 paired with it".

Once every leaf at the declared slot has a binding, closure
(`(closed R)`) derives — the activator becomes a CSP-style
cardinality constraint.
