## What it is, and why it's weird here (2026-06-15)

**The algorithm.** Arc- (2-)consistency (E14): for a constraint between two
variables X, Y, drop a *value* x from X's domain if no y in Y supports it.
Path- (3-)consistency (E15): for a *pair* of values (x for X, y for Y), drop the
**pair** if no third variable Z has a value consistent with both. k-consistency
generalises to k-tuples.

**Worked Zebra example.** Constraint A: "Green is immediately right of Ivory."
Constraint B: "the Green house drinks Coffee."
- *Arc* (E14) from A alone: House-1 can't be Green (nothing to its left) ⇒
  derive `(not (colour H1 Green))` — a single-value prune.
- *Path* (E15): take the pair of cells (House-2-colour, House-3-colour). The
  joint assignment (H2=Ivory, H3=Green) satisfies A. Bring in a third variable,
  House-3-drink: B forces H3-drink=Coffee; if another clue already fixed
  H3-drink=Tea, then the **pair** (H2=Ivory, H3=Green) is path-inconsistent ⇒
  prune it. In ein-bot terms that is the **2-literal nogood**
  `{(colour H2 Ivory), (colour H3 Green)}` — "these two can't both hold."

**Why it seems weird — because it is, for this engine:**

1. It prunes *tuples*, not single candidates. ein-bot's alive set is single
   `(R a b)` facts; there is no "domain of pairs" structure. The natural home
   for a pruned pair is a **multi-literal nogood** — which the engine **already
   has** (`_nogoods`, kept subsumption-minimal).
2. So E15 ≈ *eagerly* populating that nogood store by k-consistency propagation,
   versus the current **lazy** "emit a nogood when a commitment dies."
3. And per [E7](s1.9.e7_learned_clause.md)'s measurement, the lattice's deaths
   are **singletons** and the nogood store is **already Apriori-minimal** — the
   engine discovers exactly the (minimal) nogoods it needs, lazily, when it
   needs them. An eager k-consistency pass would compute many pair/triple
   nogoods that never gate the search, at O(n³)+ cost.

So E15 is the k-tuple generalisation of an entry (E14) that is itself largely
*subsumed by saturation*, and its output duplicates a structure the engine
already builds lazily-and-minimally. Recommendation: keep cold / lean toward
**reject** — promote only if some puzzle exhibits a measured residual of
2-consistent-but-3-inconsistent pairs that the lazy nogood store misses (no
such puzzle exists today).

## Lattice re-grounding (2026-06-15)

Same family as [E14](s1.9.e14_arc_consistency.md): a **space-shrinker** that
tightens the alive set before the cardinality BFS, cutting
enterings/saturations. Path-consistency (3-consistency) prunes harder than
AC-3 but at O(n³)+ cost. Strictly behind E14 — only worth it if AC-3 lands and
leaves a measured residual of 2-consistent-but-3-inconsistent atoms on some
puzzle. Cold on zebra2 (and behind E14).

## Context

Generalises arc-consistency to k-consistency: tuples of k
variables checked together. Path-consistency = 3-consistency.
Expensive (O(n³)+ in domain size); tightens the alive set
further than AC-3.
