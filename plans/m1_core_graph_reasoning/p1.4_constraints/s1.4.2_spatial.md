# S1.4.2 — Spatial constraints (1-D lattice)

**Phase:** P1.4 (constraints).
**Estimate:** 3-4 days.
**Depends on:** S1.4.1 (constraint engine machinery).
**Resolves question:** [M1 Q17 — spatial-relation formalisation](../../open_questions.md#q17--spatial-relation-formalisation).
**Implements:** the [PoC's open question on spatial constraints](../../../docs/ideas/05-zebra-puzzle-graph-reasoner.md#open-question-recorded-in-the-readme).

## Context

The PoC has `left_to` / `right_to` / `next_to` as ordinary
relations and sneaks spatial semantics into the *square* inference
rule. That works for Zebra but doesn't compose: it can't natively
say *"Ivory is somewhere to the left of Green"* (not adjacent) or
*"the zebra is in an odd-numbered house"*.

The fix is a small **1-D position lattice**: each spatial entity
gets a `pos : ℤ` attribute; spatial relations are
arithmetic constraints on `pos`. Cheap to implement, trivial to
SMT-emit in M3, and avoids dragging in Allen's 13 interval relations
when we only have one dimension.

The cost: a *small* amount of arithmetic propagation. Manageable
because positions are bounded (5 houses → `pos ∈ {1, 2, 3, 4, 5}`).

## Tasks

### Task T1.4.2.1 — Position attribute + bounds

`src/ein_bot/constraints/spatial.py`. The ontology IR gains:

```ein
(type House :positional :bounds (1 5))
```

Loading this attaches a `pos` integer attribute to every `House`
instance, with an initial domain of `{1..5}`. The graph stores
the domain on the node as a `:pos-domain` extra-fact.

### Task T1.4.2.2 — Spatial relations as arithmetic constraints

The IR exposes these spatial relations; each compiles to an
arithmetic constraint on `pos`:

| IR form                        | constraint                  | example                          |
|--------------------------------|-----------------------------|----------------------------------|
| `(right-of ?a ?b)`             | `pos(a) > pos(b)`           | "Ivory is to the right of Green" |
| `(immediately-right-of ?a ?b)` | `pos(a) = pos(b) + 1`       | "second house"                   |
| `(left-of ?a ?b)`              | `pos(a) < pos(b)`           |                                  |
| `(immediately-left-of ?a ?b)`  | `pos(a) = pos(b) - 1`       |                                  |
| `(next-to ?a ?b)`              | `abs(pos(a) - pos(b)) = 1`  |                                  |
| `(between ?a ?b ?c)`           | `pos(b) < pos(a) < pos(c) ∨ pos(b) > pos(a) > pos(c)` | |
| `(at-position ?a ?n)`          | `pos(a) = n`                | "Norwegian in 1st house"         |

`(at-position …)` is the form for "lives in the first house" once
houses are recognised as positional.

### Task T1.4.2.3 — Bounded propagator

A simple interval / domain propagator. Each constraint reduces the
domain of `pos(?a)` and `pos(?b)`:

```python
class SpatialPropagator:
    def propagate(self, c: Constraint, g: Graph) -> list[DomainUpdate]: ...
```

`DomainUpdate(node_id, removed_values)` — and removing the last value
from a domain emits a violation (no feasible position). Removing all
but one emits a "forced" fact via the `forced-by-unique-position`
rule (P1.3 S1.3.2 T1.3.2.9), so the chain ties into the regular
saturation engine.

### Task T1.4.2.4 — Open-question case: "Ivory left of Green"

Worked example test (`tests/constraints/test_spatial.py`):

```ein
(facts
  (instance Ivory Color) (instance Green Color)
  (rel has-color House_? Ivory)   ;; somewhere
  (rel has-color House_? Green)
  (left-of Ivory Green))
```

Combined with Zebra's other clues, this should narrow the houses
of Ivory and Green correctly. The test asserts that the engine
produces the constraint, propagates domains, and (combined with the
other clues from `examples/zebra.ein`) deduces `pos(Ivory) = 4 ∧
pos(Green) = 5`.

### Task T1.4.2.5 — Trace integration

Spatial propagator updates emit Firings the trace renderer can
display:

```
[3] spatial-propagate: pos(Ivory) ∈ {1..5} → {1..4}
    Because (left-of Ivory Green) and pos(Green) ∈ {1..5}.
```

`:why` template stored in the rule registry so P1.6's renderer
just looks it up.

### Task T1.4.2.6 — Tests

`tests/constraints/test_spatial.py`:

- Each of the seven IR forms above with a 5-position domain.
- Domain narrowing.
- Empty-domain → violation with provenance.
- Allen-style "between" via `between(b, a, c)`.
- Property test: random spatial constraints on 5 positions —
  the propagator agrees with brute-force enumeration.
