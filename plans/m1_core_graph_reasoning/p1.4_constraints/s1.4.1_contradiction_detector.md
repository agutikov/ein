# S1.4.1 — Contradiction detector

**Phase:** P1.4 (constraints).
**Estimate:** 1-2 days.
**Depends on:** P1.2 (`Fact` + `Layer` + `Provenance`),
P1.3 (saturator — shipped; this stage extends it with on-demand
contradiction detection).
**Resolves question:** none new; closes the
"negative-fact representation + detection" remit of the original
P1.4 plan after [S1.4.0 review](s1.4.0_review.md).
**Implements:** the contradiction signal Idea 03's *contradictions*
task class consumes.

## Context

The S1.4.0 review collapsed the original P1.4 scope from "full
constraint engine" to "contradiction detector": one focused
component that scans the KB for `(X, (not X))` pairs and reports
them. Everything else (Constraint IR forms, `verify_incremental`,
spatial 1-D lattice, …) is either covered by P1.3 rules or
migrated to followups / P1.5.

Negative facts already exist in the KB after P1.3 saturation:
`type-exclusivity` (T2 on `co-located`) produces e.g.
`(not (co-located Norwegian Spaniard))`. The detector is the
component that notices when one of those negatives clashes with a
positive assertion in the same layer.

A `(X, (not X))` pair encodes a **branch failure** under M1's
append-only-KB model: there's no way to retract `X` once asserted,
so the only resolution is for whatever caused the conflict (most
often a hypothesis fork) to be unwound. P1.5's hypothesis loop is
the unwinder; P1.4's detector is what tells it to unwind.

## Tasks

### Task T1.4.1.1 — `Contradiction` record + detector class

`src/ein_bot/inference/contradiction.py`:

```python
from dataclasses import dataclass
from ein_bot.kb.entities import Fact, Layer
from ein_bot.kb.store import KnowledgeBase


@dataclass(frozen=True)
class Contradiction:
    """A `(X, (not X))` pair found in the same layer.

    Both facts carry their own provenance — the trace renderer
    (P1.6) reads them to explain *why* each fact was asserted, and
    the hypothesis loop (P1.5) reads `positive.provenance` to
    decide which branch produced the conflict.
    """
    positive: Fact         # the X fact
    negative: Fact         # the wrapping (not X) fact;
                           #   negative.args[0] == positive
    layer: Layer           # the shared layer (REASONING, FACT, …)


class ContradictionDetector:
    """Pure scanner over `kb.facts` looking for X / (not X) pairs.

    Construct once per KB; call `detect()` whenever you need the
    current pair list (typically: after a saturation cycle, before
    a hypothesis-loop branch decision).
    """
    def __init__(self, kb: KnowledgeBase) -> None:
        self.kb = kb

    def detect(self) -> tuple[Contradiction, ...]:
        """All `(X, (not X))` pairs across all layers."""

    def detect_layer(self, layer: Layer) -> tuple[Contradiction, ...]:
        """Pairs scoped to one layer."""

    def has_contradiction(self) -> bool:
        """Fast yes/no — short-circuits on first pair found."""
```

The algorithm:

1. Iterate `kb._facts_by_relation.get("not", ())` — every fact
   whose head is `not`.
2. For each negative fact `n`, read its first arg `n.args[0]` —
   that's the inner positive fact (Q40 / R9 nested-Fact widening).
3. Use `kb._fact_by_id(inner.relation_name, inner.args)` to look up
   whether the positive proposition is also in the KB.
4. If both exist in the **same layer** (e.g. both in REASONING),
   emit `Contradiction(positive=found, negative=n, layer=found.layer)`.
5. Layer-mismatched pairs (e.g. positive in FACT, negative in
   REASONING) are NOT contradictions — they're the rule firings'
   consequences. Only same-layer pairs constitute a branch failure.

Performance: O(|`not`-headed facts|). On Zebra with 120 negatives,
each lookup is O(deg(rel)) on `_facts_by_relation` — bounded by
the number of facts with that relation name. Sub-millisecond.

### Task T1.4.1.2 — Saturator integration

The Saturator (`src/ein_bot/inference/saturator.py`) gains an
optional helper:

```python
class Saturator:
    def contradictions(self) -> tuple[Contradiction, ...]:
        """Convenience: detect contradictions in the current KB.

        P1.5's hypothesis loop calls this between branch saturations
        to decide retraction. The Saturator owns the KB reference;
        callers don't need to construct a ContradictionDetector
        themselves.
        """
        from .contradiction import ContradictionDetector
        return ContradictionDetector(self.kb).detect()
```

This is the only integration P1.5 needs; the detector itself stays
independent of the saturator.

### Task T1.4.1.3 — Tests

`tests/inference/test_contradiction.py`:

- **Empty KB** — `detect() == ()`.
- **No-conflict KB** — a positive only, or a negative only, or a
  positive and negative whose inner content differ; all return
  `()`.
- **Same-layer conflict** — assert `(rel A B)` and
  `(not (rel A B))` both in REASONING; detector finds exactly one
  pair, both fact references resolvable.
- **Cross-layer "non-conflict"** — positive in FACT, negative in
  REASONING (e.g., `type-exclusivity` derived the negative but the
  positive is a puzzle assertion). The detector reports **nothing**
  for this case — by design, layer separation prevents
  cross-layer "contradiction" misfires. (P1.5 will introduce
  branch-scoped layers later; that's where the cross-source pair
  becomes meaningful.)

  Note: an end-to-end "Zebra contradiction" — asserting
  `(co-located Norwegian Spaniard)` on top of zebra.ein — would
  produce a same-layer conflict (the asserted positive joins
  whichever layer the assertion lands in; type-exclusivity
  produces the negative in REASONING; if the assertion is in
  REASONING too, the detector catches it). The cross-layer test
  documents the *non*-trigger by design.
- **Multi-conflict KB** — several pairs; verify all are reported.
- **Layer-scoped detection** — `detect_layer(Layer.REASONING)`
  filters correctly.
- **`has_contradiction()` short-circuits** — performance smoke
  test (with many `not` facts, first conflict found stops the
  scan).
- **Nested-fact safety** — a `(not (rel a (other inner)))` where
  the inner is itself a nested Fact (Q40) is handled by the
  recursive `Fact.__eq__` — equality on the nested args propagates.
