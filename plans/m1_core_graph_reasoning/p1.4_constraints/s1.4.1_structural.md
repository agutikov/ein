# S1.4.1 — Structural constraints

**Phase:** P1.4 (constraints).
**Estimate:** 2-3 days.
**Depends on:** P1.2 (graph + provenance), P1.1 (constraint IR
form from S1.1.1).
**Implements idea:** [05 §Constraints](../../../docs/ideas/05-zebra-puzzle-graph-reasoner.md#constraints-as-the-readme-puts-it).

## Context

The PoC named two constraints:

- **single-type** — each object node has at most one link to a
  type node.
- **number-of-attributes** — for any object of `Type_1` and any
  `Type ≠ Type_1`, at most one path from the object to a node of
  `Type`.

In Zebra these collapse to *"every node has at most one link of
every possible type"* — a per-type `allDifferent` over instances.
This stage formalises them as IR forms with a `verify` impl.

## Tasks

### Task T1.4.1.1 — Constraint IR forms

In the IR (P1.1) `(facts …)` or top-level, allow constraint
declarations:

```ein
(constraint single-attribute
  :forall ?obj of ?T_obj
  :forall ?T_attr where (!= ?T_attr ?T_obj)
  :at-most-one (?obj has-attr ?attr where (type-of ?attr ?T_attr)))

(constraint all-different
  :over (?xs... of ?T)
  :pairwise-distinct ?xs)
```

These compile into `Constraint` objects (`src/ein_bot/constraints/forms.py`):

```python
@dataclass(frozen=True, slots=True)
class Constraint:
    id:        ConstraintId
    name:      str
    kind:      Literal['at-most-one', 'all-different', 'exactly-n', ...]
    scope:     Pattern                  # over which nodes/edges
    source:    Loc | None
```

### Task T1.4.1.2 — `verify(graph)`

`src/ein_bot/constraints/verify.py`. For each registered constraint:

- enumerate its scope on the current graph;
- evaluate the predicate;
- on violation, build a `Violation(constraint_id, witness_edges)`
  where `witness_edges` is a minimal pair (or n-tuple) of edges
  that together break the constraint.

Critical: `witness_edges` must include source-layer provenance —
that's how P1.5's `hypothesis-contradiction` rule names what
clashed.

### Task T1.4.1.3 — `verify_incremental`

For the hypothesis loop and the saturator (P1.3 S1.3.3), naïve
re-verification is too slow. Add:

```python
def verify_incremental(g: Graph, last_added: list[EdgeId]) -> Verdict: ...
```

Only checks constraints whose `scope` overlaps with `last_added`.
Indexed by the constraint registry keyed on relation name + types.

### Task T1.4.1.4 — Witness narrowing

When `at-most-one` is violated by `n > 2` edges, narrow to the
*shortest* witness pair: any two edges suffice. When `all-different`
is violated, return the pair sharing a value. This keeps trace
output legible.

### Task T1.4.1.5 — Tests

`tests/constraints/test_structural.py`:

- A 4-node 4-edge graph with `single-type` constraint passes.
- Adding a second `is-type` edge from the same node triggers a
  violation with a minimal 2-edge witness.
- `all-different` over five `color(H_i)` returns a 2-edge witness
  when two houses share a color.
- `verify_incremental` returns the same verdict as `verify` for
  random graph mutations (property test).
