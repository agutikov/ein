## Why this stage exists

The S1.4.1 / S1.4.2 stage files were drafted **before** P1.3 shipped
the rule library + saturator. Several design decisions resolved
during P1.3 invalidate the bulk of the original P1.4 spec:

- The 4 M1 rules `symmetric`, `transitive`, `square-fwd`,
  `square-bwd` + `square-unique` solve the spatial problem the
  PoC's "Ivory left of Green" question raised — declaratively, no
  position lattice required (Q17 resolution).
- `type-exclusivity` (T2 on `co-located`) covers the
  "single-attribute" / "all-different" structural constraints the
  original S1.4.1 enumerated.
- The saturator already produces negative facts `(not (rel a b))`
  via R9's `Fact.args` widening — the "negative-fact representation"
  question is closed.

This stage **audits** P1.4 against the shipped reality and produces
a resolution plan. No code is written here — the output is the list
of edits and a substantially smaller follow-on stage.

## Verdict

P1.4 needs a **dramatic shrinkage**, not a rewrite. About **85%** of
the original spec is now obsolete; the remaining **15%** is one
small task — a contradiction detector that scans the KB for
`(X, (not X))` pairs. The phase shrinks from ~1 week of work to
~1-2 days.

## B. What survives — contradiction detection

One task survives, reframed:

**Original S1.4.1 T1.4.1.2** — *"`verify(graph) -> Verdict` returns
`OK | Violation(constraint_id, witness_edges)`."*

**Reframed S1.4.1 (new)** — *"Contradiction detector: scan the KB
for `(X, (not X))` pairs; return them as `Contradiction` records
the trace renderer (P1.6) and the hypothesis loop (P1.5) consume."*

Differences from the original:
- No `Constraint` entity to register against — the detector is a
  pure scan over the existing facts.
- Witness = `(positive_fact, negative_fact)` — always a pair, never
  larger.
- "Verdict" simplifies to a tuple of `Contradiction` records;
  empty tuple = no contradiction.

The output feeds P1.5's hypothesis-loop directly: when a branch
saturation completes and the detector reports a contradiction, P1.5
emits the `(contradiction-under ?h)` synthetic fact that triggers
the `hypothesis-contradiction` rule shipped in P1.3.

## D. Where the spatial cases now live

For completeness, mapping the old S1.4.2 use cases to their new
homes:

| S1.4.2 use case                                              | New home                                                                                              |
|--------------------------------------------------------------|-------------------------------------------------------------------------------------------------------|
| `(right-of ?a ?b)` derives spatial ordering                  | Existing relation + `square-fwd`/`square-bwd` rules (zebra.ein).                                       |
| `(next-to ?a ?b)` ↔ adjacency                                | `(implies right-of next-to)` + `(symmetric next-to)` activators (zebra.ein).                            |
| `(next-to ?a ?b)` corner case (House-1, House-5)             | `square-unique` rule with `(square-unique next-to House)` activator (zebra.ein).                       |
| `(next-to ?a ?b)` middle-house disjunctive case              | **P1.5 hypothesis branching** — outside M1.P1.4.                                                       |
| "Ivory left of Green" (non-adjacent ordering)                 | **P1.5 hypothesis branching** over `(right-of ?Ivory ?Green)` / `(right-of ?Green ?Ivory)` disjunct.  |
| `(between ?a ?b ?c)`                                          | Future followup — not in Zebra-class puzzles.                                                          |
| `(at-position ?a ?n)`                                         | `(co-located ?a House_n)` — no separate numeric form needed.                                          |
