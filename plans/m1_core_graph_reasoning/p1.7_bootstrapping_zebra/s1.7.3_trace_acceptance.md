# S1.7.3 — Trace-matches-human acceptance

**Phase:** P1.7 (bootstrapping).
**Estimate:** 3-4 days.
**Depends on:** S1.7.1, S1.7.2, and every prior phase.
**Implements idea:** [08 §The target trace](../../../docs/ideas/08-human-style-deductive-trace.md#the-target-trace-paraphrased).

## Context

The trace is what makes ein-bot *not just a solver*. This stage
gates M1 on a structural match between the generated trace and
the human walkthrough quoted in
[idea 08](../../../docs/ideas/08-human-style-deductive-trace.md#the-target-trace-paraphrased).

"Structural match" means: for every move the human makes (direct
fact, composition, elimination by exhaustion, forward chaining,
case analysis, reductio, symmetry), the engine's trace contains a
corresponding *named rule firing* with provenance pointing at the
same source condition. Wording need not match.

## Tasks

### Task T1.7.3.1 — Codify the target moves

`tests/integration/zebra-target-moves.yaml` — one entry per human
move:

```yaml
- id: m1
  human: "By condition (10), the Norwegian lives in the first house."
  rule:  source-fact
  source: "condition (10)"

- id: m2
  human: "From (10) and (15) it follows that the second house is blue."
  rule:  triangle-composition           # OR another from the registry
  sources: ["condition (10)", "condition (15)"]

- id: m3
  human: "What color is the first house? Not green, not white, not red, therefore yellow."
  rule:  elimination-by-exhaustion
  sources: ["condition (5)", "condition (2)"]

- ...
```

The yaml is the regression spec — every Zebra walkthrough move
maps to a rule firing the engine should produce.

### Task T1.7.3.2 — Trace audit harness

`tests/integration/test_zebra_trace_matches.py`:

1. Run `solve(zebra)` with full tracing.
2. Parse `zebra-target-moves.yaml`.
3. For each target move, find a trace step with matching `rule` and
   the expected `sources` (a subset match on source-condition
   provenance through derivation DAG).
4. Assert *every* target move maps to *at least one* step. Allow
   extra engine steps the human elided.

### Task T1.7.3.3 — Trace narrative review

Run `ein-bot solve --trace=zebra.md --diagrams=zebra-out/`. Open
the result. Manual review checklist:

- [ ] Each step's `:why` text reads as natural English.
- [ ] No step refers to a "node id" — only puzzle-language names.
- [ ] The hypothesis branches show as foldable sections.
- [ ] The symmetry-break appears once.
- [ ] The final paragraph names the puzzle answer.

Capture review notes inline as `<!-- review: ... -->` comments;
follow-up items go to [`plans/ideas.md`](../../../ideas.md).

### Task T1.7.3.4 — Performance + budget

Smoke benchmark: `ein-bot solve zebra.ein` runs in
< 2 seconds wall-clock on a laptop. If slower, profile saturator
(P1.3 S1.3.3) — likely culprit is naïve pattern matching. Optimise
only if budget exceeded.

### Task T1.7.3.5 — Three task classes integration

`tests/integration/test_zebra_three_classes.py`:

- SOLVE: unique solution, matches the canonical answer.
- GAPS: with condition (15) deleted, returns at least one
  diverging goal node, names which condition the gap depends on.
- CONTRADICTIONS: with the injected bad fact, returns a 2-3 edge
  unsat core including the injection + condition (5).
