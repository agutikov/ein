# S1.7.11 — hardcoded name `open` (third-state match sugar)

**Phase:** P1.7 (bootstrapping) — per-name kernel-purity analysis.
**Parent:** [S1.7.7](s1.7.7_kernel_purity_analysis.md) (inventory).
**Class:** logical primitive — **match-body desugaring**.
**Verdict:** **pure desugaring** — already reducible to `and`/`absent`/
`not`; the engine carries `open` only as authoring sugar.
**Status:** analysis. **Reclassification finding:** S1.7.2/S1.7.7 paired
"`closed`/`open`" as the closed-world primitive; reading the code,
`open` is **not** closed-world's counterpart — it is a *third-state
match* over an individual proposition. It belongs with `forall` (sugar),
not with [`closed`](s1.7.10_closed.md).

## Call site

- `compile.py:314` — `(open P)` in a `:match` → `_compile_premise` calls
  `_desugar_open` (`compile.py:212–226`):
  `(open P) → (and (absent P) (absent (not P)))`.

## 1. Functionality connected to the name

`(open P)` matches when `P` is in the **undecided** third state — neither
asserted nor negated in the KB. Used to gate hypothesis emission on
still-open slots ("propose only where nothing is yet committed"). It is a
*premise* form (read-only guard), never asserted or stored.

## 2. Consequences if removed

None semantic — every `(open P)` author would write the two-clause
`(and (absent P) (absent (not P)))` by hand. Removal costs only
*ergonomics* (a common pattern loses its one-token name) and the
self-documenting trace label "open". No solve, soundness, or completeness
effect — the desugaring is the engine's actual behaviour today.

## 3. How it could be implemented in ein-lang rules (or elsewhere)

It already **is** ein — `(and (absent P) (absent (not P)))` is pure
rule-body syntax over three retained primitives ([`and`](s1.7.14_and.md),
[`absent`](s1.7.17_absent.md), [`not`](s1.7.12_not.md)). So `open` is the
cleanest de-hardcoding candidate in the inventory: it needs **no Python
arm at all** if ein gains a macro/`define-syntax` facility
([P1.8 modules](../p1.8_ein_lang_modules/) / [F2](../../followups/f2_self_modifying_language.md))
— `open` becomes a library macro a puzzle imports, not a compiler branch.

## 4. How it could be implemented without sticking to the name

Two routes. **(a)** Delete the name: drop the `compile.py:314` arm and
the helper; authors write the expansion inline (no name, no sugar).
**(b)** Keep the *sugar* but move it out of the kernel: a user-space macro
`(defmacro open (P) (and (absent P) (absent (not P))))` expanded at load
— the name persists but as **user-defined data**, not a hardcoded
compiler case. Either way the kernel stops special-casing `"open"`; the
only question is whether the convenience macro lives in a stdlib.

## Decision (to record in [S1.7.7 T1.7.7.5](s1.7.7_kernel_purity_analysis.md))

`open` is **reducible now** — record it as desugaring sugar, not a kernel
primitive. Concrete step available immediately: it can leave the kernel
the moment a macro layer exists (P1.8); until then the `compile.py`
desugar arm is a harmless one-liner. Note the reclassification (away from
`closed`) in §6 of the kernel model doc.
