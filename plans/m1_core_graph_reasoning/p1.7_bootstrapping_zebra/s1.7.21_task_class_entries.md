# S1.7.21 — hardcoded names `solve` / `gaps` / `contradictions` (task-class entries)

**Phase:** P1.7 (bootstrapping) — per-name kernel-purity analysis.
**Parent:** [S1.7.7](s1.7.7_kernel_purity_analysis.md) (inventory) — **Tier B**.
**Class:** engine **task-class entry selector** (the A/B/C dispatch).
**Verdict:** **engine control vocabulary** — not an ein-lang relation
name; the de-naming question is "string → enum / first-class task class".
**Status:** analysis. Connects to
[F3 three task classes first-class](../../followups/f3_three_task_classes_first_class.md).

## Call sites (the most frequent strings in the engine: 22/25/22)

- `monotonic/solver.py` — ~69 `entry == "solve" / "gaps" /
  "contradictions"` branches threading the task class through `_explore`,
  the stop policy, result construction, and solution recording
  (`solver.py:523,565,805,883,1024,1059,1107,1145,1189,…`).
- `verdict.py:54–56` — the three verdict shapes keyed by entry.
- `monotonic/__init__.py:118` — the `solve` entry point.

## 1. Functionality connected to the names

These are the **three task classes** of [idea 03](../../ideas/03-three-task-classes.md):
**A `solve`** (find the unique/ambiguous/⊥ solution), **B `gaps`**
(enumerate diverging goal nodes — under-constraint), **C
`contradictions`** (return an unsat core — over-constraint). The single
unified set-search engine ([memory `project_set_search_unified`]) runs one
`_explore_layers` and the `entry` string selects: the **stop policy** (A
stops at the first solution-or-⊥; B/C exhaust), the **store_lattice** flag
(B/C keep the frontier), and the **verdict** projection. So the names are a
*mode selector* over one engine, not three engines.

## 2. Consequences if removed

There is nothing to "remove" — the engine *must* know which of the three
questions it is answering; the names are how the caller asks. Replacing
the strings with anything (enum, subclass) is mechanical and changes no
behaviour. The risk is purely **stringly-typed fragility**: a typo
`"contradiction"` (singular) silently mis-dispatches; the 69 scattered
comparisons are hard to keep exhaustive (a new task class means touching
all of them).

## 3. How it could be implemented in ein-lang rules (or something else)

Not applicable as *rules* — a task class is a **query about the search**,
not a fact or a derivation. But it *is* expressible as **data**: the
task class could be a field on the `(query …)` block (it already partly is
— `:goal` selects what `solve` answers; cf.
[S1.7.5](s1.7.5_query_semantics_who_vs_where.md)). The deeper
[F3](../../followups/f3_three_task_classes_first_class.md) ambition is to
make the three classes **first-class objects** — a `TaskClass` with
`stop_policy`, `keeps_frontier`, `verdict_projection` as methods — so the
engine dispatches on a value, not a string, and a fourth class is a new
object, not 69 new branches.

## 4. How it could be implemented without sticking to the names

Replace the `entry: str` with a **`TaskClass` enum or strategy object**
carrying the three behavioural axes the strings currently gate
(stop-after, store-lattice, verdict-shape). The 69 `== "…"` sites collapse
to method calls / `match` on the enum; the names survive only as the
enum's `repr` and the CLI `--mode` surface. This is the canonical
"stringly-typed → typed" refactor and the concrete first step toward
[F3](../../followups/f3_three_task_classes_first_class.md). No ein-lang
exposure changes — `--mode=gaps` still reads `gaps`.

## Decision (to record in [S1.7.7 T1.7.7.5](s1.7.7_kernel_purity_analysis.md))

Record as **Tier B engine vocabulary**: not a kernel-language purity
question (never a relation name), but a **code-quality / F3** one. Recommend
a `TaskClass` value type replacing the `entry` string as a non-gating
refactor — the same three behaviours, dispatched on a type. Defer to
[P1.7b review](../p1.7b_review_and_refactor/) or F3, not the kernel-purity
gate.
