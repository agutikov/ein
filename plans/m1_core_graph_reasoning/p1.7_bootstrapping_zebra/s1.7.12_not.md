# S1.7.12 — hardcoded name `not` (negation-as-data)

**Phase:** P1.7 (bootstrapping) — per-name kernel-purity analysis.
**Parent:** [S1.7.7](s1.7.7_kernel_purity_analysis.md) (inventory).
**Class:** logical primitive — **irreducible base** (the ⊥ calculus).
**Verdict:** **stays kernel** ([S1.7.2](s1.7.2_dynamic_vs_hardcoded.md)).
This stage records *why* it is irreducible and what the name buys.
**Status:** analysis.

## Call sites (the most pervasive name in the engine, ~19)

- `compile.py:269,322–327,353` — `(not P)` falls through to the generic
  relation handler: it compiles as a **fact pattern with head `"not"`**
  whose inner expression is a `NestedPattern` arg, matching stored
  `(not P)` facts (S1.5.8c K-Δ.1).
- `compile.py:442,462,474,487,502` — `asserted_relation` / NAF-ref
  extraction must see through `"not"` to the inner relation.
- `contradiction.py:166–190` — the **pair-⊥** detector walks
  `_facts_by_relation["not"]` matching each `(not X)` against a positive
  `X` (same layer ⇒ contradiction).
- `lookahead.py:128,144` — same pair shape, pre-fork.
- `back_prop.py:178–212,307` — death writes a `(not h)` fact
  (`relation_name="not"`) and bubbles it; the symmetric mirror writes
  `(not (R b a))`.
- `solver.py:337,340` — monotonic negated-fact writeback.
- `naf_deps.py:31` — Scope-B NAF: the literal head is `"not"` but the
  watched fact is the derived `(not (R …))`.

## 1. Functionality connected to the name

`not` is **negation as data**: `(not P)` is a *stored fact*, not a
runtime test (that is [`absent`](s1.7.17_absent.md)). Three roles ride on
the head string `"not"`: (a) the matcher treats `(not P)` as an ordinary
relation whose single arg is a nested proposition — so rules can *assert*
and *match on* negative facts; (b) the **contradiction** detector defines
⊥ as a same-layer `(X, (not X))` pair; (c) **back-prop** records a refuted
hypothesis as `(not h)` and propagates it. The name is the bridge between
"P is false" as data and the ⊥-pair that drives every branch death.

## 2. Consequences if removed

Catastrophic — `not` is not an optimisation, it is the **representation
of falsity**. With no `"not"` head there is no negative fact to store, so
back-prop cannot record a refuted hypothesis, the contradiction detector
has no pair shape, lookahead cannot pre-kill, and NAF (`absent (not …)`)
loses its target. The engine would lose the entire concept of a *proven
negative*. There is no graceful degradation: solve, gaps, and
contradictions all collapse.

## 3. How it could be implemented in ein-lang rules (or elsewhere)

It **cannot** be a rule — `not` is the object language of negation that
rules are written *in*. A rule `(rule … :assert (not P))` already uses it;
nothing more primitive expresses "P is false" without circularity. The
only design freedom is **representation**, not elimination: negation could
be a typed field on `Fact` (a `polarity` bit) rather than a wrapper
relation. But that is a *data-model* change, not a rule — and it would
**relocate**, not remove, the privileged concept.

## 4. How it could be implemented without sticking to the name

The string `"not"` could become a **distinguished `Fact` subtype** or a
`Fact.polarity ∈ {+,−}` field, so the engine branches on a *type/flag*
rather than a relation name — removing `"not"` from `compile.py` /
`contradiction.py` / `back_prop.py` string comparisons. Trade-off: today
`(not P)` composes uniformly with every fact mechanism (indexing,
provenance, nesting, rendering) **for free** precisely *because* it is an
ordinary relation; a polarity field would need each of those mechanisms
taught about it. The name is the cheapest encoding of a genuinely
primitive notion. ⇒ de-naming is possible (flag/subtype) but is pure cost
with no purity gain — `not` is *the* irreducible kernel primitive.

## Decision (to record in [S1.7.7 T1.7.7.5](s1.7.7_kernel_purity_analysis.md))

Record `not` as **kernel-irreducible** — the load-bearing definition of
falsity, not a derivation and not a search hint. The only open design note
is the polarity-field representation alternative (§4), deferred as a
data-model question, not a purity removal.
