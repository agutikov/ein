# S1.7.16 — hardcoded name `forall` (guarded universal sugar)

**Phase:** P1.7 (bootstrapping) — per-name kernel-purity analysis.
**Parent:** [S1.7.7](s1.7.7_kernel_purity_analysis.md) (inventory).
**Class:** logical primitive — **pure desugaring** over `absent`/`and`/`not`.
**Verdict:** **reducible now** — like [`open`](s1.7.11_open.md), a
compile-time rewrite; the engine carries `forall` only as sugar.
**Status:** analysis.

## Call site

- `compile.py:319–320` — `(forall ?b (G) (B))` in a `:match` →
  `_desugar_forall` (`compile.py:229–253`):
  `(forall ?b (G) (B)) → (absent (and G (absent B)))`, i.e.
  ∀x. G(x)→B(x) ≡ ¬∃x. G(x) ∧ ¬B(x). Safety: the bound `?b` must appear
  in the guard `G` (the guard grounds it); a malformed head raises
  `ValueError`.

## 1. Functionality connected to the name

`forall` is the **bounded universal** match premise: "for every `?b` the
guard `G` admits, body `B` holds". It is the workhorse behind
`total` / `domain-elimination` style rules ("every house has exactly one
colour" ⇒ a `forall` over colours). It is purely a *guard* (read-only,
never asserted), and it is exactly the negation-of-existential the
docstring states — a finite, safe ∀ over guard-grounded bindings.

## 2. Consequences if removed

No semantic loss — every `(forall …)` author would write
`(absent (and G (absent B)))` by hand. But these expansions are **harder
to read and easier to get wrong** than `open`'s (the double-negation +
the bound-var safety check the desugarer enforces). Removing the name
costs ergonomics and the safety guard `_desugar_forall` performs (bound
var must occur in the guard) — which authors would then have to honour
manually or risk an unsafe/ungrounded ∀.

## 3. How it could be implemented in ein-lang rules (or elsewhere)

It already **is** ein — the expansion target is three retained primitives
([`absent`](s1.7.17_absent.md), [`and`](s1.7.14_and.md),
[`not`](s1.7.12_not.md)). `forall` is the canonical example of "a
primitive that is sugar": it cannot be a *rule* (it is a match-body form,
not a fact), but it is a *macro* over the base NAF calculus. A
`define-syntax`/macro layer ([P1.8](../p1.8_ein_lang_modules/) /
[F2](../../followups/f2_self_modifying_language.md)) would host it in user
space, including the bound-var safety check as a macro-expansion
assertion.

## 4. How it could be implemented without sticking to the name

Same two routes as [`open`](s1.7.11_open.md): **(a)** delete the name and
inline the `(absent (and G (absent B)))` expansion (loses the safety
check); **(b)** keep `forall` as a **user-space macro**, name preserved as
data, kernel stops special-casing `"forall"`. The bound-var safety
condition argues *for* keeping the sugar (somewhere) rather than inlining:
it is real authoring protection the raw expansion lacks.

## Decision (to record in [S1.7.7 T1.7.7.5](s1.7.7_kernel_purity_analysis.md))

`forall` is **reducible** — record as desugaring sugar, not a kernel
primitive. Co-migrate with [`open`](s1.7.11_open.md) /
[`or`](s1.7.15_or.md) to a P1.8 macro stdlib when one exists; preserve the
bound-var safety check in the macro. Until then the `compile.py` desugar
arm is harmless.
