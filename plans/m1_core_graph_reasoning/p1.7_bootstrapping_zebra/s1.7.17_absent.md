## 1. Functionality connected to the name

`absent` is **negation-as-failure**: `(absent P)` matches iff `P` is *not
currently provable* in the KB ‚Äî the open-world "I cannot show P". It is
the dual of [`not`](s1.7.12_not.md) (which is "P is *proven false*", a
stored fact). `absent` is a **runtime test**, computed against the live KB,
never stored. It is the base of the whole derived-NAF story:
domain/range-elimination, totality, `forall`, and `open` all bottom out in
`absent`. Soundness rests on the fire-time re-eval (S1.5a.1).

## 2. Consequences if removed

The engine loses **negation-as-failure** entirely ‚Äî and with it
`forall`, `open`, and every elimination/totality rule that says "if no
other house can be the colour, this one is". Closed-world reasoning
([`closed`](s1.7.10_closed.md)) bottoms out here: without `absent` there is
no way to act on "nothing proves P", so the Zebra solve (which is mostly
elimination) collapses. Unlike `not`, there is **no data encoding** of
NAF ‚Äî it is inherently a *query over the current proof state*, so it
cannot even be relocated to a fact flag. Hardest-irreducible primitive
after `not`.

## 3. How it could be implemented in ein-lang rules (or elsewhere)

It **cannot** be a rule ‚Äî `absent` is a *metalogical* test ("P is not
derivable"), which a monotonic rule body cannot express (a rule can match
on what *is* true, not on the *absence* of a derivation; that requires
stratified NAF, which is the engine's job). Everything *built on* `absent`
is rules (`forall`, elimination); `absent` itself is the floor. The S1.7.4
map exists precisely because NAF over a derived relation is sound only by
engine machinery (fire-time re-eval), not by anything a rule could state.

## 4. How it could be implemented without sticking to the name

The string `"absent"` is a single compile-time recognition
(`compile.py:309`); it could be any token. But the *concept* ‚Äî a
`AbsentGuard` opcode evaluated against live KB state with fire-time
re-eval ‚Äî is irreducible. De-naming would be cosmetic (rename the head);
de-**hardcoding** is impossible without giving up NAF. The one real design
axis is **stratification policy**: today soundness leans on fire-time
re-eval + the S1.7.4 derived/declared distinction; a future engine could
make NAF *stratified by construction* (priority bands guarantee the
watched relation is saturated first), which would move the soundness
argument from runtime re-eval into a static schedule ‚Äî but `absent` the
primitive stays.

## Decision (to record in [S1.7.7 T1.7.7.5](s1.7.7_kernel_purity_analysis.md))

Record `absent` as **kernel-irreducible** ‚Äî the NAF primitive the whole
elimination calculus and `closed` soundness rest on. Note that the
[S1.7.4 map](s1.7.4_naf_dependency_map.md) is its soundness instrument and
that the only design freedom is stratification policy, not removal.
