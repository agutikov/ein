## Why this stage exists — the soundness bug S1.7.3 found

On `examples/zebra2.ein` (which **is** satisfiable — a unique 25/25 model),
the engine produces *unsound* verdicts because the acceptance signal is
**goal-pattern match** (`is_solved`), not "the model is determined":

- `monotonic_solve` returns a `Solution` whose KB is a **non-extendable
  partial dead-end** — e.g. it commits `(color-loc Green House-4)`, the
  house-only goal `(drink-loc Water ?h) ∧ (pet-loc Zebra ?h)` matches
  (Water/Zebra houses are forced regardless of colour), and it stops. The
  grid is 19/25 *resolved* but only 12/25 *correct*; completing it forces
  `Spaniard = Japanese` → contradiction. So it calls a non-solution a
  solution.
- Bolting a completeness check onto `is_solved` (the S1.7.3 experiment)
  made monotonic instead return **`Contradiction` for this SAT puzzle** —
  it rejected the partial matches and its pruning drained `alive` to empty
  without the real model surviving, and Phase 3's `if not alive: return
  _contradiction` mislabelled that as ⊥.

**Hard invariant violated:** a correct inference engine can never exhaust a
satisfiable problem to a contradiction. Both behaviours above break it (one
claims a non-model is a model; the other claims a model doesn't exist). The
fix is not a patch on `is_solved` — it is a **new, domain-agnostic
definition of "solution"**.

## The clean definition

For a KB, every generable 1-size hypothesis `h = (R a b)` (what the `guess`
hrule / `generate_hypotheses` proposes) is in exactly one state:

- **present** — `h` is asserted (derived or committed);
- **refuted** — `(not h)` is asserted;
- **open** — neither.

Then:

- **complete(kb)** ⟺ *no open hypothesis* — the generator proposes nothing
  that isn't already decided. (Built only on `generate_hypotheses` + the
  KB's positive / negated fact sets. **No `is-a`, no `total`, no relation
  signatures** — those were the encoding-specific crutches that made the
  S1.7.3 `_is_complete_model` wrong: `is-a` may be renamed, absent, or
  unneeded, and a puzzle may declare no `total` at all.)
- **consistent(kb)** ⟺ no contradiction (no `(false)`, no `X ∧ ¬X`).
- **solution node** = a saturated lattice node that is *consistent ∧
  complete*, **deduped by semantic state** (`state_hash`).

**Verdict from the count `k` of distinct solution nodes:**

| k | verdict | task class |
|---|---------|------------|
| 1 | the unique model | **SOLVE** |
| >1 | multiple models | **GAPS** (ambiguity) |
| 0 | no model | **CONTRADICTIONS** |

The query `:goal` **stops driving termination**; it becomes a *projection /
question over the solution node(s)*. This is what dissolves the original
bug: "where is Water / who keeps the Zebra" is answered by reading a
*complete* model, never by stopping as soon as a pattern matches a partial
one.

### Three task classes fall out of one exhaustive run

- **SOLVE** — return the single solution node.
- **GAPS** — `k > 1`; the **gap is the root's residual open set** (the
  contingent hypotheses — true in some models, false in others). This is a
  sharper definition of "diverging goal node" than the current one.
- **CONTRADICTIONS** — `k = 0`; report the dead nodes' unsat cores
  (the S1.7.3 root-core work feeds this).

### Validation against already-observed data

The S1.7.3 `compare_engines.py` BEFORE run is consistent with this: `gaps`
returned 15 branches → 2 distinct grids → grid[1] ×14 (the 25/25 model)
+ grid[0] ×1 (the 19/25 dead-end). Under this definition grid[0] is **not**
a solution node (it has open hypotheses), and the 14 grid[1] commitments
share one complete state → **k = 1 = unique**. The dead-end is excluded for
the principled reason (incomplete), not by a heuristic.

## Plan

### Phase A — Investigations (read-only; verify, do not assume)

Earlier in S1.7.3 incorrect claims were made about the engine (that
`_compute_alive` was an "open set", that `gaps_solve` "auto-disables the
state-hash dedup-merge"). **Treat every code fact below as unverified until
read.**

- **I1 — generator coverage.** Does `generate_hypotheses(kb)`
  (`inference/hypgen.py`) enumerate the *full* candidate-atom space for a
  puzzle, or a filtered subset? "complete = no open" is only as sound as
  the generator's coverage.
- **I2 — `state_hash` soundness.** Is `inference/canon.py`'s `state_hash`
  a canonical key over *semantic* content (decided hypotheses), invariant
  to provenance / insertion order / bookkeeping heads? Too coarse →
  undercount (false "unique"); too fine → overcount (false "ambiguous").
- **I3 — how solution nodes are recorded.** Read how a satisfying
  commitment becomes a `SolutionRecord` / `verdict.branches`
  (`lstate.solutions` in `monotonic/solver.py`) and what role, if any,
  `state_hash` / `kb_index` plays there. (Clarify the 15-vs-2 gap from the
  code — the earlier "auto-disables" statement was wrong and must be
  replaced with what the code actually does.)
- **I4 — `_compute_alive` semantics.** What does it actually return —
  open hypotheses, or present ∪ open — and how does it relate to a
  `has_open_hypothesis` predicate?
- **I5 — verdict sites.** Enumerate where each verdict is decided in
  `_explore_layers`, especially every `not alive → _contradiction` path,
  and whether "alive empty + consistent" can mean "complete model".

## Acceptance

- `zebra2.ein` → `k = 1`; the complete 25/25 model; the `--mode=solve`
  answer projects from it correctly ("Japanese keeps the zebra; Norwegian
  drinks water").
- `zebra2-minus-15.ein` → `k > 1`; GAPS reports the contingent open set.
- `ein-bugs/zebra2-bad.ein` → `k = 0`; contradiction with a tight unsat
  core.
- No SAT→⊥ (or UNSAT→Solution) regressions anywhere in the suite.

## Phase A — findings (2026-05-31, read-only; code-cited)

- **I1 — generator coverage.** `generate_hypotheses` (`hypgen.py:107`→`_generate`)
  is hrule-driven when `kb.hrules` exists (zebra2's `guess`), else a blind
  type-compatible enumerator. Its filter pipeline (`_apply_filters`) drops
  **refuted** (`_is_excluded` via `_negated_facts`), **present**
  (`_already_a_fact`), and **lookahead-doomed** (`dies_immediately`, which
  back-props `(not h)`) candidates — so it yields exactly the **open set**.
  Coverage for zebra2 = the five `*-loc` grids (the `guess` hrule's scope).
- **I2 — `state_hash`** (`canon.py:27`) is an order-insensitive hash over the
  propositional facts, excludes bookkeeping heads (`hypothesis` /
  `contradiction-under`), keeps `(layer, relation, args)`. Same model ⇒ same
  hash (the 14 grid[1] commitments share one state). *Caveats:* it's a Python
  `hash()` used as a dict key (collision ⇒ undercount; a tuple key is fully
  sound) and is layer-sensitive (fine — derived facts are all REASONING).
- **I3 — solution recording (corrects an earlier wrong claim).**
  `verdict.branches` / `lstate.solutions` append **one `SolutionRecord` per
  satisfying commitment** (`solver.py:999`), with **no state dedup** — that's
  why 15 branches. `kb_index` (`_record_setnode`) keys by `state_hash` under
  *contradictions* (states merge) but by `hash(commitment)` under *gaps*
  (deliberately separate). There is **no** "gaps auto-disables the merge" — it
  keys `kb_index` by commitment, and the solutions list is never deduped.
- **I4 — `_compute_alive` (`solver.py:393`) is `generate_hypotheses` results,
  symmetric-canonicalised.** So **`alive` IS the open-hypothesis set**, and
  `has_open_hypothesis(kb)` already exists as `bool(_compute_alive(kb))`;
  "complete" = `not _compute_alive(kb)`. The completeness predicate is free.
- **I5 — the SAT→⊥ mechanism is worse than a mislabel.** The Contradiction is
  `if not alive: return _contradiction`. In the gate-on run, monotonic's final
  **root** was 12/25 *correct* + **7 cells WRONG** (`Green@H4`, …): the
  unsound part is `_promote_forced_positives` (`solver.py:337`) committing
  **wrong forced-positives into the shared, append-only root** once the search
  was driven past first-goal-match. Contrast gaps' shared root: 12/25 correct,
  **0 wrong** (necessary facts only), with the correct 25/25 model in its
  **per-branch** snapshots.

**Architecture decision (resolves Q-A): build on the lattice, not monotonic.**
The completeness predicate (`not _compute_alive`) is trivially available, and
`state_hash` gives the k-count — but monotonic's shared-root promotion is
unsound under full search, so a *solution node* must be a **per-branch**
commitment KB (`gaps`-style, independent per commitment), never the shared
root. Phase B is scoped around the lattice exploration.

**Caveat to verify in Phase B (do not assume):** the dead-end grid[0] must be
excluded by *consistent ∧ `not _compute_alive`*. Confirm whether grid[0]'s
unresolved cells are **open** (so `_compute_alive` non-empty ⇒ excluded as
incomplete) or **all-refuted** (a totality dead-end ⇒ must be caught as a
contradiction). Verify on the real branch KBs before wiring the predicate in.

## Open questions

- **Q-A — RESOLVED (Phase A): no.** `monotonic_solve` cannot be a sound
  complete-model finder as architected — its shared-root flat-writes +
  forced-positive promotion pollute the root with wrong values once driven
  past first-goal-match (I5). The sound complete-model engine is the
  per-branch lattice. `monotonic_solve` stays a *fast first-goal-match*
  entry (its existing, narrower contract); it is **not** the answer source.
- **Q-B — exhaustiveness vs depth cap.** `k = 0 ⇒ UNSAT` only holds if
  exploration ran to complete-or-dead leaves on every branch; a truncating
  `max_set_size` could fake an UNSAT. Define the termination guarantee.
- **Q-C — goal-as-question vs goal-as-constraint.** This stage treats
  `:goal` as a question projected over models. A puzzle that means `:goal`
  as a *constraint* would instead push it into facts/rules. Confirm the
  intended semantics (likely a [S1.7.5](s1.7.5_query_semantics_who_vs_where.md)
  concern).
