## Call sites

- `closed.py:36–83` — `emit_closed`: writes `(closed R)` for **every
  declared domain relation no compiled rule positively asserts**
  (`producible_relations` walks the engine's `:assert` heads). Runs once,
  before initial saturation. Reads existing `(closed …)` to stay
  idempotent (`closed.py:68`); writes the fact (`closed.py:80`).
- `hypgen.py:328` — `apps = kb._facts_by_relation.get("closed", ())`:
  hypgen **suppresses** hypotheses on any `closed` relation.

## 1. Functionality connected to the name

A relation is *closed* when no rule can ever positively conclude a new
fact of it — its extension is fixed by `(facts)`/`(ontology)`. Closing it
tells hypgen **never to speculate** facts of that relation: a hypothesis
about a closed relation could never be confirmed by saturation, so it
would only bloat the search. `is-a` and `right-of` are auto-closed in
zebra2; `co-located`/`next-to` (rule-propagated) are not. The name does
double duty: `emit_closed` **derives** closure structurally (the "no rule
asserts R" test), and hypgen **consumes** the `(closed R)` fact.

## 2. Consequences if removed

Two halves, two consequences. Drop the **consumer** (hypgen ignores
`closed`): hypgen speculates facts of closed relations — every `is-a` /
`right-of` edge becomes a hypothesis target, a large search blow-up
(soundness intact, tractability lost). Drop the **producer**
(`emit_closed`): closure must then be **authored** (`(closed R)` by
hand) or it vanishes — and a mis-closed relation that genuinely needs
guessing would make the search **incomplete** (a real solution never
hypothesised). The S1.7.4 map flags exactly the rules whose NAF watches a
derived vs declared-only relation — the audit that catches mis-closure.

## 3. How it could be implemented in ein-lang rules (or elsewhere)

The **consumer** side is already data — `(closed R)` is an ordinary fact;
hypgen reading it is the same pattern as any property query, so "remove
the name from hypgen" just means routing it through a generic
`kb.relations_with_property` accessor (the name stays, as data). The
**producer** side — "R is closed iff no rule asserts R" — is a
*meta-property over the rule set*, second-order: it quantifies over rules,
which M1 ein cannot express (no rule can match on "the set of all rules").
So `emit_closed` is **not** expressible as an M1 ein rule; it is a
compile-time analysis. The honest move is to keep `emit_closed` as a
**derivation pass** (like the S1.7.4 NAF map — observability/analysis, not
kernel control) that *emits* the `(closed R)` data the puzzle could also
author. ⇒ consumer: yes, trivially; producer: no, it is metalogic.

## 4. How it could be implemented without sticking to the name

Closed-world soundness ultimately rests on the **NAF primitive**
`absent` ([S1.7.17](s1.7.17_absent.md)) + the derived-NAF distinction the
[S1.7.4 map](s1.7.4_naf_dependency_map.md) draws. If every
domain-elimination rule's `(absent (R …))` guard is *known sound* by that
map, then `(closed R)` is redundant **for soundness** and survives only as
a **search-pruning hint** — at which point it is pure optimisation data,
not a kernel name, and could carry any name (or be inferred on the fly by
the same producible-relations test inlined into hypgen). Pin first what
`(closed R)` does that `absent` + the map do not: it is the *generative*
side (don't propose), where NAF is the *consumptive* side (don't conclude)
— complementary, and that gap is what decides whether the name is
reducible.

## Decision (to record in [S1.7.7 T1.7.7.5](s1.7.7_kernel_purity_analysis.md))

Pin the generative-vs-consumptive split against the S1.7.4 map; decide
whether `emit_closed` stays a **derivation pass emitting data** (the
likely outcome — closure is metalogic, but the *name* in hypgen is just a
property query) or whether closed becomes a hand-authored search hint.
Re-validate the gaps / domain-elimination fixtures either way.
