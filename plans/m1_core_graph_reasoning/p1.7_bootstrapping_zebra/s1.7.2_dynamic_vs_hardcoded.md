# S1.7.2 — Dynamic-vs-hardcoded audit + decision

**Phase:** P1.7 (bootstrapping).
**Estimate:** 4-6 days (was 2-3 before T1.7.2.5 absorbed the
proto-library + zebra-compat sub-tasks on 2026-05-21).
**Depends on:** S1.7.1.
**Implements:** the user's directive — *"first hardcode everything
not yet in the PoC's input file; make a list; later decide what is
dynamic and what is hardcoded"*.

## Context

The PoC's input format (an obj-rel-obj triples file, now removed
from the active tree — see `docs/PoC/`) carried only relations: no
types, no constraints, no ontology, no a-priori facts. The current
Zebra encoding in `examples/zebra.ein` (S1.7.1) externalises all of
that into the IR. Some of it may still want to be *engine-side*
defaults rather than per-puzzle declarations.

This stage produces the **list** the user asked for: everything
that today is hardcoded somewhere, with a column for "stays
hardcoded / moves to IR / moves to puzzle library".

The list is the artefact, not a refactor. Refactor decisions become
follow-up stages or are deferred to the next milestone.

### Task T1.7.2.5 — IR encoding final call + proto-library execution

**Resolved 2026-05-21 (user-directed) — Option B: unified `is-a`
wins.** The canonical form has *no kernel-hardcoded relations
except `(relation …)` and `(rule …)`*; everything else, including
the inheritance relations `is-a` / `type` / `instance` and their
companion rules, lives in user-modifiable ein code. See memory
entry `project_canonical_zebra2` and
[`docs/kernel/ir/01-ein-graph/03_ein_model.md` §6](../../../docs/kernel/ir/01-ein-graph/03_ein_model.md#6-reserved-relation-names).

The two encodings before the decision:

- `examples/zebra.ein` — **classic**: kernel `(type …)` /
  `(instance …)` populate `kb.types` (7) + `kb.instances` (30) as
  cached entities.
- `examples/zebra2.ein` — **unified `is-a`**: no `(type …)` or
  `(instance …)`; the hierarchy is a graph of `(is-a Child Parent)`
  facts; the type structure is only visible after
  `(transitive is-a)` / `(asymmetric is-a)` /
  `(sibling-exclusive is-a)` fire.

T1.7.2.5 ships zebra2 across the engine and provides a
**proto-library** that doubles as the **zebra-original ↔
zebra2-canonical compatibility layer**: a hardcoded chunk of
ein-code the engine always loads before user code, declaring
`instance` / `type` as ordinary relations (`(relation instance T
T)`, `(relation type T)`) and supplying the rules that give them
their semantics. This is how zebra.ein keeps working without any
kernel-level special cases.

#### Sub-tasks

- **T1.7.2.5.a** — Decide the proto-library content. Minimum:
  `(relation instance T T)`, `(relation type T)`, plus
  `(transitive type)`, `(sibling-exclusive instance)`,
  `(asymmetric instance)`. Open question: do `is-a` and `instance`
  collapse to one canonical relation, or coexist?
- **T1.7.2.5.b** — Ship the proto-library as a Python string + an
  always-loaded path in `KnowledgeBase.from_ir`; record its facts
  with their own provenance kind (`kind="proto-library"`).
- **T1.7.2.5.c** — Remove the `(type …)` / `(instance …)` special
  cases from `from_ir.py` once the proto-library covers them.
  `kb.types` and `kb.instances` reduce to *derived views* over
  the `instance` / `type` Facts (computed on demand from
  `kb.names` plus the relation's signature).
- **T1.7.2.5.d** — Remove the temporary
  `KERNEL_META_RELATIONS = frozenset({"relation","rule","instance","type"})`
  hardcode from `ein.py/src/ein_bot/kb/entities.py`; once
  `instance` and `type` are real declared relations, the names
  index categorises them correctly without the special case.
  (The hardcode was introduced under the P1.5 names-index
  migration commit; that commit's TODO points here.)
- **T1.7.2.5.e** — Migrate every remaining test fixture using
  kernel `(type …)` / `(instance …)` to zebra2. The P1.5
  migration (commit 2026-05-21) covered
  `ein.py/tests/inference/test_hypothesis.py` and
  `test_multilevel.py`; this sub-task sweeps the rest of the
  suite + any demos still in classic form.
- **T1.7.2.5.f** — Acceptance: zebra.ein and zebra2.ein both
  load, saturate, and solve under the proto-library; the
  resulting traces are equivalent modulo `is-a` ↔ `instance`
  rewriting; full test suite green.
- **T1.7.2.5.g** — Decision artefact: write `docs/ir.md`
  §IR-encoding-final, recording the 2026-05-21 resolution,
  pointing at the proto-library, and documenting the
  desugaring/migration path for any future puzzle still
  authored in classic style.
