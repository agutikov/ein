## The decision — full purity (2026-05-30, user-directed)

The kernel keeps only the declarators (`relation` / `rule`, plus the
`hrule` variant) and the logical primitives (`not` `false` `and` `or`
`neq` `forall` `absent`). **Everything else leaves the kernel** and is
re-expressed as ein rules (or deleted, for `a-priori`), with `zebra2.ein`
re-validated end-to-end.

Sub-decisions locked today:

- **`type`/`instance` stay writable** as ordinary generic facts on
  user-declared relations. `(type UserType T)` is the **basic two-arg
  form** (the type + its parent, defaulting to the top type `T`);
  `(instance Thing Type)` likewise. Their *meaning* moves from the loader
  into rules authored **in `zebra.ein`** (bridging `instance`→`is-a`,
  type-parent→`is-a`, sibling-exclusivity). **Do not** rewrite `zebra.ein`
  into `is-a`.
- **No equivalence / no solve requirement on `zebra.ein`.** It is a
  *demonstrator* that type/instance are expressible in user space; it may
  fail to solve. The old **proto-library / compat-layer** framing of
  T1.7.2.5 is **dropped** — no always-loaded ein chunk, no "traces
  equivalent modulo is-a↔instance". `zebra2.ein` stays canonical.
- **`a-priori` is dropped**, not relocated — a no-op alias of `relation`
  used in zero `.ein` files (only a parser round-trip test).
- **`or` stays kernel.** Design note (user, 2026-05-30): since ein's
  *multiple-rules* semantics is already **disjunctive**, an `(or …)` in a
  `:match` should **lower into multiple rule instances** (one per
  disjunct), not a runtime branch. Tasked in
  [S1.7.6](s1.7.6_kernel_minimization.md).
- **`is-a`/`closed`/`open` are load-bearing** (zebra2's solve depends on
  inheritance-pruned hypgen + closed-world soundness). Removing them is
  real work + risk → parked as the analysis stage
  [S1.7.7](s1.7.7_kernel_purity_analysis.md), which may legitimately
  conclude an **irreducible kernel**. (`symmetric` is *not* load-bearing
  — its rule already exists; it's a dedup in S1.7.6.)
