# S1.7.2 — Dynamic-vs-hardcoded audit + decision

**Phase:** P1.7 (bootstrapping).
**Estimate:** 2-3 days.
**Depends on:** S1.7.1.
**Implements:** the user's directive — *"first hardcode everything
not yet in the PoC's input file; make a list; later decide what is
dynamic and what is hardcoded"*.

## Context

The PoC's input format (an obj-rel-obj triples file, now removed
from the active tree — see `docs/PoC/`) carried only relations: no
types, no constraints, no ontology, no a-priori facts. The current
Zebra encoding in `examples/zebra.ein` (S1.7.1) externalises all of
that into the IR. Some of it may still want to be *engine-side*
defaults rather than per-puzzle declarations.

This stage produces the **list** the user asked for: everything
that today is hardcoded somewhere, with a column for "stays
hardcoded / moves to IR / moves to puzzle library".

The list is the artefact, not a refactor. Refactor decisions become
follow-up stages or are deferred to the next milestone.

## Tasks

### Task T1.7.2.1 — Walk the codebase

Grep for and tabulate every place where the engine assumes
something puzzle-specific. Candidates:

- Pet/Color/Nationality/Drink/Cigarette type names (assumed
  alphabetical-friendly atom names in tests).
- Position bounds (`1..5`) — currently per-type `:bounds`.
- The five spatial relations + their inverses.
- The set of rule priorities (S1.3.3 banding).
- The S2 symmetry group support (S1.5.3) — only S2 is implemented.
- Per-type cardinality assumptions (`functional`, `1..1`).

Output: `docs/dynamic-vs-hardcoded.md` with a table:

| thing                         | currently lives in | declared in IR? | should move? | reason                       |
|-------------------------------|--------------------|-----------------|--------------|------------------------------|
| Position bounds `1..5`        | ontology IR        | ✓               | stays IR     | per-puzzle; no engine default needed |
| `all-different` constraint    | constraints IR     | ✓               | stays IR     | puzzle-specific              |
| S2 symmetry only              | engine             | partially       | engine, doc-as-known-limit | richer groups → F4 |
| Rule priority banding         | engine config      | ✗               | engine        | universal heuristic; revisit on a non-Zebra puzzle |
| Spatial position lattice (1-D) | engine            | partially       | engine (via `:positional`) | works for Zebra; would need 2-D variant for chess-style |
| ...                            | ...                | ...             | ...          | ...                          |

### Task T1.7.2.2 — Decision log

For each row marked "should move", create a short follow-up note
in [`plans/ideas.md`](../../../ideas.md) (Live entries). Don't act
in M1 — capture the call site + the trigger condition (e.g.,
"when adding a 2-D puzzle, generalise spatial lattice").

### Task T1.7.2.3 — Engine config seam

Promote the rule priority banding and the S2-only flag into
`src/ein_bot/config.py` so they have a single place to evolve.
This is a refactor *within* M1 to avoid surprise: the next-milestone
NL pipeline will want to override priorities.

### Task T1.7.2.4 — Tests

`tests/integration/test_audit_coverage.py`: assert that
`docs/dynamic-vs-hardcoded.md` lists every entry from a programmatic
introspection sweep. Catches drift — if someone adds a new hardcoded
assumption, the audit table must grow too.

## Acceptance

- `docs/dynamic-vs-hardcoded.md` exists with ≥ 8 rows.
- `plans/ideas.md` "Live entries" gains a section linking each
  "should move" row to its trigger.
- `src/ein_bot/config.py` exists and is imported by the engine.
- The drift test passes.
