# S1.7.15 — hardcoded name `or` (disjunction → multiple rules)

**Phase:** P1.7 (bootstrapping) — per-name kernel-purity analysis.
**Parent:** [S1.7.7](s1.7.7_kernel_purity_analysis.md) (inventory).
**Class:** logical primitive — **lowered to rules** at load.
**Verdict:** **stays kernel as a primitive, but already de-engined** —
[S1.7.6 T1.7.6.5](s1.7.6_kernel_minimization.md) lowered top-level `or` to
one rule per disjunct; no runtime `or` branch remains.
**Status:** analysis (mostly settled; this records the residue).

## Call sites

- `from_ir._match_disjuncts` (loader) — a **top-level** `(or d1 … dn)` in
  a `:match` is desugared **at load time** into one `Rule` per disjunct
  (`<rule>__or0…n-1`), exploiting ein's already-disjunctive
  multiple-rules semantics (S1.7.6 T1.7.6.5).
- `compile.py:342–343` — the residual arm: a **nested** `(or …)` (inside
  `(and …)`) emits nothing — DNF expansion is unsupported/deferred.
- `kb/pattern.py:73` — pattern-side `or` handling (loader).

## 1. Functionality connected to the name

`or` is disjunction in a `:match`. ein's design note (user, 2026-05-30):
because **multiple rules with the same conclusion are already
disjunctive**, an `(or …)` should **lower into multiple rule instances**,
not a runtime branch. So `or`'s "functionality" is entirely a
**load-time rewrite** — by saturation time there is no `or`, just N
sibling rules. The only live engine code is the *nested-or* guard that
declines to compile (DNF deferred).

## 2. Consequences if removed

Top-level: **none at runtime** — it is already gone after load; removing
the loader lowering would force authors to write the N rules by hand
(verbose, error-prone, but equivalent). Nested `(or …)` is already
unsupported, so removing its compiler arm changes nothing but a silent
skip. The name is the **lowest-risk** removal target: the engine proper
does not branch on `or` at all.

## 3. How it could be implemented in ein-lang rules (or elsewhere)

It already **is** "implemented as rules" — that is the whole S1.7.6
T1.7.6.5 lowering: `(rule r :match (or A B) :assert C)` ≡ two rules
`(rule r :match A :assert C)` + `(rule r :match B :assert C)`. ein's
disjunctive-multiple-rules semantics **is** the implementation of `or`. No
deeper primitive is needed; `or` is sugar over the rule *set*.

## 4. How it could be implemented without sticking to the name

The lowering is name-keyed only at the loader (`_match_disjuncts` sees the
head `"or"`). To drop the name from the *engine* entirely, the lowering
could move to a **user-space macro** (`defmacro or …` expanding to sibling
rules) once a macro layer exists ([P1.8](../p1.8_ein_lang_modules/)) — then
`or` is library data, not a loader case. Nested `or` (DNF) would, if ever
needed, be the same macro applied recursively, still producing only base
rules. So the name can leave the kernel completely the moment desugaring
is user-authored.

## Decision (to record in [S1.7.7 T1.7.7.5](s1.7.7_kernel_purity_analysis.md))

Record `or` as **already-de-engined**: top-level disjunction is a
load-time rewrite to the disjunctive rule set, no runtime presence; the
nested arm is dead/deferred. Candidate to move from loader into a stdlib
macro with `open`/`forall` once P1.8 lands. Lowest risk in the inventory.
