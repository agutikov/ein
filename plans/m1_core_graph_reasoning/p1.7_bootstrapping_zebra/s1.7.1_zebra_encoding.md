## Context

**Re-scoped 2026-05-30.** This stage was written pre-P1.5a/b to *design*
the Zebra encoding from scratch in the classic
`(type …)` / `(instance …)` / `:cardinality` / definite-description
vocabulary, split across `ontology.ein` / `facts.ein` / `query.ein`.
That vocabulary is dead. The canonical encoding shipped during P1.5a as
[`examples/zebra2.ein`](../../../examples/zebra2.ein) — a single file using:

- the **B1 `*-loc` bijections** (`color-loc` / `nation-loc` /
  `drink-loc` / `smoke-loc` / `pet-loc`),
- unified **`is-a`** inheritance (no kernel `type` / `instance`),
- `co-located` / `adjacent-via` activators for the 15 conditions,
- a `(query :mode solve :goal … :hrules guess …)` block.

It **solves uniquely on CPython** —
`tests/inference/monotonic/test_monotonic_skeleton.py::test_zebra2_solves_via_monotonic_backbone`
(~5 s, no `EIN_RUN_SLOW` gate; `max_set_size=2`).

So the original T1.7.1.1–.3 (ontology / facts / query authoring in classic
syntax) are **obsolete** — superseded by `zebra2.ein`. What remains is the
two *variant* fixtures the three-task-class acceptance ([S1.7.3](../p1.7a_solution_search_refactor/s1.7.3_trace_acceptance.md))
needs, authored in the **same canonical syntax**, plus a parse smoke test.

(Note the classic `examples/zebra.ein` is **not** deleted — it becomes the
demonstrator that `type` / `instance` are expressible as user-space
relations once [S1.7.6](s1.7.6_kernel_minimization.md) lands. It is not
required to solve.)

## Tasks

### Task T1.7.1.1 — (done) Canonical encoding
`examples/zebra2.ein`. No work; recorded as the baseline the variants
fork from.

### Task T1.7.1.2 — GAPS fixture  ✅
`examples/zebra2-minus-15.ein` — `zebra2.ein` minus condition (15)
("the Norwegian lives next to the blue house" — the relevant
`adjacent-via` activator). Expected: ambiguous; `gaps_solve` reports
≥ 1 diverging goal node. Keep it a thin diff of `zebra2.ein` (same
ontology + rules; drop the one fact) so it stays in sync.

### Task T1.7.1.3 — CONTRADICTIONS fixture  ✅
`examples/ein-bugs/zebra2-bad.ein` — `zebra2.ein` plus the injected
`(color-loc Green House-1 :source "injected contradiction")` colliding
with condition (6): Green is the `right-of` neighbour of Ivory, so it
cannot sit in the leftmost house, and `adjacent-via-endpoint-bwd` derives
`(not (color-loc Green House-1))` from (6) alone. The co-located bridge for
(4) (Coffee ↔ Green) then carries both the positive and the negative onto
`drink-loc Coffee House-1` — a **same-layer pair the detector flags at
d=0** (verified by `test_bad_is_unsat_at_root_saturation`), so
`contradictions_solve` gets a tight 2–3 edge unsat core fast.

**Placement (user decision 2026-05-30):** the file lives under
`examples/ein-bugs/` (alongside `zebra2-hints.ein`), **not** in
`examples/broken/`. `broken/` is strictly *lint-failure* fixtures —
`tests/test_cli.py::test_broken_fixtures` asserts each one exits non-zero
— whereas `zebra2-bad.ein` parses cleanly and is only *semantically*
unsatisfiable. Kept out of the top-level `examples/` to avoid sitting a
deliberately-broken puzzle next to the canonical encoding.

### Task T1.7.1.4 — Parse smoke test  ✅
`tests/integration/test_zebra_parse.py` (new `tests/integration/` dir,
with `__init__.py`): all three canonical files parse cleanly, load, and
carry the nine declared B1 relation signatures + a query block. The
thin-diff contract is pinned **relatively** — variants match `zebra2` on
relations / rules / ontology and differ by exactly the one FACT-layer fact
dropped (minus-15) or added (bad) — anchored by a landmark `zebra2`
fact-count (18 = the 15 conditions, (1) expanding to 4 `right-of` facts).
A malformed (unbalanced-paren) variant is rejected with `IRParseError`.
Two `EIN_RUN_SLOW`-gated checks validate the fixtures are *fit for
purpose* — `-bad` is UNSAT at d=0 root saturation, `zebra2` is not — so
they don't count against the default suite (the SOLVE/GAPS/CONTRADICTIONS
CLI acceptance proper is S1.7.3). 8 ungated + 2 gated tests, all green.
