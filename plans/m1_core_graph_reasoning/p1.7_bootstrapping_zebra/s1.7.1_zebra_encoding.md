## Context

**Re-scoped 2026-05-30.** This stage was written pre-P1.5a/b to *design*
the Zebra encoding from scratch in the classic
`(type ‚Ä¶)` / `(instance ‚Ä¶)` / `:cardinality` / definite-description
vocabulary, split across `ontology.ein` / `facts.ein` / `query.ein`.
That vocabulary is dead. The canonical encoding shipped during P1.5a as
[`examples/zebra2.ein`](../../../examples/zebra2.ein) ‚Äî a single file using:

- the **B1 `*-loc` bijections** (`color-loc` / `nation-loc` /
  `drink-loc` / `smoke-loc` / `pet-loc`),
- unified **`is-a`** inheritance (no kernel `type` / `instance`),
- `co-located` / `adjacent-via` activators for the 15 conditions,
- a `(query :mode solve :goal ‚Ä¶ :hrules guess ‚Ä¶)` block.

It **solves uniquely on CPython** ‚Äî
`tests/inference/monotonic/test_monotonic_skeleton.py::test_zebra2_solves_via_monotonic_backbone`
(~5 s, no `EIN_RUN_SLOW` gate; `max_set_size=2`).

So the original T1.7.1.1‚Äì.3 (ontology / facts / query authoring in classic
syntax) are **obsolete** ‚Äî superseded by `zebra2.ein`. What remains is the
two *variant* fixtures the three-task-class acceptance ([S1.7.3](s1.7.3_trace_acceptance.md))
needs, authored in the **same canonical syntax**, plus a parse smoke test.

(Note the classic `examples/zebra.ein` is **not** deleted ‚Äî it becomes the
demonstrator that `type` / `instance` are expressible as user-space
relations once [S1.7.6](s1.7.6_kernel_minimization.md) lands. It is not
required to solve.)
