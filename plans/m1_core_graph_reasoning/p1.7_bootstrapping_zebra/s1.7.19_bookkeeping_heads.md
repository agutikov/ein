# S1.7.19 — hardcoded names `hypothesis` / `contradiction-under` (bookkeeping carrier heads)

**Phase:** P1.7 (bootstrapping) — per-name kernel-purity analysis.
**Parent:** [S1.7.7](s1.7.7_kernel_purity_analysis.md) (inventory).
**Class:** bookkeeping carrier-head relations (graph-level provenance facts).
**Verdict:** **derivable from provenance** — these are markers the search
adds, not ontology; their special-casing is a hashing concern.
**Status:** analysis.

## Call sites

- `canon.py:21–24` — `BOOKKEEPING_HEADS = {"hypothesis",
  "contradiction-under"}`; `state_hash` **excludes** facts with these
  heads so two branches with identical propositional content but different
  branch-marker facts collapse to one search-tree node (the DAG dedup).
- `try_branch` (P1.5 hypothesis loop) **adds** `(hypothesis …)` /
  `(contradiction-under …)` carrier facts to mark "which branch is this".
- `"hypothesis"` is **also** a provenance kind (see
  [S1.7.22](s1.7.22_engine_internal_strings.md)) and a `_SPECULATIVE_KINDS`
  member (`back_prop.py:89`) — distinct use, same word.

## 1. Functionality connected to the name

These are **carrier facts** the search machinery writes into a branch's KB
to record *which hypothesis the branch is exploring* (`hypothesis`) and
*which contradiction a learned nogood sits under* (`contradiction-under`,
the Q40 carrier). They are **not puzzle ontology** — no rule matches on
them; they are the search's own marginalia. `canon.py` must exclude them
from the canonical state hash, or two otherwise-identical closed KBs would
hash differently merely because they carry different branch markers,
defeating the DAG-storage dedup.

## 2. Consequences if removed

Remove the **exclusion** (keep writing the facts): state-hash dedup
**breaks** — equivalent branches stop collapsing, the search tree no
longer becomes a DAG, redundant re-exploration inflates wall-time
(correctness intact, performance lost). Remove the **carriers
themselves**: the search loses its branch-identity / nogood-attribution
marginalia — traces and Q40 nogood learning lose their anchor. So the two
names are coupled: the carriers *require* the hash exclusion to coexist
with dedup.

## 3. How it could be implemented in ein-lang rules (or elsewhere)

These should **not** be ein relations at all — they are search
bookkeeping, and the fact that they live *as facts* (so they ride the KB's
indexing/fork machinery for free) is the only reason they have relation
heads. The cleaner model: branch identity and nogood attribution are
**provenance / SearchNode metadata**, not KB facts — then `canon.py` needs
no exclusion list because the hash already ignores non-fact metadata.
Expressing them *as rules* is meaningless (no derivation produces them; the
search engine does). So the route is the **opposite** of "make it a rule":
demote them from facts to engine-side node metadata.

## 4. How it could be implemented without sticking to the name

Move branch-marking off the fact list onto the `SearchNode` /
`Provenance` object (a `branch_id` / `under_contradiction` field). Then:
(a) no `"hypothesis"` / `"contradiction-under"` relation heads exist in
the KB; (b) `state_hash` drops `BOOKKEEPING_HEADS` entirely (it hashes
facts, and these stop being facts); (c) the `"hypothesis"` *provenance
kind* (a separate concern, S1.7.22) is unaffected. This is the honest
de-naming: the names vanish because the *data* moves to where it belongs
(node metadata), not because it is renamed.

## Decision (to record in [S1.7.7 T1.7.7.5](s1.7.7_kernel_purity_analysis.md))

Record these as **search bookkeeping mis-encoded as facts** — candidates
to migrate to `SearchNode`/`Provenance` metadata, which dissolves both the
carrier heads *and* the `canon.py` exclusion list. Not a kernel-language
question (they are never puzzle-authored); a data-model cleanup. Low
priority unless the DAG-dedup hash exclusion proves fragile.
