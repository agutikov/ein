# S1.7.8 — hardcoded names `is-a` + `T` (inheritance / hypgen type-pruning)

**Phase:** P1.7 (bootstrapping) — per-name kernel-purity analysis.
**Parent:** [S1.7.7](s1.7.7_kernel_purity_analysis.md) (inventory + spike).
**Class:** inheritance relation, engine-special.
**Verdict (prior, S1.7.2):** "load-bearing search optimisation". **Revised
by the [T1.7.7.1 spike](#spike-result-t1771--2026-06-01) (2026-06-01):
de-hardcoding `is-a` is viable** — its kernel type-pruning is *redundant*
on the path zebra2 actually uses, and *dead code* on the gate.
**Status:** spike done; design route open (§3/§4).
**Not an M1 gate** — zebra2 solves with `is-a`, and with the kernel
type-filter neutralised, **identically**.

## Design principle (user, 2026-06-01)

> The kernel **must not impose its own type system** for objects and
> relations. If there are rules defining substitution and inheritance,
> *then there is a type system — defined by the user* in ein-lang (or
> reused from a P1.8 stdlib). So from the kernel's point of view **`is-a`
> must be just another relation, nothing more.**

This stage adopts that as the target: every kernel site that *interprets*
`is-a` as a type lattice (the entity-views, the hypgen type-filter) is a
kernel-imposed type system to remove; the inheritance *semantics* stay,
but as user rules over an ordinary `is-a` relation. The spike below shows
zebra2 already works this way.

## Call sites

- `hypgen.py:45` — `INHERITANCE_RELATIONS = ("is-a",)`, the name constant.
- `hypgen.py:243,273` — `_type_compatible(kb, obj, sig_type)`: in the
  **blind enumerator** (`_raw_candidates`/`_fill_slot`), admits a
  candidate only if its `is-a` ancestry (`_ancestor_names`, `hypgen.py:378`)
  reaches the slot's declared type. **This whole path runs only when the
  puzzle declares *no* `(hrule …)`** (`_generate`, `hypgen.py:157`).
- `hypgen.py:162,490` — `_instance_like_objects`: graph leaves are
  "instance-like" iff they sit at slot 0 of an `is-a` edge and never at
  slot 1 (the candidate *object set*, a separate `is-a` use from the
  type-filter; runs on both paths).
- `kb/views.py:130,156` — `logical_types` / `logical_instances`: the
  **entity-view** (see §1). `kb/render.py:105,273` draw the is-a edge.

## Spike result (T1.7.7.1 — 2026-06-01)

Script: `ein.py/demo/spike_isa_pruning.py` (`./bench_isa_spike_pypy.sh`).
It neutralises `hypgen._type_compatible` (→ always True) and re-solves.

**On zebra2 (the M1 gate): no-op.** Baseline and neutralised are
bit-identical — `Solution k=1`, `enterings_total=11`, `raw=125`,
`emitted=56`, **`type-incompat skips = 0` even with pruning ON**. Reason:
zebra2 declares `(hrule guess (?R ?T1 ?T2) :match (and (is-a ?a ?T1)
(is-a ?b ?T2)) :assert (?R ?a ?b))`, so generation is **hrule-driven**;
the blind `_raw_candidates` path — and with it `_type_compatible` — is
**dead code** for zebra2. *The gate cannot see this filter at all* (same
blind-spot shape as [S1.7.9 symmetric](s1.7.9_symmetric.md) Q5).

**On the blind-enumerator path (no-hrule, non-`T` typed puzzles): the
filter is real.** `features/05_stdlib_domain_elim.ein` (`color-of House
Color`): baseline `type-incompat skips=18`, neutralised `=0`, root `raw`
**18 → 60 (3.3×)**; `features/04_open.ein`: `emitted` **2 → 23 (11.5×)**.
So where it runs, it prunes substantially. *Caveat:* neither puzzle
**solves** either way (`k=0`, both abort the 50 k-entering budget — they
are non-converging feature demos, not solve fixtures), so the spike
cannot show a clean "solves → blows up" flip on a committed puzzle.
**No committed solving puzzle exercises `_type_compatible`.**

Headline: per the gate's own question ("still solves in budget?") the
answer is **yes, trivially** — de-hardcoding is viable. The filter is a
real optimisation *only* on the blind path, which the canonical encoding
has already replaced with the `guess` hrule.

## 1. Functionality connected to the name

Two distinct things the kernel does with `is-a`, both forms of "imposing
a type system":

**(a) The entity-view.** `kb/views.py` reads `(is-a Child Parent)` facts
and *synthesises a type/instance taxonomy*: `logical_types` returns the
parents (RHS of `is-a`) as "the types"; `logical_instances` returns the
leaves (names that are a child but never a parent) as "the instances".
Together with the `kb.types`/`kb.instances` registries and
`_instance_like_objects`, this is the engine **interpreting `is-a` as an
inheritance forest** — deriving a typed entity-API view over what is, at
the graph level, just a set of binary facts. ("Entity-view" = this
derived types-and-instances picture, as opposed to the raw fact list.)

**(b) The hypgen type-filter.** In the blind enumerator, `_type_compatible`
walks `is-a` ancestry to keep only candidates whose object reaches the
slot's declared type. **In zebra2 this job is done instead by the `guess`
hrule** matching `(is-a ?a ?T1) (is-a ?b ?T2)` — i.e. the *exact same
type-pruning, expressed declaratively in user space over `is-a`-as-a-plain-
relation*. The kernel filter is the blind-path fallback for puzzles that
don't write such an hrule.

Transitivity of `is-a` is already a plain user rule (`(transitive is-a*)`
in zebra2). So **none of the inheritance *semantics* needs the kernel** —
(a) and (b) are the only kernel impositions, and (b) is already replaced
by an hrule on the canonical path.

## 2. Consequences if removed

**Remove the type-filter (b):** zebra2 — **nothing** (dead code, measured).
Blind-path puzzles lose 3.3× candidate pruning and lean on contradiction/
lookahead to kill type-incompatible hypotheses; tractability suffers, but
soundness does not (the filter only ever removed provably-type-wrong
candidates). The fix for those puzzles is to give them a `guess`-style
hrule, not to keep the kernel filter.

**Remove the entity-view (a):** `kb.types`/`kb.instances` and the
`logical_*` helpers lose their source. These feed renderers, some tests,
and `_instance_like_objects` (the candidate object set). They are *derived*
and recomputable from `is-a` facts by a user/stdlib rule — but removing
the kernel view means anything reading `kb.types` must instead query
`is-a` facts (or a user-defined `(type …)`). This is the larger surface
and the real work of "`is-a` is just a relation".

## 3. How it could be implemented in ein-lang rules (or elsewhere)

It already is, on the canonical path: **the `guess` hrule is the
type-pruner**, written in ein-lang, matching `is-a` as a plain relation —
the kernel contributes only the generic hrule machinery. The de-hardcoding
is therefore mostly *deletion + migration*, not new rule design:

- **Type-filter (b):** retire the blind enumerator's `_type_compatible`
  (or gate it off); require puzzles that want type-pruned guessing to use
  a `guess`-style hrule (zebra2's pattern). The kernel stops walking
  `is-a` ancestry entirely. Optionally ship the `guess` hrule + the
  `is-a`/`transitive`/`sibling-exclusive` rules as a **P1.8 stdlib module**
  so puzzles import a type system rather than the kernel embedding one.
- **Entity-view (a):** make `kb.types`/`kb.instances` either (i) thin
  queries over `is-a` facts with **no privileged status** (a convenience
  accessor, not a type system), or (ii) user-space — a puzzle that wants
  named types writes `(type …)` facts and a rule, exactly as
  [zebra.ein does since S1.7.6](s1.7.6_kernel_minimization.md). Renderers
  read whichever relation the puzzle declared inheritance with.

## 4. How it could be implemented without sticking to the name

The principle says the kernel shouldn't key on `"is-a"` at all. Two
levels:

- **Best (no name):** the kernel names *no* inheritance relation. Type
  pruning is whatever `:match` an hrule writes; the "type system" is
  wherever the puzzle/stdlib puts it. `INHERITANCE_RELATIONS`,
  `_type_compatible`, `_ancestor_names`, and the `views.py` is-a arms all
  delete. `_instance_like_objects` (the object set) becomes the one open
  question — see below.
- **Fallback (name → data):** if the blind enumerator must stay, replace
  the literal `"is-a"` with a property the engine reads generically —
  `(inheritance ?R)` — so `INHERITANCE_RELATIONS` is a KB query, not a
  Python tuple ([F5 rules-as-data](../../followups/f5_rules_as_data.md)).
  This relocates rather than removes the imposition; the principle prefers
  the first level.

**Open sub-question — the candidate object set.** `_instance_like_objects`
still needs *some* notion of "which graph nodes are guessable leaves". On
the hrule path this is moot (the hrule's `:match` supplies the objects via
`is-a`); on a name-free blind path it would need a generic signal (e.g.
"nodes of `object` category that aren't relation/type nodes"). Pin this
when/if the blind enumerator is kept.

## Companion name: `T` (the universal top type) — goes with `is-a`

User (2026-06-01): *"together with `is-a` should go `T`."* Correct, and
cleaner than `is-a` itself:

- **Reserved?** No. `T` is an ordinary capitalised atom — the grammar
  does not reserve it; the kernel doc calls it *"the convention atom
  `T`"*. It is not in the §6 reserved-names table.
- **Hardcoded?** **Exactly one Python site:** `hypgen.py:373`, inside
  `_type_compatible` — `if sig_type == obj_name or sig_type == "T":
  return True`. That is its *only* string-literal special-case in all of
  `src/` (grep-verified). Everywhere else (`(relation is-a T T)`,
  `(is-a Attribute T)`, `House is-a* T`) `T` is **data** the user-space
  typecheck rules + signatures reference.
- **Function:** the **top of the `is-a` lattice** / universal "any" type
  — a `T`-typed slot is compatible with every object (skips the ancestry
  walk), so `(relation R T T)` and unrooted ontologies still type-check.

Because its sole consumer is `_type_compatible` — the function this stage
retires — **the `"T"` hardcode disappears for free when (b) goes**, and is
**already dead on the gate** (hrule path never calls it; measured no-op).
Afterwards `T` is a pure puzzle convention: an atom naming whatever top
the puzzle's `is-a` facts declare. *Caveat:* `T` encodes an **implicit**
"compatible with anything" the explicit `is-a` forest doesn't always
state; if any blind-enumerator survives, that universal-top must become
**data** (a `(is-a ?x T)` rule, or a `(top T)` annotation read
generically), not a Python `== "T"`. On the hrule path it is moot.

## Decision (to record in [S1.7.7 T1.7.7.5](s1.7.7_kernel_purity_analysis.md))

**De-hardcoding `is-a` (and `T` with it) is viable and aligned with the
user principle.**
The spike shows the kernel type-filter is redundant (zebra2 prunes via the
`guess` hrule) and dead on the gate. Recommended route, lowest-risk first:

1. **Type-filter (b):** retire / gate `_type_compatible`; the canonical
   path is unaffected (measured). Add a no-hrule typed regression fixture
   *that actually solves* so the blind path isn't silently broken (the
   gate can't see it — like the symmetric fixture, S1.7.9 Q5).
2. **Entity-view (a):** demote `kb.types`/`kb.instances` to name-free
   queries or user-space `(type …)`; this is the bigger migration and the
   real "`is-a` = just a relation" payoff. Candidate P1.8 stdlib home for
   the inheritance rules + the `guess` hrule.

Not an irreducible kernel: the inheritance *semantics* are already user
rules; the only kernel residue is a redundant optimisation + a derived
view, both removable.
