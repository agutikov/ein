# S1.7.8 — hardcoded name `is-a` (inheritance / hypgen type-pruning)

**Phase:** P1.7 (bootstrapping) — per-name kernel-purity analysis.
**Parent:** [S1.7.7](s1.7.7_kernel_purity_analysis.md) (inventory + spike).
**Class:** inheritance relation, engine-special.
**Verdict (prior, S1.7.2):** **load-bearing** — not a mere derivation but
a hypothesis-generation optimisation. This stage tests whether that holds.
**Status:** analysis (parked with [S1.7.7](s1.7.7_kernel_purity_analysis.md));
gated by the [T1.7.7.1 spike](s1.7.7_kernel_purity_analysis.md#tasks).
**Not an M1 gate** — zebra2 solves with `is-a` as a primitive.

## 1. Functionality connected to the name

`is-a` is the inheritance forest. The engine reads it for **two**
things: (a) deriving the entity-view of "what is a subtype/instance of
what" (`views.py`), and (b) **type-pruning hypothesis generation** —
before hypgen proposes `(R x …)`, `_type_compatible` walks the `is-a`
ancestry of `x` and keeps the candidate only if it reaches `R`'s declared
slot type. Transitivity of `is-a` itself is *already* a plain user rule
(`(rule transitive is-a)`); the **hardcoded** part is exclusively the
hypgen pruning + the view derivation, both of which read the literal
relation name `"is-a"`.

## 2. Consequences if removed

Drop the pruning and hypgen proposes **type-incompatible** hypotheses
(e.g. `(drink House-1 Milk)`), leaning on contradiction rules to kill
them one branch at a time — a potential combinatorial blow-up (the
S1.7.7 worry: depth-6, ~60 candidates → 60⁶ branchings if pruning is
off). Drop the views and the entity-API surface (`kb.types`,
ancestry walks in `_ancestor_names`) loses its source — but those are
*derived* and could be recomputed from any inheritance edge. The
**soundness** of the answer is unaffected either way (pruning only
removes provably-dead candidates); the risk is **tractability**, which
is exactly what [T1.7.7.1](s1.7.7_kernel_purity_analysis.md) measures.

## 3. How it could be implemented in ein-lang rules (or elsewhere)

The pruning is a *negative* constraint — "don't hypothesise `(R x v)`
when `x` is not an `R`-compatible type". ein already expresses the
positive form: a `typecheck-arg` rule asserts `(false …)` when a fact's
argument violates the slot type. So the cleanest rule-sourced route is
**make hypgen consult the puzzle's own type guards** instead of a Python
ancestry walk: hypgen proposes broadly, then a *lookahead* over the
existing typecheck rules prunes — i.e. fold pruning into the
already-present `lookahead.py` (kill a candidate iff a type rule would
immediately derive ⊥). That keeps the type knowledge in `zebra2.ein`
rules; the engine keeps only the generic lookahead. Open risk: lookahead
per-candidate may cost more than the O(ancestry) walk it replaces — the
spike's wall-time delta decides.

## 4. How it could be implemented without sticking to the name

Replace the literal `"is-a"` with a **declarative annotation the engine
reads generically**: a puzzle marks its inheritance relation(s) with a
property fact — `(inheritance is-a)` (or reuse `(transitive R)` +
`(closed R)` already present) — and `INHERITANCE_RELATIONS` becomes a KB
query (`kb.relations_with_property("inheritance")`) rather than a Python
tuple. This is the [F5 rules-as-data](../../followups/f5_rules_as_data.md)
move: the *name* stops being privileged, but the *role* "this relation is
the type lattice the pruner walks" is still declared — relocating the
hardcode from Python into one puzzle-authored fact. Whether that counts
as de-hardcoding or just relocation is the honest question (cf. S1.7.7
Q3 for the symmetric analogue).

## Decision (to record in [S1.7.7 T1.7.7.5](s1.7.7_kernel_purity_analysis.md))

Run the [T1.7.7.1 spike](s1.7.7_kernel_purity_analysis.md) first
(neutralise `_type_compatible`, re-solve zebra2, record solve/wall-time/
hypothesis-count deltas). Then either: **(a)** route pruning through a
property-annotated generic query + lookahead (§3/§4) and re-validate
solve + idea-08 firing test + wall-time; or **(b)** record `is-a` pruning
as an **irreducible search optimisation** with the spike numbers (a
legitimate result — the name leaves the language only if the property
annotation in §4 is accepted as "data not hardcode").
