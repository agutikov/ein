## Context

The original Zebra puzzle's two questions are:

> 1. Who drinks water?
> 2. Who owns the zebra?

Both ask for a **Nationality**, not a House. The current
`(query)` block in [`examples/zebra2.ein`](../../../examples/zebra2.ein)
encodes:

```lisp
(query
  :mode solve
  :goal (and (drink-loc Water ?h_water)
             (pet-loc   Zebra ?h_zebra)))
```

— which binds `?h_water` and `?h_zebra` to House values. The
engine has the data to derive `(nation-loc ? ?h)` for both houses
(both `Water` and `Zebra` co-locate with `Norwegian` and `Japanese`
respectively in the canonical solution), but the goal pattern
stops at the House. The trace renderer ([P1.6](../p1.6_rendering_and_trace/))
would say "Water is in House-1, Zebra is in House-5" rather than
"Norwegian drinks Water, Japanese owns the Zebra".

## Design space

Three options for surfacing the partner-attribute answer:

### (1) `:project` keyword on the goal

Extend the query AST with a projection list — after the goal
binds the goal's free variables, project each binding through
a named relation chain:

```lisp
(query
  :mode solve
  :goal    (and (drink-loc Water ?h_water)
                (pet-loc   Zebra ?h_zebra))
  :project (and (nation-loc ?who_water  ?h_water)
                (nation-loc ?who_zebra  ?h_zebra)))
```

- **Pro:** Explicit; the author names exactly which projection
  they want. Composes with multi-step projections
  (`drink-loc → nation-loc → smoke-loc → …`).
- **Con:** Two-step query shape. Some redundancy with `:goal`.

### (2) `(?relation ?V ?out)` directly in the goal

Make the goal AST accept an output-binding marker — `?out` is a
free var that the engine binds to the answer after saturation.
The current code already supports free vars in the goal, but the
solver only reports the House binding; rephrase to add the
Nationality bind:

```lisp
(query
  :mode solve
  :goal (and (drink-loc Water ?h_water)
             (pet-loc   Zebra ?h_zebra)
             (nation-loc ?who_water  ?h_water)
             (nation-loc ?who_zebra  ?h_zebra)))
```

- **Pro:** No new query syntax — uses what the matcher already
  does. The trace renderer reports all four bindings.
- **Con:** Goal grows linearly with hop count. The "what we're
  looking for" intent (`?who_*`) is mixed in with the
  "what we know" anchors (`Water`, `Zebra`).

### (3) Question-shape `:ask` keyword

Highest-level, NL-flavoured:

```lisp
(query :ask (who drinks Water) (who owns Zebra))
```

— resolved by a puzzle-domain vocabulary that knows "drinks" →
`drink-loc`, "owns" → `pet-loc`, "who" → `nation-loc`. Requires
a translation table in the puzzle file or a shared ontology.

- **Pro:** Reads like the natural question.
- **Con:** A whole micro-DSL. Probably belongs in M2 (NL → IR),
  not M1.

## Recommendation

**(2) for M1**, **(1) as a forward-looking refactor**, **(3) is M2's
problem**.

(2) is zero engine work — just rewrite the goal in the puzzle.
The matcher already binds `?who_*`; the trace renderer in P1.6
will print all bindings as solution facts. It just needs
[`bench_solve --solution-facts`](../../../ein.py/demo/bench_solve.py)
to print the Nationality bindings alongside the House bindings —
which it already does for *any* fact, because it iterates the
solution leaf's full reasoning layer.

(1) becomes worthwhile when puzzles ship multi-hop projections
or when authors want the "answer-from-question" boundary made
explicit in the IR. Park until a puzzle needs it.

## Open questions

- **Q-S1.7.5.A — Same-house disambiguation. RESOLVED → first row,
  multi-answer is a malformed-puzzle signal.** `goal_bindings`
  returns all matching rows; `answer.py` renders `rows[0]`. For a
  well-formed unique `solve()` the model is a bijection, so each house
  has exactly one nation → one row. If a puzzle were malformed such
  that two nations co-locate with `Water`, `goal_bindings` yields >1
  row and the renderer reports the first — but `solve()` would have
  flagged the model non-functional (`functional`/`injective` fire
  `(false)`) long before, so this row-multiplicity is unreachable on a
  consistent model. No special handling added.
- **Q-S1.7.5.B — Kernel vs puzzle-author concern. RESOLVED →
  author-facing, no kernel grammar change.** Option (2) needs none —
  the question lives in the per-puzzle `:goal`. Even option (1)'s
  `:project` would be a *query-surface* keyword (parses for free; no
  whitelist), orthogonal to the logical kernel that S1.7.6/.7 minimise.
  Promote `:project` only if a multi-hop puzzle needs the
  anchor/question split made explicit (parked per the recommendation).
