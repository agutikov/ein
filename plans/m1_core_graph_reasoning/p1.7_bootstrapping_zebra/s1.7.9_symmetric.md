# S1.7.9 — hardcoded name `symmetric` (search-layer machinery)

**Phase:** P1.7 (bootstrapping) — per-name kernel-purity analysis.
**Parent:** [S1.7.7](s1.7.7_kernel_purity_analysis.md) (inventory + spike).
**Class:** property tag, engine-special (search layer).
**Verdict (prior):** **load-bearing / irreducible** — empirically
established by [S1.7.6 T1.7.6.4](s1.7.6_kernel_minimization.md): deleting
the search-layer sites left zebra2 untouched but **failed 31 tests** on
symmetric-hypothesis puzzles.
**Status:** analysis (parked with [S1.7.7](s1.7.7_kernel_purity_analysis.md)).
**Not an M1 gate** — zebra2's targets are `*-loc` bijections; symmetry
never fires there (the gate is blind to this name; see §Q5 below).

## Call sites — two layers

The **positive derivation** `(rule symmetric ?R) :match (?R ?a ?b)
:assert (?R ?b ?a)` is **pure user-space** and already in `zebra2.ein`.
What is hardcoded is the **search layer**, read through
`store.is_symmetric` / `store.symmetric_relations` (`kb/store.py:600,609`,
themselves keyed on the `"symmetric"` fact index):

1. **Generation** — `hypgen._candidate_fillers` emits *both* orderings
   `(R a b)` and `(R b a)` as branch candidates (`hypgen.py:267,284`).
2. **On-death mirror** — `back_prop.back_propagate(promote_symmetric=True)`
   writes `(not (R b a))` when `(R a b)` dies **and bubbles it to every
   ancestor kb** (`back_prop.py:162,298–312`). This is the cross-branch
   pruner the docstring ties to combinatorial sanity ("60⁶ branchings").
3. **Result counting** — `solution.open_hypotheses` canonicalises
   `(R a b)`/`(R b a)` to one open entry (`solution.py:40,45`), so a
   sole surviving symmetric pair counts as `k=1`, not 2.
4. **Liveness** — `solver._compute_alive` symmetric canonicalisation
   (`solver.py:325`) and `canon.py` branch-state exclusion.

## 1. Functionality connected to the name

Symmetry handled **once positively** (saturation propagates `(R a b)` →
`(R b a)`) and **three times in search**: don't enumerate the same
unordered pair twice; when one ordering is proved dead, mark the mirror
dead everywhere; count an undecided pair once. The positive rule
provably does **not** subsume the search half — it propagates *positives
within one branch*; the machinery is *negative*, *cross-branch*, and
*result-counting*.

## 2. Consequences if removed

[Measured, T1.7.6.4.](s1.7.6_kernel_minimization.md) zebra2: **identical**
(`enterings_total=11`, `k=1`, wall 6.9 s vs 6.7 s) — it has no symmetric
hypothesis target. Full suite: **31 failures** on `co-located`/`next-to`
puzzles — without canonicalisation a symmetric pair counts as **2**
ambiguity solutions (wrong `k`: `test_solve_run_exhaustive_is_ambiguity`,
`test_gaps_full_solution_frontier`); without the mirror, deaths don't
reach the counterpart (`test_shuffle_invariance`, `test_p16_contract`).
So removal is **unsound for ambiguity verdicts** and **slow**, but only on
puzzles that hypothesise on a symmetric relation — invisible to the gate.

## 3. How it could be implemented in ein-lang rules (or elsewhere)

The mirror's *fact* is rule-expressible: a **negative** rule
`(rule symmetric-neg ?R) :match (symmetric ?R) (not (?R ?a ?b))
:assert (not (?R ?b ?a))` derives `(not (R b a))` from `(not (R a b))`.
But it fires only inside the **current branch's** kb — it cannot
replicate the **ancestor-bubbling** that makes the mirror a cross-branch
pruner (S1.7.7 Q4). The other two sites — both-orderings generation and
open-set canonicalisation — are *search/counting* concerns with **no
fact-level meaning**, so no rule expresses them: they are about which
nodes the search enumerates and how the result `k` is computed, not about
what is true. ⇒ partial only: the mirror fact yes (modulo bubbling),
generation + counting no.

## 4. How it could be implemented without sticking to the name

**Canonical-only generation** (S1.7.7 Q2): if hypgen emitted *only* the
canonical ordering (`(R a b)` with `a < b`) for any relation flagged
symmetric, the mirror **and** the canonicalisation become unnecessary —
symmetry handled once, at generation. The name then survives only as a
generic flag the generator consults (`kb.symmetric_relations()` already a
query, not a literal — the literal is the **fact head** `"symmetric"`,
not Python control flow). To drop even that, the puzzle would supply a
**canonical-form function as data** ([F5](../../followups/f5_rules_as_data.md))
that the search calls to fold equivalent orderings — but that just
relocates "these orderings are equivalent" from Python keyed on
`"symmetric"` to a puzzle-supplied canonicaliser (S1.7.7 Q3). The
**completeness** question (does canonical-only still find every reachable
solution and preserve `k`?) is the gate on this redesign.

## Decision (to record in [S1.7.7 T1.7.7.5](s1.7.7_kernel_purity_analysis.md))

Either **(a)** the canonical-generation redesign — re-validate the 31
tests + a standing symmetric-hypothesis fixture (S1.7.7 Q5) + wall-time;
or **(b)** record `symmetric` search machinery as **kernel-irreducible**
(the T1.7.6.4 default), the search optimisation legitimately living in the
engine. **Add the regression fixture regardless** — the gate cannot see
this name, so a `co-located`/`next-to` solve must become a standing test
so future kernel edits don't silently break it.
