# S1.7.9 — hardcoded name `symmetric` (search-layer machinery)

**Phase:** P1.7 (bootstrapping) — per-name kernel-purity analysis.
**Parent:** [S1.7.7](s1.7.7_kernel_purity_analysis.md) (inventory + spike).
**Class:** property tag, engine-special (search layer).
**Verdict (prior):** "load-bearing / irreducible" — from
[S1.7.6 T1.7.6.4](s1.7.6_kernel_minimization.md) (deleting the sites
failed 31 tests). **Revised twice (2026-06-01):** (i) deletion is unsound
but the machinery is *reducible* — `canonical-generation` collapses the
three sites to one; (ii) **but even that one site is too much** (user
objection, see §"Kernel must impose no symmetric semantics") — the
**target is FULL removal**: the kernel keys on `is_symmetric` *nowhere*;
generation moves to a user hrule and `k`-counting to the **generic**
state-dedup. `symmetric` becomes a pure property tag like `transitive`.
**Status:** analysis refined; full-removal route concrete (§4).
**Execution → [S1.7.24](s1.7.24_dehardcode_symmetric.md)** (the symmetric
analogue of [S1.7.23](s1.7.23_retire_kernel_type_system.md)).
**Not an M1 gate** — zebra2's targets are `*-loc` bijections; the
machinery never fires there (gate blind-spot, confirmed below).

## Call sites — one positive rule + three coupled search mechanisms

The **positive derivation** `(rule symmetric ?R) :match (?R ?a ?b)
:assert (?R ?b ?a)` is **pure user-space** (in `zebra2.ein`,
`mini_zebra`, …). What is hardcoded is the **search layer**, every site
gated on `store.is_symmetric` / `store.symmetric_relations`
(`kb/store.py:600,609` — the `symmetric`-fact index, i.e. the property as
*data*). There are **three** mechanisms (the prior draft's "4th site" was
a miscount — `_compute_alive` (`solver.py:407`) just *delegates* to
`open_hypotheses`, and `canon.py` is unrelated):

1. **Generation — both orderings.** `hypgen._fill_slot` emits `(R a b)`
   *and* `(R b a)` as candidates when `(symmetric R)` (`hypgen.py:267,284`).
2. **On-death mirror.** When `(R a b)` dies, `(not (R b a))` is written
   too — two implementations: tree-path
   `back_prop.back_propagate(promote_symmetric=True)` (`back_prop.py:298–312`),
   monotonic-path `solver._emit_negated_fact_writeback` (`solver.py:322–327`).
3. **Open-set canonicalisation.** `solution.open_hypotheses` collapses
   `(R a b)`/`(R b a)` to one canonical entry (`a<b`) (`solution.py:40,45`),
   consumed by `_compute_alive` — so an undecided pair counts once (`k`).

**They form one triangle.** The mirror (2) exists *because of* the
canonicalisation (3): `_emit_negated_fact_writeback`'s own docstring says
the mirror is needed or "`_compute_alive`'s symmetric canonicalisation
would resurrect the dead entry through the mirror orientation." Generate
both (1) → count each pair once (3) → so mark both orientations dead (2).

## Empirical evidence (the blind-spot + the deletion test)

- **Deletion fails (T1.7.6.4, done).** Deleting all three sites: zebra2
  **identical** (`enterings_total=11`, `k=1`) — no symmetric hypothesis
  target; **31 other tests fail** (wrong `k`: a pair counts as 2;
  un-mirrored deaths). So naive deletion is **unsound**.
- **Gate blind-spot confirmed (this turn).** `mini_zebra`
  (`examples/branching/05_mini_zebra.ein`) — `(symmetric co-located)` +
  `(transitive co-located)`, branches on `co-located` via the blind
  enumerator — **solves**: `Solution`, `k=1`, `enterings_total=6`. zebra2
  (the M1 gate) never reaches this path. So a symmetric-hypothesis solve
  **must** be a standing fixture (Q5); `mini_zebra` is the ready-made one
  (already committed, already solving).

## Answers to the [S1.7.7](s1.7.7_kernel_purity_analysis.md) Q1–Q5

- **Q4 (bubbling general?) — GENERAL.** `_bubble_to_ancestors`
  (`back_prop.py:196`) is shared with the *primary* hypothesis death, not
  symmetric-specific. **Only the mirror *fact* `(R b a)` is
  symmetric-specific.** ⇒ Q2/Q3 are the whole question.
- **Q2 (canonical generation) — YES (by analysis).** If (1) emits *only*
  the canonical ordering `(R a b)`, `a<b`: there is no `(R b a)` candidate,
  so (3) canonicalisation is a no-op (removable) **and** (2) the mirror is
  unnecessary (no counterpart to resurrect — removable). Complete: each
  unordered pair is enumerated once. `k`-correct: each pair is one node.
  The positive saturation rule still derives `(R b a)` as a *fact*, so the
  solved KB is unchanged. The triangle collapses to **one** site.
- **Q1 (irreducible?) — NO.** "Can't just delete" (T1.7.6.4) ≠
  irreducible. Canonical-generation *replaces* the triangle; what remains
  is a single generation rule keyed on the `symmetric` *data*-property —
  the same status as `transitive`/`functional` (read generically, zero
  Python control-flow special-casing).
- **Q3 (where does equivalence live?) — the `symmetric` property + a
  generic min/max canonical form.** No per-puzzle canonicaliser needed
  (the order on names is generic), so this is *not* a relocation of the
  hardcode — it is removal of two sites, the third becoming data-driven
  generation.

## Kernel must impose no symmetric semantics (user, 2026-06-01)

The canonical-generation reduction above still has the kernel *consult*
`kb.is_symmetric` to decide the canonical ordering — i.e. the kernel
**decides that `(symmetric R)` means "unconditional arg-swap."** That is
the kernel imposing a semantics, and it is wrong for the same reason the
[`is-a` type system](s1.7.8_isa.md) was — two concrete failure modes:

- **User-defined / non-standard `symmetric`.** The hardcoded path bakes in
  *one* meaning of the tag. A user whose `(rule symmetric …)` means
  something else (conditional symmetry, a differently-named property, a
  custom body) gets a kernel that **silently disagrees with their own
  rule**: wrong `k` (a pair deduped they didn't intend), or mirror facts
  `(not (R b a))` their rule never licensed. Kernel vs user rule can
  conflict — and the kernel wins. Unsound w.r.t. the *stated* semantics.
- **Rule induction (F7).** A symmetric-aware kernel **injects** mirror
  facts and **collapses** pairs, polluting the fact set an inducer learns
  from with kernel artifacts — and it is circular: you cannot *induce* "R
  is symmetric" if the kernel already presupposes it for tagged R.
  [Induction](../../followups/f7_rule_induction.md) needs a **neutral**
  kernel: facts are only what user rules/data produce.

⇒ **The target is full removal, not reduction.** Keep symmetry entirely in
user rules; the kernel keys on `is_symmetric` nowhere. The search benefit
is recovered generically:

- **Generation → a user hrule** (the `is-a`/`guess` pattern, already
  proven in zebra2): a symmetric puzzle's hrule enumerates only canonical
  pairs, so `_fill_slot`'s both-orderings disappears and the kernel never
  consults `is_symmetric`. *The user decides what to enumerate.*
- **`k`-counting → generic state-dedup.** After commit+saturate the user's
  positive `(rule symmetric)` makes both orderings present, so a
  `(R a b)`-branch and a `(R b a)`-branch saturate to **identical KBs** →
  collapsed by the existing `canon.state_hash` solution-node dedup. The
  pair counts once because the *user's rule* established the equivalence,
  not the kernel. (Redesign must verify this subsumes the pre-saturation
  open-set canonicalisation.)
- **The mirror** is then moot (no both-orderings to prune) or the
  already-**general** ancestor-bubble (Q4) — neither symmetric-aware.

This is strictly stronger than §4's canonical-generation: that left one
`is_symmetric`-keyed site in the kernel; this removes all of them.

## 1. Functionality connected to the name

Symmetry handled **once positively** (saturation propagates `(R a b)` →
`(R b a)`) and, today, **three times in search** (the triangle above):
enumerate each unordered pair, mark both orientations dead together, count
a pair once. The positive rule does not subsume the search half *as
written* (it is negative / cross-branch / counting) — but
canonical-generation makes the search half a *generation policy*, not a
separate mechanism.

## 2. Consequences if removed

Naive deletion: **unsound** (wrong `k`) + slow on symmetric-hypothesis
puzzles (the 31 fails), invisible to the gate. *Canonical-generation* is
the principled alternative: same solutions, correct `k`, three sites → one.

## 3. How it could be implemented in ein-lang rules (or elsewhere)

The positive `(rule symmetric)` is already ein. The *search* half is not
naturally a rule (it is generation policy + result counting, not a fact
derivation) — but it does not *need* to be a rule: canonical-generation
makes it a property of the generator. A negative rule
`(not (R a b)) ⟹ (not (R b a))` could derive the mirror fact within a
branch but **cannot** replicate the general cross-branch bubble (Q4) — so
the rule route is the wrong tool; the generation route is right.

## 4. How it could be implemented without sticking to the name

The `symmetric` **name stays** — as a property tag read generically
(`kb.is_symmetric`), exactly like `transitive`/`functional`; the kernel
keeps no `== "symmetric"` *control flow*, only the property query the
generator consults to pick canonical ordering. Concretely: **canonical-
generation** (Q2) — `_fill_slot` emits `(R min(a,b) max(a,b))` once for
symmetric `R`; delete the mirror (2) and the open-set canonicalisation
(3). That is the minimal, name-respecting reduction — three coupled
Python sites → one data-driven generation choice.

**But that one site is still kernel-imposed semantics** (the kernel
deciding `(symmetric R)` ⇒ arg-swap). The stronger, preferred answer —
**full removal**, generation via a user hrule + `k` via generic
state-dedup — is in §"Kernel must impose no symmetric semantics" above.
Canonical-generation-in-the-kernel is the *fallback* if full removal
proves to lose `k`-correctness empirically.

## Decision (to record in [S1.7.7 T1.7.7.5](s1.7.7_kernel_purity_analysis.md))

**Full removal of kernel symmetric-awareness** — the kernel keys on
`is_symmetric` nowhere (user objection: no kernel-imposed semantics; sound
for user-defined `symmetric` + rule induction). Recommended execution —
**[S1.7.24](s1.7.24_dehardcode_symmetric.md)** (the symmetric analogue of
[S1.7.23](s1.7.23_retire_kernel_type_system.md) for `is-a`):

1. **Promote a standing symmetric-hypothesis fixture** — `mini_zebra`
   (or `branching/04`, `lattice/01-03`) to a gate-level solve test, since
   the M1 gate is blind here (Q5). **Do first** (locks behaviour before
   the redesign).
2. **Move generation to a user hrule** — a `guess`-style hrule that
   enumerates only canonical pairs (zebra2's proven pattern); delete
   `_fill_slot`'s both-orderings. The kernel stops consulting
   `is_symmetric` in generation.
3. **Drop the mirror + the open-set canonicalisation**; rely on the
   generic `canon.state_hash` solution-node dedup (the user's positive
   `(rule symmetric)` makes both orderings present post-saturation, so
   mirror-branches collapse generically). Re-validate: the 31 tests + the
   fixture + zebra2 (no-op) + wall-time + `k`-correctness.

Fallbacks, weaker → weakest: **(b)** if generic dedup loses `k`, keep
*canonical-generation in the kernel* (§4 — one `is_symmetric`-keyed site,
still better than three but still kernel-imposed); **(c)** if even that
fails, record the search machinery as irreducible with the T1.7.6.4
evidence.
