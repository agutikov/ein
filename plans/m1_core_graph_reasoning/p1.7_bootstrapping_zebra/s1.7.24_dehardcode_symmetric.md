## Outcome — FULL REMOVAL (2026-06-02)

The three sites are gone; symmetry lives entirely in the user's
`(rule symmetric)`. `store.is_symmetric` / `symmetric_relations` survive
only as unprivileged property queries (no search code calls them).

- **Site 1 (generation):** deleted `_fill_slot`'s `is_symmetric` consult +
  both-orderings emit. (Near-cosmetic on the blind path — both orderings
  already arise across focal objects; the kernel just stops *deciding* it.)
- **Site 2 (on-death mirror):** deleted `back_prop.is_symmetric_relation`
  + the `promote_symmetric` parameter/block, and the symmetric branch of
  `solver._emit_negated_fact_writeback`.
- **Site 3 (open-set canonicalisation):** deleted from
  `solution.open_hypotheses` — `(R a b)` / `(R b a)` are now two distinct
  open entries.

**The crux (`T1.7.24.4`) — generic `state_hash` recovery, NOT the
fallback.** SOLVE `k` was already correct (its `solution_nodes` dict keys
on `state_hash`; the two orientations saturate via the user rule to the
*same* KB and collapse). The regression was in the **gaps / contradictions
/ lattice-snapshot** entries, which keyed on *commitments* (paths), not
states. Fix, all generic (no `is_symmetric`):
- `_finalise_lattice_verdict` dedups gaps solution records by `state_hash`
  (canonical min-commitment representative → deterministic branches). This
  also fixed a **pre-existing latent over-count** the symmetric retirement
  exposed: mini_zebra gaps returned 3 branches that were the *same model*
  (one distinct `state_hash`) via three commitment paths — now 1.
- `DeadCommitment` gained a `state_hash`; `lattice_snapshot` is now
  **result-level** — `solutions` / `deads` key on post-saturation
  `state_hash` (orientation-invariant), and the inherently
  order/orientation-sensitive **learned `nogoods` are excluded** from the
  snapshot (a clause `{(R a b),…}` ≡ its mirror `{(R b a),…}` only under
  symmetry, unknowable without `is_symmetric` — an optimisation artifact,
  not part of the *result*). This restored shuffle-invariance.
- The **forced-positive cascade** no longer fires for a sole symmetric
  pair (it keys on a singleton `alive`; the pair is now two entries) — the
  model is found by a Phase-2 commitment instead. Verdict preserved;
  documented as a canonicalisation-dependent optimisation in
  `test_gaps_solve_branching_03_returns_one_model`.

The gaps **kb_index** stays per-commitment (its documented "no-merge"
contract); `proof.solutions` is the state-deduped distinct-model view —
the render test asserts both granularities.

## Origin — the user principle (2026-06-01)

> The kernel must impose **no symmetric semantics**. Hardcoding
> symmetric-aware behaviour is wrong in two cases: **(1)** a user who
> defines a different / non-standard `symmetric` rule gets a kernel that
> silently disagrees with their own rule; **(2)** future **rule induction**
> from facts would behave incorrectly — a symmetric-aware kernel injects
> mirror facts and pre-supposes the very property induction must learn.

So the target is **full removal**: the kernel keys on `is_symmetric`
**nowhere**. The positive `(rule symmetric ?R) :match (?R ?a ?b) :assert
(?R ?b ?a)` stays (pure user-space); everything else — generation,
mirroring, pair-counting — comes from user rules + generic engine
mechanisms, not kernel symmetric-awareness.

## Targets — the three sites + the property methods

All gated on `store.is_symmetric` / `store.symmetric_relations`
(`kb/store.py:600,609`):

1. **Generation — both orderings** — `hypgen._fill_slot` `symmetric =
   kb.is_symmetric(rel.name)` then emits `(R a b)` *and* `(R b a)`
   (`hypgen.py:267,284`).
2. **On-death mirror** — tree-path `back_prop.is_symmetric_relation` +
   `promote_symmetric` (`back_prop.py:162-168,220,298-312`); monotonic-path
   `solver._emit_negated_fact_writeback` `root_kb.symmetric_relations()`
   (`solver.py:322-327`).
3. **Open-set canonicalisation** — `solution.open_hypotheses`
   collapses `(R a b)`/`(R b a)` to one canonical entry (`solution.py:40,45`).

`store.is_symmetric` / `symmetric_relations` may survive as **plain,
unprivileged property queries** (a puzzle/renderer may ask "is R tagged
symmetric?"), but **no kernel search code calls them**.

## Why naive deletion is wrong, and full removal is sound

- **Naive deletion fails** (T1.7.6.4): drop all three and `k`
  double-counts (a pair = 2) and deaths don't propagate — **31 tests
  fail**. So removal needs *replacement*, not deletion.
- **Sound replacement.** Without the mirror, a dead `(R a b)`'s
  counterpart `(R b a)` still dies — committing it re-derives `(R a b)`
  (positive rule) and hits the same ⊥, one branch later. The mirror was an
  *optimisation* (early prune), never soundness. And `k` is recoverable
  **generically**: two mirror-branches both saturate (via the positive
  rule) to the **same KB** → collapsed by `canon.state_hash`
  solution-node dedup. The pair counts once because the *user's rule*
  established the equivalence, not the kernel.

## Tasks

### T1.7.24.1 — Standing symmetric-hypothesis fixture (do first)
Promote `mini_zebra` (`examples/branching/05_mini_zebra.ein` —
`(symmetric co-located)`, branches on it, solves `k=1`,
`enterings_total=6`) to a **gate-level solve test** (plus a `gaps`/
ambiguity case from `branching/04` / `lattice/01-03`). The M1 gate
(zebra2) is blind here; this locks the symmetric `k`/pruning behaviour
**before** the redesign so regressions are caught (the 31 T1.7.6.4 fails
are the warning).

### T1.7.24.2 — Generation → user hrule (the primary, kernel-neutral route)
Delete `_fill_slot`'s both-orderings (`hypgen.py:284`) and the
`kb.is_symmetric` consult (`:267`). Symmetric puzzles author a
`guess`-style hrule that enumerates **canonical pairs** (zebra2's proven
pattern) — re-author `mini_zebra` etc. with such an hrule. The kernel then
never consults `is_symmetric` in generation; *the user decides what to
enumerate*. This unifies with [S1.7.23](s1.7.23_retire_kernel_type_system.md)
(hrule-driven generation retiring the blind enumerator's special-casing).

> **Alt route (if keeping the blind enumerator):** the blind loop already
> produces both orderings via different focal objects even without site 1,
> so true canonical-only there needs an `is_symmetric`-keyed min/max step
> — which is *canonical-generation-in-kernel*, the §4 fallback, **not**
> full removal. Prefer the hrule route for neutrality.

### T1.7.24.3 — Drop the mirror (site 2)
Remove `promote_symmetric` + `is_symmetric_relation` from `back_prop.py`
and the symmetric branch of `solver._emit_negated_fact_writeback`. Deaths
propagate to the counterpart **generically** (re-derivation hits the same
⊥) or via the already-general ancestor-bubble (Q4) — neither
symmetric-aware. Record the wall-time cost of losing the early prune.

### T1.7.24.4 — Drop the open-set canonicalisation (site 3) + verify `k`
Remove the `symmetric_relations` canonicalisation from
`solution.open_hypotheses`. **The crux to verify:** does the generic
`canon.state_hash` solution-node dedup recover correct `k`, *and* does the
**forced-positive cascade** still fire? The cascade keys on `alive` being
a **singleton**; if a sole viable pair appears as two open entries
(`(R a b)`,`(R b a)`) the cascade may not trigger. Two outcomes:
- **Pass** — generic dedup + cascade behave: full removal confirmed; the
  31 tests + the T1.7.24.1 fixture green; `symmetric` is a pure tag.
- **`k`/cascade regresses** — fall back to **canonical-generation in the
  kernel** (one `is_symmetric`-keyed generation site; §4 of S1.7.9) —
  weaker (still kernel-imposed) but three sites → one. Last resort:
  record the search machinery irreducible (T1.7.6.4 evidence).

### T1.7.24.5 — Kernel-doc + reserved-names update
`docs/kernel/ir/01-ein-graph/03_ein_model.md` §6: `symmetric` is a plain
property tag like `transitive`/`functional` — the kernel keys on it in no
search path; symmetric *semantics* are the user's `(rule symmetric)` (+
optional canonical-enumeration hrule). Feeds the
[S1.7.7 T1.7.7.5](s1.7.7_kernel_purity_analysis.md) conclusion.
