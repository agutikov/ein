**Implementation deltas (2026-06-01).**

1. **Not "load-time" — post-saturation.** Almost every NAF-bearing
   rule in zebra2 (`adjacent-via-*`, `disjunctive-prune-*`,
   `*-endpoint-*`, `typecheck-arg-*`, `domain`/`range-elimination`,
   `total`, `surjective`) is activated by a *derived* fact that does
   not exist at load, so a fresh `Engine(kb).compile_all()` yields a
   near-empty map. The complete cache exists only after the Phase-1
   root saturation; emission reuses that saturator's engine cache
   (`solver._phase1_root`). `Engine.naf_dependency_map()` is callable
   anytime but documents the incompleteness.
2. **Flag defaults OFF, not on.** The suite runs under
   `filterwarnings = ["error"]` (`pyproject.toml`), so a default-on
   `warnings.warn` would fail every test that solves a derived-NAF
   puzzle. And while `closed` stays hardcoded the NAF is sound
   regardless, so the warning is advisory. Promote to default-on
   under [S1.7.7](s1.7.7_kernel_purity_analysis.md) when closure is
   de-hardcoded. Dedicated `DerivedNafWarning` category so it stays
   filterable.
3. **Scope B (negated NAF) included.** `(absent (not (R …)))` —
   `forall` / `total` / `domain-elimination` — watches the *derived*
   `(not (R …))` fact, so it's classified via a `negated_relation`
   dual of `producible_relations`. This flags the elimination /
   totality rules the literal-head reading would miss (user decision
   2026-06-01).

## Context

S1.5a.1 T1.5a.1.1 closes the NAF race at runtime — every
`AbsentGuard` is re-evaluated at fire time, so a rule whose NAF
references a derived relation is *correct*, full stop. What the
engine doesn't tell the author is **which** of their rules rely
on that re-eval. A rule with `(absent (R …))` where R is a
declared-only relation (only `(facts)` / `(ontology)` entries)
behaves identically under enqueue-time and fire-time semantics;
a rule where R is derived by another rule is only sound because
of the re-eval. The distinction matters for trace explanations
and for tuning priorities.

## Tasks

### Task T1.7.4.1 — Compile-plan NAF→relation walk

- **a.** Walk every compiled `JoinPlan` in `Engine.cache`; for
  each top-level (or nested) `AbsentGuard.sub_steps`, collect
  the relation names every `Scan`/`Join` step inside it
  references. Emit `(rule_name, activator_args, naf_relations)`
  records.
- **b.** Reuse the [`closed.producible_relations`](../../../ein.py/src/ein/inference/closed.py)
  test ("some rule positively asserts R") to classify each NAF
  relation as **derived** vs **declared-only**.
- **c.** Surface the map programmatically:
  `Engine.naf_dependency_map() -> list[NafDep]` or similar. The
  return shape should be diff-friendly for golden tests.

### Task T1.7.4.2 — Load-time warning

- **a.** After `Engine.compile_all()` (or at the first
  `Saturator.saturate()` call — whichever is the natural single
  invocation site), emit a `warnings.warn(...)` for every rule
  with at least one derived-NAF dependency. The message names
  the rule, the activator (if any), and the derived relations.
- **b.** Configurable via a `SolverConfig` flag
  (`warn_derived_naf: bool = True`); defaults on but silenceable
  for tests that intentionally exercise the case.
- **c.** Documented in
  [`docs/kernel/inference/README.md`](../../../docs/kernel/inference/README.md#naf-semantics--fire-time-re-evaluation-s15a1)
  as a paragraph under the existing NAF subsection.

### Task T1.7.4.3 — Tests

- **a.** Unit: rule with declared-only-NAF → no warning. Rule
  with derived-NAF → exactly one warning naming the right
  relation(s).
- **b.** Smoke: load `examples/zebra2.ein`, **saturate the root**
  (so rule-derived activators are compiled), and assert the
  derived-NAF rule set. Refreshed to current names + Scope B —
  the 14 rules now flagged:
  `adjacent-via-{fwd,bwd}` and `adjacent-via-{fwd,bwd}-negative`
  and `disjunctive-prune-{fwd,bwd}` and
  `adjacent-via-endpoint-{fwd,bwd}` (NAF on `next-to`, derived via
  `(symmetric next-to)` + `(includes right-of next-to)`);
  `typecheck-arg-{0,1}` (NAF on `is-a*`, derived via
  `(includes is-a is-a*)` + `(transitive is-a*)`); and the Scope-B
  negated-NAF rules `domain-elimination`, `range-elimination`,
  `total`, `surjective` (`(absent (not (*-loc …)))` over the
  `*-negative` companions' derivations). The `right-of` activators
  of the spatial rules are the declared-only siblings — asserted by
  `test_spatial_split_by_activator`.

## Open questions

- **Q-S1.7.4.A — Warning vs lint vs no-op. RESOLVED →
  `warnings.warn`, default-off.** The middle level, as proposed,
  *but gated off by default* (`warn_derived_naf=False`): the engine
  fix makes derived-NAF sound, so today it's advisory, and
  default-on is incompatible with `filterwarnings=["error"]`. A
  hard error stays rejected (would block correct rule sets). The
  silent map is always available via `Engine.naf_dependency_map()`.
- **Q-S1.7.4.B — Nested NAF handling. RESOLVED → recurse, dedupe
  per `(relation, negated)`.** `naf_relation_refs` recurses into
  nested `AbsentGuard.sub_steps` (so `forall`/`open` desugars are
  fully entered); `compute_naf_map` collects into sets keyed by the
  `(not R)` / `R` label, so a relation seen at multiple depths
  collapses. Negation *parity* across nesting is not tracked — the
  map answers "which relations are watched", not "with what
  polarity at what depth".
