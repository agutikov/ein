## Goal

Execute the clean removals: delete the dead `a-priori` alias; move
`type` / `instance` out of grammar + loader into user-space relations +
rules; collapse the redundant `symmetric` engine code onto its existing
rule; lower `or`-in-`:match` to multiple rule instances. Each task sits
behind a cheap re-validation gate.

**Re-validation gate** (after every task):
`pytest …/test_monotonic_skeleton.py::test_zebra2_solves_via_monotonic_backbone`
(~5 s CPython) stays green; note any wall-time regression.

## Tasks

### Tier 1 — mechanical removals (each behind the solve gate)

**T1.7.6.1 — drop `a-priori`.** Delete the grammar production, its
`SYMBOL`-exclusion entry, the loader arm, the AST builder, and the two
renderer references. Migrate the one parser round-trip test off it.

**T1.7.6.2 — `type`/`instance` out of grammar + loader.** Drop
`type_decl` / `instance_form`; remove `instance` (and `a-priori`) from
the `SYMBOL` exclusion so they fall through to `generic_fact`; delete the
`head=="type"` / `"instance"` loader arms and the `pattern.py` /
`compile.py` instance special-cases; `KERNEL_META_RELATIONS` →
`{relation, rule}`; `INHERITANCE_RELATIONS` drops `"instance"`. Make
`kb.types` / `kb.instances` **derived views** over the `type` /
`instance` facts (keep the entity-API surface) rather than cached
registries.

**T1.7.6.3 — author type/instance semantics in `zebra.ein`.**
`(relation type UserType Type)` (basic two-arg form: a type + its parent,
parent defaulting to the top `T`), `(relation instance Thing Type)`, plus
rules bridging them to `is-a` (instance→is-a; type-parent→is-a;
sibling-exclusivity). Acceptance: `zebra.ein` **parses + loads** — it need
not solve (too many hypotheses, missing rules; expected and recorded, not
fixed).

### Tier 2 — T1.7.6.4 — `symmetric` dedup

zebra2 already defines `(rule symmetric …)` (it fires in the idea-08 set).
Confirm the rule subsumes the engine's eager on-death mirror, then delete
the engine special-case (`solver.py:332,412`; `back_prop.py:187`;
`hypgen.py:320`). Re-validate the solve gate + the idea-08 firing test. If
the rule's timing misses an on-death case the mirror covered, extend the
rule before deleting the code.

> **Finding (2026-06-01) — premise does not hold; recommend PARK.**
> Reading the actual code (not the line numbers, which predate P1.7a),
> `symmetric` is special-cased on **two different layers** and the
> positive rule only covers one:
> - **Derivation layer** — `(rule symmetric ?rel) :match (?rel ?a ?b)
>   :assert (?rel ?b ?a)` — pure *positive* forward propagation during
>   saturation.
> - **Search layer** — (a) `hypgen._candidate_fillers` emits *both*
>   orderings as branch candidates (`hypgen.py` ~287); (b)
>   `back_prop.back_propagate(promote_symmetric=True)` writes the
>   *negative* mirror `(not (R b a))` when `(R a b)` dies **and bubbles
>   it to every ancestor kb** (`back_prop.py` ~300) — the cross-branch
>   pruning the docstring ties to combinatorial sanity ("60^6
>   branchings"); (c) `solver._compute_alive` symmetric canonicalisation;
>   (d) `solution.py` symmetric-pair dedup; (e) `canon.py` symmetric
>   branch-state exclusion.
>
> The positive rule propagates *positives within one branch*; the
> on-death mirror propagates *negatives across branches*. A negative
> rule `(not (?rel ?a ?b)) ⟹ (not (?rel ?b ?a))` could in principle
> derive the mirror fact, but it would fire only inside the current
> branch's kb — it cannot replicate the **ancestor-bubbling** that makes
> the mirror a cross-branch pruner. So the engine's symmetric machinery
> is **search-layer**, not a redundant duplicate of the derivation rule.
>
> **Empirically tested + reverted (2026-06-01, user-requested).** Deleted
> the three monotonic-path special-cases (hypgen both-orderings; the
> `solver._emit_negated_fact_writeback` mirror; the `solution.open_hypotheses`
> canonicalisation) and ran the bench + full suite:
> - **zebra2 SOLVE: identical** — `enterings_total=11`, `dead_post=2`,
>   `nogoods=2`, same `Solution k=1`, wall 6.9 s vs 6.7 s. zebra2's
>   hypothesis targets are the `*-loc` **bijections**, not symmetric
>   relations, so the machinery never fires there. The M1 gate would
>   *not* have caught the deletion.
> - **Full suite: 31 failures** on the puzzles that *do* hypothesise on
>   symmetric relations (`co-located` / `next-to`): `test_shuffle_invariance`
>   (contradiction branch-order), `test_p16_contract` gaps-branching,
>   `test_gaps_full_solution_frontier`, `test_solve_run_exhaustive_is_ambiguity`,
>   `test_linearize_real_gaps_solution`. Without canonicalisation a
>   symmetric pair counts as **2** ambiguity solutions instead of 1
>   (wrong `k`); without the mirror, deaths don't propagate to the
>   counterpart.
>
> **Conclusion: the symmetric search machinery is load-bearing and
> irreducible — reverted.** The user-space half is already pure
> (`symmetric` property tag + `(rule symmetric …)`); the *search
> optimisation* (pair canonicalisation, death-mirror, candidate
> both-orderings) legitimately lives in the engine — it is branch
> pruning + ambiguity-counting, not ontology, and the positive rule
> cannot express it. **T1.7.6.4 is reclassified with the S1.7.7
> load-bearing removals.**

### T1.7.6.5 — `or` lowering

`or` stays a kernel primitive, but an `(or …)` in a rule `:match` lowers
into **multiple compiled rule instances** (one per disjunct), exploiting
ein's already-disjunctive multiple-rules semantics — not a runtime branch
(user design note, 2026-05-30). Today `compile.py:308,311` has an unused
`or` arm; implement the desugar (or document it as the intended lowering)
and add the missing test (no M1 rule exercises `or` yet).

## Implementation notes (2026-06-01)

- **T1.7.6.1 (a-priori)** — done. Removed from grammar / AST / loader /
  both renderers; the round-trip test became `test_apriori_is_now_a_plain_symbol`.
- **T1.7.6.2 (type/instance out)** — done. `type_decl` / `instance_form`
  gone from the grammar; `instance` left the `SYMBOL` exclusion; the
  `type` / `instance` loader arms, the `pattern.py` `has_instance_pattern`
  special-case (field removed) and the `compile.py` `instance` arm are
  deleted; `KERNEL_META_RELATIONS = {relation, rule}`;
  `INHERITANCE_RELATIONS = ("is-a",)`. **`kb.types` / `kb.instances` are
  now derived in `KnowledgeBase.rebuild_indexes`** from the `(type …)` /
  `(instance …)` facts (two-pass: types first so explicit parents beat
  instance auto-vivification); `add_type` / `add_instance` deleted;
  `snapshot()` rebuilds them (no longer shared by reference). Side
  effects handled: `render._suppress` now also suppresses `(type …)`
  facts (drawn as entity boxes, not octagons); the obsolete
  `broken/instance_in_ontology.ein` fixture (tested parse-time instance
  arity rejection — now gone) was retired.
- **T1.7.6.3 (zebra.ein)** — done, **with one deviation**: zebra.ein
  declares `(relation type T T)` / `(relation instance T T)` and keeps
  its `(type …)` / `(instance …)` facts, consumed directly by the
  existing `type-exclusivity` / `square-unique` rules. It does **NOT**
  introduce `is-a` or instance→is-a / type→is-a bridge rules.
  **User decision (2026-06-01): "zebra.ein didn't have is-a before, no
  reason to have it now."** Rationale confirmed in code: the engine's
  `_ancestor_names` already walks the `kb.types` parent chain directly,
  so a `type-is-a` bridge is pure redundancy; `instance-is-a` would only
  matter if zebra.ein were solved (it need not be — it is a demonstrator).
  is-a stays zebra2's relation; the two encodings stay distinct.
  Acceptance met: `zebra.ein` parses + loads (facts=68: +7 type-decls
  now facts, +2 new relation decls). The `zebra.golden` /
  `kb_zebra_unified.dot` goldens + the zebra-count tests were refreshed.
- **T1.7.6.4 (symmetric dedup)** — investigated, empirically tested,
  **reverted, parked** (see the Finding box above). The symmetric search
  machinery is load-bearing (deleting it fails 31 tests on
  symmetric-hypothesis puzzles); the positive `(rule symmetric)` does not
  subsume it. Reclassified with S1.7.7.
- **T1.7.6.5 (or lowering)** — done. A *top-level* `(or d1 … dn)` in a
  rule `:match` is desugared at **load time** (`from_ir._match_disjuncts`)
  into one `Rule` per disjunct, named `<rule>__or0…n-1`, exploiting the
  already-disjunctive multiple-rules semantics — no runtime `or` branch
  (the `compile.py` `or` arm is now only reachable by a nested `(or …)`,
  which stays unsupported / DNF-deferred). Tests in
  `tests/inference/test_rules.py` (`test_or_in_match_lowers_to_one_rule_per_disjunct`,
  `test_or_disjuncts_fire_independently`). Kernel doc §6 reserved-names
  table updated to drop `type` / `instance` / `a-priori`.
