# S1.7.6 — Kernel minimization (full purity pass — execution)

**Phase:** P1.7 (bootstrapping).
**Estimate:** Tier 1 (1–2 days) + Tier 2 (1 day) + `or` lowering (~½ day).
**Depends on:** [S1.7.2](s1.7.2_dynamic_vs_hardcoded.md) (audit + decision).
**Status:** open — the *tractable* part of the 2026-05-30 full-purity
decision. The load-bearing removals (`is-a`, `closed`/`open`) are parked
as the end-of-phase analysis stage
[S1.7.7](s1.7.7_kernel_purity_analysis.md).
**Not an M1 acceptance gate** — zebra2 already solves with the primitives.

## Goal

Execute the clean removals: delete the dead `a-priori` alias; move
`type` / `instance` out of grammar + loader into user-space relations +
rules; collapse the redundant `symmetric` engine code onto its existing
rule; lower `or`-in-`:match` to multiple rule instances. Each task sits
behind a cheap re-validation gate.

**Re-validation gate** (after every task):
`pytest …/test_monotonic_skeleton.py::test_zebra2_solves_via_monotonic_backbone`
(~5 s CPython) stays green; note any wall-time regression.

## Tasks

### Tier 1 — mechanical removals (each behind the solve gate)

**T1.7.6.1 — drop `a-priori`.** Delete the grammar production, its
`SYMBOL`-exclusion entry, the loader arm, the AST builder, and the two
renderer references. Migrate the one parser round-trip test off it.

**T1.7.6.2 — `type`/`instance` out of grammar + loader.** Drop
`type_decl` / `instance_form`; remove `instance` (and `a-priori`) from
the `SYMBOL` exclusion so they fall through to `generic_fact`; delete the
`head=="type"` / `"instance"` loader arms and the `pattern.py` /
`compile.py` instance special-cases; `KERNEL_META_RELATIONS` →
`{relation, rule}`; `INHERITANCE_RELATIONS` drops `"instance"`. Make
`kb.types` / `kb.instances` **derived views** over the `type` /
`instance` facts (keep the entity-API surface) rather than cached
registries.

**T1.7.6.3 — author type/instance semantics in `zebra.ein`.**
`(relation type UserType Type)` (basic two-arg form: a type + its parent,
parent defaulting to the top `T`), `(relation instance Thing Type)`, plus
rules bridging them to `is-a` (instance→is-a; type-parent→is-a;
sibling-exclusivity). Acceptance: `zebra.ein` **parses + loads** — it need
not solve (too many hypotheses, missing rules; expected and recorded, not
fixed).

### Tier 2 — T1.7.6.4 — `symmetric` dedup

zebra2 already defines `(rule symmetric …)` (it fires in the idea-08 set).
Confirm the rule subsumes the engine's eager on-death mirror, then delete
the engine special-case (`solver.py:332,412`; `back_prop.py:187`;
`hypgen.py:320`). Re-validate the solve gate + the idea-08 firing test. If
the rule's timing misses an on-death case the mirror covered, extend the
rule before deleting the code.

### T1.7.6.5 — `or` lowering

`or` stays a kernel primitive, but an `(or …)` in a rule `:match` lowers
into **multiple compiled rule instances** (one per disjunct), exploiting
ein's already-disjunctive multiple-rules semantics — not a runtime branch
(user design note, 2026-05-30). Today `compile.py:308,311` has an unused
`or` arm; implement the desugar (or document it as the intended lowering)
and add the missing test (no M1 rule exercises `or` yet).

## Acceptance

- `a-priori` gone from grammar / loader / renderers;
  `KERNEL_META_RELATIONS` = `{relation, rule}`; `type` / `instance` are
  generic facts on user-declared relations.
- `zebra2.ein` solve gate green after every task (wall-time regressions
  recorded).
- `zebra.ein` parses + loads with type/instance as user-space relations +
  rules (need not solve).
- `or`-lowering implemented or documented, with a test.
- `docs/kernel/ir/01-ein-graph/03_ein_model.md` §6 (reserved relation
  names) updated to drop `type` / `instance` / `a-priori`.

The remaining purity removals (`is-a`, `closed`/`open`) →
