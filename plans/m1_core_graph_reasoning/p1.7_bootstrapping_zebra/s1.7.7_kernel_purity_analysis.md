## Context

The [S1.7.2](s1.7.2_dynamic_vs_hardcoded.md) audit classified `is-a` and
`closed`/`open` as **load-bearing engine optimizations**, not mere
derivations; [S1.7.6 T1.7.6.4](s1.7.6_kernel_minimization.md) added
**`symmetric`** after an empirical deletion test:

- **`is-a`** — transitivity is already a rule, but the hardcoded part is
  hypothesis-generation **type-pruning** (`INHERITANCE_RELATIONS`,
  `hypgen.py:43,390,506`) + the inheritance views (`kb/views.py:149,169`).
  Pruning restricts candidate bindings to type-compatible instances;
  remove it and the engine generates type-incompatible hypotheses and
  leans on contradiction rules to kill them — a potential combinatorial
  blow-up.
- **`symmetric`** — the positive `(rule symmetric)` is pure user-space,
  but the *search-layer* handling is engine-hardcoded across three sites:
  hypgen emits **both orderings** as candidates (`hypgen.py` ~287); the
  on-death **mirror** writes `(not (R b a))` when `(R a b)` dies and
  **bubbles it to every ancestor kb** (`back_prop.back_propagate(promote_symmetric=True)`
  tree-path; `solver._emit_negated_fact_writeback` monotonic-path); and
  `solution.open_hypotheses` **canonicalises** `(R a b)`/`(R b a)` to one
  open entry so ambiguity `k` counts a symmetric pair **once**. Empirically
  (T1.7.6.4): deleting all three leaves zebra2 untouched (its targets are
  `*-loc` bijections — symmetry never fires) but **fails 31 tests** on
  symmetric-hypothesis puzzles. The positive rule propagates positives
  within a branch; it cannot subsume the negative + cross-branch +
  result-counting logic.
- **`closed`/`open`** — `(closed R)` underwrites NAF / domain-elimination
  **soundness** (`inference/closed.py:67,79` auto-closure;
  `hypgen.py:334`); `open` (`compile.py:244`) is the match-body
  counterpart whose exact role must be pinned first. Intersects
  [S1.7.4](s1.7.4_naf_dependency_map.md): once closure is rule-sourced,
  the static derived-NAF map becomes the soundness audit.

Removing these as rules is real work *and* risk. This stage is the
investigation that decides whether they *can* be — ending in a written
decision, possibly an **irreducible-kernel** finding (a legitimate result
worth recording, not a failure).

type-compatibility filter a no-op) and run the zebra2 solve gate. Record:

(`MonotonicStats`)? Commit a `demo/` measurement script
(persistent-script convention). This single number gates the rest:
- still solves in budget → de-hardcoding `is-a` is viable;
- blows up → `is-a` pruning is (at least) an irreducible optimization.

> (`./bench_isa_spike_pypy.sh`). **Result: de-hardcoding viable.** On
> zebra2 the neutralisation is a **no-op** — identical `Solution k=1`,
> `enterings_total=11`, `raw=125`/`emitted=56`, **0 type-incompat skips

> enumerator that holds `_type_compatible` is **dead code** for the gate —

### T1.7.7.3 — `symmetric` design + decision

Reclassified here by [S1.7.6 T1.7.6.4](s1.7.6_kernel_minimization.md)'s
empirical test (deleting the three search-layer sites failed 31
symmetric-hypothesis tests; zebra2 was unaffected because its targets are
bijections). The question is whether the search-layer handling can be
made rule-sourced / generation-canonical, or is irreducible. Open
questions:

- **Q1 (irreducible?)** Is the symmetric search machinery separable from
  the engine at all, or — like `is-a` pruning — an irreducible search
  optimization? The positive `(rule symmetric)` provably does not subsume
  it (it propagates positives within a branch; the machinery is negative,
  cross-branch, and result-counting).
- **Q2 (canonical generation)** Could hypgen generate **only canonical
  orderings** (`(R a b)` with `a < b` for symmetric `R`) instead of
  both-then-dedup? That would make the mirror **and** the canonicalisation
  unnecessary — symmetry handled once, at generation. Does canonical-only
  generation stay **complete** (every reachable solution still found) and
  preserve the ambiguity `k`? Wall-time delta?
- **Q3 (where does "these orderings are equivalent" live?)** Pair
  equivalence is Python-hardcoded keyed on `(symmetric R)`. Can it be a
  **declarative annotation** the search consults (rules-as-data,
  [F5](../../followups/f5_rules_as_data.md)) — e.g. a canonical-form
  function the puzzle supplies — or is that just relocating the hardcode?
- **Q4 (ancestor-bubbling)** Is the cross-branch death-bubbling
  fundamentally symmetry-aware, or the *general* nogood-bubbling mechanism
  that merely also carries the mirror fact? If general, only the mirror
  fact is symmetric-specific and Q2/Q3 are the whole question.
- **Q5 (gate blind spot)** zebra2 (the M1 gate) does **not** exercise
  symmetric hypotheses (bijection targets). Should a symmetric-hypothesis
  solve (a `co-located` / `next-to` puzzle) become a **standing regression
  fixture** so future kernel edits don't silently break the symmetric path
  the gate can't see? (T1.7.6.4's deletion was green on zebra2.)

Decision: either a **canonical-generation redesign** (re-validate the 31
tests + the symmetric-hypothesis fixtures + wall-time), or record
`symmetric` as kernel-irreducible search-layer logic with the T1.7.6.4
numbers.
