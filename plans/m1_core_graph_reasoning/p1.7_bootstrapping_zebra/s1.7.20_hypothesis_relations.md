# S1.7.20 — hardcoded name `hypothesis-relations` (query-config keyword)

**Phase:** P1.7 (bootstrapping) — per-name kernel-purity analysis.
**Parent:** [S1.7.7](s1.7.7_kernel_purity_analysis.md) (inventory).
**Class:** query-config keyword — **already data** (a `:query` kw-pair).
**Verdict:** **config key, not control flow** — the name is read from the
puzzle's `(query …)` block, already the rules-as-data form.
**Status:** analysis.

## Call sites

- `hypgen.py:344–349` — `_hypothesis_relations(kb)` scans the
  `(query …)` block's `kw_pairs` for the key whose name is
  `"hypothesis-relations"`; its value (a bare SYMBOL or a `(r1 r2 …)`
  list) restricts hypgen to **only** those relations.
- `hypgen.py:351–360` — `_coerce_relation_names` parses the value.

## 1. Functionality connected to the name

`:hypothesis-relations` lets a puzzle **scope hypothesis generation** to a
named subset of relations — "only guess `co-located` facts, never
`right-of`". It is the author-facing knob behind the `closed` auto-closure
([S1.7.10](s1.7.10_closed.md)): where `closed` *derives* "don't guess R",
`:hypothesis-relations` lets the author *state* "guess only these". The
name is matched as a **keyword in the query block**, exactly the
data-driven configuration surface.

## 2. Consequences if removed

A puzzle loses the ability to **manually scope** hypgen; generation falls
back to its default (all non-closed relations with hypothesis-eligible
signatures). For zebra2 this is harmless (auto-closure already scopes
correctly); for a puzzle where auto-closure over-approximates, removal
forces a slower search (more candidate relations) but not an incorrect
one. It is an **optimisation knob**, not a soundness or completeness
lever.

## 3. How it could be implemented in ein-lang rules (or elsewhere)

It is **already** "in ein" — it lives in the `(query …)` block as a
kw-pair, i.e. puzzle data, not Python. The only hardcoded part is the
*keyword string* `"hypothesis-relations"` that `hypgen` looks for. There
is nothing to "express as a rule": it is declarative configuration, and
its rule-analogue (`(closed R)` for the complement) already exists. The
question is purely whether the **keyword vocabulary** of `(query …)` is
itself a kernel-reserved set.

## 4. How it could be implemented without sticking to the name

Two routes. **(a)** Fold it into the existing property-fact vocabulary:
instead of a query keyword, mark eligibility with a fact —
`(hypothesisable R)` (the positive dual of `(closed R)`) — read through
the same generic property accessor; then hypgen has *no* query-keyword
special-case, only a property query. **(b)** Keep it a query keyword but
treat the **query block's keyword set as open** (the engine already
tolerates unknown `:match` kw-pairs, Q32) — so `:hypothesis-relations` is
just one recognised config key among an extensible set, its handler
registered rather than inlined. Route (a) unifies it with `closed` and
removes the name from hypgen entirely.

## Decision (to record in [S1.7.7 T1.7.7.5](s1.7.7_kernel_purity_analysis.md))

Record as **already-data configuration**; the single hardcode is the
keyword string. Prefer route (a) — express hypgen scoping as a
`(hypothesisable R)` / `(closed R)` property pair read generically — so
[S1.7.10 `closed`](s1.7.10_closed.md) and this name collapse into one
property-driven hypgen-eligibility mechanism with no reserved keyword.
