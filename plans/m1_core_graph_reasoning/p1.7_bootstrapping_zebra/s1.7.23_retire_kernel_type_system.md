## T1.7.23.2 + .3 — DONE (2026-06-01)

**Kernel is now is-a-free for reasoning.** A grep of `inference/` for live
`is-a` reads returns nothing (only comments + the state-dump label); the
only remaining `is-a` reads are `kb/views.py logical_*` (deleted in .4) and
`kb/render.py` (presentation).

- **Removed** from `hypgen.py`: `_type_compatible`, `_ancestor_names`,
  `INHERITANCE_RELATIONS`, the `"T"` short-circuit, both call sites.
- **`_instance_like_objects` → `_candidate_objects`** — name-free:
  `category=='object'` **minus** relation-signature-type nodes **minus**
  `_RESERVED_PRIMITIVES = {not,false,and,or,absent,forall,neq,eq}`. The
  reserved guard is load-bearing — lookahead back-props `(not h)` *during*
  generation, injecting `not` into `kb.names`; without the guard the
  enumerator proposes `(R x not)` garbage → false ⊥.
- **Corpus migration** (the residual the name-free set exposed — type
  nodes leak when they aren't named in a signature):
  - inline CDCL fixtures (`test_monotonic_cdcl.py`): slots `T T`→`Thing Thing`
    (now a signature type → excluded; reproduces the old is-a-leaf set, so
    assertions unchanged).
  - `branching/01,03,04` (co-located relates **2** attribute types): typed
    `co-located` signature (`Color Drink` / `Color House`) → type atoms
    signature-excluded; tests pass unchanged.
  - `branching/05_mini_zebra` (co-located spans **3** types — no 2-slot
    signature works): converted to a **leaf-pattern `guess` hrule**
    `:match (and (is-a ?a ?A) (is-a ?b ?B))` activated per attribute-type
    pair — object selection in *user space*, per the 2026-06-01 user
    direction ("detect leaves in ein; ein-lang has powerful pattern
    matching"). `lattice/01–03` were already hrule-driven (unaffected).
- **Answer to "can the kernel live without `is-a`?":** yes — no reasoning
  path needs it; the kept `is-a` reads are the to-be-deleted `logical_*`
  bridge and the renderer (presentation).

## Origin — the user principle (2026-06-01)

> The kernel **must not impose its own type system** for objects and
> relations. If there are rules defining substitution and inheritance,
> *then there is a type system — defined by the user* in ein-lang (or a
> P1.8 stdlib). So from the kernel's PoV **`is-a` (and `T`) must be just
> another relation/atom, nothing more.**

[S1.7.8](s1.7.8_isa.md) split the kernel's `is-a` machinery into **(a)**
the *entity-view* (deriving a types/instances taxonomy from `is-a` facts)
and **(b)** the *hypgen type-filter*, plus the companion top-type atom
**`T`**. This stage removes all three from the kernel.

## Why it is viable (the S1.7.8 / T1.7.7.1 spike)

`ein.py/demo/spike_isa_pruning.py` neutralised `hypgen._type_compatible`
and re-solved: **zebra2 is bit-identical** (`Solution k=1`,
`enterings_total=11`). zebra2 is **hrule-driven** (`(hrule guess (?R ?T1
?T2) :match (and (is-a ?a ?T1) (is-a ?b ?T2)) :assert (?R ?a ?b))`), so
the blind enumerator holding (b)+`T` — and `_instance_like_objects` — is
**dead code** on the gate; type-pruning already lives in the `guess`
hrule, in user space. The kernel type system is exercised only by the
**blind path** (no-hrule puzzles) and by **rendering** — never the
canonical solve.

## Targets — the exact surface

**(b) + `T` — hypgen type-filter** (all in `inference/hypgen.py`, all in
the `else`/no-hrule branch of `_generate:157`):
- `INHERITANCE_RELATIONS = ("is-a",)` (`:45`)
- `_type_compatible` (`:365`), incl. the `T` universal-top short-circuit
  `sig_type == "T"` (`:373`) — the *only* `"T"` hardcode in `src/`
- `_ancestor_names` (`:378`) — walks `INHERITANCE_RELATIONS` facts **and**
  the `kb.types` parent chain (`:393`)
- call sites `_raw_candidates:243`, `_fill_slot:273`

**(a) — entity-view** (`kb/`):
- registries `kb.types` / `kb.instances` (`store.py:116-117`), derived in
  `rebuild_indexes` from `(type …)`/`(instance …)` facts (`store.py:309-346`)
- `logical_types` / `logical_instances` / `type_name` / `instance_name`
  (`kb/views.py`), exported `kb/__init__.py:29-55`
- consumers: `_instance_like_objects` (`hypgen.py:508`, unions `kb.instances`;
  blind path only), `_ancestor_names` (`hypgen.py:393`), the renderer
  (`kb/render.py:214-218`), and the zebra-count / store tests

> **Precedent:** [S1.7.6](s1.7.6_kernel_minimization.md) already removed
> `type`/`instance` from grammar+loader and made `kb.types`/`kb.instances`
> *derived views*. This stage finishes the job by removing their
> **privileged status** — nothing in the kernel should *interpret* them as
> a type lattice.

## Tasks

### T1.7.23.1 — Standing blind-path regression fixture (do first)
The M1 gate (zebra2) **cannot see** (b)/`T` (hrule path; spike = no-op),
exactly like the symmetric blind-spot ([S1.7.9](s1.7.9_symmetric.md) Q5).
Before touching anything, add a **no-hrule, non-`T`-typed puzzle that
actually solves** (the existing `features/04`,`05` abort at `k=0` —
non-converging demos, unusable as a gate). Commit it under `examples/`
with a solve test so retiring (b) is verifiable, not silent.

### T1.7.23.2 — Retire (b) + `T`
Delete `_type_compatible`, `_ancestor_names`, `INHERITANCE_RELATIONS`, and
the `"T"` short-circuit. The blind enumerator then proposes without
type-pruning and **leans on `lookahead` + contradiction rules** to kill
type-wrong candidates (sound — the filter only ever removed provably-dead
ones; tractability is the cost the fixture measures). Re-validate: zebra2
identical (spike); the T1.7.23.1 fixture still solves (wall-time
recorded). **`T` falls out for free** — `_type_compatible` was its sole
consumer; afterwards `T` is a plain atom puzzles use to name their top.

### T1.7.23.3 — The candidate object set (the S1.7.8 §4 open question)
**Decision (user, 2026-06-01): option (i) — keep the blind enumerator,
name-free object set.** `_instance_like_objects` selects guessable nodes
by a **name-free** signal (`NameRef.category == "object"`) instead of
keying on `is-a` / unioning `kb.instances`; the kernel names no
inheritance relation. No-hrule puzzles stay solvable (the T1.7.23.1
fixture remains the blind-path witness). Consequence: nodes that play a
"type" role (`House`, `Color`) are also `object`-category and get
proposed too, then killed by `typecheck-arg-*` — sound, slightly more
candidates (the cost the fixture measures). Option (ii) — deprecate the
blind enumerator, require a `guess` hrule — was **rejected**.

### T1.7.23.4 — Delete (a) — `kb.types` / `kb.instances` outright
**Decision (user, 2026-06-01): DELETE the registries entirely** — "that
was the core task of removing the kernel type system" — not merely demote
them. The deletion is a self-contained unit because the supporting
machinery exists *only* to serve the registries:

- **Registries** `kb.types` / `kb.instances` + their `rebuild_indexes`
  derivation from `(type …)` / `(instance …)` facts (`store.py:315-346`).
- **Indexes** `_types_by_parent`, `_instances_by_type`, `_facts_by_instance`
  — each has exactly one consumer: the `Type` / `Instance` entity property
  accessors (`entities.py:104,118,150`). They go with the registries.
- **Entities** `Type` / `Instance` (`entities.py`) — the type-system
  entity classes; no registry home once (a) is gone → delete (drop from
  `Fact.arg_entities`, `FactView.about`, the `kb/__init__` exports).
- **Views** `logical_types` / `logical_instances` / `type_name` /
  `instance_name` (`views.py`) + their `kb/__init__` exports.
- **Consumers** the renderer reads `is-a` facts directly (presentation
  may know the `is-a` convention — it is not kernel *reasoning*);
  hypgen's `_ancestor_names` / `_instance_like_objects` lose their
  `kb.types` / `kb.instances` reads (T1.7.23.2/.3); `store.__len__` /
  `__repr__` drop the type/instance counts; the store / render / entity
  tests assert over `is-a` facts, not the registries.

Sequencing: do **T1.7.23.2 + .3 (hypgen)** first so the kernel *reasoning*
stops touching the registries, then delete (a) with only the renderer +
views + tests left to migrate. Keep the suite green at each step.

### T1.7.23.5 — Kernel-doc + reserved-names update
`docs/kernel/ir/01-ein-graph/03_ein_model.md` §6: record that `is-a` /
`T` are **not** reserved — ordinary relation / atom; the type system is a
user concern (inline rules today; an importable stdlib once P1.8 lands).
Feeds the [S1.7.7 T1.7.7.5](s1.7.7_kernel_purity_analysis.md) conclusion.

### Follow-on (P1.8, not a task here) — extract the stdlib type module
zebra2 keeps its inheritance rules + `guess` hrule **inline** (works
today). When P1.8's imports land, [A4](../p1.8_ein_lang_modules/s1.8.a4_rule_library_location.md)/[A5](../p1.8_ein_lang_modules/s1.8.a5_migration_imports.md)

only part that stays P1.8.

## Acceptance

- `grep '"T"' src/` and `grep 'INHERITANCE_RELATIONS\|_type_compatible'
  src/` return nothing — the kernel keys on no inheritance name.
- zebra2 solves identically (spike numbers: `k=1`, `enterings_total=11`),
  with the kernel filter + entity-view gone and the inline `guess` hrule
  doing the type-pruning.
- The T1.7.23.1 blind-path fixture solves with (b) retired (wall-time
  recorded; any regression is the measured cost of dropping the filter).
- `kb.types`/`kb.instances` carry **no privileged interpretation**:
  removing them (or any consumer) does not change a solve, only the
  rendering/accessor surface, which reads `logical_*` over declared facts.
- `03_ein_model.md` §6 records `is-a`/`T` as non-reserved.
