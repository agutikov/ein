> eliminated**. D3 only removes the per-fork cold pass, which the

> **D2 AS BUILT + a completeness bug caught by the user (2026-06-14).**
> Implemented in `saturator.py`: a per-firing delta of derived relations
> threads from `step()` into `_enqueue_pass(delta_relations)`, which
> re-matches only `_relation_index[rel]` plans (+ never-yet-matched plans
> for reflective rules) instead of all plans against the full KB.
>
> **First cut was UNSOUND.** `_scan_relations` collected only *top-level
> positive* Scan/Join relations, on the false premise "AbsentGuard deltas
> only block." A **`forall`** desugars to a NESTED absent
> `(absent (and G (absent B)))` — adding a `B` fact flips the inner absent
> fail→pass, the outer absent fail→**pass**, *enabling* a firing (e.g.
> `domain-elimination` fires once the last `(not (R a b_other))` exists,
> relation `not`, which lives *inside* the forall). Missing those triggers
> stalled the eliminations → hypotheses never pruned → the zebra2 lattice
> exploded (the user saw "too many lattice nodes," pytest hung).
>
> **The fast suite did NOT catch it** — `tests/` skips the full zebra2
> solve on CPython (the 4 `EIN_RUN_SLOW` tests), so "1192 passed" was
> green on a path that never ran zebra2. **Lesson: any saturator change
> must be validated with `EIN_RUN_SLOW=1` (or PyPy), not the default
> skip-set.** [[feedback_verify_engine_claims]]
>
> **Fix:** `_scan_relations` now recurses into `AbsentGuard.sub_steps` and
> collects every relation at every nesting depth — conservative-complete:
> a plan's match can change only if a fact changes at one of its premise
> relations (positive or absent, any depth), so any such delta re-matches
> it. Validated: 1196 passed with slow tests; `enterings_total`/verdict
> identical to HEAD; 2.3× wall.

## Measured motivation (2026-06-14)

`demo/measure_redundant_firings.py` (committed) wraps `Saturator.saturate`
to tally redundant vs productive firings. zebra2 `solve(stop_after=1)`:

```
12 saturation runs, 10,694 firings — 91% REDUNDANT (9,766 / 928 productive)
  root run:   108 redundant, 269 productive   (builds the base extent)
  11 forks:  9,658 redundant,  659 productive  → 94% of fork work is waste
```

The two biggest forks fire **2,850×** each (**2,669 redundant**). So **~91%
of all firing-work is re-derivation of facts that already exist.**

## Root cause (code-precise)

`Saturator._enqueue_pass` (`saturator.py:207`) walks **every** plan in
`engine.cache` and runs `match.run(plan, self.kb)` against the **full KB**:

```python
for plan in self.engine.cache.values():
    for bindings, premises in match.run(plan, self.kb):   # full-KB re-match
        if key in self._seen or key in self.engine._fired: ...  # dedup AFTER matching
```

`step()` sets `_needs_enqueue=True` after **every productive firing**
(`saturator.py:120`), so `_enqueue_pass` runs **once per productive
firing** — the profile's **940 passes = 940 full-KB re-matches** (144 s
cumtime; `match.run` is 139 s of it; `_apply`/firing itself is only
0.38 s). The `_seen`/`_fired` sets dedup the *enqueue*, but the **matching
is redone every pass**: deriving one new fact triggers a full re-scan of
all plans against all facts to discover the handful of new bindings it
enabled. Two redundancy axes, one cause:

1. **Within a run** — ~77 enqueue passes (one per productive firing) each
   re-match the whole KB, though only the last-derived fact can create new
   matches.
2. **Across forks** — each `try_commitment_set` (`commitment.py:134`)
   builds a **fresh** `Saturator(fork)` with empty `_seen` + fresh
   `engine._fired`, so its first pass re-matches + re-fires the *entire*
   already-saturated root before touching the hypothesis delta.

## Design — make enqueue delta-driven

The fix is one idea applied at both axes: **enqueue only the plans a newly-
present fact can newly satisfy**, never a full re-scan.

### D1 — relation→plans participation index (B5)
Extend the store index (S1.8.B-idx) with `relation_name → [plan]`: which
compiled plans have that relation in a Scan/Join premise. Built from
`engine.cache` (each plan's `steps`), refreshed by `compile_all`. (The
activator side already exists — `kb._rule_apps_by_rule` /
`_rule_apps_on_relation`; this is its premise-side twin.)

### D2 — within-run incremental enqueue
Track the facts derived since the last pass (a `delta` list, or a version
stamp on each fact). `_enqueue_pass` then iterates only
`plans_touching(delta_relations)` and, within each, seeds the match from
the delta fact (the `_facts_by_rel_slot_val` bucket) rather than scanning
the relation extent. One productive firing → re-match a handful of plans
against ~1 fact, not all plans against all facts. Collapses the 940
full re-matches to ~12 initial + cheap delta passes.

### D3 — cross-fork version carry (B2.v)
The root is saturated once. A fork inherits the root's facts **and** its
saturator state — `_seen`, `engine._fired`, and a "saturated through
version V0" mark. The fork's first pass is seeded only by the hypothesis
facts (version > V0); it never re-enqueues a root binding. Kills the 94%
fork redundancy directly.

### D4 — the NAF correctness caveat (the hard part)
Incremental enqueue is sound for **monotone** premises (a derived fact
stays true; append-only). It is **not** automatically sound for
`(absent P)` premises: a hypothesis can make a previously-absent P
present, which would *retract* the justification for a root firing that
fired *because* P was absent. The current from-scratch re-saturation
re-evaluates absents in the fork's context and silently gets this right;
the delta path must replicate it. Approach: treat absent-premised plans as
a **re-validation set** — on the fork delta, re-run `absents_still_pass`
(`match.py:159`, already exists) for every root firing whose plan contains
an `AbsentGuard`, and re-open any whose absent is now violated. Bounded:
only absent-premised firings, only against the delta. **This is the
correctness gate for the whole stage** (Q-B2v.A).

### D5 — semi-naive matching (SHIPPED 2026-06-14, ~1.16×) — the deep lever

The post-D2 caller-split (`profile_solve.py --callers`, zebra2
`stop_after=1`) settled where the residual ~72% matching lives:
**`match.run` is 99.9% the saturator's `_enqueue_pass`** (134,697 calls /
45.4 s); hypgen + lookahead together are **0.04 s** — *not* a lever (which
D3 independently confirmed from the other side: the fork cold-passes are
cheap). So the remaining cost is the **within-run delta matching itself**.

D2 narrowed **which** plans a delta re-matches; it still runs each chosen
`match.run` **from the plan's entry Scan**, re-scanning that relation's
whole extent — though exactly one new fact triggered the pass. The deep fix
is **semi-naive matching**: seed the match **at** the newly-derived fact
(bind it into the premise step on its relation) and propagate the remaining
joins, so a delta costs O(the new fact's joins), not O(relation extent).
This narrows **where in the plan** the match starts — completing what D2
(*which* plans) and the B-idx participation index (bound Scans) began; it is
classic semi-naive / RETE delta evaluation.

Sketch: for plan `P` re-matched on delta relation `R`, for each Scan/Join
step `s ∈ P` with relation `R`, run `P` with `s` pre-bound to the new fact
and the join order re-rooted at `s`; union the results (the index still
narrows the *other* bound steps). The real work is **re-rooting the join
order per seed** — `JoinPlan.steps` are entry-ordered, so this needs a
matcher that can start mid-plan (or per-step seed plans precompiled).

Caveats: same NAF/completeness care as D2 (a seeded run must still see its
`AbsentGuard`s; `_scan_relations`' absent recursion still governs *which*
plans re-match). **Effort: D2-sized or larger, correctness-sensitive.**

**AS BUILT 2026-06-14 — SHIPPED, ~1.16×.** `match.run_seeded` seeds each
delta fact into a plan's positive Scan/Join premises (premises rebuilt in
original order, so provenance is identical to `run`); `_enqueue_pass`
partitions delta plans — **seed** positive-premise plans, **full-match** the
absent-flip plans (a `forall` that may flip — not seedable; the D2
completeness case, via `_absent_relations`) + reflective new plans.
Validated: **1196 passed** with `EIN_RUN_SLOW=1`, `enterings_total`/verdict
identical to HEAD, ruff clean. Win **~1.16×** (PyPy zebra2 stop_after=1
2.17 s → 1.88 s median, non-overlapping samples) — real but modest: the 91%
re-discovery was cProfile self-time that *inflates* the cheap `_seen`-skip
re-scan (PyPy's JIT makes it cheaper), and the absent-flip plans stay full.
**Phase cumulative ~3.6×** (6.7 s → 1.88 s).

## Open questions

- **Q-B2v.A — NAF re-validation soundness (the gate).** Is the
  absent-premised re-open set complete? A derived *negative* `(not P)` is a
  positive fact (monotone); only literal `(absent …)` premises are the
  risk. Enumerate the absent-premised rules (negative-completion,
  `domain/range-elimination`'s `forall`) and prove the re-validation
  covers them.
- **Q-B2v.B — version stamp vs delta list.** A monotonic introduction-
  version per fact (B2.v prose) vs a per-pass delta list. Version supports
  O(1) "facts newer than V0" queries across forks; the delta list is
  simpler within a run. Likely both (version for cross-fork, delta for
  within-run).
- **Q-B2v.C — scope.** Ship D2 (within-run) first — it benefits the root
  saturation too and is NAF-contained to one run — then D3 (cross-fork)?
  D2 alone collapses 940→~12+δ full matches.
