# S1.8.B2 — Engine lookups traverse parent chain

**Phase:** P1.8 (improvements), Theme B (perf) → COW.
**Estimate:** 1-2 days.
**Status:** parked — **NOT A LEVER** (`fork()` is `0.000s`; see the
[S1.8.B-idx](s1.8.b_indexes.md) baseline + [S1.8.B1](s1.8.b1_cow_overlay.md)).
Sequential after S1.8.B1.
**Depends on:** [S1.8.B1](s1.8.b1_cow_overlay.md).
**Blocks:** [S1.8.B3](s1.8.b3_saturator_invariants.md).

## Context

After S1.8.B1 lands the overlay class, the engine code paths
that look up facts (`_facts_by_relation` queries, contradiction
scanning, hypothesis generation) need to traverse the parent
chain transparently — they shouldn't know whether they're
running on a root KB or a deeply-forked one.

## Tasks

### Task T1.8.B2.1 — Audit lookup sites

Grep for direct dict / list accesses on KB internals:

- `kb._facts_by_relation[...]` → overlay-aware lookup.
- `kb._negated_facts` membership tests → overlay-aware.
- `kb._rule_apps_by_rule` → overlay-aware.
- Any iteration over `kb.facts` or `kb.names`.

### Task T1.8.B2.2 — Wire the overlay

Per the design choice in S1.8.B1:

- **If (i) `ForkedKB`**: the engine code stays unchanged;
  ForkedKB overrides `__getitem__`-style operations to walk
  the chain.
- **If (ii) layered backing store**: introduce a
  `KnowledgeBase.lookup_facts(relation_name)` (etc.) method
  that walks the chain internally; callers use the method
  instead of accessing the dict directly.

### Task T1.8.B2.3 — Tests

- **a.** Saturator on a forked KB sees all facts (root +
  fork's overlay) and only writes to the fork's overlay.
- **b.** Contradiction detector on a forked KB sees the
  combined fact set.

## Acceptance

- All lookup paths overlay-aware.
- No engine-level code accesses `_facts_by_relation` etc.
  directly (or, if it does, only via wrappers that walk the
  chain).
- Saturator + hypgen still pass their existing tests on a
  forked KB.

## Open questions

- **Q-S1.8.B2.A — Performance overhead.** Each lookup now
  walks a chain. For deep forks (zebra's hypothesis loop
  goes ~20 deep), that's 20 dict reads per lookup. Probably
  fine for M1; benchmark in S1.8.B4.
- **Q-S1.8.B2.B — Caching at lookup time.** Could memoize
  "last lookup for rel R returned X"; invalidate on overlay
  edit. Premature; defer.
