## Context — why these eight are inlined today

`zebra2.ein` imports only `std.macro` (`forall`/`open`) and `std.algebra`'s
**`symmetric`/`transitive`/`includes`** (the A5-tail closures). All eight
names in the TODO note are **inlined** (zebra2.ein l.86-512), with their
`derive-*` activators and negative companions. They are not imported for
two concrete reasons:

1. **`std.algebra` does not export the cardinality properties at all.** It
   ships `converse`/`imply*`/`compose`/`meet`/`join`/`identity`/`top`/
   `complement` + the *checks* `irreflexive`/`antisymmetric`/`asymmetric`/
   `connex`/`difunctional` — but **not** `functional`/`injective`/`total`/
   `surjective`/`bijective`.
2. **`std.elim`'s forms differ from zebra2's.** `std.elim` has
   `typecheck-arg-0/1 (?R ?isa ?Dom)`, `domain-elimination (?R ?isa ?OT ?VT)`,
   `no-room-left (?R ?isa ?OT ?VT)` — the **`?isa`-param, positional**
   formulation. zebra2's are `is-a`/`is-a*`-**literal**, relation-signature-
   driven, and it has **`range-elimination`** (no `no-room-left`). This is
   exactly the reconciliation [S1.8.A8](../p1.8_ein_lang_modules/s1.8.a8_general_totality.md) deferred
   as "risky to the canonical acceptance." **S1.8a.f20 is that reconciliation,
   scheduled.**

## Approach

The categorization in the TODO note is sound and maps cleanly:

- **Cardinality properties** (`functional`/`injective`/`total`/`surjective`/
  `bijective`) ARE relation-algebra — they belong in **`std.algebra`**, as
  *checks* in the existing `irreflexive`/`antisymmetric` style (fire
  `(false)` on violation), plus the `bijective` fan-out. The extensive ones
  (`total`/`surjective`) carry the universe as a `?isa Dom Ran` param, exactly
  as `top`/`complement`/`connex` already do (is-a-free bodies — the kernel
  stays is-a-free, [[feedback_kernel_is_a_free]]).
- **Elimination + typecheck** (`typecheck`/`domain-elimination`/
  `range-elimination`) are *inference* rules that consume those properties →
  **`std.elim`** (their existing home), reconciled to one formulation.

Migration method = the proven **A5-tail discipline**
([S1.8.A5](../p1.8_ein_lang_modules/s1.8.a5_migration_imports.md)): for each rule, extract to the
stdlib, import into zebra2 at the same position, and diff the **resolved KB**
(`resolve_imports` + `dump_canonical`). Migrate only the rules whose resolved
KB is **byte-identical**, so `solve()`'s verdict is unchanged *by
construction*; rules that come out as text-variants (different `:why`, the
`?isa`-param rewrite of `total`/`surjective`/elim) either get their `:why`
normalised first or stay inline with the variance recorded — never a silent
trace-visible edit.
