## Baseline — measured 2026-06-14 (redirects this stage to the front)

Fresh profile via the committed `ein.py/demo/profile_solve.py` of
**`zebra2.ein` `solve(stop_after=1)`** under CPython 3.14 + cProfile
(~145 s wall; the PyPy fast-path is ~6 s — the *attribution*, not the
absolute, is the point). The P1.8a README's "re-measure first, the
baseline moved" mandate, executed:

| subsystem            | self-time | calls               | reading |
|----------------------|-----------|---------------------|---------|
| **match/bind**       | **75 %**  | 806 k `match.run`   | the hotspot |
| saturate (driver)    | 2 %       | 10,706 `saturate()` | thin shell; cost is the matcher it calls |
| hypgen / branch      | 0 %       | —                   | |
| contradiction        | ~0 %      | 567                 | negatives are not a time sink |
| **fork/copy**        | **0.000 s** | **24 forks**      | COW (B1-B4) would optimise a 0 % line |

The 23 % "other" bucket is `isinstance` (249 M calls, 20.7 s) + `len`
(136 M, 9.9 s) — builtins called *from* `_bind_arg`, so the matcher is
effectively **~95 %** of the solve. Inside `match.py`:

| fn                  | tottime | ncalls    |
|---------------------|---------|-----------|
| `_bind_args` (:78)  | 48.7 s  | **60.4 M** |
| `_bind_arg` (:42)   | 42.3 s  | **82.2 M** |
| `_run_steps` (:97)  | 16.8 s  | 10.1 M    |

**Two stale premises the data corrects:**

1. **COW fork (B1-B4) is premature.** The 2026-05-21 rant predates the
   P1.7a sound search + P1.7b `_explore_layers` split; `fork()` is now
   0.000 s over 24 calls in a `stop_after=1` solve. Re-measure under
   `--exhaustive` (thousands of forks) before B1-B4 — but it is *not*
   the `stop_after=1` cost.
2. **Theme C (negative-fact volume) is not the cost either.** zebra2's
   saturated root is 62/360 = **17 % negatives** (not the >50 % the old
   zebra-classic estimate implied) and the detector is ~0 s. Lazy
   materialisation would not move this needle.

**The fix this stage ships.** `match.py:110-115` already narrows by
relation (`kb._facts_by_relation.get(step.relation, ())`) but then
**scans the whole relation extent** per Scan/Join, calling `_bind_args`
on every fact even when a slot is already bound — a Join's shared
variable, or a constant `Atom`/`Int` slot. That linear rescan ×
join-recursion *is* the 60 M `_bind_args`. The targeted fix is a
**participation index `(relation, slot, value) → [fact]`** consulted when
a Scan/Join step has ≥1 bound/constant slot, so the inner loop fetches
only candidate facts (often 1) instead of the full extent — precisely the
"facts by object / by arity" index in T1.8.B-idx.3 below. Maintained on
the same `_index_fact` / fork-copy path as `_facts_by_relation`.
Secondary micro-opts (hoist the redundant `len` arity check; replace the
per-arg `isinstance` dispatch with a compiled slot-kind tag) attack the
249 M `isinstance` / 136 M `len`, but the index is the order-of-magnitude
move — it cuts the *number* of binds, not their per-call cost.

## Design — participation index on bound slots (chosen 2026-06-14)

Authoritative design for the fix; the Tasks/Context below are the
original framing. Runtime-driven (no static bound-slot analysis needed —
the live `bindings` dict already says exactly what is bound), so the
change is **one new store index + one match.py helper**, no `compile.py`
change in the core.

### D1 — Data structure

```python
# store.py, alongside _facts_by_relation (the "sixth fact-derived index")
self._facts_by_rel_slot_val: dict[tuple[str, int, str | int],
                                  tuple[Fact, ...]] = {}
```

Maps `(relation_name, arg_position, arg_value) → facts of that relation
whose arg at that position equals that value`. **Keys are restricted to
atomic arg values (`str` object-names + `int`)** — exactly the values a
Join binds on. Nested-`Fact` args (the `(not (R …))` / `(and …)` heads)
are *not* keyed: a rule matching them carries a `NestedPattern` slot (not
a bound atomic slot), so it full-scans the `not`/`and` relation anyway —
keying whole Facts would buy nothing and risks the `==`/`hash` contract.
`str`/`int` are well-behaved (`a == b ⇒ hash(a) == hash(b)`), so a
hash-bucketed index and the matcher's `==` can never disagree.

### D2 — Maintenance (store.py — mirror the existing contract exactly)

Four sites, all parallel to `_facts_by_relation`:

1. **`__init__`** (`:175`) — declare the dict.
2. **`_index_fact`** (`:426`) — after the `_facts_by_relation` append, add
   ```python
   for i, val in enumerate(fact.args):
       if type(val) is str or type(val) is int:
           key = (rn, i, val)
           self._facts_by_rel_slot_val[key] = (
               *self._facts_by_rel_slot_val.get(key, ()), fact,
           )
   ```
   Rebind-to-fresh-tuple, never mutate-in-place — the **F-KB-9 invariant**
   that keeps `fork`/`snapshot` shallow copies byte-identical to a
   `rebuild_indexes()` (store.py:707-709).
3. **`rebuild_indexes`** (`:323`) — reset to `{}` and populate in the
   batch loop next to `fbr`.
4. **`_copy_fact_indexes_into`** (`:716`) — add
   `new._facts_by_rel_slot_val = dict(self._facts_by_rel_slot_val)`.
   Update the docstrings that say "five fact-derived indexes" → six
   (`:628`, `:703`).

### D3 — Lookup (match.py — the only hot-path change)

Replace the `Scan`/`Join` branch (`match.py:110-115`) candidate source:

```python
if isinstance(step, (Scan, Join)):
    for fact in _candidates(step, bindings, kb):
        new_b = _bind_args(step.arg_slots, fact.args, bindings)
        if new_b is not None:
            yield from _run_steps(rest, new_b, (*premises, fact), kb)
    return
```

```python
def _candidates(step, bindings, kb):
    """Facts to try for a Scan/Join — narrowed by the first bound slot."""
    for i, slot in enumerate(step.arg_slots):
        if isinstance(slot, Atom):
            v = slot.name
        elif isinstance(slot, Int):
            v = slot.value
        elif isinstance(slot, Var) and slot.name in bindings:
            v = bindings[slot.name]
            if type(v) is not str and type(v) is not int:
                continue          # nested-Fact binding → not keyed
        else:
            continue              # unbound / NestedPattern slot
        return kb._facts_by_rel_slot_val.get((step.relation, i, v), ())
    return kb._facts_by_relation.get(step.relation, ())   # all-free: full scan
```

Baseline picks the **first** bound slot. Refinement (deferred, optional):
pick the bound slot with the smallest bucket (`min` over the ≤arity bound
slots) for max selectivity — but zebra's binary relations have exactly
one bound join var, so first == only here.

### D4 — Correctness (behaviour-preserving, by construction)

The yielded `(bindings, premises)` stream is **identical** to today's,
because the candidate set returned by `_candidates` is always:

- a **subset** of `_facts_by_relation[rel]` (never iterate more), and
- a **superset** of the facts that bind at the chosen slot — every fact
  with `fact.args[i] == v` is in bucket `(rel, i, v)` (same `_index_fact`,
  same raw `fact.args[i]`), and

`_bind_args` still re-checks **all** slots, so any bucket member that
fails another slot is rejected exactly as before — no false positives, no
missed matches. The index mirrors the matcher's **raw `==`** (it does
*not* apply `resolve.resolve_leaf` eq-class resolution — neither does
`_bind_arg`, lines 55/58), so the two can't drift. AbsentGuard sub-plans
and `absents_still_pass` route through the same `_run_steps`, so they
inherit the speedup with no extra change.

### D5 — What it deliberately does NOT touch

- `_facts_by_relation` semantics (still the all-free-Scan source + the
  `_fact_by_id` / saturator readers) — purely additive.
- The fork/snapshot O(|facts|) shallow-copy — that path is 0.000 s
  (24 forks); COW (B1-B4) stays parked until `--exhaustive` re-measures.
- Eq-class resolution, provenance, NAF, the saturator driver.

### D6 — Expected effect + verification

Target: cut the 60.4 M `_bind_args` on Join steps to ~bucket-size
(≈1-5 vs the full ≈25-fact extent) → a **3-10× drop in matcher work**,
the dominant 75 %. Honest until measured. Verify by:

1. **Full suite green** (`run_tests.sh` / pytest 1192) — the behaviour-
   preservation gate; any index bug shows as a match-count regression.
2. **Re-run `demo/profile_solve.py`** — `_bind_args` ncalls + match/bind %
   before/after, recorded back into this Baseline.
3. **PyPy wall-clock** (`./bench_solve_monotonic_pypy.sh examples/zebra2.ein`)
   stop_after=1 before/after — the real-world number.

### D7 — Risks / edges

- **Memory** (Q-S1.8.B-idx.A): +~(atomic-arity) entries/fact, ~720 at
  zebra scale — negligible; revisit for M2 NL-scale KBs.
- **`bindings[var]` could be a `Fact`** (nested binding) — guarded:
  `type(v) is str|int` only, else fall through to the next slot / full
  scan. Correct, just unoptimised for that slot.
- **Variadic-arity relations** (`not`/`and` with differing arg counts) —
  positional keying is still sound (`_bind_args`'s length check rejects
  mismatched arity); they simply key less selectively.

### D8 — Secondary micro-opt — ATTEMPTED then ABANDONED 2026-06-14 (wash)

Hypothesis: the 249 M `isinstance` + 136 M `len` (the 23 % "other") called
from `_bind_arg` are real cost; inlining the Var/Atom/Int cases into
`_bind_args` with exact-type (`type(s) is …`) dispatch — dropping the
per-arg `_bind_arg` call + the `isinstance` ladder — would cut it.

**Built it, measured it, reverted it.** It is a **wash on both
interpreters**:

| | index-only | + D8 | |
|---|---|---|---|
| PyPy (production), median of 3 | ~4.30 s | ~4.26 s | within noise |
| CPython no-profile, 3 runs | 26.1 s (tight) | ~26 s (noisy) | within noise |

Suite stayed 1192 green either way. **The "23 %" was a cProfile
artifact**: cProfile instruments *every* call, so it inflates the apparent
cost of the 82 M tiny `_bind_arg` calls; without the profiler — and under
PyPy's tracing JIT, which already inlines + type-specialises that path —
the per-call overhead is cheap, so removing it saves nothing.

**Lesson for future matcher work:** validate micro-opts on **PyPy + no-
profile wall**, never on cProfile self-time. The index won because it cut
*algorithmic work* (candidate iterations) — interpreter-independent — not
per-call overhead. Per-call micro-opts in this matcher are not worth the
complexity; the next real lever stays **incremental saturation** — sized
2026-06-14: **91% of firing-work is redundant re-derivation** (`_enqueue_pass`
does ~940 full-KB re-matches per solve, one per productive firing), not the
bind loop. Stage: [S1.8.B2v](s1.8.b2v_incremental_saturation.md).

### Result (measured 2026-06-14) — shipped, ~1.67× on the fast path

A real, low-risk, behaviour-preserving win — **but short of the D6 3-10×
projection.** The projection assumed bound-slot rescans were most of the
matcher cost; the profile shows they were **~34 %**. Tell-tale: `_run_steps`
and `match.run` ncalls are **identical** before/after (10.16 M / 806 k) —
the index removed exactly the *rejected* binds on bound slots (~20 M),
while the *surviving* matches and the **unbound entry-Scans** (a Scan whose
args are all-free — no value to key on) are untouched.

**Where the residual time lives → the next lever.** `saturate()` runs
**10,706 times** in one `stop_after=1` solve (≈ one fork+saturate per
hypgen lookahead probe), each re-matching the whole rule set from scratch
over a near-identical KB. The matcher index speeds each probe ~1.7×; the
order-of-magnitude needs **incremental / version-based saturation** (B2.v /
B5 — process only the commitment delta, not a full re-saturation per probe)
and/or cheaper lookahead. The D8 micro-opt (compiled slot-kind tag vs the
225 M `isinstance` / 95 M `len`, ~23 % of profiled self-time) is the
smaller matcher-local follow-up. The index stays as the foundation those
build on.

## Context

User direction 2026-05-21:

> saturation that produce 150 facts and runs 1.5 seconds IS
> INSANE! Python performance is 1M-10M ops/s. Before
> saturation we have 64 facts and 8 rules. (64×8)² and even
> (64×8)³ both fall short of the time spent. WHAT IS THE
> SATURATION DOING SO LONG?

The answer involves match-bind allocation, NAF re-evaluation,
provenance bookkeeping. Indexes reduce the look-up cost from
linear-in-K to O(1) for the lookup steps that today scan KB.
