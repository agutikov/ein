# S1.8.B1 — ForkedKnowledgeBase overlay class

**Phase:** P1.8 (improvements), Theme B (perf) → sub-theme
**B2 COW**.
**Estimate:** 2-3 days.
**Status:** parked; entry of the COW sub-theme.
**Depends on:** stable KB API (P1.2 shipped).
**Blocks:** [S1.8.B2](s1.8.b2_engine_lookups.md),
[S1.8.B3](s1.8.b3_saturator_invariants.md).

## Context

P1.2's `KnowledgeBase.fork()` shallow-copies the `facts` list +
the four reverse-index dicts (O(|facts|) cost). For P1.5's
hypothesis loop, where each branch potentially forks many
sub-branches, that copy cost compounds.

User direction 2026-05-21: M1's KB is **append-only** — no
deletes, no in-place writes. That makes **true copy-on-write**
trivially correct: a fork inherits the parent's facts +
indexes by reference; the fork's own appends write to per-fork
overlays; lookups consult the fork's overlays first, then fall
through to the parent.

This stage builds the overlay class. S1.8.B2 wires engine
lookups; S1.8.B3 confirms saturator + contradiction-detector
behaviour preserved; S1.8.B4 benches.

## Tasks

### Task T1.8.B1.1 — Design

- **a.** Two implementation paths:
  - **(i) `ForkedKnowledgeBase` overlay class** as a subclass
    or wrapper around `KnowledgeBase`. Forks return overlays
    that compose; root remains a plain KB.
  - **(ii) Rework `KnowledgeBase` itself** with a layered
    backing store (every KB has an optional parent reference;
    lookups walk the chain).
- **b.** Pick one. (ii) is more invasive but more uniform; (i)
  is additive but adds a class hierarchy.

### Task T1.8.B1.2 — Implement

Per the design choice:

- **a.** Overlay storage: per-fork appended facts; per-fork
  per-index delta.
- **b.** Lookups: overlay first, then parent recursively.
- **c.** Identity / equality: fork's identity is its overlay
  chain; equality should be reference-based.

### Task T1.8.B1.3 — Tests

- **a.** Smoke: fork a KB, append facts in the fork, confirm
  parent unchanged.
- **b.** Lookup correctness: fork has facts A + B; parent has
  C + D. Lookup returns A∪B∪C∪D.
- **c.** Nested forks: fork → fork → fork. Lookups walk the
  whole chain.

## Acceptance

- Forked KB returns correct lookups across parent + overlay.
- Smoke test green; nested forks work.
- Existing tests still pass against the unforked path.
