# S1.7c.15 — Split `_LatticeLoopState` into per-entry accumulators

**Phase:** P1.7c, Track B.
**Estimate:** ~3–4 hr. **Medium–high risk** — it is the loop's mutable state;
a field aliased to the wrong entry is a P1.7a-class bug. Gate hard.
**Depends on:** P1.7b baseline. Unblocks [S1.7c.19](s1.7c.19_drop_type_ignore.md)
(the `type: ignore` vanish once each entry's state is concretely typed).
**Finding:** [F-ENG-8](../p1.7b_review_and_refactor/findings.md#eng--monotonic-search-engine-inferencemonotonic).

## Goal

`_LatticeLoopState` (`solver.py:155`) mixes solve-only fields
(`solution_nodes`, `truncated`) with lattice-only fields (`solutions`,
`kb_index`, `state_hash_merges`, `root_was_solved`); **no single run uses all
of them**, so the type carries a 30-line per-field-per-entry caveat
docstring describing which fields are live for which entry.

## Tasks

- Split into per-entry accumulator dataclasses (or a shared base + a
  `_SolveState` / `_LatticeState` pair); each phase function receives the
  accumulator it actually uses.
- Delete the per-field caveat docstring — the types now say it.

## Acceptance

- No field is read by zero entries.
- **Identical enterings + verdicts** across the 3-variant acceptance gate
  (zebra2 / minus-15 / invariant) — this is the hard net.
- Full suite green.
