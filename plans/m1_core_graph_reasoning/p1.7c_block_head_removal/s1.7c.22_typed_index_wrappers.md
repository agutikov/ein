# S1.7c.22 — Typed `_FactIndexes` / `_StaticIndexes` wrappers

> **Status: DONE — RE-SCOPED to a copy-helper (2026-06-03).** A 6-agent
> design workflow (inventory of all 58 index touch-points + a 3-design
> panel) concluded the full typed wrapper is the wrong scope **now**:
> F-KB-9's marquee value was *separating mutable from static indexes* on a
> "4 mutable / 4 static" split, but S1.7.23 deleted the type indexes, leaving
> **5 mutable + 1 static** (`_rules_by_relation`). A `_FactIndexes` /
> `_StaticIndexes` split therefore degenerates to a five-field class + a
> one-field class, and **no** wrapper design actually stops `_index_fact`
> from writing the static index — so the "structural safety" payload isn't
> delivered by any of them. The *real* remaining value is "stop duplicating
> the fork/snapshot copy block so they can't desync" (the duplication
> [S1.7c.21](s1.7c.21_snapshot_shallow_copy.md) introduced). That is captured
> in full by a single `KnowledgeBase._copy_fact_indexes_into(new)` helper —
> ~12 LOC, one file, **zero** test/accessor churn, **zero** hot-path perf
> cost (the property-delegation wrapper would add a descriptor read to the
> per-candidate `_negated_facts` membership tests, and as drafted even had a
> latent bug — turning public `kb.names` into a method). Implemented as the
> helper; the wrapper-class form is not pursued. The finding text's "8 dicts
> / 4 mutable + 4 static" framing is stale.

**Phase:** P1.7c, Track B.
**Estimate:** ~4 hr. **High risk** — the central kb data structure touched by
`__init__` / `rebuild` / `fork` / `_index_fact` / `snapshot`. Gate hardest.
**Depends on:** P1.7b baseline. **Structurally enables**
[S1.7c.21](s1.7c.21_snapshot_shallow_copy.md) and retro-justifies the
S1.7b.4 `fork` index-sharing / in-place writes.
**Finding:** [F-KB-9](../p1.7b_review_and_refactor/findings.md#kb--data-model-kbpy).

## Goal

Eight loose `_*_by_*` dicts are hand-maintained in **four** places
(`__init__` / `rebuild_indexes` / `fork` / `_index_fact`). `_index_fact`
*correctly* omits the four post-load-immutable indexes — but **nothing
enforces** the load-invariance that makes that omission correct. One wrong
edit silently corrupts index state (exactly the class of bug F-KB-3 was).

## Tasks

- Introduce two wrappers:
  - `_FactIndexes` — the four fact-derived indexes; mutable, with `add(fact)`
    + `copy()` (real copy).
  - `_StaticIndexes` — the four post-load-immutable indexes; `copy()` is
    identity (shared by reference, as `fork` already does).
- Route `__init__` / `rebuild` / `fork` / `_index_fact` / `snapshot` through
  the wrappers; the invariant is now in the **type**, not a comment.

## Acceptance

- Behaviour **identical** across the full 3-variant gate + fork-parity +
  shuffle-invariance + the S1.7b.4 double-index regression tests.
- No public KB API churn (wrappers are internal).
- Full suite green; bench ≤ baseline.
