## Why it moved (and why now)

The P1.6 note "owned by P1.6 since it lives next to the DOT renderer work"
was only half the story — the grammar it highlights is the **surface
syntax**, and P1.7c is the surface-syntax phase. The original keyword list
also still named the block wrappers `facts` / `ontology` / `reasoning` /
`rules`, which **no longer exist** ([S1.7c.4](s1.7c.4_migrate_and_drop_shim.md)).
Building it now means highlighting the *current* (flat) language from the
start, against the single source of truth.

## Goal

A lightweight TextMate / LSP-free grammar definition
(`.tmLanguage.json`, e.g. under `utils/vscode-ein/` or `editors/vscode/`)
that gives `.ein` files structural highlighting in VSCode (and any
TextMate-grammar consumer: Sublime, GitHub `linguist`, `bat`, …). No
language server, no build step.

## What to highlight (flat surface, P1.7c)

- **S-expression structure** — `( … )` nesting; atom vs `"string literal"`
  vs `42` / `1..*` (RANGE) vs `;`-comment / `#| … |#` block comment.
- **Declarators** (the closed reserved set —
  [`06_reserved_names.md`](../../../docs/kernel/ir/03-ein-lang/06_reserved_names.md)):
  `relation` · `rule` · `hrule` · `query` · `config` · `trace`
  (+ `macro` once [P1.8 S1.5.9](../p1.8_ein_lang_modules/s1.5.9_ein_lang_macros.md)
  lands). **Highlight these only in head position** (first atom after `(`),
  so a relation/atom that merely *contains* the word isn't miscoloured.
  **Do NOT highlight `ontology` / `facts` / `reasoning` / `rules`** — they
  are ordinary fact heads now, not keywords.
- **Rule-body / ⊥ primitives:** `not` · `and` · `or` · `neq` · `false` ·
  `absent` · `forall` · `open` (the `primitives.STRUCTURAL` / `SUGAR` sets);
  **predicates** `eq` · `neq`.
- **Keywords** (`:keyword` form) — `:match` · `:assert` · `:why` ·
  `:priority` · `:source` · `:rule` · `:using` · `:layer` · `:mode` ·
  `:goal` · `:hrules` · `:hypothesis-relations` · … (a single
  `:[a-z][\w-]*` rule is enough; a fixed list is optional polish).
- **Variables** — `?var` (and `?R*` with the `*` tail).
- **Wildcard** — bare `_`.
- Optional niceties: distinct scope for the `:layer` *value*
  (`ontology`/`fact`/`reasoning`) and for the `=` head.

## Tasks

- Author `<name>.tmLanguage.json` with the scopes above; key declarator
  highlighting off **head position**, not bare occurrence, and source the
  declarator + primitive lists from
  [`06_reserved_names.md`](../../../docs/kernel/ir/03-ein-lang/06_reserved_names.md)
  / [`primitives.py`](../../../ein.py/src/ein_bot/inference/primitives.py)
  so the one closed set stays authoritative (a stray copy drifting from the
  grammar is the main failure mode).
- A minimal `package.json` + `language-configuration.json` registering the
  `.ein` extension, comment tokens (`;`, `#| |#`), and bracket pairs —
  enough to load as an unpacked VSCode extension (no marketplace publish).
- A short README: how to symlink/install the unpacked extension; the note
  that the grammar is *regex* highlighting, not a parser (it will happily
  colour syntactically-invalid ein — correctness is the loader's job).

## Acceptance

- Opening `examples/zebra2.ein` in VSCode (extension installed) colours:
  declarator heads, `?vars`, `:keywords`, strings, comments, and `( )`
  nesting — and does **not** specially colour `ontology` / `facts` /
  `rules` used as fact heads.
- The declarator + primitive lists in the grammar match
  `06_reserved_names.md` (a checklist diff, or a tiny test that greps both).
- Loads as an unpacked extension with no errors in the VSCode dev console.
