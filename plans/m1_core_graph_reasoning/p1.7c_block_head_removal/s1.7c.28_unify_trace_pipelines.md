# S1.7c.28 — Unify the two trace pipelines

> **Status: DONE — RE-SCOPED (2026-06-03).** The doc's task ("have
> `render_trace` consume `parse_trace_steps`; delete `_trace_premises`") is
> **unsafe** and was not done: `_trace_premises` decomposes a `:using`
> clause into **atoms** — `(nation-loc Norwegian H1)` → three DOT nodes
> (`nation-loc`, `Norwegian`, `H1`) — whereas `parse_trace_steps` produces
> **one FactRef** `("nation-loc", ("Norwegian","H1"))` per premise. Routing
> the DOT renderers through `TraceStep`s would change the trace-graph
> structure (a byte-break), and `parse_trace_steps` has **no internal
> caller** anyway (it's a round-trip utility used by tests) — so the
> "two live consumers of one path" premise is also false. What IS shared is
> the *generic step-field extraction* (leading-symbol name + `{kw: value}`
> map); that now lives once as `SForm.leading_symbol()` / `SForm.kw_map()`,
> used by `render_trace`, `_render_trace_dag`, AND `parse_trace_steps`'
> `_parse_step`, while each keeps its own value interpretation (atoms for
> DOT via `_trace_premises`, FactRefs for `TraceStep` via `_parse_using`).
> `_trace_premises` is kept atom-only. `render_trace` output verified
> byte-identical across all 6 views on a real `gaps_solve` trace.
