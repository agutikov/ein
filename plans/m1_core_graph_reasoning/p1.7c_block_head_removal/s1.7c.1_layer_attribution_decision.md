# S1.7c.1 — Decide layer attribution for flat forms (the gating design call)

**Phase:** P1.7c, Track A (block-head removal).
**Estimate:** ~½ day — a **design decision + a written rationale**, no code

**Status:** **DONE 2026-06-02** — decision recorded below
([§ Decision](#decision-recorded-2026-06-02)).
**Depends on:** nothing — this is the crux all later Track-A stages build on.
**Resolves:** [README Q1](README.md#open-design-questions-the-reason-its-not-trivial)
(layer attribution) — the one the README flags as "**the crux**".

## The problem

Today the wrapper block heads *carry* the [`Layer`](../../../ein.py/src/ein/kb/entities.py)
of their children: `(ontology …)` → `Layer.ONTOLOGY`, `(facts …)` →
`Layer.FACT`, `(reasoning …)` → `Layer.REASONING` (`from_ir.py:408-412`).
Drop the wrappers and a flat fact has **no layer signal**. Q1 asks where the
signal comes from instead.

## What the layer is actually load-bearing for (investigated 2026-06-02)

The whole premise that "the layer split is load-bearing for the contradiction
detector" is **only half true**, and the half that matters is *not*
ONTOLOGY-vs-FACT:

- **`REASONING` is the only inference-distinguished layer.** The detector's
  pairing logic (`lookahead.py:139,145`) flags `(not g)` / `g` as a
  contradiction **iff the existing fact is `Layer.REASONING`** — *"cross-layer
  is not a contradiction; see the detector's module docstring."* Given
  (authored) facts never pair this way; only engine-derived ones do.
- **No inference path branches on `ONTOLOGY` vs `FACT`.** Grep of
  `inference/` for those two values returns only *producers* and *render*:
  `closed.py:87` **writes** `CLOSED` as ONTOLOGY; `state_dump.py:120` reads
  ONTOLOGY for **dump styling**. The loader itself already treats them as one
  population — `from_ir.py:331`: `layer in (Layer.FACT, Layer.ONTOLOGY)`.
- **`REASONING` is engine-only on input.** Authored `(reasoning …)` is
  "usually empty; the engine dumps it after a solve" (grammar header). So
  authored flat input never needs to *write* REASONING.

**Conclusion:** for *authored* flat forms the only distinction to recover is
ONTOLOGY-vs-FACT, and it is **purely provenance / render** (the `--layers` /
`--colour-by layer` views, `slice.py:200`, `render.py:364`) — not soundness.

## The decision to make (recommended resolution)

Derive the layer from the **provenance annotation already on the form**, which
is the *same* signal `from_ir.py:325-331` uses to pick the `Provenance` kind:

| flat form carries… | layer | rationale |
|---|---|---|
| `:rule` / `:using` | `REASONING` | derived — engine working-memory dump |
| `:source "(N)"` | `FACT` | an explicit, numbered problem statement |
| neither (schema-shaped: `relation`, or an unannotated structural fact) | `ONTOLOGY` | implicit background assumption |

This is fully derivable, needs no new syntax, and keeps the per-fact
`Fact.layer` field (so every existing render/filter path is untouched).

**Alternatives to weigh and record (pick one, justify in writing):**
1. **Derive from provenance** (above) — recommended; zero new surface.
2. **Per-form `:layer ontology|fact` keyword** — explicit, but adds a reserved
   keyword and verbosity to every line the author wants in ONTOLOGY.
3. **Collapse authored input to a single given-layer** (ONTOLOGY≡FACT for
   authored facts; engine still writes REASONING) — simplest, but loses the
   author's explicit/implicit distinction in the renderer.

## Decision (recorded 2026-06-02)

**Chosen: option 1 (derive from provenance) as the default, with option 2
(`:layer` keyword) as a targeted, authoritative override.** A hybrid — but a
disciplined one: the keyword is *only* written where derivation provably
disagrees with the authored intent, so the common case carries zero new
surface (option 1's promise) while the edge cases stay byte-identical
(option 3's failure mode avoided).

### Why pure option 1 is insufficient (the empirical finding)

The recommendation above (pure derivation) was tested against the **whole
example corpus** with the acceptance oracle — for every fact, compare the
current (block-assigned) `fact.layer` to what the derivation rule
(`:rule`/`:using`→REASONING; `:source`→FACT; else→ONTOLOGY) would assign.
**Pure derivation mismatches 23 facts across 8 files**, in exactly the three
edge-case shapes this stage was told to anticipate:

| # | edge case (named in the original task) | files | n | current | pure-derive |
|---|---|---|---|---|---|
| 1 | **FACT-intent fact with no `:source`** | `features/01,03,04,05` | 13 | FACT | ONTOLOGY ✗ |
| 2 | **authored REASONING fact, no `:rule`** | `saturation/hypothesis-contradiction/{coloc,next-to,right-of}-disproved` | 6 | REASONING | ONTOLOGY ✗ |
| 3 | **structural ONTOLOGY fact that *does* want a source** | `zebra.ein` (the 4 condition-(1) `right-of`) | 4 | ONTOLOGY | FACT ✗ |

So pure option 1 **fails this stage's own acceptance** ("reproduces the
current layer assignment for every `examples/*.ein` fact"). The default
direction was still chosen correctly — **ONTOLOGY** as the no-annotation
default minimises overrides, because the unannotated population is
overwhelmingly ontology (every `is-a`, `relation`, `bijective`, property-tag
fact in `zebra2.ein` and every saturation fixture derives ONTOLOGY for free).
A FACT default would mismatch *hundreds* of ontology facts; ONTOLOGY mismatches
only the 13 in case 1.

### The rule (`_layer_of(form)` spec for [S1.7c.3](s1.7c.3_loader_flat_routing.md))

```
_layer_of(form):
    if form has an explicit  :layer <ontology|fact|reasoning>  →  that Layer   # authoritative
    elif form has  :rule  or  :using                           →  REASONING    # engine-derived
    elif form has  :source                                     →  FACT         # numbered problem statement
    else                                                       →  ONTOLOGY     # implicit background
```

- `:layer` is read by `_layer_of` only; `_fact_args` already drops every
  `KwPair`, so it never becomes a fact arg, and the provenance arm
  (`from_ir.py:325-331`) is untouched (it keys on `:source`/`:rule`, not
  `:layer`) — so an added `:layer fact` on an unsourced fact still yields
  `Provenance.from_source(source=None)`, **identical** to the wrapped load.
- The `:layer` value lexes as a plain `SYMBOL` (`ontology`/`fact`/`reasoning`
  are not in the `not|and|or|neq` exclusion), so no grammar change is needed to
  *carry* it; it parses as an ordinary `kw_pair` today.

### Why the hybrid, not full option 2 or option 3

- **Not full option 2** (`:layer` on every ontology line): the verbosity the
  README rejects. The hybrid writes `:layer` on **23 facts total**, not the
  hundreds of unannotated ontology facts.
- **Not option 3** (collapse ONTOLOGY≡FACT for authored input): violates the
  hard **KB-preserving** constraint — cases 1 and 3 above would silently flip
  `fact.layer`, and the migration's load-both-ways-diff oracle (S1.7c.3/.4)
  would (correctly) reject it.
- **Bonus — it fixes a latent dumper bug.** `state_dump._fact_to_sform`
  *already emits* `:layer reasoning` on derived facts, but the wrapped loader
  **never read it** (layer came from the block), so dump→reload silently
  dropped REASONING for any derived fact whose provenance wasn't rule-kind
  (e.g. `:hypothesis`). Honouring `:layer` makes the round-trip in
  [S1.7c.3](s1.7c.3_loader_flat_routing.md)'s acceptance actually hold.

### Edge cases the loader/migration must handle (enumerated)

1. **FACT fact with no `:source`** → migration appends `:layer fact`
   (provenance unchanged — still `from_source(None)`).
2. **ONTOLOGY fact with a `:source`** → migration appends `:layer ontology`
   (provenance keeps `source="condition (1)"`).
3. **authored REASONING fact with no `:rule`/`:using`** → `:layer reasoning`.
4. **`:source` is *not* made mandatory** on problem statements — an unsourced
   authored fact is legal; it simply needs `:layer fact` to stay FACT-layer
   (otherwise it reads as implicit ONTOLOGY, which is a defensible default but
   not byte-identical to the old `(facts …)` placement).
