# S1.3.3 — Saturation engine + banded priorities

**Phase:** P1.3 (inference rules).
**Estimate:** 3-4 days.
**Depends on:** [S1.3.1](s1.3.1_rule_dsl.md) (compiler + matcher +
engine driver), [S1.3.2](s1.3.2_ten_core_rules.md) (rule families).
**Resolves question:** [M1 Q15 — rule ordering](../open_questions.md#q15--rule-ordering)
+ [Q41 — priority scale + placement](../open_questions.md#q41--rule-priority--scale-and-placement).

## Context

S1.3.1 ships `Engine.step()` — pop one compiled-program firing and
apply. S1.3.3 wraps `step()` in a **saturation loop** that fires
rules until no rule produces a new fact, in priority order, with
deduplication and ordered firing output for the trace renderer.

[Q15](../open_questions.md#q15--rule-ordering) resolved
forward-chaining with a static priority per rule.
[Q41](../open_questions.md#q41--rule-priority--scale-and-placement)
(2026-05-20) locked the scale and placement:

- **Scale**: 100/200/300/900 banded.
- **Placement**: per-rule inside the `(rule …)` form (already in
  zebra.ein after the priority migration in S1.3.1 T1.3.1.8).
- **Order**: lower fires first.

There is **no** static-config priority table. The engine reads
`rule.priority` from `kb.Rule.priority`.

## Tasks

### Task T1.3.3.1 — `Saturator`

`src/ein_bot/inference/saturator.py`:

```python
@dataclass
class Saturator:
    kb:     KnowledgeBase
    engine: Engine            # from S1.3.1

    def saturate(self) -> Iterator[Firing]: ...
    def step(self) -> Firing | None: ...
    def is_stalled(self) -> bool: ...
    def solved(self) -> bool: ...   # query-mode predicate; plugged in by P1.5/P1.7
```

`Firing` is the dataclass from
[S1.3.1 T1.3.1.5](s1.3.1_rule_dsl.md#task-t1315--firing-record--assert).

`saturate()` is an iterator — the trace renderer (P1.6) pulls
firings one at a time and emits per-step DOT snapshots.

Algorithm (forward-chaining with priority queue):

1. Initialise a priority queue keyed by
   `(kb.Rule.priority, FIFO-tiebreaker)`. Lower priority fires
   first.
2. For each `(rule, activator)` cache entry in the engine, enqueue
   candidate firings from `engine.run(rule, activator)`.
3. On pop: re-verify the firing still produces a *new* fact
   (the KB may have grown since the candidate was queued; cheap
   re-check via `kb.facts.contains((rule, args))`). If duplicate,
   drop the candidate but record a "redundant" marker for the
   trace (T1.3.3.3).
4. If verified, apply the firing: `kb.add_fact(...,
   layer=Layer.REASONING, provenance=Provenance.from_rule(...))`.
   Yield the `Firing`.
5. Newly added facts may unlock other rules — enqueue candidates
   from any (rule, activator) whose pattern's relation names
   include the new fact's head.
6. Terminate when the queue is empty.

Performance: O(N × |rules|) re-matches per step is fine for Zebra
scale (≤100 facts, 6-7 rules). No Rete network for M1.

### Task T1.3.3.2 — Priority bands documentation

Document the 4 bands (Q41 resolution) in this stage file + the
P1.3 README. **Per-rule placement** is enforced — no static config
table; the saturator reads `kb.Rule.priority`.

| band       | priority | rules in M1                                                    |
|------------|---------:|----------------------------------------------------------------|
| propagate  | 100      | `symmetric`, `implies`                                          |
| derive     | 200      | `transitive`, `square-fwd`, `square-bwd`                        |
| eliminate  | 300      | `type-exclusivity`                                              |
| hypothesis | 900      | `hypothesis-contradiction` (fires only from P1.5 fork/retract) |

Within-band fine-grained ordering (e.g., 110 / 120 for two
propagators) is open and per-puzzle — document the convention,
don't bake in within-band values.

### Task T1.3.3.3 — Idempotency + dedup + redundant markers

A rule may produce the same conclusion via two different bindings
(e.g. `symmetric` fires twice on a single pair). `kb.add_fact`
already dedupes by `(relation_name, args)`. The saturator:

- Skips applying duplicates (no second insertion).
- Records the firing with `Firing.redundant=True` so the trace
  shows the engine considered it but doesn't double-display the
  conclusion.

### Task T1.3.3.4 — Stall detection

```python
def is_stalled(self) -> bool:
    """No firing in the most recent cycle has produced a new fact."""
```

`is_stalled()` is the trigger for P1.5 hypothesis branching. The
engine exposes it so the P1.5 loop can call
`while not solved() and not is_stalled(): step()`.

### Task T1.3.3.5 — REASONING-layer write path

The saturator writes each derived fact via `kb.add_fact(...,
layer=Layer.REASONING, provenance=Provenance.from_rule(...))`. The
`Firing` dataclass carries the resolved premise Facts so trace
generation can render them without re-resolving by id.

For Q40's promoted `hypothesis-contradiction` rule: when the rule
fires (via the matcher, since Q40 Option A made it a normal rule),
the asserted `(not <hypothesis-fact>)` lands in the parent KB's
REASONING layer with provenance pointing at the contradiction
premises. The branch retract (mark `provenance.kind='rejected'`
on all branch facts) is **P1.5's** responsibility — the saturator
emits the negation; P1.5 propagates the rejection.

### Task T1.3.3.6 — Tests

`tests/inference/test_saturator.py`:

- Triangle-only saturation on a 3-edge graph reaches transitive
  closure.
- Mixed-rule saturation on zebra.ein's initial facts fires rules
  in priority-band order (propagate → derive → eliminate),
  asserted via the order of yielded Firings.
- Re-running `saturate()` on an already-saturated KB yields zero
  firings.
- `is_stalled()` flips false after a fact is added, true after
  closure.
- Redundant-firing marker: when the same `(rel, args)` could be
  derived by two distinct bindings, the second firing is recorded
  with `redundant=True` and no double-insertion.

## Acceptance

- `pytest tests/inference/test_saturator.py` ≥ 12 tests, green.
- The saturator's API takes `KnowledgeBase`, not `Graph`. No
  `priority_class: Mapping[str, int]` argument. The signature is
  `Saturator(kb).saturate() -> Iterator[Firing]`.
- Zebra.ein's initial saturation completes (no infinite loop)
  and the trace shows the expected firings for the priority-band
  ordering.
- A tiny stand-alone benchmark in `scripts/bench_saturate.py`
  reports < 200 ms wall-clock for the Zebra `fact`-layer initial
  saturation on a laptop. Informational; not a hard CI gate.
