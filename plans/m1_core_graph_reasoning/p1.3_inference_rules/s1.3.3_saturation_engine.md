# S1.3.3 — Saturation engine + rule ordering

**Phase:** P1.3 (inference rules).
**Estimate:** 3-4 days.
**Depends on:** S1.3.1 + S1.3.2.
**Resolves question:** [M1 Q15 — rule ordering](../../open_questions.md#q15--rule-ordering).

## Context

A registry of rules and a matcher are not yet an engine. The
saturation engine **fires rules until no rule can produce a new
edge**, honouring rule priorities, deduplicating conclusions, and
emitting an *ordered* sequence of firings (needed by P1.6's trace
renderer — the order is part of the explanation).

Working answer from
[M1 Q15](../../open_questions.md#q15--rule-ordering): forward-chaining
queue with a static priority class per rule. Cheap propagation rules
first; search-like / hypothesis rules last. Revisit if the trace
shows obviously bad ordering.

## Tasks

### Task T1.3.3.1 — `Saturator`

`src/ein_bot/rules/saturator.py`:

```python
class Saturator:
    def __init__(self, graph: Graph, registry: Registry,
                 *, priority_class: Mapping[str, int] | None = None): ...
    def saturate(self) -> Iterator[Firing]: ...
    def step(self) -> Firing | None: ...
```

`Firing` carries `(rule_name, bindings, derived_edge_id, premise_ids)`.
`saturate` is an iterator — the caller (trace renderer) can pull
firings one at a time and emit per-step DOT snapshots.

Algorithm (forward-chaining):

1. Initialise a priority queue keyed by `(rule.priority, FIFO)`.
2. For each `Rule` × current `Graph`, compute matches lazily and
   enqueue candidate firings.
3. On pop, *verify* the firing still applies (the graph may have
   changed since the candidate was queued; cheap re-match).
4. If verified, apply: insert the new edge with provenance, yield
   `Firing`, and enqueue any *newly enabled* rules. Otherwise drop.
5. Terminate when the queue is empty.

Performance is fine for Zebra-scale (100 edges, 10 rules) — we
budget O(N × |rules|) re-matches per step; no Rete network for M1.

### Task T1.3.3.2 — Priority classes

Three priority bands; static map. Pasted into the engine config:

| band       | priority | rules                                                              |
|------------|---------:|--------------------------------------------------------------------|
| propagate  | 100      | arc-consistency, inverse-relation, equality-substitution           |
| derive     | 200      | triangle-composition, spatial-square, global-cardinality           |
| eliminate  | 300      | all-different-exclusion, elimination-by-exhaustion, forced-by-unique-position |
| hypothesis | 900      | hypothesis-contradiction (only fires from P1.5)                    |

Lower number fires first. Hypothesis-contradiction is reserved for
the hypothesis loop, not normal saturation.

### Task T1.3.3.3 — Idempotency + dedup

A rule may match in multiple ways producing the same conclusion
edge. After applying, the graph's `add_edge` already returns the
existing id if the `(rel, args)` tuple is duplicated — but the
*firing* should still be reported (with a "redundant" marker) so
the trace shows the engine considered it. Marker prevents the
trace renderer from re-displaying the conclusion.

### Task T1.3.3.4 — Quiescence vs progress

Detect when saturation stalls **with the puzzle unsolved**. That's
the trigger for hypothesis branching ([P1.5](../p1.5_hypothesis_loop/README.md)).
The engine exposes:

```python
saturator.is_stalled()    # no rule produced a new edge in the last cycle
saturator.solved()        # query-mode dependent: P1.5/P1.7 plug in the predicate
```

### Task T1.3.3.5 — Tests

`tests/rules/test_saturator.py`:

- Triangle-only saturation on a 3-edge graph reaches transitive
  closure.
- Mixed-rule saturation on a 10-edge graph fires in priority order
  (assertion on the order of `Firing`s).
- Re-running saturate on an already-saturated graph yields zero
  firings.
- `is_stalled()` flips false when a fact is added, true again
  after closure.
