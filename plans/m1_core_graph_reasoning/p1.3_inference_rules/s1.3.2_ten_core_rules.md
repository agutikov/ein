## M1 core — 7 rules

Each rule below lives **inline in `examples/zebra.ein`** (Q30
deferred a universal library to P1.8). Each entry shows:

- the rule name (used in `:why` templates and trace output);
- the rule type (T1 / T2 / T3 — see
  [`docs/kernel/ir/01-ein-graph/02_rules.md`](../../../docs/kernel/ir/01-ein-graph/02_rules.md));
- its `:priority` band (Q41 100/200/300/900);
- its activator pattern (the rule-application fact that enables
  the rule for a specific relation, if T2 polymorphic);
- what `:assert` produces.

| # | Rule              | Type            | Params   | Activators                                              | `:assert`                          | Priority      |
|---|-------------------|-----------------|----------|----------------------------------------------------------|-------------------------------------|---------------|
| 1 | `symmetric`       | T2 polymorphic  | `(?rel)` | `(symmetric co-located)`, `(symmetric next-to)`           | positive `(?rel ?b ?a)`             | 100 propagate |
| 2 | `transitive`      | T2 polymorphic  | `(?rel)` | `(transitive co-located)`                                 | positive `(?rel ?a ?c)`             | 200 derive    |
| 3 | `implies`         | T2 polymorphic  | `(?p ?q)`| `(implies right-of next-to)`                              | positive `(?q ?a ?b)`               | 100 propagate |
| 4 | `square-fwd`      | T2 polymorphic  | `(?R)`   | `(square-fwd right-of)`                                   | positive `(co-located ?b ?y)`       | 200 derive    |
| 5 | `square-bwd`      | T2 polymorphic  | `(?R)`   | `(square-bwd right-of)`                                   | positive `(co-located ?a ?x)`       | 200 derive    |
| 6 | `type-exclusivity`| non-generic     | `()`     | none — fires for every `(instance ?a ?T) (instance ?b ?T)` | **negative** `(not (co-located …))` | 300 eliminate |
| 7 | `hypothesis-contradiction` | non-generic | `()` | none — matches synthetic `(hypothesis ?h)` + `(contradiction-under ?h)` facts emitted by P1.5 | **negative** `(not ?h)` in the parent layer | 900 hypothesis |

Six T2 activators in the ontology → six compiled programs from
S1.3.1's compiler. `type-exclusivity` runs unconditionally over
the cartesian product of instance pairs sharing a type.
`hypothesis-contradiction` is dormant in normal saturation; P1.5's
fork/contradict machinery emits the activator facts.

## Tasks

### T1.3.2.1 — `symmetric`

```lisp
(rule symmetric (?rel)
  :match  (?rel ?a ?b)
  :assert (?rel ?b ?a)
  :why    "{?rel} is symmetric: {?a} ↔ {?b}."
  :priority 100)
```

Activators (in zebra.ein ontology): `(symmetric co-located)`,
`(symmetric next-to)`. T2 polymorphic — the compiler emits two
specialized programs.

### T1.3.2.2 — `transitive`

```lisp
(rule transitive (?rel)
  :match  (and (?rel ?a ?b)
               (?rel ?b ?c)
               (neq ?a ?c))
  :assert (?rel ?a ?c)
  :why    "{?rel} is transitive: {?a} →{?rel}→ {?b} →{?rel}→ {?c}."
  :priority 200)
```

The migrated form (Q32 dropped `:where`); `neq` dispatches via
the predicate registry. Activator: `(transitive co-located)`.

### T1.3.2.3 — `implies`

```lisp
(rule implies (?p ?q)
  :match  (?p ?a ?b)
  :assert (?q ?a ?b)
  :why    "{?p} implies {?q}: from {?a} →{?p}→ {?b}, derive {?a} →{?q}→ {?b}."
  :priority 100)
```

Two-parameter T2. Activator: `(implies right-of next-to)`. The
compiler binds `?p = right-of` and `?q = next-to`, producing a
specialized program with both names baked in.

### T1.3.2.4 — `square-fwd`

```lisp
(rule square-fwd (?R)
  :match  (and (?R ?a ?b)
               (?R ?x ?y)
               (co-located ?a ?x))
  :assert (co-located ?b ?y)
  :why    "square ({?R}, fwd): {?a}↔{?x} bridges {?b}↔{?y}."
  :priority 200)
```

Activator: `(square-fwd right-of)`. The compiler bakes `right-of`
into the two scan steps; `co-located` is a literal head in the
third step (always uses `kb.facts.by_relation["co-located"]`).

### T1.3.2.5 — `square-bwd`

```lisp
(rule square-bwd (?R)
  :match  (and (?R ?a ?b)
               (?R ?x ?y)
               (co-located ?b ?y))
  :assert (co-located ?a ?x)
  :why    "square ({?R}, bwd): {?b}↔{?y} bridges {?a}↔{?x}."
  :priority 200)
```

Activator: `(square-bwd right-of)`. Mirrors T1.3.2.4 with the
co-located bridge on the opposite leg. The user-facing trace
distinguishes "fwd" / "bwd" via `:why`.

### T1.3.2.6 — `type-exclusivity`

```lisp
(rule type-exclusivity ()
  :match  (and (instance ?a ?T)
               (instance ?b ?T)
               (neq ?a ?b))
  :assert (not (co-located ?a ?b))
  :why    "{?a} and {?b} are distinct instances of {?T} — distinct slots."
  :priority 300)
```

Migrated form (Q32: `:where` → positional `(neq …)`).
Non-generic — no activator; the compiler emits one program that
scans `kb.facts.by_relation["instance"]` ×
`kb.facts.by_relation["instance"]` with a join on `?T` and a
guard on `(neq ?a ?b)`. The asserted negative fact `(not
(co-located ?a ?b))` is a `Fact` whose first arg is a nested
`Fact` (R9's widening — Q40 Option A's data-model precondition).

### T1.3.2.7 — `hypothesis-contradiction`

```lisp
(rule hypothesis-contradiction ()
  :match  (and (hypothesis ?h)
               (contradiction-under ?h))
  :assert (not ?h)
  :why    "Assumed {?h}; branch derived a contradiction. Therefore (not {?h})."
  :priority 900)
```

Q40 Option A promoted this rule into the M1 core. `?h` binds to a
relational node (a nested `Fact`); the matcher's NestedPattern
handling unifies it with the synthetic facts emitted by P1.5's
fork / contradict machinery.

The rule lives in zebra.ein (or a hypothesis-specific puzzle
file); P1.5 owns the fork-and-emit side and the branch retraction
(marking branch facts `provenance.kind='rejected'`).

### T1.3.2.8 — Negative tests + audit

`tests/inference/test_no_false_positives.py`. For each rule, a
hand-built KB that *almost* matches but shouldn't fire (wrong
type, missing activator, partial bindings, neq violations).
Catches over-eager matchers.

### T1.3.2.9 — Optional: re-evaluate `forced-by-unique-position`

The original T1.3.2.9 (`forced-by-unique-position`, "single")
was deferred. Under Q33's `neq + not` framing, uniqueness
`single ?x P(?x)` is expressible as:

```lisp
(and P(?x)
     (not (and P(?y) (neq ?y ?x))))
```

R3 audits whether this form produces useful firings on Zebra's
trace. If yes — and the engine handles the nested-`not` pattern
naturally — promote into the M1 core as an 8th rule. If no —
keep deferred and document why.

Outcome of this audit is captured in this stage file's "Decisions"
section at task completion time.

## Deferred to followups

The 4 rules below from the original 10-taxonomy depend on
primitives Q33 deferred. They ship in a future followup adjacent
to F4 — **not** P1.4, **not** P1.7. Each gets a one-paragraph stub
preserving the design thinking.

### `all-different-exclusion` (deferred)

Family: exclusivity / `allDifferent`. Needs variadic args
`?xs...` — not in current grammar.

```ein
(rule all-different-exclusion
  :match (and (hrel allDifferent ?xs...)
              (= ?xi ?v))
  :assert (forall ?xj in ?xs, ?xj != ?xi => (!= ?xj ?v))
  ...)
```

When the variadic grammar lands in a followup, this rule slots
in. The Zebra walkthrough's *"H1 = Yellow ⇒ H1 ≠ Red, H2 ≠ Yellow,
…"* step is achievable in M1 via `type-exclusivity` + the
implicit pairwise distinctness over instances of the same type;
explicit `allDifferent` is sugar for that.

### `elimination-by-exhaustion` (deferred)

Family: elimination by exhaustion. Needs `dom`, set diff,
`forall` quantification.

```ein
(rule elimination-by-exhaustion
  :match (and (dom ?a ?d)
              (forall ?v in (?d \ ?last), (!= ?a ?v)))
  :assert (= ?a ?last)
  ...)
```

Depends on [Q26](../open_questions.md#q26--compound--virtual-node-kinds-for-higher-order-rules)
for `dom` (compound node kinds) + Q33 (set diff, forall).

### `arc-consistency-propagate` (deferred)

Family: arc consistency. Needs `dom`, `functional`, `in`
machinery — same Q26 + Q33 dependency family.

### `global-cardinality` (deferred)

Family: global cardinality. Needs `count` aggregate + `forall` +
variadic.
