# S1.3.2 — Ten core rule families

**Phase:** P1.3 (inference rules).
**Estimate:** 1-2 weeks.
**Depends on:** S1.3.1 DSL + matcher.
**Resolves question:** [M1 Q5 — what "enough" means](../../open_questions.md#q5--what-enough-means)
(partial — finalised in P1.7).
**Implements idea:** [06 §A rough taxonomy of inference rules used in practice](../../../docs/ideas/06-inference-rules-completeness.md#a-rough-taxonomy-of-inference-rules-used-in-practice).

## Context

The PoC ships two rules (triangle, square). The Zebra walkthrough
in [idea 08](../../../docs/ideas/08-human-style-deductive-trace.md)
uses six. The taxonomy in
[idea 06](../../../docs/ideas/06-inference-rules-completeness.md)
lists ten families that recur in human reasoning on logic puzzles.
Implementing all ten gives **explanation-completeness on Zebra** —
the M1 acceptance target.

Each rule below gets:
- a name (used in `:why` templates and trace output);
- an IR `(rule …)` definition in `examples/rules.ein`;
- a positive test case and a negative ("doesn't fire here") test.

## Tasks

### Task T1.3.2.1 — `triangle-composition`

Family: composition / transitivity (idea 06 row 1).

```ein
(rule triangle-composition
  :match (and (rel ?r ?a ?b)
              (rel ?r ?b ?c)
              :where (transitive ?r))
  :assert (rel ?r ?a ?c)
  :why    "From {p0} and {p1}, since `{r}` is transitive, `{a} {r} {c}`.")
```

The PoC's original triangle rule. `(transitive ?r)` is a predicate
read from the ontology's `:transitive` flag.

### Task T1.3.2.2 — `equality-substitution`

Family: equality (idea 06 row 2).

```ein
(rule equality-substitution
  :match (and (= ?a ?b) (rel ?r ?a ?c))
  :assert (rel ?r ?b ?c)
  :why    "Since `{a} = {b}`, we can substitute in {p1}: `{b} {r} {c}`.")
```

For M1 this fires off explicit `(= …)` facts; the union-find from
S1.2.1 is the substrate. E-graph promotion is deferred.

### Task T1.3.2.3 — `all-different-exclusion`

Family: exclusivity / `allDifferent` (idea 06 row 3).

```ein
(rule all-different-exclusion
  :match (and (hrel allDifferent ?xs...)
              (= ?xi ?v))
  :assert (forall ?xj in ?xs, ?xj != ?xi => (!= ?xj ?v))
  :why    "By `allDifferent({xs})`, the value `{v}` is held only by `{xi}`, so other members differ from it.")
```

The IR shorthand `?xs...` is variadic over hyperedge args. The
worked Zebra trace step *"H1 = Yellow ⇒ H1 ≠ Red, H2 ≠ Yellow, …"*
fires this rule.

### Task T1.3.2.4 — `elimination-by-exhaustion`

Family: elimination by exhaustion (idea 06 row 4).

```ein
(rule elimination-by-exhaustion
  :match (and (dom ?a ?d)             ;; domain of ?a is set ?d
              (forall ?v in (?d \ ?last), (!= ?a ?v)))
  :assert (= ?a ?last)
  :why    "Since `{a}` differs from {ruled-out}, and `dom({a}) = {d}`, the only remaining value is `{last}`.")
```

The "not green, not white, not red ⇒ yellow" step. Bookkeeping for
`dom` lives in S1.4 (constraints).

### Task T1.3.2.5 — `hypothesis-contradiction`

Family: assume + ⊥ (idea 06 row 5). Special: the rule itself doesn't
*derive*; it *retracts* under engine control. P1.5 wires it.

```ein
(rule hypothesis-contradiction
  :match (and (hypothesis ?h) (contradiction-under ?h))
  :assert (not ?h)
  :why    "Assume `{h}` — derives ⊥, so `not {h}`.")
```

### Task T1.3.2.6 — `arc-consistency-propagate`

Family: arc consistency (idea 06 row 7).

```ein
(rule arc-consistency-propagate
  :match (and (rel ?r ?a ?b) (functional ?r) (dom ?b ?dom-b)
              (!= ?b ?other) (in ?other ?dom-b))
  :assert (dom ?b (remove ?other ?dom-b))
  :why    "By functionality of `{r}` and the known `{b} ≠ {other}`, drop `{other}` from `dom({b})`.")
```

Removes infeasible values from candidate domains; powers the
"only-one-left" downstream firing of `elimination-by-exhaustion`.

### Task T1.3.2.7 — `spatial-square`

Family: spatial / interval (idea 06 row 8). The PoC's "square".
Implementation uses the position lattice from
[P1.4 S1.4.2](../p1.4_constraints/s1.4.2_spatial.md), so `:assert`
is a Python callable rather than a pure pattern.

```ein
(rule spatial-square
  :match (and (rel right-of ?a ?b)
              (immediate ?a)
              :where (in-positions ?a ?b))
  :assert! python:ein_bot.rules.spatial.assert_position_succ
  :why    "Since `{a}` is immediately right of `{b}`, pos(`{a}`) = pos(`{b}`) + 1.")
```

The `!` marks an escape-hatch rule (audited specially in the test
suite — every `python:` rule needs a doc reference).

### Task T1.3.2.8 — `global-cardinality`

Family: global cardinality (idea 06 row 9).

```ein
(rule global-cardinality
  :match (and (hrel exactly-n ?xs... ?v ?n)
              (count (?xi in ?xs, (= ?xi ?v)) ?k)
              :where (= ?k ?n))
  :assert (forall ?xj in ?xs, (not-set ?xj ?v) => (!= ?xj ?v))
  :why    "Exactly {n} of {xs} equal `{v}`; we already have {k} matches, so the rest cannot.")
```

"Exactly one zebra in five houses" → once we know which, the others
are eliminated.

### Task T1.3.2.9 — `forced-by-unique-position`

Family: forced-by-unique-position (idea 06 row 10).

```ein
(rule forced-by-unique-position
  :match (and (dom ?a ?d) :where (single ?d))
  :assert (= ?a (the ?d))
  :why    "Only one possible value for `{a}` remains: `{the-d}`.")
```

The dual of exclusivity — when only one candidate survives, *assign*
it. Tight loop with `arc-consistency-propagate` and
`elimination-by-exhaustion`.

### Task T1.3.2.10 — `inverse-relation`

Family: bonus (utility). The PoC has paired `right-of` / `left-of`,
`next-to` going both ways. Lifting that into a rule:

```ein
(rule inverse-relation
  :match (rel ?r ?a ?b) :where (inverse ?r ?rinv)
  :assert (rel ?rinv ?b ?a)
  :why    "`{r}` and `{rinv}` are inverses; from {p0} we get `{b} {rinv} {a}`.")
```

Demoted to T10 because the engine can also be told via ontology
that `inverse-of(left-of, right-of)`; this rule fires the inverse
implicitly.

### Task T1.3.2.11 — Negative tests + audit

`tests/rules/test_no_false_positives.py`. For each rule, a hand-built
graph that *almost* matches but shouldn't fire (wrong type, missing
:where, partial bindings). Catches over-eager matchers.

  (or analogous) entry explaining its implementation; audit
  script in `tests/rules/test_python_rules_documented.py`.
