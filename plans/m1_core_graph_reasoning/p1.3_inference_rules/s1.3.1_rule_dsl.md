# S1.3.1 — Rule DSL + pattern matcher

**Phase:** P1.3 (inference rules).
**Estimate:** 4-5 days.
**Depends on:** P1.1 IR, P1.2 graph.
**Resolves question:** [M1 Q4 — rule presentation language](../../open_questions.md#q4--rule-presentation-language).

## Context

The rule presentation language has to balance four requirements:

1. **Structural** — rules are graph rewrites, not arbitrary Python.
   This is what lets the trace planner say "rule X fired with
   premises P, Q".
2. **Readable in plain text** — a junior collaborator should be
   able to add a new rule without reading the engine.
3. **Escape hatch** — at least one rule (spatial arithmetic in
   P1.4) is hard to express purely structurally. The DSL allows
   a Python callable as `:assert` for those cases.
4. **GBNF-able** — M2.P2.3 will lift the DSL into a GBNF grammar so
   LLMs can *propose* new rules.

The DSL spec lives in the IR (P1.1 S1.1.1 T1.1.1.2); this stage
implements the matcher and the registry.

## Tasks

### Task T1.3.1.1 — Pattern AST + compile-from-IR

`src/ein_bot/rules/pattern.py`. Take a `(:match …)` IR sub-form
and compile it to an internal `Pattern` AST:

```python
@dataclass(frozen=True, slots=True)
class Var: name: str               # ?a → Var('a')
class Atom: name: str              # bare atom matches itself
class Hedge:
    rel:  str | Var
    args: tuple['PNode', ...]
class And: terms: tuple['PNode', ...]
class Where: clauses: tuple[WhereClause, ...]   # :where (transitive ?r) ...

PNode = Var | Atom | Hedge | And | Where
```

Plus a typed-binding environment `Bindings = dict[VarName, NodeId]`.

### Task T1.3.1.2 — Matcher

`src/ein_bot/rules/match.py`. Given a `Pattern` and a `Graph`, yield
all valid `Bindings`. Strategy:

- Pick the most-selective term first (heuristic: smallest indexable
  set on `g.by_rel[rel]` / `g.by_arg[a]`).
- Variable propagation: once `?a` is bound, downstream uses of `?a`
  filter by `arg == a`.
- `:where` clauses are predicates on the bindings; checked at the
  end of each candidate.

Targeted complexity: `O(|matches|)`-ish for the common Zebra
arity-2 case. Performance work itself is deferred — for M1 we just
need correctness on a 100-edge graph.

### Task T1.3.1.3 — Rule registry

`src/ein_bot/rules/registry.py`:

```python
@dataclass(frozen=True, slots=True)
class Rule:
    name:     str
    match:    Pattern
    assert_:  AssertSpec                 # IR template OR python callable
    why:      str                        # :why "From {0} and {1}, ..."
    priority: int = 100                  # P1.3 S1.3.3 picks the queue order

class Registry:
    rules: dict[str, Rule]
    @classmethod
    def from_ir(cls, ir_forms) -> 'Registry': ...
    def fire(self, name: str, bindings: Bindings, g: Graph) -> EdgeId: ...
```

`fire` produces:
1. the conclusion edge in the **reasoning** layer;
2. a `Provenance(kind='rule', rule=name, premises=...,
   bindings=bindings)` (P1.2 S1.2.3 records).

`AssertSpec` is normally an IR template — substituting `?vars` with
their bound atoms — but can be a callable for arithmetic
conclusions (P1.4).

### Task T1.3.1.4 — Engine integration shim

Tiny driver `src/ein_bot/rules/engine.py`:

```python
def fire_once(g: Graph, reg: Registry) -> int:
    """Try every rule on the current graph; return # of new edges."""
```

S1.3.3's saturation engine builds on this.

### Task T1.3.1.5 — Tests

`tests/rules/test_match.py`:

- match a single binary edge → 1 binding;
- match a triangle pattern over a 3-edge graph → expected bindings;
- `:where` clauses prune correctly;
- variable reuse forces equality of bindings (`(rel ?r ?a ?b)` +
  `(rel ?r ?b ?a)` matches only on symmetric edges);
- `fire()` creates an edge with full provenance.
