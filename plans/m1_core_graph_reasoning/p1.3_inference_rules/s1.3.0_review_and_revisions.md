# S1.3.0 — Review of P1.3 against shipped P1.1 + P1.2, with revisions

**Phase:** P1.3 (inference rules).
**Type:** *meta-stage* — preceeds S1.3.1; produces no code.
**Status:** open, 2026-05-19.
**Depends on:** P1.1 (shipped), P1.2 (shipped).
**Blocks:** S1.3.1, S1.3.2, S1.3.3 (must be rewritten in light of this
review before implementation).

## Why this stage exists

The S1.3.1 / S1.3.2 / S1.3.3 stage files were drafted **before** P1.1
shipped the grammar/IR and **before** P1.2 shipped the data model
(`KnowledgeBase`, `Fact`, `Rule`, `Pattern`, `Provenance`,
auto-vivified relations, `kb.fork`, `DerivationDAG`, …). Several
design decisions resolved during P1.1/P1.2 invalidate parts of the
P1.3 spec; several P1.3 tasks introduce types that the codebase
already has; several new design questions surfaced that the original
P1.3 doesn't acknowledge.

This stage **audits** P1.3 against the shipped reality and produces a
resolution plan. No code is written here — the output is the list of
edits and new questions that bring P1.3 into alignment.

## Verdict

P1.3 needs a **non-trivial rewrite pass** before implementation can
start. Roughly **70%** of the spec stands; **30%** needs rework.
The plan as written would (a) duplicate types already in
`src/ein_bot/kb/`, (b) implement a rule (`spatial-square` with a
Python escape hatch) that Q17 explicitly retired, and (c) leave
several new design questions un-asked.

## B. Type & module duplication caused by P1.2 actually shipping

These are the **biggest issues** — the plan introduces classes the
codebase already has.

### B.1 `Pattern` is already shipped

[`src/ein_bot/kb/pattern.py`](../../../../src/ein_bot/kb/pattern.py)
ships `Pattern` as a *structural lift*:

```python
@dataclass(frozen=True)
class Pattern:
    expr: IRNode
    variables: tuple[str, ...]
    relation_names: tuple[str, ...]
    type_names: tuple[str, ...]
    has_instance_pattern: bool
```

T1.3.1.1 proposes a *separate* AST with `Var/Atom/Hedge/And/Where`.
**Reconcile.** Either extend `kb.Pattern` with an executable AST
view, or build the matcher to consume `kb.Pattern.expr` directly
(walk the IR). The plan **must not** introduce a second `Pattern`
class — that splits the type system.

### B.2 `Rule` is already shipped

`kb.entities.Rule` (P1.2 S1.2.1) holds `name`, `params`,
`match: Pattern | None`, `assert_: Pattern | None`, `why`,
`priority`, `loc`, `_kb`. T1.3.1.3 proposes another `Rule` with
`match`, `assert_`, `why`, `priority` — same fields, same scope. **Same class.** The
matcher's job is to *consume* `kb.Rule.match`, not redefine `Rule`.
The plan should clarify the runtime path:
`kb.Rule → compiled matcher → Firing`.

### B.3 `Registry` duplicates `KnowledgeBase.rules`

KB already has `kb.rules: dict[str, Rule]` (S1.2.1). The plan's
`Registry.from_ir()` re-loads what KB already loaded.
**Drop `Registry`** as a separate concept. The saturator takes the
KB directly and reads `kb.rules`. If a compiled-matcher cache is
needed, it's a `MatcherCache` keyed by rule name — not a parallel
registry.

### B.4 Module path: `src/ein_bot/rules/` vs `src/ein_bot/inference/`

Kernel docs ([`docs/kernel/inference/`](../../../../docs/kernel/inference/))
point at `inference/`. The plan uses `rules/`. Pick **`inference/`**:
rule *definitions* live in KB (already), the *engine* is
`inference/`. Tests follow → `tests/inference/`.

## F. Scope reconsideration

Explanation-completeness on Zebra (the [Q5](../open_questions.md#q5--what-enough-means)
target) needs only the **6 rules already in zebra.ein**:
`symmetric`, `transitive`, `implies`, `square-fwd`, `square-bwd`,
`type-exclusivity`. The 10-rule taxonomy from idea 06 is broader
than Zebra requires.

Two paths:

- **(a) Narrow P1.3 acceptance** to the 6 Zebra rules + a *seam* for
  the other 4. Aggregation / variadic / domain primitives are
  deferred to **followups** (Q33 resolved 2026-05-20: not P1.4, not
  P1.7 — M1 ships only `eq` + `neq` predicates).
- **(b) Keep all 10**, but accept that 3 of them (T1.3.2.3 / .4 /
  .8) are blocked by Q26 + Q33 and must wait for followups. P1.3
  ships the 6 + the engine; the other 3 ship in a followup. (T1.3.2.9
  is re-evaluable under `neq + not` per Q33 — may rejoin the M1 core.)

The two paths produce **the same shipping order**; they differ in
*where* the 4 deferred rules are listed (P1.3 with a "blocked" tag,
or P1.4 with an "imported from P1.3 backlog" tag).

**Recommendation:** path (a) — keep P1.3's scope tight to what's
declaratively expressible under the M1 data model. Cleaner stage
boundaries, fewer cross-stage `:blocked-by` annotations. Filed as
[Q42](#g-consolidated-open-questions-for-p13) below.

**Side observation (2026-05-20, user direction) — P1.4 may
collapse into P1.3.** Constraints reduce to rules with `(not X)` in
`:assert`: since M1's KB is append-only (no fact deletion), firing
`:assert (not X)` on a branch where `X` exists or gets derived is a
**branch-failure signal** — exactly what a constraint violation
should produce. Zebra.ein's `type-exclusivity` rule already
follows this pattern. If the unification holds end-to-end, P1.4's
content shrinks to (a) a contradiction detector scanning for
`(X, (not X))` pairs and (b) the negative-fact representation —
small enough to fold into P1.3's saturator. Tracked in
[`plans/ideas.md`](../../../../plans/ideas.md) as a live entry
("P1.4 constraints — collapse into rules with `(not …)` in
`:assert`?") for revisiting after P1.3 ships.

### Detail per question

**Q29 — Pattern direction (detailed).**

When a rule fires, the engine takes its `:match` clause and hunts
the KB for fact-tuples that satisfy it. The clause exists in the
codebase as `kb.Pattern.expr: IRNode` — the raw S-expression tree
the loader produced. The question is what the *matcher* does with
that tree.

Three answers are on the table:

- **Option A — Walker.** The matcher walks `kb.Pattern.expr`
  directly, dispatching on `IRNode` subtype (`SForm` / `Atom` /
  `Var` / `KwPair`) at each step. Bindings accumulate in a
  `dict[str, str]` as the walk descends. No second representation;
  no compile step.
- **Option B — Cached executable program in `inference/`.** The
  matcher lazily compiles `kb.Pattern.expr` into an indexed program
  (a sequence of `ScanRelations` / `JoinByVar` / `GuardPredicate`
  opcodes, or a Rete-style network). The compiled form lives in
  `inference/`, cached by `Pattern.expr` identity. `kb.Pattern`
  stays untouched; matching is a private concern of `inference/`.
- **Option C — Extend `kb.Pattern`.** Add the compiled form as a
  field of `kb.Pattern` itself (e.g. `compiled: MatcherProgram |
  None`, populated on first match via `object.__setattr__` like the
  `_kb` back-pointer). One canonical home; pattern and its compiled
  form travel together.

### Worked example — `transitive` rule's `:match` clause

From [zebra.ein](../../../../examples/zebra.ein):

```lisp
(rule transitive (?rel)
  :match  (and (?rel ?a ?b)
               (?rel ?b ?c)
               :where (neq ?a ?c))
  :assert (?rel ?a ?c)
  ...)
```

After the loader runs, the `:match` clause exists in memory as
roughly:

```text
SForm( head = Atom("and"),
       args = ( SForm( head=Var("rel"), args=(Var("a"), Var("b")) ),
                SForm( head=Var("rel"), args=(Var("b"), Var("c")) ),
                KwPair( name="where",
                        value=SForm( head=Atom("neq"),
                                     args=(Var("a"), Var("c")) ) ) ) )
```

`kb.Pattern` already lifts the metadata over this expression:

```python
Pattern(
    expr=<the SForm above>,
    variables=("rel", "a", "b", "c"),
    relation_names=(),             # ?rel is a Var, not literal
    type_names=(),
    has_instance_pattern=False,
)
```

What each option *does* with this pattern at firing time:

**Option A walker** dispatches on each subterm:

```python
def match(node, bindings, kb):
    if isinstance(node, SForm) and node.head == Atom("and"):
        # Conjunction — match positional args left-to-right,
        # threading bindings; apply :where guards at the end.
        positional, guards = split_where(node)
        yield from match_seq(positional, bindings, kb, guards)
    elif isinstance(node, SForm) and isinstance(node.head, Var):
        # T2: ?rel is the head variable. Scan every fact, bind
        # ?rel to the fact's relation_name, then unify args.
        for fact in kb.facts_layer.all():
            new_b = bind(node.head.name, fact.relation_name, bindings)
            if new_b is not None and (new_b := bind_args(node.args, fact.args, new_b)):
                yield new_b
    elif isinstance(node, SForm) and isinstance(node.head, Atom):
        # T1: literal head. Use kb.facts.by_relation[node.head.name] as the candidate set.
        for fact in kb.facts.by_relation.get(node.head.name, ()):
            if (new_b := bind_args(node.args, fact.args, bindings)):
                yield new_b
```

Every saturation step re-walks the IR; the `isinstance` dispatch
tree is re-evaluated for every candidate. Conceptually small.

**Option B compiler** turns the same pattern into a join plan
*once per activator*. `?rel` is *not* a free variable here — it's a
rule parameter, bound by an activator fact like
`(transitive co-located)` (see the next section for the activator
mechanism). So the compile step substitutes the activator's
bindings into the pattern and bakes the resulting relation name
into the program:

```python
# For activator (transitive co-located) → bindings = {"rel": "co-located"}
program = JoinPlan([
    Scan(relation="co-located", arg_slots=["a", "b"]),     # ?rel resolved
    Join(relation="co-located", arg_slots=["b", "c"],      # ?rel resolved
         shared_vars={"b"}),                                # only ?b is the join var
    Guard(predicate="neq", arg_slots=("a", "c")),
])
```

At firing time, `program.run(kb)` walks the opcodes — indexed
lookups on `kb.facts.by_relation["co-located"]`, no IR inspection,
no `isinstance` dispatch. Compile cost amortised across every match
attempt for that (rule, activator) pair across all saturation cycles.

**Option C** is functionally identical to B but lives at
`kb.Pattern.compiled` instead of in a separate cache inside
`inference/`. Same opcodes, different home.

### Where T2 polymorphism shows up — correctly

The head-Var `?rel` in `(?rel ?a ?b)` is T2 relation-polymorphic,
but **`?rel` is not a free variable ranging over all KB relations**.
It is a **rule parameter** — declared in the rule's parameter list
`(rule transitive (?rel) ...)` — and it is bound by an explicit
**activator fact** in the ontology layer:

```lisp
(transitive co-located)               ;; binds ?rel = co-located
```

Without `(transitive co-located)`, the `transitive` rule sits
dormant — even if the KB has hundreds of `co-located` facts. The
activator is what authorises the rule to apply to that relation.
This is **rule application** in zebra.ein's design — rules
propagate through explicit activator facts, not through scanning.

Concretely, the engine processes a polymorphic rule as follows:

1. Find every fact whose head matches the rule's name and whose
   arity matches the rule's parameter list.
   *Example*: for `transitive (?rel)`, scan `kb.facts.by_relation["transitive"]`.
2. For each such **activator fact**, bind the rule's parameters to
   the activator's args (positional).
   *Example*: `(transitive co-located)` → bindings `{"rel": "co-located"}`.
3. Substitute those bindings into the rule's `:match` and `:assert`
   clauses — `?rel` becomes a concrete relation name.
4. Run the (now-specialized) clauses against the KB.

So `transitive` + activator `(transitive co-located)`:

| stage                | substance                                                          |
|----------------------|--------------------------------------------------------------------|
| activator fact       | `(transitive co-located)`                                          |
| bound parameters     | `?rel = co-located`                                                |
| specialized `:match` | `(and (co-located ?a ?b) (co-located ?b ?c) (neq ?a ?c))`           |
| specialized `:assert`| `(co-located ?a ?c)`                                                |

`?a`, `?b`, `?c` remain free — they're match-time variables, not
parameters. But `?rel` is **fully resolved before matching begins**.

In zebra.ein's ontology block, the activators are:

```lisp
(symmetric  co-located)
(transitive co-located)
(symmetric  next-to)
(implies    right-of next-to)
(square-fwd right-of)
(square-bwd right-of)
```

Six activators total. The engine treats each as a separate
specialization of the named rule. `transitive` ships with one
specialization (`co-located`); `square-fwd` ships with one
(`right-of`); `implies` ships with one (`right-of → next-to`); and
so on.

### What this implies for the matcher direction

With per-activator specialization, **the matcher's compile unit is
per (rule, activator-binding) pair**, not per `Pattern` alone:

- The compiled program for `transitive + (transitive co-located)`
  is distinct from a hypothetical `transitive + (transitive
  some-other-rel)` — same source rule, different baked-in relation.
- The compiled program for `square-fwd + (square-fwd right-of)`
  knows it scans `kb.facts.by_relation["right-of"]` — an indexed
  lookup, not a scan over all relations.

This **kills option A (walker)**. A walker that re-dispatches on
`?rel` per saturation step ignores the structural fact that the
activator already resolved `?rel`. It would either:

- iterate all relations and check each (wasteful — the activator
  already named the relevant ones), or
- include a special case to "look up `?rel` from the activator at
  match time" — at which point the walker is doing the compile
  step's work on every match attempt, with no caching.

**Option B with per-(rule, activator) compile** is the right
granularity. Sketch:

```python
# At rule-application time (load-time, after rules + ontology are loaded):
for rule in kb.rules.values():
    for activator in kb.facts.by_relation[rule.name]:
        bindings = dict(zip(rule.params, activator.args))   # {"rel": "co-located"}
        program = compile_pattern(rule.match.expr, bindings)
        cache[(rule.name, activator.args)] = program

# At firing time (per saturation step):
for (rule_name, args), program in cache.items():
    yield from program.run(kb)
```

The compiled program for `transitive + co-located` (Q33's `neq` is
a registry predicate, not a fact):

```python
program = JoinPlan([
    Scan(relation="co-located", arg_slots=["a", "b"]),     # ?rel baked in
    Join(relation="co-located", arg_slots=["b", "c"],      # ?rel baked in
         shared_vars={"b"}),
    Guard(predicate="neq", arg_slots=("a", "c")),
])
```

Note `relation="co-located"` is **hard-coded** in the program. The
matcher does an indexed `kb.facts.by_relation["co-located"]` lookup
— not a scan over all facts. The activator has effectively turned a
polymorphic T2 rule into a fast T1-style matcher *specialized for
co-located*.

`kb.Pattern.relation_names` and `kb.Pattern.variables` still serve
the **planner** — deciding when re-compilation is needed (e.g., a
new activator fact appears mid-saturation → compile a new entry; a
rule's match clause references a relation that doesn't yet exist in
the KB → mark the cache entry stale).

### Trade-offs

| dimension                          | A — walker                            | B — inference-side cache       | C — kb.Pattern.compiled field |
|------------------------------------|---------------------------------------|--------------------------------|-------------------------------|
| lines of code                      | smallest                              | medium                         | medium                        |
| performance per match              | `isinstance` dispatch per IR node     | one compile, then linear opcodes | same as B                  |
| kb ↔ inference coupling            | none                                  | inference → kb only            | kb → inference (inverted!)    |
| testability of kb in isolation     | full                                  | full                           | reduced (needs compiler)      |
| future-work runway (Rete, e-graph) | trivial promotion to B                | natural extension point        | requires extending kb         |
| memory cost                        | none                                  | one program per rule           | one program per Pattern node  |

The C-style coupling is the most damaging trade-off: once
`kb.Pattern` has a compiled-program field, kb's import graph
includes `inference/`, kb tests need an `inference/` fixture, and
the kernel-docs ordering (kb canonical, inference downstream) gets
inverted. Reject on architectural grounds.

A vs B was reframed by Q29's resolution (2026-05-20): with
per-activator specialization, A becomes structurally wrong (it
re-discovers the activator binding on every match attempt) — not
just slow. B is the right granularity *because* the activator
mechanism splits a polymorphic rule into N specialized matchers,
one per binding. The compile step makes this structural fact
explicit in the program shape.

### Resolved 2026-05-20 — option B with per-activator compile

User direction 2026-05-20: **option B**, with the compile unit being
**per (rule, activator-binding) pair**. The activator fact binds the
rule's parameter list *before* matching begins, so the compiler can
substitute the parameters and produce a specialized matcher with
relation names baked in.

**Why not A (walker).** A walker re-dispatches on `kb.Pattern.expr`
every saturation step. With T2 polymorphism + activators, the
walker has to either iterate every relation in the KB (wasteful —
the activator already named which) or special-case "look up `?rel`
from the activator". The second is doing the compile step's work
without caching it. Either way, A is the wrong granularity for the
activator-driven model.

**Why not C (extend `kb.Pattern`).** kb is the canonical substrate;
the compiled program is engine state. Putting compiled programs on
`kb.Pattern` inverts the dependency direction (kb → inference)
([feedback memory `graph-canonical`](../../../../docs/kernel/README.md)).

**Concrete compile model:**

- **Cache key**: `(rule.name, activator.args)`. One entry per
  (rule, activator) pair.
- **Compile trigger**: at load time, after the KB is fully loaded
  (rules + ontology). Eager compilation; zebra.ein has 6 activators
  → 6 programs. (Lazy compile is also fine; pick on simplicity.)
- **Re-compile**: if a new activator fact arrives mid-saturation
  (e.g., REASONING-layer rule producing an activator), compile a
  new cache entry. The cache is append-only for the lifetime of a
  KB instance; on `kb.fork()`, the fork inherits the cache by
  reference (programs are pure functions of the rule + bindings).
- **Cache invalidation**: not needed in M1 — facts are append-only
  per layer; programs don't depend on which facts exist, only on
  which activators do.

**Where it lives in code** (Q39 → flat `inference/`):

```
src/ein_bot/inference/
├── compile.py     # compile_pattern(pattern_expr, bindings) → JoinPlan
├── match.py       # JoinPlan, Scan, Join, Guard runtime
├── firing.py      # Firing dataclass (carries rule, activator, bindings, derived fact)
├── engine.py      # saturator driver, owns the cache
└── predicates.py  # eq + neq (Q33)
```

The compile pass is small — for M1's 6 activators × small `:match`
clauses, the entire program cache is a few dozen opcodes. Engineering
cost: medium (one compile module + a small runtime); performance:
matched directly to the activator-binding granularity that the rule
system actually uses.

**Q31 — `:why` substitution.** Zebra.ein uses `{?rel}` (ein-var
style, `?` prefix). The original P1.3 plan used `{r}` / `{p0}`
(Python format).

**Resolved 2026-05-20** (user direction): adopt the `{?var}`
notation. The `?` is *part of the reference*, not a prefix that's
stripped — `?var` is simultaneously the node name and the var name.
This keeps a single textual identity across the rule body and the
trace template:

```lisp
(rule transitive (?rel)
  :match  (and (?rel ?a ?b)
               (?rel ?b ?c)
               :where (neq ?a ?c))
  :assert (?rel ?a ?c)
  :why    "{?rel} is transitive: {?a} →{?rel}→ {?b} →{?rel}→ {?c}.")
```

Inside the `:why` string, `{?rel}` substitutes to the bound node
name (e.g. `co-located`), `{?a}` to the bound argument (e.g.
`Norwegian`), and so on. The `?` is preserved through binding —
there is no separate "bare var name" form. This is what zebra.ein
already ships.

Grammar implications for R2: the template tokeniser scans for `{` …
`}` delimited sequences whose body starts with `?` followed by a
var-name (the same lex rule as the rest of ein-lang's `?var`). Any
`{x}` *without* a leading `?` is a literal `{x}` (not substituted).

**Q32 — `:where` benefits over canonical `(and …)`.**

The candidate forms:

```lisp
;; Form A — :where as kwpair inside (and …)  (zebra.ein today)
(and (?rel ?a ?b)
     (?rel ?b ?c)
     :where (neq ?a ?c))

;; Form B — all premises positional
(and (?rel ?a ?b)
     (?rel ?b ?c)
     (neq ?a ?c))
```

Both are accepted by the current grammar (kwpairs are legal children
of any SForm; the question is purely *semantic*, not syntactic).
Four candidate benefits of Form A:

### 1. Distinguishes guards from generators

*Claim:* `:where` premises evaluate *after* positional premises
bind their vars; they filter without scanning. Positional premises
*generate* bindings.

This distinction is real and necessary for any matcher. `(neq ?a ?c)`
cannot generate bindings — there are no `(neq A B)` facts to scan.
The matcher must know to evaluate it as a guard, not a fact-scan.

**But the matcher gets this knowledge from one of two sources:**

- A **predicate registry**: the engine ships a known set of built-in
  predicates (`neq`, `lt`, `gt`, …). The matcher dispatches: "is this
  premise's head in the registry?" → "yes" → evaluate as guard. No
  need for `:where`.
- Or **`:where` itself**: syntactic special-casing. Anything inside
  `:where` is a guard; anything outside is a fact-scan.

**Verdict: benefit only without a registry.** With a registry, this
benefit dissolves entirely.

### 2. Prevents auto-vivification

*Claim:* under P1.2's open-world model
([S1.2.1](../p1.2_typed_hypergraph/s1.2.1_data_model.md)), unknown
relation heads auto-vivify into `kb.relations`. Positional
`(neq ?a ?c)` would register a phantom `neq` relation;
`:where (neq ?a ?c)` would not.

This *is* a current behaviour — loading Form B as written would
register `neq` in `kb.relations` (with `declared=False`) and the
matcher would scan `kb.facts.by_relation["neq"]`, finding zero
facts → silent no-match.

**But again**: a predicate registry tells the loader to *skip*
auto-vivification for known predicates. Same outcome without
`:where`.

**Verdict: benefit only without a registry.**

### 3. Readability for humans

*Claim:* `:where` labels a clause as a guard, making intent obvious.

Weakly true. `:where (neq ?a ?c)` reads as "where `?a` and `?c`
differ"; `(neq ?a ?c)` reads as "and `?a` ≠ `?c`". Both are clear;
the difference is small. Slightly larger win for longer guard
blocks (`:where (and (lt ?x 10) (gt ?x 5))`), but the natural
heading for THAT is still just one `:where` per rule.

**Verdict: weak; subjective; not load-bearing.**

### 4. Future-proofing for user-defined predicates

*Claim:* if later the engine grows user-defined predicates (F5
territory), `:where` is the natural extension point.

A registry handles user-defined predicates equally well (register a
Python callable or a rule-as-predicate; matcher dispatches). Neither
form blocks the other.

**Verdict: weak; both approaches scale.**

### Synthesis — the underlying design choice

All four candidate benefits reduce to **one** decision:

> Do we ship a **predicate registry**, or do we use `:where` as the
> syntactic special-case for predicates?

| dimension                  | predicate registry (drop `:where`)              | `:where` as syntactic special-case                       |
|----------------------------|--------------------------------------------------|----------------------------------------------------------|
| canonical form             | Form B (all positional)                          | Form A (`:where (…)`)                                    |
| matcher dispatch           | by lookup in registry                            | by kwpair name on the parent `(and …)`                   |
| loader / auto-vivification | suppress for registry hits                       | suppress for `:where`-children                           |
| grammar surface            | smaller (one fewer keyword for the matcher to interpret) | adds `:where` semantics on top of generic kwpair grammar |
| source of truth            | single (the registry)                            | scattered (every rule annotates its own predicates)      |
| extension to user predicates | "register a callable"                          | "put it inside `:where`"                                 |
| Datalog / CSP precedent    | matches both (Datalog has built-in predicates; no `WHERE` keyword in body) | unusual — most Lisp-y rule DSLs use positional + registry |

The registry path is **strictly more general** — anything `:where`
can express, the registry expresses too, and the registry handles
cases `:where` doesn't (e.g., dispatching on a predicate inside a
*nested* form, where `:where` doesn't reach).

### Resolved 2026-05-20 — drop `:where`, ship predicate registry

User direction 2026-05-20: drop `:where`; ship the predicate
registry. Q24's original "sugar for inline `(and …)` premise"
framing is retroactively confirmed — `:where` was a hint that the
shipped form would treat the clause specially, but the registry
makes that hint redundant.

Concrete deliverables for R2:

1. **`src/ein_bot/inference/predicates.py`** — built-in registry.
   Initial set: `neq`, `lt`, `gt`, `le`, `ge`, `not`. Each entry is
   a Python callable `(bindings, args) → bool` that the matcher
   invokes once all `args` are bound. Extensible: `register(name,
   callable)` for user-supplied predicates (F5 territory).
2. **Loader change** (`src/ein_bot/kb/from_ir.py`): when an unknown
   head appears in a `:match` body, check the registry first; only
   auto-vivify into `kb.relations` if absent from the registry.
   Prevents phantom `neq` / `lt` / … relations.
3. **Matcher change** (R2's new `inference/match.py`): walking a
   conjunction of premises, dispatch each premise by
   `(head ∈ registry)`:
   - registry hit → defer to *after* all `args` are bound;
     evaluate as guard.
   - registry miss → treat as fact-scan over
     `kb.facts.by_relation[head]` (or `kb.facts_layer.all()` if the
     head is a `Var`).
4. **Migration**: rewrite zebra.ein's `transitive` rule from
   `:where (neq ?a ?c)` → positional `(neq ?a ?c)`. One rule, one
   line.

Grammar surface stays unchanged — kwpairs remain legal SForm
children. The matcher simply ignores any leftover `:where` kwpairs
(or rejects them if R2 chooses strict mode). The grammar can
forbid `:where` *inside `:match`* in a later cleanup pass.

### M1 built-in predicate registry

```python
# src/ein_bot/inference/predicates.py — M1 contents
predicates = {
    "eq":  lambda b, args: resolve(args[0], b) == resolve(args[1], b),
    "neq": lambda b, args: resolve(args[0], b) != resolve(args[1], b),
}
```

Two binary predicates. That's the entirety of the M1 registry.

### What's not a predicate (and why)

| construct                       | role                                                                          | where it lives                  |
|---------------------------------|-------------------------------------------------------------------------------|---------------------------------|
| `(rel ?a ?b)` pattern matching  | the matcher's core operation; finds bindings against KB facts                  | matcher (`inference/match.py`)  |
| existence `(x)`                 | already pattern matching — `(foo ?a ?b)` matches iff that fact exists          | matcher                         |
| `(not (rel ?a ?b))` in `:match` | negation-as-failure — "no such fact in KB"                                     | matcher (structural special-case) |
| `(not (rel ?a ?b))` in `:assert`| assertion of a negative fact                                                   | asserter (structural)           |
| `forall ?x P(?x)`               | already `:match (P(?x))` — the clause is implicitly universally quantified    | rule semantics                  |
| `exists ?x P(?x)`               | already `:assert (P(?x))` — adding a fact IS the existential                  | rule semantics                  |
| `single ?x P(?x)`               | derivable: `(and P(?x) (not (and P(?y) (neq ?y ?x))))`                        | expressible as rule body        |

### Why minimal — user's reasoning (2026-05-20)

- **`forall` is already `:match`.** The `:match` clause is
  implicitly universally quantified over candidate bindings — every
  binding that satisfies `:match` produces a firing. No `forall`
  primitive needed.
- **`exists` is already `:assert`.** Adding a fact to the KB via
  `:assert` IS the existential claim. No `exists` primitive needed.
- **`single` reduces to `neq` + `not`.** Uniqueness ≡ "no
  competitor": `(and P(?x) (not (and P(?y) (neq ?y ?x))))`.
- **Numbers are out of scope for M1.** Zebra-class puzzles don't
  need arithmetic; the spatial structure is encoded declaratively
  via `square-fwd` / `square-bwd`
  ([Q17](../open_questions.md#q17--spatial-relation-formalisation)).
- **Sets are out of scope for M1.** Sets, domains, aggregations,
  variadic args (`?xs...`) — followup territory (cross-linked at
  [Q26](../open_questions.md#q26--compound--virtual-node-kinds-for-higher-order-rules)).

### Consequences for R3

The deferred 4 rules from §F path-(a) split into two groups:

- **Re-evaluable inside M1**: T1.3.2.9 (`forced-by-unique-position`,
  "single"). Under the user's `neq + not` framing, "only one X
  satisfies P" is `(and P(?x) (not (and P(?y) (neq ?y ?x))))`.
  R3 evaluates whether this is *useful for Zebra* and either
  promotes it back into the M1 core or keeps it deferred for
  ergonomic reasons.
- **Stay deferred — to followups, not P1.4**:
  T1.3.2.3 (variadic `?xs...` — `all-different-exclusion`),
  T1.3.2.4 (`dom` + set diff — `elimination-by-exhaustion`),
  T1.3.2.8 (`count` aggregate — `global-cardinality`). Numeric,
  set, and variadic reasoning is **not in M1's scope at all** —
  not P1.4, not P1.7. A followup phase (likely a new F-numbered
  followup adjacent to F4) owns them.

**Q40 — Hypothesis-rule premises (detailed).**

The original P1.3 plan's T1.3.2.5 sketches a rule:

```lisp
(rule hypothesis-contradiction
  :match  (and (hypothesis ?h) (contradiction-under ?h))
  :assert (not ?h)
  :why    "Assume {?h} — derives ⊥, so not {?h}.")
```

The premises `(hypothesis ?h)` and `(contradiction-under ?h)` don't
correspond to any fact in the shipped data model. In P1.2 S1.2.3,
*hypothesis status is a property of provenance, not a fact*:

```python
fact.provenance.kind == 'hypothesis'   # this fact is a branch assumption
fact.provenance.branch == 42            # of which branch
```

Q40 asks: how does the rule see this? Two options.

### Worked example — Zebra branch contradiction

To ground both options, here's a concrete contradiction in a Zebra
fork:

**Parent KB** has source fact `(co-located Norwegian House_1)`.

**Engine forks branch B42** to test the candidate
`(co-located Norwegian House_2)`:

```python
fork = kb.fork()
fork.add_fact(("co-located", ("Norwegian", "House_2")),
              layer=Layer.REASONING,
              provenance=Provenance.from_hypothesis(branch=42, loc=...))
```

**Saturation in B42** runs the 6 rules:

1. `symmetric (co-located)` fires on both:
   `(co-located House_1 Norwegian)`, `(co-located House_2 Norwegian)`.
2. `transitive (co-located)` fires on
   `(co-located Norwegian House_1)` + `(co-located Norwegian House_2)`
   (via the symmetric closure) → derives `(co-located House_1 House_2)`.
3. `type-exclusivity` already has `(not (co-located House_1 House_2))`
   in the parent (House_1, House_2 are distinct instances of `House`).

**Contradiction**: B42 now contains both `(co-located House_1
House_2)` and `(not (co-located House_1 House_2))`. The detector
([side observation on P1.4 collapse](../../../../plans/ideas.md))
finds the conflict; the engine must retract B42 and emit
`(not (co-located Norwegian House_2))` into the parent.

The question is: how does this "fire" — through the matcher, or
through an engine procedure?

### Option A — Synthetic facts, normal matcher dispatch

When forking, the engine emits *synthetic facts* into the branch:

```lisp
(hypothesis (co-located Norwegian House_2))
```

When the contradiction detector fires, it emits:

```lisp
(contradiction-under (co-located Norwegian House_2))
```

The `hypothesis-contradiction` rule's `:match` clause sees both via
the normal matcher (per Q29's compile model): `?h` binds to the
*relational node* `(co-located Norwegian House_2)`, the rule fires,
`:assert` adds `(not (co-located Norwegian House_2))` to the parent.

**This is kernel-canonical, not exotic.**
[`docs/kernel/ir/01-ein-graph/03_ein_model.md` §3](../../../../docs/kernel/ir/01-ein-graph/03_ein_model.md)
states the ein model's two flavours of node:

| flavour       | example                          | role in args                                 |
|---------------|----------------------------------|----------------------------------------------|
| **named**     | `Norwegian`, `House_1`           | atom — what `Fact.args` admits today          |
| **relational**| `(co-located Norwegian House_1)` | another node — what Option A needs admitted  |

So `(hypothesis (co-located Norwegian House_2))` is just a fact
whose first arg is a *relational node* — the kernel says this is a
node like any other. No compound-node-kind promotion is involved
(Q26 is about *virtual* / *computed* node kinds — sets, slot-
projections, top/bottom of lattices — different concern). Q33
deferred numeric / set *predicates*, not the kernel's existing
node-as-arg duality.

**What Option A actually requires** (small, M1-scoped):

1. **`Fact.args` widening** (P1.2 patch): from
   `tuple[str | int, ...]` to `tuple[str | int | Fact, ...]`.
   Fact identity stays by `(relation_name, args)`; recursive
   equality propagates via existing `Fact.__eq__`. ~5 lines + a
   handful of tests.
2. **Loader change** (`from_ir.py`): when an arg is itself an
   SForm, recursively build a `Fact` for it instead of treating
   it as an atom. ~10-20 lines.
3. **Matcher extension** (Q29 compile model): a pattern arg that
   is itself an SForm compiles to a `MatchFact(rel, arg_slots)`
   sub-program; at runtime, this matches against `Fact`-valued
   args. The unification rule generalises naturally — a `Fact`-
   valued arg unifies with a `Fact` pattern iff their
   `(relation_name, args)` tuples unify pointwise. ~30 lines.
4. **Engine emission**: on fork, emit
   `(hypothesis <hypothesis-fact>)`; on contradiction, emit
   `(contradiction-under <hypothesis-fact>)`. Both are normal
   facts with `provenance.kind='rule'` (the synthetic-fact rule is
   the engine's fork / contradict procedure).

Total effort: roughly one day of code; well within P1.3 scope.

**Trace clarity**: the synthetic facts appear in the KB and show
up in the trace as ordinary `(hypothesis …)` / `(contradiction-under …)`
firings. The user-facing explanation reads:

> *Tried `(co-located Norwegian House_2)` as a hypothesis. In the
> branch, derived `(co-located House_1 House_2)` and
> `(not (co-located House_1 House_2))` — a contradiction. By
> `hypothesis-contradiction`: `(not (co-located Norwegian House_2))`
> in the parent.*

**Verdict for A**: viable in M1 with a small data-model widening +
matcher extension. Aligned with the kernel principle that every
part of the KB is a node ([feedback memory `graph-canonical`](../../../../docs/kernel/README.md)).

### Option B — Engine procedure, no matcher dispatch

The `hypothesis-contradiction` rule is **not a normal matched
rule**. It's an engine procedure that runs when the contradiction
detector signals. Pseudo-code in `inference/engine.py`:

```python
def on_contradiction(fork: KnowledgeBase, conflict_pair: tuple[FactId, FactId]):
    # 1. Find the hypothesis fact that authorised this branch.
    h_fact = next(f for f in fork.facts_layer.reasoning()
                  if f.provenance.kind == 'hypothesis'
                  and f.provenance.branch == fork.branch_id)

    # 2. Emit the negation into the parent.
    parent = fork.parent
    parent.add_fact(
        ("not", (h_fact.relation_name, h_fact.args)),
        layer=Layer.REASONING,
        provenance=Provenance.from_rule(
            rule='hypothesis-contradiction',
            premises_raw=[
                (h_fact.relation_name, h_fact.args),       # the hypothesis
                conflict_pair[0],                           # X
                conflict_pair[1],                           # (not X)
            ],
            bindings=(('h', f'{h_fact.relation_name}({...})'),),
            loc=None,
        ),
    )

    # 3. Mark all branch facts as rejected.
    for f in fork.facts_layer.reasoning():
        if f.provenance.branch == fork.branch_id:
            f.provenance = Provenance.rejected(branch=fork.branch_id, loc=f.provenance.loc)
```

The `kb.Rule` named `hypothesis-contradiction` can still exist (for
trace rendering + documentation), but its `:match`/`:assert` are
**not consumed by the matcher**. The engine emits the firing
directly. The user-facing trace shows:

> *Step N. Tried assumption `(co-located Norwegian House_2)`.
> Branch B42 derived `(co-located House_1 House_2)` and
> `(not (co-located House_1 House_2))` — contradiction. By
> hypothesis-contradiction: `(not (co-located Norwegian House_2))`
> in the parent.*

Identical user experience to A; cleaner machinery.

### Bands

| band     | priority | what fires here                                                    |
|----------|---------:|--------------------------------------------------------------------|
| propagate | 100      | cheap, structural propagation — symmetric, inverse, implies        |
| derive    | 200      | composition / join — transitive, square-fwd, square-bwd            |
| eliminate | 300      | exclusion / narrowing — type-exclusivity, all-different-exclusion  |
| hypothesis | 900     | reserved for hypothesis-contradiction (P1.5; not in normal saturation) |

The banded scale leaves room *inside* each band (e.g., 110 / 120 /
130) for fine-grained ordering of multiple rules in the same band —
useful if a specific puzzle needs to prefer one propagator over
another. The exact within-band numbering is open and per-puzzle;
R4 documents the convention but doesn't bake in within-band values.

### Per-rule placement

Priority lives in the rule definition, not a saturator config:

```lisp
(rule transitive (?rel)
  ...
  :priority 200)               ;; derive band
```

Rationale: the rule's *behaviour band* is a property of the rule
itself, not of the engine that runs it. A rule moved between
puzzles keeps its band. Per-rule placement also keeps the rule
definition self-documenting — readers see the priority next to
`:match` / `:assert` without consulting an external table.

### Ordering — lower fires first

A queue ordered by `(priority, FIFO)` pops 100 before 200 before
300. Propagation runs first; elimination runs last; hypothesis is
reserved for P1.5.

## R10 — Flatten `relation` / `a-priori` args (no inner group when no body follows)

Patch to already-shipped P1.1 grammar. Companion to R9: where R9
widens P1.2's data model, R10 tightens P1.1's surface syntax to
remove a noise-only paren-pair.

### Origin

Surfaced 2026-05-20 during the kernel ein-model review
([`docs/kernel/ir/01-ein-graph/03_ein_model.md` §7.2](../../../docs/kernel/ir/01-ein-graph/03_ein_model.md#72-relation-declaration-shape--body-form-and-args-grouping)).
Under [Q27](../open_questions.md#q27--relation-body-form)'s resolution
(M1 ships form (b) — properties as separate facts, not bundled
inside the relation), the inner `(…)` group around relation args has
no purpose: the trailing-body block it was disambiguating from
doesn't exist in form (b). The wrapper is pure noise.

### Direction

**Strict-flat.** The grammar accepts only `(relation R T1 T2 …)`;
the wrapped form `(relation R (T1 T2))` is rejected at parse time.

Rationale:

- Matches the M1 design rule-of-thumb "syntax should be as
  protective as possible" ([`m1_core_graph_reasoning/README.md`](../README.md)
  §"Design rules-of-thumb").
- Pre-release codebase — migration cost is one-line edits in two
  example files + a handful of doc files.
- Form (a)'s wrapper survives in its own grammar production when
  the body-form sugar is eventually introduced (Q27, deferred).
  The two forms stay syntactically distinguishable.

The lenient alternative (accept both, dump flat) was considered and
declined — keeping two valid spellings indefinitely is grammar noise
that the loader and matcher would have to ignore on every parse.

### Edits

- **`src/ein_bot/ir/grammar.lark`**: flatten the two affected
  productions.

  Token disambiguation: `R` and the `Ti` args lex as SYMBOL; any

## R11 — Slot-indexed inherit-rel pattern (one rule per argument slot)

Documentation-only revision; no M1 code impact (Zebra doesn't use
`inherit-rel`). Captures a design pattern surfaced in the same
kernel review as R10.

### Origin

Surfaced 2026-05-20 in
[`docs/kernel/ir/01-ein-graph/04_jack_drinks_coffee.md`](../../../docs/kernel/ir/01-ein-graph/04_jack_drinks_coffee.md)
— the worked-example file's `inherit-rel` rule covered only the
LHS (arg #1) case of is-a propagation. The "any × any" closure
(*any human can drink any drink*) was implicit in the prose but
missing from the rule definition.

### Pattern

For an N-ary relation `R(T1, T2, …, TN)`, is-a inheritance over the
relation requires **N independent rules** — one per argument slot.
Each rule matches an `is-a ?inst ?Ti` edge and substitutes `?inst`
for `?Ti` in slot `i`:

```lisp
(rule inherit-rel-slot-i (?r)
  :match  (and (?r ?a1 … ?Ti … ?aN)
               (is-a ?inst ?Ti))
  :assert (?r ?a1 … ?inst … ?aN))
```

The full *"any × any × … × any"* closure emerges from **saturation
chaining** — no separate "cartesian" rule is needed. Applying
slot-`i` then slot-`j` (for any i ≠ j) over the cartesian product of
is-a edges produces the full closure as the chained fixpoint.

For the binary case in `04_jack_drinks_coffee.md`:

- `inherit-rel-lhs`: propagates over arg #1.
- `inherit-rel-rhs`: propagates over arg #2.
- Closure `(can-drink Jack Coffee)` emerges from chaining either
  order (lhs → rhs or rhs → lhs).

### Why one rule per slot, not a single all-slots rule

A single rule with a variadic match (`(?r ?args…)` plus a slot-index
parameter) would require:

- Variadic args `?xs…` — deferred to followups under
  [Q33](../open_questions.md#q33--predicate-primitives--minimal-set-for-m1).
- A way to refer to "slot k of a fact" — a compound node kind, parked
  at [Q26](../open_questions.md#q26--compound--virtual-node-kinds-for-higher-order-rules).

N slot-indexed rules sidestep both. The cost is N rule declarations
per inherited relation; for the binary `can-drink` case this is N=2.
For arity-3 relations (e.g. `between`) it grows to N=3, which is
still manageable inline.

### Why this is not an M1-core rule

`inherit-rel` doesn't ship in `examples/zebra.ein` — Zebra's
inheritance pattern goes through `co-located` + `square-fwd/bwd`
([Q17](../open_questions.md#q17--spatial-relation-formalisation)),
not through is-a-propagation. The slot-indexed pattern is
**pedagogical** for the kernel docs (worked example) and a
**template** for future puzzles whose rule libraries do use
is-a-driven inheritance.

The followup home for this generalisation is
[F4 Q36 — relation inheritance / rule polymorphism](../../followups/f4_cross_cutting.md);
when a future puzzle authors a slot-indexed `inherit-rel` family,
F4 Q36 covers the design-space audit.
