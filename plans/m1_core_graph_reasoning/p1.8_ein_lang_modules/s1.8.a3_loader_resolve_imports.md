# S1.8.A3 — Loader: resolve imports

## Context

The grammar (S1.8.A2) produces `Import` nodes; the loader has
to actually *resolve* them: open the imported file, parse it,
recursively resolve its imports, merge symbols into the
current KB, detect conflicts.

Per S1.8.A1, the exact resolution + merge semantics depends on
the file-vs-symbol-level + namespace-scoping + conflict-policy
choices.

## Tasks

### Task T1.8.A3.1 — Path resolver

A `_resolve_import(path: str, *, importing_file: Path) → Path`
function:

- **a.** Apply the path-resolution rule from S1.8.A1.4 (repo-
  relative / file-relative / env).
- **b.** Error path: import not found, circular import.
- **c.** Cache: each path resolved once per `solve()` call.

### Task T1.8.A3.2 — Recursive loader

`load_kb_from_path(path)` becomes recursive:

- **a.** Parse the file.
- **b.** Process `Import` nodes first, before the rest of the
  top-level forms.
- **c.** For each import: resolve → load recursively → merge
  symbols.

Cycle detection: maintain a `currently_loading` set; reject
re-entry.

### Task T1.8.A3.3 — Merge semantics

Per S1.8.A1 namespace scoping decision:

- Flat namespace: all symbols go into the same registries.
- Module-scoped: prefix names with module path before
  insertion.
- Aliased: use the alias prefix.

Conflict detection per S1.8.A1.3 policy.

### Task T1.8.A3.4 — Diagnostics

- **a.** Clear error messages for: import not found, circular
  import, conflict-without-override.
- **b.** Each error carries the `Loc` of the offending
  `(import …)` form.

### Task T1.8.A3.5 — Tests

- **a.** Single-import smoke: file A imports file B; symbols
  from B are visible in A's KB.
- **b.** Transitive: A → B → C.
- **c.** Circular: A → B → A. Reject.
- **d.** Conflict: A defines `foo`, imports B which also
  defines `foo`. Per the policy, error / warn / use one.
- **e.** Symbol-level filter (if (b)): only listed symbols
  arrive.

## Acceptance

- Two-file project loads end-to-end.
- Errors are clear + located.
- Existing zebra2.ein still loads (no imports — should be
  backward-compatible).
- Cross-file tests pass.

## Open questions

- **Q-S1.8.A3.A — Side effects.** If imported file has `(facts
  …)` blocks, do those facts get merged into the importing
  KB? Usually yes for a library; usually no if the imported
  file is conceptually a "module of rules only". Decide what
  the convention is.
- **Q-S1.8.A3.B — Re-export.** If A imports B and C imports
  A, does C see B's symbols? Default: yes (transitively
  visible) unless a re-export annotation says otherwise.
