# S1.8.A10 — Type / domain matching

**Phase:** P1.8 (improvements), Theme A.
**Estimate:** ~~2-3 days, moderate engine work~~ → **zero engine change** — both
halves turned out to be stdlib ein.
**Status:** **DONE — both halves are STDLIB; the engine was never touched.** The
relation-arg typecheck (T10.1) was tried as a kernel check and **reverted** (it
reintroduced is-a coupling — see
[§ The kernel-purity finding](#the-kernel-purity-finding-2026-06-04)); it already
exists as `std.elim`'s `typecheck-arg-{0,1}`. The rule-domain check (T10.2) —
once thought to need engine work — ships as pure ein in `std.algebra` +
`std.typing` (2026-06-04): the signature *is* a matchable `(relation R A B)` fact
and an activator-bound relation head already compiles to a concrete scan, so
neither a `(domain …)` DSL nor a matcher change was needed. See
[§ T10.2 resolution](#t102-resolution-pure-stdlib-2026-06-04).
**Depends on:** stdlib content shipped (A6-A9) so the demand
for type-checking is concrete.
**Blocks:** nothing on M1's critical path; informs P1.20 docs
(the type/domain semantics needs documenting).

## The kernel-purity finding (2026-06-04)

A first cut shipped T10.1 as a **load-time kernel check** (`from_ir`,
flag-gated) that read `(is-a arg SigType)` facts to validate declared-relation
args. The user caught that this **reintroduces is-a special-casing in the
kernel** — exactly what
[S1.7.23](../p1.7_bootstrapping_zebra/s1.7.23_retire_kernel_type_system.md)
removed ("kernel reasoning is-a-free; a puzzle's inheritance rules ARE its type
system, in user space"). It was **reverted** (the flag + the loader check + its
test).

The principle it violated: **type-checking inherently needs the instance-type
relation (`is-a`), which is a user-space convention.** The kernel may consult a
relation's *arity* (signature length, is-a-free) but not its *types*. So
relation-arg typechecking belongs in stdlib ein — and **it already exists
there**: `std.elim`'s `typecheck-arg-{0,1}` do precisely this check, in ein,
where naming the hierarchy relation is fine. (Reparameterized 2026-06-04: those
rules now take the instance-type relation as a `?isa` *parameter* rather than a
hardcoded `is-a` literal — so even the stdlib *bodies* are is-a-free, and a
puzzle passes `is-a` or the transitive `is-a*` per its hierarchy. Same move the
converse check uses.) T10.1 as a kernel feature is therefore **rejected on
purity grounds + redundant**. (The only remaining kernel `is-a` literals are in
`kb/render.py` — DOT presentation of inheritance edges, a display concern.)

## T10.2 resolution: pure stdlib (2026-06-04)

The earlier "genuinely deferred — needs engine work" verdict rested on three
premises, **all false** (caught by the user):

1. *"It's about the activator binding, not facts."* The signature **is** a fact.
   `(relation R A B)` declarations live in `kb._facts_by_relation` and rules
   already match them: `functional-negative` reads one signature,
   `disjunctive-prune-{fwd,bwd}` join *two* `(relation …)` decls on shared
   type-vars (zebra2-hints.ein). So a converse check is an ordinary rule
   activated by the `(converse …)` fact.
2. *"It needs a per-rule `(domain …)` predicate DSL."* No — it needs `and` /
   `neq` / `absent` over the `(relation …)` facts, all of which ein already has.
3. *"It needs the matcher's no-scan-bound-relation-head limitation lifted."* No.
   That limit (`compile._compile_relation`'s `else: return []`) only bites a
   *truly free* relation head. A head that's a **rule parameter** bound by the
   activator is baked into a concrete `Scan` at compile time (compile.py:211),
   exactly as `co-located` / `adjacent-via` already do. The hierarchy relation
   rides in as a parameter `?isR*`, so it scans fine — and the rule body names
   **no `is-a` literal**, making the dependency on the is-a* closure explicit.

**What shipped** — two files, per the user's "(b) and (c) in different stdlib
files":

- **`std.algebra` (variant b)** — `converse-illtyped-{dom,ran} (?R1 ?R2 ?isR*)`:
  read both signatures off the `(relation …)` facts, `(false)` when R2's
  signature isn't R1's reverse, up to the subtype relation `?isR*`. A `(neq …)`
  guard absorbs the identical-type case, so an *irreflexive* `is-a*` still passes
  exact-type converses (right-of/left-of) without a self-loop.
- **`std.typing` (variant c)** — `(type-hierarchy is-a*)` declared once drives
  `type-hierarchy-converse`, a NON-generic rule that derives both check
  activators for every `(converse …)` pair (relying on A9 reflective). Plus
  `(reflexive R)` → `derive-reflexive` + `reflexive-{dom,cod}`, the
  reflexive-closure rule (a *generic* rule can't multi-assert — S1.8.A11 — so
  the single knob fans out non-generically, the `bijective-properties` idiom).

Pinned by `tests/inference/test_converse_typecheck.py` (incompatible→⊥, exact-ok,
subtype-ok-with-hierarchy, subtype-rejected-without-it, the one-knob driver,
reflexive both-positions). The kernel stays is-a-free; `Relation.signature` is
consulted in **user space**, exactly where [S1.7.23](../p1.7_bootstrapping_zebra/s1.7.23_retire_kernel_type_system.md)
put types. The Tasks / Acceptance sections below describe the **rejected**
kernel-flag design and are retained only as the historical record.

## Context

Per [P1.8 README Theme A § Engine support](README.md#standard-library):

`match._bind_args` never consults `Relation.signature`; nothing
enforces declared signatures — no relation-arg check *and* no
rule-domain check. A generic stdlib (rules parametrised over
relations) wants both, to catch:

- `(imply R1 R2)` where R1's signature is `(House Color)` and
  R2's is `(Person Pet)` — incompatible.
- `converse R1 R2` where the arg orderings can't be inverted.
- Bijective shorthand asserts that don't satisfy the bijective
  relation's argument types.

M1 is deliberately untyped here (open-world); the stdlib is
where typing earns its keep.

## Tasks

### Task T1.8.A10.1 — Relation-arg type-check

- **a.** In `_bind_args` (or a sibling) consult
  `Relation.signature`. If a binding violates the signature,
  reject the firing.
- **b.** Config flag `:enable-relation-arg-typecheck` (default
  **off** for M1 backward-compat; the stdlib can opt-in
  per-rule).
- **c.** Tests: facts that violate signature; with flag on,
  the firing is rejected.

### Task T1.8.A10.2 — Rule-domain check

- **a.** For rules parametrised over relations (like
  `(rule converse (?R1 ?R2) …)`), check that the relations
  bound to `?R1` and `?R2` have compatible signatures (e.g.,
  `converse` needs `R1` and `R2` with arg orderings that can
  be swapped).
- **b.** Per-rule annotation: a `(domain …)` clause on the
  rule definition listing the relation-domain constraint.
- **c.** Tests: `(converse house-color pet-dog)` (incompatible
  sigs) rejects with a clear error.

### Task T1.8.A10.3 — Diagnostics

- Clear error messages: "rule X requires relations with
  compatible signatures, got R1=(House,Color) and
  R2=(Person,Pet)".
- Location info: point at the activator fact's `Loc`.

### Task T1.8.A10.4 — Documentation

- **a.** Document the per-rule `(domain …)` annotation syntax
  in [S1.20.B kernel API ref](../p1.20_kernel_docs/s1.20.b_kernel_api_reference.md).
- **b.** Document the flag in
  [`docs/kernel/inference/`](../../../docs/kernel/inference/).

## Open questions

- **Q-S1.8.A10.A — Type narrowing in patterns.** Does the
  match propagate type info from premise to conclusion? E.g.,
  if `(?R ?a ?b)` matched on `house-color`, can `?a` be
  inferred as House? Borderline interpreter territory; defer
  to F4 Q35.
- **Q-S1.8.A10.B — Sub-typing.** zebra2's `is-a*` closure
  creates a sub-typing hierarchy. Does the type check respect
  it? Probably yes for the user-facing concept; implementation
  may simplify by checking only declared types.
