# S1.8.A10 — Type / domain matching

**Phase:** P1.8 (improvements), Theme A.
**Estimate:** 2-3 days — moderate engine work.
**Status:** parked; opt-in. M1 is deliberately open-world
untyped; this stage is the typed counterpart for the stdlib's
generic rules.
**Depends on:** stdlib content shipped (A6-A9) so the demand
for type-checking is concrete.
**Blocks:** nothing on M1's critical path; informs P1.10 docs
(the type/domain semantics needs documenting).

## Context

Per [P1.8 README Theme A § Engine support](README.md#standard-library):

`match._bind_args` never consults `Relation.signature`; nothing
enforces declared signatures — no relation-arg check *and* no
rule-domain check. A generic stdlib (rules parametrised over
relations) wants both, to catch:

- `(imply R1 R2)` where R1's signature is `(House Color)` and
  R2's is `(Person Pet)` — incompatible.
- `converse R1 R2` where the arg orderings can't be inverted.
- Bijective shorthand asserts that don't satisfy the bijective
  relation's argument types.

M1 is deliberately untyped here (open-world); the stdlib is
where typing earns its keep.

## Tasks

### Task T1.8.A10.1 — Relation-arg type-check

- **a.** In `_bind_args` (or a sibling) consult
  `Relation.signature`. If a binding violates the signature,
  reject the firing.
- **b.** Config flag `:enable-relation-arg-typecheck` (default
  **off** for M1 backward-compat; the stdlib can opt-in
  per-rule).
- **c.** Tests: facts that violate signature; with flag on,
  the firing is rejected.

### Task T1.8.A10.2 — Rule-domain check

- **a.** For rules parametrised over relations (like
  `(rule converse (?R1 ?R2) …)`), check that the relations
  bound to `?R1` and `?R2` have compatible signatures (e.g.,
  `converse` needs `R1` and `R2` with arg orderings that can
  be swapped).
- **b.** Per-rule annotation: a `(domain …)` clause on the
  rule definition listing the relation-domain constraint.
- **c.** Tests: `(converse house-color pet-dog)` (incompatible
  sigs) rejects with a clear error.

### Task T1.8.A10.3 — Diagnostics

- Clear error messages: "rule X requires relations with
  compatible signatures, got R1=(House,Color) and
  R2=(Person,Pet)".
- Location info: point at the activator fact's `Loc`.

### Task T1.8.A10.4 — Documentation

- **a.** Document the per-rule `(domain …)` annotation syntax
  in [S1.10.B kernel API ref](../p1.10_kernel_docs/s1.10.b_kernel_api_reference.md).
- **b.** Document the flag in
  [`docs/kernel/inference/`](../../../docs/kernel/inference/).

## Acceptance

- Type-checking + domain-checking flags work; tests cover
  positive + negative cases.
- M1 acceptance unchanged with flags off.
- Stdlib generic rules can opt-in.
- Diagnostics are clear + located.

## Open questions

- **Q-S1.8.A10.A — Type narrowing in patterns.** Does the
  match propagate type info from premise to conclusion? E.g.,
  if `(?R ?a ?b)` matched on `house-color`, can `?a` be
  inferred as House? Borderline interpreter territory; defer
  to F4 Q35.
- **Q-S1.8.A10.B — Sub-typing.** zebra2's `is-a*` closure
  creates a sub-typing hierarchy. Does the type check respect
  it? Probably yes for the user-facing concept; implementation
  may simplify by checking only declared types.
