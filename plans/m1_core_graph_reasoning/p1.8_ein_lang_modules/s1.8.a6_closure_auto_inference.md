# S1.8.A6 — Closure auto-inference

**Phase:** P1.8 (improvements), Theme A.
**Estimate:** 2-3 days (the original S1.5.5 estimate).
**Status:** **DONE 2026-06-04** — shipped as `std.closure`, **import-gated**
(not config-gated); soundness caveat sharpened. See [§ As built](#as-built-2026-06-04).

## As built (2026-06-04)

`infer-closure` ships in `ein_bot/stdlib/closure.ein` (`std.closure`):
parameter-less, `:match (and (functional ?R) (total ?R)) :assert (closed ?R)`.
Opt-in by `(import std.closure :symbols (infer-closure))`; hypgen reads the
derived `(closed R)` via `_is_closed` with no extra wiring.

**Two deviations from the original plan, both grounded in code:**

1. **Import-gated, not config-gated (T1.8.A6.2 changed).** The doc predates
   imports (A1–A3). Gating a *rule's* firing on a config flag would mean the
   kernel hardcoding a stdlib rule name in the saturator — against the
   purity arc. Now the **import is the opt-in** (default-off = don't import).
   The pre-existing `SolverConfig.enable_auto_closure` was a declared-but-inert
   placeholder (never consumed; no `.ein` set it) whose docstring claimed to
   gate this very rule — **retired** to avoid an actively-misleading no-op flag.

2. **Soundness caveat sharpened (T1.8.A6.4).** `functional ∧ total` is an
   *operational* witness that is sound **only when R's extension is determined
   by saturation/elimination**. For a relation that needs hypothesis
   **branching** — the bijective attribute relations of the Zebra puzzle —
   closing it prunes the very branches the search relies on, so the puzzle
   fails to solve (a **completeness** hazard, not a wrong-answer one). Hence it
   is opt-in and **must NOT be imported into the zebra family**; the strong
   (semantic) "extension can't grow" sense still needs the count predicate the
   eq/neq matcher lacks (out of M1 — A10 / F1b). This is now a boxed warning in
   `closure.ein` and the `std.closure` README row.

Tests: `tests/inference/test_infer_closure.py` — functional+total → closed;
functional-only / total-only → not closed; not-imported → not closed;
parameter-less shape; the hypgen skip hand-off; local-redefinition dup error.
Full suite green; nothing closed by default (zebra2 untouched). The original
framing follows.
**Depends on:** [S1.5.4 T1.5.4.1](../p1.5_hypothesis_loop/s1.5.4_hypgen_improvements.md#task-t1541--relation-closure-declarative-closed-r-activator)
ships (manual `(closed R)` activator); [S1.5.8](../p1.5_hypothesis_loop/s1.5.8_totality_domain_elimination.md)
ships (`(total R)` + domain-elimination) so the witness
`functional ∧ total ⇒ closed` is operationally sound.
**Blocks:** nothing on M1's critical path; informs stdlib
reference [S1.20.C](../p1.20_kernel_docs/s1.20.c_stdlib_api_reference.md).

## Context

The deferred S1.5.5 work. Two corrections from the 2026-05-22
review (recorded in [P1.8 README Theme A § Standard library](README.md#standard-library)):

- **Parameter-less.** `(rule infer-closure () :match … :assert
  (closed ?R))` — not `(?R)`. The latter would be a param
  list; the engine then expects activator facts and never
  fires. `?R` is a free match var, like `hypothesis-
  contradiction ()`.
- **`functional ⇒ closed` is too weak.** Use **`functional ∧
  total ⇒ closed`** as the sound operational witness.

Two senses of "closed":

- **Strong / semantic**: extension can't grow. Needs cardinality
  predicate the eq/neq-only matcher lacks.
- **Operational**: hypgen needn't branch on `R`. M1's `(closed
  R)` is operational; `functional ∧ total` is the sound
  witness post-S1.5.8.

## Tasks

### Task T1.8.A6.1 — Ship the rule

In the stdlib (per S1.8.A4 layout):

```lisp
(rule infer-closure ()
  :match (and (functional ?R) (total ?R))
  :assert (closed ?R)
  :why "{?R} is functional + total, so its extension is fixed.")
```

Parameter-less. Free `?R` match var.

### Task T1.8.A6.2 — Config flag

- **a.** Add `:enable-auto-closure` to `SolverConfig` (S1.5.4
  T1.5.4.4). Default **off** — explicit `(closed R)` still
  required unless the user opts in.
- **b.** Gate the rule's activation on the flag.

### Task T1.8.A6.3 — Tests

- **a.** Puzzle with `(functional R)` + `(total R)` + flag
  on: `(closed R)` derived; hypgen skips R.
- **b.** Same puzzle with flag off: `(closed R)` not derived;
  hypgen branches on R.
- **c.** Puzzle with `(functional R)` only: flag on or off,
  `(closed R)` not derived (totality required).

### Task T1.8.A6.4 — Document the soundness argument

The "two senses of closed" discussion lives in
[`docs/kernel/inference/`](../../../docs/kernel/inference/) (or
[S1.20.D](../p1.20_kernel_docs/s1.20.d_inference_engine_docs.md)
once it ships): why `functional ∧ total ⇒ closed` is sound for
the operational sense, and what would be needed for the
semantic sense (cardinality predicate).
