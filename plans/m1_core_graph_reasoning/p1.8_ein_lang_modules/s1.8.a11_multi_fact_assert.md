# S1.8.A11 — Multiple facts asserted by a single rule

**Phase:** P1.8 (ein-lang / stdlib), Theme A.
**Status:** **DONE 2026-06-04** — shipped via a **load-time split** (a third
option, neither of the two below): a top-level `(and c1 … cn)` `:assert` is
lowered to one `Rule` per conjunct, named `<rule>__and<j>`, exactly mirroring
the proven `(or …)`-in-`:match` → `__or<i>` split (`kb.from_ir`,
`_assert_conjuncts`). The two splits compose as a cross-product
(`__or<i>__and<j>`).

> **AS BUILT (2026-06-04).** Chosen over the doc's options 1/2 because
> `JoinPlan.assert_template` is read across **six** engine modules (compiler
> introspection, `lookahead`, `hrule`, `verdict`, the saturator redundant-check,
> `firing`), so changing it to a tuple ripples through the engine core — high
> risk for an M1 ergonomic feature. The split reuses an existing, proven
> mechanism with **zero engine churn**; each conjunct fires as an ordinary
> single-assert rule sharing the parent `:match` / `:why`.
>
> **Generic-rule limitation (made a loud load error).** The split renames the
> rule to `<name>__and<j>`, whose generic-rule activator would have to be
> `(<name>__and<j> …)` — but a puzzle writes `(<name> …)`. So a multi-fact
> assert on a *generic* rule (non-empty params) can't activate; the loader now
> **rejects it with a clear error** (rather than splitting into rules that
> silently never fire). This is the same latent limitation `(or …)`-in-`:match`
> has for generic rules (never hit in the corpus). The doc's motivating example
> is generic — lifting the limitation needs the firing-level representation
> below, deferred.
>
> **Trace-grouping not delivered.** The conclusions surface as `__and<j>`
> sibling firings, exactly as `(or …)` surfaces as `__or<i>` siblings — NOT one
> rule node fanning out to N facts. Grouping them into a single rule
> *application* is S1.6.2's concern (and is the natural place for the
> `Firing.derived: tuple` / group-id representation if/when it's wanted).
>
> Tests: `tests/inference/test_multi_assert.py` (split, three conjuncts,
> positive+negative conclusions, or×and cross-product, generic rejection,
> generic-single-assert still OK). Full suite 1139 green; no regression
> (single-assert rules keep their bare name).

## Why

Today a rule asserts exactly one fact per match:

- `Rule.assert_: Pattern | None` — a single conclusion pattern
  (`kb/entities.py`).
- `JoinPlan.assert_template` — a single template (`inference/compile.py`).
- `Firing.derived: Fact` — a single derived fact (`inference/firing.py`);
  `fire()` produces one `Firing` per binding.

So a rule that should conclude several facts at once is either split
into N rules (duplicated `:match`) or shows up as N independent firings
that merely share a rule name. The trace renderer (S1.6.2) then has to
reconstruct "these N firings came from one rule application"
heuristically.

## Surface syntax — SKETCH (confirm)

> The exact surface form is **to be confirmed** — recorded here as a
> placeholder from the design discussion, not a settled grammar. Replace
> with the intended ein-lang.

```ein
(rule place-and-exclude (?rel)
  :match  (and (?rel ?a ?x) (neq ?x ?y))
  :assert (and (?rel ?a ?x)
               (not (?rel ?a ?y)))      ; two conclusions, one firing
  :why    "{?a} is {?x}, so it is not {?y}")
```

`:assert (and C1 C2 …)` lists several conclusion patterns; one match
fires all of them as a single rule application.

## Representation — two options

1. **One firing, many derived.** `Firing.derived: tuple[Fact, ...]`;
   `assert_` becomes a tuple of patterns (or an `and`-pattern expanded
   at compile time). Cleanest provenance grouping (one application = one
   node fanning out to N facts; S1.6.2 renders it directly). Touches
   every `Firing.derived` consumer (detector, dumper, trace, tests).
2. **N firings, shared group-id.** Keep `derived: Fact`; add a
   `group_id` / `application_id` so consumers can cluster the N firings
   under one rule application. Lower-churn; the renderer groups by id.

## Open questions

- Which representation (1 vs 2)? — 1 is cleaner, 2 is lower-churn.
- Do all asserted facts share the **same** premises/provenance, or can
  a later conclusion depend on an earlier one within the same firing?
- Does saturation dedup / the contradiction detector need changes when
  one firing introduces several facts (especially a positive + its
  `not`)? **P1.7c interaction:** derived facts now get `Layer.REASONING`
  via their `:rule` provenance (`_layer_of`), and the detector pairs `g` /
  `(not g)` as a contradiction *iff* the existing fact is `REASONING` (per
  [S1.7c.1](../p1.7c_block_head_removal/s1.7c.1_layer_attribution_decision.md)'s
  finding, `lookahead.py:139,145`) — so a single firing asserting both `g`
  and `(not g)` would self-contradict at the REASONING layer. Confirm that's
  intended (it likely is — such a rule *is* inconsistent).
- Interaction with the `imply` family (S1.8.A7) and totality /
  domain-elimination (S1.8.A8), which are single-assert today.

## Acceptance (draft)

- A rule with `:assert (and …)` parses, compiles, and fires all
  conclusions from one match.
- The trace (S1.6.2) renders the multi-fact firing as a single rule
  application (one rule node → N derived facts), not N look-alike
  firings.
- zebra2 continues to solve (no regression for single-assert rules).
