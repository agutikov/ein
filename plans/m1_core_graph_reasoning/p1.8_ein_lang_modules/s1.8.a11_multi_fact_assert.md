# S1.8.A11 — Multiple facts asserted by a single rule

**Phase:** P1.8 (ein-lang / stdlib), Theme A.
**Status:** design stub — captured 2026-05-29 from the "where did we
plan multiple facts fired from one rule?" question. The answer was
*nowhere*: the current model is **one fact per firing**. This stage
records the extension.
**Depends on:** P1.3 rule DSL + compiler (`compile_rule`), the `Firing`
model.
**Relates to:** [S1.5.9 ein-lang macros](s1.5.9_ein_lang_macros.md)
(parser sugar — distinct from this); [P1.6 S1.6.2](../p1.6_rendering_and_trace/s1.6.2_dot_state_transitions.md)
(how a multi-fact firing renders in the trace).

## Why

Today a rule asserts exactly one fact per match:

- `Rule.assert_: Pattern | None` — a single conclusion pattern
  (`kb/entities.py`).
- `JoinPlan.assert_template` — a single template (`inference/compile.py`).
- `Firing.derived: Fact` — a single derived fact (`inference/firing.py`);
  `fire()` produces one `Firing` per binding.

So a rule that should conclude several facts at once is either split
into N rules (duplicated `:match`) or shows up as N independent firings
that merely share a rule name. The trace renderer (S1.6.2) then has to
reconstruct "these N firings came from one rule application"
heuristically.

## Surface syntax — SKETCH (confirm)

> The exact surface form is **to be confirmed** — recorded here as a
> placeholder from the design discussion, not a settled grammar. Replace
> with the intended ein-lang.

```ein
(rule place-and-exclude (?rel)
  :match  (and (?rel ?a ?x) (neq ?x ?y))
  :assert (and (?rel ?a ?x)
               (not (?rel ?a ?y)))      ; two conclusions, one firing
  :why    "{?a} is {?x}, so it is not {?y}")
```

`:assert (and C1 C2 …)` lists several conclusion patterns; one match
fires all of them as a single rule application.

## Representation — two options

1. **One firing, many derived.** `Firing.derived: tuple[Fact, ...]`;
   `assert_` becomes a tuple of patterns (or an `and`-pattern expanded
   at compile time). Cleanest provenance grouping (one application = one
   node fanning out to N facts; S1.6.2 renders it directly). Touches
   every `Firing.derived` consumer (detector, dumper, trace, tests).
2. **N firings, shared group-id.** Keep `derived: Fact`; add a
   `group_id` / `application_id` so consumers can cluster the N firings
   under one rule application. Lower-churn; the renderer groups by id.

## Open questions

- Which representation (1 vs 2)? — 1 is cleaner, 2 is lower-churn.
- Do all asserted facts share the **same** premises/provenance, or can
  a later conclusion depend on an earlier one within the same firing?
- Does saturation dedup / the contradiction detector need changes when
  one firing introduces several facts (especially a positive + its
  `not`)?
- Interaction with the `imply` family (S1.8.A7) and totality /
  domain-elimination (S1.8.A8), which are single-assert today.

## Acceptance (draft)

- A rule with `:assert (and …)` parses, compiles, and fires all
  conclusions from one match.
- The trace (S1.6.2) renders the multi-fact firing as a single rule
  application (one rule node → N derived facts), not N look-alike
  firings.
- zebra2 continues to solve (no regression for single-assert rules).
