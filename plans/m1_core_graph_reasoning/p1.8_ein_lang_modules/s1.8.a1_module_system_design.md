## Context

P1.3 review (2026-05-20) deferred the module-system design here.
The three coupled questions:

1. Where do rule definitions live — inline per puzzle, in a
   universal library, or hybrid?
2. How does one `.ein` file reference rules in another?
3. How are conflicts resolved when a puzzle redefines a library
   rule?

P1.3 proceeds under "rules are inline per puzzle" (current
zebra.ein convention); this stage revisits.

## Tasks

### Task T1.8.A1.1 — File vs symbol-level imports

Two extreme designs:

- **(a) File-level**: `(import "stdlib/sugar.ein")` includes
  everything from that file. Simple, no name selection.
- **(b) Symbol-level**: `(import "stdlib" :symbols (symmetric
  transitive functional))` selectively pulls names. More
  control, more verbose.

Possible hybrid: `(import "stdlib/sugar.ein")` for whole-file +
`(use stdlib/sugar :symbols (...))` for partial.

Decide based on: M1 puzzle scale (small — file-level wins),
expected stdlib growth (large — symbol-level wins eventually).

### Task T1.8.A1.2 — Namespace scoping

- **(a) Flat namespace**: imports merge into one global
  symbol table. Conflicts → error or warning.
- **(b) Module-scoped**: imported symbols live under
  `module/symbol` (e.g. `stdlib/symmetric`). Verbose but
  clear.
- **(c) Aliasing**: `(import "stdlib" :as std)` then
  `std/symmetric`. Lisp-like.

### Task T1.8.A1.3 — Conflict policy

When a puzzle's local rule clashes with an imported rule of the
same name:

- **(a) Error** — refuse to load.
- **(b) Warn + local wins** — most permissive.
- **(c) Warn + imported wins** — counterintuitive; skip.
- **(d) Explicit `(override …)`** — require the puzzle to flag
  the intent.

### Task T1.8.A1.4 — Path resolution

- **(a) Repo-relative**: `(import "stdlib/sugar.ein")` resolves
  to a fixed path under the engine.
- **(b) File-relative**: relative to the importing file.
- **(c) Environment / install-relative**: discoverable like
  Python's `sys.path`.

For M1, (a) is enough. Future M2 NL-output may want (c).

### Task T1.8.A1.5 — Loading semantics

- **(a) Eager**: imports loaded immediately on parse.
- **(b) Lazy**: imports resolved when needed.
- **(c) Caching**: each file loaded once per `solve()` call.

### Task T1.8.A1.6 — Decision record

Write up the chosen design in a short decision doc; this
becomes the spec for S1.8.A2 (grammar) + S1.8.A3 (loader).
