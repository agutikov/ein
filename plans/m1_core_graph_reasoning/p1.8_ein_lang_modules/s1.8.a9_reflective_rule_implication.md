# S1.8.A9 — Reflective rule-implication

**Phase:** P1.8 (improvements), Theme A.
**Estimate:** 1-2 days — small code change + tests.
**Status:** **DONE 2026-06-04 — no code change needed.** The capability

**Depends on:** [S1.8.A4](s1.8.a4_rule_library_location.md)

## Finding (2026-06-04)

The premise below — that `Saturator._enqueue_pass` iterates a *frozen*
`engine.cache`, so a derived activator gets no plan — is **stale**. The shipped
`saturator.py:_enqueue_pass` calls `self.engine.compile_all()` at the **start of
every pass** (its docstring even names the `(bijective R)` → `(functional R)`
case), and `compile_all` → `_activators_for` reads the **live**
`kb._rule_apps_by_rule` index, which `_index_fact` updates as facts are written.
So a fact a rule *derives* — `(symmetric foo)` — is an activator for the
matching generic rule on the very next pass, with **no extra code**. (This
landed incidentally, most likely with the S1.5a.2a stats-determinism fix that
made `_activators_for` read the live KB index rather than `rule.applications`.)

Verified empirically and pinned by `tests/inference/test_reflective_rule.py`:

- **Direct** — a rule asserts `(symmetric foo)`; the generic `symmetric` rule
  then fires `(foo A B) ⇒ (foo B A)`.
- **Indirect (`imply1`)** — `(imply1 functional symmetric)` + `(functional foo)`
  → `(symmetric foo)` → `symmetric` fires.
- **Termination (Q-S1.8.A9.A)** — a rule that derives its own activator reaches
  a fixpoint; the `_fired` / `_seen` dedup bounds the loop (no runaway).

So there is **no fix and no flag**: reflective rule-implication is unconditional
(it's how the saturator already works) and terminating. The
`:enable-reflective-rule-implication` gate (T1.8.A9.2) is dropped — gating it
would mean *disabling* the saturator's compile-refresh, which other behaviour
(the `bijective` expansion feeding `domain-elimination`) relies on. The
remaining value is the regression test (done) + the doc cross-reference in
`_enqueue_pass`.

The original framing follows, kept for the F5 lineage and the
compile-time-vs-runtime trade-off note (Q-S1.8.A9.B).

## Context

Per [P1.8 README Theme A § Engine support](README.md#standard-library):

For `imply1` to implement implication *between rules* (derive
`(symmetric foo)` → the `symmetric` rule then fires on `foo`),
a *derived* fact must be able to activate a rule. It currently
**cannot**: `Engine.compile_all()` snapshots `rule.applications`
**once**; `Saturator._enqueue_pass` then iterates the frozen
`engine.cache`, so a derived activator gets no plan.

The store side is fine (`_index_fact` live-updates
`_rule_apps_by_rule`); only the engine cache is static.

Fix (~6 lines): after a productive firing whose conclusion head
is a rule name, call `engine.compile_for(rule, derived_fact)`
(it exists, idempotent) + set `_needs_enqueue`.

Note `(imply functional closed)` itself does *not* need the
fix — `closed` feeds hypgen (a live-index reader), not a
rule. The fix is needed when the *conclusion* of `imply1` is a
rule activator (like `(symmetric foo)`).

## Tasks

### Task T1.8.A9.1 — The fix

In `inference/saturator.py` (or `engine.py`, wherever the
firing-commit path lives):

- **a.** After committing a firing, check whether the
  conclusion's head matches a registered rule name.
- **b.** If yes, call `engine.compile_for(rule, derived_fact)`.
- **c.** Set `_needs_enqueue` so the saturator picks up the
  new plan in its next sweep.

### Task T1.8.A9.2 — Config gate

Add `:enable-reflective-rule-implication` to `SolverConfig`.
Default **off** — opt-in for the experimentation.

### Task T1.8.A9.3 — Tests

- **a.** Direct: assert `(functional foo)`, then via a custom
  `imply1` derive `(symmetric foo)`, expect the `symmetric`
  rule to fire on `foo` (`(foo a b) ⇒ (foo b a)`).
- **b.** Indirect via `imply1`: `(imply functional symmetric)`
  + `(functional foo)` → `(symmetric foo)` → `symmetric` fires.
- **c.** Counter-test: with flag off, the chain stalls at
  `(symmetric foo)` not firing.

### Task T1.8.A9.4 — Document the contract

Saturator + engine docs (S1.20.D) document the contract: which
facts activate rules, and when re-compilation happens.

## Acceptance

- Fix lands behind flag.
- Tests cover direct + indirect cases.
- M1 acceptance unchanged with flag off (regression check).
- Documentation reflects the new behaviour when flag on.

## Open questions

- **Q-S1.8.A9.A — Termination.** Reflective rule-implication
  opens a loop: a rule activates another rule that activates
  another. Does saturation still terminate? Argument: M1's KB
  is finite; the rule registry is finite; activator facts
  are finite. So the loop bottom-out at some fixpoint. Confirm
  with a stress test.
- **Q-S1.8.A9.B — Compile-time vs runtime.** Could the
  compile-cache be re-built once per saturate-pass start
  instead of incrementally? Simpler, but slower. Trade-off
  depends on how often derived activators arrive.
