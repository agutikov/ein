# S1.8.A2 — Grammar / parser extensions for imports

**Phase:** P1.8 (improvements), Theme A.
**Estimate:** 1-2 days — pure grammar + AST work.
**Status:** parked; sequential after S1.8.A1.
**Depends on:** [S1.8.A1](s1.8.a1_module_system_design.md)
ships with module-system design decisions.
**Blocks:** [S1.8.A3](s1.8.a3_loader_resolve_imports.md)
(loader implementation needs the grammar in place).

## Context

Add the `(import …)` (and/or `(use …)`) **top-level flat form(s)** to
`ein.py/src/ein_bot/ir/grammar.lark`. Surface depends on
S1.8.A1's decisions.

**P1.7c context (2026-06-02).** The surface is now a **flat sequence of
forms** classified by head against a **closed declarator set**
([`06_reserved_names.md`](../../../docs/kernel/ir/03-ein-lang/06_reserved_names.md),
the parser's source of truth) — the `(ontology …)` / `(rules …)` / `(facts …)`
**block wrappers were removed** in
[S1.7c.4](../p1.7c_block_head_removal/s1.7c.4_migrate_and_drop_shim.md). So
`import` / `use` are not "inside a block"; they are **new top-level
declarators**, and adding each one is the standard P1.7c **4-point
registration** (grammar `?form` + `SYMBOL` negative-lookahead + reserved-names
doc + loader head-switch) — see Task T1.8.A2.1. Collision-safe: no
`examples/*.ein` or fixture uses `(import …)` / `(use …)` as a fact head
(grepped 2026-06-03), so promoting them to declarators misclassifies nothing.

## Tasks

### Task T1.8.A2.1 — Grammar + the P1.7c 4-point declarator registration

- **a.** **Grammar** (`ein.py/src/ein_bot/ir/grammar.lark`) — add
  `import_form` (and `use_form` if symbol-level imports were chosen) to the
  flat `?form` alternation. Path appears file-level only — there are no
  wrapper blocks to nest inside (P1.7c):
  ```diff
    ?form: relation_decl | rule_decl | hrule_decl | query_form
         | config_form | trace_form
  +      | import_form                      // ← new flat declarator (+ use_form)
         | fact_form
  + import_form: "(" "import" STRING (":symbols" "(" SYMBOL+ ")")? ")"
  ```
- **b.** **`SYMBOL` negative-lookahead** (`grammar.lark` ~line 176) — add
  `import` (+ `use`) to
  `/(?!(?:not|and|or|neq|rule|hrule|query|config|trace)\b)…/` so a malformed
  `(import)` is a *parse* error, not a silent fact named `import` falling
  through to `generic_fact`.
- **c.** **[`06_reserved_names.md`](../../../docs/kernel/ir/03-ein-lang/06_reserved_names.md)**
  — add `import` (+ `use`) to the closed Declarators table (it currently has
  no slot for either); the parser/loader key on that table.
- **d.** **Loader head-switch** (`ein.py/src/ein_bot/kb/from_ir.py`
  `load()`, ~line 415) — add an `import` / `use` branch. (The branch's *body*
  — resolve + merge — is [S1.8.A3](s1.8.a3_loader_resolve_imports.md); this
  stage just routes the head so it isn't bucketed as a fact.)

### Task T1.8.A2.2 — AST classes

In `ein.py/src/ein_bot/ir/ast.py`:

- **a.** New `Import` dataclass with fields `path: str` +
  optional `symbols: tuple[str, ...]` + optional alias.
- **b.** Update the Lark Transformer to construct `Import`
  nodes.

### Task T1.8.A2.3 — Dump support

The IR round-trip (`to_ir()` in the dumper) needs to write
`(import …)` forms back out. Same shape as other top-level
forms.

### Task T1.8.A2.4 — Round-trip tests

In `tests/test_ir_parser.py`:

- A puzzle with one `(import "stdlib")` form parses + dumps
  identically.
- A puzzle with multiple imports does the same.
- A malformed import (missing path string) rejects with a
  clear error.

## Acceptance

- Grammar accepts the new form(s) at the top level (flat — not inside any
  wrapper).
- `import` / `use` appear in
  [`06_reserved_names.md`](../../../docs/kernel/ir/03-ein-lang/06_reserved_names.md)'s
  closed Declarators table and in the `from_ir.load()` head-switch.
- AST constructs `Import` nodes correctly.
- A malformed `(import)` (no path) is rejected at *parse* time (the `SYMBOL`
  negative-lookahead guard), not silently lowered to a fact head.
- Round-trip parse + dump byte-identical (modulo
  whitespace).
- Tests green.

## Open questions

- **Q-S1.8.A2.A — Quoted path or atom?** Probably quoted
  string (matches the file-path content); atom-style would
  conflict with the bareword convention.
- **Q-S1.8.A2.B — Glob imports?** `(import "stdlib/*.ein")`
  pulls every file under `stdlib/`. Convenient but
  underspecified — skip for M1 unless trivial.
