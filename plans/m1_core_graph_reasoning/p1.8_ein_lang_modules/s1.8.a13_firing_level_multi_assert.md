## Why the split is the wrong layer

`:assert (and F1 F2 F3)` is just "emit three facts from one match." But the
engine's firing model is **one fact per firing**:

- `JoinPlan.assert_template: object | None` — a *single* compiled template
  ([`compile.py:120`](../../../ein.py/src/ein_bot/inference/compile.py)).
- `Firing.derived: Fact` — a *single* derived fact; `fire()` builds one fact and
  returns one `Firing` ([`firing.py:55,148`](../../../ein.py/src/ein_bot/inference/firing.py)).

So A11 lowered multi-assert at **load time** by *cloning the rule* — one rule per
conjunct, named `<rule>__and<j>`, plus a cross-product with the `or`-in-match
split `<rule>__or<i>__and<j>`
([`from_ir.py:388-430`](../../../ein.py/src/ein_bot/kb/from_ir.py)). This is the
wrong layer, and it costs us three ways:

1. **Generic multi-assert is impossible.** Generic rules are dispatched by *name*
   — a fact `(foo X)` activates the rule literally named `foo`
   (`store.py:430`, `rn in self.rules`, exact match). The split renames the rule
   to `foo__and0`, so `(foo X)` reaches nothing; the loader rejects it outright
   ([`from_ir.py:405`](../../../ein.py/src/ein_bot/kb/from_ir.py)). You cannot
   write a relation-polymorphic rule that scans `(?R ?a ?b)` and concludes
   several facts per edge — the archetypal "place a value, exclude the rest"
   constraint, generic over R.
2. **Phantom rules pollute everything keyed by rule.** `kb.rules` carries
   `__and0` / `__or1__and0` siblings; provenance, `_rule_apps_by_rule`, the
   agenda, and any by-name view see N look-alikes instead of one rule.
3. **Provenance lies about structure.** N facts that are *one* logical conclusion
   surface as N independent firings that merely share a name; the trace renderer
   reconstructs "these came from one application" heuristically (S1.6.2).

## The model — one rule, many plans, many asserts

Adopt A11's **option 1** (firing-level). A rule keeps its single name and
identity; the disjunction/conjunction lower at **compile** time into plans and
templates, never into new rules.

```
Rule  (unchanged name/params)
  match:  Pattern            # may be a top-level (or d1 … dm)
  assert: Pattern            # may be a top-level (and c1 … ck)
        │  compile_rule(rule, activator)
        ▼
list[JoinPlan]               # ONE plan per match disjunct  (m plans)
  each JoinPlan:
    assert_templates: tuple  # k compiled conclusion templates (shared)
        │  matcher yields (bindings, premises) per plan
        ▼
fire() → ONE Firing
  derived: tuple[Fact, ...]  # the k facts, one Provenance (one application)
```

A `(rule x :match (or d1 d2) :assert (and c1 c2))` becomes **2 plans, each
emitting 2 facts** — not the 4 phantom rules `x__or{0,1}__and{0,1}`.

### Concrete shape changes

| site | now | A13 |
|------|-----|-----|
| `JoinPlan` | `assert_template: object \| None` | `assert_templates: tuple[object, ...]` |
| `compile_rule` | `Rule → JoinPlan` | `Rule → list[JoinPlan]` (one per `or` disjunct) |
| `fire()` | builds 1 fact, 1 `Firing` | builds k facts, 1 `Firing(derived=tuple)` |
| `Firing.derived` | `Fact` | `tuple[Fact, ...]` |
| `from_ir` loader | `_match_disjuncts` / `_assert_conjuncts` → `__or/__and` rules + generic reject | keep the **AST** as written (or/and intact); no split, no reject |
| `store.py:430` | unchanged | unchanged — the rule keeps its name, so **generic activation just works** |

Lowering moves from the **loader** (rule-renaming) to the **compiler**
(plan-listing). The loader stops splitting; `compile_rule` reads the top-level
`(or …)` to emit m plans and the top-level `(and …)` to fill `assert_templates`.

## Tasks

### T1.8.A13.1 — `Firing.derived: tuple[Fact, ...]` + `fire()` fan-out
Make `fire()` walk `plan.assert_templates`, `build_fact` each under the *same*
`bindings` + `Provenance.from_rule` (one application), `add_and_index_fact` each,
and return one `Firing` whose `derived` is the tuple of stored facts. Keep a
`Firing.derived` length-1 path byte-equivalent to today for single-assert rules.

### T1.8.A13.2 — `compile_rule → list[JoinPlan]`, multi-template, multi-plan
`JoinPlan.assert_templates`. `compile_rule` lowers a top-level `:match (or …)`
to one plan per disjunct (sharing the templates) and a top-level `:assert (and …)`
to k templates. `engine.compile_all` stores a *list* of plans per rule.

### T1.8.A13.3 — Remove the load-time split + the generic reject
Delete the `__or<i>__and<j>` cross-product and the
`len(assert_nodes) > 1 and params` rejection in `from_ir.py`; keep `(or …)` /
`(and …)` as ordinary AST on the single `Rule`. (`_match_disjuncts` /
`_assert_conjuncts` move into the compiler, or are re-expressed there.)

### T1.8.A13.4 — Saturator: per-plan agenda, multi-fact redundancy, fired-key
- Enqueue/run each of a rule's plans (the `_fired` key gains a **plan/disjunct
  discriminator** so two disjuncts with the same bindings don't alias).
- Redundancy: a firing is redundant iff **every** templated fact already exists;
  if any is new, fire and emit only the new ones (define partial-overlap, see
  Open Qs). The fire-time NAF re-check (`absents_still_pass`) runs once per match.

### T1.8.A13.5 — `Firing.derived` consumers → iterate the tuple
Wide-but-shallow sweep of every `.derived` reader:
- **contradiction / lookahead** (`lookahead.py:139,145`) — check each derived
  fact; a single firing asserting both `g` and `(not g)` self-contradicts at the
  REASONING layer (the A11 open question — confirm intended).
- **trace** (`linearize.py:77`, `render.py`, `ast.py`, `relevance.py`) — render
  one application → N facts as one node fanning out (the payoff).
- `naf_deps.py`, `monotonic/state_dump.py`, `render/slice.py`, and tests asserting
  `firing.derived.relation_name`.

### T1.8.A13.6 — Migrate `test_multi_assert.py` + re-home its intent
The split tests (`set(kb.rules) == {"x__or0__and0", …}`) become "one rule, m
plans, k asserts; one match emits k facts." **Add the previously-impossible
generic case** as a *passing* test: `(rule place-and-exclude (?rel) :match (?rel
?a ?x) … :assert (and (?rel ?a ?x) (not (?rel ?a ?y))))` now loads, activates via
`(place-and-exclude R)`, and fires both facts.

### T1.8.A13.7 — Docs
Update [S1.8.A11](s1.8.a11_multi_fact_assert.md) (mark the load-time split
superseded by A13), the reserved-name / primitives notes if `__and`/`__or`
appear, and the engine inference docs' firing model.

## Acceptance

- No `__and<j>` / `__or<i>` rule names anywhere in `kb.rules` or provenance;
  `(or …)`/`(and …)` rules carry their single source name end-to-end.
- A **generic** rule with `:assert (and …)` loads, activates by its bare name,
  and fires all conclusions from one match (the A11 reject is gone).
- A rule with `:match (or …)` runs m match plans under one name.
- One match → one `Firing` → a `derived` tuple sharing one `Provenance`.
- zebra2 acceptance unchanged (Solution k=1) — `bijective-properties` /
  `type-hierarchy-converse` now fire as single multi-fact rules; verify the
  resolved verdict and the saturation firing-count are equivalent.
- Full suite green.

## Open questions

- **Q-A13.A — partial-overlap redundancy.** When some templated facts are new and
  some already exist, fire once and emit only the new (one application, fewer
  facts), or treat any-new as fully productive? Pin the dedup + the `_fired` key.
- **Q-A13.B — intra-firing contradiction.** A firing that derives `g` and
  `(not g)` is inconsistent by construction — should the contradiction detector
  flag it as ⊥ immediately (likely yes), and does ordering within the tuple
  matter to the detector?
- **Q-A13.C — provenance granularity.** All k facts share one `Provenance`
  (same premises/bindings — no intra-firing data dependency, per A11). Confirm no
  consumer needs per-fact premises.
- **Q-A13.D — nested `or`.** Top-level `(or …)` in `:match` lowers to plans;
  a *nested* `or` (inside `and`) still needs DNF and stays unsupported
  (`compile.py:278`) — A13 does not change that.
- **Q-A13.E — agenda fairness / priority.** m plans of one rule share the rule's
  priority; confirm the tiebreaker stays deterministic across plans (it keys on
  registration/enqueue order — the plan discriminator must be stable).
