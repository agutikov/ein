## Why this stage exists

S1.5.8 as originally drafted (2026-05-22) introduced
`(total ?R ?OT ?VT)` — a *sub-domain totality* declaration. The
2026-05-23 design discussion rejected sub-domains as a kernel
concept: a "sub-domain of co-located restricted to (House, Color)"
*is* a named derived relation, and admitting it implicitly through
a property activator is the same construct as introducing a hidden
relation, just spelled differently. **B1 chosen**: drop sub-domains;
refactor zebra2 into specific binary relations that carry their
properties globally.

The discussion also surfaced four further design points the original
S1.5.8 did not address:

1. Totality / functionality / etc. are properties of a *projection*
   of the relation, not of the relation as a whole. For
   `(R Person House)` the four projections (Person → House,
   House → Person) carry independent totality / functionality
   verdicts. M1's existing `(functional ?R)` rule is implicitly
   "functional :on arg1" — works for binary by coincidence,
   doesn't scale.
2. N-ary / hyper-relations (`(cartesian-to-polar x y r a)`,
   `(sum x y z)`) exist; the framework must not preclude them.
3. Symmetric hyper-relations (`(coplanar p q r)`) exist
   mathematically; M1 won't need them but the design must name
   the case.
4. Operating on relation structure from inside rules (the
   `(func dom ?R)` direction) is meta-programming the kernel
   doesn't need. The narrower need — "what type sits at position
   `i` of relation R?" — falls out of auto-derived reflection
   facts.
5. Argument well-formedness — a kernel rule that contradicts
   when a relation fact's args don't match the declared
   signature. Catches modeling errors at saturation time with a
   named firing rather than via a downstream mystery
   contradiction; rests on the same reflection machinery as point
   #4.

This stage produces the framework, the zebra2 refactor, the
typecheck rule, and the S1.5.8 rule redesign before any code
lands.

## Tasks

### Task T1.5.8a.1 — Per-projection property framework

The kernel decision: properties like `total`, `functional`,
`injective`, `surjective` take an optional `:on` (input arg or
arg group) and optional `:to` (output arg or arg group).

#### Surface form

```lisp
;; Binary, default — :on arg1 implied, :to arg2 implied.
(functional live-in)            ; ≡ (functional live-in :on (1) :to (2))
(total      live-in)            ; ≡ (total live-in :on (1))

;; Binary, the other direction.
(functional live-in :on (2) :to (1))    ; House → Nationality functional too
(total      live-in :on (2))            ; every House has someone (in zebra)

;; N-ary, must be explicit.
(functional cart-to-polar :on (1 2) :to (3 4))
(total      cart-to-polar :on (1 2))

;; Multiple projections on the same relation.
(functional sum :on (1 2) :to (3))   ; z = x + y
(functional sum :on (1 3) :to (2))   ; y = z − x
(functional sum :on (2 3) :to (1))   ; x = z − y
```

`:on` and `:to` take tuples of 1-based arg positions. The empty `:to`
is rejected; `:to` defaults to "everything not in `:on`" when omitted
for arities ≤ 3 (sugar for the common case).

#### Semantics

| property               | meaning                                                                                  |
|------------------------|------------------------------------------------------------------------------------------|
| `(total R :on I)`      | ∀ tuples t indexed by I that exist in the type product of those positions, ∃ a fact in R extending t |
| `(functional R :on I :to J)` | for fixed t over I, at most one tuple over J extends t to a fact                    |
| `(injective R :on I :to J)`  | for distinct t, t' over I, the J-tuples are distinct (or undefined)                  |
| `(surjective R :on I :to J)` | every J-tuple in its type product is reached by some I-tuple                         |

`(symmetric R)` and `(transitive R)` stay as whole-relation
primitives — see Q-S1.5.8a.2.

#### Sub-tasks

- **a.** Grammar update — `(functional R :on (i …) :to (j …))` and
  the three siblings. Defaults: binary `:on arg1, :to arg2`;
  n-ary requires explicit `:on`.
- **b.** Loader auto-derives `(arity ?R ?n)` from the
  `(relation R T1 … Tn)` declaration; rejects projection-tuples
  whose positions are out of range.
- **c.** Each property activator stays an **ontology fact**, not
  a rule kwarg — preserves the "property tags are
  rule-application facts" M1 design rule.
- **d.** Existing `(functional ?R)` rule in the demo `.ein` files
  (branching/01-11) keeps working under default-binary semantics
  — *no demo edits required*. Verified manually by re-reading
  the rule body `(?R ?a ?b) (?R ?a ?c) ⇒ ⊥`: that *is* the
  arg1-projection rule, which the default covers.

### Task T1.5.8a.2 — Relation reflection facts

Replace the lispy `(func dom ?R)` direction (which would need
tuple constructors + `car` / `cdr` accessors) with auto-derived
reflection facts a regular `:match` can read.

#### Auto-derived at load

For every `(relation R T1 … Tn)`:

```lisp
(arity     R n)
(arg-type  R 0 T1)        ; 0-indexed in the engine, 1-indexed only in the :on/:to surface (sugar)
(arg-type  R 1 T2)
…
(arg-type  R (n-1) Tn)
```

#### Use sites

```lisp
;; Domain-elimination, rewritten against this framework
;; (compare to S1.5.8 T1.5.8.2 — sub-domain version):
(rule domain-elimination (?R)
  :match  (and (functional ?R :on (?i) :to (?j))
               (total      ?R :on (?i))
               (arg-type   ?R ?i ?OT)
               (arg-type   ?R ?j ?VT)
               (is-a ?a ?OT)
               (only-remaining ?R ?a ?i ?j ?v))
  :assert (?R-with-?v-at-?j-and-?a-at-?i)
  :priority 400
  :why    "Every {?OT} is mapped by {?R} to some {?VT};
           {?a} has all but {?v} excluded — therefore ({?R} … {?v} …).")
```

Two unresolved sub-points the implementation phase has to handle:

- **i. Constructing the asserted fact.** The conclusion's positional
  shape depends on `?i` / `?j` (a 4-ary relation could place the
  forced value at any position). The current `:assert` syntax
  doesn't have positional construction. S1.5.8 owns the fix —
  candidate: a built-in `(assoc-fact ?R (?i ?a) (?j ?v))` form,
  or `:assert` with `:positional ((?i ?a) (?j ?v))`. Picked at
  implementation time; for binary M1 needs, the conclusion is
  always `(?R ?a ?v)` with `?i=1, ?j=2`, so the simple form
  works.
- **ii. `only-remaining` signature widens.** From the S1.5.8 draft's
  `(only-remaining ?R ?a ?VT ?v)` to `(only-remaining ?R ?a ?i ?j ?v)`
  — the predicate now knows *which* projection it's scanning. For
  binary M1 use the extra `?i` / `?j` are constant (1 / 2), but
  carrying them keeps the rule encoding faithful for future n-ary.

#### Sub-tasks

- **a.** Loader auto-asserts `(arity …)` + `(arg-type …)` for
  every declared relation; these go into the **ontology** layer
  (read-only, not subject to revision in the fact / reasoning
  layers).
- **b.** Document the position-indexing convention (0-indexed in
  engine, 1-indexed in surface `:on (…)` — pick one and stop
  flipping). Recommend **0-indexed engine, 0-indexed surface**;
  arithmetic for projection tuples is cleaner and the user-facing
  surface is rare enough that "arg 0" reads fine.
- **c.** Note in `docs/kernel/ir/01-ein-graph/02_rules.md` that
  relation reflection is a kernel feature, not a stdlib rule —
  these facts are present in *every* puzzle without import.
- **d.** Forward pointer: when F5 (rules-as-data) lands, the same
  pattern auto-derives `(rule-arg-type rule i T)` /
  `(rule-arity rule n)` for rules. Not in M1 scope.

### Task T1.5.8a.3 — Relation-argument typecheck rule

A kernel well-formedness rule that fires `(false)` whenever a
relation fact is asserted with an arg whose direct type doesn't
match the relation's declared signature. Catches puzzle-author
errors (e.g., `(house-color House-1 Englishman)` mixing
Nationality into a Colour slot) at saturation time with a named
firing, instead of via a downstream mystery contradiction.

#### The rules (binary form for M1)

```lisp
(rule rel-arg-typecheck-dom ()
  :match  (and (relation ?R ?Dom ?Ran)
               (?R ?a ?b)
               (not (is-a ?a ?Dom)))
  :assert (false)
  :priority 110
  :why    "{?R} arg 0 expects {?Dom}; {?a} is not a {?Dom}.")

(rule rel-arg-typecheck-ran ()
  :match  (and (relation ?R ?Dom ?Ran)
               (?R ?a ?b)
               (not (is-a ?b ?Ran)))
  :assert (false)
  :priority 110
  :why    "{?R} arg 1 expects {?Ran}; {?b} is not a {?Ran}.")
```

Priority **110** — inside the 100 propagate band (Q41 banded
scale: 100=propagate, 200=derive, 300=eliminate, 900=hypothesis)
but fine-grained-after the 100 propagation rules so a fact lands
in the KB first and then gets type-checked. Within the
*well-formedness* sub-band; document alongside the existing 100
entries in [S1.3.3 T1.3.3.2](../p1.3_inference_rules/s1.3.3_saturation_engine.md#task-t1332--priority-bands-documentation).
The empty activator pattern `()` is correct — `?R` is bound by
the `:match` pattern itself, not a rule param. (M1's higher-order
matcher already supports relation-name variables in pattern head
position; see the existing `(rule functional (?R) :match (and (?R ?a ?b) …))`.)

#### Subtlety — direct vs transitive is-a

The rule uses `(is-a ?a ?Dom)` — a *direct* parent match.
zebra2 intentionally omits `(transitive is-a)` (see
`examples/zebra2.ein` ~line 196: activating transitive is-a
makes `sibling-exclusive` fire cross-boundary and emit ~1300
spurious negatives). So the rule fires correctly **iff** the
relation's declared types are the **direct parents** of the
args:

- `(relation house-color House Color)` + `(house-color House-1 Yellow)`
  + `(is-a House-1 House)` + `(is-a Yellow Color)` → silent. ✓
- `(relation house-color House Color)` + `(house-color House-1 Englishman)`
  with no `(is-a Englishman Color)` → fires `(false)`. ✓
- `(relation R Attribute Attribute)` + `(R House-1 Yellow)`:
  House-1's direct parent is House, not Attribute. The rule would
  spuriously fire **unless** transitive is-a is activated for this
  case — which it isn't, and which can't be flipped on without
  cascading damage.

Constraint that falls out: the **B1 zebra2 refactor must declare
every relation at the direct-parent level its args occupy** —
`(relation house-color House Color)`, never
`(relation house-color Attribute Attribute)`. Recorded as a
sub-task on [T1.5.8a.5](#task-t158a5--zebra2-refactor--specific-binary-relations).

#### Symmetric / n-ary cases

- For `(symmetric R)` with same-typed args (`(relation R T T)`),
  both rules check the same type; symmetry is irrelevant.
- For n-ary relations the binary rule shape doesn't generalise.
  When variadic `?args...` + an `nth` predicate land (Q33
  followup), one rule replaces the pair:

  ```lisp
  (rule rel-arg-typecheck ()
    :match  (and (arg-type ?R ?i ?T)
                 (?R ?args...)
                 (nth ?args ?i ?a)
                 (not (is-a ?a ?T)))
    :assert (false)
    :priority 110)
  ```

  Forward-pointer documented; M1 ships only the two binary rules.

#### NAF caveat

`(not (is-a ?a ?Dom))` uses M1's negation-as-failure semantics
(absence of the positive). This is **sound** for zebra2 because
is-a in zebra2 is a closed, fully-listed ontology — the absence
of `(is-a Englishman Color)` is the truth, not a derivation gap.
For puzzles with open-world is-a (a future possibility), this
rule's NAF would over-fire; flag in
`docs/kernel/ir/01-ein-graph/02_rules.md` as a constraint on the
ontology layer's closed-world reading.

#### Sub-tasks

- **a.** Ship both rules in the canonical M1 stdlib slot (inline
  in each puzzle's `(rules …)` block until the P1.8 stdlib lands).
- **b.** Test: zebra2 with one deliberately mis-typed fact (e.g.,
  `(house-color House-1 Englishman)`) returns `Contradiction`
  whose `unsat_core` cites the relation decl + the offending
  fact, and the trace shows `rel-arg-typecheck-ran` as the firing.
- **c.** Test: zebra2 without mis-typed facts — both rules
  silent (zero firings logged).
- **d.** Cross-link from [T1.5.8a.5](#task-t158a5--zebra2-refactor--specific-binary-relations)'s
  sub-tasks the direct-parent-types constraint discovered above.
- **e.** Forward-pointer note in
  `docs/kernel/ir/01-ein-graph/02_rules.md` — the variadic
  general form parks behind the Q33 deferral.

### Task T1.5.8a.4 — N-ary grammar acceptance + symmetric-hyper scoping

#### N-ary admissions

Grammar already accepts `(relation R T1 T2 … Tn)` for any arity
(P1.1 T1.1.1.2 — relation-decl is a head + atom list). What this
stage adds:

- **Property activators** parse with `:on` / `:to` for any arity.
- **`(arg-type R i T)` / `(arity R n)`** auto-derivation works
  for any arity.
- **The M1 rule library remains binary-only.** Specifically, the
  domain-elimination rule, the spatial rules
  (`square-fwd` / `square-bwd` / `square-unique`), and the
  zebra2 ontology all stay binary. The framework permits more;
  the rule library does not exercise it.

#### Symmetric hyper-relations — scoped out

`(symmetric R)` for n > 2 means "the fact is permutation-invariant
over its args" (`(coplanar p q r) ≡ (coplanar q p r) ≡ …`). M1's
rule library has zero n > 2 relations and so zero use for
symmetric-hyper, but the framework must not preclude:

- `(symmetric R)` stays a **whole-relation primitive** —
  declares the symmetric group Sym(arity(R)) acts on R's facts.
- For binary R, this is the existing behaviour (Sym(2) is just
  swap, exactly what `(symmetric co-located)` does today).
- For n > 2 R, the loader accepts it but no M1 rule consumes
  the n > 2 case. Document the future-proofing in
  `docs/kernel/ir/01-ein-graph/02_rules.md` and stop.
- **Partial symmetry** (`(symmetric R :over (1 2))` — only the
  first two args swap) is out of scope; if a puzzle ever needs
  it, that's an F4 sub-question, not an M1 one.

#### Sub-tasks

- **a.** Grammar test — `(relation R T1 T2 T3 T4)` round-trips.
- **b.** Loader test — `(symmetric R)` on a 3-arity relation
  parses without error; nothing in the rule library fires on it.
- **c.** Document the n-ary + symmetric-hyper scope-out under
  `docs/kernel/ir/01-ein-graph/02_rules.md` § "Properties on
  projections".

### Task T1.5.8a.5 — zebra2 refactor — specific binary relations

The concrete output of B1: replace zebra2's one symmetric+transitive
`co-located` with five House-rooted specific relations, each
globally `(total)` + `(functional)` on both projections (=
bijections). Spatial relations stay House-on-House. Worked through
below; final form lands in `examples/zebra2.ein` as part of S1.5.8
implementation.

#### Relation decls + properties

```lisp
;; Attribute-vs-House relations: each is a bijection
;; (5 of each type, paired 1-1 with houses).
(relation house-color  House Color)
(functional house-color :on (0) :to (1))   ; each house has at most one colour
(functional house-color :on (1) :to (0))   ; each colour at most one house
(total      house-color :on (0))            ; every house has some colour
(total      house-color :on (1))            ; every colour belongs to some house

(relation house-nation House Nationality)
(functional house-nation :on (0) :to (1))
(functional house-nation :on (1) :to (0))
(total      house-nation :on (0))
(total      house-nation :on (1))

(relation house-drink  House Drink)         ; same shape
(relation house-smoke  House Cigarette)
(relation house-pet    House Pet)

;; Spatial relations: House-on-House.
(relation right-of House House)
(functional right-of :on (0) :to (1))       ; at most one house to the right
(functional right-of :on (1) :to (0))       ; at most one house to the left
;; NOT total — rightmost has no right-neighbour, leftmost has no left.
(square-fwd    right-of)
(square-bwd    right-of)

(relation next-to  House House)
(symmetric  next-to)
(implies    right-of next-to)
(square-unique next-to House)
```

#### Puzzle conditions — old vs new

The 15 Zebra conditions re-expressed:

| #  | English                                                  | old (co-located)                          | new (specific)                           |
|----|----------------------------------------------------------|-------------------------------------------|------------------------------------------|
| 1  | Englishman lives in the red house                        | (co-located Englishman Red)              | (∃ ?h. house-nation ?h Englishman ∧ house-color ?h Red) — **constraint**, not fact |
| 2  | Spaniard owns the dog                                    | (co-located Spaniard Dog)                | (∃ ?h. house-nation ?h Spaniard ∧ house-pet ?h Dog) |
| …  |                                                          |                                           |                                          |
| 9  | Milk is drunk in the middle house                        | (co-located House-3 Milk)                | (house-drink House-3 Milk) — direct fact |
| 10 | Norwegian lives in the first house                       | (co-located Norwegian House-1)           | (house-nation House-1 Norwegian) — direct |
| 11 | Chesterfields next to fox owner                          | (∃ ?h ?h'. … next-to)                    | similar, via two specific relations + next-to |
| 13 | Lucky Strike drinker drinks orange juice                 | (co-located Lucky_Strike Orange_Juice)   | (∃ ?h. house-smoke ?h Lucky_Strike ∧ house-drink ?h Orange_Juice) |
| 14 | Japanese smokes Parliaments                              | (co-located Japanese Parliaments)        | (∃ ?h. house-nation ?h Japanese ∧ house-smoke ?h Parliaments) |
| 15 | Norwegian next to blue                                   | (∃ ?h ?h'. next-to … co-located …)       | (∃ ?h ?h'. house-nation ?h Norwegian ∧ house-color ?h' Blue ∧ next-to ?h ?h') |

#### What this exposes

- **Cross-attribute conditions are now existential constraints,
  not single facts.** Conditions 1, 2, 3, 4, 5, 6, 7, 8, 11, 12, 13,
  14, 15 — most of the puzzle — bind through a House existential.
  The IR needs an existential-binding `:match` form (already present
  via the rule's existential `?h`), but the *fact* layer in
  `(facts …)` currently expects atomic facts.
- **Two options for the facts layer**:
  - (i) **Atomic-fact-only** (cleanest): the puzzle's `(facts …)`
    block contains only direct facts — conditions 9, 10. The other
    13 conditions land as **rules** (existential matchers that
    contradict if violated, or assert the joint pair if confirmed).
    Heavier in rule count but clean separation.
  - (ii) **Allow `exists` in facts**: introduce `(exists ?h
    (and (house-nation ?h Englishman) (house-color ?h Red)))` as
    a fact form. Smaller IR change but introduces existentials
    in the fact layer.
- **Recommendation**: (i). The "exists then conjoin" pattern is
  exactly what `hypothesis-contradiction` rules look like
  (P1.3 T1.3.2 hypothesis row); the existing pattern works.

- **b.** Draft `examples/zebra2.ein` in the new encoding (do

- **c.** Cardinality budget — the cross-attribute conditions are

- **d.** Drop `co-located` from zebra2. Trace text in

- **e.** Audit `examples/zebra.ein`. zebra.ein also uses

- **f.** **Direct-parent types throughout** (constraint imported
  from [T1.5.8a.3](#task-t158a3--relation-argument-typecheck-rule)).
  Every relation in zebra2 is declared at the level its args

### Task T1.5.8a.6 — S1.5.8 redesign against the framework

S1.5.8 now ships against a clean framework. The shape changes:

- **`domain-elimination` rule** — uses `arg-type` + `arity` +
  per-projection `total` / `functional`; binds the survivor at
  the correct position; conclusion-fact construction handled by
  T1.5.8a.2 sub-task (i). For M1's binary use, the rule body is
  exactly the snippet in [T1.5.8a.2](#task-t158a2--relation-reflection-facts).
- **`only-remaining` predicate** — signature widens to
  `(only-remaining ?R ?a ?i ?j ?v)`. The `StructuralScan` opcode
  scans `(arg-type ?R ?j ?VT)` to find the value-type, then
  walks instances of `?VT` against stored `(not (?R … ?a … ?w …))`
  facts. For binary M1 use, `?i=0, ?j=1`, so the scan reads
  exactly today's S1.5.8 draft.
- **All-excluded contradiction** — unchanged; T1.5.8.3 stays.
- **Activator pair** — `(total ?R :on (?i))` +
  `(functional ?R :on (?i))` together trigger the rule. Functional
  is needed because without it, multiple values could co-exist and
  "only one remaining" doesn't force a unique conclusion. The
  S1.5.8 draft elided this — the rule needs *both*.

#### Sub-tasks

- **a.** Rewrite [S1.5.8 T1.5.8.1](s1.5.8_totality_domain_elimination.md#task-t1581--totality-declaration)
  in place: drop the `(total ?R ?OT ?VT)` 3-arg sub-domain form;
  adopt the projection form from T1.5.8a.1.
- **b.** Rewrite [S1.5.8 T1.5.8.2](s1.5.8_totality_domain_elimination.md#task-t1582--domain-elimination-rule--matcher-capability)
  in place: drop `?OT` / `?VT` from the rule signature; use the
  `arg-type` reflection; require `functional` *and* `total`.
- **c.** Strike S1.5.8's "IMPORTANT NOTE!" block (the embedded
  open-question about sub-domains) — resolved by this stage's
  B1 decision; cross-reference Q-S1.5.8.D resolution back here.
- **d.** Update S1.5.8 acceptance — the trace's "therefore …"
  text now reads the specific-relation vocabulary directly
  (Norwegian *drinks* Water; Japanese *owns* Zebra), not via
  co-located.

## Open questions parked here

- **Q-S1.5.8a.1 — Naming: `functional` vs `exclusive`.** The
  2026-05-23 discussion used "exclusivity" for the at-most-one
  property; the existing M1 vocabulary uses `functional` (the
  branching/01-11 demos and the
  [S1.5.4a fix](s1.5.4a-fix-single-parent.md) all assume this
  name). Two existing names already start with "exclusiv":
  `sibling-exclusive` and `type-exclusivity`, both meaning
  *pairwise distinctness over instances of a type* — a different
  thing. **Recommend: keep `functional`** for the at-most-one
  property; the discussion's "exclusivity" was informal. Confirm
  at ship and stop.
- **Q-S1.5.8a.2 — `(symmetric R)` / `(transitive R)` — derived
  or primitive?** `(symmetric R)` for binary could be derived
  from "all (i, j) projections are functional and total in
  matching pairs", but that's awkward. `(transitive R)` doesn't
  reduce to per-projection at all (it's a 3-arg relation between
  pairs of facts). **Recommend: keep both as primitives**, no
  projection args. Update
  [docs/kernel/ir/01-ein-graph/02_rules.md](../../../docs/kernel/ir/01-ein-graph/02_rules.md)
  to call them "whole-relation properties" alongside the
  "projection properties" family.
- **Q-S1.5.8a.3 — Does zebra2 keep a derived `co-located`?**
  Spatial rules (`next-to`, `right-of`) only need House identity;
  no rule needs cross-attribute composition that today's
  `(transitive co-located)` was supplying. **Recommend: drop
  `co-located` from zebra2 entirely.** If a future puzzle needs
  cross-attribute composition, write the composition rule
  explicitly — the elegance of "one big symmetric+transitive
  relation" was always doing the work of the specific relations
  under the hood, and the trace fidelity argument (idea 08)
  favours making that work visible.
- **Q-S1.5.8a.4 — Position indexing: 0-based or 1-based?** The
  user-facing surface (`:on (1 2)`) and the engine (`Fact.args`
  is 0-indexed by Python convention). Pick one for both.
  **Recommend: 0-indexed everywhere.** Surface use of `:on` is
  rare; engine code wins consistency.
- **Q-S1.5.8a.5 — `:assert` positional construction.** The
  `domain-elimination` rule's conclusion is `(?R … ?v at ?j … ?a at ?i …)`
  — positional construction the current `:assert` doesn't
  support. For binary M1 the conclusion is fixed at
  `(?R ?a ?v)`. **Defer to S1.5.8 implementation**: ship the
  binary special case; the n-ary general case is a future-work
  syntax addition (forward pointer to P1.8 / F2).
