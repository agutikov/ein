## Context

S1.5.4b removed the "slot-already-used" filter from
`_hypotheses_for` because it implicitly assumed the relation is
functional on that slot — sound for Zebra-style puzzles, wrong as
soon as a multi-image relation (`friends-with`) shows up. The
principled architecture: let user rules express the functional
constraint via `(functional R)` / `(sibling-exclusive …)`,
let those rules' `:assert (not h)` populate `kb._negated_facts`,
let Filter C (T1.5.4.7's `negated_fact` filter) drop `h` in O(1).

The cost of that fix is that **the rules need to actually fire**
to populate `_negated_facts`, and firing them is what S1.5.4's
Topic B Tier B was trying to avoid (the "pay for fork +
saturate + detect just to learn this hypothesis dies in one rule
step" complaint that motivated the whole stage).

S1.5.6 closes the loop:

1. **Sibling-exclusive 2-arg rewrite** (T1.5.6.1) makes the rule
   express what it actually means: "objects sharing a parent
   under one relation are forbidden from co-occurring under
   *another* relation". The current 1-arg form hardcodes the
   exclusion relation to `co-located`; the 2-arg form lets the
   rule library declare which relation pairs are exclusive.
2. **One-step lookahead** (T1.5.6.2) reads the activators of
   negative-asserting rules (including the new
   `(sibling-exclusive ?S ?X)`) and pre-detects which
   hypotheses die against the parent KB *without* paying for a
   fork. The TODO observation: *"engine must know that
   `sibling-exclusive` rule affects both `is-a` and `co-located`
   hypothesis"* — the 2-arg rewrite is what makes this
   observable via the activator's `?exclusive-under` argument.

The two land together because lookahead reading the *old* 1-arg
form learns nothing useful (the rule body hardcodes the
conclusion relation, opaque to the matcher's pre-firing simulator);
the 2-arg form makes the impact relation a first-class binding
the lookahead can index.

## Tasks

### Task T1.5.6.1 — Sibling-exclusive two-argument rewrite

Generalise the rule from one parameter (the inheritance relation)
to two (inheritance + exclusion).

#### Current form (zebra2.ein, S1.5.4 baseline)

```lisp
(rule sibling-exclusive (?rel)
  :match  (and (?rel ?a ?P)
               (?rel ?b ?P)
               (neq ?a ?b))
  :assert (not (co-located ?a ?b))   ;; co-located is hardcoded
  :why    "{?a} and {?b} share parent {?P} under {?rel} —
           exclusive slots."
  :priority 300)
```

Activator: `(sibling-exclusive is-a)`. The conclusion relation
`co-located` is baked into the rule body — the engine can't tell
from the activator which hypothesis-relation the rule kills.

#### Target form

```lisp
(rule sibling-exclusive (?siblings-via ?exclusive-under)
  :match  (and (?siblings-via ?a ?P)
               (?siblings-via ?b ?P)
               (neq ?a ?b))
  :assert (not (?exclusive-under ?a ?b))
  :why    "{?a} and {?b} share parent {?P} under {?siblings-via}
           — exclusive under {?exclusive-under}."
  :priority 300)
```

Activators in zebra2 ontology:

```lisp
(sibling-exclusive is-a co-located)   ;; same-type instances aren't co-located
(sibling-exclusive is-a is-a)         ;; siblings aren't is-a each other (a leaf-only stronger form of asymmetric is-a)
```

**Variable naming.** Per the TODO: *"carefully pick rel var names
according to meaning"*. The two parameters carry distinct roles:

- `?siblings-via` — the relation that defines the "siblings"
  group (parent-sharing under this relation).
- `?exclusive-under` — the relation siblings are forbidden from
  participating in pairwise.

Avoid generic names like `?rel1` / `?rel2`. The mnemonic for
readers: *"siblings via X are exclusive under Y"* reads back the
activator argument order directly.

- **d.** Cascading edit to S1.5.5 if it's already shipped — the

#### Open question on the rename

Q-S1.5.6.A — Some rule-library readers may prefer
`?inheritance` / `?exclusion` over `?siblings-via` /
`?exclusive-under`. The former pair is shorter; the latter pair
makes the *role* in the rule's logic explicit (the first
argument's variable participates in the `:match` premise; the
second's variable participates only in the `:assert`).
Settle when T1.5.6.1.a is written; document the choice in the
rule's `:why` template.

### Task T1.5.6.2 — `_dies_immediately(kb, h)` one-step simulator

Per the original T1.5.4.0 design — a Python predicate that runs
*one rule step* with `h` substituted into each negative-asserting
rule's premises, checking whether the assertion would be
`(not h)` (h's death) or `(not f)` for some `f` already in the
parent KB (some other contradiction).

Belongs *between* T1.5.4.7's filter pipeline and `try_branch`
in `_explore` — applied to the alive-set candidate before paying
for a fork.

#### Design

- `_dies_immediately(kb: KnowledgeBase, hyp: Fact) -> bool` in
  `inference/hypgen.py`, alongside `_is_excluded`.
- Iterates each rule whose `:assert` head is `not` (registry
  pre-computed at solve() entry):
    1. Substitute `hyp` into each `:match` premise that pattern-
       matches `hyp`'s relation name (post-T1.5.6.1, this
       includes the polymorphic activator binding so
       `sibling-exclusive`'s `?exclusive-under` matches `hyp`'s
       relation).
    2. Run the remaining premises against the parent KB's
       facts using the existing `inference/match.py` machinery —
       but **observe** the result rather than firing.
    3. If the resulting `:assert` is `(not hyp)` exactly, or
       `(not f)` for an `f` already in
       `kb._facts_by_relation[f.relation_name]`, return True.
- The match is the same machinery `Saturator` uses; the only
  difference is we don't write the result back. Reuses
  `compile.compile_pattern` + `match.run` with `hyp` injected as
  a frozen fact.

#### Why T1.5.6.1 is the prerequisite

Without the 2-arg rewrite, the lookahead applied to
`sibling-exclusive` (1-arg form) reads the body and finds
`:assert (not (co-located ?a ?b))` — hardcoded relation
`co-located`. The lookahead can only ever pre-detect
`co-located` hypothesis deaths via this rule, not `is-a` ones.
The TODO's call to "carefully pick rel var names" makes the
exclusion relation a binding; the lookahead simulator can then
match `hyp.relation_name` against the bound `?exclusive-under`
and prove the death.

#### Implementation notes (deviations from the pre-ship spec)

- **Wired as a hypgen filter, not in `_explore`.** Per the
  2026-05-22 direction "S1.5.6 is a hypgen feature" — the
  lookahead is the last filter in `hypgen._apply_filters` (gated
  by `enable_pre_branch_lookahead`); a doomed candidate is
  dropped + counted (`HypGenStats.filtered["lookahead_killed"]`).
  T1.5.6.2.c's "dead-leaf SearchNode per kill" is superseded — a
  hypgen filter has no branch to record.
- **All rules, not only `:assert (not …)` rules.** The headline
  win — a candidate `h` that joins the saturated KB under
  `transitive` / a `square` rule to derive an already-negated
  fact — needs *positive*-asserting rules. The simulator
  (`inference/lookahead.py`) injects `h` into every `:match`
  premise it unifies with, runs the rest one rule firing, and
  flags four contradiction shapes: `(false)`, `(not h)`,
  `(not g)` for an existing REASONING-layer `g`, and a positive
  `f` with `(not f)` already at REASONING. REASONING-layer guards
  mirror the P1.4 detector so a cross-layer `(X, ¬X)` never
  false-kills a live hypothesis.
- Files touched: new `inference/lookahead.py`; `hypgen.py` filter
  wiring; the 2-arg `sibling-exclusive` rewrite in
  `examples/zebra2.ein`, the five `examples/branching/` demos,
