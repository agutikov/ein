# S1.5.3 — Symmetry detection placeholder

**Phase:** P1.5 (hypothesis loop).
**Estimate:** 2-3 days.
**Depends on:** S1.5.1 + S1.5.2.
**Resolves question:** [M1 Q6 — symmetry pre or post trace](../../open_questions.md#q6--symmetry-breaking).

## Context

The Zebra walkthrough contains a "doesn't matter which way the
houses are numbered, only the order matters" remark
([idea 08 target trace](../../../docs/ideas/08-human-style-deductive-trace.md#the-target-trace-paraphrased)).
A solver that explores both numbering directions doubles the search
and produces a trace twice as long. The fix is symmetry breaking.

Per [M1 Q6](../../open_questions.md#q6--symmetry-breaking):
**ship (a) pre-trace canonicalisation with a `(symmetry-class …)`
escape-hatch annotation** so the trace planner can later label or
suppress.

## Tasks

### Task T1.5.3.1 — Detect declared symmetries

The ontology IR gains:

```ein
(symmetry house-numbering
  :group "S2"           ;; flip orientation
  :on House
  :note "doesn't matter which way the houses are numbered")
```

Loader registers a `Symmetry(group, scope, note)` value. For M1 we
support only `S2` (reflection); larger symmetry groups (`S_n`,
rotations) are tracked as a future need but not implemented.

### Task T1.5.3.2 — Canonicalisation hook in `hgen`

Before generating hypotheses (S1.5.1 T1.5.1.2), apply declared
symmetries: if the relevant orbit hasn't been broken yet, pin one
choice as the canonical orientation. Concrete Zebra example: if no
`House_i` has a fixed position yet, decree `pos(House_1) = 1` (or
equivalently `pos(House_5) = 5` — only one suffices).

Pinning emits a fact with `Provenance(kind='symmetry', group='S2')`
so the trace planner can label it as a "without loss of generality"
step.

### Task T1.5.3.3 — Trace annotation

The search tree gets a per-node `symmetry_break: SymmetryRef | None`
field, populated at the canonicalisation step. P1.6 reads it.

### Task T1.5.3.4 — Tests

`tests/engine/test_symmetry.py`:

- A toy 2-house puzzle with the declared `S2` symmetry — solver
  explores half as many branches as without it.
- The trace records the WLOG step exactly once.
- Disabling the symmetry declaration doubles the branch count.

### Task T1.5.3.5 — Document the escape hatch

`docs/symmetry.md`: a half-page on what's supported (`S2`), what
isn't (general groups), and how to plug in a richer detector later.
The point is to acknowledge the limitation explicitly, not pretend
the engine handles all symmetries.
