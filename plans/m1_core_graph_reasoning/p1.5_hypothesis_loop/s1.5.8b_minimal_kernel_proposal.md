# S1.5.8b — Minimal-kernel proposal for domain-elimination

**Phase:** P1.5 (hypothesis loop).
**Type:** *meta-stage* — alternative to
[S1.5.8a](s1.5.8a_relation_projections_design.md); no code
ships unless this design is adopted.
**Status:** open, 2026-05-23.
**Depends on:** [S1.5.7](s1.5.7_back_prop_unconditional.md)
shipped (re-saturation + return-on-derived-positive — the
mechanism the new rules fire inside).
**Blocks:** [S1.5.8](s1.5.8_totality_domain_elimination.md)
implementation, *in lieu of* S1.5.8a.
**Implements idea:**
[idea 06 — inference-rule completeness](../../../docs/ideas/06-inference-rules-completeness.md);
[idea 08 — human-style trace](../../../docs/ideas/08-human-style-deductive-trace.md).

## Why this stage exists

[S1.5.8a](s1.5.8a_relation_projections_design.md), the design
stage drafted earlier today, lands several new pieces of
machinery in the Python kernel: a `StructuralScan` opcode for
the matcher; a kb-aware `only-remaining` predicate that walks
`_facts_by_relation["not"]`; loader-time auto-derivation of
`(arg-type R i T)` / `(arity R n)` facts; a `:on` / `:to`
keyword sugar layer for projection properties; and a
direct-parent-types constraint imported from the typecheck rule.

This stage asks the dual question: **what is the minimum kernel
delta that lets S1.5.8 ship as a small block of stdlib ein, with
all the elimination logic expressed in ein rules rather than
Python code paths?**

The motivation isn't elegance alone. M1's own rules-of-thumb
already point this way:
- **"Generic > syntactic when both work"** ([M1 README](../README.md#design-rules-of-thumb-load-bearing-across-all-m1-phases)).
- **"Rules are graph rewrites"** — the engine fires *named
  rules*, not bespoke Python branches.
- **"Property tags are rule-application facts"** — not kwargs
  on declarations, not loader-special-cases.
- **"Syntax should be as protective as possible"** — fewer
  kernel forms, more visible behaviour.

The 2026-05-23 design discussion surfaced six new vocabulary
items in the S1.5.8a / minimal-kernel sketch: `number-atoms`,
`arg-type`, `is-a*`, `forall`, `absent`, `arity`. This stage
audits each, identifies which need kernel support and which
collapse to stdlib ein, and proposes the smallest kernel that
covers domain-elimination plus the relation-argument typecheck.

## Vocabulary audit

### `number-atoms` — already in grammar; semantics-as-atoms

`ein.py/src/ein_bot/ir/grammar.lark` already accepts `INT`
(`/-?[0-9]+/`) as a valid atom inside a fact / pattern (see
[grammar L181](../../../ein.py/src/ein_bot/ir/grammar.lark)).
Today the parser produces Python `int` values for them. The
matcher's identity comparisons (`==`) and the existing `eq` /
`neq` predicates handle them correctly because Python `int`
equality coincides with atom equality for non-arithmetic use.

The user's concern — "ok if it's not a number, just a string" —
is correctly answered by **never introducing arithmetic
primitives**. `0` and `1` in `(arg-type ?R 0 ?T1)` are
**positional identifiers**: their printed form happens to be
digits, but the engine never adds, subtracts, compares-by-order,
or otherwise manipulates them numerically. Only `eq` / `neq`
fire. The atom `0` is to the atom `1` exactly as `Red` is to
`Blue` — distinct identifiers with no further structure.

#### Sub-tasks

- **a.** No grammar change needed. Confirm the parser doesn't
  silently widen `INT` to numeric semantics in `predicates.py` —
  audit `eq` / `neq` to confirm they're identity comparisons
  only.
- **b.** Document in
  [`docs/kernel/ir/03-ein-lang/01_grammar.md`](../../../docs/kernel/ir/03-ein-lang/01_grammar.md):
  digit-leading atoms are identifiers; arithmetic predicates
  (`<`, `+`, `*`, …) are deliberately not provided. M2 / M3
  may revisit when puzzle classes need numeric reasoning.
- **c.** Forward-pointer to F4 Q33 (predicate primitives) —
  when ordered / arithmetic predicates become needed, this is
  the seam.

### `arg-type` — derived stdlib relation, not a kernel feature

`(arg-type ?R ?i ?T)` is "relation `?R` declares type `?T` at
argument position `?i`". A *stored fact* in the KB, in the
ontology layer, derived from the puzzle's `(relation R T1 T2)`
declaration by two stdlib rules:

```lisp
(rule reflect-arg-type-0 ()
  :match  (relation ?R ?T1 ?T2)
  :assert (arg-type ?R 0 ?T1)
  :priority 100)

(rule reflect-arg-type-1 ()
  :match  (relation ?R ?T1 ?T2)
  :assert (arg-type ?R 1 ?T2)
  :priority 100)
```

No kernel change: `(relation ?R ?T1 ?T2)` is already a fact in
the ontology layer (zebra2's `(ontology …)` block contains
exactly these), and the higher-order matcher already binds
relation-name vars (see `(rule functional (?R) :match (?R ?a ?b))`
in every demo). Reflection emerges from existing machinery.

This avoids S1.5.8a's "loader auto-derives `arg-type` /
`arity`" path. The kernel learns nothing new; the puzzle's
relation declarations are simply re-read by stdlib rules at
saturation time.

#### Sub-tasks

- **a.** Ship the two reflection rules in the M1 stdlib slot
  (inline in each puzzle until P1.8 imports).
- **b.** N-ary note: for arity > 2 the binary reflection rules
  miss positions 2..n-1. When variadic match (`?args...`)
  lands per Q33, a single generic rule replaces these two.
- **c.** Forward-pointer in
  [`docs/kernel/ir/02_rules.md`](../../../docs/kernel/ir/01-ein-graph/02_rules.md)
  to F5 / P1.8: reflection is a special case of "rules read
  the ontology layer". F5 (rules-as-data) extends the pattern.

### `is-a*` and `(closure …)` — two-parallel-relations facility

#### When two parallel relations are the right answer

If a puzzle wants `(transitive R)` and merges everything back
into `R`, R becomes its own closure after saturation — that's
the simple case. But sometimes a puzzle wants **both granularities
of the same hierarchy** present in the KB at once: a direct
relation `R` that other rules can match without ancestor noise,
*and* a transitively-closed `R*` for rules that want ancestor
reach.

zebra2's `(transitive is-a)` is intentionally omitted today
because the existing `sibling-exclusive is-a` rule would
over-fire under transitive closure (~1300 spurious negatives:
every leaf shares root `T` as an ancestor, every leaf pair
appears as a "sibling"). The two-parallel-relations design
keeps `is-a` direct (sibling-exclusive stays correct) while
making `is-a*` available for whoever wants transitive ancestor
reach.

M1's zebra2 under B1 doesn't actually use this — all relations
are declared at direct-parent types, so direct `is-a` suffices.
But the facility is small (4 rules + 2 grammar chars) and the
forward-compat option value is real, so ship it as **optional
library** alongside the 6 M1-blocking rules.

#### The user-facing declaration

```lisp
(ontology
  (relation is-a T T)         ; direct, kept non-transitive
  (closure is-a is-a*))       ; ONE fact, expands to everything below
```

One declaration. Internally, four library rules (T1.5.8b.3.b)
expand `(closure is-a is-a*)` into the full machinery:
- `(relation is-a* T T)` — `is-a*` declared with same signature
- `(transitive is-a*)` — `is-a*` is transitive
- `(subrel is-a is-a*)` — every `is-a` edge lifts to `is-a*`
- the `subrel` rule fires on each `is-a` fact, populating
  `is-a*`
- the existing `transitive` rule then saturates `is-a*`

After fixpoint: `is-a` holds only direct parent edges (~30 in
zebra2); `is-a*` holds the full transitive closure (~90 in
zebra2).

#### Multi-emission decomposition — why 4 rules, not 1

The natural single-rule encoding would be:

```lisp
;; Does NOT compile — :assert is single-fact in M1.
(rule closure (?R ?R*)
  :match  (relation ?R ?a ?b)
  :assert (and (relation ?R* ?a ?b)
               (transitive ?R*)
               (subrel ?R ?R*)))
```

M1's `:assert` accepts one fact-shape only (verified in
`compile.py:346 _lower_assert`). To emit three setup facts
from one trigger, decompose into three always-active rules
that share `(closure ?R ?R*)` as their *premise* (not their
activator name):

```lisp
;; Three setup rules — all activator-empty `()`, all read the
;; trigger fact as a premise. One (closure is-a is-a*) fact in
;; the puzzle fires all three.
(rule closure-decl-relation ()
  :match  (and (closure ?R ?R*) (relation ?R ?a ?b))
  :assert (relation ?R* ?a ?b)
  :priority 100
  :why    "{?R*} inherits {?R}'s signature")

(rule closure-emit-transitive ()
  :match  (closure ?R ?R*)
  :assert (transitive ?R*)
  :priority 100
  :why    "{?R*} declared transitive")

(rule closure-emit-subrel ()
  :match  (closure ?R ?R*)
  :assert (subrel ?R ?R*)
  :priority 100
  :why    "{?R} ⊆ {?R*}")

;; And the lift rule that does the actual edge transfer:
(rule subrel (?Sub ?Super)
  :match  (?Sub ?a ?b)
  :assert (?Super ?a ?b)
  :priority 100
  :why    "{?Sub} ⊆ {?Super}: lift edge")
```

4 rules total. If M1 ever wants single-rule multi-emission as
an idiom (3+ patterns demanding it), the kernel addition
`(and …)` in `:assert` collapses the three `closure-emit-*` /
`closure-decl-*` rules into one — a clean refactor. Until
then, decomposition.

#### Why three rules for emission instead of one trampoline fact

An alternative: emit a single `(closure-active …)` driver
fact, then have ordinary rules cascade off it. That's the same
4-rule count with an intermediate fact for no gain. The
3-rules-share-premise pattern is the cleaner shape.

#### Grammar extension — `*` in identifier tail

`is-a*` doesn't parse under the current SYMBOL regex
(`[A-Za-z][A-Za-z0-9_-]*`). One-char extension:

```diff
- SYMBOL:   /(?!(?:instance|not|and|or|neq|relation|a-priori)\b)[A-Za-z][A-Za-z0-9_-]*/
+ SYMBOL:   /(?!(?:instance|not|and|or|neq|relation|a-priori)\b)[A-Za-z][A-Za-z0-9_*-]*/
- VAR:      /\?[A-Za-z][A-Za-z0-9_-]*/
+ VAR:      /\?[A-Za-z][A-Za-z0-9_*-]*/
```

`*` is purely a character in names; no Kleene / arithmetic
semantics. `is-a*` is to `is-a` as `Old_Gold` is to `Old` —
distinct identifiers sharing a prefix. No tokenisation
ambiguity (S-expression whitespace/paren delimiters; INT /
RANGE / STRING have non-overlapping leading chars).

#### Sub-tasks

- **a.** Ship the 4 closure-facility rules (`subrel` +
  `closure-decl-relation` + `closure-emit-transitive` +
  `closure-emit-subrel`) as **optional** library pieces in
  the M1 stdlib block. zebra2 doesn't activate them; they
  cost zero firings on puzzles that don't declare
  `(closure …)`.
- **b.** Extend SYMBOL / VAR regex per the diff above (2-char
  grammar change). Test: `(relation is-a* T T)` and `?R*`
  round-trip through the parser.
- **c.** Document the closure idiom in
  [`docs/kernel/ir/01-ein-graph/02_rules.md`](../../../docs/kernel/ir/01-ein-graph/02_rules.md):
  user-facing form is one `(closure R R*)` declaration;
  under the hood it expands to the 4-rule decomposition.
- **d.** Forward-pointer to a future kernel option:
  `(and …)` in `:assert` would collapse the 3
  `closure-emit-*` / `closure-decl-*` rules into one
  multi-fact rule. ~20 LOC kernel addition. Worth it iff 3+
  patterns demand it; for just closure, decomposition wins.

### `forall ?bound (guard) (body)` — **the one substantive kernel addition**

A guarded universal premise. Semantics:

> For every binding of `?bound` such that `guard` is satisfied,
> `body` is satisfied.

Truth conditions:
- For each `?bound` binding that satisfies `guard`, the
  matcher evaluates `body` (which is itself a conjunction of
  premises) with `?bound` substituted in.
- The forall premise is **true** iff every such `?bound`
  binding yields a satisfied body.
- Empty guard-bindings: vacuously **true** (no `?bound` to
  check).
- `?bound` does not escape — it is local to the forall.
- Variables that appear free in `guard` or `body` and are NOT
  `?bound` must already be bound by an outer premise (safety).

This is the construct that lets us express "every other value
of type `?VT` for object `?a` is denied under `?R`" without
inventing `only-remaining` as a Python predicate. Used by:

- `domain-elimination` — fires when exactly one survivor.
- `no-room-left` — fires when zero survivors (contradiction).

#### Example — happy-parent

A small standalone rule, isolated from the
`domain-elimination` setup so the `forall` premise is the
only moving part: assert `(happy-parent ?p)` whenever every
child of `?p` is happy.

```lisp
(rule happy-parent ()
  :match  (and (person ?p)
               (forall ?c
                 (child-of ?c ?p)
                 (happy ?c)))
  :assert (happy-parent ?p)
  :priority 200
  :why    "all of {?p}'s children are happy")
```

Reading: `?p` is bound by the outer `(person ?p)` premise; the
forall iterates `?c` over the guard `(child-of ?c ?p)` (which
uses the already-bound `?p`); the body `(happy ?c)` is checked
for each `?c`. The rule fires iff *every* such `?c` satisfies
the body.

Trace against this KB:

```lisp
(person Alice)
(child-of Bob   Alice)
(child-of Carol Alice)
(happy Bob)
(happy Carol)
```

1. Matcher binds `?p = Alice` from `(person ?p)`.
2. forall enumerates `?c` over `(child-of ?c Alice)` →
   bindings `{?c = Bob, ?c = Carol}`.
3. For each, the body `(happy ?c)` is matched:
   - `?c = Bob`   → `(happy Bob)`   ∈ KB ✓
   - `?c = Carol` → `(happy Carol)` ∈ KB ✓
4. forall passes.
5. Rule fires; KB gains `(happy-parent Alice)`.

Edge cases:

- **Add `(child-of Dave Alice)` with no `(happy Dave)`** — the
  forall fails on `?c = Dave` (body unsatisfied); the rule
  does *not* fire; Alice is correctly not yet a
  happy-parent. If `(happy Dave)` later lands during
  saturation, the rule re-checks and fires.
- **No children at all** — forall over an empty enumerator is
  vacuously true (classical ∀ reading); Alice qualifies as a
  happy-parent. If the modelling wants "must have at least
  one child", add an existential to the outer match:
  `(and (person ?p) (child-of ?some ?p) (forall …))`.
- **Multiple `(person …)` facts** — the rule fires
  independently for each, with its own forall evaluation.

The same shape, with `?p` ↦ `?a`, `?c` ↦ `?v_other`, and
`(happy ?c)` ↦ `(not (?R ?a ?v_other))`, is how
`domain-elimination` reads "every other value is denied" in
[the stdlib block](#the-stdlib-ein-block).

#### Implementation sketch

Mirrors `NegativeGuard` in `inference/compile.py` /
`inference/match.py`:

```python
# compile.py — new opcode
@dataclass(frozen=True)
class ForallGuard:
    bound: str                              # the ?bound var name
    guard_steps: tuple[object, ...]         # compiled guard sub-plan
    body_steps:  tuple[object, ...]         # compiled body sub-plan

# match.py — new arm in _run_steps
if isinstance(step, ForallGuard):
    all_ok = True
    for guard_b, guard_premises in _run_steps(step.guard_steps, bindings, premises, kb):
        # For each guard-satisfying binding extension, check body.
        body_ok = any(True for _ in _run_steps(step.body_steps, guard_b, guard_premises, kb))
        if not body_ok:
            all_ok = False
            break
    if all_ok:
        yield from _run_steps(rest, bindings, premises, kb)
    return
```

~30 LOC including a compile-time safety check that `?bound`
appears positively in `guard`. The guard's bindings must be
finite for termination — in practice that means `guard`
contains at least one fact-scan over `?bound`, which the
existing `Scan` already enforces.

#### Sub-tasks

- **a.** Add `ForallGuard` opcode to `compile.py`.
- **b.** Add `_compile_premise` arm for `(forall ?bound
  (guard) (body))` heads. Reject at compile time if `?bound`
  doesn't appear in `guard`'s positive subgoals.
- **c.** Add `_run_steps` arm in `match.py`.
- **d.** Tests under
  `ein.py/tests/inference/test_forall.py`: empty domain
  (vacuous-true), single-element domain, all-pass, one-fail,
  body referencing outer bindings, nested forall.
- **e.** Document under
  [`docs/kernel/ir/03-ein-lang/01_grammar.md`](../../../docs/kernel/ir/03-ein-lang/01_grammar.md)
  the surface form, and under
  [`docs/kernel/inference/`](../../../docs/kernel/inference/)
  the matcher semantics.

### `absent P` — explicit NAF; **subsumes the current `(not P)` NAF default**

Today's M1 matcher special-cases `(not P)` in `:match` as
negation-as-failure (`compile.py:210` `head_name == "not"` →
`NegativeGuard`). **Zero current rules use this**: a grep across
`examples/zebra2.ein` and `examples/branching/*.ein` finds
56 `:match` lines and zero `(not P)` premises in them. The
special case is dead weight.

The proposal: **remove the special case**, freeing `(not P)`
in `:match` to mean what it means everywhere else — a fact
pattern matching a stored `(not P)` fact — and introduce
`(absent P)` as the explicit NAF.

This matters because `domain-elimination` needs to match
*stored* `(not (R a v))` facts (the ones `type-exclusivity`
writes), not test "is `(R a v)` currently underived". The two
are different: a stored `(not P)` is a positive commitment;
NAF on `P` is the absence of a positive — which is wrong inside
a hypothesis-branched search where "not yet derived in this
branch" doesn't mean "false".

Under the proposed semantics:
- `(not P)` in `:match` — matches a stored fact `(not P)`. Same
  as every other pattern.
- `(absent P)` in `:match` — fires iff `P` matches no fact in
  the KB (the same `NegativeGuard` machinery, renamed).
- `(not P)` in `:assert` — unchanged; writes the negative fact.

#### Migration

Zero rules to migrate. The dead-weight NAF in `(not P)` had no
in-tree users.

The new typecheck rule (T1.5.8a.3, or its T1.5.8b.5 successor
below) DOES need NAF — for closed-world is-a — and explicitly
uses `(absent (is-a ?a ?Dom))`. Explicit is better than
default-NAF: it flags the closed-world assumption at the use
site.

#### Sub-tasks

- **a.** Delete the `head_name == "not"` → `NegativeGuard` arm
  in `compile.py:210`. `(not P)` in :match now compiles as an
  ordinary fact pattern with relation name `"not"` and the
  inner pattern as nested arg.
- **b.** Rename `NegativeGuard` → `AbsentGuard`. Register
  `absent` as the head keyword in `_compile_premise`.
  Semantics unchanged from today's NAF — sub-plan tried, parent
  continues iff sub-plan yields zero.
- **c.** Update `match.py`'s arm to match the rename. The
  truth-condition code is unchanged.
- **d.** Tests: `test_absent.py` — current NAF tests
  (if any) migrate; add a "stored (not P) does NOT satisfy
  (absent P)" test to pin the semantic distinction.
- **e.** Document in
  [`docs/kernel/inference/`](../../../docs/kernel/inference/) —
  `(not P)` is the fact; `(absent P)` is the absence of `P`;
  the two are independent.

### `arity` — **not needed** for M1

`(arity R n)` was proposed alongside `arg-type` as a
loader-derived reflection fact. The S1.5.8 / S1.5.8b rule set
**never matches on it**. Drop. When n-ary support lands (Q33
followup), one stdlib rule re-introduces it.

#### Sub-tasks

- **a.** Don't ship arity in the M1 stdlib.
- **b.** Forward-pointer in
  [`docs/kernel/ir/01-ein-graph/02_rules.md`](../../../docs/kernel/ir/01-ein-graph/02_rules.md):
  arity reflection lives behind Q33 variadic; not in M1.

## The stdlib ein block

The S1.5.8 implementation ships as exactly this block, inlined
into each puzzle that wants domain-elimination (until P1.8's
import mechanism lands):

```lisp
(rules
  ;; ── Reflection: arg-type from (relation …) declarations.
  (rule reflect-arg-type-0 ()
    :match  (relation ?R ?T1 ?T2)
    :assert (arg-type ?R 0 ?T1)
    :priority 100
    :why    "{?R} arg 0 is typed {?T1}")

  (rule reflect-arg-type-1 ()
    :match  (relation ?R ?T1 ?T2)
    :assert (arg-type ?R 1 ?T2)
    :priority 100
    :why    "{?R} arg 1 is typed {?T2}")

  ;; ── Relation-argument typecheck — closed-world is-a NAF.
  (rule typecheck-arg-0 ()
    :match  (and (relation ?R ?Dom ?Ran)
                 (?R ?a ?b)
                 (absent (is-a ?a ?Dom)))
    :assert (false)
    :priority 110
    :why    "{?R} arg 0 expects {?Dom}; {?a} is not a {?Dom}.")

  (rule typecheck-arg-1 ()
    :match  (and (relation ?R ?Dom ?Ran)
                 (?R ?a ?b)
                 (absent (is-a ?b ?Ran)))
    :assert (false)
    :priority 110
    :why    "{?R} arg 1 expects {?Ran}; {?b} is not a {?Ran}.")

  ;; ── Domain elimination — every-other-excluded ⇒ force the survivor.
  ;; ── `(not (?R ?a ?v_other))` matches the STORED neg fact written
  ;; ── by `type-exclusivity` (semantics: `(not P)` in :match is now
  ;; ── a stored-fact pattern, not NAF).
  (rule domain-elimination ()
    :match  (and (functional  ?R 0 1)
                 (total       ?R 0)
                 (arg-type    ?R 0 ?OT)
                 (arg-type    ?R 1 ?VT)
                 (is-a ?a ?OT)
                 (is-a ?v ?VT)
                 (forall ?v_other
                   (and (is-a ?v_other ?VT)
                        (neq ?v_other ?v))
                   (not (?R ?a ?v_other))))
    :assert (?R ?a ?v)
    :priority 400
    :why    "every {?VT} except {?v} excluded for {?a} — therefore ({?R} {?a} {?v}).")

  ;; ── All-excluded ⇒ contradiction.
  (rule no-room-left ()
    :match  (and (functional  ?R 0 1)
                 (total       ?R 0)
                 (arg-type    ?R 0 ?OT)
                 (arg-type    ?R 1 ?VT)
                 (is-a ?a ?OT)
                 (forall ?v
                   (is-a ?v ?VT)
                   (not (?R ?a ?v))))
    :assert (false)
    :priority 110
    :why    "{?a} has every {?VT} excluded — no value left.")

  ;; ── Closure facility (OPTIONAL — zebra2 doesn't activate it).
  ;; ── User writes one (closure R R*) declaration; the four
  ;; ── rules below expand it into the full machinery so R
  ;; ── stays direct while R* becomes the transitive closure
  ;; ── of R. Requires the *-in-identifier grammar extension
  ;; ── (T1.5.8b.2.d).
  ;; ──
  ;; ── These rules fire iff (closure …) is declared; otherwise
  ;; ── zero cost.

  (rule subrel (?Sub ?Super)
    :match  (?Sub ?a ?b)
    :assert (?Super ?a ?b)
    :priority 100
    :why    "{?Sub} ⊆ {?Super}: lift edge")

  (rule closure-decl-relation ()
    :match  (and (closure ?R ?R*) (relation ?R ?a ?b))
    :assert (relation ?R* ?a ?b)
    :priority 100
    :why    "{?R*} inherits {?R}'s signature")

  (rule closure-emit-transitive ()
    :match  (closure ?R ?R*)
    :assert (transitive ?R*)
    :priority 100
    :why    "{?R*} declared transitive")

  (rule closure-emit-subrel ()
    :match  (closure ?R ?R*)
    :assert (subrel ?R ?R*)
    :priority 100
    :why    "{?R} ⊆ {?R*}"))
```

**10 rules** — 6 M1-blocking for zebra2 (reflection × 2,
typecheck × 2, `domain-elimination`, `no-room-left`) + 4
optional closure-facility (`subrel`,
`closure-decl-relation`, `closure-emit-transitive`,
`closure-emit-subrel`). The closure block fires only when
the puzzle declares `(closure R R*)`; zebra2 doesn't.

Each rule is named, traceable, and fires through the existing
saturator. No Python-side branching. Multi-emission from one
trigger is expressed as 3 always-active rules sharing the
trigger as a premise — see [(b) decomposition rationale](#multi-emission-decomposition--why-4-rules-not-1).

## Worked example — `domain-elimination` firing on zebra2

Setup (post-B1 zebra2):
- `(relation house-color House Color)`
- `(functional house-color 0 1)` + `(total house-color 0)`
- Five Houses, five Colors, all `(is-a …)` direct.
- Type-exclusivity has stored
  `(not (house-color House-1 Red))`,
  `(not (house-color House-1 Green))`,
  `(not (house-color House-1 Ivory))`,
  `(not (house-color House-1 Blue))`.

Saturation trace:
1. **Priority 100**: `reflect-arg-type-0/1` fire on
   `(relation house-color House Color)` → KB gains
   `(arg-type house-color 0 House)` +
   `(arg-type house-color 1 Color)`.
2. **Priority 110**: `typecheck-arg-0/1` scan every
   `(house-color ?a ?b)` fact. For `(house-color House-1 Yellow)`,
   the `absent (is-a House-1 House)` premise fails (the fact is
   stored), so the rule is silent. ✓
3. **Priority 400**: `domain-elimination`'s outer pattern
   matches `?R = house-color, ?a = House-1, ?v = Yellow,
   ?OT = House, ?VT = Color`. The forall iterates `?v_other`
   over `{Red, Green, Ivory, Blue}` (the four Color instances
   ≠ Yellow). For each, the body
   `(not (house-color House-1 ?v_other))` matches the stored
   neg → body passes. forall succeeds. **Rule fires**:
   `:assert (house-color House-1 Yellow)`.

Trace text in the markdown trace:
> **domain-elimination** — every Color except Yellow excluded
> for House-1 — therefore (house-color House-1 Yellow).

This is the "Therefore the first house is yellow" line idea-08
asks for, produced by a named saturation rule firing with the
4 negatives + the Yellow `(is-a)` fact as explicit premises.

## Comparison with S1.5.8a

| concern                    | S1.5.8a                                                   | S1.5.8b                                              |
|----------------------------|-----------------------------------------------------------|------------------------------------------------------|
| New kernel opcodes         | `StructuralScan`                                          | `ForallGuard`, `AbsentGuard` (rename)                |
| New kernel predicates      | `only-remaining` (kb-aware, binding-yielding)             | none                                                 |
| Loader-time auto-derivation | `arg-type` + `arity`                                      | none — reflection via stdlib rules                   |
| Sub-domain concept         | Rejected (B1) but residual in `?OT`/`?VT` plumbing        | Gone entirely; `forall` reads what it iterates over  |
| Property facts             | `:on (i) :to (j)` keyword sugar                           | Positional: `(functional R 0 1)`, `(total R 0)`      |
| `(not P)` in :match        | NAF (default) + StructuralScan for stored-neg             | Uniform fact-match; NAF moved to explicit `absent`   |
| Migration cost             | All existing `(not P)` :match uses kept as NAF            | Zero migration (no existing `(not P)` :match uses)   |
| Rule count                 | ~2 rules in stdlib (domain-elim, no-room-left)           | 6 M1-blocking + 4 optional closure-facility (10 total) |
| Trace fidelity             | Custom firing wrapping `StructuralScan`                   | Named-rule firings at every band; vanilla saturator  |
| N-ary path                 | `StructuralScan` already kb-aware, generalises easily      | Variadic match + `?args...` predicate needed (Q33)   |
| Code added to kernel       | ~150 LOC (scan + predicate + loader + sugar)              | ~30 LOC (forall opcode)                              |

S1.5.8a is *more general at the kernel* (the scan opcode admits
arbitrary structural predicates); S1.5.8b is *more general at
the rule layer* (forall composes with existing primitives). The
trade is "kernel power" vs "rule expressiveness".

For M1's needs — Zebra plus one trace-fidelity criterion —
S1.5.8b is the smaller, more declarative answer. For M2+ where
the rule set widens, the forall pattern is a familiar idiom
(Datalog with safe-negation does exactly this) while
StructuralScan is bespoke.

## Open questions parked here

- **Q-S1.5.8b.1 — `is-a*` re-entry condition.** Direct-parent
  types cover B1 zebra2. The first puzzle that legitimately
  needs ancestor-typed relations (e.g., a relation typed against
  `Attribute` because it must accept Color or Nationality
  arguments interchangeably) re-introduces `is-a*` as a 2-rule
  stdlib closure. M1 doesn't need it; M2 may.
- **Q-S1.5.8b.2 — Compile-time safety vs runtime fail for
  `forall` unsafety.** A forall whose `?bound` isn't grounded
  by `guard` has infinite bindings (every atom in the universe).
  Compile-time reject is cleaner; runtime "domain too large"
  fail is more permissive. Recommend **compile-time reject** —
  if `?bound` doesn't appear positively in `guard`'s scan
  arguments, the compiler raises and the puzzle fails to load.
- **Q-S1.5.8b.3 — Positional `(functional R 0 1)` vs sugar
  `(functional R :on (0) :to (1))`.** Positional is
  matcher-trivial. The keyword sugar S1.5.8a proposed is
  reader-friendly but adds parser complexity. Recommend
  **positional for M1**; keyword sugar as a future P1.8 polish
  if user feedback wants it.
- **Q-S1.5.8b.4 — Activator-fact storage layer.** Where do
  `(functional R 0 1)` / `(total R 0)` live — ontology layer
  or facts layer? `(symmetric …)` / `(transitive …)` already
  live in `(ontology …)` blocks. Recommend **ontology** for
  consistency; the typecheck and domain-elim rules read them
  the same way regardless.
- **Q-S1.5.8b.5 — `(not (?R ?a ?v))` semantics in :assert.**
  Today's behaviour stores a `(not …)` fact. Unchanged by this
  proposal. Confirm no rule writes `(not P)` expecting NAF
  semantics — grep shows no such use.
- **Q-S1.5.8b.6 — Stratification of typecheck and reflection.**
  Reflection (priority 100) must complete before
  typecheck/domain-elim (110, 400) reads `arg-type`. The
  banded saturator already enforces this. Confirm by test
  fixture.
