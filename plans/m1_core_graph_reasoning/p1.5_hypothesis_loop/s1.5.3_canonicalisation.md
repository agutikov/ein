# S1.5.3 — Canonical fact-list hash + state-dedup + alive-branch termination

**Phase:** P1.5 (hypothesis loop).
**Estimate:** 3-4 days.
**Depends on:** [S1.5.1](s1.5.1_saturate_branch.md),
[S1.5.2](s1.5.2_multilevel.md).
**Predecessor:** [S1.5.0 review](s1.5.0_review.md) §F2.

## Context

S1.5.2 builds a search **tree** with leaf verdicts (`solution` /
`dead`) and interior nodes stamped `open`. This stage:

1. **Promotes interior-node verdicts** via the alive-branch
   protocol — "interior node is solution iff ≥ 1 child path
   reaches a solution; dead iff all children are dead". The
   resulting verdict tree is the proof.
2. **Adds canonical-state hashing** so sibling branches that
   saturate to the same KB collapse to one node. The structure
   becomes a DAG, not a tree — same final state via different
   proof paths is one proof, not two.
3. **Gates `solve()` return** on the alive-branch predicate —
   *all alive branches must be fully explored* before the engine
   declares a solution. Returning early on the first
   goal-matching branch is unsound: a deeper hypothesis level
   might surface a contradiction that invalidates the apparent
   solution.

Per [S1.5.0 §F2.3](s1.5.0_review.md#f23-alive-branch-termination):

> Termination must explore every alive branch to exhaustion. A
> branch is **dead** iff `ContradictionDetector` finds a same-
> layer `(X, (not X))` pair. A branch is **fully-explored** iff
> dead, OR has no more hypotheses to generate AND no
> contradiction surfaced. The puzzle is **solved** iff exactly
> one fully-explored, non-dead branch exists AND every other
> branch is dead.

## Tasks

### Task T1.5.3.1 — Canonical fact-list hash

`src/ein_bot/inference/hypothesis.py` (or a new
`inference/canon.py` if the hypothesis module grows large):

```python
def state_hash(kb: KnowledgeBase) -> int:
    """Order-insensitive hash of the KB's fact list.

    Hashes ONLY the facts (S1.5.0 §F2.1). Ontology, rules, and
    query are constant across branches; the trace is intentionally
    per-branch (the dedup target).

    Args order INSIDE each fact tuple is preserved — (right-of A B)
    and (right-of B A) are different facts with different semantic
    meanings. Only the OUTER list of facts is sorted to get a
    canonical order across the set.

    Nested-Fact args (Q40) hash recursively via Fact.__hash__,
    which Python already wires through tuple equality.
    """
    return hash(tuple(sorted(
        (f.layer.value, f.relation_name, _hashable_args(f.args))
        for f in kb.facts
    )))


def _hashable_args(args):
    return tuple(
        _hashable_args(a.args) if isinstance(a, Fact) else a
        for a in args
    )
```

Cost: O(|facts| log |facts|) per hash. Zebra has ~200 facts
post-saturation → microseconds.

### Task T1.5.3.2 — State index + dedup at fork time

The `_TreeBuilder` from S1.5.2 gains a `_state_index: dict[int,
BranchId]`. When `_explore` is about to create a new child:

```python
def _explore(...):
    ...
    # NEW: hash the post-saturation state.
    h = state_hash(kb)
    if h in builder.state_index:
        # Already explored this exact state via a different path.
        # Don't recurse; point the parent at the existing node.
        existing_id = builder.state_index[h]
        builder.add_alias(parent_id=parent_id,
                          hypothesis=hypothesis,
                          target_id=existing_id)
        return existing_id

    node_id = builder.alloc()
    builder.state_index[h] = node_id
    ...
```

`builder.add_alias(...)` records that the parent has a child
*pointing into* an existing subtree, without re-exploring it. The
SearchTree's `nodes: dict[BranchId, SearchNode]` becomes a DAG
in storage — multiple parents can point to the same SearchNode.

Hash collisions: Python's `hash()` is 64-bit; collisions are
astronomically rare for Zebra-scale state counts (<10^5 distinct
states). If a future puzzle pushes against this, swap in SHA-256
on a deterministic byte-serialisation of the same tuple.

### Task T1.5.3.3 — Alive-branch verdict promotion

After `_explore` returns and the builder's leaf verdicts are set,
walk the tree bottom-up to settle interior-node verdicts:

```python
def _promote_verdicts(builder: _TreeBuilder, root: BranchId) -> None:
    """Bottom-up walk: interior nodes inherit from children.

    A node is:
    - `dead` if all its children are dead (no alive subtree below).
    - `solution` if any child path reaches a solution.
    - `ambiguous` if children include both solution and dead
      branches in a way that means GAPS mode would report
      multiple survivors.
    """
    def walk(nid: BranchId) -> Literal["solution", "dead"]:
        node = builder.nodes[nid]
        if not node.children:
            return node.verdict       # leaf: solution or dead
        child_verdicts = [walk(c) for c in node.children]
        if any(v == "solution" for v in child_verdicts):
            builder.set_verdict(nid, "solution")
            return "solution"
        builder.set_verdict(nid, "dead")
        return "dead"

    walk(root)
```

Aliased children (from dedup) consult the existing node's
already-settled verdict; cycles are impossible because
`state_index` keys are post-saturation states (each branch
strictly extends its parent's fact set, so two siblings' states
are either equal — and dedup'd — or strictly different).

### Task T1.5.3.4 — Termination gating in `solve()`

The S1.5.2 `solve()` calls `_promote_verdicts()` before extracting
solutions:

```python
def solve(kb, *, mode=None, max_depth=6):
    ...
    tree = builder.finalize()        # S1.5.2 shape
    _promote_verdicts(builder, tree.root)
    tree = builder.snapshot()        # immutable view

    if any(n.verdict == "open" for n in tree.nodes.values()):
        # max_depth cutoff — some interior nodes unresolved.
        return Ambiguity(
            branches=tuple(tree.solutions()),
            unresolved=tuple(n for n in tree.nodes.values()
                            if n.verdict == "open"),
        )

    # All nodes have terminal verdicts. Standard mode handling
    # from S1.5.2 applies — but now uses promoted verdicts.
    ...
```

The `Ambiguity` value gains an `unresolved` field so callers can
deepen if they want.

### Task T1.5.3.5 — Tests

`tests/inference/test_canonicalisation.py`:

- **state_hash determinism** — same KB hashes to the same value
  across calls; permuting fact insertion order doesn't change the
  hash; different layers count as different states.
- **state_hash distinguishes nested args** — `(not (R A B))` and
  `(not (R B A))` hash differently (the outer-set sort doesn't
  reorder inner args).
- **Dedup collapses equivalent siblings** — two hypotheses that
  saturate to the same KB produce one tree node (the second is
  aliased). Verify `len(tree.nodes) < tree-without-dedup-count`.
- **Verdict promotion: all-dead → dead** — interior node with
  three dead children inherits `dead`.
- **Verdict promotion: any-solution → solution** — interior node
  with one solution and two dead children inherits `solution`.
- **`solve()` doesn't return early** — set up a KB where the
  first hypothesis appears to solve but the *second* hypothesis
  on the same level surfaces a contradiction in the goal-matching
  branch. `solve()` returns `Contradiction`, not `Solution`.
- **max_depth → Ambiguity with `unresolved`** — verify the
  returned value includes the open nodes.
- **End-to-end Zebra** — `solve(zebra_kb, mode=Mode.SOLVE)`
  returns a `Solution` with `(co-located Norwegian Water)` and
  `(co-located Japanese Zebra)` in the KB. Tree-node count <
  some sane bound (target: < 500 with dedup).

## Open questions parked here

- **Q-S1.5.3.A — Alive-hyps in the state hash (added 2026-05-24).**
  Can two distinct paths through the search tree reach the same
  post-saturation KB but carry **different alive hypothesis sets**?
  Under M1's monotonic semantics (append-only KB, no fact retraction,
  rules don't depend on path history), the saturated KB IS a
  pure function of its fact set — so the alive set at exit should
  be a pure function of the KB as well, regardless of how the KB
  was reached. *If* that's true, the current state-hash dedup is
  sound. *If* not, the dedup is unsound for the [T1.5.4.5/8
  alive-inherit](s1.5.4_hypgen_improvements.md) optimisation and
  the fix is to include the alive set in the canonical hash:
  collapse only paths whose `(state, alive)` pair matches. Track
  in [P1.9 E22](../p1.9_hypothesis_loop_followups/README.md).
  Resolve before T1.5.4.8 ships in production-default mode, or
  document the invariant.
