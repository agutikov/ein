# S1.5.5 — Closure auto-inference via explicit rules

**Phase:** P1.5 (hypothesis loop).
**Estimate:** 1-2 days.
**Depends on:** [S1.5.4 T1.5.4.1](s1.5.4_hypgen_improvements.md#task-t1541--relation-closure-declarative-closed-r-activator)
(the `(closed R)` activator the rule's `:assert` head feeds into),
[S1.5.4 T1.5.4.4](s1.5.4_hypgen_improvements.md#task-t1544--config-head--config--top-level-form)
(the `:enable-auto-closure` config knob),
[S1.5.4a Part 1](s1.5.4a-fix-single-parent.md) (rule rename
`single-parent` → `functional` so the inference rule references
the canonical name).
**Split from:** the original T1.5.4.2 task body in
[S1.5.4](s1.5.4_hypgen_improvements.md).

## Context

Closure *recognition* (T1.5.4.1) doesn't care whether `(closed R)`
was authored by the puzzle or asserted by a rule firing — it just
reads `kb._facts_by_relation["closed"]`. S1.5.5 supplies the rule
that produces those firings automatically, so a puzzle author who
writes `(functional is-a)` doesn't *also* have to remember
`(closed is-a)`.

Why split from S1.5.4: the manual activator's empirical win is the
load-bearing claim (~25-30% of root hypotheses for zebra2 are
is-a noise that closure kills). Until that win is measured against
the demo suite, layering automatic derivation on top adds a
second variable to the experiment. Once S1.5.4 ships and T1.5.4.6
records the node counts, S1.5.5 lands as a pure ergonomic
improvement — the author can stop double-declaring.

## Tasks

### Task T1.5.5.1 — `infer-closure-from-functional` rule

The simplest closure rule: a **functional** relation (one whose
image is unique per slot-0 value) whose leaves are all already
declared is closed by construction — no new fact `(R x ?y)` can
be admitted without violating the functional constraint.

```lisp
;; in the puzzle's (rules …) block, or in the proto-library once
;; T1.7.2.5 ships:
(rule infer-closure-from-functional (?R)
  :match  (functional ?R)
  :assert (closed ?R)
  :why    "{?R} is functional and every leaf is declared with one
           parent — no further (?R …) facts are admissible."
  :priority 50)
```

The assertion fires at root-saturation time, *before* hypothesis
enumeration runs, so the generator sees a populated `(closed ?R)`
set on its first call.

Sub-tasks:

- **a.** Ship `infer-closure-from-functional` in the canonical
  rule library (the proto-library once
  [T1.7.2.5](../p1.7_bootstrapping_zebra/s1.7.2_dynamic_vs_hardcoded.md)
  lands; inline in each puzzle's `(rules …)` block until then).
  Note: relies on [S1.5.4a Part 1](s1.5.4a-fix-single-parent.md)
  having landed — the rule must reference `(functional ?R)`, not
  `(single-parent ?R)`.
- **b.** Acceptance test: load a zebra2-style puzzle that declares
  `(functional is-a)` but no explicit `(closed is-a)` — observe
  `(closed is-a)` in the REASONING layer after root saturation
  and zero `is-a` hypotheses at any depth.

### Task T1.5.5.2 — Config flag `:enable-auto-closure`

The rule lands behind a gate so the first turn-on is a controlled
experiment.

Sub-tasks:

- **a.** Add `enable_auto_closure: bool = False` to `SolverConfig`
  (T1.5.4.4). Default off — auto-inference can over-fire on rule
  libraries that don't expect it (see Q-S1.5.5.A).
- **b.** Hypothesis-generator side: nothing changes — T1.5.4.1's
  `_is_closed` reads `kb._facts_by_relation["closed"]` regardless
  of who wrote the fact.
- **c.** Saturator side: when the flag is false, skip
  `infer-closure-from-functional` (and any future closure
  inference rule) during the priority-banded saturation. Simplest
  shape: tag the rule with a category in its `(rule …)` body and
  the saturator filters by `solver_config.enable_auto_closure`.
- **d.** Acceptance: a zebra2 puzzle with `(functional is-a)` and
  `:enable-auto-closure true` shows `(closed is-a)` in REASONING;
  with `:enable-auto-closure false` it does not. Branch counts
  drop in the former and match S1.5.4 baseline in the latter.

### Task T1.5.5.3 — Richer closure rules (parked, open)

Richer closure conditions need primitives M1's matcher doesn't
yet have:

- *"Every leaf of `R` already has its required parent"* needs
  **negation-as-failure** or a **coverage count** predicate.
- *"Every (slot-0 instance, slot-1 type) pair has at least one
  fact"* needs an **aggregate-over-relation** predicate.

Neither is in M1; both are tracked under
[F7 rule induction §C — rule-set sufficiency](../../followups/f7_rule_induction.md)
for the long-term framing. M1 trusts `(functional ?R)` as a
sufficient (but not necessary) witness; mismatched ontologies
(functional declared but leaves unmentioned) fall back to
T1.5.4.1.c's sanity-check warning.

**Anti-pattern note.** If a future closure inference is tempting
to add as a Python branch in `_hypotheses_for` ("if relation is
foo then skip"), back up and write the rule instead. The engine
reading `(closed foo)` from REASONING is the canonical path; the
puzzle author or rule library decides what's closed. T1.5.4.2.d
(now this section) is the load-bearing convention.

## Acceptance

- The `infer-closure-from-functional` rule fires on a puzzle that
  declares `(functional is-a)` + `:enable-auto-closure true`,
  producing `(closed is-a)` in REASONING.
- Branch counts on the demo suite drop *strictly more* than the
  S1.5.4 baseline (auto-inference catches anything the author
  forgot to mark closed; in zebra2 itself the gain is small
  because the author already adds `(closed is-a)`, but a
  zebra-original-style puzzle without manual closure sees the
  full T1.5.4.6 collapse).
- Default-off behaviour: a puzzle without
  `:enable-auto-closure true` matches S1.5.4 baseline exactly
  (the rule doesn't fire).
- `pytest ein.py/tests/inference/test_auto_closure.py` ≥ 3 tests
  (rule fires when expected; doesn't fire when flag off; produces
  warning per T1.5.4.1.c if a closed relation gains a new fact
  later in saturation).

## Open questions

- **Q-S1.5.5.A — Default the flag on?** Once T1.5.5.1 + T1.5.4.1
  prove sound across the demo suite and the Zebra puzzle, is
  there a reason to *not* default `:enable-auto-closure` to true?
  The concern is libraries that mark a relation `(functional R)`
  without intending it as a closure signal (e.g. for SMT
  reduction in M3). Park until M3 has an opinion.
- **Q-S1.5.5.B — One rule per closure-witness, or one
  parameterised rule?** Future closure witnesses (cardinality,
  coverage) would each need their own rule under the M1 matcher.
  Once M2/M3 land a richer predicate library, consolidate to a
  single parameterised rule with `(closure-witness ?R ?witness)`
  premises. Park.
