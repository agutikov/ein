## Context

The "dead-hypothesis cache" is the `_negated_facts` set (Tier A
in S1.5.4's filter pipeline). When a branch dies *unconditionally*,
the engine should assert `(not h)` into the **parent** KB so the
next sibling — and every descendant on every level — sees `h`
excluded by the same O(1) Filter-C check. No separate cache, no
extra plumbing — the dead-hypothesis cache *is* the negation
index.

Two reasons this didn't ship inside S1.5.4:

1. **Soundness depends on a transitive premise walk.** The
   shallow read — *"if the unsat-core contains no
   `provenance.kind == 'hypothesis'` fact, the death is
   unconditional"* — is wrong when an unsat-core fact is itself
   *derived* from a hypothesis-kind premise via a rule firing
   (its own provenance is `kind = 'rule'`, but its `justified_by`
   chain terminates in a hypothesis). The safe version walks
   `Provenance.justified_by` transitively until a
   source/ontology/fact terminal or a hypothesis terminal is
   reached; unconditional iff *no* leaf is a hypothesis. This is
   E6 in [S1.5.4's parked-ideas catalog](s1.5.4_hypgen_improvements.md#conflict-driven-learning-satcdcl-inspired).
2. **The write is irreversible relative to the parent's KB.**
   Once `(not h)` is written into the parent, it persists across
   sibling iterations and into the parent's parent. If the
   unconditional read is wrong even once, the engine permanently
   excludes a valid hypothesis — a correctness bug masquerading
   as a performance improvement.

Defaulting the flag **off** in S1.5.7 means the first turn-on is
a controlled experiment: a single puzzle, a single
`solve(..., config=SolverConfig(enable_back_prop_unconditional=True))`
call, with `bench_solve --diff-trees` comparing the dead-leaf set
to the gated-off run. Any divergence in alive-branch verdicts is a
ship blocker for the default-on flip.

### Architecture shift — the consume loop

`_explore` today (`inference/hypothesis.py`) is a *static*
recursive descent: `_candidates_for(kb)` is read once, every alive
candidate becomes a child `SearchNode`, recursion goes strictly
downward. Back-prop breaks that shape. The moment a dead sibling
writes `(not h)` into the parent, the parent's *own* state has
changed mid-enumeration — new negatives shrink the candidate set,
and (once [S1.5.8](s1.5.8_totality_domain_elimination.md)'s
`domain-elimination` rule lands) new *positive* facts appear.
Solving stops being a single-pass b0-rooted tree walk and becomes
an **iterative process**: explore one branch, consume the facts it
back-propagates, repeat.

```text
explore(kb):
  loop:
    saturate(kb)                       # initial pass, then re-saturation
    if contradiction(kb): return dead
    candidates = _candidates_for(kb)   # _prune_alive re-filters vs _negated_facts
    if no candidates: return leaf-verdict(kb)
    progressed = False
    for h in candidates:
      result = try_branch(kb, h)
      if result is dead and is_unconditional_death(result):
        back-prop (not h) into kb            # T1.5.7.2
        record the dead child SearchNode
        progressed = True                    # kb gained a (not h) fact
    if progressed:
      continue loop          # re-saturate; domain-elimination etc. may fire
    # fixpoint: no sibling died unconditionally this pass —
    # the still-alive candidates are a genuine disjunction
    for h in alive candidates: spawn child node, recurse explore
    return interior-node
```

**Return on a derived positive.** When the re-saturation pass
derives a *non-hypothetical* positive fact for a slot that was open
— provenance that walks back to source/rule terminals only, *no*
hypothesis leaf (the same transitive test as T1.5.7.1) — that fact
is a **deduction, not a branch**. It is recorded as a firing on the
*current* node and the loop continues at the *same depth*: no child
`SearchNode`, no extra tree level. This is the zebra-walkthrough
shape — *"Therefore the first house is yellow"* immediately
followed by *"From this it follows … — saturation"*: a forced move
folds back into the current level's saturation, and only a genuine
disjunction ever spends a tree level. The `SearchTree` is still the
proof object (S1.5.0 §F), but its interior nodes now correspond to
*real choice points* only — the tree is shallower and reads like a
human trace.

## Tasks

### Task T1.5.7.1 — Transitive premise walk (E6)

Sound unconditional-vs-conditional read for a contradiction's
unsat-core.

#### Design

```python
def is_unconditional_death(
    kb: KnowledgeBase, unsat_core: frozenset[Fact]
) -> bool:
    """True iff no transitive premise in the unsat-core chain
    terminates at a hypothesis-kind fact.

    Walks each fact's Provenance.justified_by until reaching
    facts with kind in {'source', 'rule'} — for 'rule', recurses
    into the rule firing's premises. Memoised per fact so the
    walk is O(|reachable facts|).
    """
    visited: set[int] = set()
    def reaches_hypothesis(f: Fact) -> bool:
        fid = id(f)
        if fid in visited:
            return False
        visited.add(fid)
        if f.provenance is None:
            return False
        if f.provenance.kind == "hypothesis":
            return True
        for premise in f.provenance.justified_by or ():
            if reaches_hypothesis(premise):
                return True
        return False
    return not any(reaches_hypothesis(f) for f in unsat_core)
```

#### Sub-tasks

- **a.** Implement `is_unconditional_death` in
  `inference/hypothesis.py` (or a new
  `inference/back_prop.py` if it grows neighbours). Memoise per
  call via the visited set above; that's enough for the per-
  branch use site.
- **b.** Unit tests:
  - unsat-core is a single `source` fact + a derived
    `rule`-kind fact whose chain terminates at sources →
    unconditional.
  - unsat-core contains a `hypothesis`-kind fact directly →
    conditional.
  - unsat-core contains a `rule`-kind fact whose chain reaches
    a hypothesis two steps deep → conditional (this is the case
    the shallow read misses).
  - cyclical-justification edge case — defensive: `visited`
    set prevents infinite recursion if provenance ever
    self-references (shouldn't happen but the engine doesn't
    invariant-check).
- **c.** Document in
  `docs/kernel/inference/README.md` (currently a stub) the
  unconditional-death predicate and why it's stricter than the
  obvious `provenance.kind` check.

### Task T1.5.7.2 — Back-prop write on unconditional death

The actual `_negated_facts` write that turns a dead branch into a
cache entry for its siblings.

#### Design

In `_explore`, after `try_branch` returns Dead:

```python
if (cfg.enable_back_prop_unconditional
        and is_unconditional_death(parent_kb, result.unsat_core)):
    not_h = Fact(
        relation_name="not",
        args=(result.hypothesis,),
        layer=Layer.REASONING,
        provenance=Provenance.from_rule(
            rule_name="<back-prop-unconditional>",
            justified_by=tuple(result.unsat_core),
            bindings={},
        ),
    )
    parent_kb.add_fact(not_h)
    parent_kb._index_fact(not_h)        # ⟵ updates _negated_facts
    # T1.5.7.3 symmetric promotion fires here too
```

The next sibling's T1.5.4.7 `negated_fact` filter drops `h` in
O(1); ditto every descendant of every sibling.

#### Sub-tasks

- **a.** Add `enable_back_prop_unconditional: bool = False` to
  `SolverConfig` (T1.5.4.4). Default **off** — per the soundness
  concern above.
- **b.** Wire the write in `_explore` as sketched above. The
  synthetic provenance `<back-prop-unconditional>` makes the
  back-propped fact distinguishable in the search tree's
  REASONING-layer dump — important for the P1.6 trace renderer's
  "why is this hypothesis excluded?" explanation.
- **c.** Counter — add `back_propped` to T1.5.4.7's
  `HypGenStats.filtered` keyspace so the bench-solve summary
  shows "how many hypotheses did we skip *because* an earlier
  sibling died and back-propped". Distinguishes the cache-hit
  case from the original-source `(not h)` case.
- **d.** `bench_solve --diff-trees` mode — compare the search
  tree under `:enable-back-prop-unconditional false` vs `true`.
  The alive-branch verdicts must be identical; the dead-leaf
  count under `true` should be strictly ≤ the `false` count
  (cached siblings produce zero dead leaves). Any verdict
  divergence is a correctness bug.

### Task T1.5.7.3 — Symmetric-dead promotion

If `(R A B)` is unconditionally dead and `R` is symmetric,
back-propagate `(not (R B A))` too. The symmetric counterpart's
death is *also* unconditional under the same reasoning — `R`
is a symmetric relation, so `(R B A)` would saturate to `(R A B)`
which we just proved unconditional.

#### Sub-tasks

- **a.** After T1.5.7.2.b writes the primary `(not h)`,
  consult `_is_symmetric(parent_kb, h.relation_name)`. If true,
  build the symmetric counterpart and write it too with the
  same `<back-prop-unconditional>` provenance (justified_by
  extended with the original `h_fact` so the symmetric write's
  derivation walks back to the original death).
- **b.** Counter — increments `back_propped` by 2 for symmetric
  cases. Document this in the `--hyp-stats` output so readers
  don't double-count.
- **c.** Acceptance: a puzzle with `(symmetric co-located)` and
  a contradicted `(co-located White H1)` shows
  `(not (co-located White H1))` and `(not (co-located H1 White))`
  in the parent's REASONING layer after the back-prop fires;
  subsequent forks at the same or deeper level don't enumerate
  either ordering.

### Task T1.5.7.5 — Re-saturate the parent after a back-prop write

A back-propped `(not h)` is only an O(1) Filter-C entry until the
parent is **re-saturated** — once re-saturated it is a *premise*
the rule engine can chain from. This is the step that lets
[S1.5.8](s1.5.8_totality_domain_elimination.md)'s `domain-elimination`
fire: four `(not (co-located Norwegian X))` back-props plus a
`(total …)` activator derive `(co-located Norwegian Water)`.

#### Design

After the sibling sweep in `_explore` writes its back-propped
negatives (T1.5.7.2 / T1.5.7.3 / T1.5.7.4), re-run the saturator
on the parent KB:

```python
sat = Saturator(parent_kb)          # fresh — append-only KB, so correct
new_firings = tuple(sat.saturate())
```

Re-filtering needs no new code: `_candidates_for` already calls
`_prune_alive`, which re-checks `_negated_facts` +
`fact_already_exists` against the current KB. The iterative loop
(T1.5.7.6) re-invokes `_candidates_for` each pass, so the candidate
set shrinks for free.

#### Sub-tasks

- **a. Cadence.** Re-saturate **once per sibling sweep**, not once
  per dead sibling. A sweep can kill several siblings; one
  re-saturation after the sweep amortises cost and avoids
  mid-sweep candidate-list churn. (Per-sibling re-saturation is
  simpler but O(N) saturations and re-orders firings unhelpfully
  in the trace.)
- **b. Saturator instance.** Use a *fresh* `Saturator(parent_kb)`.
  The KB is append-only, so a fresh compile + enqueue is correct;
  reusing the stale instance risks missing matches unlocked by the
  new `(not …)` facts. Cost is Zebra-scale-trivial.
- **c. State-hash re-index.** Re-saturation mutates `kb`; the
  `builder.state_index[state_hash(kb)]` entry recorded at
  `_explore` entry is now stale. Re-index the node under the new
  hash after each re-saturation (or defer hashing to loop
  quiescence — pick one and document it).
- **d. Counter.** Add `resaturations` to the `HypGenStats`
  keyspace (T1.5.4.7) so `bench_solve --hyp-stats` shows how many
  extra saturation passes back-prop costs.
- **e. Tests.** A 3-candidate puzzle where killing 2 siblings
  back-props 2 negatives; assert the re-saturated parent contains
  both `(not …)` facts *and* any fact a loaded rule derives from
  them. (`domain-elimination` itself is exercised in S1.5.8 — here
  use a trivial test-only rule that asserts a positive from a
  `(not …)` premise, so T1.5.7.5 is independently testable.)

### Task T1.5.7.6 — Iterative `_explore` loop + return on derived positive

Restructure `_explore`'s `for h in candidates` body into the
consume loop sketched in *Architecture shift* above.

#### Design

- The loop: `saturate → detect → candidates → sweep → if any
  unconditional death, re-saturate and loop; else descend`.
- **Forced-deduction detection.** After re-saturation (T1.5.7.5),
  diff the KB's REASONING layer against the pre-re-saturation
  snapshot. For each *new* positive fact, run the
  `reaches_hypothesis` walk from T1.5.7.1: if it terminates only
  at source/rule leaves (no hypothesis), the fact is a **forced
  deduction**.
- **Where forced facts go.** A forced fact is recorded in the
  *current* `SearchNode.firings` — it is a saturation step, not a
  branch. No child `SearchNode` is allocated for the hypothesis
  that "would have" produced it. The P1.6 trace renderer already
  threads `firings` as the inter-branch-point "saturation" prose.
- **Genuine choice point.** Only when the consume loop reaches a
  fixpoint with ≥ 2 still-alive candidates *and* no forced
  positive does the recursion descend — one child per alive
  candidate, exactly as today.
- **The one-survivor case.** If exactly one candidate is alive and
  re-saturation has not derived it as a positive, S1.5.7 alone
  still descends into it as a 1-child branch (correct, but spends
  a tree level). S1.5.8's `domain-elimination` is what removes
  that last tree level: with a `(total …)` activator the survivor
  is *derived*, return-on-positive fires, and the forced move
  costs zero depth. S1.5.7 ships the plumbing; S1.5.8 supplies the
  rule. This dependency is why S1.5.8's acceptance — not S1.5.7's —
  owns the "zebra solves with no forced-move tree levels" check.

#### Sub-tasks

- **a.** Restructure `_explore` into the loop; preserve the
  existing `dead` / `solution` / `open` leaf-verdict logic and the
  S1.5.3 state-dedup short-circuit.
- **b.** `reaches_hypothesis`-based forced-deduction detection on
  the re-saturation delta (reuses T1.5.7.1's walk verbatim).
- **c.** Forced firings recorded on the current node; assert no
  child node is created for a forced move.
- **d. Flag-gated no-op.** With `enable_back_prop_unconditional`
  off, nothing back-props, so `progressed` is always false and the
  loop runs exactly once — behaviour byte-identical to S1.5.6.
  Acceptance (T1.5.7.7) must show identical trees flag-off.
- **e. Tests.** A puzzle plus a test-only positive-deriving rule:
  4 of 5 siblings die unconditionally, re-saturation derives the
  5th positive; assert the tree has *no* depth+1 node for it and
  the firing is on the current node. (The real `domain-elimination`
  version is S1.5.8 T1.5.8.5.)

## Open questions parked here

- **Q-S1.5.7.A — Promote the flag to default-on?** After the
  controlled-experiment validation (T1.5.7.2.d) shows zero
  verdict divergence across the demo suite + Zebra, flip the
  default. The decision needs a corpus of puzzles bigger than the
  M1 suite to be confident; defer to P1.7's Zebra integration or
  M2's NL-generated puzzles.
- **Q-S1.5.7.B — Cross-`solve()` persistence?** The
  `_negated_facts` writes are scoped to a single `solve()` call.
  A future "REPL" or "incremental solve" use case might want to
  persist them across calls — but that requires invalidation
  semantics (what if the user edits the puzzle between calls?).
  Park until a use case appears.
- **Q-S1.5.7.C — Visualisation of the cache.** The P1.6 trace
  renderer should distinguish "this branch was tried and died"
  from "this hypothesis was skipped because an earlier sibling
  died". The provenance `<back-prop-unconditional>` /
  `<lookahead-dies-immediately>` marker should drive a special
  edge style in the search-tree DOT render. Spec lives in P1.6;
  this task notes the requirement.
