# S1.5.4b — Drop Filter B (slot-already-used)

**Phase:** P1.5 (hypothesis loop), prerequisite to
[S1.5.4](s1.5.4_hypgen_improvements.md).
**Estimate:** S (~½ day code, plus test-fixture sweep).
**Driver:** user feedback 2026-05-21 — "Filter B is obviously
wrong, number of relations where object participate is limited by
rules."

## Context

`generate_hypotheses` runs three filters per candidate
(`ein.py/src/ein/inference/hypgen.py:83`):

| filter | predicate | sound? | shipped via |
|--------|-----------|--------|-------------|
| A — type compatibility | `_type_compatible(obj, sig_type)` walks `is-a` / kernel `Type` ancestry | yes; structural | `_hypotheses_for:94` |
| **B — slot-already-used** | `(rel.name, slot_idx) in existing(obj)` | **no; assumes the relation is functional on that slot** | `_hypotheses_for:96-97` |
| C — `(not h)` exclusion | `(fact.rel_name, fact.args) in kb._negated_facts` | yes; O(1) check | `_fill_slot:159, 171` |

Filter B says: "if `obj` already sits at `(rel, slot)` in some
fact, don't enumerate `(rel obj ?)` further." That's correct *only
when the relation is functional on that slot* — at most one fact
`(rel obj *)` may hold. For a future puzzle with `(friends-with A
B)` where A can have multiple friends, Filter B incorrectly prunes
`(friends-with A C)`, `(friends-with A D)`, … even though they're
admissible.

The principled architecture (per the canonical-zebra2 direction —
[[project-canonical-zebra2]]): let user rules express the
functional constraint, let the rule's `:assert (not h)` (or its
direct-⊥ variant under [S1.5.4a](s1.5.4a-fix-single-parent.md)
Part 2) populate `kb._negated_facts`, let Filter C drop `h` in
O(1). Filter B short-circuits this in Python, which violates
T1.5.4.2.d's anti-pattern note (decisions about admissibility belong
in user rules, not in the generator).

## The fix

Delete the two-line filter:

```python
# ein.py/src/ein/inference/hypgen.py
def _hypotheses_for(kb: KnowledgeBase, obj_ref) -> Iterator[Fact]:
    existing = {                                 # ← compute but use only for
        (f.relation_name, i)                     #   diagnostic / future heuristic;
        for f in obj_ref.as_arg                  #   no longer drives the skip.
        for i, a in enumerate(f.args)
        if isinstance(a, str) and a == obj_ref.name
    }
    for rel in kb.relations.values():
        if not rel.signature:
            continue
        for slot_idx, sig_type in enumerate(rel.signature):
            if not _type_compatible(kb, obj_ref.name, sig_type):
                continue
-           if (rel.name, slot_idx) in existing:    # ← Filter B, REMOVE
-               continue
            yield from _fill_slot(kb, rel, slot_idx, obj_ref)
```

The `existing` set computation can stay (cheap; useful for future
heuristics like ordering hypotheses by "how many slots already
occupied"); only the skip-on-membership goes.

Once removed, the generator emits the full type-compatible
candidate set. The same-call `seen` dedup in `generate_hypotheses`
itself still suppresses obvious duplicates (both Alice and Bob
enumerating `(r Alice Bob)` from their respective focal-object
perspectives), and Filter C still catches `(not h)` exclusions.

## Cascade — T1.5.4.5's `_prune` needs the same fix

The S1.5.4 alive-set inheritance plan
([`s1.5.4_hypgen_improvements.md`](s1.5.4_hypgen_improvements.md#task-t1545--inherit-alive-hypothesis-set-compute-once-at-b0)
T1.5.4.5) currently mirrors Filter B as its second prune trigger:

> 2. **Slot-occupied** — for every new positive REASONING fact
>    `(r a b)` introduced on this path, drop `(r a *)` and
>    `(r * b)` from alive.

Under this stage's fix, **that prune trigger must also go**. The
S1.5.4 plan needs an editorial pass:

- Remove "slot-occupied" from T1.5.4.5 `_prune`'s elimination
  triggers.
- The only remaining trigger is "newly negated" — any `h ∈ alive`
  whose key is in `kb._negated_facts \ parent_kb._negated_facts`.
- Document in T1.5.4.5 that the principled functional-constraint
  enforcement lives in user rules + back-prop (T1.5.4.3), not in
  `_prune`.

## Cost of the fix

For functional relations (the zebra-style case): the hypothesis
set grows because previously-suppressed candidates are now emitted.
Each tries → forks → saturates → dies via the
`functional` / `sibling-exclusive` / `asymmetric` rule → contributes
a dead leaf. Without compensating optimisations, the demo node
counts grow.

The compensations land alongside:

- **T1.5.4.3 back-prop** — first death of `(R A B)` writes
  `(not (R A B))` into the parent KB; subsequent siblings/descendants
  skip via Filter C. Functional deaths become one-shot per
  hypothesis-key.
- **T1.5.4.1 `(closed R)`** — for is-a (the bulk of the noise),
  closure short-circuits enumeration entirely.

So the sequencing is: ship S1.5.4b, *then* T1.5.4.1 and T1.5.4.3
in close succession so the temporary node-count regression is
quickly recovered.
