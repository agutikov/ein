; S1.5.7b — Stable-alive (and stable-conditional-dead) caching in the
; `_consume` loop. Performance follow-up to S1.5.7: today the consume
; loop re-runs `try_branch` on every still-alive AND every
; conditionally-dead candidate in every sweep pass, even though under
; the M1 rule set re-saturation cannot change either verdict. Adds a

; S1.5.7 T1.5.7.6 and S1.5.8 T1.5.8.3 both need it. Not M1-blocking —

# S1.5.7b — Stable-alive caching in the `_consume` loop

**Phase:** P1.5 (hypothesis loop), follow-up to
[S1.5.7](s1.5.7_back_prop_unconditional.md).
**Status:** parked 2026-05-23 — discovered while sanity-checking
`bench_solve --hyp-stats` against the
[10_backprop_on / 11_backprop_off](../../../examples/branching/10_backprop_on.ein)
pair (commit a37541b). Not M1-blocking.
**Estimate:** ~1-1½ days (≈25-line change in `_consume` for the
alive + conditional-dead cache, plus the
[`Firing.derives_positive`](#task-t157b5--promote-_derives_positive-to-a-kernel-predicate)
kernel promotion and its two-call-site refactor; unit-test surface
small because S1.5.7's existing tests already pin verdict-equivalence
flag-on vs flag-off).
**Depends on:** [S1.5.7](s1.5.7_back_prop_unconditional.md) shipped
(the consume loop + `back_propagate` + `is_unconditional_death`).
**Composes with:** [S1.5.8](s1.5.8_totality_domain_elimination.md)
— the cache's invalidation guard fires precisely when
`domain-elimination` (or any future rule that consumes `(not …)`
positively) derives a non-negative fact during re-saturation.

## Context

### Symptom

`bench_solve --hyp-stats` on the demo 10/11 pair shows back-prop ON
costing **more** `try_branch` calls than back-prop OFF, even though
tree node count is correctly lower:

| metric             | demo 10 (back-prop ON) | demo 11 (back-prop OFF) |
|--------------------|------------------------|--------------------------|
| `tree nodes`       | **32**                 | 50                       |
| `branches taken`   | **110**                | 70                       |
| `branches alive`   | 68                     | 30                       |
| `branches dead`    | 42                     | 40                       |

`tree nodes` is the metric back-prop is *designed* to shrink and
does. `branches taken` (== total `try_branch` invocations) tells a
different story because of the loop's structure.

### Root cause — the consume loop's per-pass full re-sweep

`solver.py:_consume` ([s1.5.7 T1.5.7.6](s1.5.7_back_prop_unconditional.md#task-t1576--iterative-_explore-loop--return-on-derived-positive)):

```python
while True:
    candidates = _candidates_for(kb)
    if not candidates: break
    unconditional = []
    for h in candidates:
        result = try_branch(kb, h, branch_id=builder.peek_id())  # ⟵ EVERY pass
        if (not result.is_alive()
                and is_unconditional_death(...)):
            back_propagate(kb, h, result.unsat_core)
            unconditional.append((h, result))
    if not unconditional:
        for cid in _descend(kb, nid, depth, max_depth, builder, mode):
            _add(cid)
        break
    # record dead, re-saturate ONCE per sweep
    ...
    node_firings = node_firings + tuple(Saturator(kb).saturate(...))
```

Each sweep pass calls `try_branch` on *every* still-pending
candidate, including the ones that returned alive last pass. Under
M1's rule library the alive verdict can't change between passes (see
**Invariant**, next), so those re-attempts are pure waste.

### Invariant — alive verdict is stable across `(not …)`-only re-saturation

Between sweep N and sweep N+1 the parent KB gains exactly:

1. The `(not h_i)` facts written by `back_propagate` for the
   unconditional deaths in sweep N (T1.5.7.2 / T1.5.7.3).
2. Whatever re-saturation derives from those `(not h_i)`s
   (T1.5.7.5).

Under the M1 rule library no rule's `:match` consumes a top-level
`(not X)` to derive a *positive* conclusion — re-saturation can
only propagate further `(not …)`s (e.g. via T1.5.7.3 symmetric
promotion). An alive candidate's verdict came from its fork's
saturation reaching fixpoint without contradiction; adding more
`(not …)`s to the parent's `_negated_facts` can only *introduce*
contradictions on subsequent forks, never *remove* them. So:

> **Stable-alive invariant** (M1): If `try_branch(kb, h)` returns
> alive at sweep N and the only change to `kb` between sweeps N and
> N+1 is the addition of `(not …)` facts (directly or via
> re-saturation), then `try_branch(kb, h)` would return alive at
> sweep N+1 with the same `result.kb` modulo the new `(not …)`s.

This means the "try every alive candidate every pass" structure
exists only to be defensively correct under
[S1.5.8](s1.5.8_totality_domain_elimination.md)'s
`domain-elimination` — which CAN derive new positive facts via
re-saturation and *can* therefore flip an alive verdict (the
return-on-derived-positive case
[T1.5.7.6](s1.5.7_back_prop_unconditional.md#task-t1576--iterative-_explore-loop--return-on-derived-positive)
explicitly handles). Under M1 alone the defensive re-sweep is
unconditional waste.

## Tasks

### Task T1.5.7b.1 — `alive_verified_at` cache + re-saturation generation counter

Cache each candidate's alive verdict by the re-saturation
generation it was verified against; on the next pass skip candidates
whose recorded generation matches the current one.

#### Design

```python
def _consume(kb, nid, firings, depth, max_depth, builder, mode):
    child_ids, seen_children = [], set()
    node_firings = firings

    resat_gen = 0
    alive_verified_at: dict[tuple[str, tuple], int] = {}

    def _add(cid):
        if cid not in seen_children:
            seen_children.add(cid); child_ids.append(cid)

    while True:
        candidates = _candidates_for(kb)
        if not candidates: break

        # Skip candidates whose verdict was verified after the
        # current re-saturation generation — stable-alive invariant.
        to_check = [
            h for h in candidates
            if alive_verified_at.get((h.relation_name, h.args), -1) < resat_gen
        ]
        if not to_check:
            # All remaining candidates are stable-alive — hand off
            # to descend over the full candidate set.
            for cid in _descend(kb, nid, depth, max_depth, builder, mode):
                _add(cid)
            break

        unconditional = []
        for h in to_check:
            result = try_branch(kb, h, branch_id=builder.peek_id())
            key = (h.relation_name, h.args)
            if result.is_alive():
                alive_verified_at[key] = resat_gen
            elif is_unconditional_death(
                    result.kb, result.unsat_core,
                    own_hypothesis=result.hypothesis):
                back_propagate(kb, h, result.unsat_core)
                unconditional.append((h, result))
            # else: conditional death — let _descend handle on
            # the final pass (today's behaviour).

        if not unconditional:
            for cid in _descend(kb, nid, depth, max_depth, builder, mode):
                _add(cid)
            break

        for h, result in unconditional:
            cid = builder.alloc()
            builder.add(SearchNode(
                id=cid, parent=nid, hypothesis=h,
                kb_snapshot=result.kb, firings=result.firings,
                verdict="dead", children=(),
                unsat_core=result.unsat_core,
            ))
            _add(cid)

        new_firings = tuple(Saturator(kb).saturate(max_steps=10_000))
        node_firings = node_firings + new_firings
        resat_gen += 1
        if any(_derives_positive(f) for f in new_firings):
            alive_verified_at.clear()    # S1.5.8 invalidation

    return child_ids, node_firings
```

`_derives_positive(firing)` returns `True` iff the firing's
conclusion fact has `relation_name != "not"` — i.e. the firing
added a positive, not a negation. Under M1 it always returns
`False`, so the cache is never invalidated.

#### Sub-tasks

- **a.** Add `alive_verified_at` + `resat_gen` to `_consume` as
  sketched.
- **b.** Implement `_derives_positive(firing)` next to
  `is_unconditional_death` in `inference/back_prop.py` (or
  `inference/solver.py` — wherever the helper-cluster ends up
  living after S1.5.8 lands).
- **c.** Update [S1.5.7 T1.5.7.6.d](s1.5.7_back_prop_unconditional.md#task-t1576--iterative-_explore-loop--return-on-derived-positive)'s
  "flag-gated no-op" claim: with the flag off the consume loop runs
  exactly once and skips the cache entirely; with the flag on the
  cache shrinks the per-call work but does not change the verdict.

### Task T1.5.7b.2 — Counter + bench observability

The win is invisible without a counter. Two new keys on
`HypGenStats` (T1.5.4.7):

- `alive_cached_skips` — `try_branch` calls *avoided* by the cache.
- `cache_invalidations` — number of times the cache was cleared
  because re-saturation derived a non-`(not …)` fact. Always 0 under
  M1; > 0 only once S1.5.8 ships.

`bench_solve --hyp-stats` prints both. The acceptance check is:

```
branches taken (S1.5.7b on)
    = branches taken (S1.5.7b off) − alive_cached_skips
```

i.e. the cache is the *only* source of try_branch reduction; tree
node counts must match flag-on vs flag-off.

### Task T1.5.7b.4 — Cache conditional-dead verdicts too

The stable-alive invariant has a mirror for conditional dead:

> **Stable-conditional-dead invariant** (M1): If `try_branch(kb, h)`
> returns dead at sweep N and the unsat-core's transitive walk
> reaches an *ancestor* hypothesis (the conditional case —
> `is_unconditional_death(...) == False`), and the only change to
> `kb` between sweeps N and N+1 is the addition of `(not …)` facts,
> then `try_branch(kb, h)` is dead-conditional at sweep N+1 too.

Reason: the ancestor-hypothesis context that made `h` dead is fixed
for the duration of this `_consume` call (ancestors are baked into
the inherited `kb.alive` / current KB); the only between-sweep
delta is `(not …)`s, which can only *add* contradictions to `h`'s
fork, never remove them. So conditional-dead is just as stable as
alive — and the same invalidation guard (re-saturation deriving a
non-`(not …)` fact) covers both.

**Scope.** This task adds value only at **interior** `_consume`
calls. At b0 there is no ancestor hypothesis, so
`is_unconditional_death` is true for every death — there are *no*
conditional deads at b0 to cache. The cache shape itself is
identical at every level; the b0 cache simply stays empty in its
conditional-dead half.

**Cross-boundary subtlety.** Today's `_consume` hands off to
`_descend` at fixpoint, and `_descend` runs its own
`for h in candidates: result = try_branch(...)` loop (solver.py:493)
that re-attempts every still-pending candidate — including the
conditional deads `_consume` already knows about. The cache must
either be plumbed through to `_descend` or `_consume` must
**allocate the conditional-dead SearchNodes itself** before
handing off, so `_descend`'s candidate set already excludes them.
Pick the second — same shape as the unconditional-dead handling
already in `_consume`, no `_descend` signature change.

#### Design

Extend `alive_verified_at` into a verdict-bearing cache:

```python
verdict_at: dict[tuple[str, tuple], tuple[Literal["alive", "cond-dead"], int]] = {}

for h in to_check:
    key = (h.relation_name, h.args)
    result = try_branch(kb, h, branch_id=builder.peek_id())
    if result.is_alive():
        verdict_at[key] = ("alive", resat_gen)
    elif is_unconditional_death(...):
        back_propagate(kb, h, result.unsat_core)
        unconditional.append((h, result))
    else:
        # Conditional dead: record now so _descend doesn't re-try.
        verdict_at[key] = ("cond-dead", resat_gen)
        cid = builder.alloc()
        builder.add(SearchNode(
            id=cid, parent=nid, hypothesis=h,
            kb_snapshot=result.kb, firings=result.firings,
            verdict="dead", children=(),
            unsat_core=result.unsat_core,
        ))
        _add(cid)

# Re-check exclusion logic — same skip predicate for both verdicts:
to_check = [
    h for h in candidates
    if verdict_at.get((h.relation_name, h.args), (None, -1))[1] < resat_gen
]

# Hand-off to _descend with cond-dead candidates already removed:
cond_dead_keys = {k for k, (v, _) in verdict_at.items() if v == "cond-dead"}
remaining = [h for h in candidates
             if (h.relation_name, h.args) not in cond_dead_keys]
# _descend operates on `remaining` (alive set only).
```

`_descend` keeps its current signature; the filtering happens in
`_consume` before the handoff (e.g. by mutating `kb.alive` to drop
the cond-dead keys, or by an explicit `exclude=` kwarg — pick
whichever falls out cleanly from the integration).

#### Sub-tasks

- **a.** Promote `alive_verified_at` to the

- **b.** Allocate conditional-dead SearchNodes inline in `_consume`

- **c.** Filter conditional-dead keys out of the candidate set

### Task T1.5.7b.5 — Promote `_derives_positive` to a kernel predicate

`_derives_positive(firing)` (introduced in T1.5.7b.1) answers "did
this rule firing add a positive fact, not just propagate a
`(not …)`?" The same question is asked by:

- **S1.5.7 T1.5.7.6's return-on-derived-positive** — "for each
  *new* positive fact" in the re-saturation delta is exactly this
  predicate, applied to the firing rather than the fact.
- **S1.5.8 T1.5.8.3 `domain-elimination`** — the rule's downstream
  consumers (trace renderer, firing classifier) need to distinguish
  "domain-elimination derived a positive forced fact" from
  "symmetric promotion propagated another `(not …)`".

Three call sites = factor it out now (instead of waiting for the
second caller per the usual defer-until-friction rule).

#### Design

Home it as a method on `Firing` (P1.3's class — already carries
the concluded fact). One-line implementation:

```python
# inference/firing.py (or wherever Firing lives)
@dataclass(frozen=True)
class Firing:
    rule:        str
    bindings:    dict[str, Any]
    conclusion:  Fact
    ...
    def derives_positive(self) -> bool:
        """True iff the conclusion is a positive fact, not a `(not …)`.

        Re-saturation produces only `(not …)` propagation under the
        M1 rule library — a `True` return marks a firing that goes
        beyond negation-cache filling: a forced positive deduction
        (S1.5.7 T1.5.7.6's return-on-positive trigger, or S1.5.8's
        `domain-elimination` output).
        """
        return self.conclusion.relation_name != "not"
```

Call sites that adopt it as part of S1.5.7b:

- `_consume`'s cache invalidation guard:
  `if any(f.derives_positive() for f in new_firings): verdict_at.clear()`.
- T1.5.7.6's forced-deduction detection (replaces the current
  inline `"for each new positive fact"` loop predicate).

#### Sub-tasks

- **a.** Add `Firing.derives_positive` (find the dataclass first;
  if it lives in `inference/saturator.py` or a sibling module, that
  one).
- **b.** Update `_consume`'s invalidation guard to use it.
- **c.** Update T1.5.7.6's return-on-derived-positive site to use
  it (purely a refactor — same semantics, named predicate).
- **d.** Test: a unit test on `Firing.derives_positive` over the
  two cases (positive conclusion → True; `(not X)` conclusion →
  False). Existing T1.5.7.6 / T1.5.7b tests cover the integrated
  behaviour.

## Design notes — questions resolved during planning

- **Cache value: `bool + gen`, not `BranchResult`.** The cached
  `BranchResult.kb` would be a fork from the moment of first
  `try_branch`; its `_negated_facts` is stale relative to later
  back-prop writes. `_descend` re-forks from the *current* parent
  so it doesn't use the cached fork; the consume loop only reads
  `result.is_alive()` off the cached entry. Keeping just the
  verdict + generation number is cheaper and sidesteps any
  stale-fork hazard (e.g. a stale fork ending up as a
  `SearchNode.kb_snapshot` would break P1.6's trace fidelity even
  if the verdict is right). T1.5.7b.4's extension stays
  `(verdict, gen)` for the same reason — never store the fork.
