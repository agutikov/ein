# S1.5.2 — Multilevel branching + search tree as proof object

**Phase:** P1.5 (hypothesis loop).
**Estimate:** 3-4 days.
**Depends on:** [S1.5.1](s1.5.1_saturate_branch.md) (single-branch
test cycle + hypothesis-generation + Q40 wiring).
**Successor:** [S1.5.3](s1.5.3_canonicalisation.md) (dedup +
alive-branch termination).
**Predecessor:** [S1.5.0 review](s1.5.0_review.md) §F (the
proof-object framing this stage implements).

## Context

S1.5.1 ships `try_branch(parent_kb, hypothesis)` — one round of
"fork, saturate, detect, emit Q40 facts". This stage wraps it in a
recursive descent that builds the **search tree**, the M1 engine's
proof object.

Per [S1.5.0 §F](s1.5.0_review.md#f-search-tree-is-the-proof-object),
the tree is not just a debug artefact. **It IS the proof** of the
puzzle's solution (or, in *contradictions* mode, of its
unsolvability):

- Every **dead** branch's unsat-core is a constructive proof of
  `¬H` for the hypothesis H that seeded it. The
  `hypothesis-contradiction` rule (P1.3 Q40 Option A) materialises
  this proof step explicitly.
- The **surviving** branch is the constructive part: every fact in
  its REASONING layer carries provenance back to source facts.
- The **tree as a whole** witnesses uniqueness: solution X is
  unique because every alternative branches is refuted, each with
  its own constructive sub-proof.

P1.6's trace renderer will serialise the entire tree, dead
branches included. The original P1.5 plan didn't take this
framing — it treated dead branches as noise. S1.5.2 corrects that:
dead branches are first-class proof artefacts.

This stage **does not implement dedup** — S1.5.3 owns that. The
tree built here is structurally a tree (no shared sub-graphs);
S1.5.3 turns it into a DAG via state-hash collapse.

## Tasks

### Task T1.5.2.1 — `SearchNode` + `SearchTree`

`src/ein_bot/inference/hypothesis.py` (same module as S1.5.1):

```python
from typing import Literal

BranchId = int

@dataclass(frozen=True)
class SearchNode:
    """One node in the search tree — a fork point plus its verdict.

    Each node represents *one branch* of exploration: the KB the
    branch ran on (post-saturation), the hypothesis fact that
    seeded it (None for the root), and the verdict.

    `verdict='dead'` → the branch contradicted; `unsat_core`
    carries the source-frontier facts; the branch is part of the
    uniqueness proof for the surviving sibling(s).

    `verdict='solution'` → the branch's KB satisfies the query's
    goal (mode-aware via S1.5.1's `is_solved`); the branch is the
    constructive part of the proof.

    `verdict='open'` → the branch was seeded but not yet explored
    (depth limit, or in-progress). Open nodes are NOT proof
    artefacts; they're work items.
    """
    id:         BranchId
    parent:     BranchId | None
    hypothesis: Fact | None         # None for the root
    kb_snapshot: KnowledgeBase      # post-saturation state
    firings:    tuple[Firing, ...]  # what fired during this branch's saturation
    verdict:    Literal["solution", "dead", "open"]
    children:   tuple[BranchId, ...]
    unsat_core: frozenset[Fact] = frozenset()  # set only for `dead`


@dataclass(frozen=True)
class SearchTree:
    """Immutable view of the completed search.

    Built bottom-up by `solve()` as each branch completes. Live
    branches don't appear in the final tree — only solved or dead
    nodes. (The 'open' state exists only mid-search; on return
    from `solve()` every node has a final verdict, or the depth
    limit triggered an Ambiguity for the unfinished subtree.)
    """
    root:  BranchId
    nodes: dict[BranchId, SearchNode]

    def solutions(self) -> tuple[SearchNode, ...]:
        return tuple(n for n in self.nodes.values() if n.verdict == "solution")

    def dead_branches(self) -> tuple[SearchNode, ...]:
        return tuple(n for n in self.nodes.values() if n.verdict == "dead")
```

Identity by `id` — a monotonic integer assigned at fork time.
Immutability via frozen dataclass; mutations during search live
on a builder, not on the value type.

### Task T1.5.2.2 — Recursive descent + `max_depth` guard

```python
def _explore(
    kb: KnowledgeBase,
    parent_id: BranchId | None,
    hypothesis: Fact | None,
    depth: int,
    max_depth: int,
    builder: _TreeBuilder,
    mode: Mode,
) -> BranchId:
    """Recursive search. Returns the BranchId of the node added."""
    sat = Saturator(kb)
    firings = list(sat.saturate(max_steps=10_000))

    node_id = builder.alloc()

    # Termination ladder, in order:
    contradictions = sat.contradictions()
    if contradictions:
        unsat_core = kb.unsat_core(c.positive for c in contradictions)
        builder.complete(node_id, parent_id, hypothesis, kb, firings,
                         verdict="dead", unsat_core=frozenset(unsat_core))
        return node_id

    if is_solved(kb, mode):
        builder.complete(node_id, parent_id, hypothesis, kb, firings,
                         verdict="solution")
        return node_id

    if depth >= max_depth:
        # Treat as open + return — S1.5.3 will report Ambiguity if
        # any 'open' nodes remain at top level.
        builder.complete(node_id, parent_id, hypothesis, kb, firings,
                         verdict="open")
        return node_id

    # Generate hypotheses and recurse.
    children = []
    for h in generate_hypotheses(kb):
        result = try_branch(kb, h, branch_id=builder.next_branch_id())
        if result.is_alive():
            child_id = _explore(result.kb, node_id, h, depth + 1,
                                max_depth, builder, mode)
        else:
            child_id = builder.add_dead(parent_id=node_id,
                                        hypothesis=result.hypothesis,
                                        kb=result.kb,
                                        firings=result.firings,
                                        unsat_core=result.unsat_core)
        children.append(child_id)

    builder.set_children(node_id, tuple(children))
    # Verdict on this node is computed AFTER children settle —
    # S1.5.3 alive-branch termination decides between
    # "solution-via-children" and "dead-because-all-children-dead".
    builder.complete(node_id, parent_id, hypothesis, kb, firings,
                     verdict="open")
    return node_id
```

**Note**: at this stage the parent's verdict is `"open"` until
S1.5.3 fills in the alive-branch logic. The output of S1.5.2 is a
*tree shape with leaf verdicts* (every leaf either `solution` or
`dead`); interior nodes are stamped `open` and S1.5.3 promotes
them based on their children.

This split keeps S1.5.2 focused on the *shape* and S1.5.3 on the
*decision*.

### Task T1.5.2.3 — Search-tree → trace IR

The IR's existing `(trace …)` form has the sub-forms we need
([S1.5.0 §F](s1.5.0_review.md#search-tree-ir--round-trip-through-existing-trace)):

```lisp
(trace
  (branch-open root :note "initial state")
  (step s1 :rule symmetric :using (cond10) :derives (co-located House-1 Norwegian))
  …
  (branch-open b1 :on (co-located Englishman House-1) :branch 1)
  (step …)
  (contradiction c1 :unsat-core (b1 c10))
  (branch-close b1 :dead)
  (branch-open b2 :on (co-located Englishman House-2) :branch 2)
  …
  (branch-close b2 :solution)
  (branch-close root :solution))
```

`SearchTree.to_ir()` walks the tree, emitting `branch-open` /
`branch-close` pairs nested per parent/child relationship, with
`step` entries for each `Firing` and `contradiction` entries for
each dead branch's unsat-core.

`SearchTree.from_ir(trace_form)` parses the same back into a
`SearchTree` value — closes the loop so P1.6's trace renderer can
re-load a saved trace.

### Task T1.5.2.4 — `Ambiguity` + `Contradiction` packaging

The top-level `solve()` wraps `_explore`'s tree-building and
extracts the right verdict per mode:

```python
def solve(kb: KnowledgeBase, *, mode=None, max_depth=6) -> Verdict:
    mode = mode or _mode_from_query(kb) or Mode.SOLVE
    builder = _TreeBuilder()
    _explore(kb, None, None, 0, max_depth, builder, mode)
    tree = builder.finalize()  # S1.5.3 fills in alive-branch verdicts

    solutions = tree.solutions()
    dead_branches = tree.dead_branches()

    if mode is Mode.SOLVE:
        if len(solutions) == 1:
            return Solution(kb=solutions[0].kb_snapshot, trace=tree)
        if len(solutions) == 0:
            unsat = frozenset.union(*(d.unsat_core for d in dead_branches),
                                    frozenset())
            return Contradiction(unsat_core=unsat,
                                 branches=tuple(dead_branches))
        return Ambiguity(branches=tuple(
            Solution(kb=s.kb_snapshot, trace=tree) for s in solutions
        ))

    if mode is Mode.GAPS:
        # GAPS returns all surviving branches.
        return Ambiguity(branches=tuple(
            Solution(kb=s.kb_snapshot, trace=tree) for s in solutions
        ))

    if mode is Mode.CONTRADICTIONS:
        # CONTRADICTIONS always returns the dead branches' cores.
        unsat = frozenset.union(*(d.unsat_core for d in dead_branches),
                                frozenset())
        return Contradiction(unsat_core=unsat,
                             branches=tuple(dead_branches))
```

The `tree` value flows through all three return types — it's the
proof artefact regardless of what the caller is asking for. The
mode just decides what *summary* to expose alongside.

### Task T1.5.2.5 — Tests

`tests/inference/test_multilevel.py`:

- **Depth-1 success** — a 3-instance puzzle solvable by one
  hypothesis level; verify tree has root + 1 alive child + N dead
  siblings.
- **Depth-2 success** — puzzle requiring two hypothesis levels;
  verify root + child + grandchild path, with intermediate
  dead branches.
- **Depth-limit `Ambiguity`** — set `max_depth=1` on a depth-2
  puzzle; verify the returned verdict is `Ambiguity` and the open
  nodes are visible in the tree.
- **All-dead `Contradiction`** — a deliberately-unsolvable KB;
  every branch dies; verdict is `Contradiction` with the union of
  unsat-cores.
- **GAPS mode** — Zebra minus one condition (e.g., remove
  condition 15); verify GAPS returns at least one `Solution` (or
  multiple if the removal genuinely creates ambiguity).
- **CONTRADICTIONS mode** — Zebra plus a deliberately-conflicting
  positive (e.g., `(co-located Englishman House-1)` on top of the
  Norwegian-in-H1 assertion); verify the dead branches surface
  with provenance traces.
- **Tree → trace IR round-trip** — build a 5-node tree; call
  `to_ir()`; parse the result via `from_ir()`; verify the
  reconstructed tree is structurally equal.
- **Dead branches are first-class** — verify a returned `Solution`
  carries the full tree (dead branches inspectable via
  `solution.trace.dead_branches()`).
- **Q40 protocol in nested branches** — at each level, the
  `(hypothesis …)` and `(contradiction-under …)` synthetic facts
  appear in the correct branch's KB.
