# S1.5.8 — Totality + domain elimination

**Phase:** P1.5 (hypothesis loop).
**Estimate:** 3-5 days (the matcher capability in T1.5.8.2 is the

## Context

`type-exclusivity` (P1.3 T1.3.2.6, priority band 300) ships the
*forward* exclusion direction: an established `(co-located a v)`
negates every sibling `(not (co-located a v'))`. The **reverse** —
*N−1 values negated ⇒ the N-th is forced* — is the move the human
walkthrough writes as "Therefore, the first house is yellow" and
"the Norwegian drinks water". P1.3 catalogued it twice —
`elimination-by-exhaustion` (T1.3.2.4) and `forced-by-unique-position`
(T1.3.2.9) — and **deferred both**: T1.3.2.4 needs a `(dom …)`
compound node (blocked on Q26); T1.3.2.9 was left for an R3 audit
that never ran. S1.5.8 un-defers it, scoped to exactly what Zebra
needs.

Two ingredients:

1. **Totality.** Exclusion alone gives *at most one* value per
   slot (injectivity). Concluding the survivor additionally needs
   *at least one* (the slot must take **some** value) — totality.
   In Zebra this is the puzzle frame: "there are five houses, each
   a different colour, …" — every House has a Colour, a
   Nationality, a Drink, a Smoke, a Pet. Totality is an
   **ontology-layer property fact**, in the same family as
   `(symmetric R)` / `(square-fwd R)` (M1 rule-of-thumb: property
   tags are rule-application facts, not declaration kwargs).

2. **The rule.** `domain-elimination` reads, for an object `a` and
   a value type `VT` it is total over under `R`, the set of
   `(not (R a v))` facts; when all VT-instances but one are
   excluded, it asserts `(R a v_last)` as a REASONING-layer fact
   with a named, traceable firing.

Why this is a P1.5 stage and not a P1.3 one: the rule is only
*useful* once S1.5.7 back-propagates unconditional `(not h)` facts
into the parent and re-saturates — that is what accumulates the
N−1 negatives at a single KB the rule can fire over. P1.3 owns the
rule-family taxonomy; S1.5.8 owns making this specific rule fire
inside the hypothesis loop, and the matcher capability it needs.

## Tasks

### Task T1.5.8.1 — Totality declaration

A property fact stating that an object type's slot under a
relation is total over a value type.

#### Design

Surface form — a property fact, activator-style:

```lisp
(total co-located House Color)
(total co-located House Nationality)
(total co-located House Drink)
(total co-located House Smoke)
(total co-located House Pet)
```

`(total ?R ?OT ?VT)` reads "every instance of `?OT` is
`?R`-related to some instance of `?VT`". It activates
`domain-elimination` (T1.5.8.2) the way `(symmetric R)` activates
`symmetric`. The declaration lives in the puzzle's `(ontology …)`
block.

#### Sub-tasks

- **a.** Decide arity / shape. `(total ?R ?OT ?VT)` (3-arg, as
  above) vs a 2-arg `(total ?R ?VT)` that infers `?OT` from `?R`'s
  declared domain. Recommend the explicit 3-arg form —
  `co-located` is symmetric and spans many type-pairs, so the
  directed (object-type → value-type) reading must be stated, not
  inferred.
- **b.** Add the `(total …)` facts to the canonical puzzle files
  (`examples/zebra.ein` and `examples/zebra2.ein`). In zebra2's
  `is-a` encoding confirm the value types resolve (Color etc. are
  `is-a` targets) — adjust the spelling if zebra2 needs it, but
  keep **one** rule for both encodings (encoding-agnostic).
- **c.** Loader: `total` auto-vivifies as an open-world relation
  like any property head; no kernel change. Confirm it is not
  mistaken for a fact whose arguments need type-checking against a
  declared `total` relation.

### Task T1.5.8.2 — `domain-elimination` rule + matcher capability

The rule, and the matcher feature it needs.

#### The rule

```lisp
(rule domain-elimination (?R ?OT ?VT)
  :match  (and (instance ?a ?OT)
               (only-remaining ?R ?a ?VT ?v))
  :assert (?R ?a ?v)
  :priority 400
  :why    "Every {?OT} is {?R}-related to some {?VT}; for {?a},
           every {?VT} except {?v} is excluded — therefore
           ({?R} {?a} {?v}).")
```

Activator `(total ?R ?OT ?VT)` binds `?R` / `?OT` / `?VT`.
Priority **400** — strictly after the `300` eliminate band,
because the rule needs `type-exclusivity` to have *finished*
producing `(not …)` facts before it counts survivors. (New band;
note it in the saturator's priority doc — S1.3.3.)

#### The matcher gap

`(only-remaining ?R ?a ?VT ?v)` is the hard part. It must: scan
the `(instance ?w ?VT)` facts, check for each whether
`(not (R a w))` is a *stored fact*, and bind `?v` to the unique
non-excluded one. M1's matcher cannot do this today:

- The predicate registry (`inference/predicates.py`) is Q33-capped
  at `eq` / `neq`, and a `Predicate` returns `bool` — it cannot
  *bind* `?v`.
- `(not P)` inside `:match` is hard-wired to negation-as-failure
  (`compile.py` `head_name == "not"` → `NegativeGuard`). NAF tests
  *absence of a positive*; it cannot match a *stored* `(not …)`
  fact — and absence-of-positive is unsound for the loop (a fact
  underived at a branch is not the same as a fact proven false).

#### Sub-task — pick the mechanism (decision task)

- **Option A — binding structural predicate.** Extend the registry
  with a predicate kind that receives the KB and *yields
  bindings* (a `StructuralScan` step, distinct from `Guard`).
  `only-remaining` walks `kb._facts_by_relation["not"]` +
  `["instance"]` in Python and binds `?v`. Contained: one new step
  type, one new predicate; does not touch `not` semantics.
  **Recommended.**
- **Option B — match stored `(not …)` facts.** Teach the compiler
  that `(not <single-relation-pattern>)` in `:match` is a *scan*
  of `not`-headed facts, while `(not <conjunction/guard>)` stays
  NAF. Then `only-remaining` is expressible as a pure
  `neq + not + NAF` pattern (Q33's T1.3.2.9 framing). Rejected
  unless A proves infeasible: the disambiguation is subtle and
  overloads the `not` keyword.
- **Option C — `(dom …)` compound node.** The original T1.3.2.4
  route; blocked on Q26 (compound node kinds). Out of M1 scope.

Whichever is chosen, the rule body and `:assert` stay declarative
and the trace shows a named `domain-elimination` firing — no raw
Python in the proof.

#### Sub-tasks

- **a.** Implement the chosen mechanism (Option A unless the audit
  flips it). Land it behind the `predicates.register()` hook the
  registry already exposes for followups.
- **b.** Ship `domain-elimination` in the canonical rule library
  (inline in each puzzle's `(rules …)` block until the P1.7
  T1.7.2.5 proto-library lands).
- **c.** Priority band 400 — extend the saturator priority doc
  (S1.3.3) and `docs/kernel/inference/`.
- **d.** Tests: a single object, a 5-instance value type, 4
  `(not …)` facts pre-loaded ⇒ exactly one `domain-elimination`
  firing binding the survivor; 3-of-5 excluded ⇒ no firing;
  5-of-5 excluded ⇒ see T1.5.8.3.

### Task T1.5.8.3 — All-excluded ⇒ contradiction

If a *total* slot has **every** value excluded, the puzzle is
unsatisfiable at this state. `domain-elimination` finds zero
survivors; that must surface as a contradiction, not a silent
no-fire.

#### Sub-tasks

- **a.** When `only-remaining` finds zero candidates for a total
  slot, assert a contradiction witness (a `(not …)` against the
  totality fact, or a synthetic `(contradiction-under …)`) so the
  `ContradictionDetector` (P1.4) flags the branch dead.
- **b.** This is the CDCL-style eager conflict: composed with
  S1.5.7, an all-excluded total slot detected during parent
  re-saturation kills the *parent* branch and back-jumps — the
  conflict is attributed where it belongs instead of being found
  lazily by `_promote_verdicts` after a full descent.
- **c.** Tests: a puzzle seeded so a total slot loses every value;
  assert `solve()` returns `Contradiction` with an `unsat_core`
  citing the totality fact + the excluding sources.

### Task T1.5.8.4 — Integration with the S1.5.7 consume loop

Wire `domain-elimination` into the re-saturation pass so its
conclusions trigger T1.5.7.6's return-on-positive.

#### Sub-tasks

- **a.** Confirm `domain-elimination` is in the compile cache used
  by the parent-re-saturation `Saturator` (T1.5.7.5) — it fires on
  the back-propped `(not …)` facts the same pass they land.
- **b.** Confirm the derived `(R a v)` fact's provenance walks
  back (via `justified_by`) to source/rule terminals only — so
  T1.5.7.6's `reaches_hypothesis` test classifies it as a *forced
  deduction* and folds it into the current node's firings (no
  child node, no tree level).
- **c.** Trace: the firing's `:why` string renders in the P1.6
  markdown trace as the "Therefore …" line. Reconcile the rule
  name with S1.7.3's checklist, which looks for
  `elimination-by-exhaustion` — see Q-S1.5.8.B; recommend naming
  the rule `elimination-by-exhaustion` so the human-trace
  vocabulary and the firing match without an alias table.

### Task T1.5.8.5 — Acceptance — M1 solves Zebra, zebra-style

- The Zebra puzzle (`examples/zebra2.ein`, canonical) solves
  uniquely under `Mode.SOLVE`: `(co-located Norwegian Water)` and
  `(co-located Japanese Zebra)` in the result KB.
- Every "therefore X" / "From this it follows" move in the
  [idea 08 target trace](../../../docs/ideas/08-human-style-deductive-trace.md)
  corresponds to a named `elimination-by-exhaustion` (or other
  saturation-rule) firing — *no* forced move appears as a
  hypothesis-branch verdict.
- The `SearchTree`'s interior nodes are *genuine choice points*
  only: with S1.5.7 + S1.5.8 + the flag on, forced moves cost
  zero tree depth — compare against the flag-off tree (strictly
  shallower, identical solution).
- `solve()` on Zebra stays within a small `max_depth`: forced
  moves no longer consume the depth budget, so the depth-budget
  exhaustion the static descent risked is gone.
- `pytest ein.py/tests/inference/test_domain_elimination.py
  ein.py/tests/inference/test_totality.py` ≥ 8 tests, green;
  `ruff check` green.

## Open questions parked here

- **Q-S1.5.8.A — Auto-infer totality?** A type whose instances are
  in bijection with another (all-different + equal cardinality)
  is total by construction. Auto-deriving `(total …)` from an
  `(all-different …)` + cardinality witness would drop the manual
  declaration — but needs the aggregate-cardinality predicate M1
  doesn't have. Park to
  [F7 — rule-set sufficiency](../../followups/f7_rule_induction.md);
  M1 declares totality explicitly.
- **Q-S1.5.8.B — Rule name.** `elimination-by-exhaustion` (idea 06
  / human-trace vocabulary) vs `forced-by-unique-position`
  (T1.3.2.9) vs `domain-elimination` (this file). T1.5.8.4.c
  recommends `elimination-by-exhaustion`; confirm at ship and
  update this file's working name if so.
- **Q-S1.5.8.C — Generalise past bijection puzzles.** Totality +
  exclusion is the Zebra shape. Puzzles with partial functions (a
  slot that *may* be empty) carry no totality fact and
  `domain-elimination` correctly never fires there. Non-Zebra
  generalisation is F7 / M2 scope.
