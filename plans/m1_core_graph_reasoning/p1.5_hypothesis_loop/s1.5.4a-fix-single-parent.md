# S1.5.4a — Fix the `single-parent` rule

**Phase:** P1.5 (hypothesis loop), prerequisite to
[S1.5.4](s1.5.4_hypgen_improvements.md).
**Estimate:** Part 1 — S (~½ day). Part 2 — M (1-2 days; optional).
**Driver:** user feedback 2026-05-21 — "What `single-parent` rule are
you talking about?" and "both `:match` and `:assert` contains same
relation (?R ?a ?c); either remove it from `:match` or we have
another kind of constraint rules — direct constraint rules that emit
False or ⊥."

## Context

Two issues with the rule that S1.5.4 / the demos currently rely on:

```lisp
(rule single-parent (?rel)
  :match  (and (?rel ?a ?b) (?rel ?a ?c) (neq ?b ?c))
  :assert (not (?rel ?a ?c))
  :why    "{?rel} functional: {?a} has only one parent."
  :priority 250)
```

**Issue 1 — wrong name.** This rule is not in
`examples/zebra.ein`, `examples/zebra2.ein`, the proto-library
sketch, or any kernel module. It was introduced by me while
authoring `examples/branching/02_…05_*.ein` and the rewritten
hypothesis-loop test fixtures (commit `cda7c4a`). The canonical
rule-library vocabulary already has `symmetric`, `transitive`,
`implies`, `asymmetric`, `sibling-exclusive`, `square-{fwd,bwd,unique}`.
"single-parent" doesn't fit the family. The standard CSP/DL term is
**functional** — a relation R is functional iff for every `a`, at
most one `b` satisfies `(R a b)`.

**Issue 2 — clunky formulation.** The rule's `:match` and `:assert`
both reference `(?rel ?a ?c)`. Logically it derives "given that
`(R a b)` AND `(R a c)` both hold (b ≠ c), conclude that
`(R a c)` does NOT hold" — a contradiction encoded as a
self-negation of one of the premises. It works (the contradiction
detector pairs the asserted `(not (R a c))` with the existing
`(R a c)` in the same fork's KB), but it reads as if the rule is
talking past itself. The principled formulation is "given that
both hold (b ≠ c), we have a contradiction" — i.e. `:assert ⊥` —
which the engine **does not currently support**.

### Part 2 (optional, design call) — direct ⊥-emitting rules

A kernel widening that lets rules `:assert (false …)` (or any
chosen reserved head) to signal a contradiction directly,
without going through the `(not h)` self-negation idiom. After
this lands, the `functional` rule becomes:

```lisp
(rule functional (?R)
  :match  (and (?R ?a ?b) (?R ?a ?c) (neq ?b ?c))
  :assert (false)
  :why "{?R} not functional: {?a} has {?b} and {?c}")
```

Logically the conclusion *is* the contradiction; nothing is
self-negated.

**Sub-tasks:**

- **a.** Pick the reserved head. Candidates:
  - `false`           — short, ASCII, matches Prolog / SMT idioms.
  - `contradiction`   — descriptive but heavy. Conflicts with
    Q40's `contradiction-under` carrier (close but distinct).
  - `⊥`               — Unicode, matches academic notation; loses
    grep-ability in pure-ASCII contexts.
  Default recommendation: `false`. Decision should be recorded
  in [`docs/kernel/ir/01-ein-graph/03_ein_model.md` §6](../../../docs/kernel/ir/01-ein-graph/03_ein_model.md#6-reserved-relation-names)
  as a fourth kernel-reserved head (alongside `relation`, `rule`,
  and arguably `not`).

- **b.** Extend `ContradictionDetector` at
  `ein.py/src/ein/inference/contradiction.py:110` (`_iter`):
  - Walk `kb._facts_by_relation["false"]` in addition to `"not"`.
  - For each `(false …)` fact, yield a `Contradiction` with
    `positive=None` and `negative=that fact`.

- **c.** Reshape `Contradiction` dataclass at
  `ein.py/src/ein/inference/contradiction.py:45`:
  - `positive: Fact | None` (currently `Fact`).
  - New field `kind: Literal["pair", "direct"]` to discriminate
    the two contradiction shapes.
  - Existing callers (e.g. `unsat_core` computation in
    `_explore`) handle both shapes. For `direct`, the unsat-core
    is the premise chain of the `(false …)` fact via
    `Provenance.justified_by`; no positive to chase.

- **d.** Reformulate the `functional` rule in the canonical rule
  library and re-run the demos. Hypothesis-loop back-propagation
  (T1.5.4.3) needs to identify "which hypothesis is responsible
  for this `(false …)`" by walking the unsat-core for the
  hypothesis-kind fact — same logic as today, no special
  handling.

- **e.** Update related rules that currently use the
  self-negation idiom to use `:assert (false …)`:
  - `sibling-exclusive` actually doesn't have this issue —
    `:match` is `(is-a ?a ?P)` ∧ `(is-a ?b ?P)` and `:assert
    (not (co-located ?a ?b))` — different relations. Leave
    as-is.
  - `asymmetric` — `:match (?rel ?a ?b)` ∧ `:assert (not (?rel
    ?b ?a))`. Different arg-order; not self-negation. Leave.
  - Only `functional` (and any future rule with the same shape)
    actually wants `:assert false`.

**Open question for Part 2 design discussion (Q-S1.5.4a.A):**
should we go further and add a `:contradict h` clause as syntactic
sugar over `:match (… h …) :assert (false)`? It would make the
intent obvious at the syntax level but it's another kernel form.
Park unless a second rule in the library wants it.

**Acceptance for Part 2:**
- A test in `ein.py/tests/inference/test_contradiction.py` shows
  a `(false …)` fact produces `Contradiction(kind="direct",
  positive=None)`.
- The reformulated `functional` rule's demos return the same
  verdicts and (modulo trace-content differences) the same node
  counts as the Part 1 version.
