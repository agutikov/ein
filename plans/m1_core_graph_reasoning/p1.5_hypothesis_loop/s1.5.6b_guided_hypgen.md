# S1.5.6b — Guided hypothesis generation

**Status:** done — 2026-05-22 (implemented; see Implementation
notes). Not M1-blocking.
**Phase:** P1.5 (hypothesis loop).
**Estimate:** ~3-5 days (T1.5.6b.1 / .3 are small; T1.5.6b.2 — the
`hrule` mechanism — is the bulk and may itself spin to a followup).
**Depends on:** [S1.5.4](s1.5.4_hypgen_improvements.md) — the
`(config …)` head (T1.5.4.4) and the `hypgen` filter pipeline +
`HypGenStats` (T1.5.4.7).
**Composes with:** [S1.5.6](s1.5.6_one_step_lookahead.md) — S1.5.6
*filters* candidates after generation; S1.5.6b *steers* generation
itself. The two are orthogonal: a guided / whitelisted candidate
still passes through the S1.5.6 one-step lookahead.

## Context

`hypgen` (`inference/hypgen.py`) is a blind combinatorial
enumerator: every instance-like object × every relation × every
slot × every type-compatible filler. For Zebra that is thousands
of raw candidates (zebra2: 3462 at the root), almost all pruned by
the filter pipeline. The filters — S1.5.4's `negated_fact` /
`fact_already_exists`, S1.5.6's `lookahead_killed` — are the
*reactive* defence: they discard a bad candidate after it is
built. This stage parks the *proactive* alternative: don't build
the bad candidate in the first place. Three mechanisms, increasing
in ambition, all from the user's "P1.5 more ideas":

## Tasks

### Task T1.5.6b.1 — Relation whitelist

The cheapest guide: restrict `hypgen` to an explicit set of
relations. Zebra only ever needs `(co-located …)` hypotheses —
`is-a` is `(closed)`, `right-of` / `next-to` are given by the
puzzle or derived by rules, never guessed. Enumerating them at all
is pure waste.

This is the **dual** of S1.5.4 T1.5.4.1's `(closed R)`: `(closed R)`
*blacklists* one relation; the whitelist says *only these*. When
both express the same intent the whitelist is terser for a puzzle
(Zebra: one `co-located` entry vs a `(closed …)` on every other
relation).

#### Design — query keyword (decided 2026-05-22, user direction)

A `(query …)` keyword `:hypothesis-relations` — the whitelist is
*query-scoped*, describing how to search for *this* goal, not a
solver-global config. The value is a `(r1 r2 …)` list or a bare
SYMBOL for the one-relation case; absent ⇒ no restriction.
`_raw_candidates` skips every relation absent from the whitelist
as a `pre_candidate` skip (`relation_not_whitelisted`) — the same
tier as `closed_relation`.

Earlier sketches put it on `SolverConfig` (a `(config …)` flag)
and, before that, on an `(allow-hypothesis R)` fact; both were
dropped — the whitelist belongs with `:mode` / `:goal` as part of
the query.

#### Sub-tasks

- **a.** `hypgen._query_relations` reads `:hypothesis-relations`
  from `kb.query`.
- **b.** `_raw_candidates` skip + the `relation_not_whitelisted`
  `pre_candidate` counter.
- **c.** Demo: `examples/branching/08_relation_whitelist.ein`
  exercises it (`right-of` skipped; verdict / binding unchanged).

### Task T1.5.6b.2 — Rule-driven hypothesis generation (`hrule`)

The substantive idea: hypotheses proposed by **rules**, not by the
blind enumerator. A *hypothesis rule* (`hrule`) is a rule whose
firing yields a *candidate hypothesis* instead of a derived fact —
the puzzle (or, later, M2's NL→IR frontend) declares *which*
hypotheses are sensible to try, and in what shape.

Why: the enumerator has no semantics — it cannot express "guess a
`co-located` edge for the most-constrained still-unbound object"
or "only guess values inside this object's remaining domain". A
rule pattern can. This is the generation-side counterpart of
S1.5.8's `domain-elimination`: that rule *derives* a forced value,
an `hrule` *proposes* a speculative one.

#### Design — `(hrule …)` declaration form (decided 2026-05-22)

An `hrule` is a **variant of `rule`**: a `(hrule NAME (params)
:match … :assert …)` form, the same shape as `(rule …)`, declared
inside the `(rules …)` block. The loader routes it to a separate
`kb.hrules` registry — hrules are *used* differently, so they are
*stored* apart. Because the engine's `compile_all` walks
`kb.rules` only, an `hrule` is never compiled for the saturator
or the lookahead — the separation alone gives the skip; no marker
fact and no skip logic are needed.

`hypgen` is the sole consumer: it compiles each `hrule`, runs its
`:match` over the KB, and `build_fact`s the `:assert` template
into a candidate routed through the ordinary `_apply_filters`
pipeline (so a rule-generated candidate is still negated-fact /
lookahead filtered and counted). The candidate is *not* written
to the KB — it is a hypothesis, handed to `try_branch`.

#### Sub-tasks

- **a.** Grammar + AST: a `hrule_decl` form inside `(rules …)`.
- **b.** Loader + store: a `kb.hrules` registry; `from_ir` routes
  `(hrule …)` there.
- **c.** New `inference/hrule.py` — the `Hrules` runner; `hypgen`
  feeds its output through `_apply_filters` + `HypGenStats`.
- **d.** A worked `hrule` demo —
  `examples/branching/09_hrule.ein`.

This task is design-heavy; if the `hrule` mechanism grows past a
few days it spins out to a followup — it is rung-2
**F5 rules-as-data** territory: a rule operated on for
*generation*, not derivation.

### Task T1.5.6b.3 — Automatic generation: driven by `hrule` presence

No config flag. The rule needs no knob: **if the puzzle declares
any `hrule`, the blind combinatorial enumerator is off and
generation is rule-driven; otherwise the enumerator runs.**
`hrule` presence *is* the switch. (An earlier sketch had an
explicit `enable-auto-hypgen` flag; dropped 2026-05-22 — a flag a
puzzle would always set in lockstep with declaring an `hrule` is
redundant.)

#### Sub-tasks

- **a.** `_generate` branches on `kb.hrules`: non-empty ⇒ run the
  hrules only; empty ⇒ run the blind enumerator.
- **b.** Acceptance: a puzzle with no `hrule` enumerates exactly
  as before; a puzzle with an `hrule` draws candidates only from
  it.

## Implementation notes — done 2026-05-22

Each mechanism is a **no-op for a puzzle that does not opt in**, so
the existing suite was unchanged — 555 tests green
(`test_guided_hypgen.py`, 9 tests).

- **T1.5.6b.1 — whitelist.** A `(query :hypothesis-relations …)`
  keyword — query-scoped, per user direction; *not* a `(config …)`
  flag, *not* an `(allow-hypothesis)` fact. `hypgen._query_relations`
  reads it from `kb.query`; `_raw_candidates` skips unlisted
  relations as `pre_candidate["relation_not_whitelisted"]`.
- **T1.5.6b.2 — `hrule`.** A first-class `(hrule …)` declaration
  form (grammar `hrule_decl`, `ast.py` `hrule_decl`), loaded into
  the separate `kb.hrules` registry (`store.add_hrule`,
  `from_ir._ingest_rules`). New `inference/hrule.py` — the
  `Hrules` runner compiles `kb.hrules` and yields their `:assert`
  facts as candidates. Because hrules live outside `kb.rules`, the
  engine's `compile_all` never compiles them — **the saturator and
  the lookahead need no skip logic** (the earlier fact-marker
  design's skips were reverted).
- **T1.5.6b.3 — no flag.** `hypgen._generate` runs the hrules iff
  `kb.hrules` is non-empty, else the blind enumerator. Presence of
  an `hrule` *is* the switch; the `enable-auto-hypgen` flag of the
  first sketch was dropped.
- **Demos** — `examples/branching/08_relation_whitelist.ein`
  (`(query … :hypothesis-relations (co-located))` skips `right-of`)
  and `09_hrule.ein` (a `(hrule …)` form, enumerator auto-off).
  Both solve, `?h = H3`.
- **Tests** — `tests/inference/test_guided_hypgen.py`, 9 green.

## Open questions parked here

- **Q-S1.5.6b.A — whitelist surface form** — **resolved**: a
  `(query :hypothesis-relations …)` keyword (query-scoped).
- **Q-S1.5.6b.B — `hrule` marking** — **resolved**: a first-class
  `(hrule …)` declaration form, loaded into `kb.hrules`.
- **Q-S1.5.6b.C — augment vs replace** — **resolved**: replace —
  any `hrule` turns the blind enumerator off; no flag.
- **Q-S1.5.6b.D — alive-set interaction** — the root alive set is
  computed once and inherited (S1.5.4 T1.5.4.8). A rule-guided
  generator may want to re-propose per node; reconcile with the
  inherit-once model.
