# S1.5.4 — Hypothesis generation improvements

**Phase:** P1.5 (hypothesis loop).
**Estimate:** 3-4 days.
**Depends on:** [S1.5.1](s1.5.1_saturate_branch.md),
[S1.5.2](s1.5.2_multilevel.md). Composes with — does **not** depend
on — [S1.5.3](s1.5.3_canonicalisation.md) (state-hash dedup).
**Motivated by:** the P1.5 demo runs in
`examples/zebra2/demos/solve/` — the 2-color, 2-house puzzle that
should branch ≤ 4 times instead produces ~19 search nodes, and the
5-color/5-house variant produces ~109. The bulk of the noise comes
from is-a hypotheses (`(is-a Red Blue)`, `(is-a H1 H2)`, …) and
from re-enumerating the same already-dead candidates at every
depth.

## Context

Today's `generate_hypotheses`:

1. Walks `kb.names` filtered to `category == "object"` — instance-
   like nodes derived from `is-a` (or `instance`) leaves.
2. For each object, iterates `kb.relations.values()` and emits
   `(R obj filler)` for every type-compatible filler in the
   complementary slot, modulo `(not …)` exclusion and the symmetric
   activator.

The 2-step framing has two leaky pieces:

- **Step "pick relation" is exhaustive.** Every declared relation
  contributes hypotheses; `is-a` produces `(is-a X Y)` candidates
  for every leaf-pair even though the inheritance forest is
  structurally fixed in every demo we author. The `single-parent`
  rule kills these post-hoc, but each one costs a saturate cycle
  and a dead-leaf node.
- **Death is forgotten between branches.** `(is-a Red Blue)` dies
  at depth 1 because Red already has `(is-a Red Color)`. At depth
  2 (under any hypothesis), the generator re-emits it; saturation
  re-runs; it re-dies. Same outcome, same cost, every depth.

S1.5.4 closes three gaps that share **one data structure** — the
`(not …)` fact set on the KB (`kb._negated_facts`, O(1) lookup).
Every mechanism below either *reads* from it or *writes* into it;
the dead-hypothesis cache and the negation index are the same
thing (user observation, 2026-05-21):

- **Topic A:** a *relation-level* "don't bother enumerating
  hypotheses on this" signal — either explicit (`(closed R)`) or
  derived (cardinality + already-full coverage).
- **Topic B:** a *pre-branch* contradiction check on each generated
  hypothesis against the parent KB — kill the hypothesis BEFORE
  paying for a fork + saturate + detect cycle. User observation
  (2026-05-21): "with all generated hypothesis new branch will be
  created, but new branch will be duplicated and saturated before
  contradiction; maybe it makes sense to filter hypothesis before
  creating new branch — look for contradiction in already existing
  facts". Two tiers, A shipped under the names-index commit (O(1)
  via `_negated_facts`), B new:
    - **Tier A (shipped — O(1))** — `_is_excluded` in
      `hypgen.py` returns `(rel, args) in kb._negated_facts`. The
      set is built in `rebuild_indexes` and maintained
      incrementally in `_index_fact`; the fork-time shallow-copy is
      a `set(parent._negated_facts)`. Catches hypotheses whose
      negation was derived during the parent's saturation
      (e.g. `(co-located Red Blue)` killed by sibling-exclusive at
      the parent level).
    - **Tier B (new, T1.5.4.0)** — one-step rule lookahead: for each
      negative-asserting rule, simulate firing it with `h` injected
      into the premises against the parent's existing facts. If the
      assertion would be `(not h)` (or a `(not f)` for which `f` is
      already in the parent KB), kill `h` without forking. Catches
      hypotheses that die in one rule step but aren't pre-derivable
      without the hypothesis being present — e.g. `(is-a Red Blue)`
      under `(single-parent is-a)` when Red already has a parent.
- **Topic C:** an *unconditionally-dead hypotheses* cache. Implementation
  collapses into "back-propagate `(not h)` to the parent KB on
  unconditional death" — the cache IS the `_negated_facts` set
  Tier A is already reading. When `try_branch` returns Dead with
  an unsat-core that depends on no ancestor-hypothesis,
  `_explore` adds `(not h)` to the parent's KB before iterating
  the next hypothesis. The next sibling, and every descendant,
  sees `h` excluded by the same O(1) `_is_excluded` check.
- **Topic D:** *the hypothesis set is fixed at b0 — inherit, don't
  regenerate*. User observation (2026-05-21): under the M1
  invariant that rules don't create new objects or new
  relations, and given M1 hypotheses connect names only (no
  nested-Fact hypothesis args), every possible hypothesis is
  enumerable from the root state. Going deeper *eliminates*
  hypotheses from this set; it never adds. The natural shape
  of the loop is therefore: compute the alive set once at b0;
  every child branch inherits its parent's alive set, prunes
  the candidates eliminated *on this path*, and picks the next
  hypothesis from what remains. `generate_hypotheses` runs
  exactly once per `solve()` call.

All four keep the hypothesis loop *complete* (no false alive
verdicts) — they remove provably-redundant work.

## Implementation order

Task numbers are stable for cross-references; the order below is the
agreed implementation sequence. Picked for **incremental value** —
each step has an obvious empirical signal on the demo node counts —
and for **minimal rework**: tasks 1-3 work in both the old and the
new loop shape, so the structural rewrite (T1.5.4.5) lands once,
late, against an already-small state space.

| # | task     | rationale                                                |
|---|----------|----------------------------------------------------------|
| 1 | T1.5.4.1 | declarative `(closed R)` — biggest win for least code; the is-a hypothesis pollution is ~25-30% of root candidates, single helper kills the lot |
| 2 | T1.5.4.2 | closure inference rule — automates step 1; ein-code only, no engine change |
| 3 | T1.5.4.3 | back-prop `(not h)` on unconditional death — works with the existing per-branch `generate_hypotheses` (it reads `_negated_facts`); refactor-neutral |
| 4 | T1.5.4.5 | alive-set inheritance — the structural rewrite. Validate against the small state space steps 1-3 leave behind |
| 5 | T1.5.4.6 | acceptance — demo node counts against the target table |
| 6 | T1.5.4.0 | **defer past M1** — one-step rule lookahead. After step 1, the cases T1.5.4.0 would catch are already filtered upstream by `(closed is-a)`. The co-located deaths the user cares about need a 2-3 step chain (`symmetric` → `transitive` → contradiction), which one-step misses. Park as a P1.5 follow-up if a future rule library produces single-step death patterns |

Author directive recorded 2026-05-21, re-evaluated after S1.5.3 ship:

> automatically and then emits the statement (closed R) that
> prevents new R hypothesis generation. User-defined also possible.
> Let it be 2 step of implementation:

> - implement inference of (closed R) fact on early stage
>   - preferred to use explicit rules for prove rather than

(Reflected as T1.5.4.1 → T1.5.4.2 in the order above.)

## Tasks

### Task T1.5.4.0 — Pre-branch one-step contradiction filter

The cheapest hypothesis-pruning tier. Belongs *between*
`generate_hypotheses` and `try_branch`: before paying for a fork
+ saturate, run a *one-step* simulation of every negative-
asserting rule with `h` substituted into the premises against
the parent KB. If the simulation would produce a contradiction,
skip `h` entirely.

Design:

- `_dies_immediately(kb, hyp) -> bool` next to `_is_excluded` in
  `hypgen.py`. Iterates each rule with `:assert (not …)`; for each,
  build the premise set with `h` as one of the facts; run the
  match against the parent KB's existing facts; check whether the
  assertion would be `(not h)` (h's death) or `(not f)` for a
  positive `f` already in the parent KB (some other contradiction).
- The match itself is the same machinery used by the saturator
  (`inference/match.py`); we just don't *apply* the result, only
  observe it.
- For the puzzles in `examples/branching/`, this turns most of the
  `is-a` hypothesis-deaths from "fork + saturate + detect (slow)"
  into "match one rule (fast)". Empirically these are the bulk of
  P2/P3's 100+ dead leaves.

Sub-tasks:

- **a.** Implement `_dies_immediately(kb, hyp)` covering rules
  whose `:assert` head is `not`. Handle the two distinct
  contradiction shapes:
    1. The asserted negative IS `(not <hyp>)` exactly.
    2. The asserted negative IS `(not <f>)` and `<f>` is already
       in `kb._facts_by_relation[f.relation_name]`.
- **b.** Have `_explore` consult `_dies_immediately` between
  `generate_hypotheses(kb)` and `try_branch(...)`. If true,
  either:
    - skip silently (cheapest; the dead branch never appears in
      the tree) — preserves all *alive*-branch verdicts;
    - OR record a dead leaf without a fork (richer proof but
      one extra SearchNode per kill) — recommended for now so
      the trace still shows "we considered and rejected".
  Pick the latter; the trace renderer ([P1.6](../p1.6_rendering_and_trace/README.md))
  needs the dead branches for the explanation gate.
- **c.** Co-design with T1.5.4.3: when `_dies_immediately` fires,
  it's *by definition* an unconditional death — also insert the
  key into the dead-hypothesis cache so later levels skip without
  even running the lookahead.
- **d.** Acceptance: enable the filter, rerun the demos, check the
  node counts collapse per the table in T1.5.4.6. P2 in particular
  should drop from 18 nodes to single digits because the bulk of
  the is-a hypothesis-deaths are exactly this one-step pattern.

Open question: parameterise the lookahead depth? One step is
enough for `(single-parent R)` / `(sibling-exclusive R)` /
`(asymmetric R)` death patterns — adding more depth converges
toward full saturation (Tier C in the topic intro, not worth
doing). M1 ships one step only.

### Task T1.5.4.1 — Relation closure: declarative `(closed R)` activator

A new property-fact: `(closed R)` declares that the relation `R`
admits no further true facts beyond what's already in the KB. The
hypothesis generator skips `R` entirely when picking the relation
step.

```lisp
(closed is-a)             ;; the inheritance forest is fixed
(closed right-of)         ;; the spatial lattice is fixed
```

Sub-tasks:

- **a.** `_is_closed(kb, r_name)` helper alongside `_is_symmetric`;
  looks up the `(closed R)` property fact via
  `kb._facts_by_relation["closed"]`.
- **b.** `_hypotheses_for` skips relations where `_is_closed(kb,
  rel.name)` is true. Document the contract: a closed relation
  contributes zero hypotheses; the engine assumes the puzzle
  author has fully populated it.
- **c.** Linter / sanity check: if a `(closed R)` is declared and
  saturation later derives a new `R`-fact (under any branch),
  emit a warning — the closure assumption was unsound for that
  puzzle. M1 doesn't have to *block* the run; the warning is
  enough for debugging.

### Task T1.5.4.2 — Closure inference via explicit rules

Per the directive at the head of this section, closure
*inference* is delivered as user-readable rules in ein-code, not
as a hardcoded Python predicate. The closure-recognition machine
in T1.5.4.1 doesn't care whether `(closed R)` was authored by the
puzzle or asserted by a rule firing — it just looks up
`kb._facts_by_relation["closed"]`. T1.5.4.2 supplies the rules
that produce those firings.

The simplest such rule: a single-parent (functional-on-slot-0)
relation whose leaves are all already declared is closed by
construction.

```lisp
;; (rules …)
(rule infer-closure-from-single-parent (?R)
  :match  (single-parent ?R)
  :assert (closed ?R)
  :why    "{?R} is functional and every leaf is declared with one
           parent — no further (?R …) facts are admissible."
  :priority 50)
```

The closure assertion fires at root-saturation time, *before*
hypothesis enumeration runs, so the generator sees a populated
`(closed ?R)` set on its first call.

Sub-tasks:

- **a.** Ship `infer-closure-from-single-parent` in the canonical
  rule library (the proto-library once
  [T1.7.2.5](../p1.7_bootstrapping_zebra/s1.7.2_dynamic_vs_hardcoded.md)
  lands; inline in each puzzle's `(rules …)` block until then).
- **b.** Acceptance test: load a zebra2-style puzzle that declares
  `(single-parent is-a)` but no explicit `(closed is-a)` —
  observe `(closed is-a)` in the REASONING layer after root
  saturation and zero `is-a` hypotheses at any depth.
- **c.** Open question: richer closure rules. Express "every
  leaf of `R` already has its required parent" in the rule
  language requires *negation-as-failure* or a *coverage count*
  primitive — neither is in M1's matcher (see
  [F7 rule induction §C — rule-set sufficiency](../../followups/f7_rule_induction.md)
  for the long-term framing). For M1, trust `(single-parent ?R)`
  as a sufficient witness; mismatched ontologies (functional
  declared but leaves unmentioned) fall into the
  *T1.5.4.1.c sanity-check warning* path.
- **d.** Anti-pattern note: if you reach for a Python branch in
  `_hypotheses_for` that says "if relation is is-a then skip",
  back up and write the rule instead. The engine reading
  `(closed is-a)` from REASONING is the canonical path; the
  puzzle author or rule library decides what's closed.

### Task T1.5.4.3 — Back-propagate `(not h)` on unconditional death

The "dead-hypothesis cache" is the `_negated_facts` set (Tier A).
When a branch dies *unconditionally*, the engine should assert
`(not h)` into the **parent** KB so the next sibling — and every
descendant on every level — sees `h` excluded by the same O(1)
`_is_excluded` check. No separate cache, no extra plumbing.

Unconditional vs ancestor-conditional death is read off the
unsat-core's `Provenance.kind` field:

- A fact is **ancestor-conditional** if its `provenance.kind ==
  "hypothesis"` (seeded by some ancestor branch).
- A fact is **unconditional** otherwise (ONTOLOGY / FACT / a
  REASONING-fact derived purely from unconditional inputs).

If the unsat-core contains **no** ancestor-conditional facts, the
hypothesis would die in every branch that ever tested it.

Sub-tasks:

- **a.** In `_explore`, after `try_branch` returns Dead, walk
  `result.unsat_core` checking for any fact with
  `provenance.kind == "hypothesis"`. If none, the death is
  unconditional.
- **b.** On unconditional death, build a `(not h)` fact (head
  `"not"`, args `(h_fact,)`) at the parent KB's REASONING layer
  and call `parent.add_fact(...)` + `parent._index_fact(...)`.
  The `_negated_facts` incremental update fires automatically;
  subsequent `generate_hypotheses(parent)` calls drop `h` via
  Tier A.
- **c.** Symmetric-dead promotion (formerly T1.5.4.6): if
  `(R A B)` is unconditionally dead and `R` is symmetric,
  back-propagate `(not (R B A))` too. Same insert, same
  index, same O(1) lookup downstream. Cuts the symmetric-
  duplicate work in half on the eliminate path.
- **d.** Acceptance shape: re-running P3 should show the engine
  fork `(co-located White H1)` once, see it die, back-propagate
  `(not (co-located White H1))`, and then never re-enumerate it
  under any deeper branch. Empirically combine with T1.5.4.0
  pre-branch filter — the two together stop the search well
  before it explodes.

### Task T1.5.4.5 — Inherit alive-hypothesis set; compute once at b0

The structural change of S1.5.4. Implements Topic D from the intro.

**Pre-condition (M1 invariant).** Holds under the current rule
library and the canonical-zebra2 direction:

- Rules don't create new objects. Their `:assert` produces facts
  whose args are existing names (or nested existing facts via
  Q40 — but the hypothesis generator excludes nested-Fact args by
  construction).
- Rules don't create new relations. `(relation N S₀ S₁)`
  declarations come from ontology IR, not from rule firings.
- Hypotheses connect names only — `_fill_slot` iterates
  `_instance_like_objects` and string-fills both slots; no
  nested-Fact hypothesis args.

Under these clauses, **the set of admissible hypotheses is
finite and fully determined at root saturation**. Deeper branches
*eliminate* candidates — they never extend the candidate space.

#### Algorithm shift

Current shape:

```
_explore(kb, …):
    for h in generate_hypotheses(kb):    # ⟵ re-enumerated every call
        result = try_branch(kb, h)
        ...
```

Target shape:

```
solve(root_kb, …):
    root_alive = frozenset(generate_hypotheses(root_kb))   # ⟵ ONCE
    _explore(root_kb, parent=None, alive=root_alive, …)

_explore(kb, parent, alive, …):
    alive = _prune(alive, kb, parent_kb)                  # path-specific deaths
    for h in alive:
        result = try_branch(kb, h)
        if result.alive:
            child_alive = alive - {h}                     # consumed
            _explore(result.kb, kb, child_alive, …)
        else:
            _record_dead_leaf(...)
            # Topic C: if unconditional, parent_kb gains (not h);
            # next-sibling's prune drops the rest.
```

`_prune(alive, kb, parent_kb)` removes hypotheses that became
inadmissible *on the path between parent and this state*:

1. **Newly negated** — any `h ∈ alive` with `(h.relation_name,
   h.args) ∈ kb._negated_facts \ parent_kb._negated_facts`.
   Caught by the same `_is_excluded` predicate as Tier A.
2. **Slot-occupied** — for every new positive REASONING fact
   `(r a b)` introduced on this path, drop `(r a *)` and
   `(r * b)` from alive. Mirrors the `existing(obj)[rel, slot]`
   skip the original generator performs per-object. Matters
   for relations where the rule library encodes a functional
   constraint (`single-parent` / `sibling-exclusive`) — when
   the constraint hasn't yet fired but the slot is occupied
   in this branch's KB, future hypotheses on the same slot are
   redundant.

#### Sub-tasks

- **a.** `solve()` computes `root_alive = frozenset((h.relation_name,
  h.args) for h in generate_hypotheses(root_kb))` after the root
  saturation. Pass through to `_explore` as a new positional arg.
- **b.** `_explore` accepts `alive: frozenset[tuple[str, tuple]]`.
  At entry it does the inheritance prune; the picked `h` is
  removed before recursing.
- **c.** `_prune` implementation in `inference/hypgen.py` (the
  natural home — symmetric to `generate_hypotheses`). Takes the
  current alive set + the kb + the parent kb. Returns the new
  alive set.
- **d.** Delete the per-branch `generate_hypotheses(kb)` call from
  `_explore`. Keep `generate_hypotheses` as the root-time
  enumerator (and used by `bench_solve`'s "root hyps" preview).
- **e.** Composition with Topic C: when `_explore` back-propagates
  `(not h)` into the parent's KB on unconditional death, the
  parent's `_negated_facts` is updated; the next sibling's
  inheritance-prune drops `h` (and symmetric pair, via Topic C's
  `_is_symmetric` promotion) from its alive set automatically.
  No separate signalling needed.
- **f.** Document the M1 invariant. Add an assertion at solve()
  entry: log a warning (or fail-fast) if any rule's `:assert`
  pattern introduces a relation name not declared in the
  ontology. Future rule libraries that violate the
  fixed-relations clause must either revert to per-branch
  enumeration or extend the alive set incrementally.

#### Performance picture

For the current zebra2.ein:
- `root_alive` size ≈ 3458 (from the bench_solve verbose output).
- Per-branch cost: O(|new-facts|) prunes + O(1) `_is_excluded`
  membership × O(|alive|) candidate iteration. Alive shrinks
  monotonically per path.
- Versus current: O(|leaves| × |relations| × |fillers|) per
  branch regardless of depth. Quadratic-ish in the puzzle's
  fact count.

Composes with T1.5.4.0 (pre-branch filter — applied to the
inherited alive set before picking) and T1.5.4.3 (back-prop —
shrinks the parent's alive set for downstream siblings).

#### Open question

Q-S1.5.4.D — **what happens when the invariant breaks**? F5
(rules-as-data) and any rule library that asserts new
`(relation …)` declarations would violate the fixed-relations
clause. Two reasonable responses:
  - re-enumerate from scratch when a rule fires that affects
    the relation registry (rare in practice);
  - declare such rules out-of-scope for the inheritance optimisation
    and require explicit per-puzzle opt-in.

Park until F5 lands.

## Open questions parked here

- **Q-S1.5.4.A — Closed-by-default for kernel-meta relations?**
  Once `instance` and `type` ship as ordinary relations via the
  proto-library ([P1.7 T1.7.2.5](../p1.7_bootstrapping_zebra/s1.7.2_dynamic_vs_hardcoded.md)),
  should the proto-library also assert `(closed instance)` /
  `(closed type)`? It would make zebra-original puzzles behave
  the same way as zebra2's `(closed is-a)`-style — fewer junk
  hypotheses for free. Default answer: yes, but the puzzle can
  override by NOT importing that line of the proto-library. Park
  until the proto-library lands.
- **Q-S1.5.4.B — Generalisation: relation-level pre-screening for
  hypothesis viability.** Should the generator also skip relations
  whose signature has no *unfilled* slot-instance pair? E.g.
  `(relation right-of House House)` where every House at slot 0
  already has a `right-of` fact emitted. Generalises the "is-a is
  closed because every leaf has a parent" idiom. Defer until F7
  rule induction picks it up.
- **Q-S1.5.4.C — Per-hypothesis vs per-state caching.** S1.5.3
  caches at state-hash granularity (a whole post-saturation KB
  reached twice → collapse). S1.5.4 caches at hypothesis-key
  granularity (a single `(R A B)` known to fail unconditionally).
  Both compose; they don't subsume each other. Confirm with a
  small benchmark on demo 05.
