# S1.5.4 — Hypothesis generation improvements

**Phase:** P1.5 (hypothesis loop).
**Estimate:** 3-4 days.
**Depends on:** [S1.5.1](s1.5.1_saturate_branch.md),
[S1.5.2](s1.5.2_multilevel.md). Composes with — does **not** depend
on — [S1.5.3](s1.5.3_canonicalisation.md) (state-hash dedup).
**Motivated by:** the P1.5 demo runs in
`examples/zebra2/demos/solve/` — the 2-color, 2-house puzzle that
should branch ≤ 4 times instead produces ~19 search nodes, and the
5-color/5-house variant produces ~109. The bulk of the noise comes
from is-a hypotheses (`(is-a Red Blue)`, `(is-a H1 H2)`, …) and
from re-enumerating the same already-dead candidates at every
depth.

## Context

Today's `generate_hypotheses`:

1. Walks `kb.names` filtered to `category == "object"` — instance-
   like nodes derived from `is-a` (or `instance`) leaves.
2. For each object, iterates `kb.relations.values()` and emits
   `(R obj filler)` for every type-compatible filler in the
   complementary slot, modulo `(not …)` exclusion and the symmetric
   activator.

The 2-step framing has two leaky pieces:

- **Step "pick relation" is exhaustive.** Every declared relation
  contributes hypotheses; `is-a` produces `(is-a X Y)` candidates
  for every leaf-pair even though the inheritance forest is
  structurally fixed in every demo we author. The `single-parent`
  rule kills these post-hoc, but each one costs a saturate cycle
  and a dead-leaf node.
- **Death is forgotten between branches.** `(is-a Red Blue)` dies
  at depth 1 because Red already has `(is-a Red Color)`. At depth
  2 (under any hypothesis), the generator re-emits it; saturation
  re-runs; it re-dies. Same outcome, same cost, every depth.

S1.5.4 closes both gaps:

- **Topic A:** a *relation-level* "don't bother enumerating
  hypotheses on this" signal — either explicit (`(closed R)`) or
  derived (cardinality + already-full coverage).
- **Topic B:** an *unconditionally-dead hypotheses* cache, populated
  when a branch's unsat-core depends on no ancestor-hypothesis.
  Subsequent enumeration consults the cache and skips known-bad
  `(rel, args)`.

Both keep the hypothesis loop *complete* (no false alive verdicts) —
they remove provably-redundant work.

## Tasks

### Task T1.5.4.1 — Relation closure: declarative `(closed R)` activator

A new property-fact: `(closed R)` declares that the relation `R`
admits no further true facts beyond what's already in the KB. The
hypothesis generator skips `R` entirely when picking the relation
step.

```lisp
(closed is-a)             ;; the inheritance forest is fixed
(closed right-of)         ;; the spatial lattice is fixed
```

Sub-tasks:

- **a.** `_is_closed(kb, r_name)` helper alongside `_is_symmetric`;
  looks up the `(closed R)` property fact via
  `kb._facts_by_relation["closed"]`.
- **b.** `_hypotheses_for` skips relations where `_is_closed(kb,
  rel.name)` is true. Document the contract: a closed relation
  contributes zero hypotheses; the engine assumes the puzzle
  author has fully populated it.
- **c.** Linter / sanity check: if a `(closed R)` is declared and
  saturation later derives a new `R`-fact (under any branch),
  emit a warning — the closure assumption was unsound for that
  puzzle. M1 doesn't have to *block* the run; the warning is
  enough for debugging.

### Task T1.5.4.2 — Relation closure: derived from cardinality

Some closures are mechanical given the rule library. If the puzzle
declares `(single-parent is-a)` (per-leaf functional under is-a)
AND every leaf appears at slot 0 of some `(is-a leaf parent)`
fact, then is-a is *structurally* closed at hypothesis-loop
entry — no hypothesis could ever add a valid is-a edge without
removing an existing one (which the append-only KB doesn't permit).

Sub-tasks:

- **a.** A small structural predicate
  `_relation_implicitly_closed(kb, rel)` — checks for known
  closure-implying rule patterns (`single-parent`, future
  `functional`, `exactly-one`) and verifies the coverage.
- **b.** Combine with T1.5.4.1's explicit signal: skip iff either
  is true.
- **c.** Open question (parked): generalise to multi-slot
  cardinality (e.g. `(exactly-one is-a-slot-0)`). For M1 we
  cover only the single-parent / functional pattern that the
  rule library already names.

### Task T1.5.4.3 — Unconditionally-dead hypothesis cache

When `try_branch` returns Dead, examine its `unsat_core`:

- A fact is **ancestor-conditional** if its `provenance.kind ==
  "hypothesis"` (i.e. seeded by some ancestor branch).
- A fact is **unconditional** otherwise (ONTOLOGY / FACT / a
  REASONING-fact derived purely from unconditional inputs).

If the unsat-core contains **no** ancestor-conditional facts, the
hypothesis's death is *unconditional* — it would die in every
branch that ever tested it. Cache its key.

Sub-tasks:

- **a.** `dead_hypotheses: set[tuple[str, tuple]]` on the
  `_TreeBuilder` (or threaded through `_explore` as a context
  arg). Persists across the whole `solve()` call.
- **b.** Inside `_explore`, after a child branch reports Dead with
  an unconditional unsat-core, add `(h.relation_name, h.args)`
  to `dead_hypotheses`.
- **c.** `generate_hypotheses` consults the same cache and skips
  yielded facts whose key is in it. Threaded by the same context
  arg.
- **d.** "Provenance has hypothesis-kind" check needs the
  `Provenance.kind` literal to include `"hypothesis"` (S1.5.1
  set this; verify). Walk the unsat-core, OR the recursive
  closure of REASONING facts via their justification chain if
  S1.5.4 finds direct-provenance not granular enough.

### Task T1.5.4.4 — Symmetric-dead promotion

If `(R A B)` is unconditionally dead and `R` is symmetric, then
`(R B A)` is also unconditionally dead. Promote on insert.

Sub-task:

- **a.** When adding `(R A B)` to `dead_hypotheses`, also add `(R
  B A)` iff `_is_symmetric(kb, R)`. Cheap; cuts the
  symmetric-duplicate work in half on the eliminate path.

### Task T1.5.4.5 — Acceptance

Run the five demos in `examples/zebra2/demos/solve/` and confirm
the node counts collapse to the human-reasonable shape:

|------|---------------|---------------------|
| 01_saturate_only        | 1                | 1                   |
| 02_one_dead_one_alive   | 19               | ≤ 5                 |

Bindings (answers) stay identical — S1.5.4 is purely a pruning

## Open questions parked here

- **Q-S1.5.4.A — Closed-by-default for kernel-meta relations?**
  Once `instance` and `type` ship as ordinary relations via the
  proto-library ([P1.7 T1.7.2.5](../p1.7_bootstrapping_zebra/s1.7.2_dynamic_vs_hardcoded.md)),
  should the proto-library also assert `(closed instance)` /
  `(closed type)`? It would make zebra-original puzzles behave
  the same way as zebra2's `(closed is-a)`-style — fewer junk
  hypotheses for free. Default answer: yes, but the puzzle can
  override by NOT importing that line of the proto-library. Park
  until the proto-library lands.
- **Q-S1.5.4.B — Generalisation: relation-level pre-screening for
  hypothesis viability.** Should the generator also skip relations
  whose signature has no *unfilled* slot-instance pair? E.g.
  `(relation right-of House House)` where every House at slot 0
  already has a `right-of` fact emitted. Generalises the "is-a is
  closed because every leaf has a parent" idiom. Defer until F7
  rule induction picks it up.
- **Q-S1.5.4.C — Per-hypothesis vs per-state caching.** S1.5.3
  caches at state-hash granularity (a whole post-saturation KB
  reached twice → collapse). S1.5.4 caches at hypothesis-key
  granularity (a single `(R A B)` known to fail unconditionally).
  Both compose; they don't subsume each other. Confirm with a
  small benchmark on demo 05.
