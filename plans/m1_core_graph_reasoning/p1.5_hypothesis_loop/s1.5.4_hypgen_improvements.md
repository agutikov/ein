# S1.5.4 — Hypothesis generation improvements

**Phase:** P1.5 (hypothesis loop).
**Estimate:** 3-4 days.
**Depends on:** [S1.5.1](s1.5.1_saturate_branch.md),
[S1.5.2](s1.5.2_multilevel.md). Composes with — does **not** depend
on — [S1.5.3](s1.5.3_canonicalisation.md) (state-hash dedup).
**Motivated by:** the P1.5 demo runs in
`examples/zebra2/demos/solve/` — the 2-color, 2-house puzzle that
should branch ≤ 4 times instead produces ~19 search nodes, and the
5-color/5-house variant produces ~109. The bulk of the noise comes
from is-a hypotheses (`(is-a Red Blue)`, `(is-a H1 H2)`, …) and
from re-enumerating the same already-dead candidates at every
depth.

## Context

Today's `generate_hypotheses`:

1. Walks `kb.names` filtered to `category == "object"` — instance-
   like nodes derived from `is-a` (or `instance`) leaves.
2. For each object, iterates `kb.relations.values()` and emits
   `(R obj filler)` for every type-compatible filler in the
   complementary slot, modulo `(not …)` exclusion and the symmetric
   activator.

The 2-step framing has two leaky pieces:

- **Step "pick relation" is exhaustive.** Every declared relation
  contributes hypotheses; `is-a` produces `(is-a X Y)` candidates
  for every leaf-pair even though the inheritance forest is
  structurally fixed in every demo we author. The `single-parent`
  rule kills these post-hoc, but each one costs a saturate cycle
  and a dead-leaf node.
- **Death is forgotten between branches.** `(is-a Red Blue)` dies
  at depth 1 because Red already has `(is-a Red Color)`. At depth
  2 (under any hypothesis), the generator re-emits it; saturation
  re-runs; it re-dies. Same outcome, same cost, every depth.

S1.5.4 was originally drafted with **four** topics, all reading
or writing the same `(not …)` fact set on the KB
(`kb._negated_facts`, O(1) lookup) — the dead-hypothesis cache and
the negation index are the same thing (user observation
2026-05-21). Per the implementation-order call recorded
2026-05-21 (see this stage's history), only the cheap,
hard-to-misjudge pieces stay; the topics whose payoff depends on
larger engine changes split into their own stages so each lands
against the empirical signal it's meant to produce:

- **Topic A — `(closed R)` relation-level activator.** Manual
  ships here as T1.5.4.1 — no config option needed; `(closed R)`
  is an ordinary fact, declarable or removable in the puzzle's
  `(ontology …)` block. **Auto-inference of `(closed R)` from
  rule patterns** defers to
  [S1.5.5](s1.5.5_closure_auto_inference.md) so the explicit form
  proves its node-count claim before derivation layers on top.
- **Topic B — pre-branch contradiction check on each hypothesis.**
  Two tiers; **only Tier A ships here**:
    - **Tier A (over negated facts)** — `_is_excluded(kb, h) ⇔
      (h.relation_name, h.args) ∈ kb._negated_facts`. Already O(1)
      via the `_negated_facts` index. Under Topic D, it applies
      once at b0 (filtering the root alive set) and re-checks
      per-path inside `_prune`. Ships as part of the
      raw-enumeration + filter-counter refactor (T1.5.4.7) so the
      "filtered by this condition" count is visible.
    - **Tier B (one-step rule lookahead)** — defers to
      [S1.5.6](s1.5.6_one_step_lookahead.md). Bundled there with
      the **sibling-exclusive 2-arg rewrite** that surfaces during
      that work: S1.5.4b's Filter-B removal causes a node-count
      regression that S1.5.6 is the proper fix for, and the
      rewrite is what lets the engine know `sibling-exclusive`
      affects both `is-a` and `co-located` hypothesis-deaths.
- **Topic C — back-propagate `(not h)` on unconditional death.**
  Defers to [S1.5.7](s1.5.7_back_prop_unconditional.md). The
  *unconditional-vs-conditional* read needs a transitive premise
  walk (E6) to be sound, and that walk is cheaper to land
  alongside the back-prop write than separately. Config flag
  default **off** — ships disabled so the first turn-on is a
  controlled experiment.
- **Topic D — inherit alive set, generate once at b0.** Core
  mechanism shipped as commit `40b8dd4` (`alive_hyps` threaded
  through `_explore`). Three polish items remain in S1.5.4 under
  T1.5.4.8: (1) store the alive set on the KB, not as a recursion
  parameter; (2) `--print-alive`-style inspection knob; (3)
  enable/disable config flag, default **on**.

Two cross-cutting tasks ship in S1.5.4 to support the above:

- **T1.5.4.4 — config head.** The Topic A/B/C/D gates need a
  place to live. A new top-level `(config …)` IR form parses into
  a `SolverConfig` dataclass; `solve(kb, *, config=…)` accepts a
  programmatic override.
- **T1.5.4.7 — instrumentation counters.** User observation
  2026-05-21: *"we can't 'not generate' some hypothesis directly,
  we can only filter, so count raw generated hypothesis and
  filtered by every condition."* Refactor `generate_hypotheses`
  into a raw enumeration followed by named filter passes, each
  bumping a counter. Surfaces in `bench_solve --hyp-stats` and in
  the `SearchTree`'s root metadata.

What ships in S1.5.4 keeps the hypothesis loop *complete* (no
false alive verdicts) — the deferred items are pure optimisations
on top of the same correctness guarantee.

## Tasks

### Task T1.5.4.0 — Pre-branch one-step contradiction filter ➡ S1.5.6

**Moved to [S1.5.6](s1.5.6_one_step_lookahead.md)** (2026-05-21).
The Tier-B lookahead's payoff hinges on the **sibling-exclusive
two-argument rewrite** — without it the engine can't pre-detect
that, say, a `(co-located X Y)` hypothesis is killed by an
existing `(is-a X T)` + `(is-a Y T)` pair via the
sibling-exclusive rule. S1.5.6 ships both together: the rewrite,
the `_dies_immediately(kb, h)` helper, and the empirical recovery
of the node-count regression that S1.5.4b (Filter B removal)
introduces.

### Task T1.5.4.1 — Relation closure: declarative `(closed R)` activator

A new property-fact: `(closed R)` declares that the relation `R`
admits no further true facts beyond what's already in the KB. The
hypothesis generator skips `R` entirely when picking the relation
step. **No config flag needed** — `(closed R)` is itself a fact
the puzzle author adds to or removes from `(ontology …)`; flipping
behaviour is a one-line IR edit, not an engine knob.

```lisp
(closed is-a)             ;; the inheritance forest is fixed
(closed right-of)         ;; the spatial lattice is fixed
```

Sub-tasks:

- **a.** `_is_closed(kb, r_name)` helper alongside `_is_symmetric`;
  looks up the `(closed R)` property fact via
  `kb._facts_by_relation["closed"]`.
- **b.** `_hypotheses_for` skips relations where `_is_closed(kb,
  rel.name)` is true. Document the contract: a closed relation
  contributes zero hypotheses; the engine assumes the puzzle
  author has fully populated it. Surface the skip count via the
  T1.5.4.7 `closed_relation` filter counter.
- **c.** Linter / sanity check: if a `(closed R)` is declared and
  saturation later derives a new `R`-fact (under any branch),
  emit a warning — the closure assumption was unsound for that
  puzzle. M1 doesn't have to *block* the run; the warning is
  enough for debugging.

### Task T1.5.4.4 — Config head — `(config …)` top-level form

A 7th top-level IR form `(config …)` carrying solver-level
knobs, parsed into a `SolverConfig` dataclass that
`solve(kb, *, config=…)` accepts as an override. The IR form is
the persistent default a puzzle ships with; the kwarg is the
caller's per-run override.

```lisp
(config
  :enable-alive-inherit             true   ; T1.5.4.8 — Topic D
  :enable-pre-branch-negated        true   ; T1.5.4.7 — Topic B Tier A
  :enable-pre-branch-lookahead      true   ; S1.5.6 — Topic B Tier B
  :enable-back-prop-unconditional   false  ; S1.5.7 — Topic C
  :enable-auto-closure              false  ; S1.5.5 — auto (closed R)
  :print-alive                      false  ; T1.5.4.8 — diagnostic
)
```

Sub-tasks:

- **a.** Grammar — add `(config …)` as a sibling of `(ontology …)`
  / `(facts …)` / `(reasoning …)` / `(rules …)` / `(query …)` /
  `(trace …)` in `src/ein/ir/grammar.lark`. Body is a flat
  `kw-pair*` (same shape as `(query …)`).
- **b.** `SolverConfig` dataclass in
  `src/ein/inference/config.py` with one boolean field per
  flag above; sensible defaults that match the per-task defaults
  in the catalog. Document the default per field with the
  ship-stage that owns the flag.
- **c.** Loader — `kb/from_ir.py` recognises `(config …)`, parses
  it into `SolverConfig`, stashes it on the KB (e.g.
  `kb.config: SolverConfig | None`).
- **d.** `solve(kb, *, mode=None, max_depth=6, config=None)` —
  resolves the effective config as `config or kb.config or
  SolverConfig()`. Threads through `_explore` and `try_branch`.
- **e.** Acceptance: a config-only puzzle (`(config :print-alive
  true)` on top of a zebra2-style ontology) round-trips through
  parse + dump; `solve(..., config=SolverConfig(print_alive=False))`
  overrides; flags actually gate the behaviour they name (one
  smoke test per flag asserting the gated code path runs / doesn't
  run via a counter assertion).

Q-S1.5.4.E records the design choice for this task; the form
proposed here is the leading candidate but the kwarg-only
variant is not closed off.

### Task T1.5.4.7 — Raw enumeration + per-filter counters

User observation 2026-05-21: *"I think we can't 'not generate'
some hypothesis directly, we can only filter, so count raw
generated hypothesis and filtered by every condition."* Refactor
`generate_hypotheses` from interleaved emit+filter into a clean
two-phase pipeline whose intermediate sizes are countable.

Today the filters are scattered: type-compat (`_type_compatible`,
inside `_hypotheses_for`), `(not …)` exclusion (`_is_excluded`,
inside `_fill_slot`), same-call dedup (`seen` set in
`generate_hypotheses`), and the symmetric-emit duplicator. Each is
a `yield`/`if-continue` woven into the generator; no per-filter
counters exist.

Sub-tasks:

- **a.** Refactor `generate_hypotheses` into
  `_raw_candidates(kb)` (no filters; emits every
  `(obj, rel, slot, filler)` shape that's syntactically a valid
  hypothesis) followed by a `_filter_pipeline(raw, kb)` that
  applies named filters in order:
    1. `same_self`         — skip `(R x x)` (already in `_fill_slot`).
    2. `type_compat`       — skip when filler's type doesn't unify
       with the slot's declared type.
    3. `closed_relation`   — skip when `(closed rel)` (T1.5.4.1).
    4. `negated_fact`      — skip when `(not h)` in `kb._negated_facts`
       (Topic B Tier A; absorbs `_is_excluded`).
    5. `seen_in_call`      — collapse symmetric / cross-object
       duplicates the same way today's `seen` does.
- **b.** `HypGenStats` dataclass in `inference/hypgen.py` (or a
  new `hypgen_stats.py` if the module grows):
  ```python
  @dataclass(frozen=True)
  class HypGenStats:
      raw: int                                # |_raw_candidates|
      filtered: dict[str, int]                # filter name → count dropped
      emitted: int                            # final yielded count
  ```
  Returned alongside the alive set from `generate_hypotheses`
  (tuple return; existing callers ignore the second element).
- **c.** Wire to `bench_solve --hyp-stats` — print the stats line
  at b0 (the only place generation happens, post-T1.5.4.8 KB-
  storage). Optionally per-pruned-path under `--hyp-stats=verbose`.
- **d.** Stash on the SearchTree's root node metadata so P1.6's
  trace renderer can include the breakdown.
- **e.** Acceptance: on `examples/zebra2.ein`, the raw count
  reproduces the pre-refactor "what `generate_hypotheses` would
  have emitted without filters" number; the sum of `filtered.values()
  + emitted` equals `raw`. A test in
  `tests/inference/test_hypgen_pruning.py` constructs a small KB
  with two `(not …)` facts and asserts `filtered["negated_fact"]
  == 2`.

### Task T1.5.4.8 — Alive-set polish (Topic D ship cleanup)

Three follow-ups to the shipped T1.5.4.5 mechanism. Per the
2026-05-21 author note, all three land together so the alive set
becomes a first-class part of the KB rather than a recursion
parameter.

Sub-tasks:

- **a.** Move `alive_hyps` onto `KnowledgeBase`. The root KB
  carries the materialised `frozenset[tuple[str, tuple]]`;
  forks inherit by reference (alive is immutable per branch —
  `_prune` returns a new frozenset rather than mutating the
  parent's). Replace the `alive=` recursion parameter in
  `_explore` with `kb.alive`.
    - Rationale: the KB already carries `_negated_facts` (the
      symmetric write side); the alive set is the read side. Having
      both on the same object keeps the invariants together.
    - Cascading edit: `_prune` becomes a `KnowledgeBase` method
      (or stays a free function in `hypgen.py` taking the parent
      kb's alive snapshot — pick whichever keeps the import graph
      acyclic).
- **b.** `--print-alive` diagnostic flag. CLI flag on `bench_solve`
  + config field `:print-alive` (T1.5.4.4) — when true, every
  `_explore` entry prints the inherited alive size and the per-
  filter `_prune` drop counts (which compose with T1.5.4.7's
  counters). Default off; intended for developer inspection of a
  single puzzle's branch behaviour.
- **c.** `enable-alive-inherit` config flag (T1.5.4.4). When
  false, `_explore` reverts to per-branch
  `generate_hypotheses(kb)` (the pre-`40b8dd4` shape). Default
  **on**; the flag exists as an escape hatch for puzzles whose
  rule library violates the M1 invariant (no rule-created
  relations / objects) until F5 (rules-as-data) lands.
- **d.** Document the M1 invariant in
  `docs/kernel/inference/README.md` (currently a stub) — the
  three pre-conditions under which alive-inherit is sound, the
  warning emitted on violation, the workaround (`config
  :enable-alive-inherit false`).

when T1.5.4.0 / T1.5.4.2 / T1.5.4.3 were planned for this stage.

that closes the targets:

  rewrite) collapses dead-branch fan-out where the death is
  predictable in one rule firing. Most of the demo-03/04/05 gap
  is exactly this pattern (cross-attribute co-located deaths via
  the sibling-exclusive chain).
- **S1.5.7** (back-prop on unconditional death) caches the
  `(not h)` write from the first dead sibling so descendants
  skip via Filter C in O(1).
- **S1.5.5** (auto-closure inference) trims the cases where the
  puzzle author didn't think to add `(closed R)` explicitly.

S1.5.4 ships the **structural** changes (config head + counters +
inherited alive set + `(closed R)` activator + filter-pipeline
refactor); the **node-count target** ships when S1.5.5/6/7 land.

## Open questions parked here

- **Q-S1.5.4.A — Closed-by-default for kernel-meta relations?**
  Once `instance` and `type` ship as ordinary relations via the
  proto-library ([P1.7 T1.7.2.5](../p1.7_bootstrapping_zebra/s1.7.2_dynamic_vs_hardcoded.md)),
  should the proto-library also assert `(closed instance)` /
  `(closed type)`? It would make zebra-original puzzles behave
  the same way as zebra2's `(closed is-a)`-style — fewer junk
  hypotheses for free. Default answer: yes, but the puzzle can
  override by NOT importing that line of the proto-library. Park
  until the proto-library lands.
- **Q-S1.5.4.B — Generalisation: relation-level pre-screening for
  hypothesis viability.** Should the generator also skip relations
  whose signature has no *unfilled* slot-instance pair? E.g.
  `(relation right-of House House)` where every House at slot 0
  already has a `right-of` fact emitted. Generalises the "is-a is
  closed because every leaf has a parent" idiom. Defer until F7
  rule induction picks it up.
- **Q-S1.5.4.C — Per-hypothesis vs per-state caching.** S1.5.3
  caches at state-hash granularity (a whole post-saturation KB
  reached twice → collapse). S1.5.4 caches at hypothesis-key
  granularity (a single `(R A B)` known to fail unconditionally).
  Both compose; they don't subsume each other. Confirm with a
  small benchmark on demo 05.
- **Q-S1.5.4.D — What happens when the alive-inherit invariant
  breaks?** Rules-as-data ([F5](../../followups/f5_rules_as_data.md))
  or any rule library that asserts new `(relation …)`
  declarations would violate the fixed-relations clause T1.5.4.5
  rests on. Two reasonable responses:
  (a) re-enumerate the alive set when a rule fires that affects
      the relation registry (rare in practice);
  (b) declare such rules out-of-scope for the optimisation and
      require explicit `:enable-alive-inherit false` per puzzle.
  M1 ships (b) as the escape hatch — the warning T1.5.4.8.d
  documents tells the author to flip the flag. Park (a) until F5
  lands and a measurable use case appears.
- **Q-S1.5.4.E — Config head shape: IR form vs solver kwarg vs
  both?** T1.5.4.4 proposes both: `(config …)` as a 7th top-level
  IR form (the persistent default a puzzle ships with) +
  `solve(kb, *, config=…)` kwarg (the caller's override).
  Alternative A — kwarg only, no IR form; configs live in the
  caller (test, CLI, REPL). Alternative B — IR form only, no
  kwarg; callers edit the IR or load a config-only IR file.
  Alternative C — kwarg + per-rule property facts (`:flag (…)`)
  scattered across the rule library, no top-level form. The
  "both" proposal matches `(query …)`'s shape — a top-level form
  with a kwarg override — and keeps configurations homoiconic
  (subject to the same parser, dump, DOT rendering as the rest
  of the IR). Decide before T1.5.4.4.a lands; if "kwarg only"
  wins, T1.5.4.4.a/c collapse and T1.5.4.7/.8 acceptance loses
  the "IR round-trips" requirement.
