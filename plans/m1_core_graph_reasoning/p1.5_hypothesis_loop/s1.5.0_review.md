# S1.5.0 — Review of P1.5 against shipped P1.2–P1.4

**Phase:** P1.5 (hypothesis loop).
**Type:** *meta-stage* — produces no code; precedes the S1.5.x
rewrites.
**Status:** open, 2026-05-21 (this audit).
**Depends on:** P1.2 (shipped), P1.3 (shipped), P1.4 (shipped 2026-05-21).
**Blocks:** S1.5.1, S1.5.2, S1.5.3 (each needs revisions before
implementation can start).

## Why this stage exists

The S1.5.x stage files were drafted **before** the P1.3 rule
engine, the P1.4 contradiction detector, and the Q40 nested-fact
matcher had shipped. Several design decisions resolved during
P1.2 / P1.3 / P1.4 invalidate parts of the original P1.5 spec:

- The engine has no `verify()` API — P1.4 ships
  `ContradictionDetector.detect()` instead (different shape, fewer
  features).
- Q33 (2026-05-20) deferred all domain / cardinality primitives
  to followups; P1.5's original "most-constrained node via
  `dom(?a)`" hypothesis-generation strategy has no substrate to
  build on.
- Q40 Option A made `hypothesis-contradiction` a normal rule that
  the matcher fires when the engine emits `(hypothesis ?h)` +
  `(contradiction-under ?h)` synthetic facts. The original plan
  didn't describe this integration — it predates Q40.
- API names drift: `Graph` → `KnowledgeBase`, `g.fork()` →
  `kb.fork()`, `EdgeId` → `Fact` references, `Registry` →
  `kb.rules`, `Violation` → `Contradiction`.

Plus a few smaller drifts (module path, symmetry IR shape).

## Verdict

P1.5 needs a **substantial rewrite**, not a complete redesign.
The *outer-loop algorithm* (lazy branching, recursive hypothesis,
backjump on contradiction) is sound and stays. What changes:

1. **Re-base on shipped APIs** — KnowledgeBase, Saturator, Fact,
   ContradictionDetector, kb.unsat_core (already shipped in P1.2
   S1.2.3).
2. **Replace the domain-based hypothesis strategy** with one that
   uses the shipping data model only.
3. **Wire the Q40 hypothesis-contradiction rule** into the
   fork/contradict path — the engine already supports it; the loop
   just needs to emit the synthetic facts.
4. **Settle the symmetry IR shape** — either ship a grammar
   extension or defer the IR-form work and hand-canonicalise.

Rough estimate: stays at ~2 weeks. S1.5.1 grows slightly (it
absorbs the Q40 wiring); S1.5.3 shrinks if symmetry is deferred.

## B. The domain-shaped hole — two-step hypothesis generation

The original T1.5.1.2 strategy:

> 1. Pick the most-constrained unsolved node — the one with the
>    smallest live domain (`dom(?a)`). Ties broken by node id.
> 2. For each value `v ∈ dom(?a)`, produce a candidate fact
>    `(= ?a v)` with `kind='hypothesis'` provenance.
> 3. Eliminate candidates that immediately violate a constraint
>    without saturation (`verify_incremental`).

is unimplementable as written because Q33 deferred all
domain / cardinality / set-arithmetic primitives. M1 has no
`dom(?a)`, no "live domain", no "smallest" comparison.

### Replacement — two-step generation, fail-fast by relation count

The shipping engine generates hypotheses in **two steps**: pick an
**object** (instance), then pick a **relation + slot + filler**.
The fail-fast heuristic is dual to CSP's "smallest domain": pick
the *most-constrained* object — the one with the **most existing
facts mentioning it**. A new hypothesis on a well-constrained
object is more likely to clash with what's already there, so dead
branches die early.

#### B.1 Object selection

```python
def pick_object(kb):
    """Most-constrained-first: instance with the largest fact-count."""
    return max(
        kb.instances.values(),
        key=lambda obj: len(kb._facts_by_instance.get(obj.name, ())),
    )
```

For Zebra after initial saturation: Norwegian, Blue, House-1 (all
involved in spatial chains) lead the count. The loop picks one of
them and works on its remaining hypothesis sites.

#### B.2 Relation selection — `possible(obj) − existing(obj)`

For the chosen object, enumerate `(R, slot_idx)` tuples where:

- **possible(obj)** = `{ (R.name, i) : R ∈ kb.relations,
  i ∈ range(len(R.signature)), R.signature[i] == obj.type_name }`
  — the relation slots the object is type-compatible with.
- **existing(obj)** = `{ (f.relation_name, i) : f ∈ obj.facts,
  i ∈ range(len(f.args)), f.args[i] == obj.name }`
  — the slots the object already occupies in some fact.

`candidates(obj) = possible(obj) − existing(obj)` is the set of
slot positions where new hypotheses can live.

For Norwegian (a Nationality / Attribute instance) in Zebra,
`possible` includes `(co-located, 0)`, `(co-located, 1)`,
`(next-to, 0)`, `(next-to, 1)`, `(right-of, 0)`, `(right-of, 1)`
— 6 slots. `existing` includes `(co-located, 0)` (from
`(co-located Norwegian House-1)`), `(next-to, 0)` (from
`(next-to Norwegian Blue)`). So `candidates` includes the 4
remaining slots — plenty of hypothesis sites.

#### B.3 Arg enumeration

For each `(R, slot_idx) ∈ candidates(obj)`, fill the **other**
slot(s) with type-compatible instances:

```python
def enumerate_hypotheses(kb, obj, R_name, slot_idx):
    R = kb.relations[R_name]
    other_slots = [i for i in range(len(R.signature)) if i != slot_idx]
    # Recursive: for arity > 2, multi-fill. M1 only has arity-2.
    for other_i in other_slots:
        for other_v in _instances_of_type(kb, R.signature[other_i]):
            args = _build_args(obj.name, slot_idx, other_v.name, other_i)
            if _is_excluded(kb, R.name, args):
                continue            # (not (R args)) already in KB
            yield Fact(relation_name=R.name, args=args,
                       layer=Layer.REASONING,
                       provenance=Provenance.from_hypothesis(branch=...))
```

`_is_excluded` consults `kb._facts_by_relation["not"]` to skip
candidates whose negation is already derived (e.g.,
type-exclusivity has pre-negated most same-type co-located pairs).

#### B.4 Symmetric relations — emit BOTH orderings

For `(symmetric R)` activator, the candidate generator emits both
`(R a b)` and `(R b a)` as **separate** hypothesis facts, not just
one canonical ordering. Reasoning: emitting both costs one extra
hypothesis branch (small memory cost); emitting only one risks
**delaying contradiction** that would surface earlier through the
other ordering's seed. Trade memory for time.

Saturation's `symmetric` rule firings will derive the missing
direction anyway; the gen-time duplication just front-loads it.

#### B.5 What this loses vs the original spec

- No *domain narrowing* mid-saturation — domains aren't
  represented, just enumerated on demand from `possible(obj) −
  existing(obj)`.
- Arc consistency, unit propagation, etc. don't apply — those are
  Q33-deferred. The branching strategy compensates by being more
  eager about enumeration, but is slower than a real CSP solver
  on hard puzzles. For Zebra-class problems with ≤ 5 hypothesis
  levels, it's fine.

## C. Q40 wiring — the missing integration

P1.3 ships the `hypothesis-contradiction` rule:

```lisp
(rule hypothesis-contradiction ()
  :match  (and (hypothesis ?h)
               (contradiction-under ?h))
  :assert (not ?h)
  :priority 900)
```

Q40 Option A established that **the engine emits synthetic
`(hypothesis ?h)` and `(contradiction-under ?h)` facts at fork
and contradict moments**; the rule then handles the negation in
the parent layer via the matcher's nested-fact unification.

P1.5's job is to be the **emitter**:

- On `fork(hypothesis_fact)`:
  - In the fork's REASONING layer, write the hypothesis_fact
    itself with `Provenance.from_hypothesis(branch=N)`.
  - In the fork's REASONING layer, write the synthetic
    `(hypothesis <hypothesis_fact>)` fact — a relational-node arg
    (R9 widening lets us).
- On the fork's saturation completing AND the contradiction
  detector finding ≥ 1 `Contradiction`:
  - In the fork's REASONING layer, write the synthetic
    `(contradiction-under <hypothesis_fact>)` fact.
  - Re-run the saturator one cycle. The
    `hypothesis-contradiction` rule fires; its asserted `(not
    <hypothesis_fact>)` enters the **parent** KB's REASONING
    layer (this is the only fact P1.5 propagates upward).
  - Mark the fork's branch-scoped facts with `kind='rejected'`
    via `Provenance.rejected(branch=N)` for the trace renderer.
- On clean quiescence:
  - The fork survives. If `solved()` ⇒ keep; else recurse.

The plan needs an explicit section on this protocol — the engine
already does its half, P1.5 just orchestrates.

## D. Symmetry — not an engine concern (S1.5.3 dropped)

**User direction 2026-05-21:** symmetry handling does not belong
in M1. It's an **ontology-writer** concern, not an engine-runtime
concern. Once a `.ein` file declares `House-1`, …, `House-5` plus
the spatial facts `(right-of House-2 House-1)` etc., the
left-to-right orientation is *already committed* by construction.
The engine sees unique named instances; there is no second
orientation for it to explore.

The "WLOG numbering" remark in the human walkthrough
([idea 08 target trace](../../../docs/ideas/08-human-style-deductive-trace.md#the-target-trace-paraphrased))
documents the ontology writer's choice — not a derivation step the
engine made. In M2's NL → IR pipeline, this same decision moves
upstream to the parser that maps NL position references to named
House instances.

**Consequence:** S1.5.3 is **dropped from M1**. No grammar
extension, no property-fact convention, no hand-canonicalisation.
The engine does no symmetry detection at all.

[Q6 — symmetry pre vs post-trace](../open_questions.md#q6--symmetry-breaking)
is updated accordingly (the engine-time canonicalisation working
answer is replaced by "ontology-writer concern").

Future puzzles with non-trivial symmetry (logic-grid with row/
column flip, chessboard with rotation) inherit the same separation:
the ontology writer picks a convention; the engine consumes the
committed form. If a future puzzle's ontology has *latent*
symmetry that can't be committed at write time — e.g., a graph
isomorphism on an abstract group — that's a research direction
beyond M1's scope.

## E. Mode + query plumbing

The original T1.5.1.4 plug-in is correct in shape but needs to
wire to the shipped `kb.query: Query | None`:

- `kb.query.kw_pairs` carries the parsed `(query :mode … :goal …)`
  contents.
- The `:mode` value is an `Atom` (e.g. `Atom("solve")`); read it
  to pick `Mode = SOLVE | GAPS | CONTRADICTIONS`.
- The `:goal` value is an `IRNode` (`SForm` for most goals); free
  vars are the unbound positions the loop tries to determine.

`Saturator.solved()` already exists as a stub returning False; the
loop sets `solved()` per mode by calling a callback or by
swapping in a configured Saturator subclass / sibling. Cleaner
shape: the loop owns the `solved()` predicate; the Saturator
remains mode-agnostic.

## F. Search tree IS the proof object

User-direction reframing (2026-05-21): **the search tree the
hypothesis loop builds is not just a search artefact — it's a
constructive proof** of the puzzle's solution (or, in
*contradictions* mode, of its unsolvability).

- Every **dead** branch's unsat-core is a literal proof of `(not
  H)` for the hypothesis H that seeded it — Q40's
  `hypothesis-contradiction` rule materialises this proof step:
  "assumed H; derived ⊥; therefore (not H)".
- The **surviving** branch is the constructive part: every fact
  in REASONING carries provenance back to the source facts that
  justified it.
- The **tree as a whole** is the proof object that witnesses
  uniqueness: solution X is unique because every alternative is
  refuted, each with its own constructive sub-proof.

Consequences:

1. **P1.6 trace renderer** must serialise the whole tree (live
   branches and dead branches alike); dead branches are not noise
   to filter out — they're the proof of uniqueness.
2. **Idea-03's three task classes** map cleanly: `solve` returns
   the unique survivor + its proof; `gaps` returns the multiple
   survivors with per-branch proofs; `contradictions` returns the
   dead branches' unsat-cores directly. The same tree services
   all three.
3. **Idea-08's "explanation-completeness"** is about traversing
   the tree, not just the successful path. The human walkthrough's
   "if Yellow were in H_3, then …, contradiction; so Yellow is
   not in H_3" steps are dead-branch unsat-cores.
4. **F1 (categorical formulation)** gains a concrete target: the
   proof object is a morphism in some `Proof` category.

### Search-tree IR — round-trip through existing `(trace …)`

The `(trace …)` top-level form is already in the grammar with
sub-forms `step`, `branch-open`, `branch-close`, `contradiction`,
`symmetry-class`. The Python `SearchTree` is a structural view
over those forms.

| `SearchNode` field        | Trace IR form                                         |
|---------------------------|-------------------------------------------------------|
| `verdict='solution'`      | the `branch-close` with no preceding `contradiction`   |
| `verdict='contradiction'` | `(contradiction step-N :unsat-core (…))`               |
| `verdict='open'`          | `branch-open` without a matching `branch-close`        |
| `hypothesis: Fact \| None` | `(branch-open step-N :on <hypothesis-fact>)`          |
| `children`                | sibling `branch-open` / `branch-close` pairs           |
| `state_hash` (S1.5.3)     | `:state-hash` kw on `branch-open` (optional, for dedup)|

The IR is the canonical serialisation of the proof; P1.6 reads it
to render markdown traces and DOT diagrams.

## F2. Canonical state + dedup + alive-branch termination

User direction (2026-05-21): redundancy collapse needs a canonical
state and a state index; termination must wait for all alive
branches; full-tree exploration *modulo dedup* is the proof.

### F2.1 Canonical state — hash the fact list only

In M1 the **ontology** (types, instances, declared relations,
rule-application property facts), **rules**, and **kb.query** are
constant across branches — no branch mutates them. The **trace**
is intentionally per-branch (same end-state via different
proof-paths is exactly the equivalence we want to catch).

So the canonical state is **only the `facts` list**:

```python
def state_hash(kb):
    return hash(tuple(sorted(
        (f.layer.value, f.relation_name, _hashable_args(f.args))
        for f in kb.facts
    )))

def _hashable_args(args):
    return tuple(
        _hashable_args(a.args) if isinstance(a, Fact) else a
        for a in args
    )
```

**Args order *inside each tuple* stays positional** — `(right-of A
B)` and `(right-of B A)` are different facts with different
semantic meanings. Only the **outer list** of facts is sorted to
produce canonical order across the set.

Nested-Fact args (Q40) hash recursively via the helper.

Cost on Zebra: ~200 facts × O(args) per tuple, plus the sort.
Sub-millisecond.

### F2.2 State index for dedup

A side table `state_hash → BranchId` mapped on the search tree.
When a new branch finishes saturation, compute its hash; if it
matches an existing branch's, **collapse** — point the parent at
the existing branch rather than adding a new tree node. Dedup
turns the tree into a DAG; the proof remains valid (same proof
of the same fact set via two paths is still one proof).

### F2.3 Alive-branch termination

A `solve()` invocation that returns early on the first
goal-matching branch is **wrong**: a deeper hypothesis level in
the same branch might surface a contradiction that invalidates
the apparent solution. Termination must explore **every alive
branch to exhaustion**.

A branch is **dead** iff `ContradictionDetector` finds a same-
layer `(X, (not X))` pair in its KB. A branch is **fully-explored**
iff:

- it's dead, OR
- it has no more hypotheses to generate AND no contradiction has
  surfaced.

The puzzle is **solved** iff:

- there exists exactly one fully-explored, non-dead branch, AND
- every other branch is dead.

If multiple non-dead fully-explored branches survive → `Ambiguity`
(GAPS mode reports them). If zero → `Contradiction` (the puzzle
itself is unsolvable; CONTRADICTIONS mode reports all the dead
branches' unsat-cores).

The "explore all alive" predicate is the M1 acceptance gate for
P1.5 — without it, the engine can declare false solutions on
puzzles where the goal is locally satisfied but globally
contradicted.
