# S1.5.0 — Review of P1.5 against shipped P1.2–P1.4

**Phase:** P1.5 (hypothesis loop).
**Type:** *meta-stage* — produces no code; precedes the S1.5.x
rewrites.
**Status:** open, 2026-05-21 (this audit).
**Depends on:** P1.2 (shipped), P1.3 (shipped), P1.4 (shipped 2026-05-21).
**Blocks:** S1.5.1, S1.5.2, S1.5.3 (each needs revisions before
implementation can start).

## Why this stage exists

The S1.5.x stage files were drafted **before** the P1.3 rule
engine, the P1.4 contradiction detector, and the Q40 nested-fact
matcher had shipped. Several design decisions resolved during
P1.2 / P1.3 / P1.4 invalidate parts of the original P1.5 spec:

- The engine has no `verify()` API — P1.4 ships
  `ContradictionDetector.detect()` instead (different shape, fewer
  features).
- Q33 (2026-05-20) deferred all domain / cardinality primitives
  to followups; P1.5's original "most-constrained node via
  `dom(?a)`" hypothesis-generation strategy has no substrate to
  build on.
- Q40 Option A made `hypothesis-contradiction` a normal rule that
  the matcher fires when the engine emits `(hypothesis ?h)` +
  `(contradiction-under ?h)` synthetic facts. The original plan
  didn't describe this integration — it predates Q40.
- API names drift: `Graph` → `KnowledgeBase`, `g.fork()` →
  `kb.fork()`, `EdgeId` → `Fact` references, `Registry` →
  `kb.rules`, `Violation` → `Contradiction`.

Plus a few smaller drifts (module path, symmetry IR shape).

## Verdict

P1.5 needs a **substantial rewrite**, not a complete redesign.
The *outer-loop algorithm* (lazy branching, recursive hypothesis,
backjump on contradiction) is sound and stays. What changes:

1. **Re-base on shipped APIs** — KnowledgeBase, Saturator, Fact,
   ContradictionDetector, kb.unsat_core (already shipped in P1.2
   S1.2.3).
2. **Replace the domain-based hypothesis strategy** with one that
   uses the shipping data model only.
3. **Wire the Q40 hypothesis-contradiction rule** into the
   fork/contradict path — the engine already supports it; the loop
   just needs to emit the synthetic facts.
4. **Settle the symmetry IR shape** — either ship a grammar
   extension or defer the IR-form work and hand-canonicalise.

Rough estimate: stays at ~2 weeks. S1.5.1 grows slightly (it
absorbs the Q40 wiring); S1.5.3 shrinks if symmetry is deferred.

## B. The domain-shaped hole — Q33 deferral

The original T1.5.1.2 strategy:

> 1. Pick the most-constrained unsolved node — the one with the
>    smallest live domain (`dom(?a)`). Ties broken by node id.
> 2. For each value `v ∈ dom(?a)`, produce a candidate fact
>    `(= ?a v)` with `kind='hypothesis'` provenance.
> 3. Eliminate candidates that immediately violate a constraint
>    without saturation (`verify_incremental`).

is unimplementable as written because Q33 deferred all
domain / cardinality / set-arithmetic primitives. M1 has no
`dom(?a)`, no "live domain", no "smallest" comparison.

### Replacement strategy — three candidate sources

The shipping engine can enumerate hypothesis candidates from three
existing surfaces, no new primitives required:

#### B.1 Goal-var enumeration

The `(query :goal …)` block names the goal pattern. Free vars in
the goal whose values aren't fixed after saturation are
candidate-hypothesis sites. Their candidate values are *all
instances of the type the var's position would accept*.

For Zebra:
```lisp
(query :goal (and (co-located ?water-drinker Water)
                  (co-located ?zebra-owner Zebra)))
```
Free vars: `?water-drinker`, `?zebra-owner`. Their `co-located`-
position arity-2 candidates are every Nationality instance
(`Englishman`, `Spaniard`, `Ukrainian`, `Norwegian`, `Japanese`).

Pick the first goal var with multiple still-consistent candidates.
"Still consistent" = the saturated KB doesn't already contain
`(not (co-located <candidate> Water))`.

#### B.2 Disjunctive consequences from rules

The puzzle's IR sometimes asserts a disjunctive fact whose
consequence the M1 engine refuses to materialise monotonically.
The canonical case: `(next-to A B)` in zebra has the unmaterialised
consequence `(right-of A B) ∨ (right-of B A)`. P1.3 explicitly
declined to ship this as a rule (it requires branching).

P1.5 picks the disjunction up: for each `(next-to ?a ?b)` fact
where neither `(right-of ?a ?b)` nor `(right-of ?b ?a)` is in the
KB (positively or negatively), hypothesise both directions.

Each is a `kind='hypothesis'` fact in the branch's REASONING layer.

#### B.3 Type-instance enumeration over fact-rels

For a relation `R` with signature `(T1, T2)` and an under-bound
position (one of arg1/arg2 of an asserted fact is fixed, the
other is free or unknown), enumerate instances of the free
position's type.

For Zebra: `(co-located Englishman ?house)` — if the engine
hasn't placed Englishman in a specific house, hypothesise each
House instance.

### Hypothesis-selection heuristic

Three sources × first-fail-ish selection (smallest count of
still-consistent candidates) replaces "smallest domain". A
candidate counts as "still consistent" if neither `(not <fact>)`
nor an equivalent type-exclusivity-derived negative already sits
in the saturated KB.

### What this loses vs the original spec

- No actual *domain narrowing* mid-saturation — domains aren't
  represented, just enumerated on demand.
- Arc consistency, unit propagation, etc. don't apply — those
  are Q33-deferred. The branching strategy compensates by being
  more eager about enumeration, but it's slower than a real CSP
  solver on hard puzzles. For Zebra-class problems with ≤ 5
  hypothesis levels (per the original acceptance), it's fine.

## C. Q40 wiring — the missing integration

P1.3 ships the `hypothesis-contradiction` rule:

```lisp
(rule hypothesis-contradiction ()
  :match  (and (hypothesis ?h)
               (contradiction-under ?h))
  :assert (not ?h)
  :priority 900)
```

Q40 Option A established that **the engine emits synthetic
`(hypothesis ?h)` and `(contradiction-under ?h)` facts at fork
and contradict moments**; the rule then handles the negation in
the parent layer via the matcher's nested-fact unification.

P1.5's job is to be the **emitter**:

- On `fork(hypothesis_fact)`:
  - In the fork's REASONING layer, write the hypothesis_fact
    itself with `Provenance.from_hypothesis(branch=N)`.
  - In the fork's REASONING layer, write the synthetic
    `(hypothesis <hypothesis_fact>)` fact — a relational-node arg
    (R9 widening lets us).
- On the fork's saturation completing AND the contradiction
  detector finding ≥ 1 `Contradiction`:
  - In the fork's REASONING layer, write the synthetic
    `(contradiction-under <hypothesis_fact>)` fact.
  - Re-run the saturator one cycle. The
    `hypothesis-contradiction` rule fires; its asserted `(not
    <hypothesis_fact>)` enters the **parent** KB's REASONING
    layer (this is the only fact P1.5 propagates upward).
  - Mark the fork's branch-scoped facts with `kind='rejected'`
    via `Provenance.rejected(branch=N)` for the trace renderer.
- On clean quiescence:
  - The fork survives. If `solved()` ⇒ keep; else recurse.

The plan needs an explicit section on this protocol — the engine
already does its half, P1.5 just orchestrates.

## D. Symmetry — grammar work vs deferral

S1.5.3 introduces a new top-level IR form:

```ein
(symmetry house-numbering :group "S2" :on House :note "…")
```

The current grammar doesn't have this form. Options:

1. **Grammar extension** — add `symmetry_decl` to ontology /
   top-level, update the loader, add `Symmetry` to entities.
   ~half a day of grammar work, plus tests + golden regen.
2. **Hand-canonicalisation** — for the one Zebra symmetry,
   hardcode a "pin House_1 to position 1" rule in P1.5's
   `hgen`. No new IR; no grammar churn. Defer the general
   symmetry IR to followups.
3. **Reuse activator-fact convention** — `(symmetric R)`-style
   property facts already exist. Add `(symmetry-class R …)` or
   `(canonical R …)` as a property fact whose loader sets up the
   canonical pin. Less grammar work; matches the existing pattern.

Recommendation: **option 3** (activator-fact convention). It's
consistent with how the existing rule library marks properties
(`(symmetric next-to)`, `(transitive co-located)` etc.) and
avoids a new grammar primitive.

Q6 (resolved earlier) is "pre-trace canonicalisation"; either
option satisfies that.

## E. Mode + query plumbing

The original T1.5.1.4 plug-in is correct in shape but needs to
wire to the shipped `kb.query: Query | None`:

- `kb.query.kw_pairs` carries the parsed `(query :mode … :goal …)`
  contents.
- The `:mode` value is an `Atom` (e.g. `Atom("solve")`); read it
  to pick `Mode = SOLVE | GAPS | CONTRADICTIONS`.
- The `:goal` value is an `IRNode` (`SForm` for most goals); free
  vars are the unbound positions the loop tries to determine.

`Saturator.solved()` already exists as a stub returning False; the
loop sets `solved()` per mode by calling a callback or by
swapping in a configured Saturator subclass / sibling. Cleaner
shape: the loop owns the `solved()` predicate; the Saturator
remains mode-agnostic.

## F. Search-tree IR — round-trip through existing forms

The `(trace …)` top-level form is already in the grammar with
sub-forms `step`, `branch-open`, `branch-close`, `contradiction`,
`symmetry-class`. The original S1.5.2 plan introduces a separate
`SearchTree` Python class; it should explicitly map to those
trace sub-forms for the round-trip.

Mapping:

| `SearchNode` field        | Trace IR form                                         |
|---------------------------|-------------------------------------------------------|
| `verdict='solution'`      | implicit (the branch-close with no contradiction-decl) |
| `verdict='contradiction'` | `(contradiction step-N :unsat-core (…))`               |
| `verdict='open'`          | the branch-open without a matching branch-close        |
| `hypothesis: Fact | None` | `(branch-open step-N :on <hypothesis-fact>)`           |
| `children`                | sibling `branch-open` / `branch-close` pairs           |

The Python `SearchTree` is then just a structural view over the
trace's branch-open / branch-close nesting.
