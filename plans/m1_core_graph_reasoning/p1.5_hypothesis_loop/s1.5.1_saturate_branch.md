# S1.5.1 — Solve driver + two-step hypothesis gen + Q40 wiring

**Phase:** P1.5 (hypothesis loop).
**Estimate:** 5-6 days.
**Depends on:** P1.2 (`KnowledgeBase`, `kb.fork`, `Provenance`),
P1.3 (`Saturator`, `Engine`, `hypothesis-contradiction` rule),
P1.4 (`ContradictionDetector`).
**Resolves question:** [M1 Q19 — hypothesis branching strategy](../../open_questions.md#q19--hypothesis-branching-strategy).
**Predecessor:** [S1.5.0 review](s1.5.0_review.md) — the audit that
re-shaped this stage's scope.

## Context

This stage builds the **outer loop** that takes over when P1.3's
saturator stalls without solving. It implements three coupled
mechanisms:

1. **Lazy branching** (Q19) — saturate first; branch only when no
   rule fires and the puzzle isn't solved.
2. **Two-step hypothesis generation** (S1.5.0 §B) — pick the most
   constrained object first (proxy for CSP's smallest-domain
   heuristic), then enumerate its still-available `(relation,
   slot, filler)` triples.
3. **Q40-Option-A protocol** (S1.5.0 §C) — emit synthetic
   `(hypothesis ?h)` + `(contradiction-under ?h)` facts so the
   `hypothesis-contradiction` rule shipped in P1.3 can fire and
   propagate `(not H)` to the parent layer.

The recursive descent (multilevel) is in S1.5.2; the canonicalisation
+ alive-branch termination is in S1.5.3. This stage produces a
*single-step* driver that the recursion in S1.5.2 will wrap.

## Tasks

### Task T1.5.1.1 — `solve` driver

`src/ein/inference/hypothesis.py`:

```python
from collections.abc import Iterator
from dataclasses import dataclass, field
from enum import Enum

from ein.kb.entities import Fact
from ein.kb.store import KnowledgeBase


class Mode(Enum):
    SOLVE          = "solve"
    GAPS           = "gaps"
    CONTRADICTIONS = "contradictions"


@dataclass(frozen=True)
class Solution:
    kb:    KnowledgeBase
    trace: tuple[Firing, ...]

@dataclass(frozen=True)
class Ambiguity:
    branches: tuple[Solution, ...]

@dataclass(frozen=True)
class Contradiction:
    unsat_core: frozenset[Fact]
    branches:   tuple["SearchNode", ...]


Verdict = Solution | Ambiguity | Contradiction


def solve(
    kb: KnowledgeBase,
    *,
    mode: Mode | None = None,
    max_depth: int = 6,
) -> Verdict: ...
```

`mode` defaults to whatever the KB's `(query :mode …)` says, with
`SOLVE` as the fallback when there's no query. `max_depth` bounds
the recursion (Zebra needs ≤ 3 in practice; 6 leaves headroom).

`Contradiction.branches` carries the dead branches the engine
chased — that's what idea-03's *contradictions* task class
returns to the user. `Contradiction.unsat_core` is the union of
the dead branches' `kb.unsat_core(…)` calls.

### Task T1.5.1.2 — Two-step hypothesis generation (`hgen`)

The hypothesis generator is the heart of S1.5.0 §B. Pure function
over a saturated KB:

```python
def generate_hypotheses(kb: KnowledgeBase) -> Iterator[Fact]:
    """Yield candidate hypothesis facts in priority order.

    Implements S1.5.0 §B's two-step framework:
      1. Pick instance with the most facts mentioning it.
      2. Enumerate (R, slot_idx) ∈ possible(obj) − existing(obj).
      3. Fill the other slots with type-compatible instances;
         prune any candidate whose negation is already in the KB.

    Yields lazily — the loop can break after the first surviving
    candidate to fork on it.
    """
    obj = _pick_object(kb)
    if obj is None:
        return
    for r_name, slot_idx in _candidate_slots(kb, obj):
        for filler in _enumerate_fillers(kb, r_name, slot_idx, obj):
            fact = _build_hypothesis(r_name, slot_idx, obj, filler)
            if _is_excluded(kb, fact):
                continue        # (not (R …)) already in KB
            yield fact
```

#### Sub-task T1.5.1.2a — Object selection

```python
def _pick_object(kb: KnowledgeBase):
    """Most-constrained-first: instance with the most facts."""
    if not kb.instances:
        return None
    return max(
        kb.instances.values(),
        key=lambda obj: (
            len(kb._facts_by_instance.get(obj.name, ())),
            obj.name,        # tiebreak deterministically
        ),
    )
```

Cost: O(|instances|) per call. Zebra has 30 instances.

#### Sub-task T1.5.1.2b — Candidate slot enumeration

```python
def _candidate_slots(kb, obj) -> Iterator[tuple[str, int]]:
    """`possible(obj) − existing(obj)`."""
    existing = {
        (f.relation_name, i)
        for f in kb._facts_by_instance.get(obj.name, ())
        for i, a in enumerate(f.args)
        if a == obj.name
    }
    for R in kb.relations.values():
        for i, T in enumerate(R.signature):
            if T == obj.type_name and (R.name, i) not in existing:
                yield R.name, i
```

#### Sub-task T1.5.1.2c — Filler enumeration

For arity-2 relations (the M1 case), the "other slot" is uniquely
determined. Iterate type-compatible instances:

```python
def _enumerate_fillers(kb, r_name, fixed_slot, obj):
    R = kb.relations[r_name]
    other_slot = 1 - fixed_slot      # arity-2 only in M1
    other_type = R.signature[other_slot]
    for filler in kb.instances.values():
        if filler.name == obj.name:
            continue                 # skip self-edge (caller's choice)
        if filler.type_name == other_type:
            yield filler
```

#### Sub-task T1.5.1.2d — Symmetric R: emit BOTH orderings

For relations with a `(symmetric R)` activator fact in the KB,
the generator emits both `(R obj filler)` and `(R filler obj)` as
separate hypothesis branches. **Trades memory for time** — both
will saturate the same closure anyway via the `symmetric` rule;
front-loading the two seeds lets contradictions surface earlier
than they would through one seed alone (see S1.5.0 §B.4).

Detection: scan `kb._facts_by_relation.get("symmetric", ())` for a
fact with `args == (r_name,)`.

#### Sub-task T1.5.1.2e — Exclusion check

```python
def _is_excluded(kb, fact):
    """True iff (not fact) is already in the KB (any layer)."""
    not_facts = kb._facts_by_relation.get("not", ())
    for n in not_facts:
        if not n.args:
            continue
        inner = n.args[0]
        if isinstance(inner, Fact) and (
            inner.relation_name == fact.relation_name
            and inner.args == fact.args
        ):
            return True
    return False
```

Cost: O(|`not`-facts|). Zebra has ~120 negatives after
saturation. Acceptable.

### Task T1.5.1.3 — Single-branch test cycle

The driver's inner loop, per hypothesis:

```python
def try_branch(parent_kb: KnowledgeBase,
               hypothesis: Fact,
               branch_id: int) -> BranchResult:
    """Fork the parent KB, seed with hypothesis, saturate, detect."""
    fork = parent_kb.fork()

    # Step 1: write the hypothesis fact with hypothesis provenance.
    h_fact = Fact(
        relation_name=hypothesis.relation_name,
        args=hypothesis.args,
        layer=Layer.REASONING,
        provenance=Provenance.from_hypothesis(branch=branch_id),
    )
    fork.add_fact(h_fact)
    fork._index_fact(h_fact)

    # Step 2: emit synthetic (hypothesis <h_fact>) — Q40 Option A.
    synthetic_h = Fact(
        relation_name="hypothesis",
        args=(h_fact,),
        layer=Layer.REASONING,
        provenance=Provenance.from_hypothesis(branch=branch_id),
    )
    fork.add_fact(synthetic_h)
    fork._index_fact(synthetic_h)

    # Step 3: saturate.
    sat = Saturator(fork)
    firings = list(sat.saturate(max_steps=10_000))

    # Step 4: contradiction check.
    contradictions = sat.contradictions()
    if contradictions:
        # Step 5: emit (contradiction-under <h_fact>) — Q40 protocol.
        contra_fact = Fact(
            relation_name="contradiction-under",
            args=(h_fact,),
            layer=Layer.REASONING,
            provenance=Provenance.from_hypothesis(branch=branch_id),
        )
        fork.add_fact(contra_fact)
        fork._index_fact(contra_fact)
        # Step 6: one more saturation cycle — fires
        # `hypothesis-contradiction` which asserts (not h_fact)
        # for propagation to the parent on retract.
        extra_firings = list(sat.saturate(max_steps=100))
        firings.extend(extra_firings)
        return BranchResult.dead(
            branch_id=branch_id,
            hypothesis=h_fact,
            unsat_core=fork.unsat_core(
                p.positive for p in contradictions
            ),
            firings=tuple(firings),
        )

    return BranchResult.alive(
        branch_id=branch_id,
        hypothesis=h_fact,
        kb=fork,
        firings=tuple(firings),
    )
```

The hypothesis-contradiction rule (P1.3) does most of the proof
work — this function just emits the two synthetic facts at the
right moments. The `(not h_fact)` derivation lands in the fork's
REASONING layer; S1.5.2's recursion propagates the rejection by
**not** writing the fork's hypothesis fact into the parent (the
fork is discarded except for its unsat-core record).

### Task T1.5.1.4 — `solved()` predicate per mode

Wire `Saturator.solved()` (currently a stub) to the mode + the
KB's query:

```python
def is_solved(kb: KnowledgeBase, mode: Mode) -> bool:
    if mode is Mode.CONTRADICTIONS:
        return False             # always run to exhaustion
    if kb.query is None:
        return False
    goal = _query_goal(kb.query)
    if goal is None:
        return False
    # Compile the goal as a pattern; check whether at least one
    # binding satisfies it in the current KB. SOLVE wants exactly
    # one match; GAPS accepts ambiguity but each goal-var must
    # have at least one candidate.
    matches = list(_match_goal(kb, goal))
    if mode is Mode.SOLVE:
        return len(matches) == 1
    if mode is Mode.GAPS:
        return len(matches) >= 1
    return False
```

Reads `kb.query.kw_pairs` for `:mode` and `:goal`. The matcher
reuses `inference.compile` + `inference.match` modules from
S1.3.1.

### Task T1.5.1.5 — Tests

`tests/inference/test_hypothesis.py`:

- **Trivial pass-through** — KB already solved (e.g., a 1-fact
  puzzle where the goal pattern matches once). `solve(kb)`
  returns `Solution` without forking.
- **Single-hypothesis branch** — 2-instance puzzle, one rule
  fires only after a hypothesis is seeded. Verify one fork,
  saturation, return.
- **Branch dies via contradiction** — hypothesis triggers
  `type-exclusivity` to derive `(not h)`; the
  `hypothesis-contradiction` rule fires; `try_branch` returns
  `BranchResult.dead` with non-empty unsat-core.
- **Object selection picks max-fact-count instance** — set up a
  KB where one instance has 3 facts and another has 1; verify
  the gen picks the first.
- **Symmetric R emits both orderings** — verify the gen yields
  both `(R a b)` and `(R b a)` when `(symmetric R)` is in the KB.
- **Exclusion check uses `(not …)`** — pre-seed a `(not h)` fact,
  verify the gen skips that hypothesis.
- **Mode-aware `solved()`** — three small KBs, one per mode; the
  predicate returns the expected verdict.
- **Q40-protocol end-to-end** — manually trace through a single
  branch test cycle on a tiny KB; verify the `(hypothesis …)`
  and `(contradiction-under …)` synthetic facts appear in the
  fork's REASONING layer in the expected order.
