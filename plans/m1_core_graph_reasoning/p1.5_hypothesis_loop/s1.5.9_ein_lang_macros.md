# S1.5.9 — ein-lang pattern macros

**Phase:** P1.5 (hypothesis loop) — follow-up to S1.5.8c.
**Type:** kernel feature + stdlib migration. Not M1-blocking
([S1.5.8c](s1.5.8c_final_decision.md) ships forall/open as
SForm desugaring; this stage moves them to ein).
**Status:** parked — 2026-05-24.
**Depends on:** [S1.5.8c.3a / .3b](s1.5.8c_final_decision.md#34--parser-sugar-forall-b-g-b)
shipped (the SForm desugaring for forall/open is the

but preserves the semantics exactly).
**Blocks:** P1.8 stdlib import work (macros are the natural

## Why this stage exists

S1.5.8c shipped `(forall ?b (G) (B))` and `(open P)` as
compile-time parser sugars (T1.5.8c.3a / .3b). The
implementation is a pair of hardcoded arms in
`inference/compile.py`'s `_compile_premise` that synthesise
the desugared SForm tree and recursively compile it.

This works but pulls the sugar into Python. Adding a new
pattern alias means another arm in compile.py, more imports,
more test churn. The kernel acquires every piece of syntactic
sugar as a permanent extension. That contradicts S1.5.8c's
own "minimal kernel, everything else in stdlib ein" framing.

The macro system rehomes forall / open as **ein source
code**: a small generic AST-rewrite engine in the loader
applies user-defined `(macro …)` declarations to every
rule's `:match` clause before the compiler sees it. The
compiler stays ignorant of forall / open. New sugar is
added by writing more `(macro …)` declarations — no kernel
edit, no Python touch.

User direction recorded 2026-05-24:
> "I dont want SForm desugaring for forall and open if we
> can use ein stdlib."

→ confirmed: do the work in this stage; S1.5.8c stays the
shipped mechanism for now.

## Design

### Grammar — new `(macros …)` top-level + `(macro …)` decl

```diff
+ ?form: ... | macros_form
+ macros_form: "(" "macros" macro_decl* ")"
+ macro_decl:  "(" "macro" SYMBOL macro_params value ")"
+ macro_params: "(" VAR+ ")"
```

`(macro NAME (?p1 ?p2 …) BODY)` defines an alias: when the
expander sees `(NAME a1 a2 …)` in any rule's `:match`, it
substitutes `?p1 → a1`, `?p2 → a2`, … into a fresh copy of
BODY and replaces the invocation with the result.

Surface example — the entire stdlib sugar file:

```lisp
(macros
  (macro forall (?b ?G ?B)
    (absent (and ?G (absent ?B))))

  (macro open (?P)
    (and (absent ?P) (absent (not ?P)))))
```

Three lines per macro. Inspectable by any ein author; no
Python touch required to add or modify.

### AST — registry

A new field on `KnowledgeBase`:

```python
self.macros: dict[str, Macro] = {}
```

```python
@dataclass(frozen=True)
class Macro:
    name: str
    params: tuple[str, ...]   # var names in declaration order
    body: IRNode              # the body AST template
    loc: Loc | None = None
```

### Loader — two-pass

Pass 1 — read every `(macros …)` block, build `kb.macros`.

Pass 2 — read rules; before storing each rule, walk its
`:match` AST and apply `_expand_macros(node, kb.macros)`:

```python
def _expand_macros(node: IRNode, macros: dict[str, Macro]) -> IRNode:
    if isinstance(node, SForm):
        head = node.head
        if isinstance(head, Atom) and head.name in macros:
            m = macros[head.name]
            if len(node.args) != len(m.params):
                raise IRLoadError(
                    f"macro {m.name}/{len(m.params)} called with "
                    f"{len(node.args)} args"
                )
            subst = dict(zip(m.params, node.args))
            expanded = _substitute(m.body, subst)
            # Re-expand: macro bodies can invoke other macros.
            return _expand_macros(expanded, macros)
        # Recurse into args.
        new_args = tuple(_expand_macros(a, macros) for a in node.args)
        return SForm(head=head, args=new_args)
    if isinstance(node, KwPair):
        return KwPair(key=node.key, value=_expand_macros(node.value, macros))
    return node  # Var/Atom/Int/etc. are leaves
```

`_substitute(template, subst)` walks the template tree;
when it hits a `Var` whose name is in `subst`, it returns
the substitute (which may itself be a complex SForm).

### Revert compile.py

Delete the `head_name == "forall"` and `head_name == "open"`
arms (T1.5.8c.3a / .3b code). Delete the `_var_in_ast`
helper if no other use. The compiler returns to handling only
the kernel premise forms (`absent`, `not`, `and`, `or`,
`instance`, predicates, generic relation patterns).

### Stdlib sugar file

Ship `examples/stdlib/sugar.ein` with the two macros above.
Each puzzle that wants the sugar either:

1. Includes the macros block inline at the top of the file
   (M1 idiom — explicit, no imports).
2. *(Later, post-P1.8 imports)* — declares `(import "stdlib/sugar")`.

## Worked example

After this stage, `examples/features/03_forall.ein` looks like:

```lisp
(macros
  (macro forall (?b ?G ?B)
    (absent (and ?G (absent ?B)))))

(rules
  (rule undefeated ()
    :match  (and (player ?p)
                 (forall ?q
                   (and (player ?q) (neq ?p ?q))
                   (beats ?p ?q)))
    :assert (undefeated ?p)
    :why    "..."))
```

Same surface as today. The macros block at the top makes
the desugaring inspectable; everything below is unchanged.

For the loader's perspective: pass 1 reads the macro,
recording `forall : (?b ?G ?B) → (absent (and ?G (absent ?B)))`.
Pass 2 reads the undefeated rule. Its :match contains
`(forall ?q (and …) (beats …))` — the expander substitutes
`?b → ?q, ?G → (and (player ?q) (neq ?p ?q)), ?B → (beats ?p ?q)`
into the template `(absent (and ?G (absent ?B)))`. Result:
`(absent (and (and (player ?q) (neq ?p ?q)) (absent (beats ?p ?q))))`.
Compile sees this expanded form, no `forall` involved.

### Task T1.5.9.4 — Safety: forall bound-var grounding check

The S1.5.8c.3a SForm desugaring includes a safety check
(`_var_in_ast`) that the bound var appears in the guard.
The macro form loses this check — `forall` is just a macro,
no special validation.

Three options:

- **(a)** Keep the safety check in the macro expander as a
  per-macro hook. Macros can declare validators in Python.
  Cross-cuts the "macros are pure stdlib" cleanness.
- **(b)** Add a general "scan must ground its bound vars"
  validator that runs after macro expansion. Catches the
  same class of error structurally — a Scan whose slot has
  unbound vars after the parent's premises is rejected.
- **(c)** Drop the safety check. A buggy forall produces a
  silently-wrong rule. Author beware.

**Recommend (b)** — structural validation is more general
and applies to hand-written nested `absent` patterns too.

## Open questions parked here

- **Q-S1.5.9.1 — Macros block placement.** Strict
  before-rules (file-order requirement) vs anywhere
  (two-pass loader). Recommend two-pass; cheaper for
  authors.
- **Q-S1.5.9.2 — Macro name collisions.** A macro named
  `not` would clobber the kernel keyword. Reject at load
  time — kernel meta-primitive names (`not`, `and`, `or`,
  `absent`, `instance`, etc.) are reserved macro-side.
- **Q-S1.5.9.3 — Recursive macros.** Bodies can invoke
  other macros (handled by re-expansion). A macro that
  expands to itself loops infinitely; cap recursion depth
  (~50) and reject with a clear error.
- **Q-S1.5.9.4 — Hygiene.** If a macro body introduces a
  fresh Var (e.g., `?gensym`), and the macro is invoked
  twice in the same rule, both expansions share the same
  Var name — possible cross-contamination. M1's
  forall/open macros don't introduce fresh vars, so this
  isn't a problem today; revisit when user macros do.
- **Q-S1.5.9.5 — Variadic macros.** `(macro and-many (?xs...) ...)`
  with rest-params. Out of M1 scope (Q33 variadic deferral).
  Macros stay fixed-arity for now.
- **Q-S1.5.9.6 — Relation-poly heads in patterns / goals.**
  M1's compile.py rejects `(?R ?a ?b)` patterns where `?R`
  isn't bound by an activator — compile.py line 257-261:
  "Unbound head var — not supported in M1's activator model".
  This bites both rule `:match` (the typecheck / domain-elim
  stdlib rules had to work around it by taking the relation
  name as an activator arg, T1.5.8c.4) and query `:goal`
  clauses (e.g. `(query :goal (?R Rex Animal))` in
  `examples/features/02_star_in_identifiers.ein`). The latter
  is marked `;;TODO S1.5.9` inline. Resolving needs a new
  matcher / asserter step type that iterates over all
  relations at runtime. Not in scope here but lands
  naturally alongside macros — both unblock more
  "everything in stdlib ein" work and feed each other:
  with relation-poly heads, more rule patterns become
  expressible; with macros, those rules' sugar lives in ein.
