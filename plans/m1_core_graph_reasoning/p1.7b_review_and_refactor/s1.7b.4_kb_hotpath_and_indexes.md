# S1.7b.4 ‚Äî KB hot-path & index refactor

**Phase:** P1.7b (Review & refactor).
**Estimate:** 2‚Äì3 days. Independent of S1.7b.2/3/5, but **bench-gated**.
**Depends on:** nothing structural; do the typed-wrapper task (T6) last as it
re-homes the others.
**Findings:** [F-KB-1/2/3/4/5/6/9/13](findings.md#kb--data-model-kbpy).
**Why bench-gated:** every change here is on the per-commitment search hot
path (`fork`/`add_fact`/`_index_fact` run inside the saturation loop, per
branch). The point is to *speed up or hold*, never regress ‚Äî so capture a PyPy
baseline first and compare after each task.

## Hot-path facts (grep-established, re-verify before editing)

- `fork()` per commitment branch; `snapshot()` per recorded solution;
  `_index_fact()` after every fact add; `rebuild_indexes()` only at load +
  inside `snapshot`.
- The inference layer reads `_facts_by_relation`, `_negated_facts`,
  `_rule_apps_by_rule`, `_facts_by_instance`, `names`. It **never** reads
  `_types_by_parent`, `_instances_by_type`, `_rules_by_relation`,
  `_rules_by_type` (entity `@property` accessors only), and no
  `add_relation/add_rule/add_hrule` caller exists outside the loader ‚Üí those
  four indexes are **immutable post-load**. Guard this invariant with a comment
  (or an `assert` at the fork seam) since T1/T5 lean on it.

## Tasks

### T1.7b.4.1 ‚Äî Fix the `add_fact` / `_index_fact` double-index (F-KB-3) ‚Äî **bug**

`add_fact` (`store.py:257`) may return a *pre-existing* Fact on the dedup path;
every caller then unconditionally `_index_fact(stored)` (`firing.py:171`,
`back_prop.py:213`, `commitment.py:115`, `closed.py:81`, `solver.py:{348,400,
1070}`), appending the same Fact to the indexes twice (no membership guard at
`store.py:462`). Make the contract single-owner:

```python
def add_fact(self, fact) -> Fact:
    existing = self._fact_by_id(fact.relation_name, fact.args)  # F-KB-4 below
    if existing is not None:
        return existing
    _attach(fact, self)
    self.facts.append(fact)
    self._index_fact(fact)          # add_fact now owns indexing
    return fact
```

Drop the now-redundant `kb._index_fact(stored)` from all seven callers.
**New regression test:** a fact derivable by two rules (e.g. a symmetric
closure) appears **exactly once** in `kb._facts_by_relation[rel]` and once in
the relevant `names` entry after saturation (fails before ‚Äî duplicate entries;
passes after).

### T1.7b.4.2 ‚Äî O(n)‚ÜíO(deg) dedup (F-KB-4)

The dedup in T1 uses `_fact_by_id` (`store.py:570`, an O(deg(relation))
index lookup) instead of the old `for existing in self.facts` linear scan
(`:269-273`). Quadratic saturation ‚Üí near-linear. (Folded into T1's snippet.)

### T1.7b.4.3 ‚Äî In-place index writes (F-KB-5)

`_index_fact` (`store.py:469-529`) rebuilds whole dicts via `{**d, k:‚Ä¶}` per
fact, and `{**self.names, name:‚Ä¶}` **per name per fact** ‚Üí O(\|names\|) per
add. Since `fork()`/`snapshot()` already give each kb its own dict objects
(`store.py:670-676`), in-place assignment is isolated and O(1):

```python
self._facts_by_relation[rn] = (*self._facts_by_relation.get(rn, ()), fact)
self.names[name] = NameRef(...)        # in the _bump_* helpers
```

Keep the `NameRef`/tuple **values** immutable (only the dict binding changes).
This is the single biggest per-fact win.

### T1.7b.4.4 ‚Äî `fork()` shares the four load-invariant indexes (F-KB-1)

`fork()` (`store.py:668-673`) currently `dict(...)`-copies `_types_by_parent`,
`_instances_by_type`, `_rules_by_relation`, `_rules_by_type`. Share them by
reference like `types`/`rules` (`:652-656`). Small magnitude (these dicts are
tiny) but free and provably safe given the post-load invariant above. Continue
copying the genuinely-mutated `_facts_by_relation`/`_facts_by_instance`/
`_rule_apps_*`/`names`/`_negated_facts`.

### T1.7b.4.5 ‚Äî `snapshot()` shallow-copies instead of rebuilding (F-KB-6)

`snapshot()` (`store.py:697-750`) calls a full `rebuild_indexes()` (`:749`)
though it copies `facts` verbatim from a kb whose indexes are already
consistent. Replace with the same shallow-copy pattern as `fork` (share the
four static indexes, copy the mutable ones). Warm path (per solution), so
secondary to T1-T3 but the same reasoning.

### T1.7b.4.6 ‚Äî Decompose `rebuild_indexes` + fuse the double walk (F-KB-2)

`rebuild_indexes` (`store.py:278-441`, 164 ln / 61 br) builds 8 indexes inline
and walks `self.facts` **twice** (`:351-356` and `:382-388` collect nearly the
same head/arg groupings). Split into `_derive_types_instances()`,
`_build_type_indexes()`, `_build_fact_indexes()`, `_build_names_index()`,
`_build_rule_indexes()`; fuse the two fact walks into one pass that emits both
the Fact-tuple maps and the `NameRef` head/arg maps. `rebuild_indexes` becomes
an ordered list of calls.

### T1.7b.4.7 ‚Äî Typed index wrappers (F-KB-9) + `Any` annotations (F-KB-13)

- Introduce `_FactIndexes` (the 4 mutable maps, `add(fact)`/`copy()`) and
  `_StaticIndexes` (the 4 load-invariant maps, `copy()` = identity/shared).
  `fork` ‚Üí `new.fact_indexes = self.fact_indexes.copy(); new.static_indexes =
  self.static_indexes` ‚Äî the perf intent of T3/T4/T5 becomes **encoded in the
  type**, not a comment, and `_index_fact`'s append idiom lives in one place.
  The wrappers live in a new `kb/indexes.py`, shrinking the 769-line
  `store.py` (the "too-long file" axis). Do this **last** (it re-homes T1-T6);
  keep public accessor behaviour.
- F-KB-13: annotate `Query.config`/`alive`/`consume_stats`/`kw_pairs`
  (`store.py:98,133,141,149`) via `if TYPE_CHECKING:` imports instead of `Any`
  (the module already imports `DerivationDAG`/`FactView` that way).
