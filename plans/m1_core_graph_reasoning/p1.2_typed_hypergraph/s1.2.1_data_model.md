## Context

The IR (P1.1) is a syntax tree. The engine doesn't reason over
S-expressions; it reasons over **domain entities** — types,
relations, rules, instances, facts — with explicit cross-references
between them. The data model is the in-memory representation of
those entities with *navigable* relationships so the engine, the
trace renderer, and the LLM prompt generator (M2) can ask questions
like:

- *Which rules apply to `right-of`?*
- *Which relations does the `square-fwd` rule mention?*
- *Which types does `type-exclusivity` quantify over?*
- *Which facts reference the `Norwegian` instance?*
- *Which instances of `House` exist?*

…without re-walking the IR each time.

**This is not a generic graph library.** `Graph + Node + Edge` is the
*rendering*, not the data model. The data model is a small entity-
relational schema; nodes/edges are derivable from it on demand
(`docs/ir.md` §6, S1.1.4). The PoC's `Node`-`Edge`-`State` framing is
not carried over.

## Entities

| entity     | source IR form                              | identity      |
|------------|---------------------------------------------|---------------|
| `Type`     | `(type Name [Parent])` in ontology          | name          |
| `Instance` | `(instance Subj Type)` in ontology          | name          |
| `Relation` | `(relation Name (T1 T2 …))` in ontology     | name          |
| `Rule`     | `(rule Name params …)` in rules             | name          |
| `Fact`     | `(rel args …)` in ontology / facts / reasoning | (rel, args)   |

Names are interned globally (one `Type` per name, etc.). Re-loading
the same IR is idempotent.

## Cross-references

The KB pre-computes these so each is O(1) lookup or O(deg) iteration:

```python
type.parent      -> Type | None
type.children    -> tuple[Type, ...]      # direct subtypes
type.ancestors() -> Iterator[Type]        # parent chain
type.instances   -> tuple[Instance, ...]  # via (instance _ Type) facts
type.rules       -> tuple[Rule, ...]      # rules whose pattern names this type

instance.type    -> Type
instance.facts   -> tuple[Fact, ...]      # facts mentioning this instance
                                          # (any position)

relation.signature -> tuple[Type, ...]    # from (relation _ (T1 T2 …))
relation.properties -> tuple[Fact, ...]   # rule-application facts targeting it
                                          # e.g. (symmetric R) → property of R
relation.rules     -> tuple[Rule, ...]    # rules whose pattern names this rel,
                                          # OR whose name is the head of a
                                          # property fact on this rel
relation.facts     -> tuple[Fact, ...]    # facts asserting an instance of this
                                          # relation, all layers

rule.params         -> tuple[Var, ...]    # from (rule Name (?p1 ?p2 …))
rule.match          -> Pattern            # :match clause
rule.assert_        -> Pattern            # :assert clause (`assert` is a kw)
rule.why            -> str
rule.priority       -> int | None
rule.relations      -> tuple[Relation, ...]   # named in match/assert
rule.types          -> tuple[Type, ...]       # touched via Instance patterns
                                              # — i.e. types of vars bound by
                                              # `(instance ?x ?T)` patterns
rule.applications   -> tuple[Fact, ...]   # property-facts that invoke this
                                          # rule on a specific relation

fact.relation     -> Relation
fact.args         -> tuple[Instance | Var, ...]
fact.layer        -> Layer                # ONTOLOGY | FACT | REASONING
fact.provenance   -> Provenance | None    # see S1.2.3
```

The reverse-indexed views are precisely the API the trace renderer
and the rule scheduler need. They are *materialised* once at KB load
time; mutation (new derived facts) incrementally maintains them.

## Tasks

### Task T1.2.1.1 — Entity dataclasses

`src/ein_bot/kb/entities.py`. Frozen dataclasses for `Type`,
`Instance`, `Relation`, `Rule`, `Fact`, `Pattern`. Each holds direct
fields (name, signature, params, etc.); cross-reference fields are
populated by the `KnowledgeBase` (next task), not by the dataclass
itself. Equality and hashing are by *name* (or `(rel, args)` for
Fact), not by identity.

### Task T1.2.1.2 — `KnowledgeBase` and indexes

`src/ein_bot/kb/store.py`. The `KnowledgeBase` is the registry plus
all cross-reference indexes:

```python
class KnowledgeBase:
    types:     dict[str, Type]
    instances: dict[str, Instance]
    relations: dict[str, Relation]
    rules:     dict[str, Rule]
    facts:     list[Fact]                 # all layers; layer is a field

    # Reverse indexes — keyed by the *target* entity:
    _instances_by_type:  dict[str, list[Instance]]
    _facts_by_relation:  dict[str, list[Fact]]
    _facts_by_instance:  dict[str, list[Fact]]
    _rules_by_relation:  dict[str, list[Rule]]
    _rules_by_type:      dict[str, list[Rule]]
    _properties_by_rel:  dict[str, list[Fact]]   # rule-application facts
```

Indexes are rebuilt after every batch ingest. Single-fact additions
(derived facts in reasoning layer) update incrementally.

### Task T1.2.1.3 — Cross-reference query API

The reverse indexes are exposed as *attributes / methods on the
entities*, not as raw dicts:

```python
# read
norwegian = kb.instances["Norwegian"]
norwegian.type                   # Type("Nationality")
norwegian.facts                  # tuple[Fact, ...]

right_of = kb.relations["right-of"]
right_of.rules                   # (Rule("square-fwd"), Rule("square-bwd"),
                                 #  Rule("implies"))
right_of.properties              # ((square-fwd right-of), (square-bwd right-of),
                                 #  (implies right-of next-to))

type_exclusivity = kb.rules["type-exclusivity"]
type_exclusivity.relations       # (Relation("instance"), Relation("co-located"))
type_exclusivity.types           # (Type("Attribute"), …)  — any T such that
                                 # the rule patterns `(instance ?_ ?T)`
```

The entity carries a back-pointer to its `KnowledgeBase` so these
properties resolve at access time.

### Task T1.2.1.4 — Load from IR

`src/ein_bot/kb/from_ir.py`. `KnowledgeBase.from_ir(ir_forms)` walks
the parsed IR and populates the KB. Order is fixed:

1. `(ontology …)` schema heads → `Type`, `Relation`
2. `(ontology …)` instance forms → `Instance`
3. `(ontology …)` rule-application facts → `Fact` (layer=ONTOLOGY)
4. `(ontology …)` other facts (pairwise structural etc.) → `Fact`
5. `(rules …)` → `Rule`
6. `(facts …)` → `Fact` (layer=FACT, with `:source`)
7. `(reasoning …)` → `Fact` (layer=REASONING, with `:rule` / `:using`)
8. `(query …)` → kept as a separate `Query` object
9. After all ingest: rebuild indexes once.

Validation lives here (or in a future T1.2.1.6): every `Instance`
type must be a declared `Type`; every relation name in a `Fact`
should be a declared `Relation` (warn if unknown — open-world);
every `Rule`'s `:match` / `:assert` is a parseable `Pattern`.

### Task T1.2.1.5 — Pattern as a sub-entity

`Pattern` is the IR `:match` / `:assert` clause lifted into a typed
object that knows its bound variables, its mentioned relations, and
its mentioned types. Used by `Rule.relations` / `Rule.types`.

```python
class Pattern:
    expr:       IRNode                       # the SForm from the IR
    variables:  tuple[Var, ...]              # bound by this pattern
    relations:  tuple[Relation, ...]         # mentioned by name
    types:      tuple[Type, ...]             # mentioned via (instance _ T)
```

Pattern's matching semantics (binding, backtracking) is **deferred to
P1.3**. This task only builds the structural view.

### Task T1.2.1.6 — Equality-class hooks (placeholder)

Reserve a `ClassId` per `Instance` (defaults to its own id) and a
union-find over `Instance`. Do *not* yet fire equality propagation;
when an inference rule asserts `(= ?a ?b)`, call
`kb.classes.union(a, b)`. The engine doesn't yet act on classes;
that's reserved for an optional future phase
([F4](../../followups/f4_cross_cutting.md)).

## What this stage does NOT cover

- **Three layers as views** — S1.2.2. The `Layer` enum is just a
  field on `Fact` here; the filtered views land next.
- **Pattern matcher** — P1.3. `Pattern` here is structural-only.
- **Hypothesis fork** — S1.2.2 + P1.5. The KB needs to support
  cheap forking of the reasoning layer, but the mechanism (which
  facts are shared, which are deep-copied) is decided in S1.2.2.
- **Transactional mutation / rollback** — S1.2.2 + P1.5.
- **Provenance objects** — S1.2.3.
- **DOT rendering** — S1.1.4 (forward) + P1.2 inverse (from_dot).

## Acceptance

For `examples/zebra.ein` (after `kb = KnowledgeBase.from_ir(parse(...))`):

- `len(kb.types)` == 7 (Attribute + 6 leaf types).
- `len(kb.instances)` == 25.
- `len(kb.relations)` == 3 (co-located, right-of, next-to).
- `len(kb.rules)` == 6 (symmetric, transitive, implies, square-fwd,
  square-bwd, type-exclusivity).
- `kb.relations["right-of"].rules` is exactly
  `{square-fwd, square-bwd, implies}` (by name).
- `kb.relations["next-to"].rules` is
  `{symmetric, square-fwd, square-bwd}`. (`implies` mentions
  `next-to` only as its second arg in a rule-application fact, not
  in the rule body — boundary case worth a test.)
- `kb.rules["type-exclusivity"].relations` includes `instance` and
  `co-located`.
- `kb.types["House"].instances` is `{House_1, …, House_5}`.
- `kb.instances["Norwegian"].facts` includes both the implicit
  `(instance Norwegian Nationality)` (ontology) and the explicit
  `(co-located Norwegian House_1 :source "(10)")` (facts).
- `len(kb.facts)` matches IR fact count exactly (no double-counting).
- `pytest tests/kb/test_entities.py` ≥ 30 tests covering the
  cross-references above + edge cases (orphan instance with no facts,
  type with no instances, rule with no applications).

## Design rules-of-thumb (load-bearing for everything in this stage)

These are the user-stated principles the data model encodes. Captured
here so downstream stages (S1.2.2..S1.2.4, P1.3, P1.5) inherit them
without rediscovery.

| principle                                          | manifestation in this stage                                                              |
|----------------------------------------------------|------------------------------------------------------------------------------------------|
| **Types and relations are first-class**            | `Type` and `Relation` are top-level entities with stable identity and back-pointers      |
| **Rules are graph rewrites**                       | `Rule` is an entity; `:match` / `:assert` are `Pattern` objects; rewriting lives in P1.3 |
| **Generic > syntactic**                            | No `?var:T` sugar; type constraints expressed via `(is-a ?var T)` premises (F4 Q35)      |
| **Syntax should be as protective as possible**     | Per-entity validation in T1.2.1.4 — undeclared types/relations rejected (not silently)   |
| **What has types**                                 | Only `Instance.type`. Vars are typed via match premises; relations have a `signature` (tuple of types) but are not themselves "typed" in the instance-of sense |
| **Composable typed-var syntax (`?a:T`) — deferred**| Postponed; pattern language already expresses this (F4 Q35)                              |

## Note on the ontology IR split (Q22)

The IR currently packs four kinds of declaration into one
`(ontology …)` block: type/instance, relation-signature,
rule-application facts (`(symmetric R)`), and structural facts. The
load order in T1.2.1.4 already separates them internally. A
*syntactic* split into three sub-heads (relation-decls,
relation-defs, types-and-objects) is a P1.1 question, not P1.2 —
parked as [M1 Q22](../../open_questions.md#q22--ontology-ir-sub-head-split).
The data model here is robust either way: it consumes already-split
IR or already-mixed IR via the same entity loader.

## Open question recorded here

When a rule mentions a relation only as a *parameter* (`(rule
symmetric (?rel) :match (?rel ?a ?b))` — `?rel` is a Var, not a
Relation name), should that count as `rule.relations`? Currently
NO — `rule.relations` lists only relations mentioned **by name**.
The "applies to via property fact" link is `Rule.applications` →
`Fact.relation`. Documented; revisit if the trace renderer wants a
union view.
