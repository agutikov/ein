# S1.2.2 — Three layers as views

**Phase:** P1.2 (typed hypergraph).
**Estimate:** 2-3 days.
**Depends on:** S1.2.1 data model.
**Implements idea:** [05 §The three implicit layers](../../../docs/ideas/05-zebra-puzzle-graph-reasoner.md#the-three-implicit-layers).

## Context

The PoC tangled three semantically distinct knowledge populations
together. Per `plans/README.md` glossary and the IR's three-block
split (S1.1.1, 2026-05-18), they are now first-class:

| layer       | content                                                       | block name        |
|-------------|---------------------------------------------------------------|-------------------|
| `ONTOLOGY`  | schema (Type, Relation) + implicit facts (instance, rule-app) | `(ontology …)`    |
| `FACT`      | explicit puzzle conditions, `:source "(N)"`                   | `(facts …)`       |
| `REASONING` | derived facts produced by rule firings + hypotheses           | `(reasoning …)`   |

Naming them explicitly buys: (a) the DOT renderer can pick one layer
to show without redrawing the world; (b) the trace planner can say
"of the 23 REASONING facts, derive these 5 in this order"; (c)
hypothesis branching only ever copies the REASONING layer; (d) the
trace shows provenance correctly — `:source` from FACT, `:rule` /
`:using` from REASONING.

## Tasks

### Task T1.2.2.1 — `Layer` enum + fact tagging

`src/ein_bot/kb/layer.py`:

```python
class Layer(IntEnum):
    ONTOLOGY  = 1
    FACT      = 2
    REASONING = 3
```

Every `Fact` (S1.2.1) carries a `layer` field. This task only
formalises the enum and audits that the IR loader sets it correctly
per top-level form:

- `(ontology …)` fact children → `ONTOLOGY`
- `(facts …)` children          → `FACT`
- `(reasoning …)` children       → `REASONING`
- inference rule firings at runtime → `REASONING`

### Task T1.2.2.2 — Layer views over the KB

Add the three filtered accessors:

```python
class KnowledgeBase:
    def ontology(self)  -> FactView: ...
    def facts(self)     -> FactView: ...
    def reasoning(self) -> FactView: ...
    def all_layers(self) -> FactView: ...
```

`FactView` is a read-only filter exposing
`.relation(name) -> Iterator[Fact]`, `.about(instance)`, `.matching(pattern)`,
etc. The pattern matcher (P1.3) restricts to the view's layer when
called from a view.

### Task T1.2.2.3 — Hypothesis branch = reasoning-layer copy

Add `kb.fork() -> KnowledgeBase` that:

- Shares `types`, `instances`, `relations`, `rules`, and the
  ONTOLOGY / FACT slices of `facts` **by reference** (immutable
  post-load).
- Deep-copies only the REASONING slice + the indexes that point
  into it.
- Inherits provenance for shared facts; new derived facts get fresh
  ids.

This is the primitive [P1.5](../p1.5_hypothesis_loop/README.md)
builds on. Its existence in P1.2 (not P1.5) forces the data model to
support it from day one.

### Task T1.2.2.4 — Layer-aware tests

`tests/kb/test_layers.py`. Verify:

- An ONTOLOGY fact never appears in `kb.facts()` view.
- A FACT-layer fact never appears in `kb.reasoning()` view.
- `kb.fork().reasoning()` mutations don't leak back to the parent.
- Cross-reference queries that span layers (e.g.
  `instance.facts` — facts mentioning Norwegian regardless of
  layer) return entries from all layers; layer-scoped queries
  (`kb.facts().about(norwegian)`) return only the FACT layer.

## Acceptance

- All three views return the correct subset.
- `kb.fork()` runtime is O(|reasoning|), not O(|all|).
- `pytest tests/kb/test_layers.py` ≥ 12 tests, green.
