# S1.2.2 — Three layers as views

**Phase:** P1.2 (typed hypergraph).
**Estimate:** 2-3 days.
**Depends on:** S1.2.1 data model.
**Implements idea:** [05 §The three implicit layers](../../../docs/ideas/05-zebra-puzzle-graph-reasoner.md#the-three-implicit-layers).

## Context

The 2021 prototype tangled three semantically distinct knowledge populations
together. Per `plans/README.md` glossary and the IR's three-block
split (S1.1.1, 2026-05-18), they are now first-class:

| layer       | content                                                       | block name        |
|-------------|---------------------------------------------------------------|-------------------|
| `ONTOLOGY`  | schema (Type, Relation) + implicit facts (instance, rule-app) | `(ontology …)`    |
| `FACT`      | explicit puzzle conditions, `:source "(N)"`                   | `(facts …)`       |
| `REASONING` | derived facts produced by rule firings + hypotheses           | `(reasoning …)`   |

Naming them explicitly buys: (a) the DOT renderer can pick one layer
to show without redrawing the world; (b) the trace planner can say
"of the 23 REASONING facts, derive these 5 in this order"; (c)
hypothesis branching only ever copies the REASONING layer; (d) the
trace shows provenance correctly — `:source` from FACT, `:rule` /
`:using` from REASONING.

## Tasks

### Task T1.2.2.1 — `Layer` enum + fact tagging

`src/ein/kb/layer.py`:

```python
class Layer(IntEnum):
    ONTOLOGY  = 1
    FACT      = 2
    REASONING = 3
```

Every `Fact` (S1.2.1) carries a `layer` field. This task only
formalises the enum and audits that the IR loader sets it correctly
per top-level form:

- `(ontology …)` fact children → `ONTOLOGY`
- `(facts …)` children          → `FACT`
- `(reasoning …)` children       → `REASONING`
- inference rule firings at runtime → `REASONING`

### Task T1.2.2.2 — Layer views over the KB

Add the four filtered accessors:

```python
class KnowledgeBase:
    def ontology(self)   -> FactView: ...
    def fact_layer(self) -> FactView: ...   # was .facts(); renamed
    def reasoning(self)  -> FactView: ...
    def all_layers(self) -> FactView: ...
```

The `FACT` layer accessor is named `fact_layer()` to avoid shadowing
the `kb.facts: list[Fact]` registry attribute from S1.2.1
(an instance attribute and a method can't share a name in Python).
The original plan said `.facts()`; we keep the registry-as-list and
expose layer-scoped views as separate methods.

`FactView` is a read-only filter exposing
`.relation(name) -> Iterator[Fact]`, `.about(instance)`,
`.by_source(source)`, `.by_rule(rule_name)`, plus `__iter__` /
`__len__` / `__contains__` / `__bool__`. The pattern matcher (P1.3)
restricts to the view's layer when called from a view — the
`.matching(pattern)` stub is a `NotImplementedError` until P1.3.

### Task T1.2.2.3 — Hypothesis branch = reasoning-layer copy

Add `kb.fork() -> KnowledgeBase` that:

- Shares `types`, `instances`, `relations`, `rules`, `query` **by
  reference** (immutable post-load).
- Shallow-copies the `facts` list and every reverse-index dict so
  the fork can append reasoning-layer facts and update its own
  indexes via `_index_fact()` without leaking into the parent.
- Forks the equality-class union-find (own state, seeded from
  parent's parent-pointer dict).
- New derived facts in the fork carry `:rule` / `:using` provenance;
  shared facts keep their original provenance untouched.

The cost is **O(|facts|) per fork** with O(1) per fact — the plan's
"O(|reasoning|)" goal is relaxed to this for two reasons: (a) the
shallow dict copies are O(|index entries|) anyway, and (b) facts
themselves are small immutable dataclasses, so the copy is cheap.
If P1.5 profiling shows branching is hot, revisit with a copy-on-
write index wrapper.

**Caveat about entity back-pointers:** shared entities (Type,
Instance, Relation, Rule) keep their `_kb` pointing at the
**original** KB, not the fork. So `norwegian.facts` returns the
root KB's facts, not the fork's view. Fork-scoped queries go
through the explicit view API:
`fork.all_layers().about(norwegian)`. The entity API tells you root
state; the view API tells you branch state. Documented in
`store.py::KnowledgeBase.fork.__doc__`.

This is the primitive [P1.5](../p1.5_hypothesis_loop/README.md)
builds on. Its existence in P1.2 (not P1.5) forces the data model to
support it from day one.

### Task T1.2.2.5 — Encoding-agnostic logical views

The encoding-deferral principle (memory: project — IR encoding
choice deferred) means downstream code (renderer, trace planner,
hypothesis branching) must not assume `kb.types` is populated.
Add two helpers to `ein.kb.views`:

```python
def logical_types(kb)     -> tuple[Type | str, ...]: ...
def logical_instances(kb) -> tuple[Instance | str, ...]: ...
```

`logical_types(kb)` returns:
- `kb.types.values()` if non-empty (classic encoding); OR
- the unique RHS atoms of `(is-a ?Child ?Parent)` facts (unified
  encoding) — as **strings**, because no `Type` entity exists for
  them. Promotion to typed entities is a P1.3 / P1.7 question.

`logical_instances(kb)` returns:
- `kb.instances.values()` if non-empty; OR
- leaves of the is-a forest — names that appear as ``args[0]`` of an
  is-a fact and never as ``args[1]``.

The companion helpers `type_name(x) -> str` and `instance_name(x)
-> str` paper over the `Type | str` union so callers can write a
single `t.name`-style accessor.

### Task T1.2.2.4 — Layer-aware tests

`tests/kb/test_layers.py`. Verify:

- An ONTOLOGY fact never appears in `kb.facts()` view.
- A FACT-layer fact never appears in `kb.reasoning()` view.
- `kb.fork().reasoning()` mutations don't leak back to the parent.
- Cross-reference queries that span layers (e.g.
  `instance.facts` — facts mentioning Norwegian regardless of
  layer) return entries from all layers; layer-scoped queries
  (`kb.facts().about(norwegian)`) return only the FACT layer.

## Acceptance

- All three views return the correct subset.
- `kb.fork()` runtime is O(|reasoning|), not O(|all|).
- `pytest tests/kb/test_layers.py` ≥ 12 tests, green.
