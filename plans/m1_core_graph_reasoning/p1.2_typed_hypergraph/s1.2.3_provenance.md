## Context

Without provenance the *contradictions* task class
([idea 03](../../../docs/ideas/03-three-task-classes.md)) returns
"UNSAT" — useless. With provenance it can return a minimal
contradictory subset *and* the chain of derivations leading to
the conflict. Same machinery powers the trace generator
([P1.6](../p1.6_rendering_and_trace/README.md)): every "It follows
that …" sentence resolves to one provenance record.

Granularity is fixed at **per-fact**, per the working answer in
[M1 Q18](../../open_questions.md#q18--provenance-granularity). The
full derivation DAG falls out by transitive closure over premise
facts.

## Tasks

### Task T1.2.3.1 — Provenance record type

`src/ein_bot/kb/provenance.py`:

```python
@dataclass(frozen=True, slots=True)
class Provenance:
    id:         ProvenanceId
    kind:       Literal['source', 'rule', 'hypothesis', 'rejected']
    # 'source':     came from the user's input (Loc reference + sentence id)
    source:     Loc | None         = None
    sentence:   str | None         = None        # "condition (10)"
    # 'rule':       derived by a rule firing
    rule:       str | None         = None        # e.g. 'triangle-composition'
    premises:   tuple[FactId, ...] = ()           # the facts the rule consumed
    bindings:   dict[str, str]     = field(default_factory=dict)  # ?vars → instance names
    # 'hypothesis': speculative branch
    branch:     BranchId | None    = None
```

Each `Fact.provenance` (S1.2.1) points at one of these. ONTOLOGY and
FACT layer facts get a `kind='source'` record on load (carries the
`Loc` and the `:source` sentence). REASONING-layer facts get
`kind='rule'` or `kind='hypothesis'`.

### Task T1.2.3.2 — Derivation DAG queries

`KnowledgeBase.derivation_dag(fact) -> DerivationDAG`:

- BFS from `fact` through `provenance.premises`.
- Terminates at facts whose provenance is `kind='source'`.
- Returns a `DerivationDAG` value: nodes are facts; arcs go from
  premise to conclusion. Hashable, comparable, printable in DOT
  (P1.6 will format it).

`KnowledgeBase.unsat_core(conflicting_facts) -> set[Fact]`:

- Takes a set of mutually contradictory facts (typically returned
  by the constraint checker when verify() fires).
- Returns the minimal set of source-provenance facts that, together,
  derive them. Computed as the union of source-kind terminals of
  each `derivation_dag()`.

### Task T1.2.3.3 — DerivationDAG render

A small helper returning DOT. Used by:

- `ein-bot trace explain <fact>` CLI;
- the markdown trace renderer in P1.6;
- the contradictions output in P1.5.

Render rules: rule-derived facts shown as boxes labelled with rule
name + truncated bindings; source-facts shown as ellipses with the

### Task T1.2.3.4 — Provenance round-trip in IR

The provenance kw-pairs are PARTIALLY grammar-legal:

- ONTOLOGY / FACT facts use `:source "…"` — fully supported,
  round-trips cleanly.
- REASONING facts use `:rule <Name> :using (<fact-ref>+)` — supported
  on the parse side BUT with grammar limitations on `<fact-ref>`:
  - **Atom-id form** (`:using (c10 c15)`) parses but as the wrong
    shape — `c10` becomes the head atom of an SForm and `c15` an
    arg, instead of two refs to facts named `c10` / `c15`.
    Resolving atom-ids to Facts also needs an atom-id → Fact
    registry which doesn't exist yet (would arrive as a `:id <atom>`
    annotation on Facts, naturally landing with the trace renderer
    in P1.6).
  - **Compact `(rel args)` form** (`:using ((co-located A B) (co-located C D))`)
    is what the data model uses internally (Provenance.premises_raw)
    but the current grammar *rejects* it: a kw-pair value must be a
    headed list (head must be a SYMBOL/VAR/WILDCARD/EQ), not a
    headless list of headed lists. Fix is a P1.1 grammar tweak —
    add a `headless_list: "(" generic_list+ ")"` alternative to the
    `?value` production.

**Decision for S1.2.3:** *defer IR round-trip of `:using`*. The
internal `Provenance.from_rule(...)` API works end-to-end; the engine
(P1.3 / P1.5) constructs derived facts with provenance directly,
and `kb.derivation_dag(fact).to_dot()` produces the DAG SVG without
needing an IR text round-trip. When P1.6 / P1.7 needs the round-trip
(e.g., for engine dumps that can be re-loaded), pick one of:

- **A** — Atom-id form + a new `:id <atom>` Fact annotation. The
  loader builds an atom-id → Fact lookup table; the dumper emits
  `:id <auto-name>` on every fact + uses those names in `:using`.
- **B** — Compact form via a grammar tweak. Headless list
  becomes legal as a kw-pair value.

For now the loader silently ignores atom-id `:using` clauses and the
compact form simply doesn't parse. Documented in
`from_ir.py::_ingest_facts` and tested via direct Fact construction
in `tests/kb/test_provenance.py`.

### Task T1.2.3.5 — Tests

`tests/kb/test_provenance.py`:

- Adding a derived fact with premises produces a recoverable DAG.
- Multi-hop derivation: A → B → C; `derivation_dag(C)` includes A.
