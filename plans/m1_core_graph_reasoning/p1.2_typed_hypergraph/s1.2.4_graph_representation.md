# S1.2.4 — Graph representation of the data model

**Phase:** P1.2 (entity data model + three layers + provenance).
**Estimate:** 3-4 days.
**Depends on:** S1.2.1 (entity model + cross-refs), S1.2.2 (layer views),
S1.2.3 (provenance) — needs the KB before it can fuse facts across forms.
**Implements idea:** [02-graph-as-formal-substrate](../../../docs/ideas/02-graph-as-formal-substrate.md),
[05-zebra-puzzle-graph-reasoner](../../../docs/ideas/05-zebra-puzzle-graph-reasoner.md).
**Visual target:** [`docs/PoC/README.md`](../../../docs/PoC/README.md) —
the PoC's SVGs ("clustered with two-direction arcs", "linked with type
nodes", "house color constraint", the triangle/square inference
diagrams) all show types, instances, relations, and inferred edges
*fused into one* graph with cross-references — not stacked tiles.

## Context

[S1.1.4](../p1.1_ir_language/s1.1.4_ir_to_dot.md) ships an IR → DOT
renderer with one `digraph` per top-level form. That works for
diagrams of a *single* form (a rule's match/assert overlay, an
ontology UML view, a facts-only snapshot), and `utils/render_examples.sh`
emits them per-variant.

It cannot produce the **unified** view shown in `docs/PoC/README.md`:

- `Norwegian` (instance in ontology) is the SAME node as `Norwegian`
  in `(co-located Norwegian Red :source "(2)")` (fact) and as
  `Norwegian` in the inferred `(drinks Norwegian Water)` (reasoning).
- The type edge `Norwegian ─is-a→ Nationality`, the fact edges
  `Norwegian ─co-located→ Red`, and the rule-firing edges all live
  on one canvas.
- Relations are visually distinguished (per the PoC: `is`-edges
  black, `next`-edges blue, `right`-edges red, etc.).
- Layer styling: ontology edges plain, fact edges solid coloured,
  reasoning edges dashed / brighter.

The S-expression renderer can't fuse them because the AST has no
*identity* across forms — each digraph is built from its own
sub-tree, and atoms with the same name in different forms are
treated as independent. The KB (S1.2.1) introduces `Instance`
objects with stable identity and cross-references; fusing is then
trivial.

`gvpack` was investigated and rejected — it produces a grid of
independent sub-graphs ("separate facts and ontology side-by-side"
rather than the merged view the PoC shows).

## Tasks

### Task T1.2.4.1 — Design the unified-graph schema

`docs/ir.md` §6.5 (new sub-section "Unified KB graph"). Specifies
the encoding for the PoC-style view:

- **Nodes**:
  - Each `Type` → `box` (one node per type, named by `Type.name`).
  - Each `Instance` → `oval` (named by `Instance.name`).
  - Each non-binary `Fact` → `octagon` (Levi-bipartite hyperedge node).
- **Edges**:
  - `Instance ─instance-of→ Type` (dashed, `arrowhead=empty`).
  - Binary `Fact(relation, a, b)` → `a ─relation→ b` (one direct
    edge; no Levi-bipartite hyperedge node for arity 2).
  - n-ary `Fact` → one octagon node + n labelled edges.
- **Styling by layer** (`Fact.layer`):
  - `ONTOLOGY`  → plain solid.
  - `FACT`      → solid, edge label includes the `:source` short id
    ("(2)" etc.).
  - `REASONING` → dashed, edge label includes the `:rule` name.
- **Colour by relation** — one stable colour per relation name
  (CRC-hash → palette index, mirroring the PoC's
  `hash_color` helper). Per-relation colour is consistent across all
  three layers so the eye groups by relation.
- **Suppressed by default**:
  - Rule definitions (`Rule.match` / `Rule.assert`) — render those
    separately via the existing `render_rule` (S1.1.4).
  - Rule-application facts (`(symmetric co-located)` etc.) — they
    are *meta* facts; expose them as relation-node *attributes* (a
    small badge near the relation's edges) rather than first-class
    nodes.
  - Provenance details (visible in derivation-DAG view, S1.2.3).

Decision artefact: a one-page schema with a worked Zebra fragment in
DOT, before the implementation lands.

### Task T1.2.4.2 — `KnowledgeBase.to_dot()` implementation

`src/ein_bot/kb/render.py`. Signature:

```python
def to_dot(
    kb: KnowledgeBase,
    *,
    layers: Iterable[Layer] = (ONTOLOGY, FACT, REASONING),
    colour_by: Literal["relation", "layer", "none"] = "relation",
    include_types: bool = True,
    include_instances: bool = True,
) -> str: ...
```

Walks the KB's entities (`kb.types`, `kb.instances`, `kb.facts`
filtered by `layers`), emits one `digraph` covering everything. Uses
the cross-reference views from S1.2.1:

- For each `Instance i`, emit one node; emit `i → i.type` edge.
- For each `Fact f` in the requested layers, emit per the schema.
- For each `Relation r`, choose its colour once via
  `_hash_color(r.name)`; reuse across all facts of `r`.

Layer filter lets the same renderer serve:

- `to_dot(kb, layers={ONTOLOGY})` — the type system + instance
  enumeration alone.
- `to_dot(kb, layers={ONTOLOGY, FACT})` — the puzzle as posed,
  no derivations.
- `to_dot(kb)` (default) — everything including derived facts;
  reasoning-layer edges dashed.

### Task T1.2.4.3 — CLI + script wiring

- Extend `ein-bot ir dot` (or add `ein-bot kb dot`) so the unified
  view is reachable from the command line:
  `ein-bot kb dot <file.ein> --layers=ontology,fact > all.dot`.
- Update `utils/render_examples.sh` to call this for each example —
  the per-variant `_all.svg` becomes the *unified* view, not the
  gvpack grid. Layout: `fdp` (force-directed; PoC's images use the
  fdp aesthetic).

### Task T1.2.4.4 — Tests + visual regression

`tests/kb/test_render.py`:

- Shape mapping: each documented shape appears at least once for a
  Zebra-derived KB.
- **No node duplication**: `Norwegian` appears exactly once in the
  unified graph (gvpack failure case).
- Cross-form fusion: an `Instance` referenced by both an ontology
  fact and a facts-block fact has one in-degree from the type plus
  one out-degree from the fact.
- Layer filtering: `layers={ONTOLOGY}` produces strictly fewer
  edges than `layers={ONTOLOGY, FACT}`.
- Colour stability: rendering the same KB twice yields the same
  colour per relation (CRC-deterministic).
- `dot -Tcanon` parse-back round-trip (skip if `dot` not on PATH).

**Visual regression** — store a small golden Zebra fragment's
unified-graph DOT under `tests/golden/kb_zebra_unified.dot` and
compare via `dump`-style diff. Refresh on intentional change.

  not the same as the unified working-memory view here).

- **Markdown trace generation** — P1.6.
- **Per-step snapshots** — P1.6 (trace renderer iterates the unified
  view at each engine step).
