**Phase:** P1.20 (kernel documentation), Theme C.
**Estimate:** 2-3 days, mostly outlining until P1.8 stdlib lands.
**Status:** parked; **unblocked** — the stdlib shipped (P1.8a
S1.8a.f20), so this can be written against the real modules.
**Depends on:** the shipped stdlib at
[`ein.py/src/ein/stdlib/`](../../../ein.py/src/ein/stdlib/) —

**Blocks:** nothing on M1's critical path; gates [M2b](../../m2b_presentation/README.md)

## Context

P1.8 Theme A bundles the stdlib content: closure auto-inference
(S1.8.A6), `imply` family + `converse` + algebra lemmas
(S1.8.A7), general totality / `domain-elimination` library
form (S1.8.A8), reflective rule-implication (S1.8.A9), and
the macro system (S1.5.9). Each is a separately useful piece
of authoring vocabulary; collectively they form ein's stdlib.

This stage writes a reference page (or pages) parallel to
S1.20.B's kernel reference. The structure: one entry per
stdlib rule / macro, with the same format
(name, signature, semantics, example, composition).

### Task T1.20.C.2 — Per-item entry (as items ship)

For each stdlib rule / macro shipped under P1.8:

- **Name + parameters.**
- **Signature** (what the activator looks like; what types it
  binds).
- **Semantics** (what facts the activator triggers; which
  rules fire).
- **Worked example** (1-2 line puzzle snippet using it).
- **Composition** (which other stdlib + kernel atoms it
  interacts with).

### Task T1.20.C.3 — Stdlib quick-reference card

Like the kernel quick-ref card (S1.20.B.3), but for stdlib.
"I want X — which stdlib activator?".

### Task T1.20.C.4 — Importing the stdlib

The import mechanism has shipped (S1.8.A2 / S1.8a.f20): document
the real surface — per-module `(import std.<module> :symbols (…))`
with `:symbols` auto-closure, as `examples/zebra2.ein` uses.
