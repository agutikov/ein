# S1.20.A2 — Idiomatic data-model backfill

**Phase:** P1.20 (kernel documentation), Theme A.
**Estimate:** 3-4 days — actual writing, not just refactoring.
**Status:** parked; sequential after S1.20.A1 split.
**Depends on:** [S1.20.A1](s1.20.a1_ir_doc_split.md).
**Blocks:** the rest of Theme A is structurally complete after
this; Themes B-D can proceed in parallel.

## Context

After S1.20.A1 splits 02-data-model into idiomatic +
python-impl, the idiomatic level often consists of "look at the
Python class" pointers — the idiomatic spec is implicit in the
implementation. This stage writes the missing layer:

- **Collections.** Which collection shape backs each entity
  type? Sets, ordered lists, tuples? What's the invariant?
- **Indexes.** Reverse-index structure: `_facts_by_relation`,
  `_negated_facts`, `_rule_apps_by_rule`, `_facts_by_obj`. What
  do they map, what queries do they support, what's the
  amortised complexity?
- **Algorithm diagrams.** ASCII / mermaid diagrams for: fact
  insertion, rule firing path, hypothesis fork, contradiction
  detection.
- **Mathematical pseudocode.** Where the algorithm is not
  obvious from data shape — e.g. the saturator's main loop:
  worklist + dequeue + match + assert + enqueue dependents.

## Tasks

### Task T1.20.A2.1 — Enumerate the collections

For each data-model entity (Instance, Type, Relation, Rule,
Fact, Pattern, Provenance, Hypothesis, …):

- **a.** What's the collection shape (set / list / dict / …)?
- **b.** Identity rule (`__eq__` semantics).
- **c.** Ordering invariants if any.

### Task T1.20.A2.2 — Document indexes

The KB carries multiple reverse indexes; document each:

- **a.** Name; what it maps (`key → value-set`).
- **b.** Maintenance: insert (always O(1)? amortised?), delete
  (M1 is append-only — no deletes).
- **c.** Query: which engine path consults it?
- **d.** Constraints / invariants.

### Task T1.20.A2.3 — Algorithm diagrams

Write mermaid (or ASCII) diagrams for:

- **a.** Fact insertion: lookup → dedup → index update →
  watcher notification.
- **b.** Rule firing: activator lookup → match → bind → assert
  → recursive insertion.
- **c.** Hypothesis fork: KB.fork → saturate → contradiction
  detect → verdict.
- **d.** Contradiction detection: scan for `(X, (not X))`
  pairs; provenance walk for unsat-core.

### Task T1.20.A2.4 — Mathematical pseudocode

For each non-trivial algorithm:

- **a.** Saturator main loop.
- **b.** Match-bind (pattern → fact, with type-compat).
- **c.** Hypothesis-loop driver (saturate-then-branch).

## Acceptance

- Every file in `02-data-model/` reads as a *language-agnostic*
  spec — a non-Python implementer could re-create the engine
  from these docs + the surface syntax.
- Diagrams render correctly (mermaid blocks if used, ASCII
  otherwise).
- Pseudocode is syntactically free of language-specific
  conventions (no `self.`, no `@dataclass`).

## Open questions

- **Q-S1.20.A2.A — Diagram format.** ASCII (universal, no
  rendering deps) vs mermaid (richer, requires mermaid-aware
  viewer). Pre-existing precedent in `docs/`: mostly ASCII +
  DOT for graph diagrams. Stick with ASCII + DOT for
  consistency.
- **Q-S1.20.A2.B — Where does notation come from?** Some pseudo-
  code resembles SETL / Z / TLA+. Pick one informal convention
  and document it once.
