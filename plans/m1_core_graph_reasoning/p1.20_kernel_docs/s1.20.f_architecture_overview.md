# S1.20.F — Architecture overview

**Phase:** P1.20 (kernel documentation), Theme F.
**Estimate:** 1-2 days.
**Status:** parked; benefits from S1.20.A + D + E having shipped
to populate the diagrams from coherent docs.
**Depends on:** ideally [S1.20.D](s1.20.d_inference_engine_docs.md)
ships (the inference doc gives the engine boxes for the
data-flow diagram); [S1.20.E](s1.20.e_user_vs_dev_split.md)

**Blocks:** [M2b paper](../../m2b_presentation/README.md) —
the architecture page is one of the natural figures for the
write-up.

## Context

The existing [`docs/kernel/README.md`](../../../docs/kernel/README.md)
is a **reading-order** doc — it tells you what to read in what
order. This stage adds a **structural** doc — it tells you
*where to look* in the codebase + docs to make a typical change.

Audience: a contributor unfamiliar with the codebase reads
this page first to know "where does X live?".

## Tasks

### Task T1.20.F.1 — Data-flow diagram

End-to-end: `.ein source → parser → AST → loader → KnowledgeBase
→ saturator + hypgen → trace renderer`. ASCII / DOT diagram
showing the boxes + arrows + which Python module owns each box.

### Task T1.20.F.2 — Module dependency diagram

`ir/` ↔ `kb/` ↔ `inference/` ↔ rendering. Arrows show
"uses". Highlight the kernel boundary (everything in `ir/` +
`kb/` + `inference/`) vs the surface (CLI, demos, tests).

### Task T1.20.F.3 — Milestone boundary diagram

Which modules each milestone adds / depends on:

- **M1**: ir, kb, inference, basic rendering.
- **M2**: nl_to_ir (parser/LLM client), llm-infra, gbnf.
- **M3**: smt_lib backend, hybrid driver.
- **M1b**: gui (separate process, talks to M1 via artefacts).
- **M2b**: paper-specific scripts (probably under `bench/`
  or `papers/`).
- **followups**: out-of-tree until promoted.

### Task T1.20.F.4 — "Where do I look?" cookbook

Common change requests:

- "Add a new rule" → which files?
- "Add a new top-level IR form" → grammar.lark + ast.py +
  loader + tests.
- "Change saturation order" → inference/saturator.py +
  inference/engine.py.
- "Add a new rendering target" → rendering/ + cli/.
- "Add a new CLI subcommand" → cli/.

Per request: bullet list of files to touch + 1-line guide
to each.

### Task T1.20.F.5 — File placement

`docs/kernel/architecture.md` (or `docs/architecture.md` if
broader than kernel). Cross-link from `docs/kernel/README.md`.

## Acceptance

- Three diagrams (data-flow, dependency, milestone boundary)
  rendered + checked in.
- Cookbook covers ≥ 5 common change requests.
- A new contributor can locate where to make a typical change
  by reading only this page + the cookbook entry.

## Open questions

- **Q-S1.20.F.A — Diagram format.** ASCII (universal) vs DOT
  (renderable, but needs the SVG checked in too) vs mermaid
  (richer, requires a renderer). Default: DOT, matching
  the existing `utils/render_knowledge_graph.sh` pattern.
- **Q-S1.20.F.B — Drift maintenance.** Architecture diagrams
  go stale as the code evolves. Add a "last verified for
  commit SHA" footer to each diagram; revisit after each
  milestone ship.
