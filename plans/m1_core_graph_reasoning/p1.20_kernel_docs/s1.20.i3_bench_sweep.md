**Phase:** P1.20 (kernel documentation), Theme I (stage 3 of 4).
**Estimate:** 1.5-2 days, mostly machine time (the sweep).

## Context

`zebra2.ein` solves in ≈2 s under PyPy with the default config (line 54:
`(config :enable-pre-branch-lookahead true)`). So the README's 3600 s/cell
timeout is wildly oversized — keep a generous outer guard only as the
"won't terminate if off" sentinel, and bench perf levers with a tight
budget. Instruments (confirmed in I1): `ein_pypy.sh solve <variant>
--stats [--exhaustive] --max-time <budget>`; recorded `MonotonicStats`
counters: `enterings_total` / `_alive` / `_dead_pre` / `_dead_post`,
`layers_explored`, `forced_positives`, `saturate_count`, `solution_nodes`,
`exhausted` — plus wall-time + RSS.

## Tasks

### Task T1.20.I3.1 — Generate the variant fixtures

One `examples/zebra2-no<FLAG>.ein` (or a generated temp set) per measurable
lever, flipping its value in the existing `(config …)` block:
- the two perf booleans (`enable-pre-branch-lookahead`,
  `enable-lookahead-kill-cache`) → `false`;
- the I2 feature flags (`enable-path-nogoods`, `enable-symmetric-mirror`,
  …) → `false`;
- value-choice sub-rows: `hypgen-scoring most-constrained`,
  `lattice-order score-sum` (with `hypgen-scoring popularity`).
Keep them generated/scriptable, not 12 hand-maintained copies that drift
from `zebra2.ein`.

### Task T1.20.I3.2 — The sweep harness

A runner in [`utils/`](../../../utils/) (e.g. `utils/feature_matrix.py`,
alongside `profile_solve.py` / `symmetric_bench.py`) that, per variant:
runs the solve under the budget, parses `--stats`, records the metrics +
wall-time + RSS, and emits a machine-readable results file (JSON/CSV).
Repeat N times per cell and report median (wall-time is noisy).

### Task T1.20.I3.3 — Budget + mode policy

- **Exhaustive** (`--exhaustive`, `stop_after=None`) for the matrix — a
  disabled pruning lever must show the *full* blow-up, not a lucky early
  stop. (Resolve against Q-S1.20.I.B.)
- Perf budget ≈ 120 s/cell; outer "broken if off" guard ≈ 600 s. A cell
  hitting the outer guard is the **broken-if-off sentinel** (records as ∞).
- **PyPy** for wall-clock (the engine is CPU-bound; ~3-6× CPython).

### Task T1.20.I3.4 — Capture the raw artifact

Commit the results file (machine-stamped: host, interpreter, commit SHA)
so I4's table is reproducible and the provenance is explicit.

## Acceptance

- One reproducible result per measurable lever (default-on vs off), with
  the recorded counters + median wall-time + RSS in a committed artifact.
- Each correctness-load-bearing lever's "off" run is shown to break /
  hit the outer guard; each perf lever's slowdown factor is measured.
- The harness re-runs from a single command (documented in the artifact).

## Open questions

- **Q-S1.20.I.B — Exhaustive vs `stop_after=1`?** Exhaustive is the honest
  measure of a pruning lever (default above); but `stop_after=1` is the
  shipped fast path, so a second `stop_after=1` column may better reflect
  real authoring. **Default: exhaustive primary; optional fast-path column
  if cheap.**
- **Q-S1.20.I.C — Results home + portability.** Numbers are machine-
  specific. **Default: commit the artifact + bake the table into
  [`features.md`](../../../docs/kernel/inference/features.md), stamped with
  host/SHA**, and note they're indicative not absolute.
