## Context

The reader can now author facts; K2 introduces the central idea —
**a rule pattern-matches facts and asserts new facts**, i.e. *rewrites the
graph* — using the **simplest, most intuitive** rules first, on small
fixtures, one at a time. This is the difficulty-graded entry to the rule
material; the family taxonomy comes in [K3](s1.20.k3_rule_families.md).

Pedagogically these are the right openers because they're one-premise or
symmetric and visualise cleanly:

- **`symmetric`** — `(R a b) ⟹ (R b a)`; fixture
  [`saturation/symmetric/`](../../../examples/saturation/symmetric/)
  (couple / friends / neighbours). (A `std.algebra` property rule —
  introduce the import idea here.)
- **`transitive`** — `(R a b) ∧ (R b c) ⟹ (R a c)`; fixture
  [`saturation/transitive/`](../../../examples/saturation/transitive/)
  (taxonomy / mealtimes / colocation-chain).
- **`co-located`** — the zebra2 "two attributes share a house" join; the
  first puzzle-shaped rule, motivating the move to `zebra2`.

## Tasks

### Task T1.20.K2.1 — Chapter `docs/guide/02_first_rules.md`

For each opener rule, the **tri-representation per rule** (from K1) plus a
**graph before / after one firing**:

- **Ein-lang form** — the `(rule …)` (or stdlib import) + a one-paragraph
  plain-English summary of *when it fires* and *what it concludes*.
- **NL framing** — the everyday sentence the rule encodes.
- **Graph before / after** — a small DOT of the relevant KB slice
  immediately before and after the firing (single-instance, not full
  puzzle), in both the **compact** and **canonical Levi** views (the
  reader flips between them — P1.6's compact/Levi framing).

### Task T1.20.K2.2 — The graph-pair tooling

Establish the repeatable recipe the guide uses for every before/after:
- `ein render rule <name> <file>` for the rule's own graph form (compact /
  Levi via the render flags);
- [`render/slice.py`](../../../ein.py/src/ein/render/slice.py)
  (`render_state` / `render_slice`) for the KB-state slice before vs after.
- If no single-firing "before/after" helper exists, K2 adds a thin
  `utils/` wrapper (or documents the two-snapshot recipe) so K3 reuses it.
  (Flag — confirm against P1.6 S1.6.1's renderer before hand-rolling.)

### Task T1.20.K2.3 — "Run it yourself" boxes

Each rule chapter ends with the literal command
(`ein saturate <fixture>` / `ein solve …`) so the reader reproduces the
firing — the by-example contract.
