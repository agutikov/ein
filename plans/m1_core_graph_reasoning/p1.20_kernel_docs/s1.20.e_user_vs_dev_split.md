## Context

Today's `docs/kernel/` mixes "what users of ein-lang need to
know" with "what implementers of the engine need to know".
The split lives roughly at "would a non-Python author care?".
If yes → user docs; if no → developer docs.

After S1.20.A1 splits `02-data-model/` into idiomatic + python-
impl, the boundary becomes literal: idiomatic = user-facing,
python-impl = dev-facing. This stage extends the principle
across the rest of the docs.

## Tasks

### Task T1.20.E.1 — Per-page audience audit

For each existing file under `docs/kernel/`, declare audience:

- `glossary.md` → user (and dev — universal).
- `README.md` → both (reading-order serves both).
- `ir/01-ein-graph/` → user (semantics).
- `ir/02-data-model/` → user (idiomatic).
- `ir/02b-python-impl/` → dev.
- `ir/03-ein-lang/01_grammar.md` → user.
- `ir/03-ein-lang/02_patterns.md` → user.
- `ir/03-ein-lang/06_kernel_api.md` → user (per S1.20.B).
- `inference/` (idiomatic part) → user-curious / dev.
- `inference/` (python-impl part) → dev.

### Task T1.20.E.2 — Cosmetic: audience banner per page

A 1-line audience banner at the top of each page: "Audience:
ein-lang authors" or "Audience: engine contributors". Easy
visual cue for readers.

### Task T1.20.E.3 — Reading-order

Two reading orders, not one:

- **User reading order**: glossary → ein-graph → ein-lang →
  kernel API → stdlib API → inference (idiomatic).
- **Dev reading order**: glossary → ein-graph → data-model
  (idiomatic + python-impl) → ein-lang → inference (full) →
  contributing.

Update `docs/kernel/README.md` to expose both.

### Task T1.20.E.4 — Optional: directory split

Should user vs dev be physically separated (`docs/kernel/user/`
+ `docs/kernel/dev/`)? Probably not — the cross-references are
heavy; keep them in one tree, mark audience per page.

## Acceptance

- Every page in `docs/kernel/` has a declared audience.
- `docs/kernel/README.md` exposes two reading orders.
- A first-time user reads only the user-tagged pages and can
  author a puzzle.
- A first-time contributor reads only the dev-tagged pages
  (plus the user pages for context) and can make a typical
  engine change.
