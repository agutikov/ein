# S1.20.B — Kernel API reference

**Phase:** P1.20 (kernel documentation), Theme B.
**Estimate:** 2-3 days.
**Status:** parked; can run in parallel with Theme A's S1.20.A2.
**Depends on:** the kernel surface stabilises (post-P1.7) and
P1.5a closes any open semantic questions (the NAF re-eval
invariant, in particular).
**Blocks:** [S1.20.C](s1.20.c_stdlib_api_reference.md) (the
stdlib reference depends on the kernel reference for which atoms
are kernel vs stdlib).

## Context

Today the kernel's primitives and semantically-loaded atoms
(`rule`, `relation`, `T`, `not`, `eq`, `neq`, `closed`,
`absent`, `forall`, `open`, `false`, `true`, …) are documented
piecemeal across the three IR sub-trees + the engine source.

This stage ships a dedicated reference page (or pages) that
lists each, its arity / signature, semantic role, and where it may
appear (declarator vs `:match`/`:assert` vs fact). (Flat-form since
P1.7c: there are no `(rules …)` / `(facts …)` block contexts.)

The completeness test: a puzzle author can author new puzzles
without reading source.

## Tasks

### Task T1.20.B.1 — Enumerate the kernel atoms

Sweep `ein.py/src/ein/inference/compile.py` and other engine
modules for atoms that have semantic meaning to the engine
(not just appear as names in puzzles). Tabulate:

- Declarators (closed reserved set, flat-form since P1.7c — the
  former `(ontology …)`/`(facts …)`/`(reasoning …)`/`(rules …)`
  block wrappers are GONE): `relation`, `rule`, `hrule`, `query`,
  `config`, `trace`, `macro`, `import`. A fact is any head outside
  this set; its layer is per-fact (`:layer`, else from provenance).
- Pattern primitives: `not`, `and`, `or`, `absent`, `eq`,
  `neq`, `instance`, `forall` (macro), `open` (macro).
- Property facts (rule activators): `symmetric`, `transitive`,
  `functional`, `total`, `closed`, `bijective`, `converse`,
  `square-fwd` / `square-bwd` / `square-unique` (zebra-scoped),
  `sibling-exclusive`, `type-exclusivity`, …
- Logical primitives: `T` (top type), `false` (direct ⊥), maybe
  `true` (identity unit).

Decide kernel vs stdlib boundary: kernel = always available,
parser-level; stdlib = opt-in via `(import std.…)`. Document the
line.

### Task T1.20.B.2 — Per-atom reference page

One section per atom (a long page or several pages):

- **Name + arity.**
- **Signature** (types of args, return shape).
- **Semantics** (what the engine does when it sees this atom).
- **Where it appears** (declarator? fact? `:match`-only?
  `:assert`-only?).
- **Worked example** (a 1-2 line snippet showing typical use).
- **Composition** (which other atoms it interacts with).
- **Open questions** (any unresolved corners).

### Task T1.20.B.3 — Quick-reference card

A 1-2 page summary table: "I need to express X — which atom?"
Reverse-index from intent to atom. Especially useful for the
property facts (when do you write `(symmetric R)` vs
`(converse R R)` vs `(functional R)`?).

## Acceptance

- Every kernel atom has a documented entry.
- A puzzle author who reads only this page + the ein-lang
  grammar can author a non-trivial puzzle (smoke-test against a
  fresh test author or LLM).
- Quick-reference card fits on a printable page.

## Open questions

- **Q-S1.20.B.A — Where does the boundary go?** Kernel
  property facts (`functional`, `symmetric`, …) are sometimes
  treated as stdlib (rules that activate on them) and sometimes
  as kernel (the kernel's structural constraint language).
  Pick a convention; document it.
- **Q-S1.20.B.B — Auto-generation.** Can the reference be
  *generated* from the source (the rule definitions + property
  declarations)? Probably yes, partly; auto-gen for the
  basic-fact half is feasible, narrative semantics still
  hand-written.
