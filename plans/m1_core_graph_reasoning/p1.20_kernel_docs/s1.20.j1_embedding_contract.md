## Context

Theme J is the **Python-level** embedding surface — distinct from
Theme B (the IR/S-expression kernel API) and Theme D (engine
*internals*). The acceptance bar (README): *a downstream user can
import `ein`, load a `.ein`, run `solve`, and read the verdict
without reading the kernel internals.*

### Re-base audit (2026-06-16) — the live surface vs the 2026-05-27 sketch

The Theme-J prose was written pre-lattice-rewrite and names APIs that
have since moved. Audited against `ein.py/src/ein/`:

| README sketch (2026-05-27) | live surface (2026-06-16) |
|----------------------------|---------------------------|
| "import `ein`" then use it | `ein/__init__.py` exposes **only `__version__`** — no facade. Real imports are per-subpackage (see flow below). |
| `solve(kb, mode=…)` | `solve(root_kb, *, stop_after=None, max_set_size=5, config=None, dumper=None, max_time=None, max_enterings=None, store_lattice=False, on_budget="raise")` in [`inference/monotonic/solver.py:128`](../../../ein.py/src/ein/inference/monotonic/solver.py), re-exported as `ein.inference.monotonic.solve`. **No `mode=`** — the verdict *type* is read from the solution count `k`. (A separate `Mode` enum + `is_solved(kb, mode)` live in `verdict.py` for goal-checking, not for `solve`.) |
| "construct a `KnowledgeBase`, load IR" | `KnowledgeBase.from_ir(forms)` / `.from_file(path)` ([`kb/store.py:197,209`](../../../ein.py/src/ein/kb/store.py)) or `ein.kb.load(forms, *, base_dir=None)` ([`kb/from_ir.py:54`](../../../ein.py/src/ein/kb/from_ir.py)). Direct `__init__` is rarely used. |
| "call `Saturator(kb).saturate()`" | correct — `Saturator(kb, engine=None)` then `.saturate(*, max_steps=None) -> Iterator[Firing]` ([`inference/saturator.py:125,218`](../../../ein.py/src/ein/inference/saturator.py)). `inference/__init__.py` re-exports **only `predicates`** (its module docstring still names a top-level `solver` module — stale). |
| "iterate `Firing`s, read provenance" | `Firing(rule, activator, bindings, derived, premises, redundant)` ([`inference/firing.py:37`](../../../ein.py/src/ein/inference/firing.py)); per-`Fact` `Provenance` + `DerivationDAG` ([`kb/provenance.py:59`](../../../ein.py/src/ein/kb/provenance.py)). |

The canonical end-to-end flow as the live code actually spells it
(verbatim from [`utils/profile_solve.py:46`](../../../utils/profile_solve.py)):

```python
from ein.ir import parse
from ein.kb.store import KnowledgeBase
from ein.inference.monotonic.solver import solve

kb = KnowledgeBase.from_ir(parse(text))
verdict, stats = solve(kb, stop_after=1, max_set_size=5)
```

## Open questions

- **Q-S1.20.J.A — Facade or subpackage imports?** Today `import ein`
  yields only `__version__`; the contract is spread across
  `ein.ir` / `ein.kb` / `ein.inference.*` / `ein.trace`. Options: (a)
  **document as-is** (P1.20 is doc-only — zero code risk; the import
  lines are slightly verbose); (b) add a thin `ein/__init__.py`
  re-export (`parse`, `KnowledgeBase`, `load`, `Saturator`, `solve`,
  `SolverConfig`, the verdict types) so the docs can teach
  `import ein; ein.solve(...)`. (b) is a tiny, backward-compatible
  ergonomics change but still a *code* touch — out of the strict
  doc-only charter. **Default: (a)**, with (b) noted as a one-line
  follow-up. Also fold in correcting the stale `inference/__init__.py`
  docstring (it names a top-level `solver` module that is now
  `monotonic/solver.py`) — borderline doc/code; defer to J3's
  reconcile note.
- **Q-S1.20.J.B — Location.** `docs/api/` (top-level, per README) vs
  `docs/kernel/api/` (with the rest of the dev-facing docs). README
  says `docs/api/`; the counter-argument is that everything else
  dev-facing lives under `docs/kernel/`. **Default: `docs/api/`**;
  revisit if Theme E's audience model argues for co-location.
- **Q-S1.20.J.C — Rust port.** If [M1a](../../m1a_rust/README.md)
  ships, the embedding contract becomes ein.rs's (PyO3 + native Rust)
  and this Python page is the legacy reference. Keep the page
