# S1.10.D — Inference engine documentation

**Phase:** P1.10 (kernel documentation), Theme D.
**Estimate:** 4-5 days — substantial new writing.
**Status:** parked; the longest single stage in P1.10.
**Depends on:** [S1.10.A1](s1.10.a1_ir_doc_split.md) ships so
the data-model split is the template; [P1.5a](../p1.5a_zebra_solution/README.md)
ships so the NAF semantic invariant is settled.
**Blocks:** [M2b](../../m2b_presentation/README.md) paper —
the inference doc is what readers reach for "how does the
engine work?".

## Context

Today's [`docs/kernel/inference/README.md`](../../../docs/kernel/inference/README.md)
is a stub. This stage fills it with two parallel layers matching
the data-model split (S1.10.A1):

- **Idiomatic** — collections (the rule-cache, the activator
  index, the firing queue), indexes (`_facts_by_relation`,
  `_negated_facts`, `_rule_apps_by_rule`), algorithm diagrams
  + mathematical pseudocode for `compile_all` / `compile_for` /
  `_apply` / `saturate` / hypothesis loop.
- **Python implementation** — files, modules (`engine.py`,
  `saturator.py`, `hypgen.py`, `compile.py`, `match.py`,
  `solver.py`), data types, functions, classes.

Composes with P1.5a's NAF re-eval invariant — once the
fire-time-vs-enqueue-time semantics is decided, that invariant
needs documenting here.

## Tasks

### Task T1.10.D.1 — Inference idiomatic level

- **a.** Collections: rule cache, activator index, firing
  queue, KB watcher set. Per-collection: shape, invariants,
  maintenance points.
- **b.** Indexes: per-index complexity, what queries it
  serves.
- **c.** Algorithm diagrams (mermaid / ASCII):
  - Engine.compile_all / compile_for.
  - Saturator main loop (worklist → match → assert → enqueue).
  - HypGen pipeline (raw → filter chain → emit).
  - Solver driver (saturate → branch → recurse → verdict).
- **d.** Pseudocode for non-trivial algorithms (match-bind,
  hypothesis loop, contradiction walk).

### Task T1.10.D.2 — Inference python-impl level

Mirror the idiomatic level with concrete code-level docs:

- **a.** File-by-file: `engine.py`, `saturator.py`, `hypgen.py`,
  `compile.py`, `match.py`, `solver.py`. One-paragraph
  intro per file + key classes.
- **b.** Data-types reference: every dataclass in inference/
  with its fields + invariants.
- **c.** Function reference for top-level APIs (`solve`,
  `saturate`, `compile_all`, `compile_for`).

### Task T1.10.D.3 — Cross-cutting invariants

- **a.** NAF semantic invariant (per S1.5a.1's resolution).
- **b.** Append-only KB invariant (no deletes, no in-place
  writes).
- **c.** Hypothesis-loop completeness (no false alive
  verdicts under M1's rules).
- **d.** Saturation termination conditions.

### Task T1.10.D.4 — Worked example

A complete walkthrough: zebra2 → load → compile → saturate →
hypothesis fork → solve. With diagrams + code references.

## Acceptance

- `docs/kernel/inference/` has 4-6 doc files (was a single
  stub).
- Idiomatic level reads as language-agnostic; python-impl reads
  as concrete file/class walkthrough.
- Cross-cutting invariants documented.
- A reader who hasn't seen the source can answer "what
  happens when rule R fires?" by walking the docs.

## Open questions

- **Q-S1.10.D.A — Auto-gen.** Should the python-impl level be
  auto-generated (Sphinx, mkdocstrings, etc.)? M1 doesn't have
  this infra; setting it up is a separate sub-task.
- **Q-S1.10.D.B — Versioning.** Engine internals evolve more
  than the kernel; the docs may go stale. Mitigate with a
  per-file "last verified for commit SHA" marker.
