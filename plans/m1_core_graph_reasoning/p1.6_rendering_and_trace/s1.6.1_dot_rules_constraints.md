# S1.6.1 — DOT for rules + constraints

**Phase:** P1.6 (rendering + trace).
**Estimate:** 2 days.
**Depends on:** P1.1 IR, P1.3 rule registry, P1.4 constraints.

## Context

The renderers in this stage produce DOT diagrams of *static*
artefacts — the rule registry and the constraint set — so the
trace can refer back to a canonical picture of the inference
calculus, not redraw it per step.

Output is inline `dot` — no SVG (see
[README §Output format](README.md#output-format--inline-dot-no-svg)).
The rule/constraint diagrams render as fenced `dot` blocks in a
rules-reference appendix of the trace (or a standalone `rules.md`):

- one block showing the rule families,
- a per-rule pattern → conclusion block,
- one block of declared constraints + scopes.

The same rules also surface as firing nodes inside the per-hypothesis
derivation slices (S1.6.2).

## Tasks

### Task T1.6.1.1 — Rule pattern → DOT

`src/ein/render/rules.py`. For each rule, render:

- **Top**: pattern (LHS) — premise edges as solid arrows.
- **Bottom**: conclusion (RHS) — derived edge as a dashed arrow.
- **Labels**: variables `?a`, `?b` carry through; relation names
  on edges.
- **Constraints** (`:where` clauses): annotation block.

Single SVG per rule + a *combined* SVG with all ten arranged in a
3×4 grid using `graphviz fdp -Goverlap=prism`.

**Attention to folded `(absent (and …))` patterns.** The
`forall` desugar produces nested guards like
``(absent (and (?R1 ?a ?b) (?R2 ?x ?y)))`` (per the S1.5.8c.3a/b
sugar that compiles to existing `AbsentGuard` machinery —
[NAF semantics in inference README](../../../docs/kernel/inference/README.md#naf-semantics--fire-time-re-evaluation-s15a1)).
A naive renderer flattens the inner conjunction into one
overlay arrow per atom and loses the "universally quantified
over the binder" reading. Render the nested guard as its own
sub-cluster (`subgraph cluster_absent`), with the inner
conjuncts each as arrows inside the cluster, and the whole
cluster annotated with `∀ ?binder. range → body` semantics so
the reader can recover the universal from the picture.

### Task T1.6.1.2 — Constraint scopes → DOT

`src/ein/render/constraints.py`. Diagram the constraint scopes:

- **Nodes**: the types/relations the constraint quantifies over.
- **Edges**: `:scope` relationships.
- **Annotations**: the predicate (`at-most-one`, `all-different`,
  `exactly-n`, …).

One `dot` block showing all constraints; useful for sanity-checking
that the engine has the constraints the puzzle requires.

### Task T1.6.1.3 — CLI plumbing

Extend `Ein`:

```sh
ein render rules examples/rules.ein            # emits a dot block to stdout
ein render constraints examples/zebra.ein      # emits a dot block to stdout
ein render rule examples/rules.ein --name=triangle-composition
```

### Task T1.6.1.4 — Tests

`tests/render/test_rules_dot.py`:

- Every rule in `examples/rules.ein` emits a `dot` block that parses
  (`dot -n` / `pydot` exits 0).
- The combined rules block references all ten rule names.
- Constraint diagram references all constraint names.

- The CLI emits an inline `dot` block that, rendered through a
