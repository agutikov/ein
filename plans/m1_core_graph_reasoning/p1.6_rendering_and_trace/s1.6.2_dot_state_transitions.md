## Context

The trace doesn't want the whole KB at each step — that's unreadable on
a 25-cell zebra grid. It wants, per hypothesis, **just what that
hypothesis touched**: the hypothesis fact(s), the KB facts they
combined with, the rules that fired, and the facts derived — a
*provenance cone*. That cone is the diagram embedded in each trace
section (S1.6.4).

Three layers (ontology / fact / reasoning) drive colouring: ontology
grey, fact black, reasoning edges hash-coloured by rule (`hash_color`);
hypothesis facts red, derived facts bold.

## Tasks

### Task T1.6.2.1 — Per-hypothesis derivation slice (primary)

`src/ein_bot/render/slice.py::render_slice(commitment, firings, kb) -> str`
emits an inline `dot` block (no SVG):

- **Seed:** the hypothesis fact(s) of the `commitment` (red).
- **Cone:** walk `firings` (the surviving-path firings for this
  commitment); for each, add its `premises` (the KB facts it consumed)
  and its `derived` fact (bold), with a firing node labelled by `rule`
  + the rendered NL `:why` (the `:why` plumbing gap is S1.6.4 T1.6.4.4).
  Include only facts reachable in this cone — **not the whole KB**.
- **Surviving-path filter:** every firing on the alive/solution path is
  shown, *including* exclusion / elimination derivations (a
  `domain-elimination` firing shows the eliminated alternatives as
  greyed premises). Refuted-branch firings are out of scope here — they
  belong to the reductio slice (T1.6.2.1b).
- **Rules as nodes.** Each firing renders as a rule-node between its
  premises and its derived fact. A rule that derives several facts
  (e.g. forward chaining) currently shows as **N firing nodes sharing
  one rule label** — one `Firing` per derived `Fact` (the model is
  one-fact-per-firing; grouped multi-fact assertion is a P1.8 ein-lang
  question — see [P1.8](../p1.8_ein_lang_modules/README.md)).

### Task T1.6.2.1b — Refuted-branch slice (folded)

Same renderer fed a `DeadCommitment`'s firing chain (see
[README §Net-new prerequisites](README.md#net-new-prerequisites-surfaced-by-the-p15b-audit),
gap 2). Terminates in a ⊥ node tied to the `unsat_core`; the
`learned_clause` labels the no-good. Rendered only inside an expanded
reductio `<details>` (S1.6.4 T1.6.4.6).

### Task T1.6.2.2 — Whole-KB snapshot (kept, flag-gated)

`render_state(kb, *, layer_filter=None) -> str` — the complete graph at
a moment, emitted as inline `dot`. **Off by default**; enabled per-step
via `--full-kb-snapshots`. Nodes coloured by type; edges by layer +
rule; provenance via `tooltip=`. `layer_filter` selects subsets (e.g.
"fact layer only").

### Task T1.6.2.3 — Transition highlight

A `since=<kb_before>` option on both renderers thickens edges absent in
the prior state — "this firing added E". Makes a step's effect visible
without a side-by-side full graph.

### Task T1.6.2.4 — Final solution grid

`render_solution(kb) -> str` — one inline `dot` of the solved state
(the answer grid), for the trace's closing section.

### Task T1.6.2.5 — Tests

`tests/render/test_slice_dot.py`:

- A 3-firing cone renders; only cone facts appear (whole-KB facts
  excluded).
- An elimination firing shows its greyed alternatives.
- `render_state` (`--full-kb-snapshots`) emits the complete graph.
- All renderers emit syntactically valid `dot` (parse with `pydot` /
  `gvpr -n`).
