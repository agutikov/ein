# S1.6.3 — Search tree DOT

**Phase:** P1.6 (rendering + trace).
**Estimate:** 2 days.
**Depends on:** P1.5 SearchTree.

## Context

The search tree from P1.5 S1.5.2 is a tree of `SearchNode`s.
Render it as a tree-shaped DOT diagram:

- root = initial state;
- internal nodes = hypotheses (`(= ?a v)`);
- leaves = Solution / Contradiction / Open;
- pruned subtrees clipped or shown greyed-out.

This is the "path through the search tree" the user named as a
required output.

## Tasks

### Task T1.6.3.1 — Tree DOT renderer

`src/ein_bot/render/search_tree.py::render_tree(tree)`:

- `rankdir=TB` (top to bottom).
- Each `SearchNode` is a record-shaped node with the hypothesis
  text + verdict colour.
- The *successful* path (root → solution) drawn thicker.
- Symmetry-broken branches shown with a "WLOG" annotation
  (S1.5.3).

### Task T1.6.3.2 — Compact + full views

Two outputs:

- `tree-full.svg` — every explored node, including contradictions.
- `tree-path.svg` — only the survivor path + immediate siblings
  pruned-by-contradiction (the "interesting" tree the trace
  references).

### Task T1.6.3.3 — Tests

`tests/render/test_search_tree_dot.py`:

- A 3-node tree (root + 2 children) renders.
- The thicker path is the survivor leaf.
- Contradiction subtrees show their unsat-core in the tooltip.

## Acceptance

- `pytest tests/render/test_search_tree_dot.py` green.
- The Zebra solve produces a path-view tree of < 10 visible
  nodes (it should be near-linear; the puzzle is mostly
  deterministic after propagation).
