# S1.5a.14 — Transitive back-prop (bubble truths to all ancestors)

**Phase:** P1.5a (zebra solution + saturator/NAF gaps).
**Estimate:** ~1 day for the negative-bubble + cache
invalidation; Phase 2 (forced-positive abort + outer solve loop)
not yet started.
**Status:** Phase 1 **shipped** 2026-05-25 in commit
[`17bd821`](https://github.com/...). Phase 2 (outer solve loop on
forced-positive abort) deferred. Phase 3 (per-level back-prop
beyond the fully-unconditional case) folded into

**Depends on:** S1.5a.11 dumper infrastructure (the
`_dumper_ctx`, `_current_parent_ctx`, `_current_depth_ctx`

**Blocks:** S1.5a.13 acceptance (zebra2 cold-solve depends on

S1.5a.15 (dead-caching builds on the same chain
infrastructure).

## Context

Pre-fix `back_propagate(kb, h, unsat_core)` wrote `(not h)` only
into the immediate consume's kb. For an unconditional death at
depth N, this writes to the depth-(N-1) parent. Ancestors at
shallower depths got nothing — even though the death's
unconditionality says `(not h)` is provably true at every
ancestor.

Consequence: deep puzzles face combinatorial blowup. The user
laid it out: depth-6 with 60 alive hyps per level = 60^6 = 4.6B
potential branchings if nothing prunes the ancestor levels.
Truth mined at depth must reach root for the search to stay
tractable.

## What shipped (Phase 1)

### `_kb_chain_ctx` — ancestor chain via ContextVar

New ContextVar in `back_prop.py`:

```python
_kb_chain_ctx: ContextVar[tuple[KnowledgeBase, ...]] = ContextVar(
    "ein_bot_kb_chain", default=(),
)
```

`_explore` (in `solver.py`) pushes its `kb` onto the chain on
entry, pops on exit (try/finally). Chain is root-first;
`chain[-1]` is always the current `_explore`'s kb (≡ the kb the
current `_consume` is operating on).

### `back_propagate` writes to all ancestors

```python
chain = _kb_chain_ctx.get()
ancestors = tuple(ak for ak in chain if ak is not kb)
for anc_kb in ancestors:
    _write_negation(anc_kb, hypothesis, unsat_core, rule_name + "-bubbled")
```

The symmetric mirror (`promote_symmetric=True`) gets the same
treatment. Provenance carries the original unsat-core's
premises_raw; ancestor-kb walks treat missing premises as
dead-ends (safe — `_walk` returns False on unresolved chains).

### Verdict-cache invalidation

`_consume`'s per-call `verdict_at` caches alive / cond-dead
verdicts. If a descendant's back-prop adds `(not h)` to an
ancestor's kb while the ancestor's consume is mid-flight,
ancestor's `verdict_at` may still mark h as alive — leading to
fruitless recursion into a now-dead branch. Second ContextVar:

```python
_consume_caches_ctx: ContextVar[tuple[dict, ...]] = ContextVar(
    "ein_bot_consume_caches", default=(),
)
```

Each `_consume` pushes its `verdict_at` dict onto the chain on
entry. `back_propagate` (after writing the bubbled negatives)
calls `_clear_ancestor_verdict_caches()` which `.clear()`s every
ancestor's dict — current sweep's `verdict_at` untouched, the
ancestor will re-classify on its next sweep.

### Forced-positive root mirror

`_consume`'s re-saturation can produce positive firings
(`Firing.derives_positive()`). When such a firing's derived
fact has `not reaches_hypothesis(kb, derived)` — its chain
doesn't depend on any hypothesis — it's unconditional and
mirrors to root only (NOT to every ancestor — see Phase 2
note below). New helper `_mirror_forced_positive` does the
write, idempotent against pre-existing facts.

## Trade-off accepted: state_hash dedup misses

Bubbling `(not h)` into an ancestor's kb mutates that ancestor's
fact set, so the ancestor's `state_hash` registered with
`builder.state_index` is now stale. Future explorations
computing `state_hash` on equivalent NEW states won't match the
stale entry → fresh node allocation instead of dedup. The
misfires are false **negatives** (tree gets bigger), not false
positives (no incorrect dedup). Accepted as the cost of correct
ancestor pruning. [S1.5a.15](s1.5a.15_dead_caching_unsat_core.md)
addresses the dedup loss with a complementary unsat-core-keyed
cache.

## What's deferred (Phase 2)

The user's design note: "If [the re-sat derives a forced
positive] you anyway will drop the branch to re-enter it on
next loop." The current ship writes the forced positive to root
but continues current exploration. The fuller behaviour is an
outer solve loop that:

1. Catches a "forced positive detected" signal from `_consume`.
2. Aborts the current branch (and all enclosing recursion via
   an exception or flag).
3. Restarts `solve()` from root with the augmented root kb.
4. Loops until no new truths are mined during a pass.

This converges by the dead-monotonicity argument (each pass
either mines new root truths and shrinks root.alive, or
quiesces). Implementation needs:
- A `ForcedPositiveAbort` exception or equivalent.
- Solve outer loop that catches it and re-runs.
- Care that the re-loop preserves the search tree so far (or
  doesn't — depending on the chosen semantics).

Spec'd at high level; see also S1.5a.15 Phase 3 (per-level
back-prop) which interacts with this.

## Verification

`examples/zebra2-hints.ein` with 7 hints active (Wikipedia
solution):

- Pre-fix: Solution in ~9s, 8 tree nodes.
- Post-fix: Solution in ~13s, 17 tree nodes.

Tree node count went up because of the dedup misses described
above. Solve time went up modestly — the additional
ancestor-clearing work pays off in larger puzzles where ancestor
pruning matters more. Full `examples/zebra2.ein` (no hints)
remains beyond the budget — S1.5a.7 (hypothesis scoring) and
S1.5a.15 (dead caching) are the next levers.

`pytest`: 634 tests passing throughout (same regression suite
as pre-fix).
