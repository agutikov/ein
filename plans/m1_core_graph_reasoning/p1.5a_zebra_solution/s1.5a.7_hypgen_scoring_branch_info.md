## Context

The post-S1.5.4 optimisations made "minimal-domain-first"
ordering effectively obsolete. The user's 2026-05-24 framing
proposes a successor: score each hypothesis by a weighted sum
of fact-counts at the relation and object level, plus a
second-level extension that sums one-hop neighbour counts.
Composes with a "prefer information-producing branches"
ordering rule.

The guiding question: how does human reasoning decide to ask
"what is the color of the first house?" first in Zebra rather
than "what drink is drunk by the cat owner?" — the engine's
heuristic should approximate the same intuition.

## Ideas (reproduced + sharpened from the parent README)

+1 idea - re-calculate hyps scores in the branch, so scores are branch-specific

### Idea 1 — Hypothesis scoring by fact-popularity sum

For a candidate `(?R ?A ?B)` (one relation + two objects):

```text
score(R)  = number of facts mentioning relation R
score(A)  = number of facts mentioning object A
score(B)  = number of facts mentioning object B

1st-level score = rel_weight * score(R)
                + obj_weight * (score(A) + score(B))
```

Weights `rel_weight` and `obj_weight` are configurable knobs;
tune empirically. A second round extends the score by summing
facts at *one hop* — the objects that the candidate's
components co-occur with in existing facts.

Worked example from the TODO note:

```text
(co-located Red House1)    ;; fact 1
(co-located House1 Milk)   ;; fact 2

score(House1) = 2, score(co-located) = 2, score(Red) = 1
score(Milk) = 1, score(Cat) = 0

hypothesis (co-located House1 Cat):
  1st-level = score(co-located) + score(House1) + score(Cat)
            = 2 + 2 + 0
  2nd-level adds score(Red) + score(Milk)
```

Intuition: hypotheses involving heavily-referenced
relations/objects are likelier to be *interesting* — they
touch more of the existing fact graph and so are more likely
to either fire useful rules or be quickly contradicted.

### Idea 2 — Branch-info ordering

Prefer branches that *produce more information*:

- If a branch's saturation produces **no new unconditional
  constraints**, defer going deeper inside it — explore shallow
  alternatives first.
- Before going deeper anywhere, eliminate everything that's
  shallow.

Composes with [S1.5.7b stable-alive caching](../p1.5_hypothesis_loop/s1.5.7b_consume_loop_stable_alive.md),
where a branch that adds nothing on re-saturation is already
flagged.

### Idea 3 — Guiding question

> How does human reasoning decide to ask first "what is the
> color of the first house?" in Zebra? Why not e.g. "what
> drink is drunk by the cat owner?"

Two observable patterns from the idea-08 walkthrough:

1. **All human questions for hypothesis are about houses.**
   Houses are the *positional carrier* — its 1..5 ordering
   grounds every other relation.
2. **Why colour, why house 1?** Conjecture: smallest-domain
   attribute crossed with the strongest positional constraint
   in the puzzle's conditions.

Both are post-hoc rationalisations; useful as **score
heuristics**, not proof arguments. Idea 1 (scoring) is the
engine analog of #1; #2 informs the relation-priority knob.

  (S1.5.4 T1.5.4.4): one of `:most-constrained` (current

  `:obj-weight` (default 1.0); used by popularity mode.
- **c.** Wire the config into `inference/hypgen.py`'s ordering
  step.

### Task T1.5a.7.2 — Implement popularity scoring

- **a.** Per-candidate score: `rel_weight * count(R) +
  obj_weight * (count(A) + count(B))`. Use existing fact
  indexes (no new index work).
- **b.** 2nd-level extension: per-candidate, sum fact-counts
  of objects in co-occurring facts. Cap the depth at 1 hop;
  caching needed if depth grows.
- **c.** Sort the raw alive set descending by score before
  emit.

### Task T1.5a.7.3 — Implement branch-info ordering

- **a.** After saturating a branch, check whether the branch
  produced any new positive facts (vs only negatives, vs
  nothing). If only nothing or only `(not …)` cascade, mark
  the branch *low-info*.
- **b.** Sort the alive set so *high-info* branches are
  explored first.
- **c.** Composes with T1.5.7b stable-alive caching — a
  stable-alive branch with no new positives is exactly
  *low-info*.

- **b.** Tabulate: tree-node count, wall time, alive-set size,

- **c.** Compare to the post-S1.5.4 baseline (most-constrained

## Open questions

- **Q-S1.5a.7.A — Tunable weights.** Are `rel_weight` and
  `obj_weight` learnable from a held-out puzzle suite? Probably
  yes (M2 NL → IR output could drive this), but out of M1
  scope. Cross-link to F7 rule induction §C.
- **Q-S1.5a.7.B — Composability with closure activators.** Does
  popularity scoring interact correctly with `(closed R)`?
  Closed relations contribute zero hypotheses; their
  `score(R)` doesn't matter. Confirm.
- **Q-S1.5a.7.C — When to update the score.** Re-compute per
  branch (every fork updates fact-counts) or amortise (compute
  at b0 only)? Composes with [T1.5.4.5/8 alive-inherit](../p1.5_hypothesis_loop/s1.5.4_hypgen_improvements.md).
  The cheapest first cut is amortised; if popularity-scoring
  helps, revisit.
