# S1.5a.5 — Relation-name refactor: `house-*` → `*-location`

**Phase:** P1.5a (zebra solution + saturator/NAF gaps).
**Estimate:** ½-1 day — pure refactor, no semantic change.
**Status:** parked behind [S1.5a.4](s1.5a.4_acceptance_zebra2_solves.md)
— do the rename only after zebra2 solves cleanly so the
refactor is verified-by-regression rather than tangled with
the harder semantic work.
**Depends on:** zebra2 solves uniquely (S1.5a.4 ✅); rename then
is a search-and-replace + re-run.
**Blocks:** nothing structural; informs M2 NL → IR's naming
conventions if landed before M2 starts.

## Context

Current zebra2.ein relation declarations:

```lisp
(relation house-color  House Color)
(relation house-nation House Nationality)
(relation house-drink  House Drink)
(relation house-smoke  House Cigarette)
(relation house-pet    House Pet)
```

The names read as "the *color* property of a house". User's
2026-05-24 framing: rename to `*-location` so the relations read
as "the *location* of the color", "the *location* of the
nation", etc.:

```lisp
(relation color-location  Color       House)
(relation nation-location Nationality House)
(relation drink-location  Drink       House)
(relation smoke-location  Cigarette   House)
(relation pet-location    Pet         House)
```

Note: this **swaps the argument order** as a corollary — the
attribute (Color, Nationality, …) comes first because the
relation is "position of *attribute*". The `(linked …)`
activator facts read more naturally too:

```lisp
;; before
(linked house-nation Englishman house-color Red ...)
;; after
(linked nation-location Englishman color-location Red ...)
```

— still phrasing "the location of the Englishman = the
location of the Red house", but with the attribute as the
slot-1 anchor.

## Open questions

- **Q-S1.5a.5.A — Argument order swap.** Should the rename
  also swap arg order so attribute is slot 0 and House is slot
  1? (`(color-location Red House_1)` reads as "the location
  of Red is House_1".) Or keep `(color-location House_1 Red)`
  for backward compat with the bijective shorthand? The user's
  TODO note implies the swap, but it's a bigger change. Pick
  at T1.5a.5.1 start.
- **Q-S1.5a.5.B — Generalisation to non-zebra puzzles.** Does
  `*-location` make sense outside positional puzzles? Probably
  not — for puzzles without a positional carrier, the
  attribute-as-functional-property naming (`house-color`) is
  more natural. Decision: this rename is **zebra-specific**;
  document it as a stylistic choice for positional puzzles,
  not a kernel convention.
