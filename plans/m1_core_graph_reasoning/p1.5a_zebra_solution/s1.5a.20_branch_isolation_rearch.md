## Vocabulary — logical vs search branch

A **logical branch** is a hypothesis path — the chain of hypotheses
``(h_1, h_2, …, h_d)`` from root down to the current frame. Same
chain, same logical branch, regardless of context.

A **search branch** is one *entering* of a logical branch — a
concrete `try_branch` call with a specific parent KB state. A
single logical branch can be entered multiple times as the parent
KB accumulates new facts (S1.5a.17 outer passes; or, in the
recursive `integrate` model, any time an ancestor's KB tightens
between sibling sweeps and the same hypothesis re-enters the
candidate set).

```
logical branch (h_1, h_2, h_3)
   ├── search branch  pass#1 entering   ← try_branch on parent_kb @ t_0
   ├── search branch  pass#2 entering   ← try_branch on parent_kb @ t_1 (richer)
   └── search branch  pass#3 entering   ← try_branch on parent_kb @ t_2
```

A search branch is entered **exactly once**; a logical branch is
entered **at least once** and may be re-entered with a richer
parent context.

### What `b{i}` is today

`b{i}` is **not** a clean isomorphism with either notion. Reading
`_TreeBuilder.alloc()`
([`solver.py:471-474`](../../../ein.py/src/ein/inference/solver.py)),
the state-hash dedup at `_explore_inner`
([`solver.py:660-663`](../../../ein.py/src/ein/inference/solver.py)),
and the eager-loop pass logic
([`solver.py:1296-1297, 1329`](../../../ein.py/src/ein/inference/solver.py)):

- `builder.alloc()` is a monotonic counter, fresh id per call.
- **State-hash dedup** keys on the **post-saturation kb state**
  of the child: if a previous `b{j}` produced the same state-hash,
  the new entering returns `b{j}` instead of allocating `b{i}`
  (DAG link).
- **Eager mode** (S1.5a.17) starts every pass with a fresh
  `_TreeBuilder`; state-hash dedup is intentionally **not**
  carried across passes.

So `b{i}` is keyed on
``(eager_pass, post_saturation_kb_state)`` — neither logical
branch nor pure search branch:

- Two distinct logical branches that saturate to the same kb
  state → share one `b{i}` (DAG-style state dedup, not
  hypothesis-path dedup).
- Same logical branch entered twice within a pass with a richer
  parent context → distinct post-sat states → distinct `b{i}`
  (the search-branch case the engine handles correctly).
- Same logical branch across eager-mode passes → always distinct
  `b{i}` (fresh `_TreeBuilder`).

### Target vocabulary in the rewrite — three distinct ids

State-hash dedup is genuine work-skipping (it avoids
re-exploring an already-known subtree) and we **keep it**. The
rewrite separates the three notions today's `b{i}` conflates:

| id | scope | when allocated | what it counts |
|---|---|---|---|
| **`s{i}` — search_branch_id** | per-entering | every `try_branch` call | one per `(parent_state, hypothesis)` entering |
| **`b{j}` — node_id** | per-state | first time a new `state_hash` is seen in this `_TreeBuilder` | one per unique post-saturation kb state (DAG-collapsed) |
| **`l(h_1, …, h_d)` — logical_branch** | per-hypothesis-path | derived field, not allocated | the path tuple itself; same value across enterings of the same path |

`try_branch` always pays its own saturation cost (the fork has
to materialise to produce a state we can hash) — but `_explore`'s
recursive subtree exploration is what state-hash dedup skips:

```
try_branch(parent_kb, h_i)               ← always runs (one per entering)
    → fork + saturate → BranchResult(kb=fork, search_branch_id=s{i}, …)
                              │
                              ▼
state_hash(BranchResult.kb)              ← parent's local index check
    │
    ├── miss → alloc node_id = b{j}      ← fresh state
    │            │
    │            ▼
    │         _explore(b{j}, …)           ← recurse; explore subtree
    │            │
    │            ▼
    │         cache (subtree_result, b{j}) on the node
    │
    └── hit  → reuse existing node_id b{j}  ← DAG link
                 │
                 ▼
              retrieve cached subtree_result on b{j}
                 │
                 ▼
            (skip _explore — work already done)
                              │
                              ▼
integrate(parent_kb, child_result, subtree_result)
    ← always runs per entering. Each entering's parent applies
      its own per-level `is_unconditional_at` decisions; the same
      subtree_result may be adopted by P_1 and partially rejected
      by P_2 depending on each parent's own hypothesis context.
```

So a node `b{j}` has:
- **One subtree** (exploration done once, by the first entering).
- **N enterings**, each with its own `s{i}` and its own
  `integrate` decision against its own parent. Different parents
  → different integration outcomes from the same subtree result
  → distinct timeline events.

The two-channel rule stands: down is `try_branch`, up is
`integrate`. State-hash dedup is a parent-side optimisation
sitting **between** them — read-only against an immutable
node cache.

### What gets cached on the node

A `SearchNode` today carries `(id, parent, hypothesis,
kb_snapshot, firings, verdict, children, unsat_core, forced)`.
Under the rewrite it adds the **subtree's bubble-up payload** so
DAG-deduped enterings can retrieve it instead of re-exploring:

```python
@dataclass(frozen=True)
class SearchNode:
    id:               int                       # node_id (b{j})
    parents:          tuple[int, ...]            # all parents (was singular `parent`)
    state_hash:       int                       # the dedup key
    hypothesis:       Fact | None
    kb_snapshot:      KnowledgeBase | None
    firings:          tuple[Firing, ...]
    verdict:          Literal["alive","dead","open","solution"]
    children:         tuple[int, ...]           # node_ids
    unsat_core:       frozenset[Fact]
    forced:           bool

    # ── NEW (T1.5a.20.5+.6) — cached subtree bubble-up ──────
    subtree_result:   SubtreeBubbleUp           # what the subtree wants the parent to adopt
    enterings:        tuple[EnteringRecord, ...] # one per `s{i}` that landed here
```

`SubtreeBubbleUp` is the cached `proposed_negatives /
forced_positives / learned_nogoods / consume_stats_delta` from
the FIRST entering's exploration. Subsequent enterings retrieve
it from the cache.

`EnteringRecord` is the per-entering audit row: `(search_branch_id,
parent_node_id, ts_ms, parent_path_condition_hash)` — lets the
dump count enterings per logical branch and per DAG-shared
node without re-running anything.

### What's preserved vs what's new

| invariant | today | rewrite |
|---|---|---|
| One saturation per entering | ✓ | ✓ |
| Subtree exploration dedup'd by post-sat kb state | ✓ | ✓ (cache moves to SearchNode, not `_TreeBuilder.state_index`) |
| Same logical branch at different parent contexts → distinct work | ✓ (different state_hash → different `b{i}`) | ✓ (different state_hash → different `b{j}`; same `logical_branch` value) |
| Different logical branches at same state → shared subtree | ✓ (single `b{i}`, single subtree) | ✓ (single `b{j}`, single subtree; distinct enterings `s{i}` per parent path) |
| Per-entering audit trail in dump | ✗ — entering collapses into the dedup'd node | ✓ — `EnteringRecord` per entering; `branches/b{j}/enterings/s{i}/` per-entering dump |
| Multi-parent DAG explicit | ✗ — `SearchNode.parent: int` (singular) | ✓ — `SearchNode.parents: tuple[int, ...]` |

## The rule

**Exactly two places** where data crosses a branch boundary:

1. **Down — `try_branch(parent_snapshot, hypothesis) → BranchResult`.**
   The only function that takes parent state and produces child
   state. Pure modulo the saturator's RNG (none in M1).
2. **Up — `integrate(parent_kb, child_result) → BranchResult?`.**
   The only function that ingests a child's outcome into a parent.
   May return its OWN `BranchResult` if the parent itself dies as
   a consequence of the integration (caller's integrate step
   bubbles further); else `None`.

**Nothing else.** No `_kb_chain_ctx`. No `_consume_caches_ctx`.
No `fork()` field that aliases a mutable container with the
parent. No descendant call that writes a fact, clause, or counter
into any kb other than its own.

Bubbling depth-N → root happens by **N integrate calls**, each
one acting only on its immediate parent. The recursion is the
fact-propagation chain, not a one-shot chain-walk.

## Target architecture

### `BranchResult` schema

Today `BranchResult` carries `(branch_id, hypothesis, kind, kb,
firings, unsat_core)`. The new schema additionally carries
**every datum the parent needs to bubble**:

```python
@dataclass(frozen=True)
class BranchResult:
    search_branch_id: int                      # `s{i}` — per-entering, monotonic
    logical_branch:   tuple[FactId, ...]       # hypothesis-path tuple (= path_condition)
    hypothesis:       Fact
    kind:             Literal["alive", "dead"]
    kb:               KnowledgeBase            # fork, fully isolated
    firings:          tuple[Firing, ...]
    unsat_core:       frozenset[Fact]
    # node_id (`b{j}`) is assigned by the parent after state-hash
    # check — not by try_branch — so it's not on BranchResult.
    # Parent integrate writes it into the SearchNode + the
    # per-entering audit record.

    # ── NEW (T1.5a.20.1) ──────────────────────────────────────
    # Facts the child wants the parent to add (back-prop writes
    # from grandchildren that bubbled into this child, plus
    # `(not own_hypothesis)` if this branch died unconditionally).
    # The PARENT decides whether to take them — child only proposes.
    proposed_negatives:  tuple[Fact, ...] = ()

    # Forced positives derived in this child's saturation whose
    # premise chain doesn't touch any hypothesis — provably true
    # at every shallower level. Parent integrates; if also forced
    # at parent's level, parent includes them in its own result.
    forced_positives:    tuple[Fact, ...] = ()

    # No-good clauses learned from any conditional deaths this
    # child saw (its own, or bubbled from grandchildren). Each
    # clause is a frozenset of (rel_name, args) hypothesis ids.
    # Parent integrates into ITS _nogoods; if root, the set is
    # the canonical no-good store.
    learned_nogoods:     tuple[frozenset[tuple[str, tuple]], ...] = ()

    # Counter delta — per-result instead of shared mutable. Parent
    # adds to its own consume_stats; root's consume_stats is the
    # canonical total.
    consume_stats_delta: ConsumeStatsDelta = ConsumeStatsDelta()

    # Subtree of search nodes the child produced — id-rebased so
    # the parent's tree builder can splice them in under a fresh
    # id range (T1.5a.20.5). Each node carries its own children's
    # ids inside the subtree.
    subtree:             SearchSubtree | None = None
```

`SearchSubtree` is a tree of `SearchNode` records with **local**
ids (rooted at 0 for the child's branch); the parent's
`_TreeBuilder` rebases them on integration.

### `try_branch` (the down channel)

```python
def try_branch(
    parent_snapshot: KnowledgeBase,
    hypothesis: Fact,
    *,
    branch_id: int,
    path_condition: tuple[FactId, ...],    # explicit, not via ContextVar
    rule_set_ref:   RuleSetRef,            # opaque; for RPC may be a hash
    saturator_steps: int = 10_000,
) -> BranchResult:
    fork = parent_snapshot.fork()          # T1.5a.20.1 — fully isolated
    # write hypothesis + saturate + detect + (if dead) re-saturate
    # → identical to today's body, but the fork shares NO mutable
    # container with the parent
    ...
```

**Key invariant:** after `try_branch` returns, `parent_snapshot`
is bit-identical to its pre-call state. The function is a pure
`(snapshot, hypothesis) → BranchResult` modulo saturator firings,
which themselves only touch `fork`.

### `integrate` (the up channel)

```python
def integrate(
    parent_kb: KnowledgeBase,
    child_result: BranchResult,
    *,
    parent_own_hypothesis: Fact | None,
    parent_path_condition: tuple[FactId, ...],
) -> BranchResult | None:
    """Integrate `child_result` into `parent_kb`. Returns a
    BranchResult if `parent_kb` itself died as a consequence
    (caller integrates further up); else None.

    Steps, in order:
      1. Adopt unconditional-dead negatives:
           if child_result.kind == "dead"
              and is_unconditional_death(child_result.kb,
                                          child_result.unsat_core,
                                          own_hypothesis=
                                          child_result.hypothesis):
              parent_kb.add_fact((not child_result.hypothesis))
              # plus symmetric mirror if applicable
      2. Adopt proposed_negatives (from deeper bubbles):
           for f in child_result.proposed_negatives:
              if is_unconditional_at(parent_kb, f.premises,
                                      own_hypothesis=
                                      parent_own_hypothesis):
                  parent_kb.add_fact(f)
              # else: the negative was unconditional at the child
              # level but is conditional here — discard at this
              # frame, do not propagate further
      3. Adopt forced_positives similarly (S1.5a.14 forced-mining
         relocated here).
      4. Adopt learned_nogoods that are still applicable at this
         level (clause minus parent's path condition).
      5. Re-saturate parent_kb once after the integration batch.
      6. Re-run ContradictionDetector(parent_kb).detect();
         if it fires, parent_kb is dead → return BranchResult.dead
         with its own unsat_core. (Closes Q2c from S1.5a.19.)
      7. Re-prune parent_kb.alive against the new _negated_facts
         (closes Q2a from S1.5a.19).
      8. Splice child_result.subtree into the parent's
         _TreeBuilder under a fresh id range.
      9. Return None if parent stays alive; else a BranchResult
         summarising the parent's death.
    """
```

The recursive bubble is automatic: a deep death's `(not h)`
becomes `proposed_negatives` in the child's result; the immediate
parent's `integrate` either adopts (if also unconditional at this
level) or drops; the parent's caller's `integrate` sees the
adopted negative as part of the parent's result; and so on.
**N integrations to reach root from depth N.** No chain-walk.

### The parent loop (rewritten `_consume` + `_descend`)

```python
def _consume(parent_kb, nid, …):
    while True:
        candidates = _candidates_for(parent_kb)
        if not candidates: break

        # Per-sweep batch of child results.
        results: list[BranchResult] = []
        for h in to_check:
            r = try_branch(parent_kb, h,
                           branch_id=builder.peek_id(),
                           path_condition=parent_path + (h_id,),
                           rule_set_ref=parent_kb.rule_set_ref)
            results.append(r)

        # Single integration pass for the whole batch.
        parent_died_results: list[BranchResult] = []
        for r in results:
            promo = integrate(parent_kb, r,
                              parent_own_hypothesis=node_h,
                              parent_path_condition=parent_path)
            if promo is not None:
                parent_died_results.append(promo)
                break    # parent is dead, no point continuing the sweep

        if parent_died_results:
            return _build_dead_result(parent_died_results[0])

        # If no unconditional death landed during integration,
        # we've reached fixpoint — hand off to `_descend` for the
        # alive-recursion pass.
        if not any(_added_unconditional(parent_kb, r) for r in results):
            return _descend(parent_kb, nid, …)
```

The function no longer touches ContextVars carrying ancestor kbs;
it only reads `parent_kb` (passed in) and `node_h` (also passed).
The recursive `_explore(child_result.kb, …)` for alive children
keeps the same shape — the recursion IS the chain.

### T1.5a.20.6 — Per-child-branch dump (BranchResult + integrate)

The current
[`state_dump.py`](../../../ein.py/src/ein/inference/state_dump.py)
emits **one aggregate** `node_resaturated` event per resat cycle
covering all children's back-prop writes + derivations. Under the
two-channel architecture every child has a full `BranchResult`
and every integration is a discrete event — both should land
verbatim on disk, per child, so the dump reader can answer
"what did THIS search branch propose, and what did the parent
adopt from it?" without inferring positionally.

**Filesystem layout** preserves the existing `branches/b{j}/`
hierarchy (one folder per state-deduped node) and adds an
`enterings/` subfolder for the per-entering records:

```
branches/b{j}/                       ← per-node (state-hash dedup, one entry per unique state)
   hypothesis.ein                    ← node-level; written by the first entering, immutable thereafter
   post_sat.ein                      ← node-level; same
   firings.jsonl                     ← node-level; saturation firings of the first entering
   verdict.json                      ← node-level; aggregates over the subtree
   subtree_result.json               ← NEW: cached SubtreeBubbleUp (the bubble-up payload subsequent enterings reuse)
   enterings/                        ← NEW: per-entering folder
       s{i_first}/
           branch_result.json        ← BranchResult for this entering
           integrate.json            ← parent's adopt/reject decision for this entering
       s{i_second}/                  ← DAG-shared subsequent entering
           branch_result.json        ← this entering's own BranchResult (note: kb may differ slightly if parent context differs; state_hash matched anyway)
           integrate.json            ← this entering's parent's adopt/reject decision
   resats/                           ← if this node itself runs resats over its own children
       <cycle>.{ein,json,_firings.jsonl}
   branches/                         ← children (the recursive part)
       b{j+k}/...
```

The crucial property: `branches/b{j}/` exists ONCE per unique
state; `enterings/s{i}/` is created **on every entering** even
when the entering DAG-links into a pre-existing `b{j}`. The
state-hash dedup saves only the node-level work; the
per-entering audit is always recorded.

Adds three artefacts per child:

- **`branches/b{j}/subtree_result.json`** — the cached subtree
  bubble-up payload. Written by the FIRST entering's
  exploration; read by subsequent enterings that DAG-link here.

  ```jsonc
  {
    "node_id":            6,
    "state_hash":         "0x4f8a...",
    "proposed_negatives": [
      { "fact": {...},
        "source": "own_unconditional" },             // (not h_own) for this node's hypothesis
      { "fact": {...},
        "source": "bubbled_from",
        "from_descendant_node": 42 }                 // bubbled up from b42 during subtree exploration
    ],
    "forced_positives": [
      { "fact": {...}, "rule": "range-elimination",
        "premises": [...] }
    ],
    "learned_nogoods": [
      { "clause": [["color-loc", ["Red","House-1"]],
                   ["smoke-loc", ["Kools","House-2"]]],
        "size": 2 }
    ],
    "consume_stats_delta": {...},
    "first_entering": "s12"                          // which entering produced this cache
  }
  ```

- **`branches/b{j}/enterings/s{i}/branch_result.json`** — the
  per-entering `BranchResult` (what came out of THIS `try_branch`
  call, before any state-hash dedup decision). Carries the
  entering-specific kb (which may differ slightly from the
  node's cached `kb_snapshot` if the parent's state differed in
  a way that didn't change the post-saturation `state_hash`):

  ```jsonc
  {
    "search_branch_id": 12,                            // s{i}
    "logical_branch":   [["color-loc", ["Red", "House-1"]]],
    "node_id":          6,                             // b{j} after state-hash dedup
    "node_dedup":       "miss" | "hit",                // was this the first entering on b{j}?
    "parent_node_id":   0,
    "hypothesis":       {...},
    "kind":             "dead",
    "firings_count":    233,
    "unsat_core":       [...]
  }
  ```

  When `node_dedup == "miss"`, this entering's subtree was
  freshly explored — `branches/b{j}/subtree_result.json` was
  written by this entering. When `node_dedup == "hit"`, this
  entering reused the existing `subtree_result.json`.

- **`branches/b{j}/enterings/s{i}/integrate.json`** — what
  happened when **this entering's parent** integrated the
  (possibly cached) subtree result:

  ```jsonc
  {
    "parent_node_id":     0,                           // integrating frame
    "integrate_seq":      27,                          // global timeline seq
    "ts_ms":              2510.4,
    "search_branch_id":   12,                          // s{i}
    "node_id":            6,                           // b{j} the subtree result came from
    "node_dedup":         "hit",                       // was the subtree fresh or cached?
    "logical_branch":     [["color-loc", ["Red", "House-1"]]],
    "negatives_adopted": [
      { "fact": {...}, "rule": "<integrate-unconditional-from-child>",
        "via_node": 6 }                                // source: subtree_result.json on b6
    ],
    "negatives_rejected": [
      { "fact": {...}, "reason": "conditional_at_parent_level",
        "blocking_hypothesis": {...},
        "via_node": 6 }
    ],
    "positives_adopted": [...],
    "positives_rejected": [...],
    "nogoods_merged":  [...],
    "nogoods_subsumed_existing": [...],                // existing was strictly larger
    "nogoods_subsumed_by_existing": [...],             // new was strictly larger
    "stats_delta_applied":      {...},
    "post_integrate_contradiction": null,              // or { unsat_core: [...] } — Q2c from S1.5a.19
    "post_integrate_alive_pruned": 3,                  // count, for Q2a from S1.5a.19
    "post_integrate_resat_firings": 12                 // count of firings in the post-adopt resat
  }
  ```

  **Two enterings that DAG-link to the same node still produce
  two distinct `integrate.json` files** because they sit at
  different `parent_node_id`s with different parent KBs — their
  adopt/reject decisions can diverge even though the subtree
  result is shared.

The per-cycle `resats/<cycle>.json` becomes a **summary**
pointing at the per-entering integrate files:

```jsonc
{
  "cycle": 1, "node_id": 0,
  "child_integrates": [
    "branches/b1/enterings/s1/integrate.json",
    "branches/b2/enterings/s2/integrate.json",
    ...
  ],
  "back_prop_writes_count": 11,
  "resat_derivations_count": 8,
  "new_firings_count": 239
}
```

Timeline entries (`00_timeline.jsonl` from T1.5a.19.3) gain
**two** new events:

```jsonc
{ "seq": 26, "ts_ms": 2502.1, "event": "try_branch",
  "search_branch_id": 12, "logical_branch": [...],
  "parent_node_id": 0, "kind": "dead", "firings": 233 }

{ "seq": 27, "ts_ms": 2510.4, "event": "integrate",
  "search_branch_id": 12, "node_id": 6, "node_dedup": "miss",
  "parent_node_id": 0, "logical_branch": [...],
  "adopted_negatives": 1, "adopted_positives": 0,
  "rejected_negatives": 0,
  "post_integrate_contradiction": false }
```

`node_dedup` ∈ {`"miss"`, `"hit"`} is the explicit answer to
"did this entering DAG-link?". A `"hit"` event tells the reader
that no fresh `_explore` ran for this entering — the cached
`branches/b{j}/subtree_result.json` was reused, but the parent
still made its own adopt/reject decisions against its own KB.

One pair of events per child entering; the parent's batch of N
child enterings produces N pairs plus ONE `node_resaturated`
event for the post-batch resat (already landed in T1.5a.19.3).

## Soundness

The recursive bubble is equivalent to today's chain-walk:

- An unconditional death at depth N writes `(not h)` to depth
  N-1's kb (via integrate step 1).
- Depth N-1's parent loop sees the new negative; if N-1's
  `is_unconditional_death(kb, …, own_hypothesis=h_{N-1})`
  returns True for any premise touching the new negative, N-1
  itself can propose the negative to its parent. The check is
  the existing predicate, applied per level instead of once at
  the death site.
- The propagation chain terminates at root or at the first level
  where the negative becomes conditional.

The current code optimises this by writing all the way up in one
call. That optimisation is **lossy** in one subtle way: it
applies the depth-N is_unconditional check and assumes the
verdict transfers to every shallower level — which IS sound but
masks the per-level adoption decision. Stratifying per level
makes the audit trail explicit (every adopt step lands in a
`node_resaturated` event with a single bubble source) and lets
S1.5a.19's adopt-rules (a per-level decision about whether to
adopt + which rule fires next) plug in naturally.

## Benefits

1. **Distributed-ready.** `try_branch` is pure
   `(snapshot, hypothesis) → BranchResult` modulo internal
   firing. T1.5a.20.6's audit makes the wire format concrete.
2. **Clear back-prop tracing in the dump.** Each `integrate()`
   call is one bubble step on one parent. The
   [S1.5a.11](s1.5a.11_state_dump.md) dumper's
   `node_resaturated` event maps 1:1 to one integrate call;
   `back_prop_writes` is the integrate step's adoption set;
   `from_dead_child` is no longer a positional inference but
   a direct argument link. Surface improvement closes the
   "where did the unconditional bubble from?" question that
   motivated S1.5a.11's T1.5a.19.3 enhancement.
3. **Adequate parent re-saturation.** The parent loop owns the
   moment when all sibling results are in; it can re-saturate
   once at the end of the batch and re-run
   `ContradictionDetector` (integrate step 6, closes
   **Q2c** from S1.5a.19) instead of mid-traversal.
4. **Alive-set refresh on resat (Q2a from S1.5a.19).**
   Integrate step 7 owns alive-pruning against the new
   `_negated_facts`; no fork sees a stale alive set after a
   resat.
5. **Eager mode (S1.5a.17) cleanup.** `BubbleAbort` raises
   exactly once, from the root frame's integrate step, and only
   when the eager flag is on. Intermediate frames don't even
   look at the flag.
6. **No more "shared mutable container" sharp edges.** Every
   shared field listed in `kb.fork()`'s docstring becomes
   either truly immutable (config, rules) or per-fork (stats,
   nogoods, committed_hypotheses). The docstring's "caveat
   about entity back-pointers" is closed via T1.5a.20.1.d.
