; S1.5a.16 — Branch-order shuffle invariance under depth cap.
; Stronger than S1.5a.2a's fixpoint property: the user's claim is
; that at ANY cut depth D, the partial information mined by the
; solver — alive set, dead set, back-propagated (not h) facts,

## The invariant

Let `solve(kb, rules, max_depth=D)` produce, on completion:

- `alive(D)`  — set of hypothesis facts still open at the frontier
- `dead(D)`   — set of hypothesis facts whose subtree closed at ≤ D
- `bubbled(D)` — set of `(not h)` facts back-propagated into root.kb
- `fired(D)`  — multiset of `(rule_name, binding)` firings per branch

> **Claim.** For any permutation σ of the candidate ordering at
> every branch point, the four sets above are identical functions
> of `(kb, rules, D)` — they do not depend on σ.

Equivalently: at depth cap D, the algorithm has *access to the
same partial information* regardless of which order it walked
the tree. Branch IDs differ; per-branch *contents* at the same
content-identified branch are the same.

This is strictly stronger than [S1.5a.2a's fixpoint
property](s1.5a.2a_statistics_determinism.md#the-invariant) —
that one says "once the loop closes, everything equalises via
re-saturation". S1.5a.16 says "the depth-D *prefix* of the loop
already equalises." A counterexample for .16 need not violate
.2a: a bug can make a `(not h)` reach root at depth 2 via path A
but only at depth 3 via path B, while the *eventual* `(not h)`
set is identical.

## Why this is suspected

Three engine mechanisms can each leak ordering into the
partial-information picture:

1. **State-hash caching** ([`builder.state_index`](../../../ein.py/src/ein_bot/inference/builder.py)).
   Two branches reaching the same dead reason via different
   saturated KBs cache as distinct dead nodes. Whichever order
   the first cache hit lands changes the *witness* state — and
   downstream `_promote_verdicts` aggregates over the witness.

2. **Back-prop dependence on visit order** ([S1.5a.14
   Phase 1](s1.5a.14_transitive_backprop.md)). The transitive
   bubble fires when a death is *fully* unconditional. A death
   discovered at depth 3 via path A may be unconditional w.r.t.
   {root, depth-1, depth-2}; via path B the same death may be
   reached at depth 2 with a smaller hypothesis chain and so
   classifies as unconditional at a shallower level. Both
   bubble to root *eventually* (good for .2a) but the *depth at
   which the negative becomes available* differs — visible at
   any D between the two.

3. **NAF fire-time re-eval** ([S1.5a.1
   T1.5a.1.1](s1.5a.1_naf_semantic_rearch.md)). The re-eval is
   done against the kb at *dequeue* time, but dequeue order
   inside a saturation pass is FIFO-by-enqueue — and the
   enqueue order depends on which rule plans matched first,
   which depends on `_facts_by_relation` insertion order, which
   depends on which back-propagated `(not h)` landed when. If
   branch A bubbled a negative into the kb before branch B
   started its saturation, B sees a tighter NAF view than it
   would have under the reverse order. Saturation is meant to
   be a fixpoint and so should still equalise *eventually* — but
   at depth D the firing multiset can differ.

None of these is a refutation; each is a hypothesis the harness
would test.

## The harness

A pytest-driven shuffle-and-diff rig, modelled on
[`tests/inference/test_branch_determinism.py`](../../../ein.py/tests/inference/test_branch_determinism.py)
(the S1.5a.1a determinism test) and the per-phase dumper
([S1.5a.11](s1.5a.11_state_dump.md)):

```python
def test_depth_bounded_shuffle_invariance(puzzle, depth, seeds):
    kb0 = parse_and_load(puzzle)
    snapshots = []
    for seed in [None, *seeds]:                 # None = default sort
        kb = kb0.fork()
        cfg = SolverConfig(
            max_depth=depth,
            candidate_order=("shuffle", seed) if seed else "sort",
        )
        tree = solve(kb, cfg)
        snapshots.append(snapshot_partial_info(tree, kb))

    ref, *others = snapshots
    for s in others:
        assert s.alive    == ref.alive
        assert s.dead     == ref.dead
        assert s.bubbled  == ref.bubbled
        assert s.fired_mset == ref.fired_mset
```

`snapshot_partial_info` projects the depth-D solve into the
content-addressed four sets above (no branch IDs, no per-run
timing). The dumper from S1.5a.11 already produces the kb +
verdict tree per node — reuse its serialisation, content-hash
the per-node records, aggregate.

`SolverConfig.candidate_order = ("shuffle", seed)` applies a
`random.Random(seed).shuffle(...)` to the list `_candidates_for`
returns, *after* the S1.5a.1a content-sort. The default `"sort"`
is the S1.5a.1a behaviour unchanged. Tests parametrise over
`seeds = [0, 1, 2, ..., 9]` (10 permutations is enough to catch
almost any order-sensitivity; cheap on small depth).

### T1.5a.16.3 — Triage the violations

Open task; scope depends on what T1.5a.16.2 surfaces. Three
likely classes (one per § "Why this is suspected" suspect),
each addressed differently:

- **State-hash cache class.** If two branches hit the same dead
  reason via different witness states and the alive set after
  cache replay differs, the cache key needs unsat-core
  refinement — folds into [S1.5a.15 Phase
  1](s1.5a.15_dead_caching_unsat_core.md).
- **Back-prop-depth class.** If a `(not h)` bubbles to a
  shallower ancestor only on some orderings, that's the case
  [S1.5a.15 Phase
  2](s1.5a.15_dead_caching_unsat_core.md#phase-2--partially-unconditional-back-prop)
  is designed for: partially-unconditional negatives need to
  bubble to the depth of their shallowest required hypothesis,
  not just to fully-unconditional root.
- **NAF cross-branch-timing class.** If firing counts differ
  because a bubbled negative landed before/after a sibling's
  saturation pass, the fix is to make saturation deterministic
  *within* a branch w.r.t. the kb-at-entry — e.g. snapshot the
  kb at branch entry and saturate against the snapshot, not
  the live kb. Bigger refactor; defer to a follow-up stage if
  it surfaces.

## Open questions

- **Q-S1.5a.16.A — Is the invariant actually claimed for the
  `fired` multiset, or just for the derived-fact sets?**
  Firings include duplicates the engine de-dupes via `_fired`;
  the *productive* multiset (firings that added a fact) is the
  more robust quantity. May relax to `productive_fired` if
  raw `fired` proves order-sensitive in benign ways.
- **Q-S1.5a.16.B — Does the harness need to gate hypgen-stats
  too?** `bench_solve --verbose` exposes per-branch `naf_dropped`
  and `lookahead_killed` counters; arguably those should also be
  shuffle-invariant. Adding them is one more `Counter` per
  snapshot; cheap. Decision: add them iff the four core sets
  pass and the counters still differ — that itself would be
  evidence of a leak the four sets miss.
- **Q-S1.5a.16.C — What about hypothesis-scoring under
  [S1.5a.7](s1.5a.7_hypgen_scoring_branch_info.md)?** Scoring
  is a function of the kb, so under fixed kb it's
  shuffle-stable. But if scoring changes branch *priority*
  (vs the post-S1.5a.1a content-sort), this invariant becomes
  "shuffle-invariance after re-scoring" — which still holds
  iff the scoring is deterministic. Note for when .7 lands.
