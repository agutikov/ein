## Context

M1 acceptance criterion #3 ([M1 README](../README.md#acceptance)):

> The markdown trace matches the target trace to within a
> checklist of *named rule firings* — every move in the human
> walkthrough has a corresponding rule firing in the engine
> trace (matching is structural, not literal).

The "target trace" is the human walkthrough in
[idea-08 §Target trace](../../../docs/ideas/08-human-style-deductive-trace.md#the-target-trace-paraphrased).
Each step ("Therefore the first house is yellow",
"Therefore the milk drinker lives in House 3", etc.) needs to
correspond to a named saturation-rule firing in the engine's
output.

Until solve actually works (S1.5a.1 + S1.5a.2), this acceptance
criterion can't be checked. After solve works, the trace's
named-rule firings need to be diffed against the human
walkthrough.

The matching is **structural**: same rule fires in the same
logical position, not the same prose. The engine's prose may
be "elimination-by-exhaustion fires on house-color
{House_1, House_2, House_4, House_5} with negative cover, leaving
Yellow at House_1"; the human prose is "Therefore the first
house is yellow". Both refer to the same firing.

## Tasks

### Task T1.5a.3.1 — Build the firing-checklist

For each "Therefore X" in idea-08, identify which named rule
should fire:

- "Therefore the first house is yellow" → `domain-elimination`
  on `house-color` slot 1 (House_1).
- "Therefore the milk drinker lives in House 3" → direct from
  condition (9), no firing — already in facts.
- "Therefore the Norwegian's neighbour is blue" → composition
  of `next-to` + spatial-cross-attr rules; identify which.
- … (15-20 entries total).

Output: a checklist as `s1.5a.3_idea08_checklist.md` (or
similar) — one row per "Therefore" with: (a) idea-08 prose, (b)
expected rule firing, (c) expected binding, (d) status (✅
landed / ❌ missing / 🚧 partial).

### Task T1.5a.3.2 — Run + diff

- **a.** Run `bench_solve examples/zebra2.ein --trace=zebra.md`.
- **b.** For each checklist row, find the matching firing in
  `zebra.md`.
- **c.** Annotate the checklist: every row should be ✅ for the
  acceptance criterion.

### Task T1.5a.3.3 — Resolve mismatches

Three failure modes:

- **Missing firing**: a rule the human used isn't in the
  engine's rule set — extend the rule library or rewrite the
  encoding.
- **Wrong order**: the firing happens, but at a different point
  in the trace — could be the engine's saturation order
  diverging from human order. Check P1.6's reordering pass
  ([S1.6.4](../p1.6_rendering_and_trace/s1.6.4_markdown_trace.md))
  and the trace renderer.
- **Wrong rule**: the engine fires a rule, but it's not what
  the human used — the engine's "elimination-by-exhaustion"
  may fire when the human said "process of elimination",
  but if the engine fires `single-parent` instead, that's a
  mismatch.

Each mismatch gets a sub-task: identify which, decide whether
to fix the engine, the encoding, or the trace renderer.

### Task T1.5a.3.4 — Stabilise the checklist

Once all rows are ✅, freeze the checklist as a regression
target. Future trace changes are diffed against it.

## Acceptance

- All idea-08 "Therefore X" rows in the checklist map to a
  named rule firing in `zebra.md`.
- The structural matching is documented (not literal — the
  checklist explicitly captures what "structural" means per row).
- The checklist file is checked in; tests (a CLI-driven
  diff job, or a pytest case) re-run it as a regression on
  every commit that touches the trace renderer or the rule
  library.

## Open questions

- **Q-S1.5a.3.A — What's the granularity of "structural
  matching"?** If the human says "Therefore Norwegian lives in
  the first house" and the engine fires three rules in
  sequence (one for each implication), do we count that as 1
  match or 3? Working answer: any *trace fragment* covering
  the human prose counts as a match; document per row.
- **Q-S1.5a.3.B — Should the trace renderer reorder firings to
  match the human walkthrough?** P1.6 S1.6.4 already has a
  reordering pass concept. Decide whether the reordering is
  on by default or gated.
- **Q-S1.5a.3.C — Reorder validity bounds.** Reordering can
  only happen between independent firings (no provenance
  dependency). The renderer needs to track provenance dag and
  refuse to reorder dependent firings. Test for the constraint.
