; S1.5a.2a — Statistics determinism. The user's invariant: at a
; given depth, the engine's amount-of-information is constant —
; so changing the branch exploration order must NOT change the
; final stats (alive counts, dead counts, firings, naf-dropped
; counters). Open branches are re-visited by design after new
; back-prop entries land, so any state that depended on order
; eventually equalises. Companion to S1.5a.1a (which made the
; order itself deterministic).

## The invariant

> **At a given depth and saturation-step count, the engine's
> derivable knowledge is a fixed point of the rule set against
> the input KB.** Therefore:
>
> 1. The set of facts derived during root saturation is the same
>    across runs — independent of process start, hash seed, or
>    the path through `kb.fork()` vs direct.
> 2. The set of alive hypothesis candidates at any depth is the
>    same across runs — independent of branch exploration order.
> 3. The final tree's per-branch counts (alive, dead,
>    contradictions, firings, `naf_dropped`) are the same
>    across runs.
>
> Open branches are re-visited by design — when back-prop writes
> a new `(not h)` into the parent, every sibling that had `h` on
> its alive set is re-checked against the now-tighter KB. So
> ordering effects on alive sets equalise at the loop's fixpoint.

Branch order can still affect *intermediate* statistics — at
step `n`, runs A and B may have visited different b1..bn. But
once the loop closes, the final accounting matches.

## Confirmed violation (2026-05-24)

### V1 — Fork vs direct saturation differs — *shipped*

**Root cause:** `Rule._kb` is bound at load time and points at the
parent KB. `KnowledgeBase.fork()` does
`new.rules = self.rules` (shared dict reference), so the rule's
`_kb` back-pointer still points at the parent after fork. The
engine's `_activators_for(rule)` consulted `rule.applications`,
which delegates to `rule._kb._rule_apps_by_rule[rule.name]` —
i.e. the *parent's* activator map. Activator facts derived
during the *fork's* saturation (e.g. `(adjacent-via-fwd …)`,
`(disjunctive-prune …)`) were therefore invisible to plan
compilation; rules they would have triggered never enqueued.

**Fix:** `engine._activators_for` now reads
`self.kb._rule_apps_by_rule.get(rule.name, ())` directly —
respecting the engine's own kb rather than the rule's load-time
binding. Single-line change in
[`engine.py`](../../../ein.py/src/ein/inference/engine.py).
Regression test [`tests/inference/test_saturator_fork_parity.py`](../../../ein.py/tests/inference/test_saturator_fork_parity.py)
pins the parity property. 628-test suite green.

**Downstream effect:** `bench_solve`'s root preview now reports
**89 root candidates** (the truthful post-saturation count) —
not 123. The "21-candidate gap to pre-B1" that
[S1.5a.2](s1.5a.2_hypgen_pre_pruning_recovery.md) was scoped to
close turns out to be **already closed** (89 < 101) under
disjunctive-prune + lookahead + negated-fact filtering, once the
preview pipeline reports the real number.

Reproducer:

```python
import sys; sys.path.insert(0, "ein.py/src")
from pathlib import Path
from collections import Counter
from ein.inference.closed import emit_closed
from ein.inference.saturator import Saturator
from ein.ir import parse
from ein.kb.store import KnowledgeBase

text = Path("examples/zebra2.ein").read_text()

kb1 = KnowledgeBase.from_ir(parse(text))
emit_closed(kb1)
list(Saturator(kb1).saturate())

kb2 = KnowledgeBase.from_ir(parse(text))
preview2 = kb2.fork()
emit_closed(preview2)
list(Saturator(preview2).saturate())

c1 = Counter(f.relation_name for f in kb1.facts)
c2 = Counter(f.relation_name for f in preview2.facts)
for k in set(c1) | set(c2):
    if c1.get(k) != c2.get(k):
        print(f"  {k}: direct={c1.get(k,0)}, fork={c2.get(k,0)}")
```

Output:

```
  not: direct=5, fork=0
  color-loc: direct=1, fork=0
```

Same input, same `emit_closed`, same `Saturator`. Forking the KB
before saturating *suppresses* 6 firings — 5 disjunctive-prune
firings (each producing `(not (color-loc …))`) and the one
`color-loc` positive derived from them via cascade.

Downstream effect: hypothesis-generator's lookahead-killed count
drops from **28 → 0**. The lookahead checks against the
fork's saturated KB, which is missing the disjunctive-prune
negatives; with no negatives, no candidates die immediately.
`bench_solve`'s "root hyps" preview reports **123 candidates**
emitted; direct hypgen reports **89**. Same puzzle.

Suspect: `Saturator` or `Engine` reads mutable state off the
parent KB that `fork()` shallow-copies without re-deriving. The
engine's `_fired` cache is fresh per Engine instance, so the
suspect is upstream — the candidate plans `compile_all` builds
against `kb._facts_by_relation` may capture a relation map the
fork shares with the parent but mutates differently.

T1.5a.2a.1 below diagnoses + fixes. The reproducer is the
acceptance test.

## Open questions

- **Q-S1.5a.2a.A — Causal chain for V1.** ✅ Resolved 2026-05-24:
  `Rule._kb` back-pointer survives fork unchanged because
  `fork()` shares the rule dict; engine consults
  `rule.applications` which delegates through that stale
  back-pointer.
- **Q-S1.5a.2a.B — Other entity-back-pointer divergences.** V1
  was the **engine-side** symptom. The underlying broken-fork
  pattern exists for every `Type._kb` / `Relation._kb` /
  `Rule._kb` / `Instance._kb` consumer that reads kb-mutable
  state — `Relation.facts`, `Relation.properties`, possibly
  `Type.instances` if `(instance …)` derivations land. The
  immediate fix (route engine through `self.kb._rule_apps_by_rule`)
  is narrow; an audit is owed. Candidate fixes for the broader
  pattern:
  - Deep-copy entities on `fork()` with fresh `_kb` references
    (one `dataclasses.replace` per entity; ~75 per fork in
    zebra2 — negligible cost; safest semantics).
  - Refactor entity properties to accept an explicit `kb`
    parameter at call site (invasive API change).
  - Add an explicit "rebind to kb" helper that callers use post-
    fork (forgettable; lint-only protection).
  Decision deferred until another consumer trips the bug.
