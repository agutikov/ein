; Diagnostic harness: capture the whole KB + alive set + config +
; firings + per-branch verdicts at every lifecycle hook. The

## What shipped

### Module — [`inference/state_dump.py`](../../../ein.py/src/ein_bot/inference/state_dump.py)

**The filesystem hierarchy mirrors the search tree.** Each
SearchNode gets a directory whose `branches/` subfolder holds its
children, recursively. Re-sat events triggered by a child's
unconditional death (S1.5.7b) land in the parent's `resats/`
subfolder, one numbered cycle per file group.

Layout::

    dumps/<puzzle>-<ts>/
      00_root_initial.ein
      01_root_saturated.ein
      01_root_saturated/{stats.json, firings.jsonl}
      02_root_hyps.ein
      02_root_hyps/hyp_stats.json
      resats/                          ← root re-sats (if any)
        001.ein
        001.json
        001_firings.jsonl
      branches/                        ← root's children
        b1/
          hypothesis.ein
          post_sat.ein
          firings.jsonl
          verdict.json
          resats/                      ← re-sats on b1
          branches/                    ← b1's children (recursive)
            b3/
              ...
      summary.json
      node.json                        ← root verdict tag

`StateDumper` exposes these hooks:

| hook                                                 | files written                                                  |
|------------------------------------------------------|----------------------------------------------------------------|
| `root_initial(kb)`                                   | `00_root_initial.ein`                                          |
| `root_saturated(kb, firings, naf_dropped)`           | `01_root_saturated.ein`, `01_root_saturated/{stats,firings}.…` |
| `root_hyps(alive, hyp_stats)`                        | `02_root_hyps.ein`, `02_root_hyps/hyp_stats.json`              |
| `node_alloc(nid, parent_id)`                         | pre-registers `<parent>/branches/b<nid>/`                      |
| `node_dump(nid, parent_id, h, kb, firings, verdict)` | `<parent>/branches/b<nid>/{hypothesis,post_sat}.ein + …`       |
| `node_resaturated(nid, cycle, …)`                    | `<nid>/resats/<cycle>.{ein,json,_firings.jsonl}`               |
| `summary(verdict, tree, config)`                     | `summary.json`                                                 |

The split between `node_alloc` and `node_dump` matters: alloc
pre-registers the node's filesystem path so child SearchNodes
allocated *during* the parent's exploration land under it, even
though the parent's own `node_dump` call happens *after* its
children's. The order is forced by `_consume` / `_descend` — they
alloc cond-dead and uncond-dead child SearchNodes mid-loop, then
the parent's interior SearchNode is added once the loop returns.

`.ein` files are full-KB snapshots — every layer (ONTOLOGY, FACT,
REASONING) collapsed into `(ontology …)` + `(facts …)` so each
file is independently readable. Provenance lands in each fact's
`:source` / `:rule` / `:hypothesis` kw-pair; the REASONING-layer
facts carry `:layer reasoning` so the reader can tell derived from
authored at a glance.

Designed for human inspection + `diff`. Not a serialisation format;
the output isn't necessarily round-trippable through `parse` (the
parser would need to know every derived fact's relation
declaration).

### Solver hookup

`solve()` gained an optional `dumper: StateDumper | None = None`
parameter (default keeps every existing caller byte-identical).
The dumper is bound through a
[`ContextVar`](../../../ein.py/src/ein_bot/inference/solver.py)
for the duration of the call, so `try_branch` and `_consume`'s
back-prop site read it without threading through `_explore` /
`_descend`'s signatures. Reset on exit via `try/finally`.

### Bench integration — `bench_solve --dump-states [DIR]`

[`demo/bench_solve.py`](../../../ein.py/demo/bench_solve.py) gained
a `--dump-states [DIR]` flag. Default DIR is `dumps/<puzzle-stem>-<timestamp>/`;
explicit DIR honoured. The script prints the dump path in its
summary so the user knows where to look.

### Tests — [`tests/inference/test_state_dump.py`](../../../ein.py/tests/inference/test_state_dump.py)

5 smoke tests on the `branching/03_five_hyps_one_alive.ein`
demo:

- `test_dumper_produces_root_phase_files` — 6 root-phase artefacts.
- `test_dumper_produces_branch_files` — every visited branch has
  its 5-file kit.
- `test_dumper_summary_json` — top-level keys + parseable verdict.
- `test_dumper_verdict_json_parseable` — per-branch verdict.json

## Use cases this unblocks

1. **"Why did the engine pick this branch first?"** — read
   `02_root_hyps.ein` for the seed alive set; the order matches
   the iteration order in `branches/`.
2. **"Why did this branch die?"** — read `branches/b<bid>/`'s
   `verdict.json` for the unsat-core, then trace it via
   `post_sat.ein`'s REASONING layer.
3. **"Did back-prop trigger here?"** — `branches/b<bid>/backprop_after.ein`
   exists iff back-prop wrote into the parent. Its `backprop.json`
   lists the negatives added.
4. **A/B comparing rule sets.** Two `bench_solve` runs against the
   same puzzle but different rule sets produce two dump trees;
   `diff -r` is enough to see what changed at each phase.
5. **Golden tree snapshots for P1.6** — the dump directory IS the
   ground truth for trace-fidelity tests; future P1.6 work can
   parse `01_root_saturated/firings.jsonl` + per-branch firings to
   diff against the human walkthrough.

## What's deferred

- **Pre-backprop snapshot.** Today only the *after* state is
  dumped on a back-prop event. Reconstruct the before state by
  reading the previous branch's `post_sat.ein` (the parent's
  state didn't change between branches). If a future user finds
  this fiddly, add `backprop_before.ein` mirroring the after-form.
- **Round-trippable .ein.** The dumps include `:layer reasoning`
  kw-pairs that the parser may or may not tolerate. Verified
  inspection-readable, not parser-readable. If a tool needs to
  re-feed a dumped KB back into the engine, write a normaliser
  that strips `:layer` / `:rule` kw-pairs before re-parsing.
- **JSON mirror of .ein files.** Asked in the design pass — opted
  for `.ein` only. If a downstream tool needs JSON, the
  serialisation helpers in `state_dump.py` (`_fact_to_sform` etc.)
  expose the building blocks for a JSON sister.
- **Streaming output for long runs.** Today every snapshot is a
  full KB write; on zebra2 with 48 branches × 3 .ein files +
  firings.jsonl, that's a few megabytes per run. Fine. If a
  puzzle blows up to thousands of branches, switch to compressed
  output or delta-only snapshots.

## Open questions

- **Q-S1.5a.11.A — When does the dumper own the directory?** The
  dumper currently creates `out_dir` on `__init__` and never
  cleans up. Tests fork into `tmp_path` so they're fine; the CLI
  user explicitly chose the directory. Add a `--dump-clean`
  option later if accidental overwrites become an issue.
- **Q-S1.5a.11.B — Per-search-tree-node identity.** Branch IDs in
  the dump come from `builder.peek_id()` in `_consume` /
  `_descend`. These align with the IDs the trace renderer (P1.6)
  uses, so cross-referencing the search tree with `branches/bN/`
  is trivial. Verify when P1.6 ships.
