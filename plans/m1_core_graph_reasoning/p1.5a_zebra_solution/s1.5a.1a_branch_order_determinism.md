## Observation

`bench_solve ./examples/zebra2.ein --progress-every 1 --max-depth 1 -v`
yields **a different b1 (and onward) every invocation**.
2026-05-24 sample, three consecutive runs:

| run | b1 hypothesis |
|-----|--------------|
| 1   | *(b1 above truncation window — first reported was b13)* |
| 2   | `(house-drink House-2 Tea)` |
| 3   | `(house-smoke House-4 Kools)` |

Same parsing, same root saturation, same `(query …)` block —
nothing changed about the puzzle between runs. The 30-branch
visit sequence differs end-to-end.

Setting `PYTHONHASHSEED=0` makes two consecutive runs **byte-
identical** down to b1, b2, b3, b4, b5: `(house-smoke House-3
Parliament)`, `(house-smoke House-5 Chesterfields)`,
`(house-smoke House-5 Old_Gold)`, `(house-pet House-3 Horse)`,
`(house-pet House-5 Snail)`. Confirms hash-randomisation as the
sole driver.

## Diagnosis

Two collaborating sites:

- [`solver.solve` :862](../../../ein.py/src/ein/inference/solver.py):
  `kb.alive = frozenset(root_alive)` — the root alive-set is
  stored as a frozenset, inherited by every fork.
- [`solver._candidates_for` :367](../../../ein.py/src/ein/inference/solver.py):
  `return list(pruned)` where `pruned` is the frozenset from
  `_prune_alive`.

`frozenset` iteration order is `hash(element)`-dependent. The
elements are `Fact` instances; `Fact` is a `@dataclass(frozen=True)`,
so its `__hash__` hashes the tuple of `args` — which are strings.
Python randomises `hash(str)` per process (PEP 456, default since
3.3), so each `python` invocation produces a different `hash(Fact)`
and a different `frozenset` iteration order. `_candidates_for`
faithfully passes that order to `_explore`, which visits the
candidates in iteration order. Hence: b1, b2, … flap across
processes.

The `_prune_alive`-returned `out = []` is built in iteration
order over `alive` (a frozenset), so the pruning step launders
the same non-determinism into every fork.

## Fix path

**Recommended (simple, surgical):** Sort the list emitted by
`_candidates_for` (and the seed in `solve`). One total ordering
key is the lexicographic tuple
`(relation_name, args)`; `args` is a tuple of strings already.
That sort is O(n log n) on a set that is at most a few hundred
candidates at the root and shrinks at every fork; negligible.

Alternatives considered + rejected:

- **Set `PYTHONHASHSEED=0`** in the entrypoint script — fixes
  the *demo* but not library callers, programmatic
  `solve(kb, …)` invocations from tests / notebooks, or anyone
  importing `ein.inference.solver` from a process started
  the standard way.
- **Store alive as a tuple/insertion-ordered dict** — works,
  but the codebase already leans on `frozenset` for O(1)
  membership in `_prune_alive` and the cache machinery.
  Switching the carrier widens the change radius for no
  additional correctness benefit over a sort.

## Tasks

  `sorted(pruned, key=lambda f: (f.relation_name, f.args))`.

  trace). Apply the same sort or note why the order doesn't
  matter.

## Open questions

- **Q-S1.5a.1a.A — Is the Fact ordering stable across
  cosmetic refactors?** `(relation_name, args)` is content-based,
  not identity-based, so renaming a relation in the puzzle (e.g.
  the S1.5a.5 `house-*` → `*-location` rename) WILL re-order
  candidates. Decision: the rename is a separate diff with its
  own golden re-baselining; the *intent* of T1.5a.1a is
  determinism *within* a fixed puzzle definition, not stability
  across schema changes.
- **Q-S1.5a.1a.B — Where else does hash-order leak?**
  Saturator `_seen` is `set[tuple[str, tuple, frozenset]]` —
  used only for membership checks, never iterated for user-
  visible ordering. Engine `_fired` same shape, same use.
  `_enqueue_pass` iterates `engine.cache.values()` (insertion-
  ordered dict). Spot-check the hypgen `seen` set
  (line 147 of `hypgen.py`) — membership only, safe. Audit
  the contradiction detector and trace renderer once they're
  more load-bearing.
