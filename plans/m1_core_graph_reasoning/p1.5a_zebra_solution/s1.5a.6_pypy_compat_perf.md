# S1.5a.6 — PyPy compatibility + perf measurement

**Phase:** P1.5a (zebra solution + saturator/NAF gaps).
**Estimate:** 1-2 days for the audit + bench; longer if
incompatibilities surface.
**Status:** parked; "measure and document" task, not "support
PyPy forever".
**Depends on:** [S1.5a.4](s1.5a.4_acceptance_zebra2_solves.md)
ships — measure against a known-correct solve.
**Blocks:** nothing; informs whether [P1.8 Theme B indexes](../p1.8_ein_lang_modules/README.md)
work is urgent.

## Context

The solver is pure Python with no native extensions; Lark
supports PyPy in recent versions; `@dataclass` works under
PyPy with some corners (slot interactions). PyPy is a natural
candidate for a free saturator speedup on the hot loop
(`_apply` + match binding).

User direction recorded 2026-05-24:

> If PyPy gives a free ≥5× on the hot loop, it changes the
> calculus on Theme B1 (indexes) in P1.8 — the saturator might
> not need the index work as urgently. Either way, document
> the finding so future perf work knows the baseline.

This is a *measurement* stage, not a "support PyPy as a
permanent constraint" stage. The output is a baseline plus a
go/no-go on PyPy as a primary perf path.

## Tasks

### Task T1.5a.6.1 — Compatibility audit

Run the existing pytest suite under PyPy3 (a recent release —
target whatever the user's host can install).

- **a.** Install PyPy + the package dependencies (`pip install
  -e .` under PyPy's venv). Document the install incantation.
- **b.** Run `pytest ein.py/tests/`; record failures.
- **c.** Categorise failures:
    - Lark grammar parser issues → almost certainly none in
      recent Lark.
    - `@dataclass` slot model corners → known PyPy/CPython
      divergence with `frozen=True` + slots; check.
    - Any C-extension imports — there should be none; confirm.
    - Numerical / float edge cases — unlikely in M1.
- **d.** Fix or document each failure. Failures the user
  decides not to fix → workarounds documented per-test or
  marked `@pytest.mark.skip_pypy`.

### Task T1.5a.6.2 — Perf measurement

- **a.** `bench_solve examples/zebra2.ein` under CPython
  (baseline) — record wall time + RSS.
- **b.** Same under PyPy — record wall time + RSS, including
  JIT warm-up (cold first run vs warm second run).
- **c.** `bench_saturate examples/saturation/**/*.ein` — broader
  saturation perf, since saturator is the hot loop.
- **d.** Branching demos under both interpreters.

Tabulate: per-demo, CPython time vs PyPy cold vs PyPy warm,
plus the speedup factor.

### Task T1.5a.6.3 — Report

Write a short results document (e.g., `s1.5a.6_pypy_results.md`)
covering:

- Compat audit status: how many tests pass, how many fail,
  which are fixed vs documented.
- Perf table: per-demo + zebra2.
- Recommendation: is PyPy a viable primary perf path? Or a
  one-off experiment? Bears on P1.8 Theme B indexes urgency.

### Task T1.5a.6.4 — Document caveats

Whatever compat workarounds were needed are recorded somewhere
durable — probably a `docs/kernel/inference/pypy_notes.md` (or
appended to the inference README).

## Acceptance

- Compat audit complete: pytest pass-rate on PyPy ≥ 95%, with
  failures categorised and (where reasonable) fixed.
- Perf table for zebra2 + branching demos, CPython vs PyPy.
- Recommendation recorded; future P1.8 Theme B planning
  references this stage's numbers.

## Open questions

- **Q-S1.5a.6.A — PyPy version.** Pin to a specific release for
  reproducibility; document. Likely PyPy 3.10+ (latest stable
  at measurement time).
- **Q-S1.5a.6.B — Cross-cutting impact on M2.** If M2's NL
  pipeline depends on llama.cpp / Python bindings to llama.cpp
  (S2.2.2), those may not work under PyPy. Note here that PyPy
  is *engine-only*; M2's NL frontend stays CPython.
