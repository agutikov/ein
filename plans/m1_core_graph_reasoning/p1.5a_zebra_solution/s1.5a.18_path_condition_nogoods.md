# S1.5a.18 — Path-condition no-good clause learning

**Phase:** P1.5a (zebra solution + saturator/NAF gaps).
**Estimate:** ~1 day for the storage + emit + subsume + filter
helpers and the pre-fork filter wire-in; ~half-day for tests +
smoke measurement. The mechanism is small (the data structure is
a set of frozensets) but touches every dead-recording site in
the solver.
**Status:** **closed 2026-05-26 (default off).** Mechanism +
tests shipped 2026-05-25 (`2d8cac3`); flag stays `False` in
M1. Measured 2026-05-26: demo 10 with flag on learns 6
clauses but prunes 0 nodes (same 32-node tree); zebra2 d=1
learns 0 clauses (all 31 dead leaves are depth-1 1-element
clauses owned by `back_propagate`). Without outer-loop
re-entry — formerly [S1.5a.17](s1.5a.17_eager_root_bubble_outer_loop.md)
(dropped) — clauses are learned-but-unused within a single
solve. The mechanism is sound; its useful regime is
[P1.5b](../p1.5b_lattice_search/)'s monotonic engine, where
the layer-by-layer outer loop consults `_nogoods` between
layers (per P1.5b README: "path-condition no-goods become
the unified CDCL prune mechanism for both engines"). The
default-flip decision moves to P1.5b. The dual-formulation
discussion below references S1.5a.15 (dropped 2026-05-26);
preserved for historical context.
**Depends on:** [S1.5a.1a](s1.5a.1a_branch_order_determinism.md)
(deterministic candidate order is a prerequisite for the
pre-fork filter — otherwise the same puzzle would learn
different clauses run-to-run).
**Blocks:** [S1.5a.13](s1.5a.13_acceptance_zebra2_solves.md) —
this stage is one of the largest remaining levers for cold-solve
on zebra2 because most deep deaths in zebra-style puzzles are
*conditional* (multi-level hypothesis chains), and the existing
back-prop captures only the unconditional subset.

## The mechanism

### Clause learning

A dead branch at depth N has a path condition: the chain of
hypothesis FactIds from root to this branch,
``(h_1, h_2, …, h_N)`` (root's first decision through the dying
candidate itself). When the branch dies:

```text
emit at root.kb._nogoods:  (not (and h_1 h_2 … h_N))
                          = frozenset{(h_1.rn, h_1.args), …, (h_N.rn, h_N.args)}
```

The clause says: "taking ALL of `{h_1, …, h_N}` jointly is
inconsistent given root's givens and the rule set." Equivalently
the disjunction ``¬h_1 ∨ ¬h_2 ∨ … ∨ ¬h_N`` holds at root.

### Pre-fork filter

Before forking on a candidate ``h_k`` at depth N (path =
``{h_1, …, h_{N-1}}``):

```text
prospective_path = path ∪ {h_k}
for clause C in root.kb._nogoods:
    if C ⊆ prospective_path:
        # taking h_k would complete a known-dead combination
        skip h_k entirely  — no try_branch, no SearchNode
        break
```

A filtered candidate never enters the SearchTree. The trace
records the pruning indirectly through the clause's presence in
``root.kb._nogoods``.

### Subsumption

Clauses are stored *minimal*: if a learned clause ``C₁ ⊆ C₂`` is
present, ``C₂`` is redundant (any future path superset of ``C₂``
is also a superset of ``C₁``). On emit:

```text
emit(new_clause):
    # subsumed?  some existing C ⊆ new_clause
    if any(C ⊆ new_clause for C in _nogoods):
        return  # nothing to do — already covered

    # subsumes existing?  remove every C' ⊇ new_clause
    _nogoods -= {C' for C' in _nogoods if new_clause ⊆ C'}
    _nogoods.add(new_clause)
```

Subsumption keeps the clause set bounded — without it,
deep-tree learning can pile up redundant supersets and inflate
the filter's per-candidate cost.

### Relationship with `(not h)` back-prop

A 1-element clause ``{h_self}`` is exactly the existing
``(not h_self)`` back-prop write — emitted on
**unconditional** deaths via :func:`back_propagate` since
[S1.5a.14](s1.5a.14_transitive_backprop.md). So:

- **Unconditional death (path-condition reduces to {h_self})**:
  current ``back_propagate`` writes ``(not h_self)`` into
  ``root.kb._negated_facts``; this stage adds *nothing new*
  (the trivial 1-element clause is subsumed by the
  ``(not h_self)`` filter already in place).
- **Conditional death (path-condition has ≥ 2 elements)**:
  current code records a `cond-dead` SearchNode with per-call
  ``verdict_at`` caching but **no root-level fact** survives
  the branch. This stage adds the multi-element clause, which
  composes across sibling subtrees that share an ancestor
  decision.

Implementation simplification: only call
:func:`emit_nogood` for clauses with **≥ 2 elements**. The
1-element case is owned by ``back_propagate``; emitting it from
both sides would double-cost subsumption checks for no win.

## Why this composes with eager mode (S1.5a.17)

A novel (non-subsumed) clause is a new piece of information at
root. Under [S1.5a.17](s1.5a.17_eager_root_bubble_outer_loop.md):

- ``emit_nogood`` bumps ``root.kb._pass_bubbled`` by 1 and raises
  :class:`BubbleAbort` (same as ``back_propagate`` /
  ``_mirror_forced_positive``).
- The outer driver discards the in-flight subtree, re-saturates
  root, restarts the pass.
- The next pass's pre-fork filter has the new clause and prunes
  matching candidates immediately.

Termination remains O(|root candidates|) outer passes — the
clause set grows monotonically, and each pass either lands a new
clause (strict information gain) or hits fixpoint.

## Why not just S1.5a.15 Phase 1

[S1.5a.15 Phase 1](s1.5a.15_dead_caching_unsat_core.md#phase-1--direct-dead-cache-by-unsat-core)
caches "future kb ⊇ unsat-core source-frontier ⇒ dead". The two
mechanisms are duals for conditional deaths whose source-frontier
contains ancestor-hypothesis facts:

| axis                       | S1.5a.15 P1 (source-frontier)               | S1.5a.18 (path-condition)               |
|----------------------------|---------------------------------------------|-----------------------------------------|
| stored                     | frozenset of ``(rel, args)`` of given facts | frozenset of hypothesis FactIds         |
| match test                 | every C fact ∈ candidate kb (O(|C|) lookups) | C ⊆ prospective path (O(|C|) set ops)   |
| typical |C| on zebra       | 5–10 (mix of givens + ancestor hypotheses)  | 2–6 (just the ancestor chain)            |
| fires at                   | ``_explore`` entry (after saturation)       | **pre-fork** (before try_branch saturates) |
| catches                    | any kb that reaches the same fact set       | only branches with same hypothesis chain |
| memory cost per clause     | O(|core|) tuple ids                          | O(depth) FactIds                         |

The path-condition test is **strictly cheaper** (smaller cores,
no fact-set lookup) and fires **earlier in the pipeline**
(pre-fork vs post-saturation) — wins on the conditional-death
hot path. But it misses two cases the source-frontier cache
catches:

1. **Dead by purely given facts** — no hypothesis required.
   `_negated_facts` already covers this via direct contradiction
   detection at root saturation, so P1's win here is small.
2. **Dead by an equivalent-but-differently-derived state** — two
   branches reach the same kb state via different hypothesis
   chains. P1's source-frontier cache hits both; path-condition
   only hits one.

Recommendation: ship S1.5a.18 first (lighter, faster, covers
the dominant case), measure on zebra2, then ship S1.5a.15 P1 only
if there's residual win on top.

## Soundness sketch

A branch's death proves the joint hypothesis-set inconsistent
*relative to the parent's saturated kb*. Specifically: given root
saturation + the rule library, the chain
``(h_1 ∧ h_2 ∧ … ∧ h_N)`` derives a contradiction. So at least
one of those hypotheses is false at root —
``¬(h_1 ∧ h_2 ∧ … ∧ h_N)``. Adding this disjunction as a
constraint to root is monotonically safe (the constraint is
implied by root + rules).

A future branch with prospective path ⊇ ``{h_1, …, h_N}``
commits *all N* hypotheses; one must be false, so the joint
state is contradictory. Filtering pre-fork is sound: the
``try_branch`` saturation would have derived the same
contradiction.

Cross-branch caveat: the source-frontier facts that produced the
death must still be in root.kb at the time of filtering. Since
all sources are given facts (Layer.FACT / Layer.ONTOLOGY) and the
solver never *removes* facts, this is automatic.

## Tasks

### T1.5a.18.1 — Storage + ContextVar + config flag

- **a.** Add ``KnowledgeBase._nogoods:
  set[frozenset[tuple[str, tuple]]]`` (root-only mutation; forks
  share by reference like
  [`committed_hypotheses`](../../../ein.py/src/ein_bot/kb/store.py)).
- **b.** Add ``_path_ctx: ContextVar[tuple[FactId, ...]]`` in
  ``solver.py`` (default empty tuple). Push the parent
  hypothesis on ``_explore`` entry for nodes at depth ≥ 1, pop
  on exit. Mirrors the ``_kb_chain_ctx`` lifecycle.
- **c.** Add ``enable_path_condition_nogoods: bool = False`` to
  :class:`SolverConfig`. Default off until measurement confirms
  the win on at least demo 10 + a constructed conditional-death
  fixture.

### T1.5a.18.2 — `nogoods.py` helpers

New module ``ein.py/src/ein_bot/inference/nogoods.py``:

- ``emit_nogood(root_kb, clause: frozenset[FactId]) -> bool``:
  subsumption-aware insert. Returns True iff a new clause was
  added (subsumption-skipped emits return False). Under eager
  mode (``_eager_pass_ctx`` set), bumps
  ``root_kb._pass_bubbled`` and raises ``BubbleAbort`` on a
  successful add — same control flow as ``back_propagate``.
- ``filter_by_nogoods(candidates, path_set, nogoods) -> list``:
  drop candidates ``h`` where any ``C ∈ nogoods`` satisfies
  ``C ⊆ path_set ∪ {(h.rn, h.args)}``. Single helper used by
  both ``_consume`` and ``_descend``.
- Single-element-clause guard: ``emit_nogood`` rejects clauses
  of size < 2 with a no-op (back-prop's domain).

### T1.5a.18.3 — Wire into the death sites

Three death sites in ``solver.py``; each gets the same emit
when ``enable_path_condition_nogoods`` is True:

- **`_consume` sweep**, after every classified death
  (unconditional **and** conditional): compute
  ``clause = path ∪ {(h.rn, h.args)}``; call ``emit_nogood``
  if ``len(clause) ≥ 2``.
- **`_descend`'s dead-leaf path**: same emit when ``try_branch``
  returns dead.
- **`_explore_inner`'s contradiction-on-entry**: same emit
  using the parent path; this is a rare defensive path
  (contradictions usually surface in ``try_branch``) but it
  records the no-good for the rare case where re-saturation
  inside ``_explore`` produced a clash.

Three pre-fork filter sites; each runs once per candidate
sweep when the flag is on:

- **`_consume` sweep**: filter the ``to_check`` list against
  the current ``_path_ctx`` before iterating.
- **`_descend`'s candidate loop**: same filter before the
  ``for h in candidates`` loop.

The flag-off path stays byte-identical (filter helper returns
the list unchanged when ``_nogoods`` is empty; emit is a no-op
under the flag check).

### T1.5a.18.4 — Tests

New ``ein.py/tests/inference/test_path_condition_nogoods.py``:

- **Unit — subsumption.** ``emit_nogood`` rejects a clause when
  a subset is already present; removes existing supersets when a
  new strict subset arrives. Stays minimal.
- **Unit — filter.** ``filter_by_nogoods`` drops candidates
  whose prospective path matches; leaves others.
- **Integration — flag-off parity.** Tree shape unchanged from
  pre-S1.5a.18 baseline on demo 10 (32 nodes).
- **Integration — flag-on emits on conditional death.** On a
  puzzle with at least one depth-2 conditional death (demo 10
  qualifies), at least one multi-element clause lands in
  ``root.kb._nogoods``.
- **Integration — eager + nogoods compose.** With both
  ``enable_eager_root_bubble`` and
  ``enable_path_condition_nogoods`` on, ``BubbleAbort`` fires
  from ``emit_nogood`` as well as ``back_propagate``, and the
  outer loop terminates. Verdict matches flag-off.

## Open questions

- **Q-S1.5a.18.A — Where does the *parent saturation state*
  matter?** A clause emitted at depth N rests on the parent's
  saturated kb at that point — specifically the rule firings the
  parent triggered. If a future re-saturation derives *more*
  facts at root, the same clause is still sound (monotonic). But
  what if root.kb changes such that the parent's saturation
  *would* have derived different facts? Today the engine never
  retracts; future ``domain-elimination``-style rules might.
  Decision: ship now under the monotonicity assumption; re-audit
  when retraction-capable rules land.
- **Q-S1.5a.18.B — Index for the filter?** Naive
  ``filter_by_nogoods`` is O(|candidates| × |clauses|). For
  zebra-scale clause counts (< 200) this is fine. If clause
  count grows, index by "smallest hypothesis id in the clause"
  so the filter only scans clauses whose smallest element is in
  the prospective path. Deferred until measurement says it
  matters.
- **Q-S1.5a.18.C — Trace representation?** Should the learned
  clauses appear in the ``(trace …)`` IR? Today
  :func:`SearchTree.to_ir` emits branch-open/branch-close events
  only; clauses are runtime side-state. A new
  ``(nogood h_1 h_2 … h_N)`` event after each death would let
  P1.6's trace renderer surface the learning explicitly. Out of
  scope for the engine ship; revisit in P1.6.
- **Q-S1.5a.18.D — Cross-puzzle clause reuse?** A clause
  learned solving puzzle X is generally NOT sound for puzzle Y
  (different givens). But for *re-runs* of the same puzzle, the
  clause set could be persisted and loaded as a head-start.
  Out of scope for M1; note for downstream perf work.
- **Q-S1.5a.18.E — Interaction with
  [S1.5a.15 P3](s1.5a.15_dead_caching_unsat_core.md#phase-3--per-level-back-prop)
  (per-level partially-unconditional back-prop)?** P3 writes
  ``(not h)`` at every ancestor at-or-below the shallowest
  required hypothesis's depth — a stronger statement than the
  conjunctive clause when ``D`` is shallow. If P3 ships, the
  conjunctive clause becomes subsumed by P3's
  ``(not h)`` writes for those cases. Composition is benign
  (subsumption catches it).
