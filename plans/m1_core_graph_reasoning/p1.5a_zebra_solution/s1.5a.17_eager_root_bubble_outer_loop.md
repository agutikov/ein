# S1.5a.17 — Eager root-bubble + outer re-entry loop

> **STATUS: DROPPED 2026-05-26.** Superseded by
> [P1.5b](../p1.5b_lattice_search/) set-indexed engines
> (monotonic + lattice). The tree-side outer re-entry loop +

## What changes

### Today (S1.5a.14 P1 — opportunistic bubble)

```
_explore(root) ──┐
  _consume       │
    try_branch h │
    death ───────┼─► back_propagate writes (not h) to every
                 │   ancestor kb (root included), clears
                 │   ancestor verdict_at caches
                 ▼
  recurse into next sibling — ancestors learn about (not h)
  next time they happen to re-sweep
```

Root learns *eventually*; control flow does not change. The
current branch keeps spending time on candidates a sibling
subtree just proved false.

### Proposed (S1.5a.17 — eager bubble)

```
_explore(root) ─────────────────────┐
  outer pass:                       │
    for h in root candidates:       │
      try subtree(h)                │
        deep death ──► bubble ──► ◄─┘ raise BubbleAbort
        deep forced+ ► bubble ──► ◄─┘ raise BubbleAbort
      catch BubbleAbort:            │
        discard partial subtree     │
        re-saturate root            │
        continue with next h        │
    end pass                        │
  if any bubble landed this pass:   │
    start new pass                  │
  else:                             │
    handoff to verdict promotion    │
```

Trigger (option 1): **any** unconditional bubble — positive or
negative, at any depth — aborts the in-flight subtree, lifts
the truth to root, and the outer loop moves on. The discarded
subtree is rebuilt on the next pass (cheaply, via state-hash
dedup) if its parent candidate is still in the root open set.

## Root-candidate state machine

Top-level (root child) SearchNodes carry one of six states,
mapped to `SearchNode.verdict` plus one new boolean:

| state       | verdict     | extra            | meaning                                                                                  | re-entry on next outer pass?           |
|-------------|-------------|------------------|------------------------------------------------------------------------------------------|----------------------------------------|
| `open`      | (no node)   | —                | candidate not yet tried this pass                                                        | yes — first try                        |
| `alive`     | `"open"`    | `partial=False`  | tried, branch saturated, subtree fully explored under depth cap, no terminal verdict     | yes — try again under tighter root.kb  |
| `dead`      | `"dead"`    | `forced=True`    | unconditional false; `(not h)` in root.kb                                                | no — `_candidates_for` filters via `_negated_facts` |
| `solution`  | `"solution"`| —                | subtree reached a goal state                                                              | no — under SOLVE the search ends here; under GAPS preserved |
| `visited`   | (no node)   | —                | candidate's saturated state-hash dedups to an existing SearchNode                        | no — DAG link only                     |
| `committed` | (no node)   | —                | candidate `h` was bubbled as a *positive* by some deeper branch; `h` is in root.kb now   | no — `_candidates_for` no longer emits |

Two adjacent gaps the new machine closes:

- **Promoted-dead → `dead`+`forced`** (root-level subset of
  S1.5a.15 T1.5a.15.4). When every subbranch of a root-child
  died and `_promote_verdicts` flips its verdict to `"dead"`,
  synthesise `(not h)` with rule provenance
  `<root-promoted-dead>` whose premise list is the union of
  the subtree's leaf-level unsat-cores (filtered to
  source-frontier facts that don't transitively depend on the
  child's own hypothesis). Without this, an outer-loop
  re-entry would replay the entire subtree. Termination of
  the outer loop rests on it. The general per-level
  promoted-dead back-prop at depths ≥ 1 is a separate stage
  (S1.5a.15 Phase 3 + T1.5a.15.4) — useful but not required
  for this stage's correctness or termination.
- **`committed`** distinguishes "h was always a given" from "h
  started as a hypothesis, then a deeper branch proved h
  unconditionally". P1.6's trace renderer needs the distinction
  to walk the proof "why is h asserted?" The marker is a thin
  bookkeeping set on `kb` (root-only): `committed_hypotheses:
  set[FactId]` populated by `_mirror_forced_positive` at root.

`visited` is already implicit via `builder.state_index` —
making it explicit as a root-state is a trace-renderer concern;
the runtime treats `state_index` hits as "no new work, link
into the DAG."

## The abort mechanism

Three implementation routes considered:

1. **`BubbleAbort` exception raised from `back_propagate` /
   `_mirror_forced_positive`, caught in the outer driver.**
   Cleanest; the abort path is one stack-unwind. Cost: catches
   need to be precise — we want to abort the *subtree*, not the
   outer driver itself. Solved by raising a typed exception
   `BubbleAbort(root_pass_id: int)` and only catching where
   `root_pass_id == current_pass`.

2. **ContextVar flag checked at every recursion boundary.**
   Polled at top of `_explore`, top of `_consume` sweep,
   between sibling iterations. More invasive (touches every
   recursion site); makes the abort cooperative rather than
   immediate. Easier to reason about with the dumper / trace.

3. **Return-value channel: every recursive call returns
   `BranchId | Aborted`.** Most type-heavy; pollutes signatures
   everywhere even when no abort fires.

Recommended: **(1)** with a narrowly typed exception. The
existing `_kb_chain_ctx` + `_consume_caches_ctx` already prove
that ContextVar plumbing works for cross-level signals; adding
exception-based control flow on top is a small surface, and
the abort is rare (≤ once per root pass per bubble).

The catch site is the new outer driver:

```python
def solve_outer(root_kb, …):
    pass_id = 0
    while True:
        try:
            pass_id += 1
            tree = _explore_root_pass(root_kb, pass_id, …)
        except BubbleAbort as e:
            if e.pass_id != pass_id:
                raise           # stale — shouldn't happen
            continue            # restart from root
        if not _bubbled_this_pass(root_kb, pass_id):
            break               # fixpoint
    _promote_verdicts(builder, root_id, mode)
    _synthesise_promoted_dead_facts(builder, root_kb)
    return _extract_verdict(...)
```

`_explore_root_pass` is the existing recursion (`_explore` →
`_consume`) modulo:
- The candidate loop at root iterates over root candidates in
  order (S1.5a.1a deterministic), skipping any in state
  `dead` / `solution` / `committed` / `visited`.
- `back_propagate` and `_mirror_forced_positive` raise
  `BubbleAbort(pass_id)` after writing — *after* so the bubbled
  facts are durable before the unwind starts.

## Termination

Each outer pass either:

- **Lands ≥ 1 bubble** → root.kb strictly grows by ≥ 1 fact
  (`(not h)` or a positive `h`). The hypothesis candidate space
  shrinks by ≥ 1 (the bubbled `h` is removed from
  `_candidates_for` either via `_negated_facts` or via being
  promoted to a given). Pass count is bounded by the size of
  the initial candidate space (≤ a few hundred for zebra-style
  puzzles).

- **Lands zero bubbles** → fixpoint; the outer loop exits.

So termination is guaranteed in `O(|root candidates|)` outer
passes, and the per-pass cost is bounded by the search-tree
size. State-hash dedup (`builder.state_index`) carries across
passes — a subtree explored in pass 1 and discarded by abort
is *not* rebuilt in pass 2 unless its parent state changed; if
the parent state changed, it gets re-explored once with the
tighter root.kb (which is the *point*).

Promoted-dead fact synthesis is what makes "lands ≥ 1 bubble"
hold for the "all children dead by exhaustion" case — without
it, a root candidate could be alive in pass 1, every subtree
dies via exhaustion (no leaf-level unconditional, just verdict
promotion), no bubble fires, and pass 2 would replay the same
work to the same dead end. With promoted-dead synthesis the
promoted `(not h)` lands at the end of the pass, qualifies as
"a bubble landed", and the next pass strictly tightens.

## Interaction with the existing bubble code

`back_propagate` (`back_prop.py:190`) and
`_mirror_forced_positive` (`solver.py:97`) already do the
write. This stage adds the *signal*: a thin wrapper that
raises after the write.

`_mirror_forced_positive` today mirrors to root **only**,
explicitly to avoid breaking ancestor `state_index` hashes
(`solver.py:953-958`). Under the eager-abort model the
intermediate ancestors are about to be discarded anyway, so
the constraint relaxes — mirror to every ancestor, matching
`back_propagate`'s negative-bubble symmetry. The state-hash
dedup concern is replaced by: discarded subtrees are not in
`state_index` (cleared on abort).

The `verdict_at` ancestor-cache invalidation in
`back_propagate` (`back_prop.py:278-279`) becomes a no-op
under eager abort — we're unwinding past those caches anyway.
Keep the call for flag-off compatibility; it's idempotent.

### T1.5a.17.3 — Promoted-dead → root fact (root-case of T1.5a.15.4)

- **a.** `_synthesise_promoted_dead_facts(builder, root_kb)`
  runs *after* `_promote_verdicts` and *before* verdict
  extraction. Walks root children; for each child whose
  verdict was flipped to `"dead"` (i.e., the leaf was not
  itself `"dead"` — the dead came from subtree exhaustion):
  - Compute the union of unsat-cores of every `"dead"` leaf in
    the subtree (the proof witnesses for each contradiction).
  - Filter the union to the source-frontier facts that *don't*
    transitively depend on the child's own hypothesis
    (`reaches_hypothesis` filtered with `own=child.hypothesis`).
  - Synthesise `(not h)` with rule provenance
    `<root-promoted-dead>` and the filtered premise list. Write
    via `_write_negation(root_kb, h, premises, ...)`.
  - Set the child's `forced=True`.
- **b.** When this synthesis writes any facts during a pass,
  set `_pass_bubbled` so the outer loop runs another pass.
  (This is what guarantees termination — exhaustion-deaths
  promote to bubbles.)
- **c.** Add a note on S1.5a.15 T1.5a.15.4 once shipped: the
  root case is owned here; the general per-level case (depths
  ≥ 1) is owned there. The two share the
  `_write_negation` helper and the `<*-promoted-dead>` rule
  prefix; provenance kind disambiguates.

## Open questions

- **Q-S1.5a.17.A — What about *intermediate*-level bubbles
  that don't change root candidates?** Under option 1 we
  abort on *any* unconditional bubble, even one that only
  affects deeper levels (e.g., a `(not h_deep)` at depth 3
  where `h_deep` was never a root candidate). The abort
  still buys us a re-saturated root.kb (if the deep negation
  cascades through the rule set, root learns more on
  re-saturation) — but pessimistically we discard a partial
  subtree to gain nothing. Option 2 (abort only on
  root-relevant bubbles) is the natural fallback if thrash
  is observed. Decision: ship option 1; measure thrash via
  outer-pass-count vs bubbled-facts-count; downgrade to
  option 2 only if the ratio is bad.
- **Q-S1.5a.17.B — Partial-subtree preservation on abort?**
  Today's plan is to discard the partial subtree on abort
  (state-hash dedup recovers most of the work on re-entry).
  An alternative is to mark the partial subtree as
  `"incomplete"` and extend it on re-entry — saves the
  redundant re-saturation but breaks the invariant that every
  SearchNode has a final verdict. The user's framing ("re-enter
  all non-dead hyps") implies discard. Defer alternative
  until measurement says it matters.
- **Q-S1.5a.17.C — Does this change `_promote_verdicts`?**
  The promotion algorithm itself is unchanged — but it now
  runs *after* the outer loop fixpoint, on a tree where every
  root child has a stable subtree. The
  `_synthesise_promoted_dead_facts` pass runs *between*
  promotion and verdict extraction. Order matters: promotion
  must run first to identify which root children are dead by
  exhaustion; synthesis must run second to mint the facts.
  Document the order explicitly.
- **Q-S1.5a.17.D — Cond-dead at depth ≥ 1 — still cached
  per-`_consume`?** Yes. Cond-dead only exists at depth ≥ 1
  (root deaths are always unconditional); the per-`_consume`
  `verdict_at` cache is fine as-is. Cross-pass survival of
  cond-dead nodes is via the SearchTree itself (the dead
  SearchNode allocated inline is durable; `_descend`'s
  `exclude_keys` set is per-`_consume` and naturally
  re-derived on re-entry from the children list).
- **Q-S1.5a.17.E — Symmetry with hypothesis scoring
  ([S1.5a.7](s1.5a.7_hypgen_scoring_branch_info.md))?**
  Eager abort plus scoring composes naturally — the outer
  loop iterates root candidates in score order, and bubbles
  re-prioritise on each pass (scores depend on root.kb,
  which just changed). Worth noting because the scoring
  stage's perf measurement should re-baseline under eager
  abort if both land.
