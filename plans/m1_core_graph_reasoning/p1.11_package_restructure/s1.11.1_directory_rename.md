## Context

The package directory is `ein.py/src/ein_bot/`. The project's
user-facing name is `ein`; "ein-bot" was a placeholder. The
outer directory `ein.py/` is also historical (from when the
repo's only Python artefact was a single file). This stage
renames the package directory and updates the build config;
imports + content edits ship in S1.11.2.

## Tasks

### Task T1.11.1.1 — Package rename

- **a.** `git mv ein.py/src/ein_bot ein.py/src/ein`.
- **b.** Edit `ein.py/pyproject.toml`: change `[project]
  name = "ein-bot"` → `"ein"`; update `[project.scripts]`
  entries that reference `ein_bot:cli` → `ein:cli` (defer the
  entry-point internals to S1.11.4); update `[tool.*]` paths.
- **c.** Re-run `pip install -e ein.py/` (verify it installs
  cleanly under the new name).

### Task T1.11.1.2 — Outer directory rename (open question)

- **a.** Open Q-S1.11.1.A — rename `ein.py/` → `ein/` or leave
  as-is. The outer `ein.py/` name is a footgun ("a Python
  file?"), but the rename touches every `cd ein.py` or
  `ein.py/tests/` reference across docs + scripts.
- **b.** Decision recorded in this stage at start. If
  proceeding: `git mv ein.py ein` (yes, the outer dir becomes
  `ein/` containing `ein/src/ein/` — possibly confusing; an
  alternative is a sibling-named rename like `ein-py/` to keep
  the implementation marker).

### Task T1.11.1.3 — Smoke test

- **a.** `from ein import …` (or `from ein.kb import …` etc.)
  works in a Python REPL.
- **b.** `pytest ein.py/tests/` (or new path) still passes.
