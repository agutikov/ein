## Goal

Three concrete fixtures + their pinned-behaviour tests:

1. `examples/lattice/01_subset_pruned.ein` — exercises
   Apriori pruning (a 3-element commitment is pruned at
   generation because a 2-subset is dead).
2. `examples/lattice/02_genuine_3set_death.ein` — exercises
   the genuine-combinatorial-core case (a 3-element
   commitment dies but no 2-subset of it is in D_2;
   Apriori cannot prune; the lattice still discovers the
   death at layer 3).
3. **State-hash-collision verification** on
   `examples/zebra2-hints.ein` (or a constructed minimal
   fixture if zebra2-hints doesn't trigger the collision).

The fixtures are *pinned* — pytest reads each fixture's
expected verdict + dump-shape assertions; future P1.5b
changes can't silently drift the behaviour.

## Fixture design

### `examples/lattice/01_subset_pruned.ein`

A small puzzle with 4 alive hypotheses `{h₁, h₂, h₃, h₄}`
where:
- `{h₁, h₂}` saturates to contradiction → emits nogood
  `frozenset({h₁_id, h₂_id})`.
- All other 2-subsets alive.
- Layer-3 candidate `{h₁, h₂, h₃}` should be pruned at
  generation because `{h₁, h₂} ∈ D_2`.
- Layer-3 candidate `{h₁, h₃, h₄}` should be explored
  normally.

Comment block in the .ein file explains the didactic
purpose:

```lisp
;;; P1.5b lattice fixture — Apriori subset pruning.
;;;
;;; Demonstrates: a layer-3 candidate {h₁, h₂, h₃} is
;;; pruned at generation time because its 2-subset
;;; {h₁, h₂} dies at layer 2. The lattice's
;;; matches_any_nogood filter catches it BEFORE the
;;; expensive try_set fork+saturate.
;;;
;;; Expected: verdict Ambiguity (or Solution, depending
;;; on whether h₃, h₄ enterings derive the goal); the
;;; per-set dump shows {h₁, h₂, h₃} marked as
;;; pruned-at-generation (dashed-grey shape in
;;; diagrams/lattice_4hyps_dedup.svg's analog).
```

Concrete rules + facts to be written; aim for a puzzle
that's testable in <100 ms and uses ~10 facts.

### `examples/lattice/02_genuine_3set_death.ein`

A puzzle where a 3-element commitment {h₁, h₂, h₃} dies
but **no 2-subset** of it is in `D_2`. The death only
surfaces when all three hypotheses are committed together —
a genuine combinatorial-core case Apriori cannot prune.

This is the case Q1.5b.2.d flagged as "unavoidable —
combinatorial core that no dedup eliminates".

Comment block:

```lisp
;;; P1.5b lattice fixture — genuine 3-element death.
;;;
;;; A puzzle where no 2-subset of {h₁, h₂, h₃} is in D_2
;;; (every 2-subset is alive), but the 3-set saturates to
;;; a contradiction. Apriori-gen produces the candidate;
;;; matches_any_nogood lets it through; try_set fires;
;;; ContradictionDetector hits.
;;;
;;; Expected: verdict ... (puzzle-specific); the per-set
;;; dump shows {h₁, h₂, h₃} in D_3 with non-empty
;;; unsat_core; the learned clause has size 3.
```

### State-hash collision verification

Run `bench_dag.py examples/zebra2-hints.ein --max-set-size 3
--dump-states ./dump/zebra2-hints-lattice` and inspect the
dump for at least one SetNode with `len(labels) > 1`.

If found:
- Pin the specific labels + canonical_set in
  `test_state_hash_collision_observed_on_zebra2_hints`.
- Document the rule mechanism behind the collision in
  this stage doc's "Findings" subsection.

If not found:
- Construct a minimal fixture
  `examples/lattice/03_state_hash_collision.ein` per the
  candidate from Q1.5b.4.c
  (3 hypotheses where rule under `h₂` makes `h₁` and `h₃`
  derivable from each other → `{h₁,h₂}` and `{h₂,h₃}`
  saturate to the same kb).
- Pin its behaviour as above.
