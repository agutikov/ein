## Motivation — the S1.5b.5 ship-note gap

S1.5b.5 shipped the monotonic backbone with a documented gap:
`branching/03` and zebra2 returned `Ambiguity` instead of
`Solution`. The reason was structural:

1. `commitment._is_unconditional` flags any fact whose chain
   reaches a committed hypothesis as conditional (sound — `h`
   might be wrong in general).
2. When `alive` shrinks to a singleton `{h_unique}`, `h_unique`
   IS in every surviving commitment — but the engine never
   merges it because the conditionality check is per-derivation,
   not per-alive-set-shape.
3. Tree-side returns `Solution` because its `Solution.kb` is
   the *surviving fork's* kb (which has the committed hypothesis
   baked in). Monotonic returns `root.kb`, which lacks it.

The fix is the classical SAT-solver **unit propagation** /
**forced-positive promotion**: when alive collapses to one
candidate, that candidate is *provably true* (every other
slot-mate has been refuted), so promote it to a root fact and
re-saturate.

## Algorithm

```
_promote_forced_positives(root_kb, alive, stats, mode):
    while len(alive) == 1:
        (rn, args) = single member of alive
        promoted = Fact(rn, args,
                        layer=REASONING,
                        provenance=from_rule("<forced-positive>", ()))
        root_kb.add_fact(promoted)
        root_kb._index_fact(promoted)
        Saturator(root_kb).saturate()    # re-saturate (cascade)
        if ContradictionDetector(root_kb).detect():
            return alive, Contradiction(...)   # promotion surfaced a conflict
        if is_solved(root_kb, mode):
            return alive, Solution(root_kb)
        alive = _compute_alive(root_kb)   # might shrink further
    return alive, None
```

Called at two integration points in `monotonic_solve`:

1. **After Phase 1's initial `_compute_alive`.** Covers the
   case where root's first saturation already determines a
   unique survivor (zebra2's case after S1.5a.19's d=0
   negative-completion rules).
2. **After each Phase 2 re-saturation that follows a merge.**
   Covers the case where merging unconditional facts shrinks
   alive to a singleton (cascade scenarios).

### Provenance choice for the promoted fact

`Provenance.from_rule("<forced-positive>", premises_raw=())`:

- `kind="rule"` (not "hypothesis") — `commitment._reaches_commitment`
  walks through rule-kind facts via their `premises_raw`. With
  empty premises, the walk terminates without finding any
  commitment element, so future try_commitment_set chains that
  pass through the promoted fact stay unconditional. (If we
  used `from_hypothesis`, every downstream derivation would be
  flagged conditional — defeating the point.)
- Empty `premises_raw` — the promotion is an engine-level
  meta-observation, not a rule firing on specific premises.
  P1.6's trace renderer can label these as
  "<forced-positive>" by checking the rule name.

### Symmetric canonicalisation in `_compute_alive`

Hypgen emits BOTH orderings of symmetric-relation pairs
deliberately (so each can be scored independently). For
forced-positive detection, the pair is *one* hypothesis — the
naive `len(alive) == 1` check never fires.

Fix: `_compute_alive` now looks up
`kb._facts_by_relation.get("symmetric", ())` for declared
symmetric relations and canonicalises 2-arg pairs to
`args[0] <= args[1]` form before adding to the alive set.
Without this, `alive = {(co-located H5 White),
(co-located White H5)}` would prevent promotion on
branching/03.

Generalisation to other equivalence classes (transitive closure
paths, rule-derived implications, …) is out of scope here. Add
canonicalisations as new puzzles surface them.

## Soundness

`alive = {h}` after symmetric canonicalisation means every
other slot-mate has been individually refuted — either:

- back-prop wrote `(not h_other)` at root (currently not
  implemented in monotonic backbone — S1.5b.6 adds the nogood
  variant);
- hypgen's lookahead filtered `h_other` (one-step death);
- a derived `(not h_other)` rule (sibling-exclusive,
  type-exclusivity, etc.) is in root after saturation.

Combined with the puzzle's slot-exclusivity constraint
(typically a `functional` or sibling-exclusive rule), `h` must
be true in every model.

The post-promotion `ContradictionDetector` catches any
inconsistency that the promotion + further saturation surfaces.
The cascade loop terminates: each iteration adds a fact
(monotone), and alive is bounded by the puzzle's finite
hypothesis space.

### Known limitation — incomplete for joint-only contradictions

The backbone (with or without forced-positive) is still
**incomplete** for puzzles whose contradiction surfaces only
at commitment size ≥ 2 AND whose alive set has size > 1 (so
forced-positive never fires). Such puzzles report Ambiguity
instead of Contradiction. Tracked under the same rubric as
S1.5b.5's general incompleteness — closes with lattice's full
enumeration (S1.5b.21+) or monotonic CDCL (S1.5b.6).

## Zebra2 — the M1 acceptance puzzle

```
$ .venv/bin/python ein.py/demo/bench_monotonic.py examples/zebra2.ein --max-set-size 2
verdict           Solution
goal bindings (from query :goal):
  h_water=House-1, h_zebra=House-5
stats
  enterings_total    1
  enterings_alive    1
  facts_merged       30
  forced_positives   0
  saturate_count     2
  layers_explored    1
  wall               ~5 s (CPython)
```

Monotonic backbone + forced-positive solves zebra2 in ~5s on
CPython (vs tree's ~50s). Tree's tree-node baseline was 32
nodes; monotonic does it with 1 try_commitment_set and one
post-merge re-saturation. The S1.5a.19 d=0 negative-completion
rules make the layer-1 alive set determined; the merge brings
30 unconditional facts into root in one shot; goal satisfied.

The monotonic engine is now a viable candidate for the
**S1.5b.8 Zebra acceptance gate** — possibly *replacing* the
tree solver as the M1 SOLVE acceptance arm.
