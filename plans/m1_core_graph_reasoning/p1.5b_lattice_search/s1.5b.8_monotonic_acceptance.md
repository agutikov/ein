## Goal

Run `bench_monotonic.py examples/zebra2.ein` and verify it:

1. Returns `Solution`.
2. Carries the correct goal bindings (`?h_water = House-1`,
   `?h_zebra = House-5`).
3. Completes within the time budget (< 60s on the laptop
   reference; ideally < 5s if S1.5a.19 has shipped).

If yes → M1 SOLVE acceptance #2 is met via monotonic; the
lattice's S1.5b.20+ work becomes optional for M1 and can
land in P1.5b at its own pace.

If no → diagnose using the dump (S1.5b.7); the dump's
`00_timeline.jsonl` shows which layer the search bogged
down at, the `summary.json` shows the cumulative stats, and
the per-layer `.ein` snapshots show what facts monotonic
managed to accumulate at root.

## Reference run

The laptop reference (from S1.5a.13 + the
[2026-05-25 zebra2 dump](../../../dump/zebra2/summary.json)):

- machine: user's laptop (the one running
  `bench_solve_pypy.sh` for the dump).
- existing tree-side baseline: 568 nodes, 472 dead, 77 open,
  0 solutions, 49.4 s (BEFORE S1.5a.19).
- target after S1.5b.6 + (ideally) S1.5a.19:
  - **Solution** verdict.
  - bindings: `?h_water = House-1, ?h_zebra = House-5`.
  - tree_nodes (monotonic equivalent =
    enterings_total): target ≤ 200.
  - elapsed: < 60 s without S1.5a.19; < 5 s with.

The "target ≤ 200 enterings" is a rough ceiling based on
the NL trace's complexity: 4 d=1 refutations + ~10
unconditional-fact propagations = ~14 enterings if monotonic
were as efficient as NL. Allow 10× for the engine's slower
inference path.

## Tasks

### T1.5b.8.1 — Cold-run benchmark

Run:

```sh
python ein.py/demo/bench_monotonic.py examples/zebra2.ein \
    --max-set-size 5 \
    --dump-states ./dump/zebra2-monotonic \
    --verbose 2>&1 | tee dump/zebra2-monotonic/console.log
```

Record:
- Verdict
- Bindings (if Solution)
- Elapsed seconds (from `summary.json`)
- Stats: `enterings_total`, `enterings_alive`,
  `enterings_dead_pre + dead_post`, `facts_merged`,
  `saturate_count`, `nogoods_emitted`, `nogoods_subsumed`,
  `layers_explored`.

### T1.5b.8.2 — Verify the verdict

If `Solution`:
- Assert bindings match the expected
  (`?h_water = House-1, ?h_zebra = House-5`).
- Inspect `summary.json["stats"]` — sanity-check counts.
- File the result as the **M1 SOLVE baseline** in
  [P1.5a S1.5a.13](../p1.5a_zebra_solution/s1.5a.13_acceptance_zebra2_solves.md)'s
  measurement section (cross-engine note).

If `Ambiguity`:
- Likely `max_set_size` too small or d=0 closure missing
  (S1.5a.19 not landed). Increase to `--max-set-size 7` and
  re-run.
- Inspect `layers/layer_<n>_post.ein` for the last layer
  before termination — what facts did monotonic accumulate
  at root? Did it derive `(color-loc Yellow House-1)`?
  `(drink-loc Water House-1)`?
- File a diagnostic note; queue S1.5a.19 if not yet
  shipped.

If `Contradiction`:
- Indicates a real bug (Zebra is solvable). Likely:
  - over-eager nogood emit on a recoverable death,
  - wrong is_unconditional check (chain reaching commitment
    when it shouldn't),
  - `_compute_alive` shrinking too aggressively after a
    spurious singleton-death.
- The dump's `00_timeline.jsonl` shows the first failing
  layer; the per-layer .ein shows what facts were present.
- Diagnose + file as P1.5b bug. Worst-case: open a new
  sub-stage S1.5b.8a for the fix.

### T1.5b.8.3 — Verify the budget

If `Solution` but elapsed > 60 s on the laptop reference,
something is genuinely slow. Possible culprits:
- Saturator running too many times (each
  `root_changed` triggers a full re-saturate). Confirm via
  `saturate_count` in stats; the bound is roughly
  `enterings_alive + 1`, but if it's much higher, there's
  a re-saturation loop.
- Apriori-gen producing too many candidates per layer. The
  filter is the only safety net; check
  `nogoods_emitted + nogoods_subsumed` count + the per-layer
  `entering` event counts.
- Per-set `try_commitment_set` saturation cost growing. The commitment-set
  primitive saturates once per set; for k=4 commitments on
  zebra2's 89-alive base, the per-call cost is
  ~500 firings (per the [2026-05-25 dump]
  (../../../dump/zebra2/summary.json)).

Tune knobs in the order:
1. Confirm CDCL is filtering aggressively
   (`enterings_total` doesn't exceed `C(89, ≤ max_set_size)`
   by much).
2. Confirm root re-saturation isn't redundant
   (single saturate per merged-fact batch, not per fact).
3. If still > budget, file a perf sub-stage
   (S1.5b.11–.19 — the reserved slots).
