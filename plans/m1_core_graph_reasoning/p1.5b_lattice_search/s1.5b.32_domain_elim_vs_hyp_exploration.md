## The two pathways

Given an *exactly-one* constraint over a type T at a slot —
e.g. "exactly one of {Red, Blue, Green, Yellow, Ivory} sits
at H_1" — the engine has two ways to land on the positive:

### Pathway A — saturation rule with `forall`

A stdlib rule shape:

```lisp
(rule domain-elimination (?R)
  :match  (and (relation ?R ?A ?B)
               (is-a ?h ?B)
               (is-a ?v ?A)
               (forall ?v_other (is-a ?v_other ?A)
                 (or (eq ?v_other ?v)
                     (not (?R ?v_other ?h)))))
  :assert (?R ?v ?h)
  :why    "domain-elim: every alternative for {?h} excluded → {?v}."
  :priority 400)
```

(The `forall` desugars to `(absent (and (is-a ?v_other ?A)
(neq ?v_other ?v) (absent (not (?R ?v_other ?h)))))` per
[NAF semantics](../../../docs/kernel/inference/README.md#naf-semantics--fire-time-re-evaluation-s15a1).)

Fires at d=0 once every alternative is explicitly negated.
Single firing, single derivation, single named rule in the
trace.

### Pathway B — explicit hypothesis exploration

The hypothesis loop forks on each candidate `(?R ?v_i ?h)`:

- Branches `b1: Red@H_1, b2: Blue@H_1, …` enumerate the type.
- Each branch saturates; four contradict (refuted by the
  puzzle's other constraints).
- The fifth branch lives; its positive `(R V H_1)` is the
  answer.

Lands at d=1 (or deeper for cascaded eliminations). N firings
across N branches (1 alive + N-1 refutations).

## Comparison axes

### Result + correctness

**Equivalent.** Under monotone saturation, both pathways
derive the same positive fact when the puzzle is uniquely
solvable. Soundness is preserved by the rule set — pathway B
ground-truths each contradiction via refutation, pathway A
*assumes* every negative is correctly derived.

Edge case: if a puzzle is unsolvable (overconstrained),
pathway B exposes it by every branch dying; pathway A exposes
it by the negative-completion failing to produce all N
alternatives (one of them stays open). Same satisfiability
verdict either way, different shape.

### Performance

**Pathway A is cheaper by a factor of |alternatives|.** B
pays N × saturation + N × contradiction-detect; A pays one
saturation pass that consumes the already-derived negatives.
On zebra2 this is the difference between the pre-S1.5a.19
568-node tree and the post-S1.5a.19 32-node tree — same
puzzle, same verdict, the rule does the elimination work
the branches used to do.

### Soundness coupling to the negative-completion rules

**Pathway A depends on the rule library being complete on
negatives.** It only fires when *every* alternative is
explicitly in `_negated_facts`. S1.5a.19 closed this for
zebra2 by adding six d=0 negative-completion rules
(`functional-negative`, `injective-negative`,
`co-located-negative`, `adjacent-via-endpoint-{fwd,bwd}`,
`adjacent-via-{fwd,bwd}-negative`). Without those rules,
pathway A is dormant.

**Pathway B is more robust.** It doesn't need the negatives
to exist in advance — refutation produces them as it goes.
A puzzle with an incomplete rule library still solves via B
(slowly), where A would silently fail to fire and the verdict
would degrade to Ambiguity / Open.

### Trace fidelity

**Pathway A matches the NL Zebra walkthrough.** The
walkthrough says *"Therefore the first house is yellow"* —
one named move, one citation of `(domain-elimination
color-loc)`. Pathway B produces N events ("hypothesise Red,
contradiction; hypothesise Blue, contradiction; …; therefore
Yellow") which the trace renderer would have to collapse to
match NL.

### Engine machinery used

| pathway | machinery touched |
|---------|---------------------|
| A | saturator, matcher, AbsentGuard (fire-time re-eval per S1.5a.1), the `domain-elimination` rule + its `forall` desugar |
| B | hypgen, `try_branch`, `back_propagate`, `ContradictionDetector`, per-fork saturator, `_consume`'s sweep / lattice's per-set integrate |

So A is "all sat, no branching"; B is "all branching, sat
inside each branch". Two engine surfaces; both shipped today.

## The question P1.5b owes an answer to

Under the **monotonic** + **lattice** engines, does the
A-vs-B distinction stay the same as on the tree-side, or does
something shift?

Specific sub-questions:

1. **Does pathway B have a natural per-set analog?** Each
   commitment set IS a hypothesis tuple; the lattice already
   exhausts the powerset, so B-style enumeration is structural.
   A-style derivation happens *inside* each set's per-set
   saturate. Is there double-counting (the lattice's outer
   loop already enumerates what A's rule would derive)?
2. **Does A still fire under commitment-set semantics?** Pathway
   A needs the negative-completion to have run; under the
   monotonic engine's apriori prefix-join, the negative-
   completion happens at root + per-layer integrate. Verify
   that A's `forall`-rule observes the same negatives the
   tree-side saturator observes.
3. **Is B's per-set saturation wasted work when A would have
   fired?** If the puzzle has the negative-completion rules
   in scope, A fires at root and the lattice never needs to
   visit the corresponding commitment sets. Measure the
   set-count delta with A enabled vs disabled.
4. **Should `domain-elimination` be a kernel feature, a
   stdlib rule, or puzzle-author boilerplate?** Today it's
   the latter — zebra2.ein declares it. Composes with
   [F5 § Kernel minimisation](../../followups/f5_rules_as_data.md#kernel-minimisation--which-inference-features-belong-in-ein-lang-vs-kernel-code)
   — `domain-elimination` is a strong candidate for "stays
   in ein-lang because the rule shape carries the
   intent better than engine code would."

### T1.5b.32.2 — Constructed-fixture measurement

A minimal puzzle (3–5 atoms per type, one `exactly-one`
constraint) where both pathways produce the same answer.
Run with:

- **A only** — `domain-elimination` rule active, hypgen
  disabled (or `:hypothesis-relations` set to nothing).
- **B only** — `domain-elimination` rule removed, hypgen
  default.
- **A + B** — both active (current zebra2 shape).

Tabulate tree-node count, set-count (lattice), wall time,
firings. The A+B row tells us how much *redundancy* the
two pathways have when both are present.

### T1.5b.32.3 — Lattice-side audit

Open the lattice's per-set saturator and confirm that A's
`forall`-rule fires inside the per-set saturate when the
per-set kb has the negatives the rule needs. If not, the
commitment-set semantics need adjustment (probably an explicit
re-saturate-after-integrate step beyond what
[S1.5b.24](s1.5b.24_lattice_integrate.md) ships).

### T1.5b.32.4 — Recommendation

Based on T1-T3: pick a default for the engines' interaction
with `domain-elimination`-shaped rules. Three candidates:

- **Pathway A as default, B as fallback.** Engines treat
  domain-elim-style rules as first-class; the lattice prunes
  the set-tree aggressively when A would have answered.
- **Keep both, no special-casing.** Current shape — the
  engines are pathway-agnostic; redundancy is the user's
  cost. Simplest.
- **B with optimistic A skip.** The lattice consults A's
  pre-conditions before forking; if A would have fired, skip
  the branch. Hybrid; needs a cheap pre-fork lookahead.

Output: a one-paragraph recommendation in
[S1.5b.0](s1.5b.0_decisions_log.md) decisions log + the
measurement table as evidence.

## Open questions

- **Q-S1.5b.32.A — How does the comparison change when the
  puzzle's negative-completion is *incomplete*?** Pathway A
  silently fails to fire; pathway B carries the load. Should
  the engine warn when A's preconditions are unmet
  (composes with the dropped [S1.5a.8 / S1.7.4](../p1.7_bootstrapping_zebra/s1.7.4_naf_dependency_map.md)
  static NAF dependency map)?
- **Q-S1.5b.32.B — Trace renderer's collapse rule.** When the
  trace renderer (P1.6 S1.6.4) sees N refutations followed
  by 1 survivor (pathway B), should it render as one
  `by-elimination` move (NL-matching) or N events (literal)?
  Trade-off: trace fidelity to NL vs trace fidelity to
  engine. P1.6 / S1.6.5 territory; flag here so the trace
  pass knows the equivalence exists.
- **Q-S1.5b.32.C — Generalisation beyond domain-elim.** The
  same A-vs-B duality applies to other "forall ⇒ ∃!"
  rules: `range-elimination`, `no-room-left`, possibly
  `cardinality-bound`-style constraints. Catalogue once
  T32.4's recommendation lands.
