## Goal

Confirm that monotonic produces the same `SOLVE`-mode
verdict + bindings as the tree-side `bench_solve.py` on every
fixture under `examples/branching/`. Pin the results in a
parametrised pytest test so future changes can't drift.

The 11 fixtures (per the [2026-05-25 file listing](../../../examples/branching/)):

```
01_saturate_only.ein
02_one_dead_one_alive.ein
03_five_hyps_one_alive.ein
04_two_levels.ein
05_mini_zebra.ein
06_lookahead_on.ein
07_lookahead_off.ein
08_hypothesis_relation_whitelist.ein
09_hrule.ein
10_backprop_on.ein
11_backprop_off.ein
```

For each, run `bench_solve.py FIXTURE.ein` and
`bench_monotonic.py FIXTURE.ein --max-set-size N`; assert:

1. **Verdict type** matches (`Solution` ≡ `Solution`, etc.).
2. **Goal bindings** (if Solution) match exactly.
3. **Verdict consistency** under the constraints
   (Q1.5b.7's monotonic-vs-lattice equivalence): if tree
   returns `Ambiguity` because of `open` leaves at the
   depth cap, monotonic should also return `Ambiguity`
   if its own `max_set_size` mirrors the tree's
   `max_depth`.

Items NOT verified at this stage:
- Per-branch refutation order — monotonic doesn't expose
  the dead-set audit the same way tree does.
- Per-firing trace — different engines produce different
  firing sequences even when verdicts match.

### T1.5b.9.2 — Author the parity test

`tests/inference/monotonic/test_monotonic_parity.py`:

```python
"""S1.5b.9 — monotonic / tree parity on branching demos."""
from pathlib import Path

import pytest

from ein_bot.inference.monotonic import monotonic_solve
from ein_bot.inference.tree.solver import solve as tree_solve
from ein_bot.inference.tree.solver import Solution, Ambiguity, Contradiction
from ein_bot.ir import parse
from ein_bot.kb.store import KnowledgeBase

REPO = Path(__file__).resolve().parents[3]

# Per-fixture expected verdict type + max_set_size needed.
# Drawn from `parity_baselines.md` (S1.5b.9.1).
FIXTURES = [
    ("01_saturate_only.ein",                Solution,    0),
    ("02_one_dead_one_alive.ein",           Solution,    1),
    ("03_five_hyps_one_alive.ein",          Solution,    1),
    ("04_two_levels.ein",                   Solution,    2),
    ("05_mini_zebra.ein",                   Solution,    3),
    ("06_lookahead_on.ein",                 Solution,    1),
    ("07_lookahead_off.ein",                Solution,    1),
    ("08_hypothesis_relation_whitelist.ein", Solution,   1),
    ("09_hrule.ein",                        Solution,    1),
    ("10_backprop_on.ein",                  Solution,    2),
    ("11_backprop_off.ein",                 Solution,    2),
]

@pytest.mark.parametrize(
    "fixture,expected_verdict,max_set_size",
    FIXTURES,
    ids=[f for f, _, _ in FIXTURES],
)
def test_monotonic_matches_tree(
    fixture, expected_verdict, max_set_size,
):
    path = REPO / "examples" / "branching" / fixture
    text = path.read_text()

    # Tree baseline
    kb_tree = KnowledgeBase.from_ir(parse(text))
    v_tree = tree_solve(kb_tree, max_depth=max_set_size)
    assert isinstance(v_tree, expected_verdict)

    # Monotonic
    kb_mono = KnowledgeBase.from_ir(parse(text))
    v_mono = monotonic_solve(kb_mono, max_set_size=max_set_size)
    assert isinstance(v_mono, expected_verdict)

    # Verdict-content equality where applicable.
    if isinstance(v_tree, Solution):
        # Goal bindings should match.
        tree_bindings = _extract_bindings(v_tree, kb_tree)
        mono_bindings = _extract_bindings(v_mono, kb_mono)
        assert tree_bindings == mono_bindings, (
            f"fixture {fixture}: tree {tree_bindings}, "
            f"monotonic {mono_bindings}"
        )

def _extract_bindings(verdict, kb):
    """Return the dict of goal-var bindings from a Solution.
    For Ambiguity/Contradiction, return None."""
    if not isinstance(verdict, Solution):
        return None
    # Look up the goal facts in kb; map ?var → bound value
    # via the verdict.kb. Implementation per existing
    # bench_solve.py's bindings extraction.
    from ein_bot.inference.tree.solver import _extract_goal_bindings
    return _extract_goal_bindings(verdict.kb)
```

If `_extract_goal_bindings` doesn't exist as a public

place it in `inference/tree/solver.py` (or common
`inference/solver_helpers.py` if generality is wanted).

## What if a fixture genuinely diverges

For SOLVE-mode uniquely-solvable demos, monotonic should
match tree. If a fixture is genuinely ambiguous (multiple
satisfying commitments — a GAPS-shaped puzzle), monotonic
returns the *first* whereas tree returns Ambiguity with
all branches. The fixture should be marked as expected
divergence:

```python
@pytest.mark.parametrize(...)
def test_monotonic_matches_tree(...):
    if fixture == "ambiguous_demo.ein":
        # Monotonic returns Solution (the first satisfying
        # commitment); tree returns Ambiguity.
        # This divergence is BY DESIGN — Q1.5b.7.
        return  # skip strict equality check
    ...
```

Document the expected divergences in `parity_baselines.md`
+ this stage's "Known divergences" subsection (if any
appear).
