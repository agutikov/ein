Confirm that monotonic produces the same `SOLVE`-mode
verdict + bindings as the tree-side `bench_solve.py` on every
fixture under `examples/branching/`. Pin the results in a
parametrised pytest test so future changes can't drift.

For each, run `bench_solve.py FIXTURE.ein` and
`bench_monotonic.py FIXTURE.ein --max-set-size N`; assert:

1. **Verdict type** matches (`Solution` ≡ `Solution`, etc.).
2. **Goal bindings** (if Solution) match exactly.
3. **Verdict consistency** under the constraints
   (Q1.5b.7's monotonic-vs-lattice equivalence): if tree

### T1.5b.9.2 — Author the parity test

`tests/inference/monotonic/test_monotonic_parity.py`:

```python
"""S1.5b.9 — monotonic / tree parity on branching demos."""
from pathlib import Path

import pytest

from ein.inference.monotonic import monotonic_solve
from ein.inference.tree.solver import solve as tree_solve
from ein.inference.tree.solver import Solution, Ambiguity, Contradiction
from ein.ir import parse
from ein.kb.store import KnowledgeBase

REPO = Path(__file__).resolve().parents[3]

# Per-fixture expected verdict type + max_set_size needed.
# Drawn from `parity_baselines.md` (S1.5b.9.1).
FIXTURES = [
    ("01_saturate_only.ein",                Solution,    0),
    ("02_one_dead_one_alive.ein",           Solution,    1),
    ("03_five_hyps_one_alive.ein",          Solution,    1),
    ("04_two_levels.ein",                   Solution,    2),
    ("05_mini_zebra.ein",                   Solution,    3),
    ("06_lookahead_on.ein",                 Solution,    1),
    ("07_lookahead_off.ein",                Solution,    1),
    ("08_hypothesis_relation_whitelist.ein", Solution,   1),
    ("09_hrule.ein",                        Solution,    1),
    ("10_backprop_on.ein",                  Solution,    2),
    ("11_backprop_off.ein",                 Solution,    2),
]

@pytest.mark.parametrize(
    "fixture,expected_verdict,max_set_size",
    FIXTURES,
    ids=[f for f, _, _ in FIXTURES],
)
def test_monotonic_matches_tree(
    fixture, expected_verdict, max_set_size,
):
    path = REPO / "examples" / "branching" / fixture
    text = path.read_text()

    # Tree baseline
    kb_tree = KnowledgeBase.from_ir(parse(text))
    v_tree = tree_solve(kb_tree, max_depth=max_set_size)
    assert isinstance(v_tree, expected_verdict)

    # Monotonic
    kb_mono = KnowledgeBase.from_ir(parse(text))
    v_mono = monotonic_solve(kb_mono, max_set_size=max_set_size)
    assert isinstance(v_mono, expected_verdict)

    # Verdict-content equality where applicable.
    if isinstance(v_tree, Solution):
        # Goal bindings should match.
        tree_bindings = _extract_bindings(v_tree, kb_tree)
        mono_bindings = _extract_bindings(v_mono, kb_mono)
        assert tree_bindings == mono_bindings, (
            f"fixture {fixture}: tree {tree_bindings}, "
            f"monotonic {mono_bindings}"
        )

def _extract_bindings(verdict, kb):
    """Return the dict of goal-var bindings from a Solution.
    For Ambiguity/Contradiction, return None."""
    if not isinstance(verdict, Solution):
        return None
    # Look up the goal facts in kb; map ?var → bound value
    # via the verdict.kb. Implementation per existing
    # bench_solve.py's bindings extraction.
    from ein.inference.tree.solver import _extract_goal_bindings
    return _extract_goal_bindings(verdict.kb)
```

If `_extract_goal_bindings` doesn't exist as a public
helper, extract the logic from `bench_solve.py`'s

place it in `inference/tree/solver.py` (or common
`inference/solver_helpers.py` if generality is wanted).

## What if a fixture genuinely diverges

For SOLVE-mode uniquely-solvable demos, monotonic should
match tree. If a fixture is genuinely ambiguous (multiple
satisfying commitments — a GAPS-shaped puzzle), monotonic
returns the *first* whereas tree returns Ambiguity with
all branches. The fixture should be marked as expected
divergence:

```python
@pytest.mark.parametrize(...)
def test_monotonic_matches_tree(...):
    if fixture == "ambiguous_demo.ein":
        # Monotonic returns Solution (the first satisfying
        # commitment); tree returns Ambiguity.
        # This divergence is BY DESIGN — Q1.5b.7.
        return  # skip strict equality check
    ...
```

Document the expected divergences in `parity_baselines.md`
+ this stage's "Known divergences" subsection (if any
appear).

### Engine bug found + fixed (in-stage)

The first measurement pass showed 4 divergences: monotonic
returned Ambiguity on `05_mini_zebra`, `07_lookahead_off`,
`10_backprop_on`, `11_backprop_off` where tree returned Solution.

**Diagnosis** (via `bench_monotonic --verbose`): on these
fixtures the goal directly references hypothesis facts (e.g.
`(co-located ?n Coffee)` matches `(co-located Bob Coffee)`
which is the committed hypothesis itself). The hypothesis
facts live in the **fork**, not in root — and the existing
`monotonic_solve` only checked `is_solved(root_kb, mode)`.
Result: the fork satisfied the goal but the engine ignored
the signal and ran to layer-exhaustion.

**Fix** (`inference/monotonic/solver.py`): on every alive
entering, check `is_solved(result.kb, mode)` against the
fork kb. On True, return `Solution(kb=result.kb)` — carrying
the fork's hypothesis-and-derivations context. This is the
behaviour
[`algorithm_layer_n.md` §3d.vii](algorithm_layer_n.md) already
prescribed; my S1.5b.5 backbone implementation used the wrong
kb. The fork-side check fires before the unconditional-facts
merge, so successful early termination skips the work of
copying those facts into root only to immediately return.

After the fix, all 4 divergences close — monotonic finds the
exact bindings tree finds (Bob/Dog, H5, Dave) at the layer
where the productive commitment lands.

### Q1.5b.7 divergence — by design, now visible

The fix exposed one expected divergence: `04_two_levels.ein`
has two satisfying branches. Tree finds both (Ambiguity).
Monotonic terminates on the lex-first one — Blue↔H3 — and
returns Solution. Both verdicts are correct under their
respective contracts (SOLVE mode for monotonic is "give me
**a** solution"; tree's Ambiguity captures the depth-cap
shape).

The parity test treats this as an asserted divergence rather
than an `xfail`: the FIXTURES row carries both engines'
expected verdict types explicitly. `parity_baselines.md`
documents the divergence with cross-references to Q1.5b.7
+ the algorithm spec.

### Test churn from the fix

The fork-side `is_solved` check broke 7 existing monotonic
tests whose fixtures' goals happened to match the committed
hypothesis directly (or the fork's saturation derivations).
Fixed by changing those fixtures' goals to an unsatisfiable
`(never ?x)` pattern (a relation declared in the ontology
but never asserted), so the fork-side check stays inert and
the tests' original assertions about layer behaviour / nogood
emission / writeback still fire. Affected fixtures:
SINGLETON_FIXTURE / MULTI_FIXTURE / ALL_DIE_FIXTURE in
`test_monotonic_cdcl.py`; `_FIXTURE` in `test_monotonic_budget.py`;
SINGLETON_FIXTURE in `test_monotonic_dumper.py`.

The `test_branching_04_returns_ambiguity` test in
`test_monotonic_skeleton.py` was renamed to
`test_branching_04_returns_first_solution` to reflect the
fixed behaviour, with a docstring + assertions referencing
Q1.5b.7.
