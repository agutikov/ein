## Ship notes 2026-05-27

The module surface below was implemented verbatim with two
small adjustments documented here. The shipped `commitment.py`
is the source of truth.

| Adjustment | Reasoning |
|---|---|
| `saturator_steps: int \| None = None` (default, no cap) instead of `int = 10_000` | The M1 ruleset is monotone — saturation terminates. The 10K cap was an arbitrary safety net; matching `Saturator.saturate`'s own `max_steps=None` default keeps the API uniform and avoids `SaturatorStepLimitError` surprises in larger fixtures. Callers can opt in to a cap explicitly. |
| `unsat_core` wrapped in `frozenset(…)` at the call site | `KnowledgeBase.unsat_core` returns `set[Fact]`; the dataclass field is typed `frozenset[Fact]`. The spec pseudocode omitted the conversion. |

### Dead-post `unsat_core` semantics (worth knowing)

The dead-post test surfaced a subtle point about
`fork.unsat_core(c.witness for c in post_contras)`. For a
pair contradiction (REASONING-layer `X` + `(not X)`),
`c.witness` returns only the **positive** (matches the
convention in
[`tree/solver.py`](../../../ein.py/src/ein_bot/inference/tree/solver.py)
lines 382 + 715). So `unsat_core` walks back from `X` only,
reaching whichever hypothesis derived `X` — **not** the
hypothesis that derived `(not X)`. In the test fixture (two
hypotheses h1, h2 each deriving one side of the conflict),
`unsat_core = {h1}`, not `{h1, h2}`. The test asserts
`at least one of {h1, h2}` rather than `{h1, h2}` to match
this.

This isn't a bug — the source frontier for back-prop /
no-good emission is reachable from the positive alone in
practice (rules typically derive positives; negatives come
from a paired structural rule). If a future engine needs the
full bipartite source frontier, pass both sides:
`fork.unsat_core(itertools.chain.from_iterable(
    (c.witness, c.negative) for c in post_contras
    if c.negative is not None))`.

### Test scaffolding notes

- `_kb(text)` + `_put(kb, fact)` helpers mirror
  `test_contradiction.py`'s canonical pattern for inline-IR
  fixtures.
- Rules use empty parameter lists `(rule name () :match …)` —
  pattern variables like `?x` are *match-side* free vars,
  not rule parameters. Rule parameters bind via activators
  (e.g., `(symmetric r)` in ontology binds `?rel = r`); when
  no parameter is needed, `()` suffices and no activator is
  required.

## Goal

Ship `try_commitment_set` as the **single shared primitive** for batched
hypothesis-set entering. Conceptually:

```python
CommitmentSetResult = try_commitment_set(root_kb, commitment)

# 1. fork = root_kb.fork()
# 2. for h in commitment: fork.add_fact(h)
# 3. ContradictionDetector(fork).detect()    ← pre-saturation
# 4. Saturator(fork).saturate()
# 5. ContradictionDetector(fork).detect()    ← post-saturation
# 6. compute unconditional facts (chain doesn't touch any element of C)
# 7. return CommitmentSetResult(...)
```

The function is **pure** in the sense that it doesn't mutate
`root_kb` or any object reachable from it (modulo the
existing `fork()` shared-by-reference fields, which the
S1.5a.20 channel-isolation rewrite addresses separately).

## Module surface

`ein.py/src/ein_bot/inference/commitment.py`

```python
"""Common commitment-set primitive — `try_commitment_set` + `CommitmentSetResult`.

Forks `root_kb`, writes every hypothesis in `commitment`
into the fork, saturates once, detects contradictions, and
extracts unconditional facts (whose derivation chain doesn't
touch any element of `commitment`).

Both monotonic and lattice engines call this. Pure-with-fork
semantics — the fork is the function's output, never reused
across calls. No state is shared between two `try_commitment_set`
invocations on the same root.
"""
from __future__ import annotations

from dataclasses import dataclass, field
from typing import Literal

from ein_bot.inference.apriori import CanonicalSetId, FactId
from ein_bot.inference.contradiction import ContradictionDetector
from ein_bot.inference.saturator import Saturator
from ein_bot.inference.tree.back_prop import reaches_hypothesis
# ↑ reaches_hypothesis is the existing pure-fact walker; left
# at its current home until S1.5a.20 pulls it into common.
from ein_bot.kb.entities import Fact, Layer
from ein_bot.kb.provenance import Provenance
from ein_bot.kb.store import KnowledgeBase

@dataclass(frozen=True)
class CommitmentSetResult:
    """Outcome of one commitment-set entering — `try_commitment_set`'s return.

    Carries the commitment, the forked + saturated kb, and the
    facts the parent (or root, in monotonic mode) may want to
    adopt.
    """
    commitment:  CanonicalSetId
    kb:          KnowledgeBase
    firings:     tuple[...]                 # type: tuple[Firing, ...]; loose hint to avoid cycle
    kind:        Literal["alive", "dead-pre", "dead-post"]
    unsat_core:  frozenset[Fact] = frozenset()

    # Facts derived during this set's saturation whose
    # provenance chain doesn't touch any hypothesis in
    # `commitment`. Provably true at root level given root +
    # rules. Engine merges these into root.
    unconditional_facts: tuple[Fact, ...] = ()

    # The set's hypothesis-derived facts (the actual `(h_i)`
    # writes for h_i ∈ commitment, NOT the saturator's
    # additions). Useful for the lattice's per-set audit.
    hypothesis_facts: tuple[Fact, ...] = ()

def try_commitment_set(
    root_kb: KnowledgeBase,
    commitment: CanonicalSetId,
    *,
    saturator_steps: int = 10_000,
) -> CommitmentSetResult:
    """Fork root, write every hypothesis in `commitment`,
    saturate, detect, extract unconditional facts.

    `commitment` is the canonical-tuple representation
    (sorted; see `apriori.CanonicalSetId`). Each element is a
    `(relation_name, args)` FactId for a positive hypothesis
    fact. The fork's saturator runs at most `saturator_steps`
    rule firings.

    Returns:
      CommitmentSetResult(kind="dead-pre", unsat_core=…) if a (X, ¬X)
        pair surfaces immediately after writing the
        hypotheses (no saturation runs).
      CommitmentSetResult(kind="dead-post", unsat_core=…) if
        saturation runs to fixpoint and the
        post-saturation kb has a contradiction.
      CommitmentSetResult(kind="alive", unconditional_facts=…,
                hypothesis_facts=…) otherwise.

    Idempotency: `try_commitment_set(root_kb, C)` produces an
    independent result every call; calling it twice on the
    same `root_kb` returns two separate `CommitmentSetResult` objects
    whose forks share no mutable state.
    """
    fork = root_kb.fork()
    hypothesis_facts: list[Fact] = []
    for (rn, args) in commitment:
        h_fact = Fact(
            relation_name = rn,
            args          = args,
            layer         = Layer.REASONING,
            provenance    = Provenance.from_hypothesis(
                branch=0,  # lattice doesn't track branch ids
                           # the same way the tree does; the
                           # commitment IS the audit key.
            ),
        )
        stored = fork.add_fact(h_fact)
        fork._index_fact(stored)
        hypothesis_facts.append(stored)

    # Pre-saturation contradiction check (apriori filter at
    # the kb level — catches any newly-negated facts that
    # crept into root between the candidate's generation and
    # this fork's creation).
    pre_contras = ContradictionDetector(fork).detect()
    if pre_contras:
        return CommitmentSetResult(
            commitment       = commitment,
            kb               = fork,
            firings          = (),
            kind             = "dead-pre",
            unsat_core       = fork.unsat_core(c.witness for c in pre_contras),
            hypothesis_facts = tuple(hypothesis_facts),
        )

    sat = Saturator(fork)
    firings = tuple(sat.saturate(max_steps=saturator_steps))

    post_contras = ContradictionDetector(fork).detect()
    if post_contras:
        return CommitmentSetResult(
            commitment       = commitment,
            kb               = fork,
            firings          = firings,
            kind             = "dead-post",
            unsat_core       = fork.unsat_core(
                                  c.witness for c in post_contras),
            hypothesis_facts = tuple(hypothesis_facts),
        )

    # Alive — extract unconditional facts.
    hyp_ids = frozenset(commitment)
    unconditional: list[Fact] = []
    for f in fork.facts:
        if not _is_new_relative_to(f, root_kb):
            continue
        if _is_unconditional(f, fork, hyp_ids):
            unconditional.append(f)

    return CommitmentSetResult(
        commitment           = commitment,
        kb                   = fork,
        firings              = firings,
        kind                 = "alive",
        unconditional_facts  = tuple(unconditional),
        hypothesis_facts     = tuple(hypothesis_facts),
    )

# ── Helpers ─────────────────────────────────────────────────

def _is_new_relative_to(
    fact: Fact, root_kb: KnowledgeBase,
) -> bool:
    """True iff `fact` exists in the fork but not in root."""
    return root_kb._fact_by_id(fact.relation_name, fact.args) is None

def _is_unconditional(
    fact: Fact, kb: KnowledgeBase,
    hypothesis_ids: frozenset[FactId],
) -> bool:
    """True iff `fact`'s derivation chain doesn't touch any
    hypothesis in `hypothesis_ids`.

    Reuses the existing `reaches_hypothesis` walker (from
    `tree/back_prop.py`) but with a custom "is-hypothesis"
    predicate: instead of "kind in {hypothesis, rejected}",
    we check whether the fact-id matches one in our
    commitment set.

    A fact is unconditional iff every chain of its
    `provenance.premises_raw` walks back to a `source` or
    un-provenanced terminal, never touching a fact whose id
    is in `hypothesis_ids`.
    """
    visited: set[FactId] = set()
    return not _reaches_commitment(
        kb, fact, visited, hypothesis_ids,
    )

def _reaches_commitment(
    kb: KnowledgeBase, fact: Fact,
    visited: set[FactId], hypothesis_ids: frozenset[FactId],
) -> bool:
    """Recursive walker — True iff some premise chain reaches
    a fact in `hypothesis_ids`."""
    key: FactId = (fact.relation_name, fact.args)
    if key in visited:
        return False
    visited.add(key)
    if key in hypothesis_ids:
        return True
    prov = fact.provenance
    if prov is None or prov.kind != "rule":
        return False
    for rid in prov.premises_raw:
        premise = kb._fact_by_id(*rid)
        if premise is not None and _reaches_commitment(
                kb, premise, visited, hypothesis_ids):
            return True
    return False
```

### T1.5b.3.2 — Unit tests

`tests/inference/test_commitment.py`:

- **Alive path on a 1-element commitment.** Root has
  `(some-fact a b)`; commitment is `{(target c d)}`; rule
  derives `(other e f)` from the hypothesis. Expected:
  `kind="alive"`, `unconditional_facts=()`,
  `hypothesis_facts=((target c d),)`, kb contains
  `(other e f)`.

- **Alive path with an unconditional derivation.** Root +
  rule library that derives a fact from purely root facts
  even when a hypothesis is committed (e.g., a transitive
  closure of a root chain triggered by saturator's
  scheduling). Expected: that fact appears in
  `unconditional_facts`.

- **Alive path with a conditional derivation.** Root + rule
  that derives a fact from the hypothesis + root facts.
  Expected: the derived fact is in `kb.facts` but NOT in
  `unconditional_facts`.

- **Dead-pre path.** Root already carries `(not h)` for one
  of the commitment's hypotheses. Expected: `kind="dead-pre"`,
  `unsat_core` includes both `h` and `(not h)`.

- **Dead-post path.** Root + rules where two committed
  hypotheses derive contradictory facts. Expected:
  `kind="dead-post"`, `unsat_core` includes the
  contradicting pair's witnesses.

- **Isolation.** `try_commitment_set(root_kb, C1)` followed by
  `try_commitment_set(root_kb, C2)` produces independent forks; mutating
  the first result's kb doesn't affect the second.

- **Empty commitment** (sentinel case). `try_commitment_set(root_kb, ())`
  is layer-0 = just root. Returns `alive` with
  `unconditional_facts=()`, `hypothesis_facts=()`, kb = a
  fork of saturated root.

## Soundness of the unconditional check

A fact `f` derived during `try_commitment_set(root_kb, C)`'s saturation
has a derivation chain rooted at:
- Some `source` / un-provenanced root fact, OR
- One of `C`'s hypothesis facts (the writes we made in step 2).

If every chain grounds out at root facts only — no chain
touches an element of `C` — then `f` is derivable from
`root_kb ∪ rules` alone. Adding it to root is sound; the
fact would have been derived eventually by the next root
saturation, given the rules.

If any chain reaches an element of `C`, `f` is **conditional**
on `C`. Adding it to root would be unsound — it might be
false in a world where `C`'s commitments don't hold.

The walker `_reaches_commitment` checks every chain in
`provenance.premises_raw` and returns True the moment any
chain hits an element of `C`'s fact-id set.

Equivalent to S1.5a.14's `reaches_hypothesis` but
parameterised by the commitment instead of the global
"is-it-a-hypothesis" check. Identical structure; minimal
duplication.
