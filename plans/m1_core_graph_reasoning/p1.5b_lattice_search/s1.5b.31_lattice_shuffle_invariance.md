## The invariant

Let `solve_lattice(root_kb, rules, max_set_size=L)` produce, on
completion:

- `nodes(L)`   — set of canonical-set-id values for every
                 visited `SetNode`
- `labels(L)`  — per node, the multilabel (set of commitment
                 sets that map to this node under dedup)
- `kb_state(L)` — per node, the `state_hash(node.kb)` (or a
                 content-projection of `node.kb`)
- `root_kb(L)` — the root accumulator's content after every
                 unconditional bubbled up
- `nogoods(L)` — the learned-clause set at termination
- `verdict`    — Solution / Solutions / Contradiction (per
                 the lattice's mode-aware verdict from
                 [Q1.5b.7](open_questions.md))

> **Claim.** For any permutation σ of the *within-layer*
> visit order (the order in which the lattice picks the next
> SetNode to expand inside a layer, controlled by
> [S1.5b.26](s1.5b.26_lattice_scoring.md)'s `lattice_order`
> knob), the six items above are identical functions of
> `(root_kb, rules, L)` — they do not depend on σ.

This is strictly stronger than S1.5b.27's per-set commutativity:
.27 says "for one set, all paths produce the same kb"; .31
says "the entire visited lattice is path-independent at any
layer cut". A counterexample for .31 need not violate .27 —
e.g. a bug where the order of integrating unconditionals into
`root_kb` decides which sets get apriori-pruned at the next
layer would shift the lattice's *shape* without violating any
single set's commutativity.

## Why this could leak

The lattice's per-set saturation is structurally commutative
(S1.5b.27 verifies). But the *engine loop* still threads
state across sets:

1. **Forced-positive integration order** ([S1.5b.25](s1.5b.25_lattice_forced_positives.md)).
   When SetNode X discovers an unconditional positive, it
   integrates into `root_kb`; subsequent sets see the
   richer root. If two sets in the same layer each discover
   different unconditionals and the integrate order matters
   (e.g. X's unconditional makes Y's unconditional redundant
   under dedup), the per-layer integration outcome can vary.

2. **CDCL no-good emit order** (composes with
   [P1.5a S1.5a.18](../p1.5a_zebra_solution/s1.5a.18_path_condition_nogoods.md)
   if the flag flips for lattice). Two failing sets at the
   same layer may emit subsuming/subsumed nogood clauses;
   the order they land changes which dominant clauses appear
   in the final `_nogoods`.

3. **State-hash dedup race**. Two sets reach equivalent kb
   states at slightly different times in the layer's walk; the
   first to hit `state_index` becomes the `representative_nid`
   and downstream events (label aggregation, multi-parent
   integrate) reference that node. The lattice's final shape is
   the same, but the `representative_nid` choice can leak into
   the per-set audit dump.

None of these are bugs in the saturation premise; each is a
hypothesis the harness would test for the engine *loop*.

## The harness

A pytest-driven shuffle-and-diff rig, modelled on
[`test_shuffle_invariance.py`](../../../ein.py/tests/inference/test_shuffle_invariance.py)
(the tree-side sibling) and the per-set dumper
([S1.5b.23](s1.5b.23_lattice_dumper.md)):

```python
def test_lattice_shuffle_invariance(puzzle, max_set_size, seeds):
    kb0 = parse_and_load(puzzle)
    snapshots = []
    for seed in [None, *seeds]:                 # None = default order
        kb = kb0.fork()
        cfg = SolverConfig(
            lattice_max_set_size=max_set_size,
            lattice_order=("shuffle", seed) if seed else "score-sum",
        )
        proof = solve_lattice(kb, cfg)
        snapshots.append(lattice_snapshot(proof, kb))

    ref, *others = snapshots
    for s in others:
        assert s.canonical_ids == ref.canonical_ids
        assert s.kb_hashes_by_id == ref.kb_hashes_by_id
        assert s.root_kb_content == ref.root_kb_content
        assert s.nogoods_set == ref.nogoods_set
        assert s.verdict == ref.verdict
        # multilabel content same; representative_nid may differ —
        # canonicalise via min(label_ids) before compare.
```

`lattice_snapshot` projects the depth-L lattice into the
content-addressed six items above (no `representative_nid`,
no per-run timing, no integrate-order numbering — those are
permitted to differ).

## Tasks

### T1.5b.31.1 — Shuffle knob in lattice config

- **a.** Extend [S1.5b.26](s1.5b.26_lattice_scoring.md)'s
  `lattice_order` config: accept a `("shuffle", seed)` tuple in
  addition to `"lex"` and `"score-sum"`. Apply a
  `random.Random(seed).shuffle(...)` to the within-layer
  candidate list, *after* the default scoring sort, so the
  seed-driven permutation is deterministic per run.
- **b.** Update `lattice/solver.py`'s within-layer selection
  step to honour the knob.

  `representative_nid` differences don't false-fail.
- **b.** Reuse [S1.5b.23](s1.5b.23_lattice_dumper.md)'s
  per-set dumper output where possible — snapshot is a
  projection of its `set/<canonical-slug>/` records.

### T1.5b.31.4 — Triage the violations

Open task; scope depends on what T1.5b.31.3 surfaces. Three
likely classes (one per § "Why this could leak" suspect),
each addressed differently:

- **Forced-positive integration order class.** If two sets at
  the same layer integrate unconditionals into `root_kb` in
  different orders and the *content* of `root_kb` after the
  layer differs, the integrate step needs to be made
  permutation-invariant (e.g. integrate all of a layer's
  unconditionals *atomically* at layer end, not as they're
  discovered). Folds into [S1.5b.25](s1.5b.25_lattice_forced_positives.md).
- **Nogood subsumption order class.** If learned-clause
  membership differs based on emit order, `emit_nogood`'s
  subsumption-aware insert needs to be order-stable (sort the
  emitted set before subsumption pass; or compute the
  subsumption-closure post-layer). Folds into
  [P1.5a S1.5a.18](../p1.5a_zebra_solution/s1.5a.18_path_condition_nogoods.md)
  if that machinery is reused; otherwise lattice-side fix.
- **Multilabel representative-nid leak.** If the snapshot
  canonicalisation in T1.5b.31.2 isn't enough — i.e., some
  downstream consumer of `representative_nid` produces
  permutation-dependent output — that consumer needs the same
  canonicalisation. Add to the lattice dumper schema.

## Open questions

- **Q-S1.5b.31.A — Multilabel canonicalisation depth.** Is it
  enough to canonicalise `representative_nid` at snapshot time,
  or does the lattice need to internally pick a deterministic
  rep (e.g. min(label_ids))? The latter is more invasive but
  removes order-leak at the source.
- **Q-S1.5b.31.B — Within-set permutation vs across-set.** This
  stage tests across-set traversal order. Within-set
  permutation (which order does saturator dequeue firings) is
  covered by S1.5b.27 + S1.5a.1a content-sort. Verify both
  knobs are honoured at the lattice level.
- **Q-S1.5b.31.C — Mode interaction.** SOLVE-mode terminates
  on first solution; the lattice snapshot at termination
  depends on which set hit the goal first. Either restrict
  this stage to GAPS / CONTRADICTIONS modes (full enumeration),
  or define a SOLVE-mode-specific invariant ("the set of
  solution-leaves found at depth-cap L is the same"). Decide
  at T1.5b.31.3 time.
