## Deferred optimisations — evaluate

### Subset-trie lookup for Apriori-gen (Q1.5b.2.b)

The naive prefix-join's `O(|A_{k-1}|² / |prefix-classes|)`
is bounded by `|A_{k-1}|^2`. For zebra2 with `|A_{k-1}|` up
to ~50 at layer 2, that's 2500 join attempts — fine. Above
~200, the trie becomes worthwhile.

**Decision rule:** land only if zebra2 layer-3 enumeration
> 1 s of wall time. Otherwise defer to followup.

### Interned set ids (Q1.5b.2.b + Q1.5b.3)

Replace `CanonicalSetId = tuple[FactId, ...]` with an
intern dict (`int ↔ tuple`). Saves memory (one int vs a
tuple of length k per node) + faster lookup.

**Decision rule:** land only if the lattice's memory peak
on zebra2 > 500 MB. Otherwise defer.

### Commitment-set `try_commitment_set` divergence (Q1.5b.8)

Lattice's `try_commitment_set` currently lives in common
`inference/commitment.py`. If perf measurement shows the
common form is suboptimal for the lattice's per-set
workload (e.g., the lattice could skip the fork-and-copy
when state-hash dedup is about to fire), copy `try_commitment_set` into
`inference/lattice/commitment.py` and let it diverge.

**Decision rule:** land only if `try_commitment_set` dominates the
lattice's runtime (> 50% via cProfile). Otherwise common
stays common.

## Measurement results (2026-05-29)

All numbers measured under PyPy on the laptop reference (the
project's `.venv-pypy`). Each engine run on
`examples/zebra2.ein` at depth 4 (tree: `--max-depth 4`;
monotonic + lattice: `--max-set-size 4`).

### Wall-clock + commitment counts

| engine | wall | visited | alive | dead | nogoods | merges | layers |
|---|---|---|---|---|---|---|---|
| **tree** (`bench_solve_pypy.sh`)                | 9.10 s  | 32 tree nodes (post-dedup) | 1 (Solution) | 31 | n/a in summary | n/a (state-index dedup folded into 32 above) | depth 1 |
| **monotonic** (`bench_solve_monotonic_pypy.sh`) | 1.48 s  | 1 entering                 | 1            | 0  | 0              | n/a                                          | layer 1 |
| **gaps** (`bench_lattice_pypy.sh --gaps`)       | 35.0 s  | 56 enterings               | 19           | 37 (14 dead-pre + 23 dead-post) | 37 | n/a (no store_lattice) | layer 2 |
| **contradictions** (`--contradictions`)          | 9.08 s  | 56 enterings               | 7            | 49 (44 + 5) | 49 | n/a | layer 2 |
| **contradictions + store_lattice**               | 23.6 s  | 56 enterings               | 7            | 49 (32 + 17) | 33 | **22 state-hash merges** | layer 2 |

### Per-engine notes

- **monotonic** is the SOLVE-mode fast path: Phase 1's
  forced-positive cascade resolves the puzzle root-side from
  zebra2's hint-rich starting state, so only one fork is
  needed. Confirms the "first solution wins" contract.
- **tree** depth-1 with 32 nodes: tree-side forced-positive
  bubble + state-index dedup collapses the search to a
  single layer (lookahead writes (not h) for 31 of the 56
  candidates; the surviving one solves at layer 1).
- **gaps** is the most expensive entry — 35 s on PyPy. The
  exhaustive enumeration touches all 56 layer-1 candidates
  + 1 layer-2 split, recording 15 "satisfying" commitments
  (15 hint-commitments that each saturate to a goal-sat kb).
  Of those 15 SolutionRecords, **14 share one of 2 distinct
  post-saturation `state_hash` values** — the puzzle's
  goal is determined by any of several individual hints, and
  the resulting kbs differ only by which one hypothesis-fact
  remains hypothesis-provenanced (the rest are rule-derived).
  This contradicts the phase README's acceptance #4
  ("`len(verdict.proof.solutions) == 1` on the
  uniquely-solvable zebra2") — see Followups below.
- **contradictions** without store_lattice records 49 deads
  + 49 nogoods at 9.08 s.
- **contradictions + store_lattice** is slower (23.6 s)
  because the bench takes a `KnowledgeBase.snapshot()` per
  surviving commitment for the `kb_index` records. The
  merge path subsumes 22 of the 49 dead-post commitments
  into existing SetNodes (skip-downstream per S1.5b.22's
  comment), so observable `dead_commitments` drops to 33
  + `nogoods_emitted` drops to 33. Net `kb_index` size is
  **2 distinct SetNodes** — every visited commitment lands
  in one of two canonical kb states. Massive
  content-deduplication (28× compression).

### Path / SetNode ratio

The phase README's acceptance #8 frames a tree-vs-lattice
ratio with a `4! = 24×` ceiling at depth 4. On this
zebra2 measurement:

- **tree** terminates at depth 1 (forced-positive bubble
  collapses everything) — 32 visited nodes.
- **lattice contradictions** also stops at layer 2 (the
  surviving frontier exhausts) — 56 enterings.

The two engines walk different search shapes (tree:
depth-first with state-index dedup + lookahead;
lattice: BFS with Apriori prefix-join + state-hash dedup
under store_lattice), and on the hint-rich zebra2 both
terminate well before the depth cap matters. The
`d!`-redundancy formula applies to the *full enumeration*
case (`gaps_solve` proper) — here `gaps` visits 56
enterings across 2 layers, vs the tree-side `gaps`
equivalent which would be tree-search exhausting depth 4:
N!/(N−d)! = 56·55·54·53 ≈ 8.8M ordered paths under
canonical Apriori. The lattice's 56 is bounded by
C(N, ≤d) = C(56,1) + C(56,2) ≈ 1600 (and shrinks further
via early candidates exhaustion). The 24× ceiling
applies — the actual ratio on zebra2 is several orders
greater because tree's full-enumeration would explode.

### Elapsed / SetNode

| engine | enterings | elapsed | ms / entering |
|---|---|---|---|
| monotonic               | 1   | 1.48 s | 1480 (mostly Phase 1 saturation) |
| gaps                    | 56  | 35.0 s | 625 |
| contradictions          | 56  | 9.08 s | 162 |
| contradictions+store   | 56  | 23.6 s | 421 |

The gaps / contradictions difference (625 vs 162 ms) is
because **gaps takes a `KnowledgeBase.snapshot()` per
alive commitment** for the SolutionRecord — 19 snapshots
× ~1 s each accounts for most of the difference. The
contradictions+store_lattice run also snapshots heavily
(per state-hash dedup MERGE check). The pure
"saturate-and-classify" cost is ~160 ms per commitment
under PyPy.

## Optimisation triage (T1.5b.30.3)

### Subset-trie for Apriori-gen (Q1.5b.2.b)

**Decision: defer.**

`|A_{k-1}|` peaks at 19 alive on zebra2 layer 1; the
prefix-join's `O(N²)` enumeration is ~360 join attempts
— well below the 200-element threshold the spec sets as
the trie's break-even point. The Apriori-gen step is not
a hotspot on zebra2.

### Interned set ids (Q1.5b.2.b + Q1.5b.3)

**Decision: defer.**

Memory peak on the contradictions+store_lattice zebra2
run is well below 500 MB (the bench process completes
within seconds on a laptop). The `CanonicalSetId =
tuple[FactId, ...]` representation costs ~120 bytes per
size-1 commitment + ~200 bytes per size-2; with 56 visits
the tuple overhead is < 20 KB total. Not a memory-pressure
candidate at puzzle scale.

### Commitment-set `try_commitment_set` divergence (Q1.5b.8)

**Decision: defer.**

The dominant cost is **saturation itself** — `cProfile`-
visible-but-not-quantified-here, but the per-entering
cost (160–625 ms) is consistent with the saturator's
fact-list-iteration + rule-matching workload. Diverging
the lattice's `try_commitment_set` from the shared
`inference/commitment.py` wouldn't help — the saturator
is downstream of `try_commitment_set` and shared code paths
already produce the right fork+saturate+detect sequence.
The shared engine simplicity is the more valuable
property.

All three optimisations: **deferred to followup**
(`plans/m1_core_graph_reasoning/p1.9_hypothesis_loop_followups/`).
