# S1.5b.6 — Monotonic CDCL (nogood emit + filter)

**Phase:** P1.5b (set-indexed search engines).
**Estimate:** S (½–1 day) — small surface; the heavy lifting
already lives in [`nogoods.py`](../../../ein.py/src/ein_bot/inference/nogoods.py).
**Status:** **implemented** 2026-05-27 (see Ship notes below).
**Depends on:** [S1.5b.5](s1.5b.5_monotonic_backbone.md)
(backbone running; nogoods slot in cleanly).
**Blocks:** [S1.5b.7](s1.5b.7_monotonic_dumper.md) (dumper

[S1.5b.8](s1.5b.8_monotonic_acceptance.md) (Zebra likely

## Goal

Two changes to the monotonic loop:

1. **Emit** — on every `result.kind in ("dead-pre",
   "dead-post")` outcome, emit `frozenset(C)` into
   `root_kb._nogoods` via `emit_nogood` (subsumption-aware).
2. **Filter** — pass `root_kb._nogoods` to `generate_layer`'s
   `nogoods` parameter; the existing filter drops candidates
   whose subset matches.

Single-element nogoods (layer-1 deaths) shrink the alive set
on the next `_compute_alive` invocation — `generate_hypotheses`
already skips `(not h)`-stored candidates.

No new mechanism; the work is pure integration.

### Edit 3 — write back the `(not h)` for size-1 deaths

When a singleton commitment `{h}` dies unconditionally, the
classical back-prop writes `(not h)` into root so that
`generate_hypotheses` (and any rule consuming negative
facts) sees the negation on subsequent saturation.

`emit_nogood` stores the clause but doesn't write to
`_negated_facts`; we add a thin helper:

```python
def _emit_negated_fact_writeback(
    root_kb: KnowledgeBase, h_id: FactId,
) -> None:
    """For a singleton dead clause {h_id}, write (not h)
    into root_kb._negated_facts so hypgen + rule premises
    see it on the next saturate.

    This mirrors what `tree/back_prop.py`'s
    `_write_negation` does for unconditional deaths in the
    tree engine. We don't import that function because it
    carries ContextVar coupling we want to avoid on the
    monotonic side; the writeback here is the minimal
    equivalent.
    """
    rn, args = h_id
    not_fact = Fact(
        relation_name="not",
        args=(_make_inner_fact(rn, args),),
        layer=Layer.REASONING,
        provenance=Provenance.from_rule(
            rule="<monotonic-unconditional>",
            premises_raw=(),
        ),
    )
    stored = root_kb.add_fact(not_fact)
    root_kb._index_fact(stored)

def _make_inner_fact(rn: str, args: tuple) -> Fact:
    """Build the wrapped Fact inside a (not …) write."""
    return Fact(
        relation_name=rn, args=args,
        layer=Layer.REASONING, provenance=None,
    )
```

## Why this stage matters

Without CDCL, monotonic explores every Apriori-generated
candidate even when its subsets have already died — wastes
saturation cost on guaranteed-dead branches. For Zebra-scale
puzzles the cumulative wasted cost can be the difference
between solving in seconds vs. minutes.

The S1.5b.8 acceptance gate (Zebra under
`bench_monotonic.py`) almost certainly needs this stage to
hit its time budget; this is why it's the next stage after
the backbone.

## Ship notes — 2026-05-27

Status: implemented at HEAD (base 102516f, S1.5b.6 changes
not yet committed at write time).

Deltas from the spec above:

- **`emit_nogood(min_size=2)` parameter, not a verbatim reuse.**
  The pre-S1.5b.6 `emit_nogood` rejected size-1 clauses
  unconditionally (tree-side guard — covered by
  `test_emit_nogood_rejects_single_element`). Q1.5b.5.c says
  the DAG side wants size-1 clauses to land too. To preserve the
  tree-side test contract, `emit_nogood` grew a keyword-only
  `min_size: int = 2` parameter; the monotonic solver passes
  `min_size=1`. Tree call sites are unchanged. No behavioural
  drift on the tree side.

- **`_emit_negated_fact_writeback` also writes the symmetric
  mirror.** The spec's helper handled only the named singleton;
  but `_compute_alive` canonicalises symmetric pairs (so a
  declared `(symmetric R)` collapses `(R a b)` and `(R b a)` to
  one alive entry), and hypgen's `_is_excluded` checks each
  orientation literally. Without writing the mirror `(not (R b a))`
  too, the dead singleton resurrects via the mirror orientation
  on the next `_compute_alive` and the empty-alive contradiction
  test stays Ambiguity. The new helper mirrors tree-side
  `back_propagate(..., promote_symmetric=True)` — same shape, no
  ContextVar coupling.

- **Phase-3 `alive` refresh.** The backbone's verdict synthesis
  read the local `alive` straight out of the last alive-entering
  recompute; if the only deaths were singletons (no facts merged),
  the writebacks never refreshed `alive` and a fully-dead Phase 2
  reported Ambiguity instead of Contradiction. Added one
  `alive = _compute_alive(root_kb)` call at Phase 3 entry. This
  is the cheapest correct point — within-layer alive is read only
  for generate_layer, which already handles dead candidates via
  the `_nogoods` subset filter (and via apriori's prefix-join
  excluding dead pairs from `a_prev`).

- **Test fixtures pass `enable_pre_branch_lookahead=False` +
  `enable_back_prop_unconditional=False`.** With both flags at
  default, the lookahead writes `(not h)` via back_prop before
  any monotonic candidate reaches `try_commitment_set`, so the
  CDCL emit path never fires. The explicit `_NO_LOOKAHEAD`
  config in `test_monotonic_cdcl.py` exercises the monotonic
  loop's emit/filter path directly. Other monotonic tests
  (skeleton, zebra2) stay on default config and remain green.

- **`MonotonicStats` grew `nogoods_emitted` + `nogoods_subsumed`.**
  Surfaced in `bench_monotonic.py`'s output block. No counter for
  "filtered by nogood at layer N" yet — would need an instrumented
  `generate_layer` callback; left for the perf round (S1.5b.30)
  if it turns out to matter for tuning.
