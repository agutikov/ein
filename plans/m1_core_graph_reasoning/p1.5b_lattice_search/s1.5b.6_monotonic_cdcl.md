# S1.5b.6 — Monotonic CDCL (nogood emit + filter)

**Phase:** P1.5b (set-indexed search engines).
**Estimate:** S (½–1 day) — small surface; the heavy lifting
already lives in [`nogoods.py`](../../../ein.py/src/ein_bot/inference/nogoods.py).
**Status:** **planned**.
**Depends on:** [S1.5b.5](s1.5b.5_monotonic_backbone.md)
(backbone running; nogoods slot in cleanly).
**Blocks:** [S1.5b.7](s1.5b.7_monotonic_dumper.md) (dumper
records nogood events),
[S1.5b.8](s1.5b.8_monotonic_acceptance.md) (Zebra likely
needs nogoods to terminate within the time budget).

## Goal

Two changes to the monotonic loop:

1. **Emit** — on every `result.kind in ("dead-pre",
   "dead-post")` outcome, emit `frozenset(C)` into
   `root_kb._nogoods` via `emit_nogood` (subsumption-aware).
2. **Filter** — pass `root_kb._nogoods` to `generate_layer`'s
   `nogoods` parameter; the existing filter drops candidates
   whose subset matches.

Single-element nogoods (layer-1 deaths) shrink the alive set
on the next `_compute_alive` invocation — `generate_hypotheses`
already skips `(not h)`-stored candidates.

No new mechanism; the work is pure integration.

### Edit 3 — write back the `(not h)` for size-1 deaths

When a singleton commitment `{h}` dies unconditionally, the
classical back-prop writes `(not h)` into root so that
`generate_hypotheses` (and any rule consuming negative
facts) sees the negation on subsequent saturation.

`emit_nogood` stores the clause but doesn't write to
`_negated_facts`; we add a thin helper:

```python
def _emit_negated_fact_writeback(
    root_kb: KnowledgeBase, h_id: FactId,
) -> None:
    """For a singleton dead clause {h_id}, write (not h)
    into root_kb._negated_facts so hypgen + rule premises
    see it on the next saturate.

    This mirrors what `tree/back_prop.py`'s
    `_write_negation` does for unconditional deaths in the
    tree engine. We don't import that function because it
    carries ContextVar coupling we want to avoid on the
    monotonic side; the writeback here is the minimal
    equivalent.
    """
    rn, args = h_id
    not_fact = Fact(
        relation_name="not",
        args=(_make_inner_fact(rn, args),),
        layer=Layer.REASONING,
        provenance=Provenance.from_rule(
            rule="<monotonic-unconditional>",
            premises_raw=(),
        ),
    )
    stored = root_kb.add_fact(not_fact)
    root_kb._index_fact(stored)

def _make_inner_fact(rn: str, args: tuple) -> Fact:
    """Build the wrapped Fact inside a (not …) write."""
    return Fact(
        relation_name=rn, args=args,
        layer=Layer.REASONING, provenance=None,
    )
```

## Why this stage matters

Without CDCL, monotonic explores every Apriori-generated
candidate even when its subsets have already died — wastes
saturation cost on guaranteed-dead branches. For Zebra-scale
puzzles the cumulative wasted cost can be the difference
between solving in seconds vs. minutes.

The S1.5b.8 acceptance gate (Zebra under
`bench_monotonic.py`) almost certainly needs this stage to
hit its time budget; this is why it's the next stage after
the backbone.
