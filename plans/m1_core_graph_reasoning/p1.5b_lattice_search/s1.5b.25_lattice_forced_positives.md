## Goal

Extend the `CommitmentSetResult` from S1.5b.3 with a
`forced_positives` field already, and have `try_commitment_set`
populate it. Integrate (S1.5b.24) already iterates the
field — this stage makes sure it's not empty.

The forced-positive mining is **already conceptually shipped**
via S1.5b.3's `unconditional_facts` field. This stage:

1. Renames `unconditional_facts → forced_positives` to
   match the lattice's vocabulary + the S1.5a.14 tree-side
   terminology.
2. Adds the symmetric `proposed_negatives` field —
   `(not X)` derivations whose chain is also
   hypothesis-free.
3. Audits the soundness of integrate adopting these at
   parent levels.

After this stage, the lattice's bubble-up payload is:

```python
@dataclass(frozen=True)
class CommitmentSetResult:
    # … existing fields …

    forced_positives:   tuple[Fact, ...] = ()  # was unconditional_facts
    proposed_negatives: tuple[Fact, ...] = ()  # NEW: (not X) derivations
    learned_nogoods:    tuple[frozenset[FactId], ...] = ()  # NEW: clauses from child's
                                                           # own consume-style processing
```

## Symmetry — positives + negatives

S1.5a.14's tree-side `_mirror_forced_positive` only mined
positives. The lattice's integrate handles negatives too:

```python
# Inside try_commitment_set, after saturation, before returning:
hyp_ids = frozenset(commitment)
forced_positives: list[Fact] = []
proposed_negatives: list[Fact] = []
for f in fork.facts:
    if not _is_new_relative_to(f, root_kb):
        continue
    if not _is_unconditional(f, fork, hyp_ids):
        continue
    if f.relation_name == "not":
        proposed_negatives.append(f)
    else:
        forced_positives.append(f)
```

The integrate step's per-level check decides which to
adopt at each parent.

## Why distinguish positives vs negatives

Two reasons:

1. **Provenance attribution.** P1.6's NL renderer reads
   `forced_positives` and `proposed_negatives` separately
   to render "X is true because…" vs "X is not true
   because…". Mixing them flattens the explanation shape.

2. **Search effect.** Adopting a positive at root may
   trigger domain-elimination → derive more positives.
   Adopting a negative shrinks the alive set immediately.
   The dumper's audit splits them so the reader can see
   which kind triggered the next layer's pruning.

  `result.forced_positives` does NOT contain it.

  unconditional at P's level. Verify integrate adopts.
- **Rejection at parent.** Child mines a fact whose chain
  is unconditional at the child's level but reaches one of
  P's own commitments. Verify integrate rejects.
