## Architectural commitments

### Two engines, one phase

P1.5b ships **both** a monotonic engine and a lattice engine —
not a choice, both. They share Apriori-gen, the
`try_commitment_set(root_kb, commitment)` commitment-set primitive, the
CDCL nogood store, and the saturation-commutativity premise.
They differ in:

- **Termination.** Monotonic stops on first goal-sat at root
  (or layer exhaustion / `max_set_size`); lattice exhausts
  to `max_set_size` regardless.
- **Storage.** Monotonic stores only `root.kb` and
  `root._nogoods`; lattice stores per-set `SetNode`s with
  multilabel + multi-parent + cached subtree results.
- **Mode support.** Monotonic = SOLVE only; lattice = SOLVE
  / GAPS / CONTRADICTIONS.
- **Proof artefact.** Monotonic = root snapshot; lattice =
  `LatticeProof` (full DAG + provenances).
- **Code size.** Monotonic target ≤ 200 LOC; lattice
  considerably more.

### Co-existence layout

```
ein.py/src/ein_bot/inference/
   ├── *.py            ← common (Saturator, ContradictionDetector,
   │                     Lookahead, hypgen, nogoods, apriori, try_commitment_set,
   │                     kb-store helpers, base dumper utilities)
   ├── tree/           ← current tree-search; reference until parity
   ├── monotonic/      ← monotonic engine (S1.5b.4–.10)
   └── lattice/            ← lattice engine (S1.5b.20–.30)

ein.py/demo/
   ├── bench_solve.py        ← unchanged; tree entry
   ├── bench_monotonic.py    ← NEW; monotonic entry
   └── bench_dag.py          ← NEW; lattice entry

ein.py/tests/inference/
   ├── *.py            ← common-kernel tests
   ├── tree/           ← tree-search tests
   ├── monotonic/      ← monotonic tests
   └── lattice/            ← lattice tests
```

Explicit selection per script. No engine-internal default
flag. Copy-on-modify: a module starts in `inference/` (common);
when an engine needs to modify it incompatibly, copy to the
engine's folder and let the copy diverge.

### Lattice ≠ true semantic lattice

Two distinct concepts, made explicit in
[Q1.5b.4](open_questions.md#q15b4--set-equivalence-dedup--state-hash-dedup):

- The **true semantic lattice** is the abstract object
  defined by the rule set + the puzzle's conditions. We don't
  materialise it.
- The **search lattice** is the finite-state artefact we
  build by dedup. State-hash merges are evidence of the true
  lattice's structure being coarser than the powerset of
  hypotheses.

Merged `SetNode`s carry **multilabel** (all canonical sets
that collapsed) + **multi-parent** (union of parents across
labels). The model captures the true lattice convergence
without claiming to construct it directly.

### Saturation-commutativity premise

The whole approach rests on:

> For any commitment set `C`, `saturate(root ∪ C)` is
> determined by `C` (the unordered set), not by the order
> hypotheses were added.

True under M1's monotone rule set (no retraction; every
firing whose premises are present eventually fires). This is
the `d!`-redundancy elimination engine — what makes set
search strictly better than ordered-path tree search.

Sanity check (lattice F6): when `--lattice-sanity-check` is
enabled, every per-candidate processing in the lattice
saturates from each parent in turn and verifies state hashes
agree. Disabled by default; ship as a regression check.

### CDCL is shared

`nogoods.py` lives in the common `inference/` folder. Every
engine (tree, monotonic, lattice) imports the same
`emit_nogood`, `matches_any_nogood`, subsumption logic.
Clause shape `frozenset[FactId]` is identical across engines.
Same subsumption policy as S1.5a.18 (strict-subset removes
superset on emit).

### Equivalence for SOLVE on uniquely-solvable puzzles

Monotonic's `root.kb` at termination ≡ the lattice's
accumulated unconditional facts for uniquely-solvable
SOLVE-mode puzzles like Zebra. Argument in
[Q1.5b.7's "Monotonic vs lattice equivalence"](open_questions.md#monotonic-vs-lattice-equivalence)
subsection.

M1 SOLVE acceptance on Zebra is achievable with monotonic
alone — lattice is the additional shipping target, not the
M1-blocking one.
