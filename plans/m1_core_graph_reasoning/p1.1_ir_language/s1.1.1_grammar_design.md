# S1.1.1 — Grammar design

**Phase:** P1.1 (IR language).
**Estimate:** 5-6 days.
**Depends on:** nothing.
**Implements idea:** [01-self-modifying-constraint-language §point 3](../../../docs/ideas/01-self-modifying-constraint-language.md),
[04-nlp-to-graph-to-solver-pipeline](../../../docs/ideas/04-nlp-to-graph-to-solver-pipeline.md).
**Resolves question:** [global Q3](../../open_questions.md#q3--surface-ir-syntax).

## Context

The IR is the central artefact of the project — every layer reads
or writes it. Per [idea 01](../../../docs/ideas/01-self-modifying-constraint-language.md)
the design preference is **S-expression with heavy semantic load on
atoms**: small kernel, homoiconic, trivially GBNF-able (M2), trivial
to walk (M1 engine), trivially mappable to SMT-LIB (M3).

## Decisions to make

- **Atom convention**: hyphenated lowercase atoms
  (`has-color`, `next-to`) or plain underscores (`has_color`)?
  Recommendation: hyphens, Lisp-conventional.
- **Quoting / strings**: needed for source-sentence provenance and
  rule reason templates only — everything else is atoms.
- **Comments**: `;` line + `#| … |#` block, SMT-LIB-compatible.
- **Reserved heads**: a fixed top-level keyword set the parser
  recognises. Anything else is application-defined.
- **IR ↔ DOT isomorphism**: the IR has a structurally isomorphic
  DOT view (up to layout). Bidirectional; reverse parse required but
  not blocking on P1.1. See [Q21](../open_questions.md#q21).

## Tasks

### Task T1.1.1.1 — Enumerate top-level forms

Lock the kernel. Draft a single-page list. Working draft:

```
(ontology
  (type <Name> [<parent>])
  (instance <Name> <Type>)
  (relation <name> <SrcType> <DstType> [:cardinality (1..1|1..*|...)])
  (a-priori <name> <SrcType> <DstType> :pattern <pattern>))

(facts
  (= <expr> <expr>)               ;; equality fact
  (rel <name> <a> <b>)             ;; binary
  (hrel <name> <a> <b> <c> ...)    ;; hyper-relation (n-ary)
  (constraint <name> ...))         ;; named puzzle-level constraint

(rules
  (rule <name>
    :match <pattern>                ;; structural LHS
    :assert <conclusion>            ;; structural RHS
    :why <reason-template>          ;; string used by trace planner
    [:priority <int>]))

(query
  :mode (solve | gaps | contradictions)
  :goal <expr>)

(trace
  (step <n> :rule <name> :using (<premise-ids>) :derives <expr>
            :source <source-ref>))
```

Each form gets a worked example in `docs/ir.md`. Reject proposed
forms with no immediate user (YAGNI). The kernel should be
parseable in ~30 lines of code.

### Task T1.1.1.2 — Pattern sub-language

`:match` and `:assert` in `rule` need a pattern language. Smallest
useful version:

- variables prefixed `?` (`?a`, `?house`);
- ground atoms match themselves;
- `(_ x y ...)` matches a hyper-relation with head wildcard;
- `:where <predicate-list>` clause for type / inequality filters.

Spike a worked example for the **triangle rule**:

```
(rule triangle-composition
  :match (and (rel ?r ?a ?b)
              (rel ?r ?b ?c)
              :where (transitive ?r))
  :assert (rel ?r ?a ?c)
  :why "From {0} and {1}, since {?r} is transitive, {?a} {?r} {?c}.")
```

Decision artefact: a half-page in `docs/ir.md` §Pattern language.

### Task T1.1.1.3 — Survey existing dialects, pick what to steal

30-minute survey. Compare against:

- **SMT-LIB 2** — `(declare-const)` / `(assert)` / `(check-sat)`.
  We deliberately diverge by keeping rule definitions in-band.
- **miniKanren** — relational atoms; closer in spirit.
- **AtomSpace** — typed atoms + node labels. Heavy.
- **Datalog / Soufflé** — `head :- body.`-style clauses; *not*
  S-expression but informs `rule` shape.

Output: a 10-line table in `docs/ir.md` §Inspirations recording
what each contributed and what was rejected.

### Task T1.1.1.6 — Rendering schema (IR ↔ DOT)

**Resolves question:** [Q21](../open_questions.md#q21).

The IR is structurally isomorphic to a subset of DOT (the EBNF ↔
railroad-diagram move). Render is mandatory; reverse parse is required
but does not block P1.1 — it lands in P1.2 alongside the typed-hypergraph
data model. Only *graph structure* is fixed by the schema; layout
(positions, rank, unspecified style choices) is free — `random_layout`
is permitted.

Fixed by this task, written into `docs/ir.md` §Rendering:

- **Node shapes** (legend): types = boxes; instances = ovals; ground
  atoms = rectangles; hyperedges = octagons; equality classes =
  double-circles; variables (`?x`) = diamonds; wildcards (`_`) =
  anonymous diamonds.
- **Hyperedge encoding**: Levi/bipartite — every n-ary relation
  becomes a node with one edge to each participant; the hyperedge's
  identity is that node, which provenance ([Q18](../open_questions.md#q18))
  references. Anchors [Q1](../../open_questions.md#q1)'s
  typed-hypergraph answer visually.
- **Ontology rendering**: UML-ish — `type` declarations as boxes,
  `instance` as ovals, `relation` schemas as dashed edges, `a-priori`
  patterns with a distinct border.
- **Rule rendering** — three modes, configurable:
  - (a) Side-by-side LHS | RHS clusters — *default for `rules.ein`
    documentation*; most explicit when reading rule libraries.
  - (b) DPO span `L ← K → R` with shared interface K — categorical
    reading per [idea 07](../../../docs/ideas/07-categorical-formulation.md);
    available behind a flag.
  - (c) Overlay (LHS solid, RHS additions dashed) — *default at
    rule-firing time inside traces*; most compact.
- **Trace rendering** — three views, configurable:
  - (a) Per-step DOT snapshots — *default*; matches M1 acceptance §2's
    `zebra/` folder.
  - (b) Aggregate DOT — final-state image with edges coloured by step.
  - (c) Derivation DAG — nodes = derived facts, edges = rule
    applications; "explanation" view per
    [idea 08](../../../docs/ideas/08-human-style-deductive-trace.md).
- **Branch rendering** (P1.5 forward-reference): search-tree view as a
  tree of states; per-state snapshots use sub-clusters per branch.

Deliverable: §Rendering chapter in `docs/ir.md` with the node-shape
legend, a Zebra-fragment fact rendered both ways (IR + DOT), the
triangle rule rendered in all three modes, and a 3-step trace fragment
rendered in modes (a), (b), and (c).

*Implementation* lives in:
- `ein_bot.ir.to_dot` — minimum renderer per this schema, in
  [S1.1.4](s1.1.4_ir_to_dot.md) (carved out of S1.1.2 on
  2026-05-18; S1.1.2 stayed text-only).
- `ein_bot.ir.from_dot` — reverse parse, required, in P1.2.
- Renderer mode flags and per-context defaults — P1.6 S1.6.1.
