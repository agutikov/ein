# S1.1.4 ‚Äî `ein_bot.ir.to_dot` renderer

**Phase:** P1.1 (IR language).
**Estimate:** 2-3 days.
**Depends on:** S1.1.2 typed AST.
**Resolves question:** [Q21](../open_questions.md#q21) (IR ‚Üî DOT) ‚Äî
forward direction only; reverse parse (`from_dot`) still lands in P1.2.

## Context

Carved out from S1.1.2 on 2026-05-18: text serialisation
(`dump_canonical`) was the only deliverable that actually had to ship
with the parser. The DOT renderer wants a non-trivial encoder (Levi-
bipartite hyperedges, three rule modes, three trace views) and is
better treated as its own stage so reviewers can read it in
isolation.

The schema is already locked in
[`docs/ir.md` ¬ß6](../../../docs/ir.md). This stage is the
*implementation* of that schema.

The renderer lives in `src/ein_bot/ir/to_dot.py`. It is the *minimum*
DOT renderer that round-trips through the schema; the per-context
defaults (rules' (a)-vs-(c) by use-site, trace's (a)-vs-(b)-vs-(c) by
flag) and the rendering library proper land in
[P1.6 S1.6.1](../p1.6_rendering_and_trace/s1.6.1_dot_rules_constraints.md).

## Tasks

### Task T1.1.4.1 ‚Äî Node-shape mapping

Pure function `_dot_shape(node: IRNode) -> str`. One line per node
type, matching the legend in `docs/ir.md` ¬ß6 "Node-shape legend".
Types ‚Üí `box`, instances ‚Üí `ellipse`, atoms ‚Üí `rectangle`,
hyperedges ‚Üí `octagon`, equality classes ‚Üí `doublecircle`, `Var` ‚Üí
`diamond`, `Wildcard` ‚Üí `diamond style=dashed`. No I/O.

### Task T1.1.4.2 ‚Äî Form renderers

Five public functions, one per top-level form:

```python
def render_ontology(form: SForm) -> str: ...
def render_facts(form: SForm) -> str: ...
def render_rule(rule_sform: SForm, *, mode: Literal["a","c"] = "c") -> str: ...
def render_trace(form: SForm, *, view: Literal["a","b","c"] = "c") -> str: ...
def render_query(form: SForm) -> str: ...
```

Each returns a complete `digraph { ... }` string. Mode `b` (DPO span)
is **out of scope** here ‚Äî the categorical view lands in P1.6.

`render_facts` uses the Levi-bipartite encoding for any n-ary `(NAME
a b c ‚Ä¶)`: emit an octagon node `hN` for the hyperedge identity,
then edges `hN ‚Üí a [label="1"]`, `hN ‚Üí b [label="2"]`, ‚Ä¶. Equality
facts (`= L R`) collapse to a `doublecircle` node linking the two
sides.

`render_rule` mode (a) uses two `subgraph cluster_lhs` / `cluster_rhs`
blocks; mode (c) emits both halves in one digraph with RHS-only
edges marked `style=dashed`.

### Task T1.1.4.3 ‚Äî Top-level `to_dot(node, **kwargs)`

Dispatches on `node.head.name` to the right form renderer. Returns
a complete DOT string. For a tuple of top-level forms, returns one
digraph per form joined by blank lines.

### Task T1.1.4.4 ‚Äî CLI wiring

Extend the existing subparser: `ein-bot ir dot <file>` reads the
file, parses, renders every form to DOT, writes to stdout. Flags
`--rule-mode=a|c` and `--trace-view=a|c` for the configurable
renderings.

### Task T1.1.4.5 ‚Äî Tests

`tests/test_ir_to_dot.py`. Cover:

1. **Shape mapping** ‚Äî each node type produces its documented shape.
2. **Per-form snapshot** ‚Äî `examples/zebra.ein` parsed and rendered;
   the digraph contains the expected node count + a hand-picked
   set of edges (`Norwegian -h_co-located_10-> House_1`).
3. **`graphviz` parse-back** ‚Äî pipe output through `dot -Tcanon` (skip
   if `dot` not in `$PATH`) and assert exit 0; this catches malformed
   DOT without committing to layout.
4. **Rule modes (a) and (c)** ‚Äî for the `transitive` rule, both modes
   produce parseable DOT and (c) contains a dashed edge.

## Acceptance

- `to_dot(parse(zebra.ein))` returns a non-empty DOT string that
  `dot -Tcanon` accepts without error.
- Every node-shape rule in `docs/ir.md` ¬ß6 has at least one test.
- `ein-bot ir dot examples/zebra.ein | dot -Tsvg > /tmp/z.svg`
  produces a non-empty SVG (manual check; not in CI).
- The reverse direction (`from_dot`) is *out of scope*; it lands in
  P1.2 alongside the typed-hypergraph data model.
