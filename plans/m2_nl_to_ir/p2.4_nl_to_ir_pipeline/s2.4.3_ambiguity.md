# S2.4.3 — Ambiguity-as-hypothesis emission

**Phase:** P2.4 (NL → IR pipeline).
**Estimate:** 3-4 days.
**Depends on:** S2.4.2.
**Resolves:** [Q8 (final implementation)](../../open_questions.md#q8--ambiguous-parses).

## Context

[Q8 working answer](../../open_questions.md#q8--ambiguous-parses):
ambiguous parses become parse-level hypotheses on the IR. This
stage *produces* those hypotheses from the LLM and *threads* them
into the rest of the pipeline.

The shape (decided in P2.1 S2.1.2):

```ein
(hypothesis-parse
  :id parse-1
  :sentence "The Norwegian lives in the first house."
  :alternatives
    ((rel lives-in Norwegian House_1)
     (at-position Norwegian 1)))
```

For Zebra the alternatives are semantically equivalent, so the M1
engine collapses them quickly via the saturation closure. For less
constrained puzzles ("the second house has a red door, or the
third does"), the alternatives are genuinely distinct.

## Tasks

### Task T2.4.3.1 — Ambiguity-aware facts grammar

Build on `grammars/facts.gbnf` from P2.3 S2.3.2 T2.3.2.2: allow
the LLM to wrap any sentence's facts in a
`(hypothesis-parse …)` when it's not confident which form to use.
The prompt explicitly says *"if a sentence has multiple plausible
readings, emit both as alternatives."*

### Task T2.4.3.2 — Promote ambiguities to engine hypotheses

`src/ein_bot/nl/promote_ambiguity.py`. After extraction +
validation:

- Wrap each `(hypothesis-parse …)` as a level-0 hypothesis in
  the engine's hypothesis machinery (M1.P1.5).
- Add a `Provenance(kind='parse-hypothesis', branch=parse_id)` so
  the trace can name which parse choice survived.
- Promote the IR to ein-bot's standard format the engine consumes.

### Task T2.4.3.3 — Constraint-free reading detector

Heuristic: a sentence containing "or", "either…or", "maybe", or
trailing question-marks gets a special pre-prompt flag asking the
LLM to consider ambiguous readings explicitly. Cheap; no model
fine-tuning.

### Task T2.4.3.4 — Tests

`tests/nl/test_ambiguity.py`:

- A puzzle sentence with "or" produces a multi-alternative
  hypothesis-parse.
- The M1 engine recognises and prunes inconsistent alternatives.
- A puzzle where exactly one alternative survives has its
  surviving alternative recoverable from the trace.

## Acceptance

- A test puzzle with a known-ambiguous sentence is parsed to two
  alternatives; the engine narrows to one and reports the
  discarded one in the trace.
- The flagging heuristic doesn't generate spurious ambiguities on
  the canonical Zebra text.
